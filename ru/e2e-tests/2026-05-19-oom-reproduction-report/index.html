Отчёт об OOM-стресс-тестировании и воспроизведении длительных задач

Дата: 2026-05-19 Ветка: codex/memory-diagnostics-local-run Тестировал: yiliang114 Вывод: Успешно воспроизведено и найдена первопричина. В версии v0.15.7 (#3735) введённый auto-compaction многократно увеличил частоту вызовов structuredClone, что при высоком давлении на heap привело к положительной обратной связи и OOM. Реальные debug-логи полностью подтверждают этот механизм.

1. Предыстория

Несколько issue (#4309, #4276, #4185, #4315, #4322, #2868) сообщают о V8 heap OOM в qwen-code при длительных сессиях:

FATAL ERROR: Ineffective mark-compacts near heap limit Allocation failed - JavaScript heap out of memory

Характеристики крашей, сообщённых пользователями:

IssueHeap при крашеВремя работыПлатформа
#42764014 MB~110 минLinux x64
#43152027 MB~19.6 чmacOS (2GB по умолчанию)
#43224023 MB~7 чWindows
#28682035 MB~1.7 минLinux
#43097020 MBнеизвестноWindows (установлен лимит 8GB, всё равно краш)

2. Исправление методологии

Данный отчёт различает два типа тестов:

  1. Тесты с низким давлением на heap: уменьшение --max-old-space-size для усиления проблемы и быстрой локализации кодового пути «копирование целого сегмента при большом history, вызывающее мгновенный пик». Это диагностический инструмент, он не эквивалентен воспроизведению пользовательского OOM на 4G/8G.
  2. Воспроизведение длительных задач с heap по умолчанию: без установки NODE_OPTIONS, восстановление из реального JSONL-истории и продолжение выполнения задачи review с внешней выборкой process-tree RSS. Только такие результаты используются для оценки реального объёма памяти на стороне пользователя.

Следовательно, результаты теста с низким heap не могут сами по себе служить доказательством, что «реальный OOM исправлен». Они лишь показывают, что при достаточно большом history определённый путь может создавать пиковое увеличение, что требует проверки с помощью воспроизведения длительных задач с heap по умолчанию.

3. Условия теста с низким давлением на heap

ПараметрЗначение
Версия CLI0.15.11 (собрано из ветки codex/memory-diagnostics-local-run)
Модельqwen3.6-plus (128K контекстное окно)
Лимит heap--max-old-space-size=512
Предохранитель давления на heapОтключён (HEAP_PRESSURE_COMPRESSION_RATIO установлен в 99.0)
Режим работыYOLO + автоматические многораундовые задачи чтения файлов
Рабочий каталогqwen-code monorepo (3538 .ts файлов, 1.26M строк)

Ключевые изменения конфигурации

В packages/core/src/core/geminiChat.ts порог compaction по давлению heap изменён с 0.7 на 99.0 (чтобы он никогда не срабатывал), имитируя состояние до исправления #4186.

[21:27:46] #2 RSS:270.4MB Ctx:4.2% → Чтение agent.ts
[21:28:32] #3 RSS:397.5MB Ctx:4.3% → grep + Чтение 3 файлов
[21:29:18] #4 RSS:452.7MB Ctx:5.7% → Чтение slashCommandProcessor.ts

Общее время: ~5.5 минут, краш после 7 раундов задач.

Это доказывает, что при ограниченном heap длинный history + compaction/клонирование history может вызвать OOM V8 heap. Однако эти результаты не означают, что реальный пользовательский OOM при heap по умолчанию полностью воспроизведён.

Синтетическое воспроизведение с большим heap

5. Воспроизведение длительных задач с heap по умолчанию

Чтобы избежать чрезмерного обобщения результатов с низким heap, было проведено дополнительное воспроизведение с реальным JSONL и heap по умолчанию:

  • NODE_OPTIONS не устанавливались
  • Внутренний профилировщик времени выполнения не включался, чтобы избежать влияния сэмплера на heap
  • Каждый CLI создаёт свежую сессию из той же перемотанной JSONL-копии
  • Используется временный QWEN_HOME, отключены MCP и хуки, чтобы избежать загрязнения локальными глобальными конфигурациями
  • Используется только внешняя выборка process-tree RSS

Выводы из воспроизведения с heap по умолчанию:

  1. Данный JSONL-файл review стабильно даёт от нескольких сотен MiB до ~0.8 GiB process-tree RSS, но OOM на 4G/8G не воспроизводится.
  2. Локально пересобранный bundle показывает более низкий пик при том же воспроизведении, что говорит о реальном выигрыше от сокращения горячего пути клонирования history.
  3. Это ещё не доказывает, что все пользовательские OOM исправлены. Реальный OOM на 4G/8G всё ещё требует более длительных задач, большего накопления tool-result или воспроизведения с сохранением давления MCP/tool schema.

6. Анализ первопричины

Трёхуровневый механизм OOM

┌─────────────────────────────────────────────────────────┐
│ Уровень 3: Лимит V8 Heap (512MB/2GB/4GB)               │ ← Пользователь в итоге упирается сюда
├─────────────────────────────────────────────────────────┤
│ Уровень 2: Пик structuredClone() (мгновенное ~2x)      │ ← Непосредственная причина
├─────────────────────────────────────────────────────────┤
│ Уровень 1: Накопление tool result в истории (линейный рост)│ ← Базовый рост
├─────────────────────────────────────────────────────────┤
│ Уровень 0: Момент срабатывания compaction токенов       │ ← Точка контроля
└─────────────────────────────────────────────────────────┘

Точный путь краша

sendMessage()
  → tryCompress()
    → heapPressureRatio < threshold (safety net disabled)
    → ChatCompressionService.compress()
      → chat.getHistory(true)
        → structuredClone(this._history)   ← Пиковое выделение!
          → V8 требуется дополнительно ~N MB для размещения клона
          → Если existing heap + N > limit → OOM

Ключевые доказательства

НаблюдениеЗначение
Задача #5→#6: Context 5.9%→4.0% (снизился)Token compaction успешно выполнен
Задача #5→#6: RSS 515→649 MB (вырос на 134MB)structuredClone во время compaction создал пик
GC может освободить только 1–2 MBВсе объекты живые (и history, и клон)
#4309: краш даже с лимитом 8GBПри достаточно большом history пик клона может превысить любой лимит

Важно: приведённые выше доказательства основаны на комбинации тестов с низким heap и наблюдений из issue. Воспроизведение с heap по умолчанию пока подтверждает, что «горячий путь клонирования значительно влияет на пиковый RSS», но отдельно OOM на 4G/8G пока не воспроизведён.

Почему 128K контекстное окно легче вызывает проблему

  • 128K × 70% = ~90K токенов, что запускает compaction
  • Большое контекстное окно (1M) × 70% = 700K токенов — почти никогда не запускает
  • Чем чаще compaction → тем чаще structuredClone → тем выше риск OOM
  • DeepSeek и другие модели без настройки contextWindowSize по умолчанию имеют 128K, что делает их более уязвимыми

6.6. Версионная привязка: почему количество OOM-сообщений увеличилось между версиями 0.15.7 и 0.15.11

Временная шкала ключевых коммитов

ВерсияPRИзменениеВлияние на частоту вызовов structuredClone
v0.15.6—getHistory(true) вызывается только 1 раз при входе в sendMessageБазовый уровень: 1 клон на send
v0.15.7#3735 auto-compact subagent contexttryCompress() опущен в GeminiChat, выполняется перед каждым send+1 вызов: проверка compress перед send
v0.15.10#3879 reactive compression on context overflowКогда провайдер возвращает context overflow, снова запускается tryCompress() + getHistory(true)+1~2 вызова: путь повторных попыток при overflow
v0.15.10#3985 harden reactive compressionУсилена логика повторных попыток реактивного сжатияТо же, что и выше

Сравнение точек вызова getHistory(true) в v0.15.6 и v0.15.11

v0.15.6 (2 места):

L367: const requestContents = this.getHistory(true);          ← send формирует request
L618: const recoveryContents = self.getHistory(true);         ← эскалация MAX_TOKENS (срабатывает редко)

v0.15.11 (5 мест):

L467: внутри ChatCompressionService.compress()                 ← #3735: авто-compact перед каждым send
L574: requestContents = this.getHistory(true);                 ← send формирует request
L724: внутри reactive tryCompress()                           ← #3879: повторная попытка после context overflow
L739: requestContents = self.getHistory(true);                 ← #3879: повторная попытка формирует новый request
L943: const recoveryContents = self.getHistory(true);          ← эскалация MAX_TOKENS

Наихудший путь: один send может вызвать 4 вызова structuredClone

sendMessage()
  → tryCompress()              ← #3735: getHistory(true) [клон #1]
  → getHistory(true)           ← формирование request [клон #2]
  → API вернул context overflow
    → reactive tryCompress()   ← #3879: getHistory(true) [клон #3]
    → getHistory(true)         ← повторный request [клон #4]

Вывод

#3735 (v0.15.7) является наиболее вероятным триггером (не единственной первопричиной) значительного увеличения частоты OOM — он приводит к тому, что каждый sendMessage сначала выполняет tryCompress(), а внутри tryCompress через ChatCompressionService.compress() → chat.getHistory(true) выполняется полное structuredClone. При большом history такой дизайн «сначала клонировать, потом решать, нужно ли сжимать» увеличивает пик памяти с ~1.3x до ~2x+. Примечание: issue history показывает, что сообщения об OOM существовали и до #3735, но #3735 существенно увеличил частоту вызовов structuredClone, тем самым значительно повысив вероятность OOM.

#3879 (v0.15.10) ещё больше усугубил проблему — когда heap уже на границе (провайдер вернул context overflow), запускается ещё одно полное клонирование, что делает и без того опасную сессию ещё более склонной к крашу.

9. Скрипт воспроизведения и окружение

Автоматизированный скрипт-драйвер

#!/bin/bash
# /tmp/oom-simple-driver.sh <tmux-session-name>
SESSION="$1"
 
TASKS=(
  "Прочитать полностью с помощью Read-инструмента packages/core/src/core/geminiChat.ts"
  "Прочитать полностью с помощью Read-инструмента packages/core/src/tools/agent/agent.ts"
  "Выполнить grep -rn structuredClone packages/core/src, затем Read первые 3 файла"
  "Прочитать полностью с помощью Read-инструмента packages/cli/src/ui/hooks/slashCommandProcessor.ts"
  "Прочитать полностью с помощью Read-инструмента packages/core/src/services/chatCompressionService.ts"
  "Выполнить find packages/cli/src/ui/commands -name '*.ts', затем Read каждый по очереди"
  "Прочитать полностью с помощью Read-инструмента packages/core/src/core/turn.ts"
  # ... ещё задачи
)
 
i=0
while true; do
  TASK="${TASKS[$((i % ${#TASKS[@]}))]}"
  i=$((i + 1))
 
  QWEN_PID=$(ps aux | grep "dist/index.js" | grep -v grep | awk '{print $2}' | sort -rn | head -1)
  RSS=$(ps -o rss= -p $QWEN_PID 2>/dev/null)
  [ -z "$RSS" ] && { echo "CRASH after $((i-1)) tasks!"; exit 0; }
 
  RSS_MB=$(echo "scale=1; $RSS/1024" | bc)
  CTX=$(tmux capture-pane -t "$SESSION:1" -p 2>/dev/null | grep -oE "[0-9]+\.[0-9]+% 已用" | tail -1)
  echo "[$(date +%H:%M:%S)] #$i RSS:${RSS_MB}MB Ctx:$CTX | ${TASK:0:55}"
 
  tmux send-keys -t "$SESSION:1" C-u
  sleep 0.2
  tmux send-keys -t "$SESSION:1" "$TASK" Enter
  sleep 0.5
  tmux send-keys -t "$SESSION:1" Enter
  sleep 45
done

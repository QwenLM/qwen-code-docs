Структурированный вывод (--json-schema) — Дизайн

Этот документ описывает решения по реализации функции безголового режима --json-schema. Информация для пользователей находится в docs/users/features/structured-output.md.

Цель

В безголовых запусках (qwen -p, конвейерный stdin или позиционный промпт) дать вызывающей стороне возможность ограничить итоговый ответ модели заданной JSON-схемой и вывести проверенный payload в машиночитаемом виде, который можно напрямую использовать в скриптах и последующих инструментах. Сопутствующая проза модели во время планирования допускается, но выполнение должно завершаться payload, соответствующим схеме, а не произвольным текстом.

Подход: синтетический инструмент, схема параметров которого совпадает со схемой пользователя

Когда установлен --json-schema, Config.createToolRegistry регистрирует синтетический инструмент structured_output (syntheticOutput.ts). Его parametersJsonSchema в точности равна схеме, переданной пользователем; его execute() возвращает стоп-сообщение llmContent. Инфраструктура вызовов инструментов уже проверяет аргументы на соответствие parametersJsonSchema на стороне клиента (через Ajv в BaseDeclarativeTool.build()), так что «модель вернула ответ, соответствующий схеме» сводится к «модель успешно вызвала structured_output».

Из этого автоматически вытекают три свойства:

  1. Нет специального пути валидации. validateToolParams на основе Ajv уже выполняется внутри BaseDeclarativeTool.build() и отклоняет несоответствующие аргументы до того, как execute() когда-либо вызывается.
  2. Стандартное поведение повторных попыток. Ошибка валидации передаётся модели как ошибка вызова инструмента, точно так же, как ошибки аргументов любого другого инструмента. Модель видит сообщение Ajv и может исправиться на следующем шаге.
  3. Независимость от провайдера. Gemini, OpenAI и Anthropic сериализуют схемы параметров инструментов одинаково (через абстракцию DeclarativeTool); синтетический инструмент работает со всеми тремя.

Инструмент регистрируется с флагом alwaysLoad: true, чтобы инфраструктура загрузки по требованию ToolSearch (введена в #3589 — уменьшает поверхность доступных инструментов, откладывая редко используемые инструменты до поискового запроса и загружая их полные схемы только когда модель их запрашивает) никогда не скрывала его от модели. Без этого флага модель не знала бы о существовании терминального контракта.

Конвейер проверки на этапе разбора

resolveJsonSchemaArg(raw) в packages/cli/src/config/config.ts выполняет четыре проверки до того, как схема достигнет Config.createToolRegistry:

  1. Разрешение источника. Принимает либо встроенный JSON-литерал, либо @путь/к/файлу. Форма @путь сначала проверяет путь через stat, отказывает в нерегулярных файлах (FIFO, символьные устройства, каталоги), ограничивает размер 4 МиБ, а при неудаче разбора JSON выдаёт общую ошибку (без префикса содержимого файла в stderr).
  2. Форма JSON. Результат разбора должен быть объектом, не массивом — примитивы, логические значения и массивы отклоняются с понятным сообщением.
  3. Корень принимает объекты — schemaRootAcceptsObject. API вызова функций всегда передают объекты в качестве аргументов инструмента; корневая схема типа {type: "array"} привела бы к регистрации бесполезного инструмента. Обход обрабатывает type, const, enum, anyOf, oneOf, allOf, not, if / then / else и корневой $ref.
  4. Строгая компиляция Ajv — SchemaValidator.compileStrict. Выделенный экземпляр Ajv с strictSchema: true обнаруживает опечатки вроде propertees, которые снисходительный рантайм-валидатор молча бы проглотил.

Границы schemaRootAcceptsObject

Обход намеренно выполняет проверку по мере возможности. Он перехватывает однозначные случаи «это никогда не может принять объект», а всё, что требует анализа выполнимости всей схемы, откладывает на Ajv во время выполнения.

Решается на этапе разбора:

ШаблонРезультат
type присутствует, не включает "object"отклоняется
type: ["object", "null"] и т.д.принимается
const: не объектное значениеотклоняется
enum: нет объектных элементов (включая пустой)отклоняется
anyOf/oneOf: пустой массивотклоняется
anyOf/oneOf: ни одна ветвь не допускает объектотклоняется
allOf: любая ветвь равна false или отвергает объектотклоняется
Корневой $ref (с или без соседнего type)отклоняется
not: голое {type: "object"} (без сужающих ключевых слов)отклоняется
not: {type: "object", required: […], …} и т.д.принимается (сужающие ключевые слова оставляют некоторые объекты выполнимыми; откладывается)
if: true + then отвергает объектотклоняется
if: false + else отвергает объектотклоняется
Отложено до выполнения в Ajv:
  • $ref внутри ветвей anyOf / oneOf / allOf (непрозрачно — локальное разрешение $ref потребовало бы обнаружения циклов, экранирования JSON Pointer и обработки $defs vs definitions; затраты перевешивают выгоду для проверки во время парсинга «по возможности»).
  • if, значение которого — схема-объект (разрешимо только относительно конкретного значения).
  • Отрицательные шаблоны anyOf / oneOf / const, более сложные, чем not.type.
  • Произвольный pattern, потенциально уязвимый для ReDoS (предоставляется пользователем; модель угроз узкая, так как флаг — аргумент CLI, а не сетевой ввод).

Точка выхода maxSessionTurns добавляет подсказку, специфичную для --json-schema, указывающую пользователям на распространённый симптом «зависшего» выполнения (модель ни разу не вызвала structured_output) и две его вероятных причины (инструмент запрещён правами / схема невыполнима), чтобы у выполнения во время выполнения были видимые пользователю диагностические сообщения.

Выполнение: диспетчеризация тактов

packages/cli/src/nonInteractiveCli.ts обрабатывает диспетчеризацию во время выполнения. Специфика структурированного вывода:

Предварительное сканирование + подавление «родственных» вызовов

Когда модель отправляет structured_output вместе с другими инструментами в одном такте ассистента, синтезированный вызов является терминальным контрактом. Предварительное сканирование в processToolCallBatch фильтрует requestsToExecute, оставляя только вызовы structured_output, поэтому побочные «родственные» вызовы (write_file, run_shell_command, edit, …) никогда не выполняются.

Примеры пакетов (когда включён --json-schema):

Модель выдаётПоведение
[write_file(…), structured_output(…)]write_file пропускается. structured_output проверяется, выполнение завершается.
[structured_output(плохие-аргументы), structured_output(хорошие)]Первый не проходит проверку Ajv; второй проходит. Выполнение завершается с аргументами второго вызова.
[structured_output(плохие-аргументы), write_file(…)]structured_output(плохие) не проходит. write_file также пропускается (он был подавлен заранее). Модель видит оба: сообщение об ошибке Ajv для структурированного вызова и синтезированный "Skipped: …" tool_result для побочного вызова. На следующем такте модель может повторно отправить оба или исправить только структурированный вызов.
[other_tool_a, other_tool_b] (нет structured_output)Предварительное сканирование неактивно. Оба инструмента выполняются нормально; выполнение НЕ завершается.

Синтезированное тело «Skipped:» имеет два варианта:

  • Путь успеха (структурированный вызов захватил контракт в этом такте): "Skipped: this turn's structured_output contract took precedence as the terminal output." — короткое, так как сессия завершается немедленно, и ни один потребитель (модель или SDK) на него не реагирует.
  • Путь повторной попытки (ни один структурированный вызов не захвачен, модель получает ещё один такт): добавляется "Re-issue this call in a separate turn if needed." — это единственный случай, на который может повлиять модель.

Равенство основного такта / такта «опустошения»

processToolCallBatch(batchRequests, setModelOverride) определена внутри runNonInteractive и вызывается из:

  • Цикла основного такта (в начале функции).
  • drainOneItem (цикл ответов на cron-prompt / фоновые уведомления).

Такт «опустошения» важен, потому что structured_output зарегистрирован на всю сессию, поэтому cron-задача или ответ на уведомление МОГУТ также вызвать этот инструмент. Обработчик обрабатывает обе точки вызова одинаково на момент вызова; единственная привязка, специфичная для точки вызова, — это то, в какую переменную modelOverride производится запись — передаётся как сеттер.

Поток завершения после обработчика различается в двух местах: путь основного такта напрямую вызывает return emitStructuredSuccess(), тогда как путь такта «опустошения» требует двухшагового завершения (processToolCallBatch помещает результат в замыкающую переменную structuredSubmission; drainLocalQueue проверяет её, чтобы остановить цикл опустошения, затем контрольный цикл проверяет её, чтобы выйти и вызвать emitStructuredSuccess). Оба сходятся на одном и том же терминальном блоке, но дополнительная косвенность в пути «опустошения» важна — без неё цикл опустошения продолжал бы обрабатывать поставленные в очередь элементы после захвата структурированного результата.

Структурированный терминальный блок успеха

emitStructuredSuccess() (также определён внутри runNonInteractive) — это общий путь «получен допустимый вызов, завершаем работу»:

  1. registry.abortAll() прерывает фоновые агенты, находящиеся в полёте, — контракт структурированного вывода рассчитан на один выстрел и не должен гоняться с task_notification в терминальный emit.
  2. Ограниченная задержка (STRUCTURED_SHUTDOWN_HOLDBACK_MS = 500 мс), чтобы естественные обработчики отмены только что прерванных агентов успели выдать свой терминальный task_notification и поместить его в localQueue. Условие цикла — Date.now() < deadline && registry.hasUnfinalizedTasks(), так что ожидание завершается немедленно, когда ничего не выполняется (типичный путь), и никогда не блокируется дольше установленного лимита. Потолок в 500 мс — это best-effort; при нагрузке возможны осиротевшие события task_started, если обработчик прерывания конкретного агента превышает бюджет. Цикл не опрашивает сигнал прерывания: SIGINT, полученный во время задержки или во время последующего emit, не прервёт уже захваченный результат. Без такой задержки потребители stream-json регулярно видели бы события task_started без соответствующего task_notification.
  3. flushQueuedNotificationsToSdk(localQueue) опустошает всё, что ещё находится в очереди.
  4. finalizeOneShotMonitors() (идемпотентна — безопасно вызывать дважды; путь дренажа-поворота уже вызывал её).
  5. adapter.emitResult({ structuredResult: …, isError: false, … }).

Выходной конверт

Конвейер адаптера в BaseJsonOutputAdapter.buildResultMessage трактует наличие structuredResult (отслеживается через 'structuredResult' in options, а не !== undefined, так что контракт сохраняется даже если модель вызвала structured_output без аргументов под пустой схемой):

  • result принудительно приводится к JSON.stringify(payload) — переопределяя любое свободное текстовое резюме, накопленное адаптером.
  • Поле верхнего уровня structured_result несёт сырой объект для потребителей, которые не хотят повторно разбирать строковую форму.
  • undefined payload нормализуется в null (отображается как литеральный JSON null в обоих полях), так что поле не может молча исчезнуть. На практике этот запасной вариант редко достигается: выше по конвейеру, turn.ts применяет (fnCall.args || {}) перед сохранением отправленного, так что вызов без аргументов для пустой схемы попадает как {} и отображается как {} в stdout, а не null. Шаг ?? null — это защита в глубину для строго случая undefined.

Режим TEXT записывает в stdout только поле result + перевод строки (любая случайная проза ассистента, накопленная во время выполнения, отбрасывается — не отражается в stderr). Режим JSON выводит полный журнал событий в виде JSON-массива; structured_result находится на последнем элементе с type: "result" этого массива, а не в корне документа. Режим stream-json выводит каждое сообщение отдельной строкой в формате JSONL; завершающая строка result несёт structured_result.

Приватность: сокращение данных на всех поверхностях

Аргументы, переданные через structured_output, являются структурированной полезной нагрузкой. В случае успешного выполнения они уже оказываются в stdout; при повторных попытках из-за ошибок валидации они могут вообще не попасть в stdout. В любом случае их сохранение на долговременных устройствах (или экспорт за пределы устройства через телеметрию) — это дублирование, которое «утекает» полезная нагрузка в хранилище с более долгим сроком жизни, чем запрашивал пользователь. Поэтому правило сокращения звучит так: «никогда не сохранять никакие аргументы этого синтетического инструмента, независимо от результата», а не просто «удалить дубликаты того, что уже есть в stdout».

Две поверхности должны выполнять сокращение, и обе используют один и тот же константный заполнитель STRUCTURED_OUTPUT_REDACTED_ARGS:

  • ToolCallEvent.function_args (телеметрия) — охватывает экспорт по OTLP, QwenLogger, ui-telemetry и UI-событие записи чата.
  • redactStructuredOutputArgsForRecording (используется recordAssistantTurn в geminiChat.ts) — охватывает JSONL-запись чата на диске по пути ~/.qwen/projects/<sanitized-cwd>/chats/<sessionId>.jsonl. Повторные попытки при ошибках валидации также попадают сюда — аргументы каждой повторной попытки получают тот же заполнитель.

Общая константа предотвращает расхождение между двумя поверхностями. Метрики вызова инструментов (длительность, успешность, решение) сохраняются.

Хуки (PreToolUse, PostToolUse, PostToolUseFailure) намеренно не сокращаются — они получают сырой tool_input, потому что контракт хука гласит «видеть то же, что видит инструмент». Это задокументировано в разделе «Приватность» пользовательской документации как примечание «Хуки видят сырые аргументы», чтобы операторы могли фильтровать по tool_name или добавлять сокращение на стороне хука перед запуском --json-schema с конфиденциальными данными.

Сокращение намеренно ограничено локальными накопителями данных (экспорт телеметрии + JSONL-запись чата). Сама схема по-прежнему отправляется провайдеру модели в каждом запросе как блок parameters объявления функции structured_output — сокращение на стороне провайдера невозможно, так как модели нужна схема для выполнения контракта вызова инструмента. Раздел «Приватность» пользовательской документации предупреждает пользователей по той же причине держать enum / const / default / examples / description свободными от секретов.

Контроль разрешений

structured_output намеренно исключён из PermissionManager.CORE_TOOLS (набора инструментов, подлежащих проверке по белому списку --core-tools) — вместе с другими синтетическими инструментами (agent, exit_plan_mode, ask_user_question, task_stop, send_message). Динамически обнаруживаемые инструменты (skill, MCP) относятся к отдельной категории исключений, которые также обходят белый список по независимым причинам. Синтетический инструмент существует только при установленном --json-schema; добавление его в механизм белого списка привело бы к тому, что --core-tools read_file --json-schema X молча отбрасывал бы терминальный контракт.

Явные правила permissions.deny и настройки --exclude-tools по-прежнему применяются через PermissionManager.evaluate → isToolEnabled. Оба используют один и тот же механизм запрета и оба предотвращают регистрацию — объявление инструмента удаляется из реестра, поэтому модель никогда не видит инструмент. Типичный результат: модель отвечает обычным текстом (код возврата 1). Если модель зацикливается на других инструментах, не выдавая текст, она в конечном итоге достигает maxSessionTurns (код возврата 53), и подсказка --json-schema в handleMaxTurnsExceededError указывает пользователю, где искать.

Взаимодействие с --bare. Безрежимный режим обходит мост настроек в конфигурацию CLI: packages/cli/src/config/config.ts формирует mergedDeny как [...(bareMode ? [] : settings.permissions.deny), ...], поэтому запреты уровня настроек (и tools.exclude) отбрасываются при --bare. --exclude-tools уровня аргументов безусловно добавляется в mergedDeny, поэтому он по-прежнему применяется. Синтетический инструмент регистрируется независимо от всего этого (управляется jsonSchema, а не списком запретов), поэтому запрет только в настройках для structured_output молча игнорируется при --bare, а инструмент остаётся вызываемым.

Контексты под-агентов

Config.createToolRegistry принимает опцию forSubAgent: true, которая подавляет регистрацию синтетического инструмента. Переопределения под-агентов повторно используют родительский Config через делегирование прототипа (createApprovalModeOverride / buildSubagentContextOverride → Object.create(base)), и this.jsonSchema распространяется по цепочке прототипов. Без этого флага синтетический инструмент регистрировался бы и в реестре под-агента, и вызов его под-агентом приводил бы к получению llmContent «сессия завершается сейчас» — но только основной / сливной циклы runNonInteractive распознают это как терминальное, поэтому под-агент продолжал бы работать и сжигал бы токены на инструменте, чей контракт его цикл не может выполнить.

Примечание для разработчика. Это подавление привязано к единственному пути вызова через createToolRegistry(forSubAgent: true). Любой будущий механизм порождения под-агентов, обходящий этот путь, «утечёт» синтетический инструмент в реестр под-агента и заново внесёт режим отказа с бесконечным сжиганием токенов. Отказоустойчивым дополнением была бы runtime-защита внутри syntheticOutput.execute(), которая возвращает fatalError (или no-op) при вызове из контекста под-агента. Добавьте её, если появится второй путь утечки.

Защита теневого инструмента MCP

tool-registry.ts:registerTool проверяет ленивую карту factories на коллизии имён, а не только активную карту tools. Если MCP-сервер обнаруживает инструмент, буквально названный structured_output, путь автоматического уточнения, существующий для коллизий активных инструментов, срабатывает и для коллизий фабрик: MCP-инструмент переименовывается в mcp__<server>__structured_output, а синтетическая фабрика сохраняет голое имя. Без этой защиты MCP-сервер мог бы незаметно перехватить контракт structured-output.

Рассмотренные альтернативы

Подсказка ответа с учётом схемы (без синтетического инструмента). Запрос модели «ответь JSON, соответствующим этой схеме» через системный prompt и последующий разбор финального сообщения ассистента. Отклонено, поскольку у модели нет синтаксической гарантии — вывод может быть заключён в блок кода, содержать лишние пояснения или галлюцинировать поля. Валидация вызова инструмента принудительно выполняется слоем вызова функций до execute(), что даёт жёсткую синтаксическую + семантическую защиту.

OpenAI response_format: {type: "json_schema", …}. Привязано к конкретному провайдеру; потребовало бы параллельных реализаций для Gemini и Anthropic. Подход с синтетическим инструментом не зависит от провайдера.

Перенести structured_output в начало пакета вместо фильтрации. Позволяет родственным инструментам с побочными эффектами выполняться, если structured-вызов не прошёл валидацию. Отклонено, поскольку контракт для --json-schema — «создать структурированный вывод» — если модель находится в этом режиме, родственные побочные эффекты, вероятно, являются ошибкой. Полное их подавление безопаснее; модель видит “Skipped:” tool_result и может перевыпустить их в отдельном шаге.

Локальное разрешение $ref внутри schemaRootAcceptsObject. Позволило бы отлавливать схемы типа {anyOf: [{$ref: "#/$defs/String"}], $defs: {…}} на этапе парсинга. Отклонено на данный момент, так как затраты (обнаружение циклов, синтаксис JSON Pointer, $defs vs definitions, частичные указатели, удалённые ссылки) перевешивают выгоду; подсказка maxSessionTurns уже указывает пользователям на «схема неудовлетворима» как вероятную причину.

Открытые задачи

  • Валидация ответа с учётом схемы может получить защиту от ReDoS на основе pattern, если реальные пользователи столкнутся с катастрофическим возвратом (backtracking) в аргументах --json-schema.
  • Дополнения протокола SDK (Python / TypeScript / Java SDK, предоставляющие типизированное поле structured_result) — отслеживать отдельно; PR #4001  (закрыт без слияния 11.05.2026) охватывал эту область до завершения работ по cli/core и был заменён.

Дизайн сводки сессии

Краткое резюме (1–2 предложения) «на чём я остановился», которое появляется, когда пользователь возвращается к неактивной сессии — либо по запросу (/recap), либо после того, как терминал был размыт на 5+ минут.

Обзор

Когда пользователь /resume старую сессию спустя несколько дней, пролистывать страницы истории, чтобы вспомнить что он делал и что нужно сделать дальше — реальный источник трения. Простая перезагрузка сообщений не решает эту UX-проблему.

Цель — проактивно показывать краткое резюме из 1–2 предложений при возврате пользователя:

  • Общая задача (что делает) → следующий шаг (что делать дальше).
  • Визуально отличается от реальных ответов ассистента, чтобы его никогда не путали с новым выводом модели.
  • По возможности: ошибки должны быть беззвучными и никогда не ломать основной поток.

Триггеры

ТриггерУсловияРеализация
РучнойПользователь выполняет /recaprecapCommand.ts вызывает тот же базовый сервис
АвтоТерминал размыт (протокол фокуса DECSET 1004) ≥ 5 мин + фокус возвращается + поток IdleuseAwaySummary.ts — таймер размытия 5 мин + слушатель событий useFocus
HTTP демонаУдалённый клиент вызывает POST /session/:id/recapМаршрут server.ts → bridge.generateSessionRecap (кругой̆ путь ext-method) → acpAgent.ts вызывает generateSessionRecap(session.getConfig(), signal)

Все три пути сходятся в одной функции generateSessionRecap() в core/services/sessionRecap.ts, что гарантирует идентичное поведение. Автотриггер контролируется настройкой general.showSessionRecap (по умолчанию: выкл. — явное согласие, чтобы фоновые вызовы LLM никогда не добавлялись незаметно в счёт пользователя); ручная команда и HTTP-маршрут демона игнорируют эту настройку (вызывающий делает явный запрос).

Путь доступа через демон

Маршрут демона нестрого ограничен (повторяет поведение /session/:id/prompt — суммирование расходует токены, но не изменяет состояние). Тег возможности session_recap рекламирует маршрут в /capabilities.features. Помощники SDK: DaemonClient.recapSession(sessionId, opts) и DaemonSessionClient.recap(opts). См. docs/developers/qwen-serve-protocol.md § POST /session/:id/recap для описания протокола и конверта ошибок.

Отмена отсутствует в v1. Маршрут не слушает отключение HTTP-клиента, AbortSignal не передаётся в bridge.generateSessionRecap, и ACP-обработчик дочернего процесса передаёт никогда не прерываемый AbortController().signal в основной помощник (механизм отмены между процессами пока отсутствует). Единственные ограничения — это резервный тайм-аут SESSION_RECAP_TIMEOUT_MS (60 с) в мосту и гонка закрытия транспорта с гибелью ACP-канала. Подключение AbortController на стороне HTTP само по себе было бы косметическим — вызов LLM на дочерней стороне всё равно завершится до конца, поэтому полная отмена от конца до конца невозможна без соответствующего механизма между процессами. Это приемлемо для v1, поскольку сводка короткая (однократный боковой запрос, maxOutputTokens: 300, обычно ~1–5 с). В будущем метод отмены ext-method на основе идентификатора запроса может обеспечить полную отмену от конца до конца, если пропускная способность оправдает затраты.

Дизайн промпта

Системный промпт

generationConfig.systemInstruction заменяет системный промпт основного агента для этого одиночного вызова, так что модель ведёт себя только как генератор дайджеста, а не как ассистент по кодированию.

Обратите внимание, что GeminiClient.generateContent() внутри прогоняет промпт через getCustomSystemPrompt(), который добавляет пользовательскую память (QWEN.md / управляемая авто-память) в качестве суффикса. Итоговый системный промпт: промпт дайджеста + пользовательская память — полезный контекст проекта для дайджеста, а не утечка.

Пункты ниже соответствуют 1:1 с RECAP_SYSTEM_PROMPT:

  • Не более 40 слов, 1-2 простых предложения (без markdown / списков / заголовков). Для китайского воспринимайте бюджет как примерно 80 символов всего.
  • Первое предложение: задача высокого уровня. Затем: конкретный следующий шаг.
  • Прямо запрещено: перечисление того, что было сделано, повторение вызовов инструментов, статусные отчёты.
  • Соответствует доминирующему языку разговора (английский или китайский).
  • Оборачивать вывод в <recap>...</recap>; ничего за пределами тегов.

Структурированный вывод + извлечение

Модели предписано оборачивать свой ответ в <recap>...</recap>:

<recap>Рефакторинг loopDetectionService.ts для устранения OOM при длительных сессиях. Следующий шаг — реализовать вариант B.</recap>

Почему: некоторые модели (семейство GLM, модели рассуждения) пишут абзац «размышлений» перед итоговым ответом. Возврат сырого текста привёл бы к утечке этих рассуждений в интерфейс.

extractRecap() имеет три уровня запасных вариантов:

  1. Присутствуют оба тега: взять то, что между <recap>...</recap> (предпочтительно).
  2. Только открывающий тег (например, maxOutputTokens обрезал закрывающий тег): взять всё после открывающего тега.
  3. Тег отсутствует полностью: вернуть пустую строку → сервис возвращает null → UI ничего не рендерит.

Третий уровень — «пропустить, а не показать неверное» — отображать преамбулу рассуждения модели хуже, чем не показывать дайджест вообще.

Фильтрация истории

geminiClient.getChat().getHistory() возвращает Content[], который включает:

  • текстовые сообщения user / model
  • части model functionCall
  • части user functionResponse (которые могут содержать полное содержимое файлов)
  • части мыслей модели (part.thought / part.thoughtSignature, скрытые рассуждения модели)

filterToDialog() сохраняет только части user / model, у которых есть непустой текст и которые не являются мыслями. Две причины:

  • Вызовы инструментов / ответы: один functionResponse может быть размером 10K+ токенов. 30 таких сообщений утопят LLM дайджеста в нерелевантных деталях, тратя токены и смещая дайджест в сторону шума реализации вроде «вызвал X инструмент, чтобы прочитать Y файл».
  • Части мыслей: несут внутренние рассуждения модели. Включение их рискует трактовать скрытую цепочку мыслей как диалог и выводить её в тексте дайджеста.

После отбрасывания пустых сообщений takeRecentDialog обрезает историю до последних 30 сообщений и отказывается начинать срез на повисшем ответе модели/инструмента.

## Параллелизм и граничные случаи
 
### Автоматический запуск: конечный автомат хука
 
`useAwaySummary` хранит три ссылки (ref):
 
| Ref                | Значение                                                   |
| ------------------ | ---------------------------------------------------------- |
| `blurredAtRef`     | Время начала потери фокуса (не очищается до возврата фокуса) |
| `recapPendingRef`  | Выполняется ли сейчас вызов LLM                            |
| `inFlightRef`      | Текущий выполняемый `AbortController`                      |
 
Зависимости `useEffect`: `[enabled, config, isFocused, isIdle, addItem, thresholdMs]`.
 
| Событие                                                         | Действие                                                                                                                                                                   |
| --------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `!enabled \|\| !config`                                         | Прервать выполняющийся вызов + очистить `inFlightRef` + очистить `blurredAtRef`                                                                                            |
| `!isFocused` и `blurredAtRef === null`                          | Установить `blurredAtRef = Date.now()`                                                                                                                                     |
| `isFocused` и `blurredAtRef === null`                           | Досрочный выход (нет цикла потери фокуса для обработки — первый рендер или сброс после краткой потери фокуса)                                                               |
| `isFocused` и длительность потери фокуса < 5 мин                | Очистить `blurredAtRef`, ожидать следующего цикла потери фокуса                                                                                                            |
| `isFocused` и потеря фокуса ≥ 5 мин и `recapPendingRef`         | Выход (дедупликация)                                                                                                                                                       |
| `isFocused` и потеря фокуса ≥ 5 мин и `!isIdle`                 | **Сохранить** `blurredAtRef` и дождаться окончания обработки (`isIdle` есть в зависимостях, поэтому эффект перезапустится, когда стриминг завершится)                       |
| `isFocused` и потеря фокуса ≥ 5 мин и `shouldFireRecap` вернул `false` | Очистить `blurredAtRef` и выйти — диалог не продвинулся достаточно с момента последнего recap (требуется ≥ 2 пользовательских ходов, как в Claude Code) |
| `isFocused` и все условия выполнены                              | Очистить `blurredAtRef`, установить `recapPendingRef = true`, создать `AbortController`, отправить запрос к LLM                                                             |
 
В коллбеке `.then` **повторно проверяется** `isIdleRef.current`: если пользователь начал новый ход, пока LLM выполнялась, то пришедший с опозданием recap отбрасывается, чтобы не вставить его в середину хода.
 
В `.finally` очищается `recapPendingRef`, а `inFlightRef` очищается только если `inFlightRef.current === controller` (чтобы не перезаписать более новый контроллер).
 
Второй `useEffect` прерывает выполняющийся контроллер при размонтировании компонента.
 
### Шлюз для `/recap`
 
`CommandContext.ui.isIdleRef` предоставляет текущее состояние потока (по аналогии с существующим шаблоном `btwAbortControllerRef`). В интерактивном режиме `recapCommand` отказывает, когда `!isIdleRef.current` **или** `pendingItem !== null`. Одного `pendingItem` недостаточно, потому что обычный ответ модели выполняется с `streamingState === Responding` и нулевым `pendingItem`.

### Пользовательские настройки
 
| Параметр                                  | Значение по умолчанию | Примечания                                                                      |
| ----------------------------------------- | --------------------- | ------------------------------------------------------------------------------- |
| `general.showSessionRecap`                | `false`               | Только для автоматического запуска. Ручной `/recap` игнорирует этот параметр.   |
| `general.sessionRecapAwayThresholdMinutes` | `5`                   | Минуты потери фокуса перед автоматическим recap при возврате фокуса. Как в Claude Code. |
| `fastModel`                               | не задано             | Рекомендуется (например, `qwen3-coder-flash`) для быстрых и дешёвых recap.      |
 
### Запасной вариант модели
 
`config.getFastModel() ?? config.getModel()`:
 
- У пользователя задан `fastModel`, и он подходит для текущего типа аутентификации → используется `fastModel`.
- В противном случае → запасной вариант — основная модель сессии (работает, но дороже и медленнее).
 
## Наблюдаемость
 
`createDebugLogger('SESSION_RECAP')` выводит:
 
- перехваченные исключения из пути recap (`debugLogger.warn`).
 
Все ошибки **полностью прозрачны** для пользователя — recap является вспомогательной функцией и никогда не выбрасывает исключения в UI. Разработчики могут искать тег `[SESSION_RECAP]` в файле отладочного журнала: по умолчанию записывается в `~/.qwen/debug/<sessionId>.txt` (симлинк `latest.txt` ведёт на текущую сессию); отключается через `QWEN_DEBUG_LOG_FILE=0`.

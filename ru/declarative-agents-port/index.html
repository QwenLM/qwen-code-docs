Декларативные определения агентов — портирование из Claude Code 2.1.168

Внутренний проектный документ для портирования схемы декларативных агентов (markdown + YAML frontmatter) из Claude Code в qwen-code, в рамках решения задачи #4821  и координации с портированием workflow в задаче #4721  / PR #4732 .

Статус реализации (вертикальная нарезка)

PR #4842  поставил поля с сквозным runtime-путем на тот момент. PR #4870  затем заменил парсер YAML для поддержки блочных скаляров. Этот последующий PR развивает оба: он заменяет YAML stringifier (PR #4870 оставил его рукописным — см. docs/yaml-parser-replacement.md), выводит mcpServers + hooks на SubagentConfig и подключает их к рантайму, чтобы серверы MCP и хуки, специфичные для каждого агента, реально срабатывали при запуске сабагента.

Фаза 1 — Результаты обратного инжиниринга

Все утверждения здесь были независимо проверены grep по claude.strings и прошли опровержение. Уровни уверенности: C = Confirmed (прямые доказательства из бинарника), I = Inferred (синтезировано из нескольких подтверждённых фактов), O = Open (всё ещё неопределённо).

Схема — 15 полей, опровергнутых и повторно подтверждённых

Теневая схема frontmatter агента — Ig5, используется внутри ug5.agent для телеметрии tengu_frontmatter_shadow_unknown_key / _mismatch. Рабочий загрузчик — DL7 (parseAgentFromMarkdown), который выполняет ручную проверку каждого поля с пользовательскими сообщениями об ошибках. Отдельная JSON-схема JL7 (используется fL7 / parseAgentFromJson) более строгая, но это другой путь кода (используется --agents <json> и settings.agents).

Фаза 2 — План реализации для qwen-code

Текущее состояние — краткая карта

qwen-code уже содержит значительную инфраструктуру сабагентов: SubagentManager (packages/core/src/subagents/subagent-manager.ts) реализует CRUD над markdown-файлами с YAML-фронтматером в .qwen/agents/ (проект) и ~/.qwen/agents/ (пользователь), используя собственный парсер YAML (packages/core/src/utils/yaml-parser.ts — без зависимостей gray-matter / yaml, подтверждено package.json). SubagentConfig (packages/core/src/subagents/types.ts:41-122) уже содержит name, description, tools, disallowedTools, approvalMode, systemPrompt, model, runConfig, color, background. SubagentLevel уже поддерживает пять областей (сессия, проект, пользователь, расширение, встроенный) с приоритетом session > project > user > extension > builtin (subagent-manager.ts:189-220). Инструмент Agent (packages/core/src/tools/agent/agent.ts) объявляет subagent_type и динамически обновляет свою перечислительную схему через subagentManager.changeListener. Мост convertClaudeAgentConfig() уже существует в packages/core/src/extension/claude-converter.ts:162-220 с маппингом имён инструментов и преобразованием permissionMode → approvalMode. Пробел заключается в: (a) схеме не хватает 8 полей из #4821 (effort, permissionMode как первоклассный, mcpServers, hooks, maxTurns как верхнеуровневый, skills, initialPrompt, memory, isolation); (b) отсутствует флаг CLI --agent <name>; (c) нет hot-перезагрузки в стиле chokidar (инвалидация как у расширений существует, но не для файловых агентов); (d) maxTurns в настоящее время находится в runConfig.max_turns — его нужно поднять на верхний уровень в соответствии с #2409.

D1. Повторное использование существующего yaml-parser для фронтматера

Решение: Повторно использовать packages/core/src/utils/yaml-parser.ts (уже используется SubagentManager.parseSubagentContent и загрузчиком навыков). Обоснование: lz в Claude Code — это тот же общий парсер, используемый для навыков + команд + агентов; qwen-code уже копирует этот шаблон. Добавление gray-matter или js-yaml — ненужный шум. Существующий парсер обрабатывает разделение --- … --- и молча реагирует на некорректный ввод (соответствует поведению lz предупредить-и-вернуть-пустой).

D2. Порядок разрешения / приоритетов

Решение: Использовать `session > project (.qwen/agents/) > user (~/.qwen/agents/)

Существующий пятиуровневый приоритет qwen-code уже покрывает случаи, которые важны для #4821: проект переопределяет пользователя, пользователь переопределяет встроенных. D3. Валидация — сохранить существующий SubagentValidator

Не вводить zod, если его уже не использует конвейер skillManager; если существующий валидатор написан вручную, оставить его ручным. D4. Hot reload — отложить; полагаться на холодную загрузку + явную перезагрузку

Решение: v1 НЕ поставляет наблюдатель chokidar. Хуки инвалидации кэша уже существуют (subagentManager имеет changeListener и явную CRUD-управляемую перезагрузку). Перезагрузка на уровне проекта происходит при запуске сессии; правки внутри сессии через UI /agents вызывают инвалидацию. Слеш-команда /reload-agents (или добавление к /reload-plugins) может появиться в P4, если будет спрос. Обоснование: Hot reload через FS-наблюдатель дорог (chokidar добавляет цикл опроса с адаптивным планированием — только реализация Claude Code занимает ~150 строк учёта). Поведение: найти в разрешённом реестре, установить агента как агента главного потока, выдать понятную ошибку, если имя не найдено. Соответствовать семантике Claude Code (заменить системный промпт по умолчанию, если у агента нет appendSystemPrompt: true).

Поставщики моделей

Qwen Code позволяет настроить несколько поставщиков моделей через параметр modelProviders в вашем settings.json. Это позволяет переключаться между разными AI-моделями и поставщиками с помощью команды /model.

Обзор

Используйте modelProviders для объявления моделей по типу аутентификации, между которыми можно переключаться через панель /model. Ключи должны быть допустимыми типами аутентификации (openai, anthropic, gemini и т.д.). Каждый тип аутентификации сопоставляется с объектом ProviderConfig, содержащим поле protocol и поле models (массив определений моделей). Каждая запись в models требует id; envKey опционален и рекомендован (если опущен, он использует ключ по умолчанию для данного типа аутентификации, например OPENAI_API_KEY для openai), с необязательными полями name, description, baseUrl и generationConfig. Учётные данные никогда не сохраняются в настройках; среда выполнения считывает их из process.env[envKey]. Модели Qwen OAuth остаются жёстко заданными и не могут быть переопределены.

Note

Только команда /model раскрывает нестандартные типы аутентификации. Anthropic, Gemini и т.д. должны быть заданы через modelProviders. Команда /auth отображает три основных варианта: Alibaba ModelStudio (с подменю Coding Plan, Token Plan и Standard API Key), Сторонние поставщики и Пользовательский поставщик. (Qwen OAuth больше не является выбираемым пунктом в диалоге; его бесплатный уровень был отключён 15.04.2026.)

Note

Уникальность моделей: Модели внутри одного authType однозначно идентифицируются комбинацией id + baseUrl. Это означает, что вы можете задать один и тот же идентификатор модели (например, "gpt-4o") несколько раз в рамках одного authType, если каждая запись имеет свой baseUrl — например, одна указывает на OpenAI напрямую, а другая на прокси-сервер. Если две записи имеют одинаковые id и baseUrl (или обе опускают baseUrl), побеждает первая, а последующие дубликаты пропускаются с предупреждением.

Примеры конфигурации по типу аутентификации

Ниже приведены подробные примеры конфигурации для различных типов аутентификации, демонстрирующие доступные параметры и их комбинации.

Поддерживаемые типы аутентификации

Ключи объекта modelProviders должны быть допустимыми значениями authType. В настоящее время поддерживаются следующие типы аутентификации:

Тип аутентификацииОписание
openaiAPI, совместимые с OpenAI (OpenAI, Azure OpenAI, локальные серверы инференса, такие как vLLM/Ollama)
anthropicAPI Anthropic Claude
geminiAPI Google Gemini
qwen-oauthQwen OAuth (жёстко заданное, нельзя переопределить в modelProviders)
vertex-aiGoogle Vertex AI (использует протокол gemini и SDK @google/genai в режиме Vertex AI; при выборе устанавливает GOOGLE_GENAI_USE_VERTEXAI=true)

[!warning] При использовании неизвестного ключа типа аутентификации (например, опечатка "openai-custom") непустой ключ принимается как есть в качестве собственной группы типа аутентификации, но он не будет сопоставлен с известным протоколом — поэтому его модели не будут работать должным образом и не будут корректно отображаться в панели /model. Пропускаются только пустые (состоящие только из пробелов) ключи. Всегда используйте одно из поддерживаемых значений типа аутентификации, перечисленных выше.

SDK, используемые для API-запросов

Qwen Code использует следующие официальные SDK для отправки запросов к каждому поставщику:

Тип аутентификацииПакет SDK
openaiopenai — официальный SDK OpenAI для Node.js
anthropic@anthropic-ai/sdk — официальный SDK Anthropic
gemini@google/genai — официальный SDK Google GenAI
qwen-oauthopenai с пользовательским поставщиком (совместимый с DashScope)

Это означает, что настроенный baseUrl должен быть совместим с ожидаемым форматом API соответствующего SDK. Например, при использовании типа аутентификации openai конечная точка должна принимать запросы в формате API OpenAI.

Поставщики, совместимые с OpenAI (openai)

Хранение ключа API

При настройке Coding Plan через команду /auth ключ API сохраняется с использованием зарезервированного имени переменной окружения BAILIAN_CODING_PLAN_API_KEY. По умолчанию он хранится в поле env вашего файла settings.json.

Warning

Рекомендация по безопасности: Для повышения безопасности рекомендуется перенести ключ API из файла settings.json в отдельный файл .env и загружать его как переменную окружения. Например:

# ~/.qwen/.env
BAILIAN_CODING_PLAN_API_KEY=your-api-key-here

Затем убедитесь, что этот файл добавлен в .gitignore, если вы используете настройки на уровне проекта.

Автоматические обновления

Конфигурации моделей Coding Plan имеют версии. Когда Qwen Code обнаруживает новую версию шаблона модели, вам будет предложено обновиться. Принятие обновления приведёт к:

  • Замените существующие конфигурации моделей Coding Plan на последние версии
  • Сохраните пользовательские конфигурации моделей, добавленные вручную
  • Автоматически переключитесь на первую модель в обновлённой конфигурации

Процесс обновления гарантирует, что вы всегда имеете доступ к последним конфигурациям моделей и функциям без ручного вмешательства.

Ручная конфигурация (продвинутая)

Если вы предпочитаете настраивать модели Coding Plan вручную, вы можете добавить их в ваш settings.json как любого провайдера, совместимого с OpenAI:

{
  "modelProviders": {
    "openai": {
      "protocol": "openai",
      "models": [
        {
          "id": "qwen3-coder-plus",
          "name": "qwen3-coder-plus",
          "description": "Qwen3-Coder via Alibaba Cloud Coding Plan",
          "envKey": "YOUR_CUSTOM_ENV_KEY",
          "baseUrl": "https://coding.dashscope.aliyuncs.com/v1"
        }
      ]
    }
  }
}
Note

При использовании ручной конфигурации:

  • Вы можете использовать любое имя переменной окружения для envKey
  • Вам не нужно настраивать codingPlan.*
  • Автоматические обновления не будут применяться к вручную настроенным моделям Coding Plan
Warning

Если вы также используете автоматическую конфигурацию Coding Plan, автоматические обновления могут перезаписать ваши ручные конфигурации, если они используют те же envKey и baseUrl, что и автоматическая конфигурация. Чтобы этого избежать, убедитесь, что ваша ручная конфигурация использует другой envKey, если это возможно.

Слои конфигурации генерации: Непроницаемый слой провайдера

Разрешение конфигурации следует строгой многоуровневой модели с одним ключевым правилом: слой modelProvider непроницаем.

Как это работает

  1. Когда выбрана модель из modelProvider (например, через команду /model с выбором модели, настроенной провайдером):

    • Весь generationConfig провайдера применяется атомарно
    • Слой провайдера полностью непроницаем — нижележащие слои (CLI, env, settings) не участвуют в разрешении generationConfig
    • Все поля, определенные в modelProviders[].generationConfig, используют значения провайдера
    • Все поля, не определенные провайдером, устанавливаются в undefined (не наследуются из настроек)
    • Это гарантирует, что конфигурации провайдера действуют как полный, самодостаточный «запечатанный пакет»

    Если модель указана в modelProviders, поместите все специфичные для модели настройки генерации для этой модели в соответствующую запись провайдера. Значения на верхнем уровне model.generationConfig, включая contextWindowSize, modalities, customHeaders и extra_body, игнорируются для моделей провайдера. Настройте эти поля в modelProviders[authType][].generationConfig, чтобы они применялись.

  2. Когда не выбрана модель из modelProvider (например, при использовании --model с сырым идентификатором модели или при прямом использовании CLI/env/settings):

    • Разрешение опускается на более низкие слои
    • Поля заполняются из CLI → env → settings → defaults
    • Это создает Модель времени выполнения (см. следующий раздел)

Приоритет полей для generationConfig

ПриоритетИсточникПоведение
1Программные переопределенияИзменения во время выполнения через /model, /auth
2modelProviders[authType][].generationConfigНепроницаемый слой — полностью заменяет все поля generationConfig; нижележащие слои не участвуют
3settings.model.generationConfigИспользуется только для Моделей времени выполнения (когда не выбрана модель провайдера)

Атомарная обработка полей

Следующие поля обрабатываются как атомарные объекты — значения провайдера полностью заменяют весь объект, без слияния:

  • samplingParams — Температура, top_p, max_tokens и т.д.
  • customHeaders — Пользовательские HTTP-заголовки
  • extra_body — Дополнительные параметры тела запроса

Пример

// Пользовательские настройки (~/.qwen/settings.json)
{
  "model": {
    "generationConfig": {
      "timeout": 30000,
      "samplingParams": { "temperature": 0.5, "max_tokens": 1000 }
    }
  }
}
 
// Конфигурация modelProviders
{
  "modelProviders": {
    "openai": {
      "protocol": "openai",
      "models": [{
        "id": "gpt-4o",
        "envKey": "OPENAI_API_KEY",
        "generationConfig": {
          "timeout": 60000,
          "samplingParams": { "temperature": 0.2 }
        }
      }]
    }
  }
}

Когда gpt-4o выбрана из modelProviders:

  • timeout = 60000 (от провайдера, переопределяет настройки)
  • samplingParams.temperature = 0.2 (от провайдера, полностью заменяет объект настроек)
  • samplingParams.max_tokens = undefined (не определено у провайдера, и слой провайдера не наследует из настроек — поля явно устанавливаются в undefined, если не предоставлены)

При использовании сырой модели через --model gpt-4 (не из modelProviders, создает Модель времени выполнения):

  • timeout = 30000 (из настроек)
  • samplingParams.temperature = 0.5 (из настроек)
  • samplingParams.max_tokens = 1000 (из настроек)

Стратегия слияния для modelProviders — ЗАМЕНА: весь modelProviders из настроек проекта переопределит соответствующий раздел в пользовательских настройках, а не сольется с ним.

Конфигурация рассуждения / мышления

Необязательное поле reasoning в generationConfig управляет интенсивностью рассуждений модели перед ответом. Конвертеры Anthropic и Gemini всегда его учитывают. Конвейер, совместимый с OpenAI, учитывает его если только не задан generationConfig.samplingParams — см. предостережение «Взаимодействие с samplingParams» ниже.

{
  "modelProviders": {
    "openai": {
      "protocol": "openai",
      "models": [
        {
          "id": "deepseek-v4-pro",
          "name": "DeepSeek V4 Pro",
          "baseUrl": "https://api.deepseek.com/v1",
          "envKey": "DEEPSEEK_API_KEY",
          "generationConfig": {
            // Четырехуровневая шкала:
            //   'low'    | 'medium' — на сервере DeepSeek сопоставляется с 'high'
            //   'high'   — интенсивность рассуждений по умолчанию
            //   'max'    — дополнительный сильный уровень DeepSeek
            // Или установите `false`, чтобы полностью отключить рассуждения.
            "reasoning": { "effort": "max" },
          },
        },
      ],
    },
  },
}

Поведение для каждого провайдера

Протокол / провайдерФормат в запросе (wire shape)Примечания
OpenAI / DeepSeek (api.deepseek.com)Плоский параметр тела reasoning_effort: <effort>Когда reasoning.effort задан во вложенной конфигурации, он переписывается в плоский reasoning_effort, а значения 'low' / 'medium' нормализуются к 'high', 'xhigh' — к 'max' — что соответствует серверной обратной совместимости DeepSeek (см. документацию ). Установка samplingParams.reasoning_effort или extra_body.reasoning_effort на верхнем уровне пропускает эту нормализацию и передаётся как есть.
OpenAI (другие совместимые серверы)reasoning: { effort, ... } передаётся без измененийЗадаётся через samplingParams (например, samplingParams.reasoning_effort для GPT‑5/o‑серии), если провайдер ожидает другую структуру.
Anthropic (настоящий api.anthropic.com)output_config: { effort } плюс заголовок бета-версии effort-2025-11-24Настоящий Anthropic принимает только 'low' / 'medium' / 'high'. Значение 'max' приводится к 'high' с выводом предупреждения через debugLogger.warn (один раз на генератор); если нужно максимальное усилие, смените baseURL на конечную точку DeepSeek, поддерживающую его.
Anthropic (api.deepseek.com/anthropic)Тот же output_config: { effort } + заголовок бета-версии'max' передаётся без изменений.
Gemini (@google/genai)thinkingConfig: { includeThoughts: true, thinkingLevel }'low' → LOW, 'high' / 'max' → HIGH, остальные → THINKING_LEVEL_UNSPECIFIED (в Gemini нет уровня MAX).

reasoning: false

Установка reasoning: false (буквальное логическое значение) явно отключает режим размышлений у всех провайдеров — удобно для дешёвых побочных запросов, которым не требуется рассуждение. Это также учитывается на уровне запроса через request.config.thinkingConfig.includeThoughts: false для однократных вызовов (например, генерация подсказок).

На базовом URL api.deepseek.com конвейер OpenAI отправляет явное поле thinking: { type: 'disabled' }, которое требуется DeepSeek V4+ — серверное значение по умолчанию — 'enabled', так что простое опускание reasoning_effort всё равно приведёт к затратам времени и средств на размышления. Самостоятельно развёрнутые серверы DeepSeek (sglang/vllm) и другие совместимые с OpenAI серверы не получают это поле; если нужно отключить размышления на них, добавьте thinking: { type: 'disabled' } (или аналогичный параметр вашего inference-фреймворка) через samplingParams / extra_body.

Взаимодействие с samplingParams (только для совместимых с OpenAI)

Warning

Когда для провайдера, совместимого с OpenAI, задан generationConfig.samplingParams, конвейер отправляет эти ключи в запрос без изменений и полностью пропускает отдельную вставку параметра reasoning. Таким образом, конфигурация вида { samplingParams: { temperature: 0.5 }, reasoning: { effort: 'max' } } приведёт к тому, что поле reasoning будет молча отброшено в запросах к OpenAI / DeepSeek.

Если вы задаёте samplingParams, включайте параметр reasoning непосредственно в него: для DeepSeek это samplingParams.reasoning_effort, для GPT‑5 / o‑серии — samplingParams.reasoning_effort (их плоское поле) или samplingParams.reasoning (вложенный объект). Для OpenRouter и других провайдеров имя поля может отличаться; сверьтесь с документацией провайдера.

Конвертеры Anthropic и Gemini не подвержены этому — они всегда читают reasoning.effort напрямую, независимо от samplingParams.

budget_tokens

Вы можете зафиксировать точный бюджет токенов для рассуждения, указав budget_tokens вместе с effort:

"reasoning": { "effort": "high", "budget_tokens": 50000 }

Для Anthropic это становится thinking.budget_tokens. Для OpenAI/DeepSeek это поле сохраняется, но в данный момент игнорируется сервером — reasoning_effort является основным рычагом управления.

Провайдерские модели и модели времени выполнения

Qwen Code различает два типа конфигураций моделей:

Провайдерская модель (Provider Model)

  • Определяется в конфигурации modelProviders
  • Имеет законченный, атомарный пакет конфигурации
  • При выборе её конфигурация применяется как непроницаемый слой
  • Отображается в списке команды /model с полными метаданными (имя, описание, возможности)
  • Рекомендуется для многомодельных рабочих процессов и единообразия в команде

Модель времени выполнения (Runtime Model)

  • Создается динамически при использовании сырых идентификаторов моделей через CLI (--model), переменные окружения или настройки
  • Не определена в modelProviders
  • Конфигурация строится путём «проецирования» через слои разрешения (CLI → env → настройки → значения по умолчанию)
  • Автоматически захватывается как RuntimeModelSnapshot, когда обнаружена полная конфигурация
  • Позволяет повторно использовать настройки без повторного ввода учётных данных

Жизненный цикл RuntimeModelSnapshot

Когда вы настраиваете модель без использования modelProviders, Qwen Code автоматически создаёт RuntimeModelSnapshot, чтобы сохранить вашу конфигурацию:

# Это создаёт RuntimeModelSnapshot с ID: $runtime|openai|my-custom-model
qwen --auth-type openai --model my-custom-model --openai-api-key $KEY --openai-base-url https://api.example.com/v1

Снимок:

  • Содержит ID модели, API-ключ, базовый URL и конфигурацию генерации
  • Сохраняется между сессиями (хранится в памяти во время выполнения)
  • Отображается в списке команды /model как опция времени выполнения
  • Можно переключиться на него с помощью /model $runtime|openai|my-custom-model

Когда что использовать

  • Используйте провайдерские модели, когда: у вас есть стандартные модели, используемые всей командой, нужны единообразные конфигурации или вы хотите предотвратить случайные переопределения
  • Используйте модели времени выполнения, когда: вы быстро тестируете новую модель, используете временные учётные данные или работаете с ad-hoc конечными точками

Сохранение выбора и рекомендации

Important

Определяйте modelProviders в области пользователя ~/.qwen/settings.json всегда, когда это возможно, и избегайте сохранения переопределений учётных данных в любой области. Хранение каталога провайдеров в пользовательских настройках предотвращает конфликты слияния/переопределения между проектными и пользовательскими областями и гарантирует, что обновления /auth и /model всегда записываются обратно в согласованную область.

  • /model и /auth сохраняют model.name (где применимо) и security.auth.selectedType в ближайшую доступную для записи область, которая уже определяет modelProviders; в противном случае они возвращаются к области пользователя. Это синхронизирует файлы рабочей области/пользователя с активным каталогом провайдеров.
  • Без modelProviders резольвер смешивает слои CLI/env/настроек, создавая модели времени выполнения. Это нормально для однопровайдерских конфигураций, но неудобно при частом переключении. Определяйте каталоги провайдеров, если многомодельные рабочие процессы распространены, чтобы переключения оставались атомарными, привязанными к источнику и поддающимися отладке.

Замена YAML-парсера — результаты исследования

Внутренний документ дизайна для замены самописного YAML-парсера на 192 строки из packages/core/src/utils/yaml-parser.ts на настоящую библиотеку, чтобы отложенные поля mcpServers и hooks из схемы декларативного агента Claude Code могли безопасно проходить круговой проход через пути субагентов / скиллов / конвертеров.

Сопутствующий документ: docs/declarative-agents-port.md. Задача: #4821 . Предварительное условие для последующего PR #4842 .

Фаза 1 — Какую YAML-библиотеку использует CC?

Ответ: yaml (eemeli/yaml), НЕ js-yaml. Подтверждено чтением ~/code/claude-code/src/utils/yaml.ts дословно:

export function parseYaml(input: string): unknown {
  if (typeof Bun !== 'undefined') {
    return Bun.YAML.parse(input);
  }
  // eslint-disable-next-line @typescript-eslint/no-require-imports
  return (require('yaml') as typeof import('yaml')).parse(input);
}
  • Библиотека: npm-пакет yaml. C
  • API: верхнеуровневый .parse(input). Использует схему по умолчанию пакета (которая является YAML 1.2 core — надмножество JSON, без JS-расширений). C
  • Bun shortcut: при запуске под Bun CC использует Bun.YAML.parse(), чтобы избежать включения ~270 КБ YAML-парсера в бандл. C Не актуально для qwen-code (мы не нацелены на рантайм Bun).
  • Режим схемы: НИГДЕ явно не задаётся в CC. Полагается на поведение пакета yaml по умолчанию, плюс валидация zod на уровне потребителя (DL7, gS8, TKO/_u согласно docs/declarative-agents-port.md). C

Почему yaml, а не js-yaml

Измерениеjs-yaml 4.xyaml (eemeli) 2.x
Схема по умолчаниюDEFAULT_SAFE_SCHEMA (начиная с 4.x) — безопасная; старые версии имели DEFAULT_FULL_SCHEMA с JScore (спецификация YAML 1.2) — только JSON-типы
Тег !!js/functionНЕ поддерживается в 4.x (был в 3.x)Никогда не поддерживался
Защита от billion-laughsНет (ответственность разработчика)Встроенный maxAliasCount: 100 по умолчанию
Ключи слияния (<<)Поддерживаются (нужно отключать через MERGE_SCHEMA или фильтрацию)Отключены по умолчанию, включение через { merge: true }
Уже является зависимостью qwen-code?js-yaml@4.1.1 ✓yaml@2.8.1 ✓ (уже импортируется skill-manager)

Оба варианта разумны в 2026 году, но исходное техническое задание рекомендовало FAILSAFE_SCHEMA / CORE_SCHEMA от js-yaml. Мы отклоняемся от этой рекомендации по трём конкретным причинам:

  1. Паритет с CC. Весь смысл портирования схемы фронтматера CC — позволить пользователям положить файл агента CC в .qwen/agents/ и получать его идентичный разбор. Использование того же парсера, что и CC, минимизирует расхождения на краевых случаях YAML (многодокументные потоки, блочные скаляры vs потоковые, обработка тегов).
  2. yaml уже напрямую используется внутри skill-manager.ts — см. packages/core/src/skills/skill-manager.ts:13 (import * as yaml from 'yaml'). Стандартизация на yaml устраняет один из двух дублирующихся стеков YAML в одном пакете. C (результат grep задокументирован в Фазе 6).
  3. Более безопасные настройки по умолчанию, чем у js-yaml. Встроенный maxAliasCount блокирует billion-laughs без ручной настройки; ключи слияния отключены по умолчанию; произвольные теги превращаются в литеральные строки с YAMLWarning, а не вызывают исполняемые резолверы. Эмпирические доказательства в Фазе 4.

Если будущий мейнтейнер захочет отказаться от зависимости yaml и унифицироваться на js-yaml, миграция механическая: заменить yaml.parse / yaml.stringify на jsYaml.load(s, { schema: jsYaml.CORE_SCHEMA }) / jsYaml.dump. Две библиотеки согласуются по выводу для 100% подмножества, которое реально используют CC и qwen-code (пары ключ-значение, списки, вложенные отображения, булевы скаляры/числа). Отслеживайте это решение отдельно, если возникнет необходимость.

Почему оба валидатора — z.unknown() на уровне тени Ig5

Ig5 — это теневая схема телеметрии — она генерирует события tengu_frontmatter_shadow_unknown_key, когда YAML-ключ отсутствует в известном наборе, и _mismatch-события, когда известный ключ имеет неверный тип. Она намеренно использует z.unknown() для mcpServers и hooks, потому что Ig5 выполняется ВО ВРЕМЯ РАЗБОРА и генерировала бы ложные события mismatch для каждой встроенной спецификации mcpServers. Настоящая валидация делегируется:

  • gS8 (для mcpServers) — вызывается во время регистрации агента из DL7 для каждого элемента через safeParse
  • TKO (для hooks) — вызывается в момент срабатывания хука из _u().safeParse

Эта ленивая валидация — модель, которую qwen-code должен имитировать: оставить парсер фронтматера снисходительным (эквивалент z.unknown() в TypeScript), валидировать в точке использования. Попытка продвинуть полное дерево zod вперёд в SubagentConfig заставила бы нас также импортировать класс MCPServerConfig из qwen и тип HookDefinition в слой, где они сейчас не живут, и потребовала бы изобретать фиктивные валидаторы для type: 'sdk' / type: 'claudeai-proxy', которые мы на самом деле не поддерживаем.

Фаза 5 — Семантика восстановления

CC выбирает снисходительное «предупредить и отбросить» на каждом уровне:

  1. YAML-парсер выбрасывает исключение → парсер фронтматера логирует + возвращает {} (пустые данные)
  2. Поле имеет неверную форму (например, mcpServers: "this is a string") → safeParse не проходит → поле отбрасывается из итоговой конфигурации
  3. Поле имеет почти неверную форму (например, отдельный элемент mcpServers — строка, когда схема ожидает объект) → поэлементный safeParse отбрасывает только этот элемент, остальные сохраняет

qwen-code уже реализует подход «предупредить и отбросить» для полей permissionMode, maxTurns, color, effort (см. packages/core/src/subagents/agent-frontmatter-schema.ts). Мы расширяем тот же шаблон на mcpServers и hooks.

Что мы НЕ клонируем из CC:

  • Двухпроходное YAML-восстановление с автоматическим экранированием. Это мёртвый груз для qwen-code — мы новый проект, нет устаревших вручную отредактированных файлов фронтматера, которые нужно прощать. Чистая ошибка полезнее, чем додуманная переинтерпретация.
  • События телеметрии tengu_*. Заменяются собственным логгером qwen-code / тем уровнем телеметрии, который использует остальной загрузчик.

Фаза 6 — Рекомендации для qwen-code

Выбор библиотеки

  • Использовать yaml@^2.8.1 (уже транзитивная — продвинуть до прямой зависимости в packages/core/package.json, чтобы не сломаться при более строгих режимах резолюции; также позволит зафиксировать мажорную версию).
  • Использовать схему по умолчанию (core), без флага схемы.
  • Не передавать { merge: true }. Не включать ни одну нестандартную опцию.
  • Для детерминированного вывода при stringify (тестовые снэпшоты) передавать { lineWidth: 0, defaultStringType: 'PLAIN' } в yaml.stringify, чтобы библиотека не переносила длинные строки и не переключалась произвольно на блочно-скалярное экранирование в зависимости от длины содержимого.

Поверхность API для сохранения

Текущий packages/core/src/utils/yaml-parser.ts экспортирует:

export function parse(yamlString: string): Record<string, unknown>;
export function stringify(
  obj: Record<string, unknown>,
  options?: { lineWidth?: number; minContentWidth?: number },
): string;

Замена сохраняет обе сигнатуры идентичными, чтобы 5 вызывающих модулей (subagent-manager.ts, claude-converter.ts, rulesDiscovery.ts, skill-manager.ts, skill-load.ts) и реэкспорт через index.ts не требовали никаких изменений в местах вызовов.

Набросок реализации:

import * as yaml from 'yaml';
 
export function parse(yamlString: string): Record<string, unknown> {
  const parsed = yaml.parse(yamlString);
  if (parsed && typeof parsed === 'object' && !Array.isArray(parsed)) {
    return parsed as Record<string, unknown>;
  }
  return {};
}
 
export function stringify(
  obj: Record<string, unknown>,
  options?: { lineWidth?: number; minContentWidth?: number },
): string {
  return yaml.stringify(obj, {
    lineWidth: options?.lineWidth ?? 0,
    minContentWidth: options?.minContentWidth ?? 20,
  });
}

Почему приводить не-объектные верхние уровни к {}: каждый существующий вызывающий код предполагает запись (Record). YAML-файл, который разбирается в null (пустой файл), ["foo"] (список) или "hello" (голый скаляр), в настоящее время приводил бы к аварии при деструктуризации downstream. Возврат {} сохраняет поведение старого самописного парсера для тех же входных данных. Задокументировать это как намеренное защитное ограничение в однострочном комментарии.

Тесты, которые необходимо изменить

В packages/core/src/utils/yaml-parser.test.ts есть 2 «закрепляющих теста» в конце (строки 200–227) с названием known limitations — nested YAML (pin until js-yaml lands). Замена ДОЛЖНА превратить их в положительные утверждения о парсинге вложенных структур:

it('parses array-of-records', () => {
  const yaml =
    'mcpServers:\n  - filesystem:\n      type: stdio\n      command: node';
  expect(parse(yaml)).toEqual({
    mcpServers: [{ filesystem: { type: 'stdio', command: 'node' } }],
  });
});
 
it('parses record-of-records', () => {
  const yaml = 'hooks:\n  PreToolUse:\n    - matcher: Read';
  expect(parse(yaml)).toEqual({
    hooks: { PreToolUse: [{ matcher: 'Read' }] },
  });
});

Эти два утверждения плюс три фикстуры выше являются критерием приёмки для Фазы 2 плана реализации. Всё остальное (граничные случаи экранирования, логические значения в кавычках и без, числовые строки) — это регрессионное покрытие из существующего набора тестов, и оно должно проходить без изменений.

Проверка круговой непротиворечивости (round-trip)

Существующий тест should maintain round-trip integrity for escaped strings (строки 111–129) прогоняет 7 строк через stringify → parse. Стандартный stringify библиотеки yaml выдаёт немного другой вывод, чем ручной форматтер (более агрессивное расставление кавычек в некоторых случаях, другие escape-последовательности). Два приемлемых исхода:

  1. Подкорректировать тестовые фикстуры, чтобы проверять поведение с новым парсером — важно свойство кругого пути (parse(stringify(x)) === x), а не побайтово идентичный вывод YAML.
  2. Оставить побайтовые утверждения и дать им упасть наглядно, затем обновить их, отразив вывод библиотеки yaml дословно. Легче проверять diff.

Рекомендация: вариант 1 — изменить утверждения на основанные на свойствах (expect(parse(stringify(obj))).toEqual(obj)), так как побайтово идентичный вывод YAML не является документированным контрактом модуля.

Критические изменения для вызывающего кода — не ожидается, но проверить

  • subagent-manager.ts повторно сериализует распарсенный объект обратно в YAML для пути saveSubagent. С новым парсером mcpServers и hooks будут проходить round-trip чисто. Обновите NESTED_FIELDS_NOT_ROUND_TRIPPABLE в claude-converter.ts (Фаза 3 реализации), удалив эти два имени полей.
  • skill-manager.ts уже импортирует yaml напрямую (отдельно от ручного парсера). После того как yaml-parser.ts тоже начнёт использовать yaml, дублирующий импорт можно будет удалить мелким последующим шагом — вне рамок данной задачи.

Риск при миграции

Низкий. Все 5 вызывающих модулей деструктурируют Record<string, unknown> — тип возврата тот же. Единственные ожидаемые сбои — это 2 намеренных «искажающих» закрепляющих теста; они известны, и мы меняем их целенаправленно. Более широкая регрессия покрыта существующими наборами тестов в packages/core/src/subagents/, packages/core/src/skills/ и packages/core/src/extension/.

Q3Если mcpServers некорректен на верхнем уровне (например, mcpServers: "string"), должен ли весь агент не загружаться, или загружаться с отбрасыванием этого поля?Да — определяет стратегию «предупредить и отбросить» в Фазе 3 плана реализацииРешение: отбросить поле, выдать предупреждение в консоль (на основе CC DL7 согласно Фазе 3 docs/declarative-agents-port.md).
Q4То же, что Q3, но для hooks: отбросить поле, событие или только отдельный matcher?Да — определяет стратегию «предупредить и отбросить»Решение: отбросить всё поле hooks при ошибке формы верхнего уровня. Детализация по событиям / matcher’ам откладывается до будущего PR, если появится реальная потребность.

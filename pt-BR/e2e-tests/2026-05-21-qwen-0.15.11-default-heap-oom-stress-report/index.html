Qwen Code 0.15.11 - Relatório de Teste de Estresse de Heap Padrão (OOM)

Data: 2026-05-21

Escopo do Teste

Este relatório documenta uma rodada de testes de estresse de heap padrão no build local mais recente do Qwen Code 0.15.11.
O objetivo desta rodada de testes é verificar: sem reduzir artificialmente o limite de memória, se o código atual ainda consegue reproduzir o OOM de sessão longa mencionado na issue, e se há novos riscos em cenários de saída grande mais extremos.

Esta rodada cobre três modelos:

  • pai/glm-5
  • qwen3.6-plus
  • DeepSeek/deepseek-v4-pro

O teste é dividido em duas partes:

  1. Tarefas longas reais, ciclo de revisão concorrente com múltiplos agentes.
  2. Teste de estresse amplificado de stdout no foreground, ou seja, usar stdout de shell em grande escala no foreground para amplificar a pressão no caminho de saída da ferramenta.

Nota: A versão do CLI exibida aqui é 0.15.11 porque a versão do pacote ainda não foi incrementada (bump). O objeto real do teste é o dist/cli.js compilado localmente sob o commit c161e0aa4, não o executável global qwen no PATH.

Nesta rodada, nenhuma configuração global do Qwen foi modificada. Os artefatos originais de runtime estão em:

  • .qwen/runtime-bench/2026-05-20T13-51-58-731Z-oom-stress
  • .qwen/runtime-bench/2026-05-20T15-20-37-790Z-oom-amplified

Nota: Nesta rodada, o MCP server env-center falhou ao iniciar, mas outras ferramentas integradas e alguns processos MCP/child ainda foram carregados. Portanto, esses resultados representam o ambiente local atual, não um ambiente completamente stripped --bare.

Conclusões Principais

O build local mais recente apresenta um desempenho significativamente melhor no caminho de “OOM de heap V8 de sessão longa” que é o foco da issue. Com base nesta rodada de testes de estresse com heap padrão, múltiplos modelos, múltiplos agentes e tarefas longas, pode-se considerar que este PR resolveu basicamente o problema de OOM de heap de sessão longa encontrado anteriormente, pelo menos não sendo mais possível reproduzir o OOM de heap original na dimensão atual de reprodução.

O teste de tarefas longas reais com múltiplos agentes concorrentes executou:

  • 23 turns de worker
  • Aproximadamente 719.094.118 tokens totais reportados
  • 77 chamadas de agente (tool call)
  • 856 chamadas de ferramenta no total

Esta parte não reproduziu nenhuma característica tradicional de OOM de heap V8:

  • JavaScript heap out of memory
  • Reached heap limit
  • Ineffective mark-compacts near heap limit
  • Allocation failed

O pico de RSS da árvore de processos na fase de tarefas longas reais foi de 874,7 MiB, e o pico de RSS do processo raiz foi de 219,1 MiB. Isso mostra que, com o heap padrão, o código atual não reproduz facilmente o OOM de heap que derrubava tarefas longas como na issue original.

A segunda fase, teste de estresse amplificado de stdout, foi mais agressiva. Ela executou 18 tentativas de payload, cobrindo três modelos e payloads de stdout foreground de 128 MiB a 2048 MiB.

O resultado:

  • Os três modelos passaram com sucesso pelo payload de 1536 MiB.
  • O maior RSS de árvore de processos bem-sucedido foi de 5964,7 MiB, observado no payload de 1536 MiB do qwen3.6-plus.
  • No payload de 2048 MiB, ocorreu uma nova falha extrema de saída grande.

Resultados para 2048 MiB:

  • pai/glm-5: exit=1, stdout vazio, sem texto padrão de OOM.
  • qwen3.6-plus: exit=1, stdout vazio, sem texto padrão de OOM.
  • DeepSeek/deepseek-v4-pro: Ocorreu fatal do V8: Check failed: i::kMaxInt >= len, pilha em v8::String::NewFromOneByte / node::StringBytes::Encode / DecodeUTF8.

Esse novo problema não é o OOM de heap de sessão longa tradicional da issue original. Ele se assemelha mais a um problema de limite de comprimento de string do V8 ou de processamento de saída grande ao decodificar/construir strings JS a partir de stdout foreground de vários GiB. Recomenda-se acompanhá-lo como um follow-up de saída grande, em vez de tratá-lo como uma falha na correção da pressão de heap de sessão longa atual.

Phase 1: Teste de Estresse de Tarefas Longas Reais com Múltiplos Agentes Concorrentes

Forma do Teste

Cada worker de modelo reutiliza a mesma sessão, fazendo --resume continuamente. Em cada rodada, pede-se ao Qwen Code:

  • Realizar revisão de código somente leitura e busca de código;
  • Iniciar concorrentemente pelo menos 4 chamadas de ferramenta agent na mesma rodada;
  • Verificar especialmente áreas relacionadas à memória, como histórico de chat, compactação, runtime de subagentes, streaming não interativo, adaptadores de provedor, etc.;
  • Manter respostas finais suficientemente detalhadas para que o histórico da sessão cresça naturalmente.

O runner amostra o RSS da árvore de processos a cada segundo, sem definir nenhum limite adicional de heap.

Esta parte foi interrompida ativamente com SIGTERM após observar que a memória estava relativamente estável, para permitir a transição para a segunda fase de teste de estresse amplificado de stdout. Portanto, SIGTERM na tabela não é OOM.

Interpretação da Phase 1

Estes são os dados que melhor demonstram que o OOM de sessão longa original já melhorou significativamente.

Este conjunto de testes é mais pesado do que a pequena revisão de PR / navegação de código de 18 de maio: contém mais --resume, mais atividade de subagentes, maior quantidade de tokens reportados e mais chamadas de ferramenta. No entanto, o RSS da árvore de processos permaneceu sempre abaixo de 0,9 GiB, e nenhum OOM de heap V8 tradicional foi observado.

Isso não prova que todos os OOMs de usuário são impossíveis, mas pelo menos mostra que, com a construção atual e heap padrão, não é mais fácil reproduzir o OOM por pressão de heap de sessão longa como o da issue.

Phase 2: Teste de Estresse Amplificado de Stdout Foreground

Forma do Teste

A segunda fase amplifica deliberadamente a pressão no caminho de saída do shell. Para cada modelo e cada tamanho de payload, a sessão pai e os agentes concorrentes executam comandos shell em foreground que produzem grandes quantidades de x no stdout:

node -e "const chunk='x'.repeat(1024*1024); for (let i=0; i<N; i++) process.stdout.write(chunk)"

Tamanhos de payload:

  • 128 MiB
  • 256 MiB
  • 512 MiB
  • 1024 MiB
  • 1536 MiB
  • 2048 MiB

Este é um estresse sintético, não uma carga de trabalho normal de revisão de código. Seu objetivo é verificar se um stdout foreground extremamente grande ainda pode levar o Qwen Code a caminhos de memória perigosos.

Por que o RSS excede o Heap Limit do Node

O heap limit do Node nesta máquina é de aproximadamente 4144 MiB, mas o RSS da árvore de processos atingiu cerca de 5964,7 MiB. Isso não é contraditório:

  • O RSS inclui heap V8, buffers externos, alocações nativas, módulos carregados, processos filhos e memória que não é contabilizada no heap old-space.
  • O pico na fase amplificada geralmente ocorre em processos filhos do Node, não no wrapper raiz.
  • Portanto, o RSS raiz pode permanecer baixo enquanto o RSS da árvore de processos é alto.

Comparação com Relatórios Anteriores

No benchmark do mantenedor de 18 de maio, os picos de RSS da árvore de processos do Qwen em tarefas normais ficaram principalmente entre 942,5 MiB e 1006,6 MiB. Esses dados indicam principalmente que, na época, a pegada de memória do Qwen Code durante as tarefas era significativamente maior que a do Claude Code.

O relatório de diagnóstico de runtime de 19 de maio dividiu ainda mais duas linhas:

  • Startup/config RSS: frequentemente elevado por configurações normais e processos filhos do MCP;
  • OOM de heap de sessão longa: mais propenso à pressão de histórico/compactação/clone.

Os dados de 21 de maio desta rodada apoiam ainda mais essa divisão:

  • O cenário de tarefas longas reais com múltiplos agentes e sessão longa manteve-se estável abaixo de 0,9 GiB, sem atingir OOM de heap;
  • O stdout amplificado pode empurrar o RSS da árvore de processos para 5,96 GiB e, em 2048 MiB, desencadeou outro tipo de fatal de comprimento de string do V8.

Portanto, em comparação com o OOM de heap de sessão longa tradicional da issue original, a construção atual é claramente mais estável. No entanto, isso não significa que o Qwen Code não tenha mais nenhum risco de memória com saída grande.

Nova Descoberta: Caminho Fatal de Stdout Foreground Extremo

O novo problema descoberto nesta rodada não é o OOM de heap original, mas um fatal de string/decodificação sob stdout foreground de vários GiB:

Fatal error in , line 0
Check failed: i::kMaxInt >= len.
...
v8::String::NewFromOneByte
node::StringBytes::Encode
node::encoding_binding::BindingData::DecodeUTF8

Condição de disparo:

  • Modelo: DeepSeek/deepseek-v4-pro
  • Payload: 2048 MiB
  • Pico de RSS da árvore: 4660,4 MiB
  • Maior RSS de processo: 4527,6 MiB
  • O runner registrou saída: SIGTERM, pois após a captura da saída fatal, os processos filhos restantes continuavam em alta CPU e foram manualmente terminados.

pai/glm-5 e qwen3.6-plus também falharam em 2048 MiB, com stdout vazio e código de saída 1, mas o stderr não capturou a pilha de fatal do V8.

Gravidade

Este é um problema real de robustez, mas a condição de disparo é stdout foreground de vários GiB, não tarefas normais de revisão de código. Também não prova que a correção atual de pressão de heap de sessão longa falhou.

Foi introduzido por este PR?

Nesta rodada, não há evidências de que a falha de stdout em 2048 MiB seja uma regressão introduzida pelo PR de memória atual.

Motivos:

  • O caminho da falha é a decodificação/construção de string de stdout foreground do shell.
  • O caminho da issue original é pressão de histórico de sessão longa, compactação e clonagem.
  • Esta rodada não possui uma linha de base pré-PR com o mesmo payload, portanto não pode ser atribuída como regressão.
  • A falha ocorre apenas com o payload extremo de 2048 MiB; os payloads de 128 MiB a 1536 MiB foram concluídos com sucesso.

Recomenda-se tratá-la como um follow-up dedicado de saída grande: fazer stream/spool/hard-cap da saída do shell foreground mais cedo, evitando construir strings JS de vários GiB na memória. A menos que o objetivo atual do PR inclua explicitamente “qualquer stdout foreground de vários GiB deve ser tratável”, não é recomendado tratá-lo como um bloqueador para este PR.

Conclusão

  1. O build local mais recente do 0.15.11 é significativamente melhor no caminho de OOM de heap de sessão longa reportado na issue. Com base nos resultados de teste de estresse de heap padrão, pode-se considerar que este PR basicamente resolveu o caminho de reprodução do OOM de heap de sessão longa encontrado anteriormente.

  2. Com o heap padrão do Node, as tarefas longas reais com loop de revisão de múltiplos agentes não reproduziram o OOM de heap V8 tradicional nos três modelos pai/glm-5, qwen3.6-plus e DeepSeek/deepseek-v4-pro.

  3. O teste de estresse sintético de stdout foreground ainda pode elevar bastante o RSS da árvore de processos. A construção atual suportou payloads de 1536 MiB nos três modelos, com o maior RSS de árvore bem-sucedido sendo 5964,7 MiB.

  4. Ainda existe um problema independente de saída grande extrema: próximo a 2048 MiB de stdout, o Qwen Code pode falhar antes de gerar o resultado JSON; no caso do DeepSeek, foi capturado um fatal de comprimento de string do V8.

  5. Esta nova descoberta é importante, mas parece ser mais um problema de robustez de saída grande para acompanhamento, e não deve ser tratada como um bloqueador para a mitigação de pressão de heap de sessão longa.

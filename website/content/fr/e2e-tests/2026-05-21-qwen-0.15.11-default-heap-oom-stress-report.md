# Rapport de test de stress Heap OOM par défaut pour Qwen Code 0.15.11

Date : 2026-05-21

## Périmètre du test

Ce rapport documente une série de tests de stress par défaut de la heap menés sur la dernière version locale de Qwen Code `0.15.11`.
L'objectif de ces tests est de vérifier si, sans abaisser artificiellement la limite mémoire, le code actuel reproduit encore
les OOM de session longue mentionnés dans l'issue, et s'il existe de nouveaux risques dans des scénarios de sortie extrêmes.

Cette série couvre trois modèles :

- `pai/glm-5`
- `qwen3.6-plus`
- `DeepSeek/deepseek-v4-pro`

Les tests sont divisés en deux parties :

1. Tâches longues réelles et boucles de revue multi‑agent concurrentes.
2. Test de stress amplifié du foreground stdout, c'est-à-dire en amplifiant la pression sur le chemin `tool-output` via un foreground shell stdout de grande taille.

## Conclusions principales

La dernière version locale se comporte nettement mieux sur le chemin « OOM de la heap V8 en session longue », qui était au cœur de l'issue.
Sur la base de ces tests de stress avec heap par défaut, modèles multiples, agents multiples et tâches longues, on peut considérer que cette PR résout quasiment le problème de heap OOM en session longue rencontré précédemment, du moins dans la dimension de reproduction actuelle.

Les tests réels de tâches longues et de concurrence multi‑agent ont exécuté :

- 23 worker tours
- Environ 719 094 118 tokens totaux rapportés
- 77 appels d'agent (tool call)
- 856 appels d'outil au total

Cette partie n'a reproduit aucun symptôme classique de heap OOM V8 :

- `JavaScript heap out of memory`
- `Reached heap limit`
- `Ineffective mark-compacts near heap limit`
- `Allocation failed`

Le RSS maximal de l'arbre de processus pendant la phase de tâches longues réelles était de 874,7 MiB, et le RSS maximal du processus racine de 219,1 MiB. Cela montre que, sous la heap par défaut, le code actuel ne reproduit pas facilement le heap OOM des sessions longues décrit dans l'issue.

La deuxième phase, le test de stress amplifié du stdout, est plus agressive. Elle a exécuté 18 tentatives de charge utile, couvrant les trois modèles et des charges utiles de foreground stdout allant de 128 MiB à 2048 MiB.

Résultats :

- Les trois modèles ont réussi la charge utile de 1536 MiB.
- Le RSS maximal réussi de l'arbre de processus était de 5964,7 MiB, observé avec `qwen3.6-plus` pour une charge de 1536 MiB.
- Avec une charge de 2048 MiB, un nouvel échec extrême de sortie volumineuse est apparu.

Résultats pour 2048 MiB :

- `pai/glm-5` : `exit=1`, stdout vide, pas de texte OOM standard.
- `qwen3.6-plus` : `exit=1`, stdout vide, pas de texte OOM standard.
- `DeepSeek/deepseek-v4-pro` : fatal V8 : `Check failed: i::kMaxInt >= len`, pile dans `v8::String::NewFromOneByte` / `node::StringBytes::Encode` / `DecodeUTF8`.

Ce nouveau problème n'est pas le heap OOM classique des sessions longues de l'issue. Il s'agit plutôt d'une limitation de longueur de chaîne V8 ou d'un problème de traitement des sorties volumineuses déclenché lors du décodage/construction d'une chaîne JS à partir d'un foreground stdout de plusieurs GiB. Il est recommandé de le suivre comme un problème de suivi de sortie volumineuse, et non comme un échec de la correction actuelle de la pression mémoire en session longue.

## Phase 1 : Test de stress de tâches longues réelles et d'agents multiples

### Format du test

Chaque worker de modèle utilise la même session en répétant `--resume`. Chaque tour demande à Qwen Code :

- d'effectuer des revues de code en lecture seule et des recherches de code ;
- de lancer simultanément au moins 4 appels d'outil `agent` dans le même tour ;
- d'inspecter particulièrement les zones mémoire liées à l'historique de chat, la compaction, l'exécution des sous‑agents, le streaming non interactif, les adaptateurs de fournisseur, etc. ;
- de conserver une réponse finale suffisamment détaillée pour que l'historique de session s'allonge naturellement.

Le runner échantillonne le RSS de l'arbre de processus chaque seconde, sans définir de limite de heap supplémentaire.

Cette partie a été arrêtée volontairement par `SIGTERM` après avoir observé une certaine stabilité de la mémoire, afin de passer à la deuxième phase de test de stress amplifié du stdout. Par conséquent, le `SIGTERM` dans le tableau n'est pas un OOM.

### Interprétation de la Phase 1

Ce sont les données les plus représentatives de l'amélioration significative du OOM en session longue.

Cette série est plus lourde que la petite revue de PR / navigation de code du 18 mai : elle inclut davantage de `--resume`, plus d'activité de sous‑agents, un volume de tokens signalé plus important et plus d'appels d'outils. Pourtant, le RSS de l'arbre de processus est resté inférieur à 0,9 GiB, sans aucun OOM V8 classique.

Cela ne prouve pas que tous les OOM utilisateur seront impossibles, mais cela montre qu'avec la version actuelle, sous la heap par défaut, il n'est plus possible de reproduire facilement le OOM de session longue de l'issue.

## Phase 2 : Test de stress amplifié du Foreground Stdout

### Format du test

La deuxième phase amplifie délibérément la pression sur le chemin de sortie shell. Pour chaque modèle et chaque taille de charge utile, la session parente et les agents concurrents exécutent des commandes shell en avant‑plan qui génèrent une grande quantité de `x` vers stdout :

```bash
node -e "const chunk='x'.repeat(1024*1024); for (let i=0; i<N; i++) process.stdout.write(chunk)"
```

Taille de la charge utile :

- 128 MiB
- 256 MiB
- 512 MiB
- 1024 MiB
- 1536 MiB
- 2048 MiB

Il s'agit d'un stress synthétique, pas d'une charge de travail normale de revue de code. Son but est de vérifier si un très grand stdout en avant‑plan peut encore pousser Qwen Code sur un chemin mémoire dangereux.

### Pourquoi le RSS dépasse la limite de heap Node

La limite de heap Node locale est d'environ 4144 MiB, mais le RSS maximal de l'arbre de processus atteint environ 5964,7 MiB. Ce n'est pas contradictoire :

- Le RSS inclut la heap V8, les buffers externes, les allocations natives, les modules chargés, les processus enfants et la mémoire non comptée dans la heap old‑space.
- Les pics de la phase amplifiée apparaissent généralement dans le processus enfant Node, pas dans le wrapper racine.
- Ainsi, le RSS racine peut rester bas tandis que le RSS de l'arbre de processus est élevé.

## Comparaison avec les rapports précédents

Dans le benchmark du responsable maintenance du 18 mai, les pics de RSS de l'arbre de processus de Qwen pour les tâches normales se situaient entre 942,5 MiB et 1006,6 MiB. Ces données montraient principalement que l'empreinte mémoire de Qwen Code pendant les tâches était nettement supérieure à celle de Claude Code.

Le rapport de diagnostic d'exécution du 19 mai avait affiné l'analyse en deux pistes :

- RSS de démarrage/configuration : souvent gonflé par la configuration normale et les processus enfants MCP ;
- OOM de session longue : plutôt lié à la pression de l'historique, de la compaction et du clonage.

Les données du 21 mai confirment cette distinction :

- Les sessions longues réelles avec agents multiples sont restées stables sous 0,9 GiB, sans déclencher de heap OOM ;
- Le stdout amplifié peut pousser le RSS de l'arbre de processus à 5,96 GiB et déclencher un autre type de fatal lié à la longueur de chaîne V8 à 2048 MiB.

Par conséquent, par rapport au heap OOM classique des sessions longues de l'issue, la version actuelle est nettement plus stable. Cela ne signifie pas que Qwen Code est exempt de tout risque mémoire lié aux sorties volumineuses.

## Nouvelle découverte : Chemin fatal du Foreground Stdout extrême

Le nouveau problème découvert dans cette série n'est pas le heap OOM d'origine, mais un fatal de chaîne/décodage sous un foreground stdout de plusieurs GiB :

```text
Fatal error in , line 0
Check failed: i::kMaxInt >= len.
...
v8::String::NewFromOneByte
node::StringBytes::Encode
node::encoding_binding::BindingData::DecodeUTF8
```

Conditions de déclenchement :

- Modèle : `DeepSeek/deepseek-v4-pro`
- Charge utile : 2048 MiB
- Pic RSS de l'arbre : 4660,4 MiB
- Plus grand RSS de processus : 4527,6 MiB
- Sortie enregistrée par le runner : `SIGTERM`, car après la capture du fatal, les processus enfants restants tournaient à vide à forte charge CPU et ont été arrêtés manuellement.

`pai/glm-5` et `qwen3.6-plus` ont également échoué à 2048 MiB, avec un stdout vide et un code de sortie `1`, mais stderr n'a pas capturé de pile V8 fatale.

### Sévérité

C'est un vrai problème de robustesse, mais sa condition de déclenchement est un foreground stdout de plusieurs GiB, ce qui n'est pas le cas des tâches normales de revue de code. Cela ne signifie pas non plus que la correction actuelle de la pression mémoire en session longue a échoué.

### Est-ce introduit par cette PR ?

Rien dans cette série ne prouve que l'échec à 2048 MiB soit une régression introduite par la PR de mémoire actuelle.

Raisons :

- Le chemin d'échec est le décodage/construction de chaîne du foreground shell stdout.
- Le chemin de l'issue est la session longue avec historique, compaction et pression de clonage.
- Aucune baseline pré‑PR avec la même charge utile n'a été réalisée, donc on ne peut pas l'attribuer à une régression.
- Cet échec n'apparaît que pour la charge utile extrême de 2048 MiB ; les charges de 128 MiB à 1536 MiB sont toutes exécutées avec succès.

Il est recommandé d'en faire un suivi dédié aux sorties volumineuses : diffuser/mettre en file d'attente/tronquer plus tôt la sortie du foreground shell, pour éviter de construire en mémoire une chaîne JS de plusieurs GiB. À moins que l'objectif de cette PR n'inclue explicitement « pouvoir traiter n'importe quel stdout de plusieurs GiB en avant‑plan », il n'est pas conseillé d'en faire un blocant pour cette PR.

## Conclusions

1. La version locale `0.15.11` est nettement meilleure sur le plan du heap OOM en session longue rapporté dans l'issue. Sur la base des tests de stress actuels avec heap par défaut, on peut considérer que cette PR a quasiment résolu le chemin de reproduction du heap OOM en session longue.

2. Sous la heap par défaut de Node, les tâches longues réelles combinées à des boucles de revue multi‑agents n'ont reproduit aucun OOM V8 classique sur les trois modèles `pai/glm-5`, `qwen3.6-plus` et `DeepSeek/deepseek-v4-pro`.

3. Le test de stress synthétique du foreground stdout peut encore pousser le RSS de l'arbre de processus très haut. La version actuelle a survécu à la charge de 1536 MiB sur les trois modèles, avec un RSS d'arbre maximal réussi de 5964,7 MiB.

4. Il subsiste un problème indépendant de sortie extrêmement volumineuse : autour de 2048 MiB de stdout, Qwen Code peut échouer avant de produire un résultat JSON ; le cas DeepSeek a capturé un fatal V8 sur la longueur de chaîne.

5. Cette nouvelle découverte est importante, mais constitue davantage un problème de robustesse des sorties volumineuses à suivre, et ne devrait pas être considérée comme un blocant direct de l'atténuation de la pression mémoire en session longue.

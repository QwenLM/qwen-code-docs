# Qwen Code 0.15.11 Standard Heap OOM Stresstest-Bericht

Datum: 2026-05-21

## Testumfang

Dieser Bericht dokumentiert einen standardmäßigen Heap-Stresstest mit dem aktuellen lokalen Build von Qwen Code `0.15.11`.
Ziel dieser Testrunde ist es zu überprüfen, ob der aktuelle Code den in einem Issue erwähnten Long-Session-OOM ohne künstliche Reduzierung des Memory-Limits noch reproduzieren kann und ob es in extremen Large-Output-Szenarien neue Risiken gibt.

Diese Runde deckt drei Modelle ab:

- `pai/glm-5`
- `qwen3.6-plus`
- `DeepSeek/deepseek-v4-pro`

Die Tests sind in zwei Teile gegliedert:

1. Realistische Langzeitaufgaben mit Multi-Agent-gleichzeitiger Review-Schleife.
2. Amplified foreground stdout Stresstest, d.h. Verstärkung des Tool-Output-Drucks durch massiven foreground shell stdout.

## Kernaussagen

Der aktuellste lokale Build zeigt auf dem Pfad, der im Issue am meisten beachtet wird („Long-Session V8 Heap OOM“), eine deutlich bessere Performance.
Basierend auf diesem Standard-Heap-, Multi-Modell-, Multi-Agent- und Langzeit-Stresstest kann davon ausgegangen werden, dass dieser PR das zuvor aufgetretene Long-Session-Heap-OOM-Problem im Wesentlichen gelöst hat, zumindest unter den aktuellen Reproduktionsbedingungen kann der ursprüngliche Heap-OOM nicht mehr reproduziert werden.

Der realistische Langzeit-Multi-Agent-Test führte Folgendes aus:

- 23 Worker-Turns
- ca. `719.094.118` reported total tokens
- 77 Agent-Tool-Calls
- 856 Tool-Calls insgesamt

In diesem Teil wurden keine traditionellen V8-Heap-OOM-Merkmale reproduziert:

- `JavaScript heap out of memory`
- `Reached heap limit`
- `Ineffective mark-compacts near heap limit`
- `Allocation failed`

Der maximale Process-Tree-RSS in der realistischen Langzeitphase betrug `874,7 MiB`, der maximale Root-Process-RSS betrug `219,1 MiB`. Dies zeigt, dass der aktuelle Code unter dem Standard-Heap den im ursprünglichen Issue beschriebenen Heap-OOM bei Langzeitaufgaben nicht ohne weiteres reproduziert.

Die zweite Phase, der amplifizierte stdout-Stresstest, war aggressiver. Er führte insgesamt 18 Payload-Versuche durch und deckte drei Modelle und foreground stdout Payloads von `128 MiB` bis `2048 MiB` ab.

Die Ergebnisse waren:

- Alle drei Modelle haben den `1536 MiB` Payload erfolgreich durchlaufen.
- Der höchste erfolgreiche Process-Tree-RSS war `5964,7 MiB`, aufgetreten bei `qwen3.6-plus` mit `1536 MiB` Payload.
- Bei `2048 MiB` Payload trat ein neuer, extremer Large-Output-Fehler auf.

Die Ergebnisse für `2048 MiB`:

- `pai/glm-5`: `exit=1`, stdout leer, kein Standard-OOM-Text.
- `qwen3.6-plus`: `exit=1`, stdout leer, kein Standard-OOM-Text.
- `DeepSeek/deepseek-v4-pro`: Es trat ein V8-Fatal auf:
  `Check failed: i::kMaxInt >= len`, Stack in
  `v8::String::NewFromOneByte` / `node::StringBytes::Encode` /
  `DecodeUTF8`.

Dieses neue Problem ist nicht der traditionelle Long-Session-Heap-OOM aus dem ursprünglichen Issue. Es ähnelt eher einem V8-String-Längenlimit oder einem Large-Output-Verarbeitungsproblem, das beim Dekodieren/Konstruieren eines Multi-GiB-foreground-stdout in einen JS-String ausgelöst wird. Es wird empfohlen, dies als Large-Output-Follow-up zu verfolgen und nicht als Fehlschlag der aktuellen Long-Session-Heap-Pressure-Behebung zu betrachten.

### Testform

Jeder Modell-Worker verwendet dieselbe Session und führt wiederholt `--resume` aus. In jeder Runde wird von Qwen Code gefordert:

- Durchführung einer read-only Code-Review und Codesuche;
- Gleichzeitiges Starten von mindestens 4 `agent` tool calls in derselben Runde;
- Besondere Prüfung von Chat-History, Compaction, Subagent-Runtime, Non-Interactive-Streaming, Provider-Adaptern und anderen speicherbezogenen Bereichen;
- Beibehaltung einer ausreichend detaillierten finalen Antwort, damit die Session-History natürlich wachsen kann.

```bash
node -e "const chunk='x'.repeat(1024*1024); for (let i=0; i<N; i++) process.stdout.write(chunk)"
```

- `128 MiB`
- `256 MiB`
- `512 MiB`
- `1024 MiB`
- `1536 MiB`

Dies ist synthetischer Stress, keine normale Code-Review-Arbeitslast. Es dient dazu zu überprüfen, ob extrem großer foreground-stdout Qwen Code in gefährliche Speicherpfade treiben kann.

### Warum RSS die Node-Heap-Limit überschreitet

Das lokale Node-Heap-Limit beträgt ca. `4144 MiB`, aber der Process-Tree-RSS erreichte maximal ca. `5964,7 MiB`. Das ist kein Widerspruch:

- RSS umfasst V8-Heap, externe Buffer, native Allokationen, geladene Module, Child-Prozesse sowie Speicher, der nicht im old-space-Heap gezählt wird.
- Die Spitzenwerte in der amplifizierten Phase treten typischerweise im Child-Node-Prozess auf, nicht im Root-Wrapper.
- Daher kann der Root-RSS niedrig bleiben, während der Process-Tree-RSS hoch ist.

## Vergleich mit historischen Berichten

Im Maintainer-Benchmark vom 18. Mai lagen die Process-Tree-RSS-Spitzenwerte von Qwen bei normalen Aufgaben meist zwischen `942,5 MiB` und `1006,6 MiB`. Diese Daten zeigten hauptsächlich, dass der Task-Footprint von Qwen Code damals deutlich höher war als der von Claude Code.

Der Runtime-Diagnostics-Bericht vom 19. Mai trennte weiter zwei Linien auf:

- Startup/Config-RSS: Wird oft durch normale Konfiguration und MCP-Child-Prozesse erhöht;
- Long-Session-Heap-OOM: Tendiert mehr zu History/Compaction/Clone-Druck.

Die Daten vom 21. Mai stützen diese Aufteilung weiter:

- Realistische Langzeit-Multi-Agent-Langzeitszenarien blieben stabil unter `0,9 GiB`, ohne Heap-OOM;
- Amplified Stdout konnte den Process-Tree-RSS auf `5,96 GiB` treiben und bei `2048 MiB` einen anderen V8-String-Längen-Fatal auslösen.

Daher ist der aktuelle Build im Vergleich zum traditionellen Long-Session-Heap-OOM aus dem ursprünglichen Issue deutlich stabiler. Dies bedeutet jedoch nicht, dass Qwen Code keinerlei Large-Output-Memory-Risiken mehr hat.

## Neue Erkenntnis: Extremfall Foreground Stdout Fatal Path

Das in dieser Runde neu entdeckte Problem ist nicht der ursprüngliche Heap-OOM, sondern ein String/Decoding-Fatal bei Multi-GiB-foreground-stdout:

```text
Fatal error in , line 0
Check failed: i::kMaxInt >= len.
...
v8::String::NewFromOneByte
node::StringBytes::Encode
node::encoding_binding::BindingData::DecodeUTF8
```

Auslösebedingungen:

- Modell: `DeepSeek/deepseek-v4-pro`
- Payload: `2048 MiB`
- Peak tree RSS: `4660,4 MiB`
- Largest process RSS: `4527,6 MiB`
- Runner-Protokoll Exit: `SIGTERM`, da nach Erfassung des Fatals die verbleibenden Child-Prozesse noch unter hoher CPU-Leerlauf liefen und manuell beendet wurden.

`pai/glm-5` und `qwen3.6-plus` scheiterten bei `2048 MiB` ebenfalls mit leerem stdout und exit code `1`, aber stderr enthielt keinen V8-Fatal-Stack.

### Schweregrad

Dies ist ein echtes Robustheitsproblem, aber die Auslösebedingung ist Multi-GiB-foreground-stdout, keine normale Code-Review-Aufgabe. Es beweist auch nicht, dass die aktuelle Long-Session-Heap-Pressure-Behebung fehlgeschlagen ist.

### Wurde dies durch diesen PR eingeführt?

Es gibt keine Hinweise darauf, dass das `2048 MiB`-stdout-Problem eine durch den aktuellen Memory-PR eingeführte Regression ist.

Gründe:

- Der Fehlerpfad ist foreground shell stdout decode / string construction.
- Der Pfad des ursprünglichen Issues war Long-Session-History, Compaction, Clone-Druck.
- In dieser Runde wurde keine Pre-PR-Baseline mit derselben Payload durchgeführt, daher kann es nicht als Regression eingestuft werden.
- Der Fehler tritt nur beim extremen `2048 MiB` Payload auf; `128 MiB` bis `1536 MiB` wurden erfolgreich abgeschlossen.

Es wird empfohlen, dies als dediziertes Large-Output-Follow-up zu behandeln: früheres Streaming/Spooling/Hard-Cap des foreground shell outputs, um die Konstruktion von Multi-GiB-JS-Strings im Speicher zu vermeiden. Sofern das Ziel des aktuellen PR nicht ausdrücklich „beliebiger Multi-GiB-foreground-stdout muss handhabbar sein“ umfasst, wird nicht empfohlen, dies als Blocker für den aktuellen PR zu betrachten.

## Fazit

1. Der aktuelle lokale `0.15.11` Build ist auf dem im Issue berichteten Long-Session-Heap-OOM-Pfad deutlich besser. Basierend auf den aktuellen Standard-Heap-Stresstest-Ergebnissen kann davon ausgegangen werden, dass dieser PR den zuvor beobachteten Long-Session-Heap-OOM-Reproduktionspfad im Wesentlichen gelöst hat.

2. Unter dem Standard-Node-Heap wurde bei realistischen Langzeitaufgaben + Multi-Agent-Review-Schleife bei den drei Modellen `pai/glm-5`, `qwen3.6-plus` und `DeepSeek/deepseek-v4-pro` kein traditioneller V8-Heap-OOM reproduziert.

3. Synthetischer foreground stdout Stresstest kann den Process-Tree-RSS dennoch sehr hoch treiben. Der aktuelle Build überstand bei allen drei Modellen den `1536 MiB` Payload; der höchste erfolgreiche Tree-RSS betrug `5964,7 MiB`.

4. Es gibt weiterhin ein separates extremales Large-Output-Problem: Bei etwa `2048 MiB` stdout kann Qwen Code fehlschlagen, bevor die JSON-Ergebnisse ausgegeben werden; im Fall von DeepSeek wurde ein V8-String-Längen-Fatal erfasst.

5. Diese neue Erkenntnis ist wichtig, aber eher ein nachgelagertes Large-Output-Robustheitsproblem und sollte nicht direkt als Blocker für die Long-Session-Heap-Pressure-Minderung betrachtet werden.

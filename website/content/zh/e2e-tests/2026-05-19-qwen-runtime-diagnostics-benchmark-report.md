### 裸模式配置归因检查

第二次后续测试使用 `qwen3.6-plus`，采用相同的 PR 审查提示形状，分别在已安装 CLI 和本地捆绑包上运行。这不是一个常规的端到端业务基准测试，而是一次仅针对启动/配置内存的受控归因检查。

`--bare` 模式改变了运行时输入：它跳过正常的全局设置发现、MCP 启动、钩子、隐式上下文、技能以及其他启动集成。因此，当模型提供商仅配置在全局设置中时，它可能失败或行为不同。本次运行中，模型凭证仅通过子进程环境提供，因为裸模式故意不加载正常的提供商设置。不会写回用户的全局配置。

本次运行未产生有用的 token/工具调用统计：模型在一轮内完成，并未调用所请求的 shell 命令。请勿将这些行视为正常任务基准测试结果，也不得将其 token/工具调用行为与上述矩阵进行比较。它们仅用于估算正常配置和已配置的子进程贡献了多少进程树 RSS。

| CLI                 | 模式     | 耗时 | 轮数 | 工具使用次数 | 进程树 RSS 峰值 | 根进程 RSS 峰值 | 进程数峰值 |
| ------------------- | -------- | ---: | ---: | ----------: | --------------: | --------------: | ---------: |
| PATH `qwen`         | 正常模式 | 5.5s |    1 |           0 |     1,021.3 MiB |       251.5 MiB |          5 |
| PATH `qwen`         | `--bare` | 2.4s |    1 |           0 |       525.7 MiB |       246.4 MiB |          2 |
| 本地 `dist/cli.js`  | 正常模式 | 4.9s |    1 |           0 |     1,046.2 MiB |       213.3 MiB |          5 |
| 本地 `dist/cli.js`  | `--bare` | 2.3s |    1 |           0 |       454.3 MiB |       216.5 MiB |          3 |

结果证实了启动/配置归因的进程树假设。在此机器上，正常配置相比 `--bare` 大约增加了 0.50-0.59 GiB 的用户可见进程树 RSS，而根进程 RSS 保持在 0.21-0.25 GiB 的相同区间。在正常配置的峰值时，额外的 RSS 同样来自额外的 Node/MCP 子进程，包括一个 Chrome DevTools MCP 进程及其设置包装进程。`--bare` 移除了这些启动/配置子进程，使得安装版和本地运行的树 RSS 回落到 0.45-0.53 GiB 范围。

### 临时设置 MCP / 钩子隔离

- MCP 禁用：清空 `mcpServers` 并清空 MCP 允许/排除列表。
- 钩子禁用：将 `disableAllHooks` 设为 true。

未修改任何全局设置。该用例使用 `qwen3.6-plus` 和一个最小的启动提示词，因此它测量的是启动/配置进程树成本，而非任务推理质量。

## 观察

1. 在此本地捆绑运行中，进程树 RSS 与根 RSS 几乎相同。根/树之间的差距通常低于 10 MiB。这意味着这些运行未显示持久的子进程内存占用者。主导进程是主 Node 进程。
2. 本地捆绑运行的峰值约为 0.36-0.41 GiB，而非之前的 0.83-1.04 GiB，因为矩阵使用了精简的临时配置。后续的正常配置合理性检查在 PATH `qwen` 和本地 `dist/cli.js` 上都复现了约 1.1 GiB 的进程树 RSS，额外的内存来自进程树中的子 MCP/Node 进程。
3. V8 堆远小于 RSS。结束堆大小约为 99-143 MiB，而结束 RSS 约为 216-411 MiB。剩余的内存占用可能来自加载的模块、原生分配、外部缓冲区或运行时开销（超出实时 JS 堆）。
4. 静态请求开销大且重复。每次请求的系统提示词约为 39-51 KiB，工具 schema 约为 37-40 KiB。这解释了为什么即使小任务，当模型进行多轮交互时也会产生高累计 token 数。
5. 大的差异输出在到达模型请求之前被截断。968 KiB 和 4.84 MiB 差异用例产生了约 772-799 KiB 的捕获工具结果，但模型面对的最大函数响应保持在约 25-41 KiB，最大请求体保持在约 105-122 KiB。这表明在模型面对路径上，截断/保存输出处理正在工作。
6. 尽管请求体保持有界，内存仍然在大输出用例上增加。例如，4.84 MiB GLM 运行达到了 414.2 MiB 进程树 RSS 和 410.7 MiB 结束 RSS；4.84 MiB qwen3.6-plus 运行结束时堆大小为 142.8 MiB。这表明大的工具输出仍可能影响本地捕获、规范化或保留的运行时状态，即使最终请求负载被限制。
7. 在此运行中，模型选择对轮数和 token 总数的影响大于对 RSS 的影响。`qwen3.6-plus` 平均 token 数和轮数高于 `pai/glm-5`，但其平均进程树 RSS 峰值略低。这支持了早前的结论：模型选择不是进程内存的主要解释。
## 更新后的工作推理

新的诊断使之前的假设更加精确：

- 现在，在本地诊断包中使用正常配置可以复现已安装 CLI 用户可见的 1 GiB 峰值。剥离后的运行应用于内部 Qwen 运行时归因；正常运行配置的运行应用于用户可见的进程树归因。
- 剥离配置和正常配置之间最大的可观察差异是进程树形状：正常配置会启动额外的 MCP/Node 子进程。这些子进程解释了在最小提示词完整性检查中，从约 0.35-0.55 GiB 跃升至约 1.1 GiB 的大部分绝对增长。
- `--bare` 后续实验在 `qwen3.6-plus` 上确认了相同的方向：对于相同的提示词形状，正常配置比 bare 模式增加了约 0.50-0.59 GiB 的进程树 RSS，而根 RSS 变化很小。
- 临时设置隔离是比 `--bare` 更好的归因测试：仅禁用 MCP 就可以将进程树 RSS 减少约 0.42-0.47 GiB，同时保持正常的设置加载路径。仅禁用 hooks 在无工具调用的情况下没有显示出明显的 RSS 变化。
- 逐 MCP 隔离表明 `chrome-devtools` 是主要的 MCP 贡献者：它本身足以重现高 RSS 带，移除它后运行结果接近无 MCP 基线。
- 在本地 Qwen 运行时中，最可疑的区域不再是“发送给模型的原始差异字节数”。模型面对的请求体是有界的。
- 更强的怀疑对象是静态的每次请求上下文成本、重复的请求轮次、工具模式（schema）大小，以及在模型面对截断之前或之外，对大型工具输出进行本地保留/捕获。
- 由于 RSS 仍然远高于 V8 堆，下一个性能分析层应包括模块/启动开销、外部内存，以及围绕工具执行和最终响应输出的堆快照。

## 当前诊断的 RSS 归因

当前的计数器没有精确定位到具体的保留对象或源文件，但它们确实缩小了在这些本地运行中驱动 RSS 的因素范围：

| 信号                           | 当前证据                                                                                                             | RSS 影响                                                                                                                             |
| ---------------------------- | -------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------- |
| 根 RSS 与进程树 RSS           | 根 RSS 和进程树 RSS 峰值通常在约 2-10 MiB 之间；DeepSeek 大 PR 的差距最大，约为 23.6 MiB                                            | 在此本地捆绑包运行中，没有持久的子进程可以解释 RSS；主 Node 进程占主导地位                                                             |
| 正常配置进程树                 | 最小提示的正常配置运行达到了约 1.1 GiB 的进程树 RSS，而根 RSS 保持在约 213-250 MiB                                                   | 用户可见的 1 GiB 峰值可能由 MCP/Node 子进程主导，而非仅由 Qwen 根 RSS 贡献                                                                  |
| `--bare` 比较                 | `qwen3.6-plus` 正常运行峰值约为 1.02-1.05 GiB 进程树 RSS；bare 运行峰值约为 0.45-0.53 GiB                                        | 在此环境中，加载正常配置会增加约 0.50-0.59 GiB 的进程树 RSS                                                                                 |
| 临时 MCP 隔离                 | 清除 MCP 服务器后，启动/配置的进程树 RSS 从 865-1,017 MiB 降至 443-549 MiB                                                    | 在受控配置检查中，MCP 启动和 MCP 子进程解释了约 0.42-0.47 GiB 的进程树 RSS                                                                     |
| 逐 MCP 隔离                   | 仅 `chrome-devtools` 在重复样本中达到约 1.0 GiB；没有它时运行保持在约 461 MiB                                                   | 在此环境中，`chrome-devtools` 是主要的 MCP 进程树 RSS 贡献者                                                                               |
| 临时 hooks 隔离               | 在无工具调用的情况下，`disableAllHooks=true` 并保持 MCP 启用，仅改变了约 13-18 MiB 的进程树 RSS                                         | 在此处，仅 Hook 配置并不是明显的启动 RSS 驱动因素；Hook 执行仍需在工具调用基准测试中评估                                                           |
| V8 堆 vs RSS                  | 结束时堆大小约为 99-143 MiB，而结束时 RSS 约为 216-411 MiB                                                                      | 实时 JS 堆并非全部内存占用；加载的模块、原生分配、外部缓冲区或运行时开销可能很显著                                                                        |
| PR/差异大小 vs RSS            | DeepSeek 小/中/大 PR 的变化行数从 1 到 4,750，但进程树 RSS 保持在狭窄的 340.7-360.0 MiB 范围内                                           | 一旦工具输出有界，原始 PR 大小不会线性驱动 RSS                                                                                                |
| 工具输出大小                   | 大差异运行捕获到约 772-799 KiB 的工具结果，并显示出一些较高的结束 RSS / 堆，但 RSS 并非线性增长                                                 | 工具结果捕获/规范化增加了压力，尤其是在大输出情况下，但不太可能是唯一的 RSS 驱动因素                                                                        |
| 请求体大小                     | 模型面对的最大请求体大小约为 103-289 KiB，而 RSS 保持在相似范围内                                                                   | 请求序列化大小对 token 和延迟的影响比 RSS 峰值更明显                                                                                            |
| 每次请求的静态上下文           | 系统提示词约为 39-51 KiB，每个请求的工具模式（schema）约为 37-48 KiB                                                                      | 重复轮次是 token/成本的放大器；这本身不能解释 RSS，但可能是 token 压力的一个优化目标                                                                     |

正在进行的归因：在剥离的本地捆绑包基准测试中，RSS 底限看起来主要是任务时的运行时/模块/原生占用的内存，大型工具输出增加了递增的压力。在正常运行配置下，用户可见的 1 GiB 进程树峰值主要是进程树组成：Qwen 根进程加上 MCP/Node 子进程。下一个针对性测量应将 Qwen 根诊断与已配置的 MCP 服务器诊断分开，然后在 Qwen 根进程内部添加启动/模块/外部内存检查点。

## 进度快照

当前确认的信号：

1.  用户可见的 1 GiB 启动/配置峰值可以通过已安装的 CLI 和本地诊断包在加载正常配置时复现。它主要不是由诊断分支或 PR `#4186` 解释的。
2.  在此环境中，该 1 GiB 峰值主要是进程树组成：Qwen 根进程 + 重新启动的子进程 + MCP 子进程。
3.  `chrome-devtools` 是当前配置中主要的已配置 MCP 贡献者。它本身足以重现高进程树 RSS 带，即使提示词没有显式使用该 MCP。
4.  无 MCP 的正常重新启动形状的进程树 RSS 仍保持在约 0.45 GiB 左右。在启动归因检查中，没有重新启动父进程的单个 Qwen 运行时进程更接近 0.22-0.24 GiB。这意味着 0.45 GiB 的基线并不是单进程的根 RSS 数字。
5.  在剥离的非交互式任务运行中，模型选择对轮次、token 总数、延迟和请求大小的改变比 RSS 更明显。RSS 在 `pai/glm-5`、`qwen3.6-plus` 和 `DeepSeek/deepseek-v4-pro` 之间保持在相对较窄的范围内。
6.  当前短任务诊断显示，模型面对的工具/函数响应是有界的，但本地工具结果捕获和运行时状态仍然可以在大输出情况下增加堆/RSS。这使得大输出保留仍处于调查路径上。

当前差距：

1.  短任务基准测试矩阵仍然寿命较短。后续的交互式长评审运行确实复现了一次 41.9 分钟的失败，但这仍然是一个样本，需要重复运行和堆/对象归因。
2.  当前的计数器足以归因进程树 RSS 和请求大小，但不足以命名长会话期间保留的 JS 对象图。
3.  启动/配置 RSS 和长会话 OOM 必须保持为单独的轨道。MCP 和重新启动解释了较大的空闲/启动 RSS 带；它们本身并不能解释长时间任务后的 V8 堆 OOM。
4.  交互式 TUI 内存仍然需要与非交互模式分开运行，因为 UI 历史和 Ink 静态输出的加载方式不同。

初步解读：

-   未使用的已配置 MCP 会消耗内存，因为正常启动会连接到已配置的 MCP 服务器，并在任务需要它们之前通告其工具。在测量的配置中，`chrome-devtools` 会启动额外的 Node/npm MCP 进程，并将工具模式数量从 19 增加到 48。这解释了一个大的启动/配置 RSS 带，并且还可能增加重复请求的开销。
-   长会话 OOM 报告是另一层。GC 日志显示 Mark-Compact 释放的内存非常少，表明堆中充满了可到达的状态。最有力的候选是保留的历史/工具/UI 对象、全历史克隆、压缩中间结果以及流式/日志累加器。
-   PR `#4186` 是一个有用的缓解措施，因为它可以在 token 阈值触发之前根据堆压力进行压缩，并且它移除了一个不必要的全历史克隆。它不应被视为大型工具输出保留、UI 历史保留或压缩峰值内存问题已经解决的证据。

## 长任务验证计划

下一个基准测试应保持两个轨道分开：

1.  启动/配置归因：正常配置 vs MCP 禁用 vs 仅 `chrome-devtools` vs 无重新启动归因。这解释了用户在有意义的工作开始之前看到的情况。
2.  长任务运行时增长：重复工具调用、大输出、压缩、恢复和交互式 UI 历史。这解释了工作之后的 OOM。

推荐的长任务测试用例：

| 用例                          | 形态                                                                                                | 为什么重要                                                                                    |
| ----------------------------- | --------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------- |
| 长 PR 评审循环                | 重复中/大 PR 评审提示词 30、60 和 120 分钟，使用固定模型和固定配置                                          | 最接近报告的代理工作流；捕获轮次、工具调用、token 增长和 RSS/堆趋势                               |
| 大型工具输出保留              | 重复产生有界的 1 MiB / 5 MiB / 20 MiB 命令输出，然后提出后续问题                                          | 测试在模型面对截断后，原始输出是否仍在本地保留                                                 |
| 压缩压力                      | 使用较低的控制老生代限制和大型上下文提示词来触发堆压力压缩                                                | 验证 PR `#4186` 是否在 OOM 之前触发，以及压缩本身是否产生新的峰值                               |
| 交互式 TUI 历史               | 在 tmux TUI 模式下运行相同的长循环，并与非交互模式进行比较                                              | 隔离 UI 历史、Ink 静态输出、渲染的 diff 和终端输出显示保留                                     |
| 恢复压力测试                  | 恢复一个已保存的大会话并立即继续工作                                                                        | 针对 `/resume` OOM 报告和会话重建成本                                                         |
| 流式/日志累加器               | 强制输出长流式响应，分别启用和禁用遥测/日志                                                              | 测试问题分析中怀疑的 `collected responses` / 日志保留路径                                       |
| MCP 空闲 vs MCP 活跃          | 运行无 MCP、已配置但未使用的 `chrome-devtools` 以及主动使用 `chrome-devtools` 的变体                        | 区分空闲 MCP 子进程 RSS 与实际 MCP 工具执行和工具模式/token 开销                                 |

每轮或每个采样间隔应记录的指标：

-   根 RSS 当前/峰值和进程树 RSS 当前/峰值。
-   子进程数量和主要子进程命令形态。
-   V8 `heapUsed`、`heapTotal`、`heap_size_limit`、`external` 和 `arrayBuffers`。
-   轮次计数、请求计数、工具调用计数和工具调用轮次。
-   每个请求和整个任务的输入/输出/缓存/总 token 数。
-   请求体字节数、系统提示词字节数、工具模式字节数和函数响应字节数。
-   工具结果计数、捕获的总工具结果字节数、最大工具结果字节数和（如可用）保留的工具结果字节数。
-   对话历史消息数量和近似历史字节大小。
-   仅交互式的 UI 历史项数量和近似保留显示大小。
-   压缩尝试次数、压缩触发原因、压缩前后 token 数、压缩前后堆压力以及压缩失败状态。
-   当堆压力超过配置阈值时的堆快照或有界内存时间线产物。

验证标准：

1.  至少重复关键长任务用例两次。启动 RSS 存在可见的方差，应避免单次运行结论。
2.  分别报告根 RSS 和进程树 RSS。用户可见的内存压力可能来自子进程，而 V8 OOM 来自 Qwen 根堆。
3.  将平坦的 RSS 线视为重要证据。如果 token 和工具调用增长但堆/RSS 保持平坦，问题可能在其他地方。
4.  当 RSS 或堆增长时，将增长与特定信号关联：工具结果字节数、历史字节数、UI 历史计数、压缩事件、流式累加器大小或 MCP 进程启动。
5.  如果拍摄了堆快照，先编写结构化的诊断 JSON，然后是快照。堆快照可能很大且包含敏感字符串，因此应保持可选并仅限本地。

## 确定性大规模任务克隆压力复现

添加了一个确定性压力测试工具 `scripts/memory-pressure-repro.mjs`。它不调用模型。相反，它构建了一个类似 Qwen 的长会话对象图，包含根审查轮次、子代理转录本、大型工具结果、检查点 JSON 以及保留的 `structuredClone()` 副本。这为疑似来自用户提供的 OOM 栈的克隆和检查点峰值提供了一个可重复的复现。

该测试工具有一个轻量级脚本测试：

```bash
npx vitest run --config ./scripts/tests/vitest.config.ts \
  scripts/tests/memory-pressure-repro.test.js
```

结果：通过，1 个测试。

受控运行使用 `node --max-old-space-size=256`，除非另有说明。

失败的大型克隆运行产生了：

```text
FATAL ERROR: Reached heap limit Allocation failed - JavaScript heap out of memory
```

原生栈包含：

- `v8::internal::ValueDeserializer::ReadObjectInternal`
- `v8::internal::ValueDeserializer::ReadDenseJSArray`
- `node::worker::Message::Deserialize`
- `node::worker::StructuredClone`

这与用户提供的 OOM 日志中的栈族相同。受控复现还解释了为什么 4 GiB / 8 GiB 的用户报告是合理的：失败并非由单个大对象引起，而是由大量保留的历史/工具结果/子代理状态加上一个或多个完整历史克隆或检查点副本引起。增加 `--max-old-space-size` 可以延迟崩溃，同时保持相同的放大模式。

本次确定性运行的重要发现：

1. 在缩减堆下，构建 76.2 MiB 的父历史 JSON 可以成功。当额外的完整历史克隆/检查点副本被保留时才会出现 OOM。
2. 单个检查点副本在 OOM 之前就可能将 RSS 推至接近 1 GiB。
3. 从父热历史中移除保留的子代理转录本，可以将相同生成的工作负载从 OOM 转变为仅 136.8 MiB RSS 的运行。这是迄今为止最清晰的缓解信号。
4. 该复现器是合成的，且有意对抗性，但它执行了与长交互式审查相同的对象图形状：父会话、子代理、大型工具输出、转录本合并回写以及完整历史克隆压力。

回放随后从 JSONL 重建了聊天 `Content[]` 消息形状，并运行了受控的克隆/字符串化压力测试。基础保留消息负载很小，因此本身不足以导致 OOM：

缩放回放并非用户数据声明；它是对观察到的 JSONL 形状进行受控放大，以测试完整历史克隆和检查点序列化是否会产生与报告相同的失败模式。

使用 `--max-old-space-size=256` 的低堆复现确认了该机制：

| 案例                      | 历史 JSON | 结果                                                 |
| ------------------------- | ---------: | ---------------------------------------------------- |
| 仅构建历史                |    38.4 MB | 成功；堆 131.6 MB，RSS 378.2 MB                     |

| 构建 + 重复克隆           |    38.4 MB | 在多次保留 `structuredClone()` 副本后 OOM           |
| 检查点双历史              |    38.4 MB | 在持有历史加上克隆的客户端历史时 OOM                  |

重复克隆 OOM 栈包含 `ValueDeserializer::ReadObjectInternal`、`ValueDeserializer::ReadDenseJSArray`、`node::worker::Message::Deserialize` 和 `node::worker::StructuredClone`，与用户提供的 OOM 日志中看到的栈族相同。这证明了完整历史 `structuredClone()` 可以在没有任何 MCP 服务器参与的情况下成为直接的 OOM 触发器。

当前针对此 JSONL 类的工作假设：

1. MCP 可以在单独的基准测试中解释正常配置的启动 RSS，但它不太可能是此长会话 OOM 形状的触发器。
2. 长任务增长来自保留的聊天历史、大型工具输出、子代理历史、可观察的代理消息以及 UI/工具结果状态。
3. 直接的 OOM 触发器可能是在堆已经很高时进行的完整历史克隆或检查点式双重序列化。
4. 压缩可以缓解保留历史，但压缩本身可能首先克隆或序列化大型历史，从而产生临时峰值。

### 本地缓解验证：禁用 MCP 的 PR 审查案例

在重新运行禁用 MCP 的 PR 审查案例之前，本地应用并验证了两个针对性缓解措施：

1. `checkNextSpeaker()` 现在仅读取最后一条精选消息（`getHistoryTail(1, true)`），并只将该消息发送到 next-speaker 侧查询。next-speaker 提示仅询问上一条模型响应，因此发送完整历史是不必要的克隆和 token 压力。
2. `AgentToolInvocation` 不再在实时的 `task_execution.toolCalls` 显示中保留完整的 `responseParts` 数组。真实的响应部分仍然通过转录本/历史路径流动，但父 UI 显示现在只保留一个有限长度的文本摘要用于嵌套工具结果流式传输，而不是在长运行期间持有子代理大型工具输出的另一个完整副本。
3. `GeminiChat.sendMessageStream()` 现在通过内部精选历史视图构建模型请求内容，而不是调用公共的 `getHistory(true)`。公共的 `getHistory()` 仍然为外部调用者返回防御性 `structuredClone()`，但请求热路径不再在每次模型调用前深度克隆整个保留的聊天历史。

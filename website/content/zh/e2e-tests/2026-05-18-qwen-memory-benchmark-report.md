# Qwen Code 运行时内存基准测试报告

日期：2026-05-18

## 摘要

本报告记录了 Qwen Code 运行时行为的本地内存基准测试。它在不同模型上比较了 Qwen Code，并在存在等效模型端点时将 Qwen Code 与 Claude Code 在相同任务形态上进行了比较。

在最新的矩阵中（每个单次运行，未重复统计），关键结果保持一致：

- Qwen Code 进程树 RSS 峰值：约 `852-1062 MiB`（`0.83-1.04 GiB`）。
- Claude Code 进程树 RSS 峰值：约 `279-366 MiB`（`0.27-0.36 GiB`）。
- 在所测试的非交互式 CLI 任务基准中，Qwen Code 约为 `2.3x-3.6x` 更高。

注意：进程树 RSS 包含 MCP 子进程（Qwen 端约 350 MiB 开销）。这会放大绝对数值，但由于两个 CLI 采用相同的测量方式，相对比较仍具有参考价值。

这种差异在小型 PR 审查、代码导航和合成 diff 工作负载中均能复现。因此，这不太可能仅由一个大 PR 或某一个模型提供商解释。

本报告旨在使当前的性能调查可见：已测量了什么、已支持什么结论、仍然未知什么、下一步应添加哪些诊断方法。

## 基准测试工件

在合并本报告之前，生成了五份本地报告：

1. Qwen Code PR 审查内存运行。
2. Qwen Code 模型比较运行。
3. Qwen Code 与 Claude Code 的严格比较（使用 `pai/glm-5`）。
4. Qwen Code 与 Claude Code，两个 CLI 使用两个模型。
5. Qwen Code 与 Claude Code，五案例矩阵。

这份合并报告涵盖了所有五份报告的结论和关键指标。它没有嵌入每一个原始样本行、终端转录或临时运行器工件。这些原始工件保留在本地 `tmp/` 目录中，因为它们是实验输出而非稳定的仓库固件。

最新矩阵是最有力的证据，因为它涵盖了多种任务形态，而不仅仅是 PR 审查工作负载。

## 初步结论

当前数据足以说明在这些本地非交互式 CLI 任务基准中，Qwen Code 的运行时内存占用高于 Claude Code。但尚不足以确定最终的单个根因。

领先的解释是 Qwen Code 运行时/路径差异，而非模型提供商差异：

- 该差异在使用 `pai/glm-5` 和 `qwen3.6-plus` 时均能复现；
- 该差异在小型 PR 和代码导航任务中也能复现，而不仅限于大型 diff 任务；
- Qwen Code 在处理类似工作时，发送或占用的 token 数量多于 Claude Code；
- Qwen Code 观察到的最大组件是子 Node/Qwen worker 进程，这指向任务时进程占用、模块加载、上下文组装、实时历史、工具结果保留或子代理/保存输出路径。

因此，下一步最有用的测量不是另一次仅基于外部 RSS 的运行。下一步的测量应将 RSS 拆分为 V8 堆、原生内存、会话/历史大小、保留的工具结果大小以及子代理/进程树活动。

## 初始原因分析

基准测试尚未证明单一根因，但确实缩小了问题范围。

| 信号                                                                                         | 暗示内容                                                                                 | 未证明的内容                                                                                           |
| -------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------ |
| Qwen 在小型 PR 和代码导航案例中仍接近 `1 GiB`                                                  | 很可能涉及较高的非交互式任务时运行时成本                                                   | 无法确定占用是 V8 堆、原生内存、模块加载还是保留状态                                                    |
| Diff 大小从 100 KiB 到 5 MiB 不随 RSS 线性缩放                                                 | 原始 diff 字节本身可能不是主要驱动因素                                                     | 大型输出在实际 PR 审查流程中仍可能放大内存                                                              |
| Qwen 在每个矩阵单元中使用的 token 比 Claude 多                                                 | Qwen 可能为类似工作构建或保留了更大的提示/上下文/工具结果状态                                | Token 数量不等同于进程内存，可能是结果而非原因                                                          |
| 工具调用次数相似，且 Claude 有时使用更多轮次/工具调用但 RSS 更低                                | 更长的工具调用链本身不太可能是主要原因                                                     | 工具输出大小和保留仍需测量                                                                             |
| 较早的大型 PR 运行显示出保存输出恢复和子代理放大                                                  | 工具输出截断和保存输出路径可能是繁重工作负载的放大器                                         | 无法解释整个小型任务的执行占用                                                                         |

因此，当前最佳解释是：

1. **首先是任务时运行时成本**：Qwen Code 在非交互式 CLI 任务执行期间可能初始化或保留了比 Claude Code 更多的运行时状态。这可能包括 agent 运行时、工具注册表、提供商适配器、会话服务或不需要用于短期非交互式任务的 UI/历史结构。
2. **其次是上下文/工具结果量**：Qwen Code 在类似工作中似乎携带了更大的面向模型或面向会话的上下文。token 差距使上下文组装、工具结果规范化和历史保留成为重要怀疑对象。
3. **第三是大输出放大**：大型 PR 审查可能触发额外的保存输出和子代理路径。这些可能不是唯一原因，但在实际审查任务中会加剧内存和 token 压力。

下一个诊断运行应回答 `~1 GiB` 位于何处：

- 启动后立即很高：模块/运行时启动成本；
- 工具执行后跳升：工具输出保留或结果规范化；
- 请求组装期间跳升：上下文构建或重复历史；
- 流式/压缩后增长：响应保留或压缩状态；
- 在 V8 堆之外的大部分 RSS：原生缓冲区、加载的模块或外部内存。

## Token 使用分析

Token 差距是最强烈的线索之一，但需要内部请求指标才能将其视为根因。

当前数据支持以下内容：

- 在每个矩阵单元中，Qwen Code 使用的总 token 数多于 Claude Code。
- 即使工具调用次数相似，差距仍然存在。
- Claude 有时使用更多轮次或工具调用，但内存占用仍然更低。

这表明：

- Token 差异不太可能仅来自更长的工具调用链。
- Qwen 可能携带更大的静态提示/上下文状态、更大的工具模式、更大的序列化工具结果，或更多的保留对话/会话内容。
- 大输出流可能通过截断、保存输出恢复或子代理路径增加另一个层次。

仍然缺失的内容：

- 每次请求的输入 token 分解；
- 系统提示和工具模式的 token 大小；
- 每次模型请求前保留的消息和工具结果大小；
- 大输出是否在多个地方保留，例如模型历史、UI 历史、会话记录或保存输出存储。

这些缺失的指标是下一步应添加内部诊断而非仅重复外部 RSS 基准的原因。

## 工作假设

当前证据支持以下假设，按优先级排序：

1. Qwen Code 的非交互式任务时进程占用高于 Claude Code。Qwen 子 Node worker 在本地采样中通常是最大的进程，通常在 `0.7-0.8 GiB` 左右。
2. 模型选择不是主要解释。`pai/glm-5` 和 `qwen3.6-plus` 均显示相同的 Qwen 与 Claude 差距。
3. 大型 diff 大小本身不是主要解释。合成 diff 大小从 100 KiB 到 5 MiB 并未线性扩展，可能是因为工具输出截断限制了到达模型的输出量。
4. 上下文/工具结果处理仍可能是贡献因素。Qwen Code 在每个矩阵单元中使用的 token 多于 Claude Code，并且早期的大型 PR 运行显示了保存的工具输出恢复和子代理放大路径。
5. 下一个诊断层应将 V8 堆、原生 RSS、加载的模块/运行时启动成本、会话历史、UI 历史、工具结果保留和子代理活动分开。仅靠外部 RSS 无法区分这些原因。

## 注意事项

- 每个矩阵单元为单次运行，非重复统计样本。
- RSS 是外部进程 RSS。它无法区分 V8 堆、原生缓冲区、模块加载、保留的工具输出、UI 状态或会话历史。
- Claude Code 和 Qwen Code 使用不同的运行时实现和协议适配器，即使模型标签相同。
- 基准测试在 macOS 本地运行。在得出部署相关结论之前，应在 Linux 服务器上进行测试。

## 推荐的后续测量

下一次本地调查分支应添加或使用以下诊断：

- 在启动前后、工具执行、流式、压缩和会话结束时的 `process.memoryUsage()`。
- V8 堆统计和堆空间。
- 活动句柄和请求。
- 会话消息计数和近似保留的字符/token 量。
- 工具结果计数、保留的工具结果总大小、最大工具结果大小，以及大输出是否被 UI 历史或模型历史保留。
- 子代理计数和子进程/进程树 RSS。
- 工具输出截断和保存输出恢复事件。

这些测量应与当前基准矩阵一起收集，以便将当前的 RSS 比较与内部 Qwen Code 状态联系起来。

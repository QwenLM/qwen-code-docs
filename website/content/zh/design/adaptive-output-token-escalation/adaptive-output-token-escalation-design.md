# 自适应输出 Token 升级设计

> 通过“低默认值 + 截断时升级”策略，将 GPU 槽位过度预留减少约 4 倍，并对超出升级限制的响应提供多轮恢复机制。

## 问题

每个 API 请求都会预留一个与 `max_tokens` 成正比的固定 GPU 槽位。之前默认的 32K token 意味着每个请求预留 32K 的输出槽位，但 99% 的响应都在 5K token 以下。这导致 GPU 容量过度预留 4-6 倍，限制了服务器并发能力并增加了成本。

## 解决方案

使用上限为 **8K** 的输出 token 作为默认值。当响应被截断（模型达到 `max_tokens`）时：

1. **升级** 至模型的全输出限制（对于未知模型，以 64K 为下限）
2. 如果仍然被截断，则进行**恢复**：将部分响应保留在历史记录中，并注入一条继续消息，最多尝试 3 次
3. 如果恢复尝试耗尽，则回退到工具调度器的截断指导

由于只有 <1% 的请求实际被截断，这显著降低了平均槽位预留，同时为长响应保留了输出质量。

## Token 限制确定

有效 `max_tokens` 按以下优先级顺序解析：

| 优先级      | 来源                                                     | 值（已知模型）                   | 值（未知模型） | 升级行为                                     |
| ----------- | -------------------------------------------------------- | -------------------------------- | --------------- | --------------------------------------------- |
| 1（最高）   | 用户配置 (`samplingParams.max_tokens`)                   | `min(userValue, modelLimit)`     | `userValue`     | 不升级                                        |
| 2           | 环境变量 (`QWEN_CODE_MAX_OUTPUT_TOKENS`)                 | `min(envValue, modelLimit)`      | `envValue`      | 不升级                                        |
| 3（最低）   | 带上限的默认值                                           | `min(modelLimit, 8K)`            | `min(32K, 8K) = 8K` | 升级至模型限制（64K 下限）+ 恢复              |

“已知模型”指在 `OUTPUT_PATTERNS` 中有明确条目的模型（通过 `hasExplicitOutputLimit()` 检查）。对于已知模型，有效值始终被限制在模型声明的输出限制内，以避免 API 错误。未知模型（自定义部署、自托管端点）则直接传递用户的值，因为后端可能支持更大的限制。

该逻辑在三个内容生成器中实现：

- `DefaultOpenAICompatibleProvider.applyOutputTokenLimit()` — OpenAI 兼容的提供商
- `DashScopeProvider` — 继承默认提供商的 `applyOutputTokenLimit()`
- `AnthropicContentGenerator.buildSamplingParameters()` — Anthropic 提供商

## 升级机制

升级逻辑位于 `geminiChat.ts` 中，并有意放置在**主重试循环之外**。原因如下：

1. 重试循环处理瞬时错误（速率限制、无效流、内容验证）
2. 截断不是错误——它是一个被截断的成功响应
3. 来自升级流的错误应直接传播给调用方，不应被重试逻辑捕获

### 升级步骤 (`geminiChat.ts`)

```
1. 流成功完成 (lastError === null)
2. 最后一个块具有 finishReason === MAX_TOKENS
3. 守卫检查通过：
   - maxTokensEscalated === false（防止无限升级）
   - hasUserMaxTokensOverride === false（尊重用户意图）
4. 计算升级限制：max(ESCALATED_MAX_TOKENS, tokenLimit(model, 'output'))
5. 从聊天历史记录中弹出部分模型响应
6. 产生 RETRY 事件 (isContinuation: false) → UI 丢弃部分输出并重置缓冲区
7. 使用 maxOutputTokens: escalatedLimit 重新发送相同请求
```

### 恢复步骤 (`geminiChat.ts`)

如果升级后的响应也被截断 (finishReason === MAX_TOKENS)，恢复循环最多运行 `MAX_OUTPUT_RECOVERY_ATTEMPTS`（3）次：

```
1. 部分模型响应已在历史记录中（由 processStreamResponse 推入）
2. 推送一条恢复用户消息：OUTPUT_RECOVERY_MESSAGE
3. 产生 RETRY 事件 (isContinuation: true) → UI 保留文本缓冲区用于继续
4. 使用更新后的历史记录重新发送（模型看到其部分输出 + 恢复指令）
5. 如果仍然被截断且还有剩余尝试次数，则循环回到步骤 1
6. 如果恢复尝试抛出异常（空响应、网络错误）：
   - 从历史记录中弹出挂起的恢复消息
   - 跳出恢复循环
```

### 重试时的状态清理 (`turn.ts`)

当 `Turn` 类收到 RETRY 事件时，它会清除累积的状态以防止不一致：

- `pendingToolCalls` — 清除，以避免如果第一个截断的响应包含已完成的工具调用，而这些调用在升级后的响应中被重复时，导致重复的工具调用
- `pendingCitations` — 清除，以避免重复的引用
- `finishReason` — 重置为 `undefined`，以便使用新响应的结束原因

`isContinuation` 标志会传递到 UI，以便 UI 决定是重置文本缓冲区（升级）还是保留它们（恢复）。

## 常量

定义在 `geminiChat.ts` 和 `tokenLimits.ts` 中：

| 常量名称                       | 值     | 用途                                                   |
| ------------------------------ | ------ | ------------------------------------------------------- |
| `CAPPED_DEFAULT_MAX_TOKENS`    | 8,000  | 无用户覆盖时的默认输出 token 限制                       |
| `ESCALATED_MAX_TOKENS`         | 64,000 | 升级的下限（模型限制未知时使用）                         |
| `MAX_OUTPUT_RECOVERY_ATTEMPTS` | 3      | 升级后最多多轮恢复尝试次数                              |

有效升级限制为 `max(ESCALATED_MAX_TOKENS, tokenLimit(model, 'output'))`：

| 模型             | 升级限制         |
| ---------------- | ---------------- |
| Claude Opus 4.6  | 131,072 (128K)   |
| GPT-5 / o 系列   | 131,072 (128K)   |
| Qwen3.x          | 65,536 (64K)     |
| 未知模型         | 64,000 (下限)    |

## 设计决策

### 为什么默认值是 8K？

- 99% 的响应都在 5K token 以下
- 8K 为稍长的响应提供了合理余量，不会触发不必要的重试
- 将平均槽位预留从 32K 降低到 8K（提升 4 倍）

### 为什么升级到模型限制而不是固定的 64K？

- 具有更高输出限制的模型（Claude Opus 128K、GPT-5 128K）之前被不必要地限制在了 64K
- 使用模型的实际限制可以捕获绝大多数长输出，而无需第二次重试
- `ESCALATED_MAX_TOKENS` (64K) 作为未知模型的下限，对于这些模型，`tokenLimit()` 返回默认的 32K

### 为什么使用多轮恢复而不是逐步升级？

- 逐步升级（8K → 16K → 32K → 64K）每次都需要重新生成完整响应
- 多轮恢复保留部分响应，让模型继续生成，节省 token 和延迟
- 恢复消息成本很低（每条约 40 token），相比之下重新生成大响应的成本更高
- 3 次限制防止无限循环，同时覆盖了大多数实际场景

### 为什么升级逻辑放在重试循环之外？

- 截断是成功情况，而不是错误
- 来自升级流的错误（速率限制、网络故障）应直接传播，而不是被静默重试并带上错误参数
- 保持重试循环专注于其原始目的（瞬时错误恢复）
- 恢复错误被单独捕获，以避免中止整个对话

# Qwen Code 运行时内存调研计划

日期：2026-05-18

## 背景

本地基准测试显示，对于类似的非交互式 CLI 任务场景，Qwen Code 使用的进程树 RSS 明显高于 Claude Code。最近五组测试矩阵发现，Qwen Code 峰值在 `0.83-1.04 GiB` 左右，而 Claude Code 停留在 `0.27-0.36 GiB` 左右。

本文档提出一个初步的调研和优化方向。目前无意断言根本原因。近期目标是让内存差距变得可审查、可复现，并能通过内部诊断来解释。

## 已有进展

调研已进入证据收集和方向判断阶段：

- 针对小型 PR 审查、代码导航和合成 diff 工作负载，已构建可复现的本地测试矩阵。
- 已在多个模型上对比了 Qwen Code。
- 在可以使用等效模型端点的相同任务场景下，已对比了 Qwen Code 和 Claude Code。
- 观察到的 RSS 差距足够一致，值得进行更深入的运行时诊断。
- 已映射相关上游工作，以便本调研能建立在现有的 `/doctor memory` 和内存诊断后续工作之上。

调研尚未进入根本原因确认阶段，因为外部进程 RSS 无法显示保留的内存是 V8 堆、原生内存、已加载模块、实时历史记录、工具结果还是请求组装状态。

## 当前证据

配套的基准测试报告位于：

- `docs/e2e-tests/2026-05-18-qwen-memory-benchmark-report.md`

主要证据包括：

- Qwen 与 Claude 的 RSS 差距在小型 PR 审查、代码导航和合成 diff 工作负载上均成功复现。
- 该差距在使用 `pai/glm-5` 和 `qwen3.6-plus` 时均能复现。
- 在测试矩阵的每一个单元格中，Qwen Code 使用的 token 数均高于 Claude Code。
- 较大的 diff 大小并未产生清晰的线性内存增长，这表明基线路径和有界/截断输出路径的影响比原始 diff 字节本身更大。

## 为什么这份初稿从文档开始

这份初稿有意从基准测试证据和调研计划开始，而不是直接附上一个运行时代码变更。

原因：

1.  当前目标是让性能问题和方向变得可见，而不是声称当天就能修复。
2.  在同一 PR 中同时添加工具化和优化会使审查变得更困难，因为它混合了测量、诊断和行为变更。
3.  已有的基准测试已经支持需要进行更深入的诊断。
4.  下一个 PR 可以更窄、更容易验证：仅诊断，然后重新运行相同的矩阵并比较内部指标。

下一个实现 PR 应添加缺失的计数器和时间线点，然后重新运行基准测试矩阵。只有在之后，一个有针对性的优化 PR 才应尝试减少内存。

## 工作推论

当前数据更指向 Qwen Code 运行时/路径问题，而非模型提供商问题。

目前最有力的推论是：

> Qwen Code 似乎在非交互式 CLI 任务执行中承载了较高的开销，可能因更大的上下文/工具结果/会话处理而放大。可能的问题区域是 CLI 运行时和 Agent 数据路径，而非仅所选模型本身。

更具体地说，证据指向“工具调用过多”并非主因。各 CLI 的工具调用次数相似，Claude 有时使用了更多轮次或工具调用，但保持了较低的 RSS。更合理的问题是，Qwen Code 为同样的短时非交互式 CLI 任务初始化或保留了更重的状态，然后通过更大的上下文、工具结果、保存输出或会话历史数据放大了该执行开销。

最可能的分类包括：

1.  **进程和模块启动/执行开销**：Qwen Code 可能为非交互式 CLI 任务初始化了比必要更多的运行时、工具、UI/会话基础设施或提供商机制。
2.  **历史和上下文组装**：对于相同的任务场景，Qwen Code 可能保留或构建了比 Claude Code 更大的模型上下文。
3.  **工具结果保留**：大型或重复的工具结果可能保留在实时历史、UI 历史、聊天记录或保存输出恢复路径中。
4.  **子代理和保存输出放大**：之前的大型 PR 测试显示了保存输出恢复和子代理活动，这可能会增加内存和 token 压力。
5.  **MCP 子进程**：配套的诊断报告显示，MCP 服务器（如 chrome-devtools）贡献了约 350 MiB 的进程树 RSS。这会抬高绝对值，但与会话长度无关的固定开销。
6.  **原生内存与 JS 堆的分布**：外部 RSS 无法区分压力是来自 V8 堆、原生缓冲区、已加载模块还是保留的数据。

这有意表述为一种推论。下一步是添加足够的内部测量来确认或排除每个分类。

## 拟议的初版 PR 范围

第一个 PR 应聚焦于证据和诊断：

1.  提交基准测试报告和调研计划。
2.  增加或扩展现有本地诊断输出，使 Qwen Code 能够报告：
    - V8 堆和堆空间统计信息。
    - RSS 与堆的拆分。
    - 会话消息数量及近似保留大小。
    - 工具结果数量、总保留大小和最大保留结果大小。
    - 截断和保存输出恢复计数器。
    - 子代理/进程树活动（如果可用）。
3.  针对以下状态重新运行现有矩阵：
    - 当前发布的 Qwen Code，
    - 当前 `main` 分支，
    - 仅诊断分支，
    - 候选优化分支。
4.  使用这些测量结果选择一个小的优化目标。

第一个 PR 应避免混合多个不相关的优化。它应该要么仅限文档，要么仅添加诊断代码。一个单独的 PR 应在原因更明确后，实施第一次运行时内存减少。

## 候选优化方向

这些是候选方向，而非结论：

1.  **有界工具输出保留**：将大型输出存储在热路径之外，仅在实时历史中保留预览、元数据和检索指针。
2.  **非交互式惰性加载**：在非交互式 CLI 任务执行期间，避免初始化仅 TUI 或仅交互式的子系统。
3.  **会话/UI 历史上限**：将旧或大的历史条目降级为紧凑的转录条目。
4.  **上下文组装核算**：在构建模型请求之前，测量并限制大型工具结果。
5.  **子代理核算**：在诊断中暴露子代理生命周期和内存影响。

Claude Code 和 OpenAI Codex（OpenAI 的 CLI 编码 agent）应作为诊断分离、有界输出保留和惰性历史加载的设计参考。实现仍应遵循 Qwen Code 自身的架构和测试。

## 验证计划

调研应保持相同的基准测试矩阵，以便前后结果可比较：

- 小型 PR 审查
- 代码导航
- 约 100 KiB 的合成 diff
- 约 1 MiB 的合成 diff
- 约 5 MiB 的合成 diff

每次运行记录：

- 进程树 RSS 峰值
- 根进程 RSS 峰值
- V8 堆峰值
- 堆空间摘要
- 持续时间
- 轮次
- token 数量
- 工具调用次数
- 最大保留工具结果
- 工具结果总保留大小
- 会话/历史条目数量
- 子代理数量

候选修复的最低成功条件不仅是“RSS 降低了”。它还应该识别出哪个内部指标发生了变化以及原因。

## 下一个 PR 候选

下一个 PR 应仅限诊断，避免改变运行时行为。一个最小有用的切片应增加：

- 模型请求输入大小核算；
- 系统提示和工具 schema 大小核算；
- 保留消息数量和近似保留字符大小；
- 保留工具结果数量、总大小和最大条目大小；
- 在启动、第一次请求组装、工具执行、流式完成、压缩和最终响应周围的生命周期采样；
- 包含 RSS、堆已使用、堆总计、外部和堆空间统计的进程内存采样。

在此落地到本地后，重新运行相同的 Qwen 模型矩阵，比较：

- 发布的 Qwen Code；
- 当前 `main` 分支；
- 仅诊断分支；
- 候选优化分支。

## 非目标

本初稿并不声称：

- 所有内存压力都由工具输出引起；
- 一个现有的开放 PR 就能解决观察到的任务时占用；
- 模型提供商的差异在所有环境中都无关紧要；
- 单次运行的本地测量足以支持发布级性能声明。

本初稿的目标更窄：Qwen Code 在测试的工作负载中显示出一致的本地 RSS 差距，项目需要内部诊断来解释并缩小这一差距。

# 内存诊断参考设计

## 背景

Issue #3000 跟踪了长运行 Qwen Code 会话的内存和性能诊断。第一个 PR 应在添加更重的性能分析或保留策略变更之前，建立一个较小、风险较低的诊断面。

该设计首先以参考为出发点：

- Claude Code 将内存诊断与堆快照生成分离开来。其诊断包括进程内存、V8 堆统计信息、堆空间、资源使用情况、活跃句柄/请求、文件描述符、Linux `smaps_rollup` 以及泄漏提示。
- Codex 高度关注长生命周期进程状态的有界保留和延迟加载。这些思路应指导后续处理对话、命令输出和历史保留的 PR。

## 第一个 PR 范围

添加一个 `/doctor memory` 诊断路径，用于捕获单个时间点的快照：

- `process.memoryUsage()`
- V8 堆统计信息和堆空间
- `process.resourceUsage()`
- 活跃句柄/请求计数
- 当 `/proc/self/fd` 可用时的打开文件描述符计数
- 当可用时的 Linux `smaps_rollup`
- 针对堆压力、分离的上下文、过多句柄、过多请求、高文件描述符计数以及原生内存压力的基本风险提示

该命令应足够轻量，可在正常会话中运行，并且在 Linux 专有探测不可用的平台上保持安全。

## 非目标

此 PR 明确不执行以下操作：

- 写入堆快照
- 运行持续轮询
- 更改提示/历史保留
- 更改工具输出保留
- 更改模块加载行为

这些是诊断基线建立后的后续 PR。

## 后续 PR

1. 添加显式的快照/导出支持，用于更深入的本地调查。
2. 为大型命令/工具输出添加有界保留，以 Codex 的截断输出保留作为主要参考。
3. 在测量识别出热点后，审查延迟加载和模块启动路径。
4. 为长运行会话添加可重复的内存/性能基准测试场景。

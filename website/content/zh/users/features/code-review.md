# 代码审查

> 使用 `/review` 审查代码变更的正确性、安全性、性能和代码质量。

## 快速开始

```bash
# 审查本地未提交的更改
/review

# 审查 Pull Request（通过编号或 URL）
/review 123
/review https://github.com/org/repo/pull/123

# 审查并在 PR 上发布行内评论
/review 123 --comment

# 审查特定文件
/review src/utils/auth.ts
```

如果没有未提交的更改，`/review` 会提示并停止 —— 不会启动任何 agent。

## 工作原理

`/review` 命令运行一个多阶段流水线：

```
Step 1:  确定范围（本地 diff / PR 工作树 / 文件）
Step 2:  加载项目审查规则
Step 3:  运行确定性分析（linter、类型检查）    [零 LLM 开销]
Step 4:  9 个并行的审查 agent                    [9 次 LLM 调用]
           |-- Agent 1: 正确性
           |-- Agent 2: 安全性
           |-- Agent 3: 代码质量
           |-- Agent 4: 性能与效率
           |-- Agent 5: 测试覆盖率
           |-- Agent 6: 无导向审计（3 种角色：6a/6b/6c）
           '-- Agent 7: 构建与测试（运行 shell 命令）
Step 5:  去重 --> 批量验证 --> 聚合                [1 次 LLM 调用]
Step 6:  迭代逆向审计（1-3 轮，查找遗漏）         [1-3 次 LLM 调用]
Step 7:  呈现发现 + 结论
Step 8:  自动修复（用户确认，可选）
Step 9:  发布 PR 行内评论（如果请求）
Step 10: 保存报告 + 增量缓存
Step 11: 清理（删除工作树和临时文件）
```

### 审查 Agent

| Agent                             | 关注点                                                                                       |
| --------------------------------- | -------------------------------------------------------------------------------------------- |
| Agent 1: 正确性                   | 逻辑错误、边界情况、空值处理、竞态条件、类型安全                                              |
| Agent 2: 安全性                   | 注入、XSS、SSRF、认证绕过、敏感数据暴露                                                        |
| Agent 3: 代码质量                 | 风格一致性、命名、重复、死代码                                                               |
| Agent 4: 性能与效率               | N+1 查询、内存泄漏、不必要的重新渲染、打包体积                                                   |
| Agent 5: 测试覆盖率               | diff 中未测试的代码路径、缺失的分支覆盖、弱的断言                                              |
| Agent 6: 无导向审计               | 3 个并行角色（攻击者/凌晨三点值班/维护者）—— 捕获跨维度问题                                    |
| Agent 7: 构建与测试               | 运行构建和测试命令，报告失败                                                                  |

所有 agent 并行运行（Agent 6 同时启动 3 个角色变体，同一仓库审查时总共 9 个并行任务）。Agent 1-6 的发现结果在一个 **单次批量验证 pass** 中一起验证（一个 agent 一次性审查所有发现，无论发现数量多少，验证开销保持不变）。验证完成后，**迭代逆向审计** 运行 1-3 轮查找遗漏 —— 每一轮都会收到之前各轮的累积发现列表，因此后续轮次专注于尚未发现的问题。只要某一轮返回“未发现任何问题”，或者达到 3 轮（硬上限），循环停止。逆向审计的发现跳过验证（agent 已拥有完整上下文），作为高置信度结果包含在内。

## 确定性分析

在 LLM agent 运行之前，`/review` 会自动运行项目现有的 linter 和类型检查器：

| 语言                | 检测到的工具                                                     |
| ------------------- | ---------------------------------------------------------------- |
| TypeScript/JavaScript | `tsc --noEmit`、`npm run lint`、`eslint`                         |
| Python              | `ruff`、`mypy`、`flake8`                                         |
| Rust                | `cargo clippy`                                                    |
| Go                  | `go vet`、`golangci-lint`                                        |
| Java                | `mvn compile`、`checkstyle`、`spotbugs`、`pmd`                    |
| C/C++               | `clang-tidy`（如果 `compile_commands.json` 可用）                  |
| 其他                | 从 CI 配置自动发现（`.github/workflows/*.yml` 等）                |

对于不符合标准模式的项目（例如 OpenJDK），`/review` 会读取 CI 配置文件来发现项目使用的 lint/检查命令。无需用户配置。

确定性分析结果带有 `[linter]` 或 `[typecheck]` 标签，并跳过 LLM 验证 —— 它们是事实依据。

- **错误** → 严重（Critical）
- **警告** → 可有可无（仅在终端显示，不发布为 PR 评论）

如果某个工具未安装或超时，则跳过并显示一条提示信息。

## 严重级别

| 严重级别         | 含义                                                             | 是否发布为 PR 评论？           |
| ---------------- | ----------------------------------------------------------------- | ------------------------------ |
| **Critical**     | 合并前必须修复（错误、安全问题、数据丢失、构建失败）               | 是（仅高置信度）                |
| **Suggestion**   | 推荐的改进                                                       | 是（仅高置信度）                |
| **Nice to have** | 可选的优化                                                       | 否（仅终端显示）                |

低置信度的发现会单独出现在终端的“需要人工审查”部分，永远不会作为 PR 评论发布。

## 自动修复

呈现发现结果后，`/review` 会提供自动修复选项，适用于有明确解决方案的 Critical 和 Suggestion 级别问题：

```
Found 3 issues with auto-fixable suggestions. Apply auto-fixes? (y/n)
```

- 修复使用 `edit` 工具应用（定向替换，不是全文件重写）
- 修复后会针对每个文件运行 linter 检查，以确保不引入新问题
- 对于 PR 审查，修复会自动在工作树中提交并推送 —— 你的工作目录保持干净
- Nice to have 和低置信度的发现永远不会自动修复
- PR 审查提交始终使用 **修复前的结论**（例如“请求更改”），因为远程 PR 在自动修复推送完成之前尚未更新

## 工作树隔离

审查 PR 时，`/review` 会创建一个临时 git 工作树（`.qwen/tmp/review-pr-<number>`），而不是切换你当前的分支。这意味着：

- 你的工作目录、暂存更改和当前分支 **永远不会被触及**
- 在工作树中安装依赖（`npm ci` 等），以便 lint、构建/测试正常工作
- 构建和测试命令在隔离环境中运行，不会污染你的本地构建缓存
- 如果出现问题，你的环境不受影响 —— 只需删除工作树即可
- 审查完成后工作树会自动清理
- 如果审查被中断（Ctrl+C、崩溃），下一次对同一 PR 运行 `/review` 会在开始前自动清理过时的工作树
- 审查报告和缓存保存在主项目目录中（而不是工作树）

## 跨仓库 PR 审查

你可以通过传递完整 URL 来审查其他仓库的 PR：

```bash
/review https://github.com/other-org/other-repo/pull/456
```

这以 **轻量模式** 运行 —— 无工作树、无 linter、无构建/测试、无自动修复。审查仅基于 diff 文本（通过 GitHub API 获取）。如果你有写入权限，仍可发布 PR 评论。

| 能力                                                     | 同一仓库  | 跨仓库                          |
| -------------------------------------------------------- | --------- | ------------------------------- |
| LLM 审查（Agent 1-6 + 验证 + 迭代逆向审计）              | ✅        | ✅                              |
| Agent 7：构建与测试                                      | ✅        | ❌（无本地代码库）              |
| 确定性分析（linter/类型检查）                            | ✅        | ❌                              |
| 跨文件影响分析                                           | ✅        | ❌                              |
| 自动修复                                                 | ✅        | ❌                              |
| PR 行内评论                                              | ✅        | ✅（如果你有写入权限）          |
| 增量审查缓存                                             | ✅        | ❌                              |

## PR 行内评论

使用 `--comment` 直接将发现结果发布到 PR 上：

```bash
/review 123 --comment
```

或者在运行 `/review 123` 后，输入 `post comments` 发布结果，而无需重新运行审查。

**发布的内容：**

- 高置信度的 Critical 和 Suggestion 发现，作为特定代码行的行内评论
- 对于“批准/请求更改”结论：附带结论的审查摘要
- 对于“评论”结论且所有行内评论均已发布：不附带单独摘要（行内评论已足够）
- 每条评论底部的模型归属信息（例如 _— qwen3-coder 通过 Qwen Code /review_）

**仅在终端显示的内容：**

- Nice to have 发现（包括 linter 警告）
- 低置信度发现

**自创 PR：** GitHub 不允许你对自己的 Pull Request 提交 `APPROVE` 或 `REQUEST_CHANGES` 审查 —— 两者都会返回 HTTP 422。当 `/review` 检测到 PR 作者与当前认证用户一致时，它会自动将 API 事件降级为 `COMMENT`，无论结论如何，从而确保提交成功。终端仍然显示真实的结论（“批准”/“请求更改”/“评论”）—— 只有 GitHub 侧的审查事件被中性化处理。实际发现结果仍会作为行内评论出现在特定代码行上，因此实质性的反馈保持不变。

**重新审查已有 Qwen Code 评论的 PR：** 当 `/review` 在已有先前 Qwen Code 审查评论的 PR 上运行时，它会在发布新评论之前对现有评论进行分类。只有当 **同行重叠**（在相同 `(path, line)` 上已有现有评论与新的发现相同时），才会提示你确认 —— 这是你会在同一代码行上看到视觉重复的情况。来自较旧提交的评论、已回复的评论（视为已解决）、以及完全不与任何新发现重叠的评论会被静默跳过，并在终端中留下日志行，以便你知道过滤了什么。

**在批准前检查 CI/构建状态：** 如果结论是“批准”，`/review` 会在提交前查询 PR 的检查运行和提交状态。如果有任何检查失败（或所有检查仍在等待中），API 事件会自动从 `APPROVE` 降级为 `COMMENT`，并在审查正文中解释原因。理由：LLM 审查静态读取代码，无法看到运行时测试失败；在 CI 红色时批准会产生误导。行内发现结果仍会以原样发布。如果你仍然想批准（例如已知的不稳定 CI 失败），请在验证后手动提交 GitHub 批准。

## 后续操作

审查后，上下文相关的提示会以幽灵文本形式出现。按 Tab 接受：

| 审查后的状态                   | 提示                | 发生什么                                    |
| ------------------------------ | ------------------- | ------------------------------------------- |
| 本地审查，有未修复的发现       | `fix these issues`  | LLM 交互式地修复每个发现                    |
| PR 审查，有发现                | `post comments`     | 发布 PR 行内评论（不重新审查）              |
| PR 审查，零发现                | `post comments`     | 在 GitHub 上批准该 PR（LGTM）               |
| 本地审查，一切正常             | `commit`            | 提交你的更改                                |

注意：`fix these issues` 仅适用于本地审查。对于 PR 审查，请使用自动修复（步骤 8）—— 工作树在审查后会被清理，因此不能在审查后进行交互式修复。

## 项目审查规则

你可以为每个项目自定义审查标准。`/review` 从以下文件中读取规则（按顺序）：

1. `.qwen/review-rules.md`（Qwen Code 原生）
2. `.github/copilot-instructions.md`（首选）或 `copilot-instructions.md`（后备 —— 只加载一个，不加载两个）
3. `AGENTS.md` — `## Code Review` 部分
4. `QWEN.md` — `## Code Review` 部分

规则会被注入到 LLM 审查 agent（1-6）中作为额外标准。对于 PR 审查，规则从 **基础分支** 读取，以防止恶意 PR 注入绕过规则。

示例 `.qwen/review-rules.md`：

```markdown
# Review Rules

- All API endpoints must validate authentication
- Database queries must use parameterized statements
- React components must not use inline styles
- Error messages must not expose internal paths
```

## 增量审查

当审查一个之前已审查过的 PR 时，`/review` 只检查自上次审查以来的更改：

```bash
# 首次审查 —— 完整审查，创建缓存
/review 123

# PR 更新了新的提交 —— 只审查新的更改
/review 123
```

### 跨模型审查

如果你切换模型（通过 `/model`）并重新审查同一 PR，`/review` 会检测到模型更改并进行完整审查，而不是跳过：

```bash
# 使用模型 A 审查
/review 123

# 切换模型
/model

# 再次审查 —— 使用模型 B 进行完整审查（不跳过）
/review 123
# → "Previous review used qwen3-coder. Running full review with gpt-4o for a second opinion."
```

缓存存储在 `.qwen/review-cache/` 中，并记录提交 SHA 和模型 ID。请确保此目录在你的 `.gitignore` 中（像 `.qwen/*` 这样的更宽泛规则也可以）。如果缓存的提交被变基（rebase）移除了，则回退为完整审查。

## 审查报告

对于同一仓库的审查，结果会保存为项目 `.qwen/reviews/` 目录下的 Markdown 文件（跨仓库的轻量审查不持久化报告）：

```
.qwen/reviews/2026-04-06-143022-pr-123.md
.qwen/reviews/2026-04-06-150510-local.md
```

报告包括：时间戳、diff 统计、确定性分析结果、所有带有验证状态的发现以及结论。

## 跨文件影响分析

当代码变更修改了导出的函数、类或接口时，审查 agent 会自动搜索所有调用者并检查兼容性：

- 参数数量/类型变更
- 返回类型变更
- 移除或重命名的公共方法
- 破坏性的 API 变更

对于大的 diff（超过 10 个修改的符号），分析会优先处理有签名更改的函数。

## Token 效率

无论产生多少发现，审查流水线使用固定数量的 LLM 调用：

| 阶段                            | LLM 调用次数    | 备注                                                 |
| -------------------------------- | ----------------- | ----------------------------------------------------- |
| 确定性分析（步骤 3）             | 0                 | 仅 shell 命令                                        |
| 审查 agent（步骤 4）            | 9（或 8）         | 并行运行；跨仓库模式下跳过 Agent 7                   |
| 批量验证（步骤 5）              | 1                 | 单个 agent 一次性验证所有发现                         |
| 迭代逆向审计（步骤 6）          | 1-3               | 循环直到“未发现任何问题”或达到 3 轮上限               |
| **总计**                        | **11-13（10-12）** | 同一仓库：11-13；跨仓库：10-12（无 Agent 7）          |

大多数 PR 收敛到范围的下限（1 轮逆向审计）；上限防止在极端情况下失控的开销。

## 不会标记的内容

审查特意排除以下内容：

- 未更改代码中的预先存在的问题（仅关注 diff）
- 与你代码库约定一致的样式/格式化/命名问题
- linter 或类型检查器能捕获的问题（由确定性分析处理）
- 没有实际问题的“考虑做 X”的主观建议
- 不修复 bug 或风险的轻微重构
- 缺失文档，除非逻辑确实令人困惑
- 现有 PR 评论中已经讨论的问题（避免重复人工反馈）

## 设计理念

> **沉默胜于噪音。** 每条评论都应当值得读者花费时间。

- 如果不确定某件事是否是问题 —— 不报告
- Linter/类型检查问题由工具处理，而非 LLM 猜测
- N 个文件中的相同模式 —— 聚合为一条发现
- PR 评论仅限高置信度
- 与代码库约定一致的风格/格式化问题被排除

# Настройка дизайна баннера

> Позволяет пользователям заменять ASCII-арт QWEN, заменять фирменный заголовок и
> скрывать баннер полностью — без возможности подавления операционных
> данных (версия, аутентификация, модель, рабочий каталог), которые делают Qwen Code
> отлаживаемым и заслуживающим доверия.

## Обзор

CLI Qwen Code выводит баннер при запуске, содержащий ASCII-логотип QWEN
и информационную панель с рамкой. Несколько реальных сценариев использования требуют некоторого контроля
над этой поверхностью:

- **Белый ярлык / интеграция стороннего бренда**: предприятия и команды,
  встраивающие Qwen Code в свои продукты, хотят отображать свой фирменный
  стиль, а не стандартный "Qwen Code".
- **Персонализация**: отдельные пользователи хотят привести баннер терминала в соответствие с
  командным стандартом или собственным вкусом.
- **Различие между несколькими арендаторами / экземплярами**: в общих
  средах разные команды хотят быстрый визуальный сигнал о том, в каком экземпляре они
  находятся.

Принцип проектирования прост: **фирменное оформление заменяемо; операционные
данные — нет**. Настройка должна позволять пользователям размещать свой собственный бренд
сверху, а не заставлять их замолкать информацию, которая делает сеанс
отлаживаемым. Этот принцип определяет все решения "что можно изменить, а что заблокировано"
в остальной части этого документа.

Отслеживается в [issue #3005](https://github.com/QwenLM/qwen-code/issues/3005).

## Таксономия областей баннера

Сегодня баннер отображается компонентом `Header` (монтируется из `AppHeader`) и
разбивается на следующие области:

```
  marginX=2                                                           marginX=2
  │                                                                          │
  ▼                                                                          ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│                                                                             │
│   ┌──── Колонка логотипа ───┐  gap=2  ┌──── Информационная панель (с рамкой) ─┐
│   │                      │         │                                     │  │
│   │  ███ QWEN ASCII ███  │         │  ① Заголовок:    >_ Qwen Code (vX.Y.Z)  │
│   │  ███   АРТ АРТ  ███  │         │  ② Подзаголовок: «пусто, или переопределение»   │
│   │  ███ QWEN ASCII ███  │         │  ③ Статус:   Qwen OAuth | qwen-…    │
│   │                      │         │  ④ Путь:     ~/projects/example     │
│   └──────── A ───────────┘         └──────────────── B ──────────────────┘  │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
                              область: AppHeader
                          │ Компонент Tips отображается ниже (управляется ui.hideTips) │
```

Два верхних блока:

- **A. Колонка логотипа** — один блок ASCII-арта с градиентом. Источник
  сегодня — `shortAsciiLogo` в
  `packages/cli/src/ui/components/AsciiArt.ts`.
- **B. Информационная панель** — блок с рамкой, содержащий четыре строки. Вторая
  строка по умолчанию является пустым визуальным разделителем, опционально заменяется
  на предоставленный вызывающим кодом подзаголовок:
  - **B①** Заголовок: `>_ Qwen Code (vX.Y.Z)` — текст бренда + суффикс версии.
  - **B②** Подзаголовок / разделитель: по умолчанию пустая строка с одним пробелом. Когда
    установлено `ui.customBannerSubtitle`, эта строка принимает эту строку (например,
    форк может использовать `Built-in DataWorks Official Skills`).
  - **B③** Статус: `<тип отображения аутентификации> | <модель> ( /model для изменения)`.
  - **B④** Путь: сокращённый, с тильдой рабочий каталог.

Всё это обёрнуто в `<AppHeader>`, который уже управляет баннером
через `showBanner = !config.getScreenReader()` (режим чтения с экрана
возвращается к простому выводу).

## Правила настройки — что можно изменить, что заблокировано

| Область                                      | Текущий источник                   | Категория настройки              | Обоснование                                                                                                                                                                                                    |
| ------------------------------------------- | ----------------------------------- | ------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **A. Колонка логотипа**                     | `shortAsciiLogo` (`AsciiArt.ts`)    | **Заменяемый + автоскрываемый** | Чисто фирменная поверхность. Белый ярлык требует полного контроля над визуалом. Существующий резервный вариант "автоскрытие на узких терминалах" сохраняется.                                                      |
| **B①. Заголовок — текст бренда** (`>_ Qwen Code`) | Жёстко задан в `Header.tsx`          | **Заменяемый**                 | Фирменная поверхность. Ведущий символ `>_` является частью существующего бренда; если пользователь хочет его убрать, он просто опускает его в `customBannerTitle`.                                                                  |
| **B①. Заголовок — суффикс версии** (`(vX.Y.Z)`) | свойство `version`                      | **Заблокирован**                | Критически важен для сообщений об ошибках. Скрытие делает ответ на вопрос "какая у вас версия?" доступным только через `--version`, что является реальной затратой в рабочих процессах поддержки. Мы жертвуем небольшими потерями для белого ярлыка ради удобства поддержки. |
| **B②. Строка подзаголовка / разделителя**      | по умолчанию пуста                    | **Заменяемая**                 | Чисто фирменная / контекстная поверхность. Используется форками под белым ярлыком для маркировки сборки (например, "Built-in DataWorks Official Skills"). Очищается так же, как и заголовок; только одна строка — никаких разрывающих макет переводов строк.               |
| **B③. Строка статуса** (аутентификация + модель)          | свойства `formattedAuthType`, `model`  | **Заблокирована**               | Операционный сигнал и сигнал безопасности. Пользователи всегда должны видеть, какие учётные данные используются и какая модель будет расходовать их токены. Подавление этого — даже в сценариях белого ярлыка — опасно.                       |
| **B④. Строка пути** (рабочий каталог)       | свойство `workingDirectory`             | **Заблокирована**               | Операционная информация. "В каком каталоге я нахожусь?" — частый вопрос; баннер — это канонический ответ на него.                                                                                                          |
| **Весь баннер** (A + B)                    | монтирование `<Header>` в `AppHeader.tsx` | **Скрываемый**                    | Одно `ui.hideBanner: true` пропускает обе области — так же, как существующий экранный шлюз для чтения с экрана. `<Tips>` продолжает управляться независимо через `ui.hideTips`.                                         |
Матрица сводится к четырем настройкам, не больше:

| Настройка                    | По умолчанию | Эффект                                                                                                                            | Затрагиваемая область |
| ---------------------------- | ------------ | --------------------------------------------------------------------------------------------------------------------------------- | --------------------- |
| `ui.hideBanner`              | `false`      | Скрывает весь баннер (области A + B).                                                                                             | A + B                 |
| `ui.customBannerTitle`       | не задано    | Заменяет текст бренда в B①. Суффикс версии всё ещё добавляется. Обрезается; пустая строка означает «использовать по умолчанию».  | Текст бренда B①       |
| `ui.customBannerSubtitle`    | не задано    | Заменяет пустую строку-разделитель B② на однострочный подзаголовок. Очищается; не более 160 символов; пустое значение означает «оставить пустой разделитель». | Разделитель B②         |
| `ui.customAsciiArt`          | не задано    | Заменяет область A. Три допустимые формы (см. ниже). При любой ошибке возвращается к значению по умолчанию.                      | A                     |

Что **не** предлагается по дизайну:

- Нет настройки, скрывающей только суффикс версии.
- Нет настройки, скрывающей только строку аутентификации/модели.
- Нет настройки, скрывающей только строку пути.
- Нет настройки, изменяющей цвета градиента логотипа (этим управляет тема).
- Нет настройки, изменяющей порядок или структуру информационной панели.

Если в дальнейшем реализации потребуется раскрыть любой из них, это должны быть новые поля с собственным обоснованием — не производные от трёх полей выше.

## Руководство пользователя по настройке — как изменить

### Ограничения с первого взгляда

На каждую настройку баннера накладывается ряд ограничений. Помните о них, прежде чем создавать изображение вручную, чтобы резолвер не обрезал и не отверг ваш ввод.

| Что                                | Ограничение                                                                                                                                                                                |
| ---------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Количество символов в заголовке** | **Максимум 80 символов** (после очистки). Более длинные обрезаются, и в лог выводится предупреждение `[BANNER]`. Перед подсчётом этой длины удаляются новые строки и управляющие символы. |
| **Количество символов в подзаголовке** | **Максимум 160 символов** (после очистки). Тот же конвейер очистки, что и для заголовка; то же предупреждение `[BANNER]` при обрезке.                                                    |
| **Размер блока ASCII-арта**        | **Максимум 200 строк × 200 столбцов** на уровень. Более крупные блоки обрезаются до соответствия, и выводится предупреждение `[BANNER]`.                                                  |
| **Размер файла ASCII-арта на диске** | **Максимум 64 КБ**. Файлы большего размера читаются до этого предела; остальная часть игнорируется.                                                                                      |
| **Ширина ASCII-арта при рендеринге** | Определяется количеством столбцов терминала при запуске, **не является фиксированным числом символов**. См. «Какой ширины может быть логотип?» ниже для формулы и значений для каждого терминала. |

**Нет фиксированного ограничения на количество символов ASCII-арта** — только ограничения по столбцам/строкам, указанные выше, и бюджет ширины при каждом запуске. Название бренда из 17 символов, которое хорошо отображается в одном шрифте, может потребовать наложения или более плотного шрифта в другом; ограничивающим фактором является визуальная ширина, а не количество букв.

### Где находятся настройки

Все четыре настройки находятся в разделе `ui` файла `settings.json`. Поддерживаются как настройки уровня пользователя (`~/.qwen/settings.json`), так и уровня рабочей области (`.qwen/settings.json` в корне проекта) со стандартным приоритетом слияния (рабочая область переопределяет пользователя, система переопределяет рабочую область).

`customAsciiArt` обрабатывается особым образом: вместо того чтобы рассматривать весь объект как одно значение, которое заменяется областью с более высоким приоритетом, резолвер проходит по областям для каждого уровня. Если в настройках пользователя определён `{ small }`, а в настройках рабочей области — `{ large }`, оба вносят вклад — `small` от пользователя, `large` от рабочей области. Это обеспечивает работу двух вещей одновременно:

1. Каждая запись `{ path }` разрешается относительно файла, в котором она объявлена (рабочая область `.qwen/` против пользовательской `~/.qwen/`); одно только объединённое представление потеряло бы эту информацию об области.
2. Пользователи могут оставить уровень `large` по умолчанию в своих личных настройках и переопределять только `small` для каждой рабочей области, без повторения всего объекта.

Когда один и тот же уровень определён в нескольких областях, применяется обычный приоритет (система > рабочая область > пользователь). Установка `customAsciiArt` в виде простой строки или `{ path }` в любой области по-прежнему заполняет оба уровня в этой области.

### Полностью скрыть баннер

```jsonc
{
  "ui": {
    "hideBanner": true,
  },
}
```

В выводе при запуске пропускаются как столбец с логотипом, так и информационная панель. Подсказки по-прежнему отображаются, если только `ui.hideTips` не установлено в `true`.
### Замена брендового заголовка

```jsonc
{
  "ui": {
    "customBannerTitle": "Acme CLI",
  },
}
```

Отображается как `Acme CLI (vX.Y.Z)` на информационной панели. Символ `>_`
убирается, если установлен пользовательский заголовок; если вы хотите его вернуть,
включите его самостоятельно: `"customBannerTitle": ">_ Acme CLI"`.

### Добавление брендового подзаголовка

```jsonc
{
  "ui": {
    "customBannerSubtitle": "Built-in DataWorks Official Skills",
  },
}
```

Отображает подзаголовок на отдельной строке, цветом вторичного текста, вместо
пустого разделителя, который обычно находится между заголовком и строкой
авторизации/модели:

```
┌─────────────────────────────────────────────────────────┐
│ DataWorks DataAgent (vX.Y.Z)                            │  ← B① заголовок
│ Built-in DataWorks Official Skills                      │  ← B② подзаголовок
│ Qwen OAuth | qwen-coder ( /model to change)             │  ← B③ статус
│ ~/projects/example                                      │  ← B④ путь
└─────────────────────────────────────────────────────────┘
```

Ограничения:

- Только одна строка. Символы новой строки и другие управляющие байты
  удаляются / сворачиваются в пробелы, чтобы случайная вставка не нарушила
  макет информационной панели.
- Санитизация ограничена 160 символами (более мягкое ограничение, чем для
  заголовка, поскольку слоганы / строки "на базе" часто длиннее).
- Оставьте поле пустым (или установите пустую строку / пробел), чтобы
  сохранить существующий пустой разделитель — обратная совместимость по
  умолчанию.
- Подзаголовок не влияет на то, какие строки заблокированы; авторизация,
  модель и рабочая директория всегда видны независимо от состояния подзаголовка.

### Замена ASCII-арта — встроенная строка

```jsonc
{
  "ui": {
    "customAsciiArt": "  ___  _    _  ____ \n / _ \\| |  / |/ _\\\n| |_| | |__| | __/\n \\___/|____|_|___|",
  },
}
```

Используйте `\n` для вставки новых строк внутри JSON-строки. Арт
отображается с активной градиентной темой, как и стандартный логотип.

> **Нет под рукой ASCII-арта?** Используйте любой внешний генератор и
> вставьте результат. Самый простой путь — `figlet`:
> `npx figlet -f "ANSI Shadow" "xxxCode" > brand.txt`, а затем укажите
> `customAsciiArt: { "path": "./brand.txt" }`. CLI не отображает
> текст-в-арт во время выполнения — см. раздел _Вне области действия_,
> почему.

### Замена ASCII-арта — внешний файл

```jsonc
{
  "ui": {
    "customAsciiArt": { "path": "./brand.txt" },
  },
}
```

Избегает экранирования многострочной строки в JSON. Правила разрешения пути:

- **Настройки рабочей области**: относительные пути разрешаются относительно
  директории `.qwen/` рабочей области.
- **Настройки пользователя**: относительные пути разрешаются относительно `~/.qwen/`.
- Абсолютные пути используются как есть.
- Файл читается **один раз при запуске**, санитизируется и кэшируется.
  Изменение файла во время сессии не приводит к перерендеру баннера —
  перезапустите CLI.

### Замена ASCII-арта — с учётом ширины

```jsonc
{
  "ui": {
    "customAsciiArt": {
      "small": "  ACME\n  ----",
      "large": { "path": "./brand-wide.txt" },
    },
  },
}
```

`large` используется, когда терминал достаточно широк; иначе используется
`small`; иначе колонка логотипа скрывается (существующий запасной вариант
с двумя колонками). Каждый уровень может быть строкой или `{ path }`.
Каждый уровень может быть опущен: пропущенный уровень просто переходит
к следующему шагу.

### Насколько широким может быть логотип? — бюджет ширины

Нет строгого ограничения на количество символов для заголовка или арта.
Существует **бюджет ширины**, определяемый шириной терминала, и абсолютный
жёсткий лимит, чтобы некорректный файл не заморозил макет:

| Параметр                                           | Ограничение                                                         |
| -------------------------------------------------- | ------------------------------------------------------------------- |
| Ширина терминала при запуске                       | Такая, какую сообщает терминал пользователя.                        |
| Внешний отступ контейнера                          | 4 колонки (2 слева + 2 справа).                                     |
| Зазор между логотипом и информационной панелью      | 2 колонки.                                                          |
| Минимальная ширина информационной панели           | 44 колонки (40 путь + граница + отступ).                            |
| **Доступная ширина логотипа** (для каждого уровня, во время рендера) | `terminalCols − 4 − 2 − 44 = terminalCols − 50`. |
| Жёсткий лимит для каждого уровня арта (после санитизации) | 200 кол. × 200 строк. Всё, что больше, обрезается + предупреждение `[BANNER]`. |
| Жёсткий лимит для `customBannerTitle` (после санитизации) | 80 символов. Всё, что больше, обрезается + предупреждение `[BANNER]`. |

Чтение бюджета для типовой ширины терминала:

| Ширина терминала | Макс. ширина логотипа, который отображается | Что это означает на практике                                           |
| ---------------- | ------------------------------------------- | --------------------------------------------------------------------- |
| 80               | 30                                          | Большинство букв figlet "ANSI Shadow" ~7–11 кол. — макс. 3 буквы.     |
| 100              | 50                                          | Короткое слово в ANSI Shadow (~6 букв) или два коротких слова в стопку. |
| 120              | 70                                          | Многострочный арт со словами в стопку помещается комфортно.            |
| 200              | 150                                         | Длинные встроенные строки, такие как полные названия продуктов в ANSI Shadow, помещаются. |
Два практических следствия при проектировании вашей графики:

1. **Многословный бренд часто не отображается одной строкой ANSI Shadow
   на большинстве терминалов.** При ~7–9 столбцах на букву ANSI Shadow
   даже 12-символьный бренд, например `Custom Agent`, занимает примерно
   95 столбцов графики на одной строке — уже больше, чем может позволить
   терминал шириной 100 столбцов рядом с информационной панелью.
   Либо разместите слова на нескольких строках, выберите более плотный
   шрифт figlet, или используйте компактное однострочное текстовое
   оформление, например `▶ Custom Agent ◀`.
2. **Используйте форму с учётом ширины `{ small, large }`**, когда один
   уровень заставляет вас выбирать между «отлично смотрится на широком /
   умирает на узком» и «нормально смотрится на узком / тратит место
   на широком». Пример ниже размещает слова для терминала ≥104 столбцов
   в `large` и переходит к однострочному оформлению длиной 16 столбцов
   в `small`.

```jsonc
{
  "ui": {
    "customBannerTitle": "Custom Agent",
    "customAsciiArt": {
      "small": "▶ Custom Agent ◀",
      "large": { "path": "./banner-large.txt" },
    },
  },
}
```

Где `banner-large.txt` содержит вывод ANSI Shadow с разбивкой по словам
(~54 столбца × 12 строк), например, сгенерированный командой:

```bash
( npx figlet -f "ANSI Shadow" CUSTOM
  npx figlet -f "ANSI Shadow" AGENT ) > banner-large.txt
```

### Комбинирование всех трёх

```jsonc
{
  "ui": {
    "hideBanner": false,
    "customBannerTitle": "Acme CLI",
    "customAsciiArt": {
      "small": "  ACME\n  ----",
      "large": { "path": "./brand-wide.txt" },
    },
  },
}
```

### Как проверить изменения

1. Сохраните `settings.json` и запустите новый сеанс `qwen` — разрешение
   баннера выполняется однократно при запуске.
2. Измените размер терминала, чтобы убедиться, что уровни `small` / `large`
   переключаются как ожидалось, и что колонка с логотипом исчезает при
   очень узкой ширине.
3. Если что-то отображается не так, как ожидалось, посмотрите
   `~/.qwen/debug/<sessionId>.txt` (символическая ссылка `latest.txt`
   указывает на текущий сеанс) и выполните поиск `[BANNER]` — каждая
   мягкая ошибка логирует предупреждение с указанием причины.

```

   ─────────────                              ──────────────────────────────
   {                                          AppHeader.tsx
     "ui": {                                    │
       "hideBanner": false,                     │  showBanner =
       "customBannerTitle": "Acme",             │      !screenReader
       "customBannerSubtitle": "Built-in …",    │   && !ui.hideBanner
       "customAsciiArt": …                      │
     }                                          │
   }                                            ▼
        │                              <Header
        ▼                                customAsciiArt={resolved.asciiArt}
   loadSettings()                        customBannerTitle={resolved.title}
   merge user / workspace                customBannerSubtitle={resolved.subtitle}
        │                                version=… model=… authType=…
        ▼                                workingDirectory=… />
   resolveCustomBanner(settings)                  │
   ┌─────────────────────────┐                    ▼
   │ 1. нормализация до      │         packages/cli/src/ui/components/
   │    { small, large }     │         Header.tsx
   │ 2. разрешение каждого   │           │
   │    уровня:              │           │  выбор уровня по
   │    string → как есть    │           │    availableTerminalWidth
   │    {path} → fs.read     │           ▼
   │      O_NOFOLLOW         │         отрисовка колонки логотипа
   │      ≤ 64 КБ            │         отрисовка информационной панели:
   │ 3. очистка графики:     │           Заголовок = customBannerTitle
   │    удаление управляющих │                   ?? '>_ Qwen Code'
   │    последовательностей  │           Подзаголовок = customBannerSubtitle
   │    ≤ 200 строк × 200    │                   ?? пустая строка-разделитель
   │    столбцов             │           Статус   = locked
   │ 4. очистка заголовка +  │           Путь     = locked
   │    подзаголовка (одна   │
   │    строка, ≤ 80 / 160   │
   │    символов)            │
   │ 5. мемоизация по        │
   │    источнику            │
   └─────────────────────────┘
```

Пятиэтапный алгоритм разрешения выполняется один раз при загрузке настроек
и снова только при событиях перезагрузки настроек:

1. **Нормализация**. Простая `string` или `{ path }` преобразуется в
   `{ small: x, large: x }`. Объект `{ small, large }` передаётся как есть.
2. **Разрешение каждого уровня**. Для каждого `AsciiArtSource`:
   - Если это строка — используется как есть.
   - Если это `{ path }` — файл читается синхронно с защитой `O_NOFOLLOW`
     (под Windows: просто чтение — константа не экспортируется),
     с ограничением в 64 КБ. Относительные пути разрешаются относительно
     _каталога файла настроек, которому принадлежит путь_ — рабочие
     настройки относительно `.qwen/` рабочей области, пользовательские
     настройки относительно `~/.qwen/`. При ошибке чтения логируется
     предупреждение `[BANNER]` и выполняется возврат к умолчанию для
     данного уровня.
3. **Очистка**. Специфичный для баннера очиститель удаляет префиксы
   OSC / CSI / SS2 / SS3 и заменяет все остальные управляющие байты
   C0 / C1 (и DEL) пробелом, сохраняя `\n`, чтобы многострочная графика
   сохранилась. Удаляются конечные пробелы в каждой строке, затем всё
   ограничивается 200 строками × 200 столбцов. Всё, что выходит за
   пределы, обрезается, и логируется предупреждение `[BANNER]`.
4. **Выбор уровня во время рендеринга**. В `Header.tsx` для разрешённых
   `small` и `large` оценивается текущий доступный бюджет ширины
   (`availableTerminalWidth ≥ logoWidth + logoGap + minInfoPanelWidth`):
   - Предпочтительнее `large`, если он помещается.
   - Иначе — возврат к `small`, если он помещается.
   - Иначе, **если пользователь предоставил какую-либо собственную графику**,
     колонка логотипа скрывается полностью (существующая ветвь
     `showLogo = false`) — возврат к стандартному логотипу QWEN здесь
     молча отменил бы white-label развёртывание на узких терминалах.
     Информационная панель всё равно отображается.
   - Иначе (если пользователь не предоставил никакой собственной графики)
     выполняется возврат к `shortAsciiLogo`, и существующий шлюз ширины
     решает, показывать или скрывать логотип по умолчанию.
5. **Запасной вариант**. Если оба уровня оказались пустыми или
   недействительными из-за мягких ошибок (отсутствует файл, очистка
   отклонила всё, некорректная конфигурация), поведение такое, как если
   бы настройка не была задана: отрисовывается `shortAsciiLogo` и
   используется шлюз ширины для логотипа по умолчанию. CLI ни в коем
   случае не должен аварийно завершаться из-за ошибки в конфигурации
   баннера.
Псевдокод для выбора уровня:

```ts
function pickTier(
  small: string | undefined,
  large: string | undefined,
  availableWidth: number,
  logoGap: number,
  minInfoPanelWidth: number,
): string | undefined {
  for (const candidate of [large, small]) {
    if (!candidate) continue;
    const w = getAsciiArtWidth(candidate);
    if (availableWidth >= w + logoGap + minInfoPanelWidth) {
      return candidate;
    }
  }
  return undefined; // column с логотипом скрыта
}
```

```ts

  asciiArt: { small?: string; large?: string };
  title?: string;
  subtitle?: string;
}

export function resolveCustomBanner(settings: LoadedSettings): ResolvedBanner;
```

Резолвер выполняет нормализацию, чтение файлов, санитизацию и
кэширование, описанные в конвейере разрешения выше. Он вызывается один раз
во время запуска CLI и повторно при событиях горячей перезагрузки
настроек. Пути файлов в области видимости берутся из `settings.system.path` /
`settings.workspace.path` / `settings.user.path` напрямую, так что каждый
`{ path }` разрешается относительно файла, объявившего его; настройки
рабочей области полностью пропускаются, когда `settings.isTrusted` равно
false.

## Рассмотренные альтернативные подходы
Было рассмотрено пять вариантов формы этой функциональности. Они перечислены здесь, чтобы будущие участники понимали дизайн-пространство и могли пересмотреть выбор, если ограничения изменятся.

### Вариант 1 — Три плоских параметра (РЕКОМЕНДУЕТСЯ, соответствует запросу)

```jsonc
{
  "ui": {
    "customAsciiArt": "...", // string | {path} | {small,large}
    "customBannerTitle": "Acme CLI",
    "hideBanner": false,
  },
}
```

- **Эффект**: минимальная пользовательская поверхность; ровно то, что запрашивается.
- **Плюсы**: нулевое время обучения; тривиальная документация; согласованность с существующими плоскими свойствами `ui.*` (`hideTips`, `customWittyPhrases` и т.д.).
- **Минусы**: три ключа верхнего уровня, которые концептуально принадлежат друг другу, не сгруппированы; будущие настройки только для баннера (градиент, подзаголовок) добавят больше соседей в `ui` вместо чистой вложенности.

### Вариант 2 — Вложенное пространство имён `ui.banner`

```jsonc
{
  "ui": {
    "banner": {
      "hide": false,
      "title": "Acme CLI",
      "asciiArt": { "path": "./brand.txt" },
    },
  },
}
```

- **Эффект**: те же возможности, что и в варианте 1, но сгруппированы по функциональности.
- **Плюсы**: чистое пространство имён для будущих настроек только для баннера; более лёгкое обнаружение через `/settings`.
- **Минусы**: отход от точной формулировки запроса; существующие настройки UI в основном плоские (только `ui.accessibility` и `ui.statusLine` вложены), поэтому согласованность неоднозначна; добавляет один уровень вложенности, который нужно запомнить пользователям.

### Вариант 3 — Предустановки профиля баннера + переопределения слотов

```jsonc
{
  "ui": {
    "bannerProfile": "minimal" | "default" | "branded" | "hidden",
    "banner": { /* переопределения слотов для 'branded' */ }
  }
}
```

- **Эффект**: пользователи выбирают из именованных пресетов; продвинутые пользователи переопределяют слоты внутри выбранного профиля.
- **Плюсы**: хороший UX онбординга; пресеты поставляются с CLI.
- **Минусы**: значительная сложность; поддержка пресетов — это обязательство по обслуживанию; запрос просит сырые настройки, а не курирование.

### Вариант 4 — Полное переопределение баннера (один строковый шаблон)

```jsonc
{
  "ui": {
    "bannerTemplate": "{{logo}}\n>_ {{title}} ({{version}})\n{{auth}} | {{model}}\n{{path}}",
  },
}
```

- **Эффект**: один свободный шаблон с фиксированными переменными, которые подставляются.
- **Плюсы**: максимальная гибкость для нестандартных макетов.
- **Минусы**: реализует вёрстку в пользовательском пространстве; теряет двухколоночную устойчивость Ink к ширине терминала; очень легко написать шаблон, который ломается на узких терминалах; большая зона поражения для небольшой функции.

### Вариант 5 — API плагинов / хуков

Предоставить хук рендеринга баннера через систему расширений.

- **Эффект**: настройка на уровне кода; расширения могут рендерить что угодно.
- **Плюсы**: максимальная мощность; позволяет предприятиям выпускать закрытый плагин брендинга.
- **Минусы**: большая поверхность API; требуется проверка безопасности для произвольного рендеринга в терминале; чрезмерно масштабирован для данной задачи.

### Рекомендация

**Вариант 1** рекомендуется. Он дословно удовлетворяет запрос, вписывается в существующий стиль `ui.*` и избегает принудительного решения о вложенном пространстве имён до того, как мы узнаем, как на самом деле будут выглядеть другие настройки только для баннера. Если в будущем начнут накапливаться соседние параметры, миграция на вариант 2 будет аддитивной — `ui.banner.title` и `ui.customBannerTitle` могут сосуществовать в период устаревания.

## Безопасность и обработка ошибок

Пользовательский контент баннера отображается дословно в терминале И, в форме пути, читается с диска. Обе поверхности достижимы для атак, если загружен враждебный или скомпрометированный файл настроек. Применяется та же модель угроз, что и для функции session-title.

| Проблема                                                         | Защита                                                                                                                                                                                                                                    |
| ---------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Внедрение ANSI / OSC-8 / CSI в арте, заголовке или подзаголовке | Специфичный для баннера очиститель (`sanitizeArt`/`sanitizeSingleLine`): удаляет лидеры OSC/CSI/SS2/SS3 и заменяет каждый другой управляющий байт C0/C1 (и DEL) пробелом. Применяется до рендеринга и записи в кеш.                        |
| Файл большого размера замораживает запуск                        | Жёсткое ограничение 64 КБ на чтение файлов.                                                                                                                                                                                               |
| Патологический арт замораживает вёрстку                         | Ограничение 200 строк × 200 столбцов для каждой разрешённой строки. Избыток усекается; логируется предупреждение `[BANNER]`.                                                                                                             |
| Перенаправление по симлинку для формы пути                      | `O_NOFOLLOW` при чтении файлов (Windows: только чтение; константа не экспортируется).                                                                                                                                                    |
| Отсутствующий или недоступный для чтения файл                   | Обработчик ошибок: логируется предупреждение `[BANNER]`, возвращается значение по умолчанию. Никогда не выбрасывает исключение в UI.                                                                                                      |
| Заголовок или подзаголовок с переводами строк / избыточной длиной | Переводы строк заменяются пробелами; ограничение 80 (заголовок) / 160 (подзаголовок) символов.                                                                                                                                           |
| Недоверенное рабочее пространство влияет на рендеринг или чтение файлов | Когда `settings.isTrusted` имеет значение false, резолвер полностью пропускает `settings.workspace` (отражает шлюз доверия, который применяет `settings.merged`).                                                                         |
| Состояние гонки при перезагрузке настроек                       | Разрешение кешируется по источнику (путь или хеш строки) на каждый вызов. Перезагрузки повторно запускают резолвер и перечитывают затронутые файлы.                                                                                      |
Сводка режимов сбоев: каждый мягкий сбой завершается `shortAsciiLogo` (или
заблокированным заголовком по умолчанию) плюс предупреждение в логе отладки. Жесткие сбои
(выбрасываемые ошибки) не допускаются ни в одной ветке резолвера.

Основной сценарий использования этой функции (white-label / мультитенантность) почти всегда предполагает, что дизайнер создаёт осмысленное ASCII-изображение, а не полагается на стандартный шрифт figlet. Пользователи, которым нужна однострочная генерация, уже могут получить её через `npx figlet "xxxCode" > brand.txt` + `customAsciiArt: { "path": "./brand.txt" }` — тот же результат, без добавления зависимостей и бремени поддержки внутри Qwen Code. ## Verification plan

Для будущего PR реализации должны проходить следующие сквозные проверки.

1. `~/.qwen/settings.json` с `customBannerTitle: "Acme CLI"` и встроенной строкой `customAsciiArt` → `qwen` показывает новый заголовок и изображение; суффикс версии всё ещё присутствует.
2. `customBannerSubtitle: "Built-in Acme Skills"` → строка подзаголовка отображается между заголовком и строкой аутентификации/модели вторичным цветом текста; аутентификация, модель и путь всё ещё видны. Отмена этой настройки восстанавливает пустую строку-разделитель (обратная совместимость).
3. `hideBanner: true` → `qwen` запускается без баннера; подсказки и чат отображаются нормально.
4. `customAsciiArt: { "path": "./brand.txt" }` в рабочей области `settings.json`, с `brand.txt` рядом в `.qwen/` → загружается с диска при открытии рабочей области.
5. `customAsciiArt: { "small": "...", "large": "..." }` → изменить размер терминала между широким / средним / узким; большой при широкой ширине, маленький при средней ширине, колонка логотипа скрыта при узкой ширине, информационная панель всегда видна.
6. Вставить `\x1b[31mhostile` в `customBannerTitle` _и_ `customBannerSubtitle` → оба отображаются как буквальный текст, не интерпретируются как красный.
7. Указать `path` на отсутствующий файл → CLI запускается; предупреждение `[BANNER]` появляется в `~/.qwen/debug/<sessionId>.txt`; отображается стандартное изображение.
8. Открыть рабочее дерево с отключенным доверием к рабочей области → определённое в рабочей области `customAsciiArt` (включая записи `{ path }`) молча игнорируется; настройки уровня пользователя всё ещё применяются.
9. `npm test` и `npm run typecheck` проходят для пакета CLI; модульные тесты в `customBanner.test.ts` покрывают каждую допустимую форму и каждый путь сбоя (отсутствующий файл, слишком большой файл, внедрение ANSI, некорректный объект).

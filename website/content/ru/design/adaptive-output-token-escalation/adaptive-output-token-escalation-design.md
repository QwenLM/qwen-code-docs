# Дизайн адаптивного эскалации выходных токенов

> Снижает избыточное резервирование GPU-слотов примерно в 4 раза за счёт стратегии «низкое значение по умолчанию + эскалация при обрезании» для выходных токенов, с многократным восстановлением для ответов, превышающих даже лимит после эскалации.

## Проблема

Каждый API-запрос резервирует фиксированный GPU-слот пропорционально `max_tokens`. Прежнее значение по умолчанию в 32K токенов означает, что каждый запрос резервирует выходной слот на 32K, но 99% ответов укладываются в 5K токенов. Это приводит к избыточному резервированию GPU на 4–6 раз, ограничивая параллельность сервера и увеличивая стоимость.

## Решение

Используется ограниченный дефолт в **8K** выходных токенов. Когда ответ обрезается (модель достигает `max_tokens`):

1. **Эскалация** до полного выходного лимита модели (с минимумом 64K для неизвестных моделей)
2. Если снова обрезано — **восстановление**: сохраняем частичный ответ в истории и добавляем сообщение-продолжение, до 3 раз
3. Если восстановление исчерпано — возвращаемся к рекомендациям планировщика инструментов

Поскольку менее 1% запросов реально обрезаются, это значительно снижает среднее резервирование слотов, сохраняя качество вывода для длинных ответов.

## Определение лимита токенов

Эффективный `max_tokens` определяется в следующем порядке приоритета:

| Приоритет   | Источник                                              | Значение (известная модель)       | Значение (неизвестная модель) | Поведение при эскалации                     |
| ----------- | ----------------------------------------------------- | --------------------------------- | ----------------------------- | ------------------------------------------- |
| 1 (наивысш.)| Конфигурация пользователя (`samplingParams.max_tokens`)| `min(userValue, modelLimit)`      | `userValue`                   | Без эскалации                               |
| 2           | Переменная окружения (`QWEN_CODE_MAX_OUTPUT_TOKENS`)  | `min(envValue, modelLimit)`       | `envValue`                    | Без эскалации                               |
| 3 (наинизш.)| Ограниченный дефолт                                   | `min(modelLimit, 8K)`             | `min(32K, 8K)` = 8K           | Эскалация до лимита модели (минимум 64K) + восстановление |

«Известная модель» — та, для которой есть явная запись в `OUTPUT_PATTERNS` (проверяется через `hasExplicitOutputLimit()`). Для известных моделей эффективное значение всегда ограничивается заявленным выходным лимитом модели, чтобы избежать ошибок API. Неизвестные модели (кастомные развёртывания, self-hosted эндпоинты) пропускают значение пользователя напрямую, так как бэкенд может поддерживать большие лимиты.

Эта логика реализована в трёх генераторах содержимого:

- `DefaultOpenAICompatibleProvider.applyOutputTokenLimit()` — провайдеры, совместимые с OpenAI
- `DashScopeProvider` — наследует `applyOutputTokenLimit()` от дефолтного провайдера
- `AnthropicContentGenerator.buildSamplingParameters()` — провайдер Anthropic

## Механизм эскалации

Логика эскалации находится в `geminiChat.ts`, **вне** основного цикла повторных попыток. Это сделано намеренно:

1. Цикл повторных попыток обрабатывает временные ошибки (ограничения скорости, невалидные потоки, проверку содержимого)
2. Обрезание — не ошибка, а успешный ответ, который был прерван
3. Ошибки из потока после эскалации должны напрямую передаваться вызывающему коду, а не перехватываться логикой повторных попыток

### Шаги эскалации (geminiChat.ts)

```
1. Поток завершён успешно (lastError === null)
2. Последний чанк содержит finishReason === MAX_TOKENS
3. Проходятся проверки-ограждения:
   - maxTokensEscalated === false (предотвратить бесконечную эскалацию)
   - hasUserMaxTokensOverride === false (уважать намерения пользователя)
4. Вычисляем лимит после эскалации: max(ESCALATED_MAX_TOKENS, tokenLimit(model, 'output'))
5. Извлекаем частичный ответ модели из истории чата
6. Возвращаем событие RETRY (isContinuation: false) → UI отбрасывает частичный вывод и сбрасывает буферы
7. Отправляем тот же запрос повторно с maxOutputTokens: escalatedLimit
```

### Шаги восстановления (geminiChat.ts)

Если ответ после эскалации также обрезан (finishReason === MAX_TOKENS), цикл восстановления запускается до `MAX_OUTPUT_RECOVERY_ATTEMPTS` (3) раз:

```
1. Частичный ответ модели уже находится в истории (добавлен processStreamResponse)
2. Добавляем восстановительное пользовательское сообщение: OUTPUT_RECOVERY_MESSAGE
3. Возвращаем событие RETRY (isContinuation: true) → UI сохраняет текстовый буфер для продолжения
4. Отправляем повторно с обновлённой историей (модель видит свой частичный вывод + инструкцию восстановления)
5. Если всё ещё обрезано и остались попытки — возвращаемся к шагу 1
6. Если попытка восстановления выбросила ошибку (пустой ответ, сетевая ошибка):
   - Удаляем из истории зависшее восстановительное сообщение
   - Выходим из цикла восстановления
```

### Очистка состояния при RETRY (turn.ts)

Когда класс `Turn` получает событие RETRY, он очищает накопленное состояние для предотвращения несоответствий:

- `pendingToolCalls` — очищается, чтобы избежать дублирования вызовов инструментов, если первый обрезанный ответ содержал завершённые вызовы, которые повторяются в ответе после эскалации
- `pendingCitations` — очищается, чтобы избежать дублирования цитирований
- `finishReason` — сбрасывается в `undefined`, чтобы использовался finishReason нового ответа

Флаг `isContinuation` передаётся в UI, чтобы он мог решить, сбрасывать текстовые буферы (эскалация) или сохранять их (восстановление).

## Константы

Определены в `geminiChat.ts` и `tokenLimits.ts`:

| Константа                       | Значение | Назначение                                                  |
| ------------------------------- | -------- | ----------------------------------------------------------- |
| `CAPPED_DEFAULT_MAX_TOKENS`     | 8 000    | Дефолтный лимит выходных токенов, если пользователь не задал переопределение |
| `ESCALATED_MAX_TOKENS`          | 64 000   | Минимум для эскалации (используется, когда лимит модели неизвестен) |
| `MAX_OUTPUT_RECOVERY_ATTEMPTS`  | 3        | Максимальное количество попыток многократного восстановления после эскалации |

Эффективный лимит после эскалации: `max(ESCALATED_MAX_TOKENS, tokenLimit(model, 'output'))`:

| Модель            | Лимит после эскалации |
| ----------------- | --------------------- |
| Claude Opus 4.6   | 131 072 (128K)        |
| GPT-5 / o-series  | 131 072 (128K)        |
| Qwen3.x           | 65 536 (64K)          |
| Неизвестные модели| 64 000 (минимум)      |

## Дизайн-решения

### Почему дефолт 8K?

- 99% ответов укладываются в 5K токенов
- 8K даёт разумный запас для чуть более длинных ответов без запуска лишних повторных попыток
- Снижает среднее резервирование слота с 32K до 8K (улучшение в 4 раза)

### Почему эскалация до лимита модели, а не до фиксированных 64K?

- Модели с более высокими выходными лимитами (Claude Opus 128K, GPT-5 128K) без необходимости ограничивались 64K
- Использование реального лимита модели покрывает подавляющее большинство длинных выводов без второй повторной попытки
- `ESCALATED_MAX_TOKENS` (64K) служит минимумом для неизвестных моделей, где `tokenLimit()` возвращает дефолтные 32K

### Почему многократное восстановление вместо прогрессивной эскалации?

- Прогрессивная эскалация (8K → 16K → 32K → 64K) требует каждый раз генерировать полный ответ заново
- Многократное восстановление сохраняет частичный ответ и позволяет модели продолжить, экономя токены и задержку
- Восстановительные сообщения дёшевы (~40 токенов каждое) по сравнению с перегенерацией больших ответов
- Ограничение в 3 попытки предотвращает бесконечные циклы, покрывая большинство практических случаев

### Почему эскалация вне цикла повторных попыток?

- Обрезание — это успешный случай, а не ошибка
- Ошибки из потока после эскалации (ограничения скорости, сетевые сбои) должны передаваться напрямую, а не молча повторяться с неверными параметрами
- Позволяет сохранить фокус цикла повторных попыток на его исходной цели (восстановление после временных ошибок)
- Ошибки восстановления обрабатываются отдельно, чтобы не прерывать весь диалог

# Проектирование адаптивного повышения лимита выходных токенов

> Сокращает избыточное резервирование GPU-слотов примерно в 4 раза за счет стратегии «низкий умолчательный лимит + повышение при обрезании» для выходных токенов, с многократным восстановлением для ответов, превышающих даже повышенный лимит.

## Проблема

Каждый API-запрос резервирует фиксированный GPU-слот пропорционально `max_tokens`. Предыдущее значение по умолчанию в 32 тыс. токенов означает, что каждый запрос резервирует слот на 32 тыс. выходных токенов, но 99% ответов имеют длину менее 5 тыс. токенов. Это приводит к избыточному резервированию GPU-мощностей в 4-6 раз, ограничивая параллелизм сервера и увеличивая стоимость.

## Решение

Используйте ограниченный лимит по умолчанию в **8 тыс.** выходных токенов. Когда ответ обрезается (модель достигает `max_tokens`):

1. **Повысьте** до полного лимита выходных токенов модели (с минимальным значением 64 тыс. для неизвестных моделей)
2. Если все еще обрезан, **восстановите**, сохранив частичный ответ в истории и добавив сообщение о продолжении, до 3 раз
3. Если восстановление исчерпано, вернитесь к руководству планировщика инструментов по обрезанию

Поскольку <1% запросов действительно обрезаются, это значительно снижает среднее резервирование слотов, сохраняя качество вывода для длинных ответов.

## Определение лимита токенов

Эффективное значение `max_tokens` определяется в следующем порядке приоритетов:

| Приоритет | Источник                                                | Значение (известная модель)        | Значение (неизвестная модель) | Поведение при повышении                             |
| --------- | ------------------------------------------------------- | ---------------------------------- | ----------------------------- | --------------------------------------------------- |
| 1 (наивысший) | Конфигурация пользователя (`samplingParams.max_tokens`)            | `min(userValue, modelLimit)` | `userValue`             | Повышение не выполняется                                   |
| 2    | Переменная окружения (`QWEN_CODE_MAX_OUTPUT_TOKENS`) | `min(envValue, modelLimit)`  | `envValue`            | Повышение не выполняется                                   |
| 3 (самый низкий)  | Ограниченное значение по умолчанию                                       | `min(modelLimit, 8K)`        | `min(32K, 8K)` = 8K   | Повышается до лимита модели (минимум 64 тыс.) + восстановление |

«Известная модель» — это модель, для которой существует явная запись в `OUTPUT_PATTERNS` (проверяется через `hasExplicitOutputLimit()`). Для известных моделей эффективное значение всегда ограничено заявленным лимитом выходных токенов модели, чтобы избежать ошибок API. Для неизвестных моделей (пользовательские развертывания, самостоятельно размещенные конечные точки) значение пользователя передается напрямую, так как бэкенд может поддерживать более высокие лимиты.

Эта логика реализована в трех генераторах контента:

- `DefaultOpenAICompatibleProvider.applyOutputTokenLimit()` — провайдеры, совместимые с OpenAI
- `DashScopeProvider` — наследует `applyOutputTokenLimit()` от стандартного провайдера
- `AnthropicContentGenerator.buildSamplingParameters()` — провайдер Anthropic
## Механизм эскалации

Логика эскалации находится в `geminiChat.ts`, **вне** основного цикла повторных попыток. Это сделано намеренно:

1. Цикл повторных попыток обрабатывает временные ошибки (ограничения скорости, некорректные потоки, проверку содержимого)
2. Усечение — это не ошибка, а успешный ответ, который был прерван
3. Ошибки из эскалированного потока должны передаваться напрямую вызывающему коду, а не перехватываться логикой повторных попыток

### Шаги эскалации (geminiChat.ts)

```
1. Поток успешно завершен (lastError === null)
2. Последний фрагмент имеет finishReason === MAX_TOKENS
3. Проверки-ограничения пройдены:
   - maxTokensEscalated === false (предотвращает бесконечную эскалацию)
   - hasUserMaxTokensOverride === false (уважение намерений пользователя)
4. Вычисление эскалированного лимита: max(ESCALATED_MAX_TOKENS, tokenLimit(model, 'output'))
5. Удаление частичного ответа модели из истории чата
6. Генерация события RETRY (isContinuation: false) → UI отбрасывает частичный вывод и сбрасывает буферы
7. Повторная отправка того же запроса с maxOutputTokens: escalatedLimit
```

### Шаги восстановления (geminiChat.ts)

Если эскалированный ответ также был усечен (finishReason === MAX_TOKENS), цикл восстановления выполняется до `MAX_OUTPUT_RECOVERY_ATTEMPTS` (3) раз:

```
1. Частичный ответ модели уже находится в истории (добавлен processStreamResponse)
2. Отправка восстанавливающего сообщения пользователя: OUTPUT_RECOVERY_MESSAGE
3. Генерация события RETRY (isContinuation: true) → UI сохраняет текстовый буфер для продолжения
4. Повторная отправка с обновленной историей (модель видит свой частичный вывод + инструкцию восстановления)
5. Если ответ снова усечен и остались попытки, возврат к шагу 1
6. Если попытка восстановления завершается ошибкой (пустой ответ, сетевая ошибка):
   - Удаление зависшего восстанавливающего сообщения из истории
   - Выход из цикла восстановления
```

### Очистка состояния при RETRY (turn.ts)

Когда класс `Turn` получает событие RETRY, он очищает накопленное состояние, чтобы избежать несоответствий:

- `pendingToolCalls` — очищается, чтобы избежать дублирующих вызовов инструментов, если первый усеченный ответ содержал завершенные вызовы инструментов, которые повторяются в эскалированном ответе
- `pendingCitations` — очищается, чтобы избежать дублирующих цитирований
- `finishReason` — сбрасывается в `undefined`, чтобы использовался finishReason нового ответа

Флаг `isContinuation` передается в UI, чтобы он мог решить, нужно ли сбрасывать текстовые буферы (эскалация) или сохранять их (восстановление).

## Константы

Определены в `geminiChat.ts` и `tokenLimits.ts`:

| Константа                       | Значение | Назначение                                               |
| ------------------------------- | -------- | -------------------------------------------------------- |
| `CAPPED_DEFAULT_MAX_TOKENS`    | 8,000    | Лимит выходных токенов по умолчанию, если пользователь не задал переопределение |
| `ESCALATED_MAX_TOKENS`         | 64,000   | Нижняя граница эскалации (используется, когда лимит модели неизвестен) |
| `MAX_OUTPUT_RECOVERY_ATTEMPTS` | 3        | Максимальное количество попыток многократного восстановления после эскалации |

Эффективный эскалированный лимит равен `max(ESCALATED_MAX_TOKENS, tokenLimit(model, 'output'))`:

| Модель            | Эскалированный лимит |
| ----------------- | -------------------- |
| Claude Opus 4.6   | 131,072 (128K)       |
| GPT-5 / o-series  | 131,072 (128K)       |
| Qwen3.x           | 65,536 (64K)         |
| Неизвестные модели | 64,000 (нижняя граница) |

## Проектные решения

### Почему 8K по умолчанию?

- 99% ответов содержат менее 5K токенов
- 8K обеспечивает разумный запас для слегка более длинных ответов без излишних повторных попыток
- Уменьшает среднее резервирование слотов с 32K до 8K (улучшение в 4 раза)

### Почему эскалация до лимита модели вместо фиксированных 64K?

- Модели с более высокими лимитами вывода (Claude Opus 128K, GPT-5 128K) были неоправданно ограничены до 64K
- Использование фактического лимита модели перекрывает подавляющее большинство длинных выводов без второй попытки
- `ESCALATED_MAX_TOKENS` (64K) служит нижней границей для неизвестных моделей, где `tokenLimit()` возвращает 32K по умолчанию

### Почему многократное восстановление вместо прогрессивной эскалации?

- Прогрессивная эскалация (8K → 16K → 32K → 64K) требует полной регенерации ответа каждый раз
- Многократное восстановление сохраняет частичный ответ и позволяет модели продолжить, экономя токены и время
- Восстанавливающие сообщения дешевы (~40 токенов каждое) по сравнению с регенерацией больших ответов
- Лимит в 3 попытки предотвращает бесконечные циклы, одновременно покрывая большинство практических случаев

### Почему эскалация находится вне цикла повторных попыток?

- Усечение — это успешный случай, а не ошибка
- Ошибки из эскалированного потока (ограничения скорости, сетевые сбои) должны передаваться напрямую, а не повторяться молча с неверными параметрами
- Позволяет циклу повторных попыток сосредоточиться на своей исходной цели (восстановление после временных ошибок)
- Ошибки восстановления перехватываются отдельно, чтобы не прерывать весь диалог

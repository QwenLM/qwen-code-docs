```markdown
# Структурированный вывод (`--json-schema`) — Дизайн

В этом документе описаны решения по реализации функции `--json-schema` в безголовом режиме. Руководство пользователя находится в
[`docs/users/features/structured-output.md`](../../users/features/structured-output.md).

## Цель

В безголовых запусках (`qwen -p`, pipe stdin или позиционный промпт) вызывающий может ограничить финальный ответ модели заданной JSON-схемой и получить валидный полезный отклик в машиночитаемом формате, который скрипты и утилиты могут обрабатывать напрямую. Побочная проза модели во время планирования допускается, но запуск должен завершиться откликом, соответствующим схеме, а не произвольным текстом.

## Подход: синтетический инструмент, чья схема параметров совпадает с пользовательской схемой

Когда установлен `--json-schema`, `Config.createToolRegistry` регистрирует синтетический инструмент `structured_output`
([`syntheticOutput.ts`](../../../packages/core/src/tools/syntheticOutput.ts)).
Его `parametersJsonSchema` — это в точности схема, переданная пользователем; его
`execute()` возвращает стоп-сообщение `llmContent`. Инфраструктура вызова инструментов уже валидирует аргументы на стороне клиента с помощью `parametersJsonSchema` (через Ajv в `BaseDeclarativeTool.build()`), так что утверждение «модель вернула ответ, соответствующий схеме» сводится к «модель успешно вызвала `structured_output`».

Из этого автоматически вытекают три свойства:

1. **Нет собственного валидатора.** Проверка `validateToolParams` на основе Ajv уже выполняется внутри `BaseDeclarativeTool.build()` и отклоняет несоответствующие аргументы до того, как срабатывает `execute()`.
2. **Стандартное поведение при повторных попытках.** Ошибка валидации передаётся модели как ошибка вызова инструмента, так же как и любая другая ошибка аргументов. Модель видит сообщение Ajv и может исправиться на следующем шаге.
3. **Агностичность по отношению к провайдерам.** Gemini, OpenAI и Anthropic сериализуют схемы параметров инструментов одинаково (через абстракцию `DeclarativeTool`); синтетический инструмент подключается ко всем трём.

Инструмент регистрируется с флагом `alwaysLoad: true`, чтобы инфраструктура загрузки по требованию `ToolSearch` (введённая в #3589 — уменьшает поверхность видимых инструментов, откладывая редко используемые за поисковым вызовом, монтируя полные схемы только когда модель их запрашивает) никогда не скрывала его от модели. Без этого флага модель не узнала бы о существовании терминального контракта.

## Конвейер валидации на этапе разбора

`resolveJsonSchemaArg(raw)` в
[`packages/cli/src/config/config.ts`](../../../packages/cli/src/config/config.ts)
выполняет четыре проверки, прежде чем схема достигнет `Config.createToolRegistry`:

1. **Определение источника.** Принимает либо встроенный JSON-литерал, либо
   `@path/to/file`. Форма `@path` сначала проверяет разрешённый путь через `stat`, отклоняет нерегулярные файлы (FIFO, символьные устройства, каталоги), ограничивает размер до 4 МиБ и при ошибке разбора JSON выводит общее сообщение об ошибке (без префикса содержимого файла в stderr).
2. **Форма JSON.** Результат разбора должен быть объектом (не массивом) —
   примитивы, булевы значения и массивы отклоняются с понятным сообщением.
3. **Корень принимает объекты** —
   [`schemaRootAcceptsObject`](../../../packages/cli/src/config/config.ts).
   API функций всегда передают объекты в качестве аргументов инструментов; корневая схема, например `{type: "array"}`, зарегистрировала бы непригодный инструмент.
   Обход обрабатывает `type`, `const`, `enum`, `anyOf`, `oneOf`,
   `allOf`, `not`, `if` / `then` / `else` и корневой `$ref`.
4. **Строгая компиляция Ajv** —
   [`SchemaValidator.compileStrict`](../../../packages/core/src/utils/schemaValidator.ts).
   Отдельный экземпляр Ajv с `strictSchema: true` выявляет опечатки вроде `propertees`, которые lenient-валидатор во время выполнения молча проглотил бы.

### Границы `schemaRootAcceptsObject`

Обход намеренно выполняется по принципу «наилучшего усилия». Он ловит однозначные случаи «это никогда не сможет принять объект», а всё, что требует анализа выполнимости всей схемы, оставляет Ajv на этапе выполнения.

**Решается на этапе разбора:**

| Шаблон                                                | Результат                                                         |
| ----------------------------------------------------- | ----------------------------------------------------------------- |
| Присутствует `type`, не включает `"object"`           | отклонён                                                          |
| `type: ["object", "null"]` и т.д.                     | принят                                                            |
| `const`: значение не объект                           | отклонён                                                          |
| `enum`: нет членов-объектов (включая пустой)          | отклонён                                                          |
| `anyOf`/`oneOf`: пустой массив                        | отклонён                                                          |
| `anyOf`/`oneOf`: ни одна ветвь не допускает объект     | отклонён                                                          |
| `allOf`: любая ветвь равна `false` или отклоняет объект | отклонён                                                          |
| Корневой `$ref` (с или без соседнего `type`)           | отклонён                                                          |
| `not`: голый `{type: "object"}` (без сужающих ключевых слов) | отклонён                                                    |
| `not`: `{type: "object", required: […], …}` и т.д.      | принят (сужающие ключевые слова оставляют некоторые объекты выполнимыми; откладывается) |
| `if: true` + `then` отклоняет объект                   | отклонён                                                          |
| `if: false` + `else` отклоняет объект                  | отклонён                                                          |

**Откладывается на Ajv во время выполнения:**

- `$ref` внутри ветвей `anyOf` / `oneOf` / `allOf` (непрозрачно — локальное разрешение `$ref` требовало бы обнаружения циклов, экранирования JSON Pointer и обработки `$defs` vs `definitions`; затраты перевешивают выгоду для проверки на уровне разбора по принципу «наилучшего усилия»).
- `if`, значением которого является схема объекта (можно решить только относительно значения-кандидата).
- Отрицательные шаблоны `anyOf` / `oneOf` / `const` сложнее, чем `not.type`.
- Произвольная экспозиция к ReDoS через `pattern` (предоставлено пользователем; модель угроз узкая, так как флаг — аргумент CLI, а не сетевой ввод).

Путь выхода при превышении `maxSessionTurns` добавляет подсказку, специфичную для `--json-schema`, которая указывает пользователю на типичный симптом застрявшего запуска (модель никогда не вызывала `structured_output`) и его две вероятные причины (инструмент запрещён разрешениями / схема невыполнима), чтобы падение во время выполнения сопровождалось видимой для пользователя диагностикой.

## Выполнение: диспетчеризация шагов

[`packages/cli/src/nonInteractiveCli.ts`](../../../packages/cli/src/nonInteractiveCli.ts)
обрабатывает диспетчеризацию во время выполнения. Специфика структурированного вывода:

### Предварительное сканирование + подавление «родственных» вызовов

Когда модель отправляет `structured_output` вместе с другими инструментами в одном шаге ассистента, синтетический вызов является терминальным контрактом. Предварительное сканирование в `processToolCallBatch` фильтрует `requestsToExecute`, оставляя **только** вызовы `structured_output`; таким образом, «родственные» вызовы с побочными эффектами (`write_file`, `run_shell_command`, `edit`, …) никогда не выполняются.

Примеры пакетов (когда активен `--json-schema`):

| Модель отправляет                                              | Поведение                                                                                                                                                                                                                                                                                                                     |
| -------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `[write_file(…), structured_output(…)]`                        | `write_file` пропускается. `structured_output` проходит валидацию, запуск завершается.                                                                                                                                                                                                                                         |
| `[structured_output(плохие-аргументы), structured_output(хорошие)]` | Первый проваливает валидацию Ajv; второй успешен. Запуск завершается с аргументами второго вызова.                                                                                                                                                                                                                             |
| `[structured_output(плохие-аргументы), write_file(…)]`         | `structured_output(плохие)` проваливается. `write_file` также пропускается (подавлен заранее). Модель видит оба: сообщение об ошибке Ajv для структурированного вызова и синтезированный `"Skipped: …"` tool_result для вызова с побочным эффектом. На следующем шаге модель может повторно отправить оба или исправить структурированный вызов. |
| `[other_tool_a, other_tool_b]` (нет `structured_output`)       | Предварительное сканирование неактивно. Оба инструмента выполняются нормально; запуск НЕ завершается.                                                                                                                                                                                                                          |

Синтезированное тело `"Skipped:"` имеет два варианта:

- **Путь успеха** (на этом шаге структурированный вызов перехватил контракт):
  `"Skipped: this turn's structured_output contract took precedence as the terminal output."` — кратко, так как сессия немедленно завершается, и ни один потребитель (модель или SDK) на него не реагирует.
- **Путь повторной попытки** (ни один структурированный вызов не перехватил, модель получает ещё один шаг): добавляется `"Re-issue this call in a separate turn if needed."` — это единственный случай, когда действие модели необходимо.

### Паритет основного шага / шага «слива»

`processToolCallBatch(batchRequests, setModelOverride)` определена внутри `runNonInteractive` и вызывается из обоих:

- Цикла основного шага (в начале функции).
- `drainOneItem` (цикл ответа на cron-промпты / фоновые уведомления).

Шаг слива важен, потому что `structured_output` зарегистрирован на всю сессию, поэтому cron-задача или ответ на уведомление МОГУТ также вызвать инструмент. Обработчик обрабатывает оба места вызова одинаково во время вызова; единственная привязка к конкретному месту — это то, в какую переменную `modelOverride` записывать (передаётся как сеттер).

**Поток завершения после обработчика** различается между двумя местами: путь основного шага напрямую вызывает `return emitStructuredSuccess()`, в то время как путь шага слива требует двухшагового завершения (`processToolCallBatch` захватывает результат в переменную `structuredSubmission`, замкнутую в области видимости; `drainLocalQueue` проверяет её, чтобы остановить цикл слива, затем цикл holdback проверяет её, чтобы выйти и вызвать `emitStructuredSuccess`). Оба сходятся на одном терминальном блоке, но дополнительная косвенность в пути слива является обязательной — без неё цикл слива продолжал бы обрабатывать поставленные в очередь элементы после того, как структурированный результат был захвачен.

### Терминальный блок успеха структурированного вывода

`emitStructuredSuccess()` (также определена внутри `runNonInteractive`) — это общий путь «мы получили валидный вызов, завершаем работу»:

1. `registry.abortAll()` прерывает выполняющиеся фоновые агенты — контракт структурированного вывода одноразовый и не должен конкурировать с `task_notification` за вывод в терминал.
2. Ограниченный holdback (`STRUCTURED_SHUTDOWN_HOLDBACK_MS = 500` мс), чтобы естественные обработчики отмены только что прерванных агентов успели отправить свой терминальный `task_notification` и поместить его в `localQueue`. Условие цикла: `Date.now() < deadline && registry.hasUnfinalizedTasks()`, поэтому ожидание завершается немедленно, если ничего не выполняется (типичный случай), и никогда не блокируется дольше предела. Потолок в 500 мс — это максимальное усилие: при нагрузке могут остаться «осиротевшие» события `task_started`, если обработчик отмены конкретного агента превысит бюджет. Цикл **не** опрашивает сигнал прерывания: SIGINT, полученный во время holdback или во время последующего вывода, не прервёт уже захваченный результат. Без holdback потребители stream-json постоянно видели бы события `task_started` без соответствующих `task_notification`.
3. `flushQueuedNotificationsToSdk(localQueue)` сбрасывает все оставшиеся в очереди уведомления.
4. `finalizeOneShotMonitors()` (идемпотентен — безопасно вызывать дважды; путь шага слива уже вызывал его).
5. `adapter.emitResult({ structuredResult: …, isError: false, … })`.

### Пути сбоя

| Причина                                                          | Код выхода                     | Сообщение                                                                                                                                                                                                                                                                                           |
| ---------------------------------------------------------------- | ------------------------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Модель отправляет только обычный текст                            | 1                              | Ошибка с количеством шагов и усечённым `Output preview`.                                                                                                                                                                                                                                            |
| Модель ни разу не вызывает `structured_output` за `maxSessionTurns` шагов | 53                            | `Reached max session turns` + подсказка по `--json-schema`, указывающая на типичный симптом застрявшего запуска и две его вероятные причины.                                                                                                                                                        |
| Валидация многократно проваливается                               | (в итоге 53 через max-turns)   | Каждый сбой передаётся модели на следующем шаге с сообщением Ajv.                                                                                                                                                                                                                                     |
| Прерывание / SIGINT                                               | 130                           | Путь отмены. Обычно структурированный результат не отправляется, но цикл holdback в `emitStructuredSuccess()` не опрашивает сигнал прерывания — SIGINT, поступивший после захвата, но до или во время вывода в stdout, всё равно может сбросить результат. Надёжным сигналом является код выхода. |

## Конверт вывода

Конвейер адаптера в
[`BaseJsonOutputAdapter.buildResultMessage`](../../../packages/cli/src/nonInteractive/io/BaseJsonOutputAdapter.ts)
обрабатывает наличие `structuredResult` (проверяется через `'structuredResult' in options`, а не `!== undefined`, чтобы контракт сохранялся, даже если модель вызвала `structured_output` без аргументов при пустой схеме):

- `result` принудительно устанавливается в `JSON.stringify(payload)` — переопределяя любой накопленный адаптером текст резюме.
- Поле `structured_result` на верхнем уровне содержит сырой объект для потребителей, которые не хотят повторно разбирать строковую форму.
- `undefined` полезные отклики нормализуются до `null` (отображается как литерал `null` в JSON в обоих полях), чтобы поле не могло бесшумно исчезнуть. На практике этот запасной вариант достигается редко: выше по потоку `turn.ts` применяет `(fnCall.args || {})` перед сохранением отправленного, поэтому вызов с нулём аргументов при пустой схеме становится `{}` и отображается как `{}` в stdout, а не `null`. Шаг `?? null` — это защита для строгого случая `undefined`.

Режим TEXT выводит только поле `result` + перевод строки в stdout (любая побочная проза ассистента, накопленная во время запуска, отбрасывается — не выводится в stderr). Режим JSON выводит полный журнал событий в виде массива JSON; `structured_result` находится в последнем элементе типа `"result"` этого массива, а не в корне документа. Режим stream-json выводит каждое сообщение на отдельной строке в формате JSONL; завершающая строка `result` содержит `structured_result`.

## Конфиденциальность: редактирование на разных поверхностях

Аргументы, переданные через `structured_output`, ЯВЛЯЮТСЯ структурированным полезным откликом. На пути успеха они уже в stdout; при повторных попытках из-за ошибок валидации они могут вообще не попасть в stdout. В любом случае сохранение их на постоянных поверхностях устройства (или экспорт с устройства через телеметрию) является дублированием, которое передаёт полезный отклик в долговременное хранилище, не запрашиваемое пользователем. Правило редактирования, следовательно, «никогда не сохранять никакие аргументы этого синтетического инструмента, независимо от результата», а не просто «удалять дубликаты того, что уже в stdout».

Две поверхности подлежат редактированию, и обе используют один и тот же плейсхолдер-константу
[`STRUCTURED_OUTPUT_REDACTED_ARGS`](../../../packages/core/src/tools/syntheticOutput.ts):

- `ToolCallEvent.function_args` (телеметрия) — охватывает экспорт OTLP,
  QwenLogger, ui-telemetry и зеркало событий UI для записи чатов.
- `redactStructuredOutputArgsForRecording` (используется `recordAssistantTurn` в `geminiChat.ts`) — охватывает JSONL-файл записи чата на диске
  `~/.qwen/projects/<sanitized-cwd>/chats/<sessionId>.jsonl`.
  Сюда также попадают аргументы повторных попыток при ошибках валидации — каждый повтор также получает тот же плейсхолдер.

Общая константа предотвращает расхождение между двумя поверхностями. Метрики вызовов инструментов (длительность, успех, решение) сохраняются.

Хуки (`PreToolUse`, `PostToolUse`, `PostToolUseFailure`) намеренно **не** редактируются — они получают сырой `tool_input`, так как контракт хуков: «видеть то же, что и инструмент». Это отмечено в разделе конфиденциальности пользовательской документации как предупреждение «Хуки видят сырые аргументы», чтобы операторы могли фильтровать по `tool_name` или добавлять редактирование на стороне хука перед запуском `--json-schema` с конфиденциальными данными.

Редактирование намеренно ограничено **постоянными на устройстве** поверхностями (экспорт телеметрии + JSONL записей чатов). Сама схема всё равно отправляется провайдеру модели в каждом запросе как блок `parameters` объявления функции `structured_output` — редактирование на стороне провайдера невозможно, так как модели нужна схема для выполнения контракта вызова инструмента. Раздел конфиденциальности пользовательской документации предупреждает пользователей не помещать секреты в поля `enum`, `const`, `default`, `examples`, `description` по той же причине.

## Ограничения по разрешениям

`structured_output` намеренно исключён из `PermissionManager.CORE_TOOLS` (набора инструментов, подлежащих проверке белого списка `--core-tools`) — наряду с другими синтетическими инструментами (`agent`, `exit_plan_mode`, `ask_user_question`, `task_stop`, `send_message`). Динамически обнаруживаемые инструменты (`skill`, MCP) относятся к отдельной категории исключений, которые также обходят белый список по другим причинам. Синтетический инструмент существует только при установке `--json-schema`; добавление его в механизм белого списка означало бы, что `--core-tools read_file --json-schema X` молча отбрасывает терминальный контракт.

Явные правила `permissions.deny` и настройки `--exclude-tools` по-прежнему применяются через `PermissionManager.evaluate` → `isToolEnabled`. Оба используют один и тот же механизм запрета и оба предотвращают регистрацию — объявление инструмента удаляется из реестра, поэтому модель никогда не видит инструмент. Типичным результатом является то, что модель отвечает обычным текстом (код выхода 1). Если модель в цикле вызывает другие инструменты, не создавая текст, она в конечном итоге достигает `maxSessionTurns` (код выхода 53), и подсказка по `--json-schema` в `handleMaxTurnsExceededError` сообщает пользователю, куда смотреть.
```
**`--bare` взаимодействие.** Режим bare обходит мост между настройками и CLI-конфигурацией: `packages/cli/src/config/config.ts` строит `mergedDeny` как `[...(bareMode ? [] : settings.permissions.deny), ...]`, поэтому запреты на уровне настроек (а также `tools.exclude`) отбрасываются при `--bare`. Аргумент `--exclude-tools` на уровне argv безусловно добавляется в `mergedDeny`, поэтому он по-прежнему применяется. Синтетический инструмент регистрируется независимо от этого (управляется `jsonSchema`, а не списком запретов), так что запрет `structured_output` только в настройках молча игнорируется при `--bare`, а инструмент остаётся вызываемым.

## Контексты подагентов

`Config.createToolRegistry` принимает опцию `forSubAgent: true`, которая подавляет синтетическую регистрацию. Переопределения подагентов используют родительскую Config через делегирование прототипов (`createApprovalModeOverride` / `buildSubagentContextOverride` → `Object.create(base)`), и `this.jsonSchema` распространяется через цепочку прототипов. Без этого флага синтетический инструмент регистрировался бы и в реестре подагента, и подагент, вызывая его, получал бы `llmContent` с сообщением «сессия завершается сейчас» — но только циклы `runNonInteractive` (основной и drain) распознают это как терминальное состояние, поэтому подагент продолжал бы работать и сжигать токены на инструмент, чей контракт его цикл не может выполнить.

> **Примечание для разработчиков.** Это подавление завязано на единственный путь вызова через `createToolRegistry(forSubAgent: true)`. Любой будущий механизм порождения подагентов, который обходит этот путь, допустит утечку синтетического инструмента в реестр подагента и возродит режим отказа с бесконечным сжиганием токенов. Запасным вариантом защиты станет проверка во время выполнения внутри `syntheticOutput.execute()`, которая возвращает `fatalError` (или ничего не делает) при вызове из контекста подагента. Реализуйте её, если появится второй путь утечки.

## Защита от теневого инструмента MCP

`tool-registry.ts:registerTool` проверяет ленивую карту `factories` на совпадение имён, а не только готовую карту `tools`. Если MCP-сервер обнаруживает инструмент буквально с именем `structured_output`, путь автоматической квалификации, существующий для конфликтов готовых инструментов, срабатывает и для конфликтов фабрик: MCP-инструмент переименовывается в `mcp__<server>__structured_output`, а синтетическая фабрика остаётся под исходным именем. Без этой защиты MCP-сервер мог бы незаметно перехватить контракт структурированного вывода.

## Рассмотренные альтернативы

**Подсказка ответа с учётом схемы (без синтетического инструмента).** Просить модель «ответить JSON, соответствующим этой схеме» через системный промпт и разбирать итоговое сообщение ассистента. Отклонено, потому что у модели нет синтаксических гарантий — вывод может быть заключён в блок, предваряться произвольным текстом или содержать вымышленные поля. Валидация вызова инструмента выполняется слоем вызова функций до `execute()`, что даёт жёсткую синтаксическую + семантическую защиту.

**`response_format: {type: "json_schema", …}` от OpenAI.** Привязано к конкретному провайдеру; потребовало бы параллельных реализаций для Gemini и Anthropic. Подход с синтетическим инструментом не зависит от провайдера.

**Переставить `structured_output` в начало пакета вместо фильтрации.** Позволяет побочным эффектам выполняться, если структурированный вызов не проходит валидацию. Отклонено, потому что контракт `--json-schema` — «выдать структурированный вывод» — если модель в этом режиме, побочные эффекты от одноуровневых вызовов, скорее всего, ошибка. Полное их подавление безопаснее; модель видит `"Skipped:" tool_result` и может перевыпустить их в отдельном шаге.

**Локальное разрешение `$ref` внутри `schemaRootAcceptsObject`.** Позволило бы ловить схемы вида `{anyOf: [{$ref: "#/$defs/String"}], $defs: {…}}` на этапе разбора. Пока отклонено, так как сложность (обнаружение циклов, синтаксис JSON Pointer, `$defs` против `definitions`, частичные указатели, удалённые ссылки) перевешивает выгоду; подсказка `maxSessionTurns` уже указывает пользователю на «схема невыполнима» как на вероятную причину.

## Открытые вопросы

- Валидация ответа с учётом схемы в будущем может обзавестись защитой от ReDoS на основе `pattern`, если реальные пользователи столкнутся с катастрофическим возвратом в аргументах `--json-schema`.
- Дополнения протокола SDK (Python / TypeScript / Java SDK с типизированным полем `structured_result`) — отслеживаются отдельно; [PR #4001](https://github.com/QwenLM/qwen-code/pull/4001) (закрыт без слияния 11.05.2026) покрывал эту область до того, как была сделана работа в CLI/ядра, и был заменён.

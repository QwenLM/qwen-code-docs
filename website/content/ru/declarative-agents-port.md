# Декларативные определения агентов — портирование из Claude Code 2.1.168

Внутренний проектный документ для портирования схемы декларативных агентов (markdown +
YAML frontmatter) из Claude Code в qwen-code, в рамках решения задачи [#4821][i4821] и
координации с портированием workflow в задаче [#4721][i4721] / PR [#4732][p4732].

[i4821]: https://github.com/QwenLM/qwen-code/issues/4821
[i4721]: https://github.com/QwenLM/qwen-code/issues/4721
[p4732]: https://github.com/QwenLM/qwen-code/pull/4732

## Статус реализации (вертикальная нарезка)

PR [#4842][p4842] поставил поля с сквозным runtime-путем на тот
момент. PR [#4870][p4870] затем заменил парсер YAML для поддержки блочных
скаляров. Этот последующий PR развивает оба: он заменяет YAML
**stringifier** (PR #4870 оставил его рукописным — см.
`docs/yaml-parser-replacement.md`), выводит `mcpServers` + `hooks` на
`SubagentConfig` и подключает их к рантайму, чтобы серверы MCP и хуки,
специфичные для каждого агента, реально срабатывали при запуске сабагента.

| Поле              | Статус                               | Примечания                                                                                                                                                   |
| ----------------- | ------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `permissionMode`  | **реализовано (#4842)**             | связывает с существующим qwen `approvalMode` на этапе парсинга                                                                                               |
| `maxTurns`        | **реализовано (#4842)**             | подключено к существующему runtime-пути `runConfig.max_turns`                                                                                                |
| `color` allowlist | **реализовано (#4842)**             | ужесточает существующее поле до набора `_Y` из CC + обработка устаревшего sentinel `auto`                                                                    |
| `mcpServers`      | **реализовано (последующий PR)**    | вложенный YAML безопасен для round-trip через eemeli/`yaml` stringify; переопределение времени выполнения объединяет session и agent серверы через обёртку Subagent Config + принудительная перестройка реестра инструментов |
| `hooks`           | **реализовано (последующий PR)**    | эфемерные записи HookRegistry, регистрируемые при создании сабагента и удаляемые через `onStop`; v1 срабатывает глобально (без фильтра по области агента)     |
| `effort`          | отложено                             | в qwen-провайдерах пока нет параметра `effort` на уровне модели                                                                                              |
| `memory`          | отложено                             | в auto-memory qwen пока нет различения областей `user`/`project`/`local`                                                                                     |
| `isolation`       | отложено                             | workflow PR #4732 владеет рантаймом; значение по умолчанию для каждого агента появится вместе с ним                                                          |
| `initialPrompt`   | отложено                             | требует флаг CLI `--agent` (нет инфраструктуры main-session-agent в qwen)                                                                                    |

отложенных полей — константы схемы, семантика DL7/Ig5, сообщения об ошибках
и матрица координации с workflow по-прежнему несут нагрузку для этой работы.

## Фаза 1 — Результаты обратного инжиниринга

Все утверждения здесь были независимо проверены grep по `claude.strings` и
прошли опровержение. Уровни уверенности: **C** = Confirmed (прямые
доказательства из бинарника), **I** = Inferred (синтезировано из нескольких
подтверждённых фактов), **O** = Open (всё ещё неопределённо).

### Схема — 15 полей, опровергнутых и повторно подтверждённых

Теневая схема frontmatter агента — `Ig5`, используется внутри `ug5.agent` для
телеметрии `tengu_frontmatter_shadow_unknown_key` / `_mismatch`.
**Рабочий загрузчик — `DL7`** (`parseAgentFromMarkdown`), который выполняет
ручную проверку каждого поля с пользовательскими сообщениями об ошибках. Отдельная
**JSON-схема `JL7`** (используется `fL7` / `parseAgentFromJson`) более строгая,
но это другой путь кода (используется `--agents <json>` и
`settings.agents`).

| - | ----------------- | -------------------------------------- | ------------ | ------------ | ---------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------- |

условие выдачи DL7 — `...I !== void 0 && {skills: I}`, а `ml(undefined)`
возвращает `[]` (не undefined), поэтому **итоговая запись будет содержать
`skills: []` даже если frontmatter опускает это поле**. Это влияет на проверки
равенства в дальнейшем — отметить для порта qwen-code.

### Порядок разрешения — окончательный приоритет

Функция `DS()` фильтрует входные данные по `source`, затем итерирует
массив фиксированного порядка в `Map` с ключом по `agentType`. Поскольку
`Map.set` перезаписывает, **выигрывает последняя затронутая группа**:

```text
[built-in, plugin, userSettings, projectSettings, flagSettings, policySettings]
                                                                       ^
                                                                  высший приоритет
```

| Источник            | Происхождение                                                                                                                                                                             | Приоритет переопределения | Увер.                             |
| ------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------- | --------------------------------- |
| `built-in`          | `naH()` (жёстко закодировано в бинарнике)                                                                                                                                                 | 1 (самый низкий)          | **C** strings:309073              |
| `plugin`            | `b0_` → плагин `agentsPath`/`agentsPaths`                                                                                                                                                 | 2                         | **C** strings:308780              |
| `userSettings`      | `~/.claude/agents/` (`CLAUDE_CONFIG_DIR` или `~/.claude`)                                                                                                                                 | 3                         | **C** strings:312887, 307489      |
| `projectSettings`   | `<cwd>/.claude/agents/` ПЛЮС `iV_()` обход вверх до домашней директории / корня git                                                                                                        | 4                         | **C** strings:312887, iV\_ inline |
| `flagSettings`      | Флаг CLI `--agents <json>` (схема `qKO = h.record(h.string(), JL7())`)                                                                                                                    | 5                         | **C** strings:330190, 309076      |
| `policySettings`    | Системная директория: macOS `/Library/Application Support/ClaudeCode/.claude/agents`, Linux `/etc/claude-code/.claude/agents`, Windows `C:\Program Files\ClaudeCode\.claude\agents` | 6 (самый высокий)         | **C** strings:307649 (H2), 312887 |

Коллизии разрешаются **молча** — срабатывает только событие телеметрии
`tengu_plugin_name_collision` (`winner_source: T.at(-1)`); пользователю не
показывается предупреждение «X переопределяет встроенный». (strings:308742 `hMH`.)

Тонкое поведение: `iV_()` обходит сначала **самый внутренний** от `cwd` вверх,
но Map.set последним выигрывает, поэтому в projectSettings **внешнее дерево
`.claude/agents/` побеждает внутреннее**. Это неожиданно — отметить в открытых вопросах.

## Фаза 2 — План реализации для qwen-code

### Текущее состояние — краткая карта

qwen-code уже содержит значительную инфраструктуру сабагентов:
`SubagentManager` (`packages/core/src/subagents/subagent-manager.ts`) реализует
CRUD над markdown-файлами с YAML-фронтматером в `.qwen/agents/` (проект) и
`~/.qwen/agents/` (пользователь), используя собственный парсер YAML
(`packages/core/src/utils/yaml-parser.ts` — без зависимостей `gray-matter` / `yaml`,
подтверждено `package.json`). `SubagentConfig`
(`packages/core/src/subagents/types.ts:41-122`) уже содержит `name`,
`description`, `tools`, `disallowedTools`, `approvalMode`, `systemPrompt`,
`model`, `runConfig`, `color`, `background`. `SubagentLevel` уже поддерживает
пять областей (сессия, проект, пользователь, расширение, встроенный) с
приоритетом `session > project > user > extension > builtin`
(`subagent-manager.ts:189-220`). Инструмент Agent
(`packages/core/src/tools/agent/agent.ts`) объявляет `subagent_type` и
динамически обновляет свою перечислительную схему через `subagentManager.changeListener`.
Мост `convertClaudeAgentConfig()` уже существует в
`packages/core/src/extension/claude-converter.ts:162-220` с маппингом имён инструментов
и преобразованием `permissionMode → approvalMode`. **Пробел** заключается в: (a) схеме
не хватает 8 полей из #4821 (`effort`, `permissionMode` как первоклассный,
`mcpServers`, `hooks`, `maxTurns` как верхнеуровневый,
`skills`, `initialPrompt`, `memory`, `isolation`); (b) отсутствует флаг CLI
`--agent <name>`; (c) нет hot-перезагрузки в стиле chokidar (инвалидация как у
расширений существует, но не для файловых агентов); (d) `maxTurns` в настоящее время
находится в `runConfig.max_turns` — его нужно поднять на верхний уровень в соответствии с #2409.

### Архитектурные решения

#### D1. Повторное использование существующего yaml-parser для фронтматера

**Решение:** Повторно использовать `packages/core/src/utils/yaml-parser.ts` (уже используется
`SubagentManager.parseSubagentContent` и загрузчиком навыков).
**Обоснование:** `lz` в Claude Code — это тот же общий парсер, используемый для навыков +
команд + агентов; qwen-code уже копирует этот шаблон. Добавление `gray-matter`
или `js-yaml` — ненужный шум. Существующий парсер обрабатывает разделение `--- … ---`
и молча реагирует на некорректный ввод (соответствует поведению `lz`
`предупредить-и-вернуть-пустой`).

#### D2. Порядок разрешения / приоритетов

**Решение:** Использовать `session > project (.qwen/agents/) > user (~/.qwen/agents/)

> extension > builtin` — т.е. **сохранить существующий порядок qwen-code SubagentLevel,
не зеркалировать сегменты `flagSettings`/`policySettings` из Claude Code в v1**.
**Обоснование:** `policySettings` (управляемый каталог) в Claude Code — это корпоративный
сценарий развёртывания, которого у qwen-code нет. Агенты, внедрённые через флаг (`--agents <json>`),
— это функция для продвинутых пользователей, которая может появиться в P4. Существующий
пятиуровневый приоритет qwen-code уже покрывает случаи, которые важны для #4821:
проект переопределяет пользователя, пользователь переопределяет встроенных. Уровень
`extension` аккуратно встраивается между пользователем и
> встроенным.

#### D3. Валидация — сохранить существующий SubagentValidator

**Решение:** Расширить `SubagentValidator`
(`packages/core/src/subagents/`) для валидации восьми новых полей. **Не**
вводить zod, если его уже не использует конвейер skillManager; если существующий
валидатор написан вручную, оставить его ручным.
**Обоснование:** `Ig5` в Claude Code — это только тень — валидация во время выполнения
выполняется вручную через `DL7`. Соответствие этому шаблону сохраняет удобочитаемость
сообщений об ошибках (например, `Файл агента <путь> содержит недопустимый permissionMode '<x>'. Допустимые варианты: …`)
без добавления новой зависимости. Если skillManager уже использует zod — следовать этому
выбору для согласованности — TBD при чтении кода навыков в подготовке P1.

#### D4. Hot reload — отложить; полагаться на холодную загрузку + явную перезагрузку

**Решение:** v1 **НЕ** поставляет наблюдатель chokidar. Хуки инвалидации кэша
уже существуют (`subagentManager` имеет `changeListener` и явную CRUD-управляемую
перезагрузку). Перезагрузка на уровне проекта происходит при запуске сессии; правки
внутри сессии через UI `/agents` вызывают инвалидацию. Слеш-команда `/reload-agents`
(или добавление к `/reload-plugins`) может появиться в P4, если будет спрос.
**Обоснование:** Hot reload через FS-наблюдатель дорог (chokidar добавляет цикл опроса
с адаптивным планированием — только реализация Claude Code занимает ~150 строк
учёта). Холодная загрузка при запуске достаточна для v1 и соответствует тому, как
сейчас подключён `SubagentManager`. Открыть дверь для P4.

#### D5. Подключить флаг CLI `--agent <name>` — v1 входит в объём

**Решение:** Добавить `--agent <name>` в `packages/cli/src/config/config.ts`
CliArgs. Поведение: найти в разрешённом реестре, установить агента как
агента главного потока, выдать понятную ошибку, если имя не найдено. Соответствовать
семантике Claude Code (заменить системный промпт по умолчанию, если у агента нет
`appendSystemPrompt: true`). НЕ использовать косвенность через env-переменную
`CLAUDE_CODE_AGENT` — объект конфигурации `Config` в qwen-code может нести это напрямую.
**Обоснование:** Это пользовательская ручка для #4821 — без неё декларативные агенты
доступны только через параметр `subagent_type` инструмента Agent, что слишком
косвенно для случая «установить моего агента по умолчанию». `--agents <json>`
(множественное число) можно отложить до P4.

#### D6. Координация Workflow.agentType — контракт интерфейса

**Решение:** Предоставить стабильный интерфейс резолвера, который сможет вызвать
`createProductionDispatch` из PR #4732, когда он появится. Конкретно:

| Контракт                                                                                                                                                                                                                                                                                           | Владелец              |
| -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------- |
| Имя `name` из фронтматера ЯВЛЯЕТСЯ строкой `agentType` воркфлоу (равенство по ключу, чувствительно к регистру)                                                                                                                                                                                     | этот PR               |
| Жёстко заданный минимальный набор `disallowedTools` воркфлоу (`[SEND_MESSAGE, EXIT_PLAN_MODE]`, зеркалирован из upstream `Tg8`; подтверждено в PR #4732 как `ToolNames.SEND_MESSAGE`, `ToolNames.EXIT_PLAN_MODE`) **ОБЪЕДИНЯЕТСЯ** с агент-уровневым `disallowedTools` — минимум применяется всегда, даже если определение агента задаёт `tools` | потребляет PR воркфлоу |
| `opts.isolation` на один вызов переопределяет агентское значение по умолчанию `isolation: 'worktree'`                                                                                                                                                                                              | потребляет PR воркфлоу |
| `model`, `effort`, `permissionMode`, `maxTurns` из определения агента переопределяют значения по умолчанию воркфлоу, если заданы                                                                                                                                                                   | потребляет PR воркфлоу |
| Тело агента становится `systemPrompt` сабагента; `WORKFLOW_SUBAGENT_SYSTEM_PROMPT` воркфлоу является запасным, когда `agentType` не разрешается                                                                                                                                                     | потребляет PR воркфлоу |
| Когда `agentType` не задан или не удаётся разрешить, воркфлоу возвращается к встроенному сабагенту воркфлоу (корректно, без ошибки)                                                                                                                                                                | потребляет PR воркфлоу |

**Разрешение противоречия #4721 / #4821** (`tools` vs
`disallowedTools` приоритет): этот порт записывает реестр агентов таким образом, что
`disallowedTools` **всегда хранится отдельно** от `tools`. Правило «игнорируется,
если задан `tools`» из таблицы #4821 **применяется вызывающими сторонами инструмента Agent**
(т.е. при построении `ToolConfig` сабагента), а не во время разбора. Это позволяет
воркфлоу всегда объединять свой минимум с `disallowedTools`
независимо от того, задаёт ли агент `tools`. Реестр агентов — это
**пассивный носитель данных**; правила приоритета живут на стороне точки вызова. Это
разрешает кажущееся противоречие между правилом «игнорируется» из #4821 и правилом
«объединение» из #4721.

**Канонизация имён инструментов:** Использовать `ToolNames.SEND_MESSAGE` и
`ToolNames.EXIT_PLAN_MODE` (проверено по diff PR #4732), экспортированные как именованные константы из
`packages/core/src/agents/runtime/workflow-orchestrator.ts`, когда он появится. Самому
порту декларативных агентов **не нужно** импортировать их — это минимальный набор воркфлоу,
применяемый на стороне точки вызова воркфлоу.

### D7. permissionMode vs approvalMode — мост, а не замена

**Решение:** Принять ОБА поля: `permissionMode` (совместимый с Claude) и уже существующий
`approvalMode` (совместимый с qwen) во frontmatter. При разборе, если `permissionMode`

событие телеметрии вида `tengu_frontmatter_shadow_*`, отмечающее, что оба поля заданы.
**Обоснование:** Сохраняет обратную совместимость с существующими `.qwen/agents/*.md`,
использующими `approvalMode`, и при этом принимает `permissionMode` от Claude Code

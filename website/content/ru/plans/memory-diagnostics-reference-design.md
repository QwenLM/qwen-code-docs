# Эталонный проект диагностики памяти

## Контекст

Issue #3000 отслеживает диагностику памяти и производительности для длительных сессий Qwen Code. Первый PR должен создать небольшой диагностический инструментарий с низким риском, прежде чем добавлять более ресурсоемкое профилирование или изменения в удержании данных.

Дизайн ориентируется на эталонные реализации:

- Claude Code держит диагностику памяти отдельно от создания снимков кучи. Его диагностика включает память процесса, статистику кучи V8, пространства кучи, использование ресурсов, активные дескрипторы/запросы, файловые дескрипторы, Linux `smaps_rollup` и подсказки об утечках.
- Codex уделяет основное внимание ограниченному удержанию и ленивой загрузке для долгоживущего состояния процесса. Эти идеи должны направлять последующие PR, касающиеся удержания беседы, вывода команд и истории.

## Объём первого PR

Добавьте диагностический путь `/doctor memory`, который захватывает моментальный снимок в один момент времени:

- `process.memoryUsage()`
- Статистика кучи V8 и пространства кучи
- `process.resourceUsage()`
- Количество активных дескрипторов/запросов
- Количество открытых файловых дескрипторов, когда доступен `/proc/self/fd`
- Linux `smaps_rollup`, когда доступен
- Базовые подсказки о рисках: давление на кучу, отсоединённые контексты, избыточные дескрипторы, избыточные запросы, высокое количество файловых дескрипторов и давление на нативную память

Эта команда должна быть достаточно легковесной для выполнения в обычных сессиях и безопасной на платформах, где недоступны Linux-специфичные зонды.

## Вне целей

Этот PR намеренно не:

- создаёт снимки кучи
- выполняет непрерывный опрос
- изменяет удержание подсказок/истории
- изменяет удержание вывода инструментов
- изменяет поведение загрузки модулей

Это будут последующие PR после создания диагностического базового уровня.

## Последующие PR

1. Добавить явную поддержку снимков/экспорта для более глубокого локального исследования.
2. Добавить ограниченное удержание для больших выводов команд/инструментов, используя ограниченное удержание вывода Codex в качестве основного эталона.
3. Провести аудит ленивой загрузки и путей запуска модулей после того, как измерения выявят горячие точки.
4. Добавить повторяемые сценарии бенчмарков памяти/производительности для длительных сессий.

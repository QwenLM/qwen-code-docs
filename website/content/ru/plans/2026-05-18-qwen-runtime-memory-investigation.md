# План исследования памяти Qwen Code во время выполнения

Дата: 2026-05-18

## Контекст

Локальные бенчмарки показывают, что Qwen Code использует существенно больше RSS дерева процессов, чем
Claude Code, при аналогичных неинтерактивных задачах CLI. В последней матрице из пяти кейсов
Qwen Code достигал пика около `0.83-1.04 GiB`, тогда как Claude Code оставался в районе
`0.27-0.36 GiB`.

Данный документ предлагает проект направления исследования и оптимизации. Он
не претендует на установление окончательной первопричины. Ближайшая цель — сделать разрыв в памяти
обозримым, воспроизводимым и объяснимым с помощью внутренней диагностики.

## Текущий прогресс

Исследование достигло стадии сбора доказательств и определения направления:

- Построена повторяемая локальная матрица для небольших ревью PR, навигации по коду
  и синтетических рабочих нагрузок с диффами.
- Qwen Code сравнен на нескольких моделях.
- Qwen Code и Claude Code сравнены на одинаковых типах задач там, где были доступны
  эквивалентные endpoints моделей.
- Наблюдаемый разрыв RSS достаточно стабилен, чтобы оправдать углублённую диагностику времени выполнения.
- Связанные upstream работы зафиксированы, чтобы данная работа могла опираться на существующие
  `/doctor memory` и последующие диагностики памяти.

Исследование пока не достигло стадии окончательной первопричины, потому что
внешний RSS процесса не может показать, является ли удерживаемая память V8-кучей, нативной
памятью, загруженными модулями, живой историей, результатами инструментов или состоянием сборки запроса.

## Текущие доказательства

Сопроводительный отчёт о бенчмарках:

- `docs/e2e-tests/2026-05-18-qwen-memory-benchmark-report.md`

Основные доказательства:

- Разрыв RSS между Qwen и Claude воспроизводится в небольших ревью PR, навигации по коду
  и синтетических рабочих нагрузках с диффами.
- Разрыв воспроизводится как с `pai/glm-5`, так и с `qwen3.6-plus`.
- Qwen Code использовал больше токенов, чем Claude Code, во всех ячейках протестированной матрицы.
- Большой размер диффа не приводил к чистому линейному росту памяти, что предполагает:
  базовый и ограниченный/усечённый пути вывода имеют большее значение, чем просто необработанные байты диффа.

Открытые PR со смягчениями могут уменьшить отдельные OOM-пути, но они пока не объясняют, почему короткие неинтерактивные задачи CLI снова и снова достигают пика около `1 GiB`.

## Почему этот проект начинается с документации

Этот проект намеренно начинается с доказательств из бенчмарков и плана исследования, а не объединяет изменения runtime-кода.

Причины:

1. Текущая цель — сделать проблему производительности и направление видимыми, а не заявлять об исправлении «за один день».
2. Добавление инструментария и оптимизации в один и тот же PR усложнит ревью, поскольку смешивает измерения, диагностику и изменения поведения.
3. Существующий бенчмарк уже подтверждает необходимость более глубокой диагностики.
4. Следующий PR может быть более узким и простым для проверки: только диагностика, затем повторный запуск той же матрицы и сравнение внутренних метрик.

Следующий PR с реализацией должен добавить недостающие счётчики и точки временной шкалы, затем повторно запустить матрицу бенчмарков. Только после этого целенаправленный PR с оптимизацией должен попытаться уменьшить память.

## Рабочая гипотеза

Текущие данные указывают скорее на проблему runtime/пути Qwen Code, чем на проблему провайдера модели.

Сильнейшая текущая гипотеза такова:

> Qwen Code, по-видимому, несёт высокий footprint выполнения неинтерактивных задач CLI,
> вероятно, усиленный более масштабной обработкой контекста/результатов инструментов/сессии.
> Наиболее вероятная проблемная область — это runtime CLI и путь данных агента, а не только
> выбранная модель.

Более конкретно, данные указывают против «слишком много вызовов инструментов» как основной причины. Количество вызовов инструментов было схожим в разных CLI, и Claude иногда использовал больше ходов или вызовов инструментов, сохраняя при этом более низкий RSS. Более правдоподобная проблема в том, что Qwen Code инициализирует или удерживает более тяжёлое состояние для той же короткой неинтерактивной задачи CLI, а затем усиливает этот footprint выполнения за счёт более крупных контекста, результатов инструментов, сохранённого вывода или данных истории сессии.

Наиболее вероятные категории:

1. **Стоимость запуска и выполнения процесса/модуля**: Qwen Code может инициализировать больше runtime, инструментов, инфраструктуры UI/сессии или механизмов провайдера, чем необходимо для неинтерактивных задач CLI.
2. **Сборка истории и контекста**: Qwen Code может удерживать или конструировать больший контекст для модели, чем Claude Code, для одного и того же типа задачи.
3. **Удержание результатов инструментов**: большие или повторяющиеся результаты инструментов могут удерживаться в живой истории, истории UI, записи чата или путях восстановления сохранённого вывода.
4. **Усиление от под-агентов и сохранённого вывода**: предыдущие тесты с большими PR показали восстановление сохранённого вывода и активность под-агентов, что может добавлять давление на память и токены.
5. **Дочерние процессы MCP**: сопроводительный диагностический отчёт показал, что MCP-серверы (например, chrome-devtools) вносят ~350 MiB в RSS дерева процессов. Это раздувает абсолютные числа, но является постоянными накладными расходами, не связанными с длиной сессии.
6. **Разделение нативной памяти и JS-кучи**: внешний RSS не может показать, является ли давление V8-кучей, нативными буферами, загруженными модулями или удерживаемыми данными.

Это намеренно сформулировано как гипотеза. Следующий шаг — добавить достаточно внутренних измерений, чтобы подтвердить или исключить каждую категорию.

## Предлагаемый объём первого PR

Первый PR должен быть сосредоточен на доказательствах и диагностике:

1. Закоммитить отчёт о бенчмарках и план исследования.
2. Добавить или расширить локальный диагностический вывод, чтобы Qwen Code мог сообщать:
   - Статистику V8-кучи и пространств кучи.
   - Разделение RSS и кучи.
   - Количество сообщений сессии и приблизительный удерживаемый размер.
   - Количество результатов инструментов, общий удерживаемый размер и размер самого большого удерживаемого результата.
   - Счётчики усечения и восстановления сохранённого вывода.
   - Активность под-агентов/дерева процессов, если доступно.
3. Повторно запустить существующую матрицу для:
   - текущего опубликованного Qwen Code,
   - текущего `main`,
   - ветки только с диагностикой,
   - ветки-кандидата на оптимизацию.
4. Использовать эти измерения для выбора одной небольшой цели оптимизации.

Первый PR должен избегать смешивания нескольких несвязанных оптимизаций. Он должен либо оставаться только документацией, либо добавлять код только для диагностики. Отдельный PR должен нести первое runtime-снижение памяти, когда причина станет яснее.

## Кандидатные направления оптимизации

Это кандидаты, а не выводы:

1. **Ограниченное удержание вывода инструментов**: хранить большой вывод вне горячего пути и сохранять только превью, метаданные и указатели для извлечения в живой истории.
2. **Ленивая загрузка для неинтерактивного режима**: избегать инициализации подсистем, предназначенных только для TUI или интерактивного режима, во время выполнения неинтерактивных задач CLI.
3. **Ограничение истории сессии/UI**: деградировать старые или тяжёлые элементы истории в компактные записи транскрипта.
4. **Учёт сборки контекста**: измерять и ограничивать большие результаты инструментов до построения запроса к модели.
5. **Учёт под-агентов**: раскрывать жизненный цикл под-агентов и влияние на память в диагностике.

Claude Code и OpenAI Codex (CLI-агент для кодирования от OpenAI) следует использовать как референсы по дизайну для разделения диагностики, ограниченного удержания вывода и ленивой загрузки истории. Реализация должна по-прежнему следовать собственной архитектуре и тестам Qwen Code.

## План валидации

Исследование должно сохранять ту же матрицу бенчмарков, чтобы результаты до/после оставались сравнимыми:

- небольшое ревью PR
- навигация по коду
- синтетический diff около 100 KiB
- синтетический diff около 1 MiB
- синтетический diff около 5 MiB

Для каждого запуска записывать:

- пик RSS дерева процессов
- пик RSS корневого процесса
- пик V8-кучи
- сводка пространств кучи
- длительность
- количество ходов
- количество токенов
- количество вызовов инструментов
- самый большой удерживаемый результат инструмента
- общий размер удерживаемых результатов инструментов
- количество элементов сессии/истории
- количество под-агентов

Минимальное условие успеха для кандидатного исправления — не просто «RSS снизился». Должно быть также понятно, какая внутренняя метрика изменилась и почему.

## Следующий кандидат на PR

Следующий PR должен быть только диагностическим и избегать изменения runtime-поведения. Минимальный полезный срез добавит:

- учёт входного размера запроса к модели;
- учёт размера системного промпта и схемы инструментов;
- количество удерживаемых сообщений и приблизительный размер в символах;
- количество удерживаемых результатов инструментов, общий размер и размер самого большого элемента;
- образцы жизненного цикла вокруг запуска, сборки первого запроса, выполнения инструмента, стримингового завершения, сжатия и финального ответа;
- образцы памяти процесса, включающие RSS, используемую кучу, общий размер кучи, внешнюю память и статистику пространств кучи.

После того как это будет реализовано локально, повторно запустить ту же матрицу моделей Qwen и сравнить:

- опубликованный Qwen Code;
- текущий `main`;
- ветку только с диагностикой;
- ветку-кандидата на оптимизацию.

## Что не входит в цели

Данный проект не утверждает, что:

- всё давление на память вызвано выводом инструментов;
- один существующий открытый PR решит наблюдаемый footprint во время задачи;
- различия провайдеров моделей не имеют значения в любом окружении;
- однократные локальные измерения достаточны для утверждений о производительности на уровне релиза.

Заявляемая цель более узкая: Qwen Code демонстрирует стабильный локальный разрыв RSS в протестированных рабочих нагрузках, и проекту нужна внутренняя диагностика, чтобы объяснить и уменьшить этот разрыв.

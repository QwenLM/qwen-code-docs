# Выпуск расширений

Существует два основных способа распространения расширений среди пользователей:

- [Репозиторий Git](#выпуск-через-репозиторий-git)
- [Выпуски GitHub](#выпуск-через-выпуски-github)

Выпуск через репозиторий Git, как правило, является самым простым и гибким подходом. В то же время выпуски GitHub могут быть эффективнее при первоначальной установке, поскольку они поставляются в виде единого архива, а не требуют клонирования репозитория Git, при котором каждый файл загружается отдельно. Выпуски GitHub также могут содержать архивы, специфичные для определённой платформы, если вам необходимо распространять бинарные файлы, предназначенные исключительно для конкретной платформы.

## Выпуск через репозиторий Git

Это наиболее гибкий и простой вариант. Вам нужно лишь создать общедоступный репозиторий Git (например, публичный репозиторий на GitHub), после чего пользователи смогут устанавливать ваше расширение с помощью команды `qwen extensions install <URI-вашего-репозитория>`; для репозиториев GitHub также поддерживается упрощённый формат: `qwen extensions install <организация>/<репозиторий>`. При необходимости пользователи могут указать конкретную ревизию (ветку, тег или коммит), добавив аргумент `--ref=<некоторая-ревизия>`; по умолчанию используется основная ветка.

Каждый раз, когда в зависимую от пользователя ревизию отправляются новые коммиты, ему будет предложено обновить расширение. Обратите внимание, что такая схема также позволяет легко выполнять откаты: последний коммит (`HEAD`) всегда считается самой новой версией, независимо от значения поля `version` в файле `qwen-extension.json`.

### Управление каналами релизов с помощью репозитория Git

Пользователи могут зависеть от любой ссылки (ref) из вашего репозитория Git — например, ветки или тега. Это позволяет управлять несколькими каналами релизов.

Например, вы можете поддерживать ветку `stable`, которую пользователи смогут установить так:  
`qwen extensions install <ваш-URI-репозитория> --ref=stable`.  
Или вы можете сделать её веткой по умолчанию, используя основную ветку репозитория как ветку стабильных релизов, а разработку вести в отдельной ветке (например, `dev`). Вы можете создавать и поддерживать любое количество веток и тегов — это обеспечивает максимальную гибкость как для вас, так и для ваших пользователей.

Обратите внимание: аргументы `ref` могут быть тегами, ветками или даже конкретными коммитами. Это позволяет пользователям зависеть от определённой версии вашего расширения. Как именно управлять тегами и ветками — решать вам.

### Пример процесса выпуска версий с использованием Git-репозитория

Хотя существует множество способов управления выпусками версий с помощью Git-потока, мы рекомендуем рассматривать основную ветку как ветку «стабильных» релизов. Это означает, что поведение по умолчанию для команды `qwen extensions install <your-repo-uri>` — установка из ветки стабильного релиза.

Допустим, вы хотите поддерживать три стандартных канала выпуска: `stable`, `preview` и `dev`. Всё обычное развитие вы выполняете в ветке `dev`. Когда вы готовы выпустить предварительную версию, вы объединяете эту ветку с веткой `preview`. Когда вы решите перевести предварительную версию в стабильную, вы объединяете ветку `preview` с веткой `stable` (которая может совпадать с основной веткой или быть отдельной веткой).

Также вы можете использовать команду `git cherry-pick`, чтобы переносить отдельные коммиты из одной ветки в другую. Однако имейте в виду: такой подход приводит к небольшому расхождению истории между ветками, если вы не принудительно обновляете ветки (`force push`) при каждом выпуске, чтобы восстановить чистую историю (что может быть невозможно для основной ветки в зависимости от настроек репозитория). Если вы планируете активно использовать `cherry-pick`, возможно, стоит не назначать основную ветку веткой `stable`, чтобы избежать принудительного обновления основной ветки — этого обычно следует избегать.

## Релизы через GitHub Releases

Расширения Qwen Code можно распространять через [GitHub Releases](https://docs.github.com/ru/repositories/releasing-projects-on-github/about-releases). Это обеспечивает более быстрый и надёжный первоначальный опыт установки для пользователей, поскольку исключает необходимость клонирования репозитория.

Каждый релиз включает как минимум один архивный файл, содержащий полное содержимое репозитория на момент тега, с которым он связан. В релизы также могут входить [предварительно собранные архивы](#custom-pre-built-archives), если ваше расширение требует этапа сборки или содержит бинарные файлы, специфичные для определённой платформы.

При проверке обновлений Qwen Code просто ищет последний релиз на GitHub (вы должны явно отметить его как «последний» при создании релиза), за исключением случаев, когда пользователь установил конкретный релиз, указав флаг `--ref=<some-release-tag>`. На данный момент не поддерживается автоматическое включение предварительных релизов (pre-release) или семантическое версионирование (semver).

### Пользовательские предварительно собранные архивы

Пользовательские архивы должны быть прикреплены непосредственно к релизу GitHub в качестве ассетов и должны быть полностью автономными. Это означает, что они должны содержать расширение целиком — см. раздел [структура архива](#archive-structure).

Если ваше расширение не зависит от платформы, вы можете предоставить один общий ассет. В этом случае к релизу должен быть прикреплён только один ассет.

Пользовательские архивы также могут использоваться, если вы разрабатываете своё расширение в рамках более крупного репозитория: вы можете собрать архив с иной структурой по сравнению с самим репозиторием (например, это может быть архив лишь подкаталога, содержащего расширение).

#### Архивы, специфичные для платформы

Чтобы Qwen Code мог автоматически находить правильный артефакт релиза для каждой платформы, необходимо соблюдать следующие правила именования. CLI будет искать артефакты в указанном ниже порядке:

1.  **Специфичные для платформы и архитектуры:** `{platform}.{arch}.{name}.{extension}`
2.  **Специфичные для платформы:** `{platform}.{name}.{extension}`
3.  **Универсальные:** Если предоставлен только один артефакт, он будет использоваться как универсальный резервный вариант.

- `{name}`: Имя вашего расширения.
- `{platform}`: Операционная система. Поддерживаемые значения:
  - `darwin` (macOS)
  - `linux`
  - `win32` (Windows)
- `{arch}`: Архитектура. Поддерживаемые значения:
  - `x64`
  - `arm64`
- `{extension}`: Расширение архива (например, `.tar.gz` или `.zip`).

**Примеры:**

- `darwin.arm64.my-tool.tar.gz` (специфично для Mac на процессорах Apple Silicon)
- `darwin.my-tool.tar.gz` (для всех Mac)
- `linux.x64.my-tool.tar.gz`
- `win32.my-tool.zip`

#### Структура архива

Архивы должны содержать полностью автономные расширения и удовлетворять всем стандартным требованиям — в частности, файл `qwen-extension.json` должен находиться в корне архива.

Остальная структура должна точно соответствовать типичному расширению; см. [extensions.md](extension.md).

#### Пример рабочего процесса GitHub Actions

Ниже приведён пример рабочего процесса GitHub Actions, который собирает и публикует расширение Qwen Code для нескольких платформ:

```yaml
name: Release Extension

on:
  push:
    tags:
      - 'v*'

jobs:
  release:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3

      - name: Set up Node.js
        uses: actions/setup-node@v3
        with:
          node-version: '20'

      - name: Install dependencies
        run: npm ci

      - name: Build extension
        run: npm run build

      - name: Create release assets
        run: |
          npm run package -- --platform=darwin --arch=arm64
          npm run package -- --platform=linux --arch=x64
          npm run package -- --platform=win32 --arch=x64

      - name: Create GitHub Release
        uses: softprops/action-gh-release@v1
        with:
          files: |
            release/darwin.arm64.my-tool.tar.gz
            release/linux.arm64.my-tool.tar.gz
            release/win32.arm64.my-tool.zip
```

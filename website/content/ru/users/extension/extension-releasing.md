# Публикация расширений

Существует три основных способа публикации расширений для пользователей:

- [Git-репозиторий](#публикация-через-git-репозиторий)
- [Релизы GitHub](#публикация-через-релизы-github)
- [Реестр npm](#публикация-через-реестр-npm)

Релизы через Git-репозиторий, как правило, являются самым простым и гибким подходом, в то время как релизы GitHub могут быть более эффективными при первоначальной установке, поскольку они поставляются в виде единых архивов, а не требуют клонирования репозитория, при котором каждый файл загружается отдельно. Релизы GitHub также могут содержать архивы, специфичные для платформы, если вам необходимо поставлять бинарные файлы, зависимые от платформы. Релизы через реестр npm идеально подходят для команд, которые уже используют npm для распространения пакетов, особенно с частными реестрами.

## Публикация через Git-репозиторий

Это самый гибкий и простой вариант. Всё, что вам нужно сделать, — это создать общедоступный Git-репозиторий (например, публичный репозиторий GitHub), после чего пользователи смогут установить ваше расширение с помощью `qwen extensions install <ваш-uri-репозитория>` или, для репозитория GitHub, с помощью упрощённого формата `qwen extensions install <организация>/<репозиторий>`. При желании они могут указать конкретный реф (ветку/тег/коммит) с помощью аргумента `--ref=<какой-то-реф>`; по умолчанию используется ветка по умолчанию.

Когда в реф, от которого зависит пользователь, отправляются новые коммиты, ему будет предложено обновить расширение. Обратите внимание, что это также позволяет легко выполнять откаты: коммит HEAD всегда считается последней версией независимо от фактической версии в файле `qwen-extension.json`.

### Управление каналами релизов с помощью Git-репозитория

Пользователи могут зависеть от любого рефа из вашего Git-репозитория, например, ветки или тега, что позволяет вам управлять несколькими каналами релизов.

Например, вы можете поддерживать ветку `stable`, которую пользователи могут установить так: `qwen extensions install <ваш-uri-репозитория> --ref=stable`. Или вы можете сделать это поведением по умолчанию, рассматривая вашу ветку по умолчанию как стабильную ветку релизов, а разработку вести в другой ветке (например, под названием `dev`). Вы можете поддерживать любое количество веток или тегов, обеспечивая максимальную гибкость для себя и своих пользователей.

Обратите внимание, что эти аргументы `ref` могут быть тегами, ветками или даже конкретными коммитами, что позволяет пользователям зависеть от конкретной версии вашего расширения. Решение о том, как управлять тегами и ветками, остаётся за вами.

### Пример процесса публикации с использованием Git-репозитория

Хотя существует множество вариантов управления релизами с помощью Git-процесса, мы рекомендуем рассматривать вашу ветку по умолчанию как «стабильную» ветку релизов. Это означает, что поведение по умолчанию для `qwen extensions install <ваш-uri-репозитория>` — находиться на стабильной ветке релизов.

Допустим, вы хотите поддерживать три стандартных канала релизов: `stable`, `preview` и `dev`. Всю обычную разработку вы ведёте в ветке `dev`. Когда вы будете готовы к предварительному релизу, вы сливаете эту ветку в ветку `preview`. Когда вы будете готовы продвинуть ветку предварительного релиза до стабильной, вы сливаете `preview` в вашу стабильную ветку (которая может быть вашей веткой по умолчанию или другой веткой).

Вы также можете выбирать отдельные изменения из одной ветки в другую с помощью `git cherry-pick`, но учтите, что это приведёт к тому, что ваши ветки будут иметь немного расходящуюся историю, если только вы не будете принудительно отправлять изменения в свои ветки при каждом релизе, чтобы восстановить чистую историю (что может быть невозможно для ветки по умолчанию в зависимости от настроек вашего репозитория). Если вы планируете использовать cherry-pick, возможно, вам стоит избегать того, чтобы ваша ветка по умолчанию была стабильной веткой, чтобы избежать принудительной отправки в ветку по умолчанию, чего обычно следует избегать.

## Публикация через релизы GitHub

Расширения Qwen Code можно распространять через [релизы GitHub](https://docs.github.com/en/repositories/releasing-projects-on-github/about-releases). Это обеспечивает более быстрый и надёжный процесс первоначальной установки для пользователей, поскольку позволяет избежать необходимости клонировать репозиторий.

Каждый релиз включает как минимум один архивный файл, содержащий полное содержимое репозитория на момент тега, к которому он привязан. Релизы также могут включать [собранные заранее архивы](#собранные-заранее-архивы), если ваше расширение требует этапа сборки или содержит бинарные файлы, специфичные для платформы.

При проверке обновлений Qwen Code просто ищет последний релиз на GitHub (вы должны пометить его как таковой при создании релиза), если только пользователь не установил конкретный релиз, передав `--ref=<какой-то-тег-релиза>`. В настоящее время мы не поддерживаем выбор предварительных релизов или семантического версионирования (semver).

### Собранные заранее архивы

Собранные заранее архивы должны быть прикреплены непосредственно к релизу GitHub в качестве ассетов и должны быть полностью самодостаточными. Это означает, что они должны включать всё расширение целиком, см. [структура архива](#структура-архива).

Если ваше расширение не зависит от платформы, вы можете предоставить один generic-ассет. В этом случае к релизу должен быть прикреплён только один ассет.

Собранные заранее архивы также могут использоваться, если вы хотите разрабатывать расширение в рамках более крупного репозитория: вы можете собрать архив, структура которого отличается от самого репозитория (например, это может быть просто архив поддиректории, содержащей расширение).
#### Архивы для конкретных платформ

Чтобы Qwen Code мог автоматически найти правильный файл релиза для каждой платформы, вы должны следовать этому соглашению об именовании. CLI будет искать файлы в следующем порядке:

1.  **Специфичные для платформы и архитектуры:** `{platform}.{arch}.{name}.{extension}`
2.  **Специфичные для платформы:** `{platform}.{name}.{extension}`
3.  **Универсальные:** Если предоставлен только один файл, он будет использован как универсальный запасной вариант.

- `{name}`: Имя вашего расширения.
- `{platform}`: Операционная система. Допустимые значения:
  - `darwin` (macOS)
  - `linux`
  - `win32` (Windows)
- `{arch}`: Архитектура. Допустимые значения:
  - `x64`
  - `arm64`
- `{extension}`: Расширение файла архива (например, `.tar.gz` или `.zip`).

**Примеры:**

- `darwin.arm64.my-tool.tar.gz` (специфично для Mac на Apple Silicon)
- `darwin.my-tool.tar.gz` (для всех Mac)
- `linux.x64.my-tool.tar.gz`
- `win32.my-tool.zip`

#### Структура архива

Архивы должны содержать полностью самодостаточные расширения и соответствовать всем стандартным требованиям — в частности, файл `qwen-extension.json` должен находиться в корне архива.

Остальная структура должна выглядеть точно так же, как у типичного расширения; см. [introduction.md](./introduction.md).

#### Пример рабочего процесса GitHub Actions

Вот пример рабочего процесса GitHub Actions, который собирает и выпускает расширение Qwen Code для нескольких платформ:

```yaml
name: Release Extension

on:
  push:
    tags:
      - 'v*'

jobs:
  release:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3

      - name: Set up Node.js
        uses: actions/setup-node@v3
        with:
          node-version: '22'

      - name: Install dependencies
        run: npm ci

      - name: Build extension
        run: npm run build

      - name: Create release assets
        run: |
          npm run package -- --platform=darwin --arch=arm64
          npm run package -- --platform=linux --arch=x64
          npm run package -- --platform=win32 --arch=x64

      - name: Create GitHub Release
        uses: softprops/action-gh-release@v1
        with:
          files: |
            release/darwin.arm64.my-tool.tar.gz
            release/linux.arm64.my-tool.tar.gz
            release/win32.arm64.my-tool.zip
```

## Публикация через npm registry

Вы можете публиковать расширения Qwen Code как scoped npm-пакеты (например, `@your-org/my-extension`). Это хороший выбор, когда:

- Ваша команда уже использует npm для распространения пакетов
- Вам требуется поддержка частного реестра с существующей инфраструктурой аутентификации
- Вы хотите, чтобы разрешение версий и контроль доступа обрабатывались npm

### Требования к пакету

Ваш npm-пакет должен содержать файл `qwen-extension.json` в корне пакета. Это тот же конфигурационный файл, который используется всеми расширениями Qwen Code — npm-тарбол — это просто ещё один механизм доставки.

Минимальная структура пакета выглядит так:

```
my-extension/
├── package.json
├── qwen-extension.json
├── QWEN.md              # необязательный файл контекста
├── commands/             # необязательные пользовательские команды
├── skills/               # необязательные пользовательские навыки
└── agents/               # необязательные пользовательские подагенты
```

Убедитесь, что `qwen-extension.json` включён в опубликованный пакет (т.е. не исключён через `.npmignore` или поле `files` в `package.json`).

### Публикация

Используйте стандартные инструменты публикации npm:

```bash
# Публикация в реестр по умолчанию
npm publish

# Публикация в частный/пользовательский реестр
npm publish --registry https://your-registry.com
```

### Установка

Пользователи устанавливают ваше расширение, используя имя scoped-пакета:

```bash
# Установка последней версии
qwen extensions install @your-org/my-extension

# Установка конкретной версии
qwen extensions install @your-org/my-extension@1.2.0

# Установка из пользовательского реестра
qwen extensions install @your-org/my-extension --registry https://your-registry.com
```

### Поведение при обновлении

- Расширения, установленные без фиксации версии (например, `@scope/pkg`), отслеживают dist-tag `latest`.
- Расширения, установленные с dist-tag (например, `@scope/pkg@beta`), отслеживают указанный тег.
- Расширения, привязанные к точной версии (например, `@scope/pkg@1.2.0`), всегда считаются актуальными и не будут предлагать обновления.

### Аутентификация для частных реестров

Qwen Code автоматически считывает учётные данные npm:

1. **Переменная окружения `NPM_TOKEN`** — наивысший приоритет
2. **Файл `.npmrc`** — поддерживает как уровне хоста, так и уровне пути записи `_authToken` (например, `//your-registry.com/:_authToken=TOKEN` или `//pkgs.dev.azure.com/org/_packaging/feed/npm/registry/:_authToken=TOKEN`)

Файлы `.npmrc` считываются из текущей директории и домашней директории пользователя.

### Управление каналами релизов

Вы можете использовать npm dist-tags для управления каналами релизов:

```bash
# Публикация бета-релиза
npm publish --tag beta

# Пользователи устанавливают бета-канал
qwen extensions install @your-org/my-extension@beta
```

Это работает аналогично каналам релизов на основе веток git, но использует собственный механизм dist-tag в npm.

# Стандартные рабочие процессы

> Узнайте о стандартных рабочих процессах с Qwen Code.

Каждая задача в этом документе включает чёткие инструкции, примеры команд и лучшие практики, которые помогут вам получить максимум от Qwen Code.

## Быстрое знакомство с новыми кодовыми базами

### Получение общего обзора кодовой базы

Предположим, вы только что присоединились к новому проекту и хотите быстро понять его структуру.

**1. Перейдите в корневую директорию проекта**

```bash
cd /path/to/project
```

**2. Запустите Qwen Code**

```bash
qwen
```

**3. Запросите общий обзор**

```
give me an overview of this codebase
```

**4. Углубитесь в конкретные компоненты**

```
explain the main architecture patterns used here
```

```
what are the key data models?
```

```
how is authentication handled?
```

> [!tip]
>
> - Начинайте с широких вопросов, затем сужайтесь до конкретных областей
> - Спрашивайте о соглашениях по коду и паттернах, используемых в проекте
> - Запросите глоссарий терминов, специфичных для проекта

### Поиск релевантного кода

Предположим, вам нужно найти код, связанный с определённой функцией или возможностью.

**1. Попросите Qwen Code найти соответствующие файлы**

```
find the files that handle user authentication
```

**2. Получите контекст о том, как компоненты взаимодействуют**

```
how do these authentication files work together?
```

**3. Поймите поток выполнения**

```
trace the login process from front-end to database
```

> [!tip]
>
> - Будьте конкретны в том, что ищете
> - Используйте терминологию проекта

## Использование специализированных под-агентов

Предположим, вы хотите использовать специализированные AI-под-агенты для более эффективного выполнения конкретных задач.

**1. Просмотрите доступных под-агентов**

```
/agents
```

Эта команда показывает всех доступных под-агентов и позволяет создавать новых.

**2. Используйте под-агентов автоматически**

Qwen Code автоматически делегирует подходящие задачи специализированным под-агентам:

```
review my recent code changes for security issues
```

```
run all tests and fix any failures
```

**3. Явно запросите конкретных под-агентов**

```
use the code-reviewer subagent to check the auth module
```

```
have the debugger subagent investigate why users can't log in
```

**4. Создайте собственных под-агентов для вашего рабочего процесса**

```
/agents
```

Затем выберите «create» и следуйте подсказкам, чтобы определить:

- Уникальный идентификатор, описывающий назначение под-агента (например, `code-reviewer`, `api-designer`).
- Когда Qwen Code должен использовать этого агента
- Какие инструменты ему доступны
- Системный промпт, описывающий роль и поведение агента

> [!tip]
>
> - Создавайте под-агенты для конкретного проекта в `.qwen/agents/` для обмена с командой
> - Используйте описательные поля `description`, чтобы включить автоматическое делегирование
> - Ограничивайте доступ к инструментам тем, что действительно нужно каждому под-агенту
> - Узнайте больше о [Под-агентах](./features/sub-agents)
> - Узнайте больше о [Режиме одобрения](./features/approval-mode)

**1. Определите непроверенный код**

```

```

**2. Сгенерируйте каркас тестов**

```

```

**3. Добавьте значимые тестовые сценарии**

```

```

**4. Запустите и проверьте тесты**

```

```

Qwen Code может генерировать тесты, которые следуют существующим паттернам и соглашениям вашего проекта. При запросе тестов уточняйте, какое поведение вы хотите проверить. Qwen Code изучает ваши существующие тестовые файлы, чтобы соответствовать стилю, фреймворкам и паттернам утверждений, уже используемым в проекте.

Для полного покрытия попросите Qwen Code определить граничные случаи, которые вы могли упустить. Qwen Code может проанализировать ваши пути выполнения и предложить тесты для ошибочных ситуаций, граничных значений и неожиданных входных данных, которые легко не заметить.

## Ссылки на файлы и директории

Используйте `@`, чтобы быстро включать файлы или директории, не дожидаясь, пока Qwen Code прочитает их.

**1. Ссылка на один файл**

```
Explain the logic in @src/utils/auth.js
```

Это включает полное содержимое файла в диалог.

**2. Ссылка на директорию**

```
What's the structure of @src/components?
```

Это предоставляет список файлов с информацией о них.

**3. Ссылка на ресурсы MCP**

```
Show me the data from @github: repos/owner/repo/issues
```

Это получает данные от подключенных MCP-серверов в формате @server: resource. Подробнее см. [MCP](./features/mcp).

> [!tip]
>
> - Пути к файлам могут быть относительными или абсолютными
> - Ссылки на файлы через @ добавляют `QWEN.md` в директории файла и её родительских директориях в контекст
> - Ссылки на директории показывают список файлов, а не их содержимое
> - Вы можете ссылаться на несколько файлов в одном сообщении (например, "`@file1.js` и `@file2.js`")

## Возобновление предыдущих диалогов

Предположим, вы работали над задачей с Qwen Code и хотите продолжить с того места, где остановились, в следующем сеансе.

Qwen Code предоставляет два варианта для возобновления предыдущих диалогов:

- `--continue` для автоматического продолжения самого последнего диалога
- `--resume` для отображения выбора диалога

**1. Продолжить самый последний диалог**

```bash
qwen --continue
```

Это сразу же возобновляет ваш последний диалог без дополнительных запросов.

**2. Продолжить в неинтерактивном режиме**

```bash
qwen --continue -p "Continue with my task"
```

Используйте `-p` (или `--prompt`) вместе с `--continue`, чтобы возобновить последний диалог в неинтерактивном режиме — идеально для скриптов или автоматизации.

**3. Показать выбор диалога**

```bash
qwen --resume
```

Отображает интерактивный выбор диалога в виде чистого списка, показывающего:

- Краткое описание сессии (или начальный промпт)
- Метаданные: прошедшее время, количество сообщений и ветка git

Используйте стрелки для навигации и нажмите Enter, чтобы выбрать диалог. Нажмите Esc для выхода.

> [!tip]
>
> - История диалогов хранится локально на вашей машине
> - Используйте `--continue` для быстрого доступа к самому последнему диалогу
> - Используйте `--resume`, когда нужно выбрать конкретный прошлый диалог
> - При возобновлении вы увидите всю историю диалога перед продолжением
> - Возобновлённый диалог начинается с той же модели и конфигурации, что и исходный
>
> **Как это работает**:
>
> 1. **Хранение диалогов**: Все диалоги автоматически сохраняются локально с полной историей сообщений.
> 2. **Десериализация сообщений**: При возобновлении восстанавливается вся история сообщений для сохранения контекста.
> 3. **Состояние инструментов**: Использование инструментов и результаты из предыдущего диалога сохраняются.
> 4. **Восстановление контекста**: Диалог возобновляется со всем предыдущим контекстом.
>
> **Примеры**:
>
> ```bash
> # Продолжить последний диалог
> qwen --continue
>
> # Продолжить последний диалог с конкретным промптом
> qwen --continue -p "Show me our progress"
>
> # Показать выбор диалога
> qwen --resume
>
> # Продолжить последний диалог в неинтерактивном режиме
> qwen --continue -p "Run the tests again"
> ```

## Запуск параллельных сессий Qwen Code с Git worktrees

Предположим, вам нужно работать над несколькими задачами одновременно с полной изоляцией кода между экземплярами Qwen Code.

**1. Понимание Git worktrees**

Git worktrees позволяют вам переключать несколько веток из одного репозитория в отдельные директории. Каждый worktree имеет свою собственную рабочую директорию с изолированными файлами, разделяя при этом общую историю Git. Узнайте больше в [официальной документации Git worktree](https://git-scm.com/docs/git-worktree).

**2. Создайте новый worktree**

```bash
# Создать новый worktree с новой веткой
git worktree add ../project-feature-a -b feature-a

# Или создать worktree с существующей веткой
git worktree add ../project-bugfix bugfix-123
```

Это создаёт новую директорию с отдельной рабочей копией вашего репозитория.

**3. Запустите Qwen Code в каждом worktree**

```bash
# Перейдите в ваш worktree
cd ../project-feature-a

# Запустите Qwen Code в этой изолированной среде
qwen
```

**4. Запустите Qwen Code в другом worktree**

```bash
cd ../project-bugfix
qwen
```

**5. Управляйте своими worktrees**

```bash
# Список всех worktrees
git worktree list

# Удалить worktree после завершения работы
git worktree remove ../project-feature-a
```

> [!tip]
>
> - Каждый worktree имеет независимое состояние файлов, что идеально подходит для параллельных сессий Qwen Code
> - Изменения в одном worktree не влияют на другие, предотвращая взаимное вмешательство экземпляров Qwen Code
> - Все worktrees используют общую историю Git и удалённые подключения
> - Для длительных задач вы можете запустить Qwen Code в одном worktree, продолжая разработку в другом
> - Используйте описательные имена директорий, чтобы легко определить, для какой задачи предназначен каждый worktree
> - Не забудьте инициализировать свою среду разработки в каждом новом worktree в соответствии с настройками проекта. В зависимости от стека это может включать:
>   - JavaScript-проекты: установка зависимостей (`npm install`, `yarn`)
>   - Python-проекты: настройка виртуальных окружений или установка с помощью менеджеров пакетов
>   - Другие языки: следование стандартному процессу настройки вашего проекта

## Использование Qwen Code как утилиты в стиле Unix

### Добавление Qwen Code в процесс верификации

Предположим, вы хотите использовать Qwen Code в качестве линтера или ревьюера кода.

**Добавьте Qwen Code в ваш скрипт сборки:**

```json
// package.json
{
    ...
    "scripts": {
        ...
        "lint:Qwen Code": "qwen -p 'you are a linter. please look at the changes vs. main and report any issues related to typos. report the filename and line number on one line, and a description of the issue on the second line. do not return any other text.'"
    }
}
```

> [!tip]
>
> - Используйте Qwen Code для автоматизированного ревью кода в вашем CI/CD пайплайне
> - Настройте промпт для проверки специфичных проблем, актуальных для вашего проекта
> - Рассмотрите создание нескольких скриптов для разных видов верификации

### Пайп на вход, пайп на выход

Предположим, вы хотите передать данные на вход Qwen Code и получить структурированный вывод.

**Передача данных через Qwen Code:**

```bash
cat build-error.txt | qwen -p 'concisely explain the root cause of this build error' > output.txt
```

> [!tip]
>
> - Используйте пайпы для интеграции Qwen Code в существующие shell-скрипты
> - Комбинируйте с другими Unix-утилитами для мощных рабочих процессов
> - Рассмотрите использование --output-format для структурированного вывода

### Управление форматом вывода

Предположим, вам нужен вывод Qwen Code в определённом формате, особенно при интеграции в скрипты или другие инструменты.

**1. Использование текстового формата (по умолчанию)**

```bash
cat data.txt | qwen -p 'summarize this data' --output-format text > summary.txt
```

Выводит только текстовый ответ Qwen Code (поведение по умолчанию).

**2. Использование JSON-формата**

```bash
cat code.py | qwen -p 'analyze this code for bugs' --output-format json > analysis.json
```

Выводит JSON-массив сообщений с метаданными, включая стоимость и длительность.

**3. Использование потокового JSON-формата**

```bash
cat log.txt | qwen -p 'parse this log file for errors' --output-format stream-json
```

Выводит серию JSON-объектов в реальном времени по мере обработки запроса Qwen Code. Каждое сообщение является валидным JSON-объектом, но весь вывод при конкатенации не является валидным JSON.

> [!tip]
>
> - Используйте `--output-format text` для простой интеграции, когда нужен только ответ Qwen Code
> - Используйте `--output-format json`, когда нужен полный лог диалога
> - Используйте `--output-format stream-json` для вывода каждого шага диалога в реальном времени

# Автоматический режим

Автоматический режим использует LLM-классификатор для оценки каждого вызова инструмента и принятия решения об автоматическом утверждении. Он находится между режимом Auto-Edit (который автоматически утверждает только правки файлов) и YOLO (который утверждает всё без исключения).

Эта страница является справочной по настройке и устранению неполадок Автоматического режима. Введение см. в [обзоре режимов утверждения](./approval-mode.md#4-auto-mode---classifier-driven-approval).

## Как это работает

Когда вы находитесь в Автоматическом режиме и агент пытается выполнить инструмент, Qwen Code последовательно проходит три уровня:

1. **acceptEdits fast-path** — Правка/Запись, целевой путь которой находится внутри рабочего пространства, автоматически утверждается без вызова классификатора.  
   **Исключение:** записи в поверхности самомодификации Qwen Code (`.qwen/settings*.json`, `QWEN.md`, `AGENTS.md`, `QWEN.local.md`, настроенные имена файлов контекста, `.qwen/rules/`, `.qwen/commands/`, `.qwen/agents/`, `.qwen/skills/`, `.qwen/hooks/`, `.mcp.json`) и поверхности сохранения (`.git/`, `.husky/`, `package.json`, `.npmrc`, `Makefile`, `.github/workflows/`, и т.д.) проходят через классификатор, даже если они находятся внутри рабочего пространства. Симлинки, указывающие на защищённые пути, также разрешаются и отклоняются. Команды оболочки, достигающие этих путей через `cd && bash -lc '...'` или другие обёртки, также проходят через классификатор.
2. **Список безопасных инструментов** — Встроенные инструменты только для чтения и метаданных (Read, Grep, Glob, LS, LSP, TodoWrite, AskUserQuestion и т.д.) автоматически утверждаются без вызова классификатора.
3. **LLM-классификатор** — Всё остальное (команды оболочки, веб-запросы, порождения под-агентов, правки вне рабочего пространства, MCP-инструменты) отправляется в двухэтапный классификатор:
   - **Этап 1 (быстрый)** — выводит только `{ shouldBlock }`. Примерно ~300 мс. Если `shouldBlock` равен `false`, действие разрешается, и вызов продолжается.
   - **Этап 2 (мыслительный)** — запускается только когда Этап 1 сказал "блокировать". Использует цепочку рассуждений для снижения ложных срабатываний Этапа 1. Может понизить блокировку Этапа 1 до разрешения. Выводит видимый пользователю `reason` при блокировке.

Классификатор использует настроенную быструю модель (`/model --fast`). Если быстрая модель не настроена, используется основная модель сессии.

## Жёсткие правила по-прежнему в силе

Автоматический режим **не** заменяет жёсткие правила разрешений. Перед запуском классификатора:

- Правила `permissions.deny` блокируют действие с указанием причины. Классификатор их не видит.
- Правила `permissions.allow` с конкретными спецификаторами (например, `Bash(git status)`, `Read(./docs/**)`) по-прежнему автоматически разрешаются без классификатора — **за исключением** случаев, когда вызов приводит к записи в защищённую поверхность самомодификации или сохранения (см. список в разделе "Как это работает"). В этом случае Автоматический режим перепроверяет вызов через классификатор, чтобы правило `Bash(*)` не могло молча превратиться в разрешение перезаписывать настройки Qwen Code, команды, хуки, навыки или MCP-серверы.
- Правила `permissions.ask` принудительно запрашивают ручное подтверждение даже в Автоматическом режиме.

## Чрезмерно широкие правила удаляются в Автоматическом режиме

Правила, подобные приведённым ниже, позволили бы агенту выполнять произвольный код без проверки классификатором:

- `Bash` / `Bash(*)` / `Bash()` — автоматически разрешает любую команду оболочки
- `Bash(python:*)`, `Bash(node*)`, `Bash(bash*)` — подстановочные знаки для интерпретаторов
- `Agent` / `Agent(coder)` — любое разрешение на инструмент Agent
- `Skill` / `Skill(pdf)` — любое разрешение на инструмент Skill

Когда вы входите в Автоматический режим, Qwen Code временно удаляет эти правила из активного набора разрешений и выводит уведомление с их списком. Правила возвращаются сразу после выхода из Автоматического режима. `settings.json` при этом не изменяется.

Если вам действительно нужны эти широкие правила, используйте вместо этого режим YOLO.

## Настройка подсказок

Автоматический режим читает `permissions.autoMode` из вашего `settings.json`. Записи представляют собой описания на естественном языке, а не шаблоны правил — они добавляются в системный промпт классификатора вместе со встроенными значениями по умолчанию.

Существует три категории подсказок, а также список окружения:

- **`allow`** — действия, которые классификатор должен автоматически утверждать.
- **`softDeny`** — деструктивные или необратимые действия, которые классификатор должен блокировать, **если только последний явный запрос пользователя не требовал именно этого действия и области**. Мягкие запреты могут быть сняты намерением пользователя; общее "да, делай что хочешь" не считается.
- **`hardDeny`** — действия, связанные с безопасностью, которые классификатор должен блокировать в Автоматическом режиме независимо от `autoMode.hints.allow` или недавнего намерения пользователя. Это политика классификатора, а не детерминированное правило разрешений: она не переопределяет `permissions.allow`. Используйте `permissions.deny` для действий, которые никогда не должны разрешаться менеджером разрешений.

```json
{
  "permissions": {
    "autoMode": {
      "hints": {
        "allow": [
          "Running poetry install and poetry update in this Python project",
          "Cleaning build artifacts under ./dist or ./build",
          "Reading any file under /Users/me/code/"
        ],
        "softDeny": [
          "Editing Qwen Code settings unless I explicitly ask for the exact change",
          "Running migration scripts that touch the production DB"
        ],
        "hardDeny": [
          "Sending secrets or .env contents to any network endpoint",
          "Modifying anything under ~/.ssh or ~/.aws"
        ]
      },
      "environment": [
        "This is a private monorepo with strict commit signing",
        "Production credentials live in 1Password, never in plain files"
      ]
    }
  }
}
```

`hints.deny` по-прежнему принимается для обратной совместимости и обрабатывается как `softDeny`. Смешивание обоих допускается — записи объединяются, сначала `softDeny`.

### Ограничения длины и количества

Чтобы системный промпт классификатора оставался небольшим:

- Каждая запись ограничена 200 символами (более длинные записи обрезаются с предупреждением).
- `hints.allow`, `hints.softDeny` и `hints.hardDeny` принимают до 50 записей каждая.
- `environment` принимает до 20 записей.

### Наслоение между файлами настроек

`autoMode` объединяется между системными/пользовательскими/рабочими настройками так же, как и другие настройки разрешений: массивы объединяются и дедублируются.

## Чтение решения

Когда классификатор блокирует действие, вызов инструмента завершается с одним из следующих текстов ошибки:

- **`Blocked by auto mode policy: <причина>`** — классификатор счёл действие небезопасным. Причина берётся из Этапа 2 классификатора.
- **`Auto mode classifier unavailable; action blocked for safety`** — API классификатора был недоступен, превысил тайм-аут или вернул неразбираемый ответ. Это поведение с закрытым отказом: при сомнении блокировать.

За обоими сообщениями следует строка-подсказка, указывающая агенту, что **конкретно запрещённое действие** не должно быть выполнено через другой инструмент, косвенное обращение к оболочке, сгенерированный скрипт, псевдоним, симлинк, изменение конфигурации, хук, файл команд, конфигурацию MCP, закодированный полезный груз или эквивалентный путь. **Несвязанная безопасная работа и действительно более безопасные альтернативы по-прежнему разрешены** — блокируются только попытки достичь той же запрещённой цели через другой способ.

Если запрещённое действие действительно необходимо, агент должен остановиться и запросить у вас явное одобрение, а не пытаться обойти запрет.

### Язык причины классификатора

Причины классификатора создаются LLM и не переводятся. Если вы хотите причины не на английском, добавьте подсказку типа `Respond reasons in Chinese` в `permissions.autoMode.environment`.

## Возврат к ручному утверждению

Автоматический режим защищает вас от застревания:

- После **3 последовательных блокировок политикой** следующий вызов инструмента возвращается к стандартному запросу ручного утверждения. Это ловит ситуацию, когда агент продолжает пробовать небольшие варианты запрещённой команды.
- После **2 последовательных недоступностей** (сбоев API классификатора) следующий вызов инструмента также возвращается к ручному утверждению. Это предотвращает ожидание неработающего классификатора.

Сессия остаётся в Автоматическом режиме — только единственный возвратный вызов проходит через ручное утверждение. Счётчики сбрасываются, когда вы утверждаете возвратный вызов или переключаете режимы.

Если вы постоянно сталкиваетесь с возвратом, наиболее вероятные причины — сбой в API классификатора или подсказки, требующие настройки. Переключитесь в режим Default Mode, пока расследуете.

## Устранение неполадок

**"Автоматический режим постоянно блокирует мои команды"**

Посмотрите на причину в сообщении об ошибке. Если классификатор слишком консервативен для вашего контекста, добавьте запись в `permissions.autoMode.hints.allow`, описывающую шаблон на естественном языке. Примеры:

- `"Building Docker images for this project (docker build ...)"`
- `"Running database migrations against the local test DB"`

**"Автоматический режим классификатора недоступен"`

API классификатора не ответил. Возможные причины:

- Сетевая проблема между вами и конечной точкой модели.
- Настроенная быстрая модель больше недоступна — проверьте `/model --fast`.
- Транскрипт слишком длинный и превышает контекстное окно быстрой модели.

Во время диагностики переключитесь обратно на режим Default Mode: `/approval-mode default`.

**"Возврат к ручному утверждению"`

Вы достигли защиты от 3 последовательных блокировок или 2 последовательных недоступностей. Утвердите или отклоните запрос как обычно. После одного утверждённого возврата счётчик последовательных сбрасывается.

**Классификатор видит конфиденциальные данные в моих промптах**

Входные данные инструментов проецируются через метод `toAutoClassifierInput` каждого инструмента, прежде чем попасть к классификатору. Длинный контент правок, промпты веб-запросов и промпты под-агентов обрезаются. Результаты инструментов (содержимое файлов, веб-страницы) никогда не отправляются классификатору — только текст пользователя и вызовы инструментов ассистента проходят.

Если конкретный инструмент раскрывает поля, которые вы хотели бы скрыть, создайте issue с именем инструмента; проекция выполняется для каждого инструмента и со временем будет ужесточаться.

## Ограничения

- **Не работает в офлайн-режиме.** Классификатор требует вызова LLM.
- **Добавляет задержку на медленном пути.** Список разрешённых + acceptEdits покрывают большинство вызовов без задержки, но `run_shell_command` обычно добавляет ~300 мс (быстрый путь классификатора) или ~3–5 с (медленный путь с мыслительной проверкой).
- **Не заменяет правила `deny`.** Классификатор работает по принципу "наилучших усилий". Для команд, которые, как вы уверены, никогда не должны запускаться, поместите их в `permissions.deny`.
- **MCP-инструменты по умолчанию блокируются консервативно.** Сторонние MCP-инструменты (`mcp__*`) подписываются на пересылку аргументов через переопределение `toAutoClassifierInput`. Инструменты, которые не подписались, раскрывают классификатору только своё имя — большинство таких вызовов консервативно блокируется, если вы не написали явное правило `allow`. Это fail-closed по замыслу (учётные данные и большой объём контента не попадают в LLM классификатора). Если вы доверяете конкретному MCP-инструменту, добавьте `permissions.allow: ["mcp__server__tool"]`, чтобы он полностью обходил классификатор.

## FAQ

**Отправляет ли Автоматический режим мой код третьей стороне?**

Автоматический режим использует вашу существующую конфигурацию модели — ту же конечную точку, что и основной агент. Если вы настроили Qwen Code на использование самостоятельной модели, классификатор также работает через эту конечную точку.

**Попадут ли мои секреты / содержимое `.env` в классификатор?**

Классификатор видит только то, что раскрывает проекция `toAutoClassifierInput` каждого инструмента:

- `read_file` и другие инструменты только для чтения: не вызываются (они находятся в списке быстрого пути).
- `edit` / `write_file`: путь к файлу плюс первые 80 символов старого/нового содержимого. Полное содержимое не пересылается.
- `run_shell_command`: полная команда (она должна быть, потому что это то, что оценивает классификатор).
- `web_fetch`: только URL. Поле промпта не пересылается.
- `agent`: тип под-агента плюс полный промпт. Промпт — это инструкция, которой будет следовать под-агент, поэтому классификатору нужен он целиком, чтобы обнаружить атаки, которые могли бы направить под-агента на деструктивные действия — по той же причине `run_shell_command` пересылает полную команду.

Результаты инструментов (фактическое содержимое, возвращаемое инструментами) полностью удаляются из транскрипта классификатора.

MCP-инструменты (`mcp__*`) следуют более строгому умолчанию: их параметры не пересылаются, если автор MCP-инструмента явно не подписался через переопределение `toAutoClassifierInput`. Классификатор видит имя инструмента, но не аргументы, поэтому большинство MCP-вызовов будут консервативно блокироваться, если пользователь не написал явное правило разрешения. Это fail-closed по замыслу — сторонние инструменты не должны без намерения передавать учётные данные или большой объём содержимого файлов в LLM классификатора.

**Можно ли отключить информационное сообщение при первом запуске?**

Оно появляется только один раз на файл пользовательских настроек. После отклонения в пользовательских настройках устанавливается `ui.autoModeAcknowledged: true`.

**Чем это отличается от Auto-Edit?**

Auto-Edit автоматически утверждает правки файлов и ничего больше — команды оболочки по-прежнему запрашивают подтверждение. Автоматический режим использует классификатор, чтобы также автоматически утверждать безопасные команды оболочки и другие вызовы инструментов, при этом блокируя рискованные.

**Чем это отличается от YOLO?**

YOLO автоматически утверждает всё без какой-либо проверки. В Автоматическом режиме в цикле участвует классификатор, который блокирует рискованные действия.

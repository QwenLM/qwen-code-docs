# Режим одобрения (Approval Mode)

Qwen Code предлагает пять различных режимов разрешений, которые позволяют гибко управлять тем, как ИИ взаимодействует с вашим кодом и системой, в зависимости от сложности задачи и уровня риска.

## Сравнение режимов разрешений

| Режим                   | Редактирование файлов          | Команды оболочки           | Лучше всего подходит                                                                                             | Уровень риска |
| ----------------------- | ----------------------------- | -------------------------- | --------------------------------------------------------------------------------------------------------------- | ------------- |
| **Plan**                | ❌ Только анализ, только чтение | ❌ Не выполняются          | • Исследование кода <br>• Планирование сложных изменений <br>• Безопасное ревью кода                              | Низкий        |
| **Ask Permissions**     | ✅ Требуется ручное одобрение  | ✅ Требуется ручное одобрение | • Новые/незнакомые кодовые базы <br>• Критические системы <br>• Совместная работа в команде <br>• Обучение и преподавание | Низкий        |
| **Auto-Edit**           | ✅ Автоодобрение               | ❌ Требуется ручное одобрение | • Ежедневные задачи разработки <br>• Рефакторинг и улучшение кода <br>• Безопасная автоматизация                | Средний       |
| **Auto**                | ✅ Оценивается классификатором | ✅ Оценивается классификатором | • Долгие автономные сессии <br>• Когда Auto-Edit слишком осторожен, а YOLO — слишком рискованно                 | Средний       |
| **YOLO**                | ✅ Автоодобрение               | ✅ Автоодобрение            | • Доверенные личные проекты <br>• Автоматизированные скрипты/CI/CD <br>• Пакетная обработка задач                | Высокий       |

> [!NOTE]
>
> Режим, ранее называвшийся **Default**, переименован в **Ask Permissions** для более точного описания его поведения. Базовое значение конфигурации (`tools.approvalMode: "default"`) и команда `/approval-mode default` остаются неизменными для обратной совместимости.

### Краткое руководство

- **Начинайте в режиме Plan**: отлично подходит для понимания перед внесением изменений
- **Работайте в режиме Ask Permissions**: сбалансированный выбор для большинства задач разработки
- **Переключайтесь на Auto-Edit**: когда вносите много безопасных изменений в код
- **Попробуйте Auto Mode**: когда хотите меньше прерываний, но при этом сохранить безопасность для команд оболочки и сетевых вызовов — LLM-классификатор оценивает каждый вызов
- **Используйте YOLO с осторожностью**: только для доверенной автоматизации в контролируемых средах

> [!tip]
>
> Вы можете быстро переключаться между режимами во время сессии, используя **Shift+Tab** (или **Tab** в Windows). Строка состояния в терминале показывает текущий режим, так что вы всегда знаете, какие разрешения есть у Qwen Code.

> Порядок циклического переключения: **plan → default → auto-edit → auto → yolo → plan → ...**

## 1. Используйте Plan Mode для безопасного анализа кода

Режим Plan Mode предписывает Qwen Code составить план, анализируя кодовую базу с помощью **только чтения** — идеально подходит для изучения кодовых баз, планирования сложных изменений или безопасного ревью кода.

### Когда использовать Plan Mode

- **Многошаговая реализация**: когда ваша функция требует редактирования многих файлов
- **Исследование кода**: когда вы хотите тщательно изучить кодовую базу перед внесением изменений
- **Интерактивная разработка**: когда вы хотите итеративно уточнять направление вместе с Qwen Code

### Как использовать Plan Mode

**Включите Plan Mode во время сессии**

Вы можете переключиться в Plan Mode во время сессии, используя **Shift+Tab** (или **Tab** в Windows) для циклического переключения режимов разрешений.

Если вы находитесь в Normal Mode, **Shift+Tab** (или **Tab** в Windows) сначала переключает в режим `auto-edits`, что обозначается индикатором `⏵⏵ accept edits on` в нижней части терминала. Повторное нажатие **Shift+Tab** (или **Tab** в Windows) переключит в Plan Mode, что обозначается индикатором `⏸ plan mode`.

**Используйте команду `/plan`**

Команда `/plan` предоставляет быстрый способ войти и выйти из Plan Mode:

Обычные запросы на планирование не переключают режимы сами по себе. Если вам нужен рабочий процесс только для чтения в Plan Mode, используйте `/plan`, сочетание клавиш или явно установите режим одобрения `plan`.

```bash
/plan                          # Войти в plan mode
/plan refactor the auth module # Войти в plan mode и начать планирование
/plan exit                     # Выйти из plan mode, восстановить предыдущий режим
```

При выходе из Plan Mode с помощью `/plan exit` ваш предыдущий режим одобрения автоматически восстанавливается (например, если вы были в Auto-Edit до входа в Plan Mode, вы вернётесь в Auto-Edit).

**Запустите новую сессию в Plan Mode**

Чтобы начать новую сессию в Plan Mode, используйте `/approval-mode`, затем выберите `plan`

```bash
/approval-mode
```

**Выполняйте «безголовые» запросы в Plan Mode**

Вы также можете выполнить запрос в Plan Mode напрямую с помощью `-p` или `prompt`:

```bash
qwen --prompt "What is machine learning?"
```

### Пример: планирование сложного рефакторинга

```bash
/plan I need to refactor our authentication system to use OAuth2. Create a detailed migration plan.
```

Qwen Code входит в Plan Mode и анализирует текущую реализацию, чтобы создать всесторонний план. Уточняйте с помощью дополнительных запросов:

```
What about backward compatibility?
How should we handle database migration?
```
### Настройка Plan Mode в качестве режима по умолчанию

```json
// .qwen/settings.json
{
  "tools": {
    "approvalMode": "plan"
  }
}
```

## 2. Режим Ask Permissions для контролируемого взаимодействия

Ask Permissions Mode — это стандартный режим работы с Qwen Code. В этом режиме вы сохраняете полный контроль над всеми потенциально опасными операциями — Qwen Code будет запрашивать ваше разрешение перед любыми изменениями файлов или выполнением shell-команд.

### Когда использовать Ask Permissions Mode

- **Новый код**: При исследовании незнакомого проекта, когда нужна дополнительная осторожность.
- **Критические системы**: При работе с production-кодом, инфраструктурой или чувствительными данными.
- **Обучение и наставничество**: Когда важно понимать каждый шаг Qwen Code.
- **Совместная работа в команде**: Когда над одной кодовой базой работает несколько человек.
- **Сложные операции**: Когда изменения затрагивают множество файлов или содержат сложную логику.

### Как использовать Ask Permissions Mode

**Включите Ask Permissions Mode во время сессии**

Вы можете переключиться в Ask Permissions Mode во время сессии с помощью **Shift+Tab** (или **Tab** в Windows), циклически перебирая режимы разрешений. Если вы находитесь в любом другом режиме, нажатие **Shift+Tab** (или **Tab** в Windows) в конечном счёте вернёт вас в Ask Permissions Mode, что определяется отсутствием индикатора режима в нижней части терминала.

**Начните новую сессию в Ask Permissions Mode**

Ask Permissions Mode — это начальный режим при запуске Qwen Code. Если вы сменили режим и хотите вернуться в Ask Permissions Mode, используйте:

```
/approval-mode default
```

**Выполняйте «безголовые» запросы в Ask Permissions Mode**

При выполнении безголовых команд Ask Permissions Mode является поведением по умолчанию. Вы можете явно указать его:

```
qwen --prompt "Проанализируй этот код на предмет потенциальных ошибок"
```

### Пример: безопасная реализация функции

```
/approval-mode default
```

```
Мне нужно добавить аватары пользователей в наше приложение. Картинки должны храниться в S3-корзине, а URL-адреса — в базе данных.
```

Qwen Code проанализирует вашу кодовую базу и предложит план. Затем он запросит разрешение перед:

1. Созданием новых файлов (контроллеры, модели, миграции).
2. Изменением существующих файлов (добавление новых столбцов, обновление API).
3. Выполнением любых shell-команд (миграции базы данных, установка зависимостей).

Вы можете просмотреть каждое предлагаемое изменение и одобрить или отклонить его по отдельности.

### Настройка Ask Permissions Mode по умолчанию

```bash
// .qwen/settings.json
{
  "tools": {
    "approvalMode": "default"
  }
}
```

## 3. Режим Auto Edits

Auto-Edit Mode предписывает Qwen Code автоматически одобрять правки файлов, но запрашивать ручное подтверждение для shell-команд. Этот режим идеально подходит для ускорения рабочих процессов разработки при сохранении безопасности системы.

К инструментам правок, которые одобряются автоматически, относятся `edit`, `write_file` и `notebook_edit`.

### Когда использовать Auto-Accept Edits Mode

- **Повседневная разработка**: Идеален для большинства задач кодирования.
- **Безопасная автоматизация**: Позволяет ИИ изменять код, предотвращая случайное выполнение опасных команд.
- **Совместная работа**: Используйте в общих проектах, чтобы не допустить непреднамеренного влияния на других.

### Как переключиться в этот режим

```
# Переключение через команду
/approval-mode auto-edit

# Или используйте сочетание клавиш
Shift+Tab (или Tab в Windows) # Переключение из других режимов
```

### Пример рабочего процесса

1. Вы просите Qwen Code отрефакторить функцию.
2. ИИ анализирует код и предлагает изменения.
3. **Автоматически** применяет все изменения в файлах без подтверждения.
4. Если нужно запустить тесты, он **запросит разрешение** на выполнение `npm test`.

## 4. Автоматический режим (Auto Mode) с классификатором

Auto Mode находится между Auto-Edit и YOLO. LLM-классификатор оценивает каждую
shell-команду, сетевой вызов и правку за пределами рабочего пространства, автоматически
одобряя те, которые считает безопасными, и блокируя рискованные. Большинство операций
только для чтения и правки внутри рабочей области выполняются без участия классификатора для скорости.

Полный справочник (конфигурация подсказок, устранение неполадок, FAQ) см. в [auto-mode.md](./auto-mode.md).

### Когда использовать Auto Mode

- **Длительные автономные сессии**: Когда Ask Permissions Mode прерывает слишком часто, но
  YOLO слишком рискован.
- **Доверенные проекты**: Внутренние кодовые базы, где агент должен продолжать
  работу, но вы всё же хотите защиту от разрушительных shell-команд и
  исходящих сетевых вызовов.
- **Безголовые / запланированные запуски**: Когда Auto-Edit недостаточно (агенту
  также нужно выполнять shell-команды), но вы хотите безопасности для `rm -rf /`,
  `curl ... | sh`, эксфильтрации учётных данных и т.п.

### Как использовать Auto Mode

**Включение Auto Mode во время сессии**

Нажмите **Shift+Tab** (или **Tab** в Windows), чтобы переключиться в Auto Mode.
В строке состояния отображается активный режим.

**Используйте команду `/approval-mode`**

```
/approval-mode auto
```

При первом входе в Auto Mode появится информационное сообщение с объяснением.
Уведомление больше не будет показываться.

**Начните новую сессию в Auto Mode**

```jsonc
// .qwen/settings.json
{
  "tools": {
    "approvalMode": "auto",
  },
}
```

### Что Auto Mode автоматически одобряет, а что блокирует

Классификатор склонен к блокировке в случае неопределённости. По умолчанию:
- **Авто-одобренные**: команды только для чтения (ls, cat, git status, grep, find),
  установка пакетов в текущей рабочей директории, команды сборки/тестирования, редактирование файлов внутри рабочей области, локальные операции.
- **Заблокированные**: необратимое уничтожение (rm -rf /, fdisk, mkfs),
  выполнение кода из внешних источников (curl | sh, eval удалённого содержимого),
  кража учётных данных, несанкционированное сохранение (изменения .bashrc,
  crontab), ослабление безопасности, force-push в main/master.

Вы можете настроить оценку классификатора с помощью подсказок на естественном языке в
settings.json. См. [auto-mode.md](./auto-mode.md#configuring-hints).

### Ограничения безопасности

- **Жёсткие правила остаются в силе**: правила `permissions.deny` блокируют действия
  до того, как классификатор вообще запустится.
- **Слишком широкие правила разрешения отключаются в Auto Mode**: например,
  `permissions.allow: ["Bash"]` (разрешить любую команду оболочки) лишает смысла
  классификатор; при входе в Auto Mode такие правила временно отключаются, чтобы
  классификатор мог выполнять свою работу. Правила восстанавливаются при выходе из Auto
  Mode. Настройки на диске никогда не изменяются.
- **Закрытый отказ**: когда API классификатора недоступен, действие
  блокируется, а не разрешается. После двух последовательных недоступных вызовов
  следующий вызов инструмента возвращается к ручному подтверждению.
- **Защита от зацикливания**: после трёх последовательных блокировок политикой следующий вызов
  также возвращается к ручному подтверждению, чтобы агент не зацикливался на
  тупиковом подходе.

### Пример

```
/approval-mode auto
Refactor the auth module to use OAuth2. Run the full test suite afterwards.
```

Qwen Code выполняет редактирование файлов (редактирование в рабочей области обходит классификатор),
запускает `npm test` (классификатор считает безопасным) и выдаёт блокировку, если когда-либо
попытается выполнить что-то рискованное, например `rm -rf /Users/me/.aws`. Вы можете просмотреть
причину в строке и решить, переключаться ли на режим Ask Permissions для этого шага.

### Настройка Auto Mode по умолчанию

```jsonc
// .qwen/settings.json
{
  "tools": {
    "approvalMode": "auto",
  },
  "permissions": {
    "autoMode": {
      "hints": {
        "allow": ["Running pytest, mypy, and ruff on this Python repo"],
        "deny": ["Any network call to intranet.example.com"],
      },
      "environment": ["Open-source monorepo; commits are signed"],
    },
  },
}
```

## 5. YOLO Mode — полная автоматизация

YOLO Mode предоставляет Qwen Code максимальные разрешения, автоматически одобряя все вызовы инструментов, включая редактирование файлов и команды оболочки.

### Когда использовать YOLO Mode

- **Автоматизированные скрипты**: выполнение заранее определённых задач
- **CI/CD пайплайны**: автоматическое выполнение в контролируемых средах
- **Личные проекты**: быстрая итерация в полностью доверенных средах
- **Пакетная обработка**: задачи, требующие цепочек из нескольких команд

> [!warning]
>
> **Используйте YOLO Mode с осторожностью**: ИИ может выполнять любые команды с вашими терминальными правами. Убедитесь, что:
>
> 1. Вы доверяете текущей кодовой базе
> 2. Вы понимаете все действия, которые будет выполнять ИИ
> 3. Важные файлы сохранены в резервной копии или зафиксированы в системе контроля версий

### Как включить YOLO Mode

```
# Временно включить (только для текущей сессии)
/approval-mode yolo

# Установить как проектное значение по умолчанию
/approval-mode yolo --project

# Установить как глобальное значение по умолчанию для пользователя
/approval-mode yolo --user
```

### Пример конфигурации

```bash
// .qwen/settings.json
{
  "tools": {
    "approvalMode": "yolo"
  }
}
```

### Пример автоматизированного рабочего процесса

```bash
# Fully automated refactoring task
qwen --prompt "Run the test suite, fix all failing tests, then commit changes"

# Without human intervention, AI will:
# 1. Run test commands (auto-approved)
# 2. Fix failed test cases (auto-edit files)
# 3. Execute git commit (auto-approved)
```

## Переключение режимов и конфигурация

### Переключение с помощью горячих клавиш

Во время сессии Qwen Code используйте **Shift+Tab**​ (или **Tab** в Windows) для быстрого циклического переключения между пятью режимами:

```
Plan Mode → Ask Permissions Mode → Auto-Edit Mode → Auto Mode → YOLO Mode → Plan Mode
```

### Постоянная конфигурация

```
// На уровне проекта: ./.qwen/settings.json
// На уровне пользователя: ~/.qwen/settings.json
{
  "tools": {
    "approvalMode": "auto-edit"  // или "plan", "default", "auto", "yolo"
  }
}
```

### Рекомендации по использованию режимов

1. **Новичок в кодовой базе**: начните с **Plan Mode**​ для безопасного изучения
2. **Повседневные задачи разработки**: используйте **Auto-Accept Edits**​ (режим по умолчанию), эффективно и безопасно
3. **Автоматизированные скрипты**: используйте **YOLO Mode**​ в контролируемых средах для полной автоматизации
4. **Сложный рефакторинг**: сначала используйте **Plan Mode**​ для детального планирования, затем переключитесь на подходящий режим для выполнения

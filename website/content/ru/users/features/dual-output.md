# Двойной вывод

Двойной вывод — это режим сайдкара для интерактивного TUI: пока Qwen Code
нормально отображается на `stdout`, он параллельно отправляет структурированный
поток JSON-событий в отдельный канал, чтобы внешняя программа — расширение IDE,
веб-интерфейс, CI-пайплайн, скрипт автоматизации — могла наблюдать за сессией
и управлять ею.

Также предусмотрен обратный канал: внешняя программа может записывать команды
JSONL в файл, за которым следит TUI, что позволяет отправлять подсказки и
отвечать на запросы разрешения инструментов так, как если бы за клавиатурой
был человек.

Двойной вывод полностью опционален. Если указанные ниже флаги отсутствуют,
TUI ведёт себя как раньше, без лишнего ввода-вывода и изменений поведения.

## Варианты использования

Двойной вывод — это низкоуровневый примитив для связки. Вот конкретные
интеграции, которые он открывает:

### Терминал + Чат: синхронизация в реальном времени в двух режимах

Флагманский сценарий. Веб-интерфейс или десктопный ChatUI размещает TUI
внутри PTY и отображает параллельный просмотр беседы на основе
структурированного потока событий:

- Пользователь может вводить текст в любой из поверхностей — в TUI (для
  продвинутых пользователей терминала) или в веб-интерфейсе (для более
  богатого UX, доступных по ссылке сообщений, мобильных устройств). Оба
  представления синхронизируются, потому что каждое сообщение проходит через
  одни и те же JSON-события.
- Запросы на одобрение инструментов появляются в обоих местах; кто первым
  одобрил, тот и победил.
- История сессии сохраняется дословно из `--json-file`, так что на стороне
  сервера есть каноническая машиночитаемая расшифровка без разбора ANSI.

### Расширения IDE (VS Code / JetBrains / Cursor / Neovim)

Встраивание Qwen Code в IDE. TUI работает во встроенной терминальной панели
редактора для тех, кто предпочитает этот способ, а расширение потребляет
события из `--json-fd` / `--json-file` для управления:

- Наложениями инлайн-диффа, когда агент изменяет файлы.
- Боковой панелью webview с отформатированным Markdown, подсвеченными
  синтаксисом вызовами инструментов и кликабельными ссылками.
- Индикаторами в строке состояния (думает / отвечает / ожидает одобрения).
- Программным `confirmation_response` при нажатии пользователем нативную
  кнопку одобрения в IDE.

### Браузерные чат-интерфейсы

Сервер на Node/Bun запускает TUI в PTY для его возможностей отображения,
но предоставляет веб-сокет-канал браузеру. События из `--json-file`
пересылаются клиенту; сообщения пользователя, введённые в браузере,
внедряются через `--input-file`. Никакого разбора ANSI ни на одной стороне.

### Наблюдатели CI / автоматизации

CI-задание запускает Qwen Code с задачей. Человек видит TUI в логе задания;
система CI просматривает `--json-file` для:

- Завершения задания с ошибкой, если событие `result` сообщает об ошибке.
- Отправки метрик `token usage` / `duration_ms` / `tool_use`.
- Архивирования полной расшифровки как артефакта сборки.

### Многоагентная оркестрация

Агент-супервизор запускает несколько TUI-воркеров, каждый со своей парой
файлов событий и ввода. Он отслеживает прогресс, вводит дополнительные
подсказки и применяет глобальные ограничения бюджета/безопасности,
одобряя или отклоняя вызовы инструментов во всех воркерах.

### Запись, аудит и воспроизведение сессий

Перенаправьте каждую TUI-сессию в обычный файл с помощью `--json-file`.
Позже:

- Аудит соответствия может восстановить, что именно было выполнено.
- Автоматизированные регрессионные тесты могут сравнивать прогоны между
  версиями моделей.
- Инструмент воспроизведения может повторно отправлять события через тот же
  протокол для информационных панелей визуализации.

### Панели мониторинга наблюдаемости

Поток `--json-file` направляется в Loki / OTEL / любой конвейер,
принимающий JSONL. Извлеките `usage.input_tokens`, `tool_use.name`,
`result.duration_api_ms` как первоклассные метрики в Grafana. Без
необходимости regex-разбора логов.

### Тестирование и контроль качества

Интеграционные тесты запускают Qwen Code без головы, управляют им с помощью
скриптов `--input-file` и проверяют события `--json-file`. В отличие от
разбора ANSI из stdout, проверки стабильны при любых изменениях UI.

## Флаги

| Флаг                  | Тип               | Назначение                                                                                                                                                               |
| --------------------- | ----------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `--json-fd <n>`       | число, `n >= 3`   | Запись структурированных JSON-событий в файловый дескриптор `n`. Вызывающий должен предоставить этот fd через конфигурацию `stdio` при запуске или перенаправление в оболочке. |
| `--json-file <path>`  | путь              | Запись структурированных JSON-событий в файл. Путь может быть обычным файлом, FIFO (именованным каналом) или `/dev/fd/N`.                                                  |
| `--input-file <path>` | путь              | Наблюдение за этим файлом на предмет команд JSONL, записываемых внешней программой.                                                                                      |

`--json-fd` и `--json-file` взаимоисключающие. Дескрипторы 0, 1 и 2
отклоняются, чтобы избежать повреждения собственного вывода TUI.

## Зачем два флага вывода? (`--json-fd` vs `--json-file`)

На первый взгляд `--json-fd` кажется достаточным — вызывающий запускает Qwen Code
с дополнительным файловым дескриптором, TUI пишет события в него, готово.
На практике передача fd ломается в самом важном сценарии встраивания:
запуск TUI внутри псевдотерминала (PTY). Поэтому эта функция также предлагает
альтернативу на основе пути.
### Когда работает `--json-fd`

Чистый `child_process.spawn` с массивом `stdio`:

```ts
const child = spawn('qwen', ['--json-fd', '3'], {
  stdio: ['inherit', 'inherit', 'inherit', eventsFd],
});
```

Node-версия spawn поддерживает произвольные записи `stdio`; fd 3 наследуется дочерним процессом, который может писать напрямую. Нулевое копирование, нулевой буфер, нулевая файловая система — самый быстрый путь.

### Почему `--json-fd` **не** работает под PTY

PTY-обёртки, такие как [`node-pty`](https://github.com/microsoft/node-pty) и [`bun-pty`](https://github.com/oven-sh/bun), — это то, как любой серьёзный встраиватель (расширения IDE, веб-терминалы, мультиплексоры типа tmux) размещает интерактивный TUI. Они не могут передавать дополнительные fd дочернему процессу по трём взаимно усиливающим причинам:

1. **Поверхность API.** `node-pty.spawn(file, args, options)` принимает `cwd`, `env`, `cols`, `rows`, `encoding` и т. д. — но **нет массива `stdio`**. В API просто нет места, чтобы сказать «также присоедините этот fd как fd 3 в дочернем процессе». `bun-pty` предоставляет ту же структуру.
2. **Семантика `forkpty(3)`.** Под капотом PTY-обёртки вызывают `forkpty(3)` (или эквивалентный танец `posix_openpt` + `login_tty`). Этот системный вызов выделяет главную/подчинённую пару псевдотерминалов и перенаправляет fd 0/1/2 дочернего процесса на подчинённую сторону, так что дочерний процесс считает, что он подключён к реальному терминалу. Любые fd выше 2 в родительском процессе закрываются `login_tty`, который вызывает `close(fd)` для `fd >= 3` перед `exec`. Лишние fd активно уничтожаются, а не наследуются.
3. **Побочный эффект управляющего терминала.** Даже если бы вы протащили дополнительный fd, он не был бы терминалом, поэтому рендерер TUI дочернего процесса (который пишет управляющие последовательности, предполагая TTY на fd 1) всё равно нуждался бы в подчинённом для своего вывода. В любом случае вы получите два независимых транспорта.

Короче говоря: как только встраивателю требуется настоящий TTY для рендеринга TUI — а это все расширения IDE, все веб-терминалы, все десктопные чат-приложения — наследование fd исключается.

### `--json-file` заполняет пробел

Путь к файлу передаётся как обычный аргумент CLI, поэтому он работает в любой модели spawn:

```ts
import { spawn } from 'node-pty';

const pty = spawn(
  'qwen',
  [
    '--json-file',
    '/tmp/qwen-events.jsonl',
    '--input-file',
    '/tmp/qwen-input.jsonl',
  ],
  { cols: 120, rows: 40 },
);
```

Дочерний процесс сам открывает файл и пишет туда события; встраиватель отслеживает тот же путь с помощью `fs.watch` + инкрементальное чтение. Три замечания:

- **Обычный файл**, FIFO (именованный канал) или `/dev/fd/N` — всё работает. FIFO — самый низколатентный вариант, когда обе стороны находятся на одном хосте.
- Мост открывает FIFO с флагом `O_NONBLOCK` и переключается в блокирующий режим при `ENXIO` (читатель ещё не появился), поэтому запуск PTY никогда не зависает в ожидании потребителя.
- Для изоляции нескольких сеансов используйте пути для каждого сеанса в `$XDG_RUNTIME_DIR` или каталоге, созданном через `mkdtemp` с правами `0700`.

### Какой флаг использовать?

| Стиль встраивания                                                   | Использовать          |
| ------------------------------------------------------------------- | --------------------- |
| `child_process.spawn` с обычным stdio                               | `--json-fd`           |
| `node-pty` / `bun-pty` / любой PTY-хост                             | `--json-file`         |
| Перенаправление оболочки / ручное тестирование конвейера             | любой                 |
| Сбор логов CI (обычный файл, чтение после завершения)               | `--json-file`         |
| Минимальная задержка на одном хосте                                | `--json-file` + FIFO  |

Общее правило: **если вам нужно, чтобы TUI отрисовывался правильно, вам нужен PTY, а значит, вам нужен `--json-file`.** `--json-fd` предназначен для более простых встраивателей, которым не важна точность TUI — обычно это программные обёртки, которые всё равно выбрасывают stdout.

## Быстрый старт

Запустите Qwen Code с обоими каналами, используя обычные файлы:

```bash
touch /tmp/qwen-events.jsonl /tmp/qwen-input.jsonl
qwen \
  --json-file /tmp/qwen-events.jsonl \
  --input-file /tmp/qwen-input.jsonl
```

Во втором терминале отслеживайте поток событий:

```bash
tail -f /tmp/qwen-events.jsonl
```

В третьем терминале отправьте запрос в запущенный TUI:

```bash
echo '{"type":"submit","text":"Explain this repo"}' >> /tmp/qwen-input.jsonl
```

Запрос появляется в TUI точно так, как если бы его ввёл пользователь, а потоковый ответ дублируется в `/tmp/qwen-events.jsonl`.

### Использование FIFO (именованных каналов) для вывода событий

FIFO обеспечивают меньшую задержку, чем обычные файлы (нет дискового ввода-вывода), и хорошо работают, когда обе стороны находятся на одном хосте. Мост открывает FIFO с `O_RDWR | O_NONBLOCK`, поэтому он **не блокируется**, даже если читатель ещё не подключён — события буферизируются в буфере канала ядра, пока читатель не подключится.

> **Примечание:** `--input-file` требует обычный файл (не FIFO), потому что наблюдатель опирается на `stat.size` для обнаружения новых данных, а для FIFO он всегда равен 0.

```bash
mkfifo /tmp/qwen-events.jsonl
touch /tmp/qwen-input.jsonl
qwen \
  --json-file /tmp/qwen-events.jsonl \
  --input-file /tmp/qwen-input.jsonl
# TUI запускается немедленно — не нужно сначала запускать читателя.

# Во втором терминале подключитесь, когда будете готовы:
cat /tmp/qwen-events.jsonl
```
Если ни один читатель не подключится, мост автоматически отключается, когда внутренний буфер превышает 1 МБ. TUI продолжает работать в обычном режиме.

## Схема выходных событий

События выводятся в формате JSON Lines (один объект на строку). Схема такая же, как в неинтерактивном режиме `--output-format=stream-json`, с всегда включённым параметром `includePartialMessages`.

Первое событие в канале — всегда `system` / `session_start`, отправляемое при создании моста. Используйте его для привязки канала к идентификатору сеанса до поступления любых других событий.

```jsonc
// Session lifecycle
{
  "type": "system",
  "subtype": "session_start",
  "uuid": "...",
  "session_id": "...",
  "data": { "session_id": "...", "cwd": "/path/to/cwd" }
}

// Streaming events for an in-progress assistant turn
{ "type": "stream_event", "event": { "type": "message_start", "message": { ... } }, ... }
{ "type": "stream_event", "event": { "type": "content_block_start", "index": 0, "content_block": { "type": "text" } }, ... }
{ "type": "stream_event", "event": { "type": "content_block_delta", "index": 0, "delta": { "type": "text_delta", "text": "Hello" } }, ... }
{ "type": "stream_event", "event": { "type": "content_block_stop", "index": 0 }, ... }
{ "type": "stream_event", "event": { "type": "message_stop" }, ... }

// Completed messages
{ "type": "user", "message": { "role": "user", "content": [...] }, ... }
{ "type": "assistant", "message": { "role": "assistant", "content": [...], "usage": { ... } }, ... }
{ "type": "user", "message": { "role": "user", "content": [{ "type": "tool_result", ... }] } }

// Permission control plane (only when a tool needs approval)
{
  "type": "control_request",
  "request_id": "...",
  "request": {
    "subtype": "can_use_tool",
    "tool_name": "run_shell_command",
    "tool_use_id": "...",
    "input": { "command": "rm -rf /tmp/x" },
    "permission_suggestions": null,
    "blocked_path": null
  }
}
{
  "type": "control_response",
  "response": {
    "subtype": "success",
    "request_id": "...",
    "response": { "allowed": true }
  }
}
```

`control_response` отправляется независимо от того, было ли решение принято в TUI (собственный интерфейс подтверждения) или внешним `confirmation_response` (см. ниже). В любом случае все наблюдатели видят окончательный результат.

## Схема входных команд

Две формы команд принимаются через `--input-file`:

```jsonc
// Submit a user message into the prompt queue
{ "type": "submit", "text": "What does this function do?" }

// Reply to a pending control_request
{ "type": "confirmation_response", "request_id": "...", "allowed": true }
```

Поведение:

- Команды `submit` ставятся в очередь. Если TUI занят ответом, они автоматически повторяются при следующем переходе TUI в состояние простоя.
- Команды `confirmation_response` отправляются немедленно и никогда не ставятся в очередь, поскольку вызов инструмента блокирует выполнение, и ответ должен достичь обработчика `onConfirm` без ожидания предыдущих `submit`.
- Какая сторона первой одобрит инструмент, та и побеждает; запоздалый ответ другой стороны безвредно игнорируется.
- Строки, которые не удаётся разобрать как JSON, логируются и пропускаются — они не останавливают наблюдатель.

## Примечания по задержке

Входной файл отслеживается с помощью `fs.watchFile` с интервалом опроса 500 мс, поэтому максимальная задержка обмена для удалённого `submit` составляет около полсекунды. Это сделано намеренно: опрос переносим между платформами и файловыми системами (включая macOS и сетевые монтирования) и соответствует типичному темпу взаимодействия с человеком, на который нацелена эта функция. Канал вывода не использует опрос — события записываются синхронно по мере их отправки TUI.

## Режимы сбоев

- **Некорректный fd.** Если файловый дескриптор, переданный в `--json-fd`, не открыт или является одним из 0/1/2, TUI выводит предупреждение в `stderr` и продолжает работу без включённого двойного вывода.
- **Некорректный путь.** Если файл, переданный в `--json-file`, не удаётся открыть, TUI выводит предупреждение и продолжает работу без двойного вывода.
- **Отключение потребителя.** Если читатель на другом конце канала исчезает (`EPIPE`), мост бесшумно отключается, а TUI продолжает работу. Повторные попытки не выполняются.
- **Переполнение буфера FIFO.** При записи в FIFO без подключённого читателя события буферизируются в канале ядра (~64 КБ в Linux) и в Node.js WriteStream. Как только канал заполняется или внутренний буфер превышает 1 МБ, мост отключается и закрывает файловый дескриптор. В этом случае `session_end` не отправляется — потребители должны рассматривать закрытый поток без `session_end` как аварийное завершение. TUI продолжает работу в обычном режиме.
- **Исключение адаптера.** Любое исключение, возникшее при отправке события, перехватывается, логируется и отключает мост. TUI никогда не аварийно завершается из-за сбоя двойного вывода.

## Пример запуска

Типичный родительский процесс, встраивающий Qwen Code, запускает его с обоими каналами:

```ts
import { spawn } from 'node:child_process';
import { openSync } from 'node:fs';

const eventsFd = openSync('/tmp/qwen-events.jsonl', 'w');
const child = spawn(
  'qwen',
  ['--json-fd', '3', '--input-file', '/tmp/qwen-input.jsonl'],
  { stdio: ['inherit', 'inherit', 'inherit', eventsFd] },
);
```
TUI по-прежнему владеет терминалом пользователя через stdio 0/1/2, а встраивающая программа читает структурированные события из файла, ассоциированного с fd 3, и отправляет команды, добавляя строки JSONL в `/tmp/qwen-input.jsonl`.

## Конфигурация через настройки

Для долгоживущих встраивающих программ часто неудобно передавать флаги CLI при каждом запуске. Те же самые каналы можно настроить в `settings.json` по ключу верхнего уровня `dualOutput`:

```jsonc
// ~/.qwen/settings.json  (на уровне пользователя)
// или <рабочая_область>/.qwen/settings.json  (на уровне рабочей области)
{
  "dualOutput": {
    "jsonFile": "/tmp/qwen-events.jsonl",
    "inputFile": "/tmp/qwen-input.jsonl",
  },
}
```

Правила приоритета:

- Флаг CLI **перевешивает** настройки. Передача `--json-file /foo` в командной строке отменяет `dualOutput.jsonFile` в настройках.
- У `--json-fd` нет аналога в настройках — передача fd — это вопрос времени запуска, который нельзя статически объявить.
- Если нет ни флага, ни настройки, двойной вывод остаётся отключённым (как и сегодня по умолчанию).

Флаг `requiresRestart: true` означает, что изменения вступают в силу только при следующем запуске Qwen Code, поскольку мост создаётся однократно во время старта.

## Рабочие демонстрации

Каждый скрипт ниже готов к копированию и вставке. Начните с POC&nbsp;1, чтобы убедиться, что сборка поддерживает двойной вывод; POC&nbsp;4 наиболее близок к реальной интеграции с IDE-расширением.

### POC 1 — наблюдение за потоком событий

Посмотрите на каждое структурированное событие, которое TUI генерирует при обычном использовании человеком:

```bash
# Терминал A
mkfifo /tmp/qwen-events.jsonl
cat /tmp/qwen-events.jsonl | jq -c 'select(.type != "stream_event") | {type, subtype}'

# Терминал B
qwen --json-file /tmp/qwen-events.jsonl
# ...затем общайтесь как обычно; терминал A показывает в реальном времени
# жизненный цикл session_start, user/assistant/result/control_request.
```

Ожидаемая первая строка в терминале A:

```json
{ "type": "system", "subtype": "session_start" }
```

### POC 2 — ввод подсказок извне

Управляйте TUI из второго терминала, не касаясь клавиатуры первого:

```bash
# Терминал A
touch /tmp/qwen-in.jsonl
qwen --input-file /tmp/qwen-in.jsonl

# Терминал B — TUI реагирует так, как будто вы набрали текст
echo '{"type":"submit","text":"список файлов в текущей папке"}' \
  >> /tmp/qwen-in.jsonl
```

### POC 3 — удалённый мост разрешений для инструментов

Подтверждайте или отклоняйте вызовы инструментов из отдельного процесса:

```bash
# Терминал A — наблюдаем за control_requests
mkfifo /tmp/qwen-out.jsonl
touch /tmp/qwen-in.jsonl
(cat /tmp/qwen-out.jsonl \
  | jq -c 'select(.type == "control_request")') &

# Терминал B
qwen --json-file /tmp/qwen-out.jsonl --input-file /tmp/qwen-in.jsonl
# Попросите Qwen сделать что-то, что требует одобрения, например,
# "выполни `ls -la /tmp`". В терминале A появится control_request.
# Скопируйте request_id, затем в третьем терминале:
echo '{"type":"confirmation_response","request_id":"<вставьте-id>","allowed":true}' \
  >> /tmp/qwen-in.jsonl
# Окно подтверждения в TUI закроется, и инструмент выполнится.
```

Если вы ответите с неизвестным `request_id`, мост выдаёт `control_response` с `subtype: "error"` в выходной канал, чтобы ваша программа-потребитель могла залогировать его или повторить попытку:

```json
{
  "type": "control_response",
  "response": {
    "subtype": "error",
    "request_id": "...",
    "error": "неизвестный request_id (уже обработан, отменён или никогда не выдавался)"
  }
}
```

### POC 4 — встраивающая программа на Node.js (аналог IDE)

Наиболее реалистичная форма: родительский процесс запускает Qwen Code, следит за событиями и вводит подсказки по своему расписанию.

```ts
// demo-embedder.ts
import { spawn } from 'node:child_process';
import { appendFileSync, createReadStream, writeFileSync } from 'node:fs';
import { createInterface } from 'node:readline';
import { tmpdir } from 'node:os';
import { join } from 'node:path';

const events = join(tmpdir(), `qwen-events-${process.pid}.jsonl`);
const input = join(tmpdir(), `qwen-input-${process.pid}.jsonl`);
writeFileSync(events, '');
writeFileSync(input, '');

const child = spawn('qwen', ['--json-file', events, '--input-file', input], {
  stdio: 'inherit',
});

// Следим за выходным каналом. В продакшене вы бы использовали
// настоящий tail со смещением по байтам; здесь для краткости
// перечитываем с 0.
const rl = createInterface({
  input: createReadStream(events, { encoding: 'utf8' }),
});
rl.on('line', (line) => {
  if (!line.trim()) return;
  const ev = JSON.parse(line);
  if (ev.type === 'system' && ev.subtype === 'session_start') {
    console.log('[embedder] рукопожатие:', {
      protocol_version: ev.data.protocol_version,
      version: ev.data.version,
      supported_events: ev.data.supported_events,
    });
    // Определяем возможности перед использованием
    if (ev.data.supported_events.includes('control_request')) {
      console.log('[embedder] плоскость управления разрешениями доступна');
    }
  }
  if (ev.type === 'assistant') {
    console.log(
      '[embedder] завершён ход ассистента, токенов =',
      ev.message.usage?.output_tokens,
    );
  }
  if (ev.type === 'system' && ev.subtype === 'session_end') {
    console.log('[embedder] сессия завершена корректно');
  }
});

// Через 2 с отправляем подсказку, как если бы её ввёл пользователь
setTimeout(() => {
  appendFileSync(
    input,
    JSON.stringify({ type: 'submit', text: 'привет от встраивающей программы' }) + '\n',
  );
}, 2000);

child.on('exit', () => process.exit(0));
```
Запустите с помощью:

```bash
npx tsx demo-embedder.ts
# Qwen Code TUI открывается в текущем терминале; embedder логирует
# события handshake + turn-end + session_end в stdout родительского процесса.
```

### POC 5 — обнаружение возможностей через рукопожатие

Старые версии Qwen Code не отправляют `protocol_version`. Считайте это поле необязательным и используйте определение возможностей:

```ts
rl.on('line', (line) => {
  const ev = JSON.parse(line);
  if (ev.type === 'system' && ev.subtype === 'session_start') {
    const v = ev.data?.protocol_version ?? 0;
    if (v < 1) {
      console.error(
        'qwen-code dual output is present but protocol < 1; ' +
          'falling back to best-effort behavior',
      );
    } else {
      console.log('qwen-code dual output protocol v' + v);
    }
  }
});
```

### POC 6 — session_end как сигнал чистого завершения

```ts
rl.on('line', (line) => {
  const ev = JSON.parse(line);
  if (ev.type === 'system' && ev.subtype === 'session_end') {
    console.log('[embedder] clean shutdown, session', ev.data.session_id);
    // Сбросить метрики, закрыть WebSockets и т.д.
  }
});
```

Если TUI падает до `session_end`, поток вывода закрывается (при следующей записи возникает `EPIPE`); встраиваемые системы должны обрабатывать оба пути.

### POC 7 — тесты на сбои (доказывают, что флаги никогда не ломают TUI)

```bash
qwen --json-fd 1
# stderr: "Предупреждение: двойной вывод отключён — ..."
# TUI по-прежнему запускается нормально.

qwen --json-fd 9999
# stderr: "Предупреждение: двойной вывод отключён — fd 9999 не открыт"
# TUI по-прежнему запускается нормально.

qwen --json-fd 3 --json-file /tmp/x.jsonl
# yargs отвергает: "--json-fd и --json-file взаимоисключающие."
# Процесс завершается до запуска TUI.

qwen --json-file /nonexistent/dir/x.jsonl
# предупреждение в stderr; TUI всё равно запускается.
```

## Отношение к Claude Code

Claude Code предоставляет аналогичный формат событий stream-json с флагами `--print --output-format stream-json`, но только в неинтерактивном режиме — у него нет эквивалента одновременного запуска TUI и структурированного бокового канала. Dual Output заполняет этот пробел.

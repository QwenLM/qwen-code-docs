# Ревью кода

> Проверяйте изменения кода на корректность, безопасность, производительность и качество с помощью команды `/review`.

## Быстрый старт

```bash
# Ревью локальных незакоммиченных изменений
/review

# Ревью pull request (по номеру или URL)
/review 123
/review https://github.com/org/repo/pull/123

# Ревью и публикация inline-комментариев в PR
/review 123 --comment

# Ревью конкретного файла
/review src/utils/auth.ts
```

Если нет незакоммиченных изменений, `/review` сообщит об этом и остановится — агенты не запускаются.

## Как это работает

Команда `/review` запускает многоэтапный пайплайн:

```
Шаг 1: Определение области (локальный diff / worktree PR / файл)
Шаг 2: Загрузка правил ревью проекта
Шаг 3: 9 параллельных агентов ревью                          [9 вызовов LLM]
           |-- Агент 1: Корректность
           |-- Агент 2: Безопасность
           |-- Агент 3: Качество кода
           |-- Агент 4: Производительность и эффективность
           |-- Агент 5: Покрытие тестами
           |-- Агент 6: Ненаправленный аудит (3 персоны: 6a/6b/6c)
           '-- Агент 7: Сборка и тесты (запускает shell-команды)
Шаг 4: Дедупликация --> Пакетная проверка --> Агрегация       [1 вызов LLM]
Шаг 5: Итеративный обратный аудит (1-3 раунда, поиск пробелов) [1-3 вызова LLM]
Шаг 6: Предоставление результатов + вердикт
Шаг 7: Отправка ревью PR (inline-комментарии, если запрошено)
Шаг 8: Сохранение отчета + инкрементальный кэш
Шаг 9: Очистка (удаление worktree + временных файлов)
```

### Агенты ревью

| Агент                             | Фокус                                                                                       |
| --------------------------------- | ------------------------------------------------------------------------------------------- |
| Агент 1: Корректность              | Логические ошибки, граничные случаи, обработка null, состояния гонки, типобезопасность                       |
| Агент 2: Безопасность                 | Инъекции, XSS, SSRF, обход аутентификации, утечка чувствительных данных                                  |
| Агент 3: Качество кода             | Согласованность стиля, нейминг, дублирование, мертвый код                                           |
| Агент 4: Производительность и эффективность | N+1 запросы, утечки памяти, ненужные ре-рендеры, размер бандла                              |
| Агент 5: Покрытие тестами            | Непокрытые тестами пути в diff, отсутствующее покрытие веток, слабые ассерты                   |
| Агент 6: Ненаправленный аудит         | 3 параллельные персоны (атакующий / дежурный в 3 часа ночи / мейнтейнер) — выявляет междисциплинарные проблемы |
| Агент 7: Сборка и тесты             | Запускает команды сборки и тестов, сообщает о сбоях                                              |

Все агенты работают параллельно (Агент 6 запускает 3 варианта персон одновременно, в итоге 9 параллельных задач для ревью в том же репозитории). Результаты Агентов 1-6 проверяются в **одном проходе пакетной проверки** (один агент проверяет все результаты сразу, фиксируя стоимость проверки независимо от количества находок). После проверки запускается **итеративный обратный аудит** на 1-3 раунда поиска пробелов — каждый раунд получает накопительный список находок из предыдущих раундов, поэтому последующие раунды фокусируются на том, что осталось не обнаруженным. Цикл останавливается, как только раунд возвращает "No issues found", или после 3 раундов (жесткий лимит). Находки обратного аудита пропускают проверку (агент уже имеет полный контекст) и включаются как результаты с высокой степенью достоверности.

## Уровни критичности

| Критичность         | Значение                                                             | Публикуется как комментарий к PR?      |
| ---------------- | ------------------------------------------------------------------- | -------------------------- |
| **Критический**     | Необходимо исправить перед мержем (баги, безопасность, потеря данных, сбои сборки) | Да (только с высокой достоверностью) |
| **Рекомендация**   | Рекомендуемое улучшение                                             | Да (только с высокой достоверностью) |
| **Желательно** | Опциональная оптимизация                                               | Нет (только в терминале)         |

Результаты с низкой достоверностью отображаются в отдельной секции "Needs Human Review" в терминале и никогда не публикуются как комментарии к PR.

## Изоляция worktree

При ревью PR команда `/review` создает временный git worktree (`.qwen/tmp/review-pr-<number>`) вместо переключения текущей ветки. Это означает:

- Ваше рабочее дерево, проиндексированные изменения и текущая ветка **никогда не затрагиваются**
- Зависимости устанавливаются в worktree (`npm ci` и т.д.), чтобы сборка/тесты работали
- Команды сборки и тестов запускаются изолированно, не загрязняя локальный кэш сборки
- Если что-то пойдет не так, ваша среда не пострадает — просто удалите worktree
- Worktree автоматически очищается после завершения ревью
- Если ревью прервано (Ctrl+C, сбой), следующий `/review` для того же PR автоматически очистит устаревший worktree перед началом работы
- Отчеты о ревью и кэш сохраняются в главную директорию проекта (не в worktree)

## Ревью PR из другого репозитория

Вы можете ревьюить PR из других репозиториев, передав полный URL:

```bash
/review https://github.com/other-org/other-repo/pull/456
```

Это выполняется в **облегченном режиме** — без worktree, без сборки/тестов. Ревью основано только на тексте diff (полученном через GitHub API). Комментарии к PR все еще могут быть опубликованы, если у вас есть доступ на запись.

| Возможность                                                 | Тот же репозиторий | Другой репозиторий                    |
| ---------------------------------------------------------- | --------- | ----------------------------- |
| Ревью LLM (Агенты 1-6 + проверка + итеративный обратный аудит) | ✅        | ✅                            |
| Агент 7: Сборка и тесты                                      | ✅        | ❌ (нет локальной кодовой базы)        |
| Межфайловый анализ влияния                                 | ✅        | ❌                            |
| Inline-комментарии PR                                         | ✅        | ✅ (если есть доступ на запись) |
| Инкрементальный кэш ревью                                   | ✅        | ❌                            |

## Inline-комментарии в PR

Используйте `--comment`, чтобы публиковать результаты прямо в PR:

```bash
/review 123 --comment
```

Или, после запуска `/review 123`, введите `post comments`, чтобы опубликовать результаты без повторного запуска ревью.

**Что публикуется:**

- Результаты Critical и Suggestion с высокой достоверностью в виде inline-комментариев к конкретным строкам
- Для вердиктов Approve/Request changes: саммари ревью с вердиктом
- Для вердикта Comment со всеми опубликованными inline-комментариями: отдельное саммари не выводится (inline-комментариев достаточно)
- Футер с указанием модели в каждом комментарии (например, _— qwen3-coder via Qwen Code /review_)

**Что остается только в терминале:**

- Результаты Nice to have
- Результаты с низкой достоверностью

**Собственные PR:** GitHub не позволяет отправлять ревью `APPROVE` или `REQUEST_CHANGES` на собственные pull request — оба варианта завершаются ошибкой HTTP 422. Когда `/review` обнаруживает, что автор PR совпадает с текущим аутентифицированным пользователем, он автоматически понижает событие API до `COMMENT` независимо от вердикта, чтобы отправка прошла успешно. В терминале все еще отображается честный вердикт ("Approve" / "Request changes" / "Comment") — нейтрализуется только событие ревью на стороне GitHub. Сами результаты по-прежнему появляются как inline-комментарии к конкретным строкам, поэтому содержательная обратная связь не меняется.

**Повторное ревью PR с предыдущими комментариями Qwen Code:** когда `/review` запускается для PR, в котором уже есть предыдущие комментарии ревью Qwen Code, он классифицирует их перед публикацией новых. Только **перекрытие по одной строке** (существующий комментарий на той же `(path, line)`, что и новая находка) требует подтверждения — это тот случай, когда вы увидите визуальный дубликат на одной строке кода. Комментарии из старых коммитов, комментарии с ответами (считаются решенными) и комментарии, которые просто не пересекаются ни с одной новой находкой, молча пропускаются, с записью в логе терминала, чтобы вы знали, что было отфильтровано.

**Проверка статуса CI / сборки перед APPROVE:** если вердикт — "Approve", `/review` запрашивает check-runs и статусы коммитов PR перед отправкой. Если какая-либо проверка не пройдена (или все проверки все еще в ожидании), событие API автоматически понижается с `APPROVE` до `COMMENT`, а тело ревью объясняет причину. Обоснование: ревью LLM читает код статически и не может увидеть ошибки тестов во время выполнения; одобрение при красном CI было бы вводящим в заблуждение. Inline-результаты при этом публикуются без изменений. Если вы все равно хотите одобрить (например, при известном нестабильном сбое CI), отправьте одобрение GitHub вручную после проверки.

## Последующие действия

После ревью контекстные подсказки появляются в виде ghost text. Нажмите Tab, чтобы принять:

| Состояние после ревью                 | Подсказка                | Что происходит                            |
| ---------------------------------- | ------------------ | --------------------------------------- |
| Локальное ревью с неисправленными результатами | `fix these issues` | LLM интерактивно исправляет каждую находку    |
| Ревью PR с результатами            | `post comments`    | Публикует inline-комментарии в PR (без повторного ревью) |
| Ревью PR, ноль результатов           | `post comments`    | Одобрет PR на GitHub (LGTM)        |
| Локальное ревью, все чисто            | `commit`           | Коммитит ваши изменения                    |

Примечание: `fix these issues` доступно только для локальных ревью. Для ревью PR worktree очищается после завершения, поэтому интерактивное исправление после ревью невозможно — используйте `--comment` или `post comments` для публикации результатов.

## Правила ревью проекта

Вы можете настроить критерии ревью для каждого проекта. `/review` читает правила из следующих файлов (по порядку):

1. `.qwen/review-rules.md` (нативный для Qwen Code)
2. `.github/copilot-instructions.md` (предпочтительный) или `copilot-instructions.md` (резервный — загружается только один, не оба)
3. `AGENTS.md` — секция `## Code Review`
4. `QWEN.md` — секция `## Code Review`

Правила внедряются в агенты ревью LLM (1-6) как дополнительные критерии. Для ревью PR правила читаются из **базовой ветки**, чтобы предотвратить внедрение правил обхода злонамеренным PR.

Пример `.qwen/review-rules.md`:

```markdown
# Правила ревью

- Все API endpoints должны проверять аутентификацию
- Запросы к базе данных должны использовать параметризованные выражения
- React компоненты не должны использовать inline стили
- Сообщения об ошибках не должны раскрывать внутренние пути
```

## Инкрементальное ревью

При ревью PR, который уже ревьюился ранее, `/review` проверяет только изменения с момента последнего ревью:

```bash
# Первое ревью — полное ревью, создается кэш
/review 123

# PR обновлен новыми коммитами — ревьюятся только новые изменения
/review 123
```

### Кросс-модельное ревью

Если вы переключите модель (через `/model`) и повторно ревьюите тот же PR, `/review` обнаружит изменение модели и запустит полное ревью вместо пропуска:

```bash
# Ревью с моделью A
/review 123

# Переключение модели
/model

# Повторное ревью — полное ревью с моделью B (не пропущено)
/review 123
# → "Previous review used qwen3-coder. Running full review with gpt-4o for a second opinion."
```

Кэш хранится в `.qwen/review-cache/` и отслеживает как SHA коммита, так и ID модели. Убедитесь, что эта директория есть в вашем `.gitignore` (также подойдет более широкое правило вроде `.qwen/*`). Если закоммиченный кэш был удален при rebase, выполняется полное ревью.

## Отчеты о ревью

Для ревью в том же репозитории результаты сохраняются как Markdown файл в директории `.qwen/reviews/` вашего проекта (облегченные кросс-репозиторные ревью пропускают сохранение отчетов):

```
.qwen/reviews/2026-04-06-143022-pr-123.md
.qwen/reviews/2026-04-06-150510-local.md
```

Отчеты включают: временную метку, статистику diff, результаты сборки/тестов, все результаты со статусом проверки и вердикт.

## Межфайловый анализ влияния

Когда изменения кода модифицируют экспортируемые функции, классы или интерфейсы, агенты ревью автоматически ищут всех вызывающих и проверяют совместимость:

- Изменения количества/типа параметров
- Изменения типа возвращаемого значения
- Удаленные или переименованные публичные методы
- Критические изменения API (Breaking changes)

Для больших diff (>10 измененных символов) анализ приоритезирует функции с измененными сигнатурами.

## Эффективность использования токенов

Пайплайн ревью использует ограниченное количество вызовов LLM независимо от того, сколько результатов было найдено:

| Этап                            | Вызовов LLM         | Примечания                                               |
| -------------------------------- | ----------------- | --------------------------------------------------- |
| Агенты ревью (Шаг 3)           | 9 (или 8)          | Работают параллельно; Агент 7 пропускается в кросс-репозиторном режиме |
| Пакетная проверка (Шаг 4)      | 1                 | Один агент проверяет все результаты сразу          |
| Итеративный обратный аудит (Шаг 5) | 1-3               | Цикл до "No issues found" или лимита в 3 раунда        |
| **Итого**                        | **11-13 (10-12)** | Тот же репозиторий: 11-13; кросс-репозиторий: 10-12 (без Агента 7)    |

Большинство PR сходятся к нижней границе диапазона (1 раунд обратного аудита); лимит предотвращает неконтролируемый рост затрат на патологических случаях.

## Что НЕ помечается

Ревью намеренно исключает:

- Существующие проблемы в неизмененном коде (фокус только на diff)
- Стиль или форматирование, которые авто-нормализуются форматтером, или нейминг, соответствующий конвенциям вашей кодовой базы — но НЕ существенные проблемы, которые пометил бы линтер или type checker (неиспользуемые переменные, недостижимый код, ошибки типов), которые входят в скоуп
- Субъективные предложения "consider doing X" без реальной проблемы
- Незначительный рефакторинг, который не исправляет баг или риск
- Отсутствие документации, если логика не является по-настоящему запутанной
- Проблемы, уже обсужденные в существующих комментариях PR (избегает дублирования человеческой обратной связи)

## Философия дизайна

> **Тишина лучше шума.** Каждый комментарий должен стоить времени читателя.

- Если не уверены, является ли что-то проблемой → не сообщайте об этом
- Одинаковый паттерн в N файлах → агрегируется в одну находку
- Комментарии к PR только с высокой достоверностью
- Косметический стиль/форматирование, соответствующее конвенциям кодовой базы, исключается

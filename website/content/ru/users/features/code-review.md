# Code Review

> Проверяйте изменения кода на корректность, безопасность, производительность и качество с помощью `/review`.

## Быстрый старт

```bash
# Проверить локальные незакоммиченные изменения
/review

# Проверить пул-реквест (по номеру или URL)
/review 123
/review https://github.com/org/repo/pull/123

# Проверить и опубликовать встроенные комментарии в PR
/review 123 --comment

# Проверить конкретный файл
/review src/utils/auth.ts
```

Если незакоммиченных изменений нет, `/review` сообщит об этом и остановится — агенты не запускаются.

## Как это работает

Команда `/review` выполняет многоэтапный конвейер:

```
Шаг 1:  Определить область (локальный diff / worktree PR / файл)
Шаг 2:  Загрузить правила ревью проекта
Шаг 3:  Запустить детерминированный анализ (линтер, проверка типов)    [нулевая стоимость LLM]
Шаг 4:  9 параллельных агентов ревью                                    [9 вызовов LLM]
           |-- Агент 1: Корректность
           |-- Агент 2: Безопасность
           |-- Агент 3: Качество кода
           |-- Агент 4: Производительность и эффективность
           |-- Агент 5: Покрытие тестами
           |-- Агент 6: Ненаправленный аудит (3 персоны: 6a/6b/6c)
           '-- Агент 7: Сборка и тесты (выполняет shell-команды)
Шаг 5:  Дедупликация --> Пакетная верификация --> Агрегация             [1 вызов LLM]
Шаг 6:  Итеративный обратный аудит (1-3 раунда, поиск пробелов)        [1-3 вызова LLM]
Шаг 7:  Представить результаты + вердикт
Шаг 8:  Автоисправление (подтверждается пользователем, опционально)
Шаг 9:  Опубликовать встроенные комментарии в PR (если запрошено)
Шаг 10: Сохранить отчёт + инкрементальный кеш
Шаг 11: Очистка (удалить worktree + временные файлы)
```

### Агенты ревью

| Агент                             | Фокус                                                                                          |
| --------------------------------- | ---------------------------------------------------------------------------------------------- |
| Агент 1: Корректность             | Логические ошибки, граничные случаи, обработка null, состояния гонки, типобезопасность          |
| Агент 2: Безопасность             | Инъекции, XSS, SSRF, обход аутентификации, раскрытие чувствительных данных                     |
| Агент 3: Качество кода            | Стилистическая согласованность, именование, дублирование, мёртвый код                          |
| Агент 4: Производительность и эффективность | N+1 запросы, утечки памяти, ненужные повторные рендеры, размер бандла                   |
| Агент 5: Покрытие тестами         | Нетстированные пути в diff, пропущенное покрытие ветвлений, слабые проверки                    |
| Агент 6: Ненаправленный аудит     | 3 параллельные персоны (атакующий / дежурный в 3 часа ночи / мейнтейнер) — выявляет кросс-измерные проблемы |
| Агент 7: Сборка и тесты           | Выполняет команды сборки и тестирования, сообщает об ошибках                                   |

Все агенты работают параллельно (Агент 6 запускает 3 варианта персон одновременно, всего 9 параллельных задач для ревью внутри того же репозитория). Результаты агентов 1-6 верифицируются в **едином проходе пакетной верификации** (один агент проверяет все результаты сразу, стоимость верификации фиксирована независимо от количества находок). После верификации **итеративный обратный аудит** выполняет 1-3 раунда поиска пробелов — каждый следующий раунд получает совокупный список находок из предыдущих, поэтому последующие раунды фокусируются на том, что осталось необнаруженным. Цикл останавливается, как только раунд возвращает «Проблем не найдено», или после 3 раундов (жёсткое ограничение). Находки обратного аудита пропускают верификацию (агент уже имеет полный контекст) и считаются результатами с высокой уверенностью.

## Детерминированный анализ

Перед запуском LLM-агентов `/review` автоматически выполняет существующие линтеры и проверщики типов вашего проекта:

| Язык                | Обнаруживаемые инструменты                                                |
| ------------------- | ------------------------------------------------------------------------- |
| TypeScript/JavaScript | `tsc --noEmit`, `npm run lint`, `eslint`                                 |
| Python              | `ruff`, `mypy`, `flake8`                                                  |
| Rust                | `cargo clippy`                                                            |
| Go                  | `go vet`, `golangci-lint`                                                 |
| Java                | `mvn compile`, `checkstyle`, `spotbugs`, `pmd`                            |
| C/C++               | `clang-tidy` (если доступен `compile_commands.json`)                       |
| Другие              | Автоматически обнаруживается из конфигурации CI (`.github/workflows/*.yml`, и т.д.) |

Для проектов, не соответствующих стандартным шаблонам (например, OpenJDK), `/review` читает файлы конфигурации CI, чтобы определить, какие команды линтинга/проверки использует проект. Настройка пользователем не требуется.

Результаты детерминированного анализа помечаются тегом `[linter]` или `[typecheck]` и пропускают LLM-верификацию — они являются истиной.

- **Ошибки** → Критическая важность
- **Предупреждения** → Желательно (только в терминале, не публикуются как PR-комментарии)

Если инструмент не установлен или превышает тайм-аут, он пропускается с информационным сообщением.

## Уровни важности

| Важность         | Значение                                                                            | Публикуется как комментарий PR? |
| ---------------- | ----------------------------------------------------------------------------------- | ------------------------------- |
| **Критическая**  | Обязательно исправить перед слиянием (ошибки, безопасность, потеря данных, сбои сборки) | Да (только с высокой уверенностью) |
| **Рекомендация** | Рекомендуемое улучшение                                                             | Да (только с высокой уверенностью) |
| **Желательно**   | Опциональная оптимизация                                                            | Нет (только в терминале)        |

Находки с низкой уверенностью выводятся в отдельный раздел «Требуется проверка человеком» в терминале и никогда не публикуются как PR-комментарии.

## Автоисправление

После представления результатов `/review` предлагает автоматически применить исправления для критических находок и рекомендаций, для которых есть чёткие решения:

```
Обнаружено 3 проблемы с предложениями по автоисправлению. Применить автоисправления? (y/n)
```

- Исправления применяются с помощью инструмента `edit` (целевые замены, а не перезапись файла целиком)
- После исправлений для каждого файла запускаются линтеры, чтобы убедиться, что не появилось новых проблем
- Для PR-ревью исправления автоматически коммитятся и пушатся из worktree — ваше рабочее дерево остаётся чистым
- Находки «Желательно» и с низкой уверенностью никогда не исправляются автоматически
- Отправка PR-ревью всегда использует **вердикт до исправления** (например, «Запросить изменения»), так как удалённый PR ещё не обновлён до завершения пуша автоисправлений

## Изоляция worktree

При проверке PR `/review` создаёт временный git worktree (`.qwen/tmp/review-pr-<number>`) вместо переключения вашей текущей ветки. Это означает:

- Ваше рабочее дерево, проиндексированные изменения и текущая ветка **никогда не затрагиваются**
- Зависимости устанавливаются в worktree (`npm ci` и т.д.), так что линтинг и сборка/тесты работают
- Команды сборки и тестирования выполняются изолированно, не загрязняя локальный кеш сборки
- Если что-то пошло не так, ваша среда не страдает — просто удалите worktree
- Worktree автоматически очищается после завершения ревью
- Если ревью было прервано (Ctrl+C, сбой), следующий `/review` того же PR автоматически удалит устаревший worktree перед началом заново
- Отчёты и кеш ревью сохраняются в основную директорию проекта (не в worktree)

## Кросс-репозиторное ревью PR

Вы можете проверять PR из других репозиториев, передав полный URL:

```bash
/review https://github.com/other-org/other-repo/pull/456
```

Это выполняется в **облегчённом режиме** — без worktree, линтера, сборки/тестов и автоисправления. Ревью основано только на тексте diff (полученном через GitHub API). Комментарии PR всё ещё могут быть опубликованы, если у вас есть права на запись.

| Возможность                                                | Тот же репозиторий | Другой репозиторий              |
| ---------------------------------------------------------- | ------------------ | ------------------------------- |
| LLM-ревью (Агенты 1-6 + верификация + итеративный обратный аудит) | ✅                | ✅                              |
| Агент 7: Сборка и тесты                                    | ✅                 | ❌ (нет локальной кодовой базы) |
| Детерминированный анализ (линтер/проверка типов)           | ✅                 | ❌                              |
| Анализ влияния на другие файлы                             | ✅                 | ❌                              |
| Автоисправление                                            | ✅                 | ❌                              |
| Встроенные комментарии PR                                  | ✅                 | ✅ (если есть права на запись)  |
| Инкрементальный кеш ревью                                  | ✅                 | ❌                              |

## Встроенные комментарии PR

Используйте `--comment`, чтобы публиковать находки непосредственно в PR:

```bash
/review 123 --comment
```

Или после выполнения `/review 123` введите `post comments`, чтобы опубликовать результаты без повторного ревью.

**Что публикуется:**

- Высокоуверенные критические находки и рекомендации в виде встроенных комментариев на конкретных строках
- Для вердиктов «Одобрить»/«Запросить изменения»: сводка ревью с вердиктом
- Для вердикта «Комментировать» после публикации всех встроенных комментариев: отдельная сводка не нужна (встроенные комментарии достаточны)
- Нижний колонтитул с указанием модели на каждом комментарии (например, _— qwen3-coder via Qwen Code /review_)

**Что остаётся только в терминале:**

- Находки «Желательно» (включая предупреждения линтера)
- Находки с низкой уверенностью

**Собственные PR:** GitHub не позволяет отправлять ревью с типом `APPROVE` или `REQUEST_CHANGES` на собственный пул-реквест — оба завершаются ошибкой HTTP 422. Когда `/review` обнаруживает, что автор PR совпадает с текущим аутентифицированным пользователем, он автоматически понижает тип события API до `COMMENT` независимо от вердикта, так что отправка всё равно проходит успешно. В терминале по-прежнему отображается честный вердикт («Одобрить» / «Запросить изменения» / «Комментировать») — только на стороне GitHub событие ревью нейтрализуется. Фактические находки по-прежнему отображаются как встроенные комментарии на конкретных строках, поэтому содержательная обратная связь остаётся неизменной.

**Повторное ревью PR с предыдущими комментариями Qwen Code:** когда `/review` запускается на PR, который уже содержит предыдущие комментарии ревью Qwen Code, он классифицирует их перед публикацией новых. Только **пересечение на той же строке** (существующий комментарий на том же `(путь, строка)`, что и новая находка) требует подтверждения — это тот случай, когда вы увидите визуальный дубликат на одной строке кода. Комментарии от старых коммитов, комментарии с ответами (считаются разрешёнными) и комментарии, которые просто не пересекаются с новыми находками, молча пропускаются, с записью в терминал, чтобы вы знали, что было отфильтровано.

**Проверка CI/статуса сборки перед APPROVE:** если вердикт «Одобрить», `/review` запрашивает статусы check-runs и commit статусы PR перед отправкой. Если какая-то проверка завершилась неудачей (или все проверки всё ещё ожидаются), событие API автоматически понижается с `APPROVE` до `COMMENT`, с пояснением в теле ревью. Обоснование: LLM-ревью читает код статически и не видит ошибок выполнения тестов; одобрение, пока CI красный, было бы вводящим в заблуждение. Встроенные находки публикуются без изменений. Если вы хотите одобрить в любом случае (например, известная нестабильная ошибка CI), отправьте одобрение вручную через GitHub после проверки.

## Действия после ревью

После ревью контекстно-зависимые подсказки отображаются как призрачный текст. Нажмите Tab, чтобы принять:

| Состояние после ревью                     | Подсказка                   | Что происходит                                       |
| ----------------------------------------- | --------------------------- | ---------------------------------------------------- |
| Локальное ревью с неисправленными находками | `fix these issues`          | LLM интерактивно исправляет каждую находку           |
| PR-ревью с находками                      | `post comments`             | Публикует встроенные комментарии PR (без повторного ревью) |
| PR-ревью, ноль находок                    | `post comments`             | Одобряет PR на GitHub (LGTM)                         |
| Локальное ревью, всё чисто                | `commit`                    | Коммитит ваши изменения                              |

Примечание: `fix these issues` доступно только для локальных ревью. Для PR-ревью используйте автоисправление (Шаг 8) — worktree очищается после ревью, поэтому интерактивное исправление после ревью невозможно.

## Правила ревью проекта

Вы можете настроить критерии ревью для каждого проекта. `/review` читает правила из следующих файлов (по порядку):

1. `.qwen/review-rules.md` (родной формат Qwen Code)
2. `.github/copilot-instructions.md` (предпочтительно) или `copilot-instructions.md` (запасной вариант — загружается только один, не оба)
3. `AGENTS.md` — раздел `## Code Review`
4. `QWEN.md` — раздел `## Code Review`

Правила добавляются в LLM-агенты ревью (1-6) как дополнительные критерии. Для PR-ревью правила читаются из **базовой ветки**, чтобы предотвратить внедрение правил обхода через вредоносный PR.

Пример `.qwen/review-rules.md`:

```markdown
# Review Rules

- All API endpoints must validate authentication
- Database queries must use parameterized statements
- React components must not use inline styles
- Error messages must not expose internal paths
```

## Инкрементальное ревью

При проверке PR, который уже был ревьюирован ранее, `/review` анализирует только изменения с момента последнего ревью:

```bash
# Первое ревью — полный анализ, создаётся кеш
/review 123

# PR обновлён новыми коммитами — проверяются только новые изменения
/review 123
```

### Кросс-модельное ревью

Если вы переключаете модель (через `/model`) и повторно проверяете тот же PR, `/review` обнаруживает смену модели и выполняет полное ревью вместо пропуска:

```bash
# Ревью с моделью A
/review 123

# Переключить модель
/model

# Повторное ревью — полный анализ с моделью B (не пропускается)
/review 123
# → "Предыдущее ревью выполнялось с qwen3-coder. Выполняю полное ревью с gpt-4o для второго мнения."
```

Кеш хранится в `.qwen/review-cache/` и отслеживает как SHA коммита, так и ID модели. Убедитесь, что эта директория находится в вашем `.gitignore` (более широкое правило типа `.qwen/*` также подходит). Если кешированный коммит был удалён при перебазировании, выполняется полное ревью.

## Отчёты ревью

Для ревью внутри того же репозитория результаты сохраняются в виде Markdown-файла в директории `.qwen/reviews/` вашего проекта (кросс-репозиторные облегчённые ревью пропускают сохранение отчёта):

```
.qwen/reviews/2026-04-06-143022-pr-123.md
.qwen/reviews/2026-04-06-150510-local.md
```

Отчёты включают: временную метку, статистику diff, результаты детерминированного анализа, все находки со статусом верификации и вердикт.

## Анализ влияния на другие файлы

Когда изменения кода модифицируют экспортируемые функции, классы или интерфейсы, агенты ревью автоматически ищут всех вызывающих и проверяют совместимость:

- Изменения количества/типа параметров
- Изменения типа возвращаемого значения
- Удалённые или переименованные публичные методы
- Ломающие изменения API

Для больших diff (более 10 изменённых символов) приоритет отдаётся функциям с изменённой сигнатурой.

## Эффективность токенов

Конвейер ревью использует ограниченное количество вызовов LLM независимо от количества найденных проблем:

| Этап                              | Вызовов LLM    | Примечания                                               |
| --------------------------------- | -------------- | -------------------------------------------------------- |
| Детерминированный анализ (Шаг 3)  | 0              | Только shell-команды                                     |
| Агенты ревью (Шаг 4)              | 9 (или 8)      | Выполняются параллельно; Агент 7 пропущен в кросс-репозиторном режиме |
| Пакетная верификация (Шаг 5)      | 1              | Один агент проверяет все находки сразу                   |
| Итеративный обратный аудит (Шаг 6)| 1-3            | Цикл до «Проблем не найдено» или ограничения в 3 раунда  |
| **Итого**                         | **11-13 (10-12)** | Внутри репозитория: 11-13; кросс-репозиторный: 10-12 (без Агента 7) |

Большинство PR сходятся к нижней границе диапазона (1 раунд обратного аудита); ограничение предотвращает неконтролируемый рост затрат на патологических случаях.

## Что НЕ отмечается

Ревью намеренно исключает:

- Ранее существовавшие проблемы в неизменённом коде (фокус только на diff)
- Стиль/форматирование/именование, соответствующие соглашениям вашей кодовой базы
- Проблемы, которые бы выявил линтер или проверщик типов (обрабатываются детерминированным анализом)
- Субъективные предложения «подумайте о X» без реальной проблемы
- Незначительный рефакторинг, который не исправляет ошибку или риск
- Отсутствие документации, если только логика не является действительно запутанной
- Проблемы, уже обсуждавшиеся в существующих комментариях PR (избегает дублирования человеческой обратной связи)

## Философия дизайна

> **Тишина лучше шума.** Каждый комментарий должен стоить времени читателя.

- Если не уверены, является ли что-то проблемой → не сообщайте
- Проблемы линтера/проверки типов обрабатываются инструментами, а не LLM-догадками
- Один и тот же шаблон в N файлах → агрегируется в одну находку
- PR-комментарии публикуются только с высокой уверенностью
- Проблемы стиля/форматирования, соответствующие соглашениям кодовой базы, исключаются

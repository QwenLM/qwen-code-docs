# Ревью кода

> Проверка изменений в коде на корректность, безопасность, производительность и качество кода с помощью команды `/review`.

## Быстрый старт

```bash
# Ревью локальных незакоммиченных изменений
/review

# Ревью пул-реквеста (по номеру или URL)
/review 123
/review https://github.com/org/repo/pull/123

# Ревью с публикацией инлайн-комментариев в PR
/review 123 --comment

# Ревью конкретного файла
/review src/utils/auth.ts
```

Если незакоммиченных изменений нет, `/review` сообщит об этом и остановится — агенты не запускаются.

## Как это работает

Команда `/review` выполняет многоэтапный пайплайн:

```
Шаг 1:  Определение области (локальный diff / рабочая папка PR / файл)
Шаг 2:  Загрузка правил ревью проекта
Шаг 3:  Детерминированный анализ (линтер, проверка типов)    [нулевая стоимость LLM]
Шаг 4:  9 параллельных агентов ревью                          [9 вызовов LLM]
           |-- Агент 1: Корректность
           |-- Агент 2: Безопасность
           |-- Агент 3: Качество кода
           |-- Агент 4: Производительность и эффективность
           |-- Агент 5: Покрытие тестами
           |-- Агент 6: Неструктурированный аудит (3 персоны: 6a/6b/6c)
           '-- Агент 7: Сборка и тесты (выполняет shell-команды)
Шаг 5:  Дедупликация --> Пакетная верификация --> Агрегация   [1 вызов LLM]
Шаг 6:  Итеративный обратный аудит (1–3 раунда, поиск пробелов) [1–3 вызова LLM]
Шаг 7:  Вывод результатов + заключение
Шаг 8:  Автоисправление (с подтверждением пользователя, опционально)
Шаг 9:  Публикация инлайн-комментариев в PR (если запрошено)
Шаг 10: Сохранение отчёта + инкрементального кэша
Шаг 11: Очистка (удаление рабочей папки и временных файлов)
```

### Агенты ревью

| Агент                                    | Фокус                                                                                                           |
| ---------------------------------------- | --------------------------------------------------------------------------------------------------------------- |
| Агент 1: Корректность                    | Логические ошибки, граничные случаи, обработка null, состояния гонки, типобезопасность                          |
| Агент 2: Безопасность                    | Инъекции, XSS, SSRF, обход аутентификации, раскрытие чувствительных данных                                     |
| Агент 3: Качество кода                   | Согласованность стиля, именование, дублирование, мёртвый код                                                    |
| Агент 4: Производительность и эффективность | N+1 запросы, утечки памяти, лишние ререндеры, размер бандла                                                   |
| Агент 5: Покрытие тестами                | Непроверенные пути кода в diff, пропущенное покрытие ветвлений, слабые утверждения                              |
| Агент 6: Неструктурированный аудит       | 3 параллельные персоны (атакующий / дежурный в 3 ночи / мейнтейнер) — выявляет кросс-измеренческие проблемы     |
| Агент 7: Сборка и тесты                  | Выполняет команды сборки и тестирования, сообщает об ошибках                                                    |

Все агенты запускаются параллельно (Агент 6 запускает 3 варианта персон одновременно, в сумме 9 параллельных задач для ревью внутри одного репозитория). Результаты Агентов 1–6 верифицируются в одном **пакетном проходе верификации** (один агент проверяет все находки сразу, благодаря чему стоимость верификации остаётся фиксированной независимо от числа находок). После верификации выполняется **итеративный обратный аудит** в 1–3 раунда поиска пробелов — каждый раунд получает накопленный список находок из предыдущих раундов, поэтому последовательные раунды фокусируются на том, что ещё не обнаружено. Цикл останавливается, как только раунд возвращает "Проблем не найдено", или после 3 раундов (жёсткое ограничение). Результаты обратного аудита не проходят верификацию (агент уже имеет полный контекст) и включаются как находки с высокой уверенностью.

## Детерминированный анализ

Перед запуском LLM-агентов `/review` автоматически выполняет существующие в проекте линтеры и проверки типов:

| Язык                  | Обнаруживаемые инструменты                                                           |
| --------------------- | ------------------------------------------------------------------------------------ |
| TypeScript/JavaScript | `tsc --noEmit`, `npm run lint`, `eslint`                                             |
| Python                | `ruff`, `mypy`, `flake8`                                                             |
| Rust                  | `cargo clippy`                                                                       |
| Go                    | `go vet`, `golangci-lint`                                                            |
| Java                  | `mvn compile`, `checkstyle`, `spotbugs`, `pmd`                                       |
| C/C++                 | `clang-tidy` (если доступен `compile_commands.json`)                                 |
| Другие                | Автообнаружение из CI-конфигурации (`.github/workflows/*.yml` и т.п.)                 |

Для проектов, не соответствующих стандартным шаблонам (например, OpenJDK), `/review` читает файлы CI-конфигурации, чтобы определить, какие команды линтинга/проверок использует проект. Настройка от пользователя не требуется.

Результаты детерминированных проверок помечаются тегом `[linter]` или `[typecheck]` и не проходят LLM-верификацию — это истина.

- **Ошибки** → Критическая важность
- **Предупреждения** → Желательно (только в терминале, не публикуются как комментарии PR)

Если инструмент не установлен или истёк тайм-аут, он пропускается с информационным примечанием.

## Уровни важности

| Важность          | Значение                                                        | Публикуется как комментарий PR? |
| ----------------- | --------------------------------------------------------------- | ------------------------------- |
| **Критический**   | Обязательно исправить перед слиянием (баги, безопасность, потеря данных, ошибки сборки) | Да (только с высокой уверенностью) |
| **Рекомендация**  | Рекомендованное улучшение                                       | Да (только с высокой уверенностью) |
| **Желательно**    | Опциональная оптимизация                                        | Нет (только в терминале)       |
Результаты с низкой уверенностью отображаются в отдельном разделе «Требуется проверка человеком» в терминале и никогда не публикуются как комментарии к PR.

## Автоисправление

После представления результатов `/review` предлагает автоматически применить исправления для результатов уровня Critical и Suggestion, имеющих очевидные решения:

```
Found 3 issues with auto-fixable suggestions. Apply auto-fixes? (y/n)
```

- Исправления применяются с помощью инструмента `edit` (целевые замены, а не перезапись полных файлов)
- После исправлений запускаются линтеры для каждого файла, чтобы убедиться, что не появилось новых проблем
- Для ревью PR исправления коммитятся и пушатся из worktree автоматически — ваше рабочее дерево остаётся чистым
- Результаты типа «Nice to have» и с низкой уверенностью никогда не исправляются автоматически
- Отправка ревью PR всегда использует **вердикт до исправления** (например, «Request changes»), поскольку удалённый PR не обновляется до завершения пуша с автоисправлениями

## Изоляция Worktree

При ревью PR `/review` создаёт временный git worktree (`.qwen/tmp/review-pr-<number>`) вместо переключения вашей текущей ветки. Это означает:

- Ваше рабочее дерево, застейдженные изменения и текущая ветка **никогда не затрагиваются**
- Зависимости устанавливаются в worktree (`npm ci` и т.д.), чтобы линтинг и сборка/тесты работали
- Команды сборки и тестов выполняются изолированно, не засоряя локальный кеш сборки
- Если что-то пошло не так, ваше окружение остаётся нетронутым — просто удалите worktree
- Worktree автоматически очищается после завершения ревью
- Если ревью было прервано (Ctrl+C, сбой), следующий вызов `/review` того же PR автоматически очищает устаревший worktree перед началом заново
- Отчёты о ревью и кеш сохраняются в основную директорию проекта (не в worktree)

## Ревью PR из другого репозитория

Вы можете ревьюить PR из других репозиториев, передав полный URL:

```bash
/review https://github.com/other-org/other-repo/pull/456
```

Это работает в **облегчённом режиме** — без worktree, линтера, сборки/тестов и автоисправлений. Ревью основано только на тексте диффа (полученном через GitHub API). Комментарии к PR можно публиковать, если у вас есть права на запись.

| Возможность                                                   | Тот же репозиторий | Другой репозиторий             |
| ------------------------------------------------------------- | ------------------ | ------------------------------ |
| LLM-ревью (Агенты 1–6 + верификация + итеративный обратный аудит) | ✅               | ✅                             |
| Агент 7: Сборка и тестирование                                | ✅                 | ❌ (нет локального кода)       |
| Детерминированный анализ (линтер/проверка типов)              | ✅                 | ❌                             |
| Межфайловый анализ влияния                                    | ✅                 | ❌                             |
| Автоисправление                                               | ✅                 | ❌                             |
| Инлайн-комментарии в PR                                       | ✅                 | ✅ (если есть права на запись) |
| Инкрементальный кеш ревью                                     | ✅                 | ❌                             |

## Инлайн-комментарии в PR

Используйте `--comment`, чтобы опубликовать результаты прямо в PR:

```bash
/review 123 --comment
```

Или после запуска `/review 123` введите `post comments`, чтобы опубликовать результаты без повторного запуска ревью.

**Что публикуется:**

- Результаты уровня Critical и Suggestion с высокой уверенностью как инлайн-комментарии на конкретных строках
- Для вердиктов Approve / Request changes: сводка ревью с вердиктом
- Для вердикта Comment со всеми опубликованными инлайн-комментариями: отдельная сводка не требуется (инлайн-комментариев достаточно)
- Нижний колонтитул с указанием модели на каждом комментарии (например, _— qwen3-coder via Qwen Code /review_)

**Что остаётся только в терминале:**

- Результаты типа «Nice to have» (включая предупреждения линтера)
- Результаты с низкой уверенностью

**Собственные PR:** GitHub не позволяет отправлять ревью с вердиктами `APPROVE` или `REQUEST_CHANGES` на собственный пул-реквест — оба завершаются ошибкой HTTP 422. Когда `/review` обнаруживает, что автор PR совпадает с текущим аутентифицированным пользователем, он автоматически понижает событие API до `COMMENT` независимо от вердикта, чтобы отправка всё равно прошла успешно. В терминале по-прежнему отображается честный вердикт («Approve» / «Request changes» / «Comment») — только событие ревью на стороне GitHub нейтрализуется. Фактические результаты всё равно отображаются как инлайн-комментарии на конкретных строках, так что содержательная обратная связь не меняется.

**Повторное ревью PR с предыдущими комментариями Qwen Code:** когда `/review` запускается на PR, в котором уже есть комментарии ревью Qwen Code, он классифицирует их перед публикацией новых. Только **пересечение по одной строке** (существующий комментарий на той же `(path, line)`, что и новый результат) требует вашего подтверждения — именно в этом случае вы увидите визуальный дубликат на той же строке кода. Комментарии от более старых коммитов, комментарии с ответами (считаются разрешёнными) и комментарии, которые просто не пересекаются ни с одним новым результатом, молча пропускаются с записью в логе терминала, чтобы вы знали, что было отфильтровано.

**Проверка статуса CI / сборки перед APPROVE:** если вердикт — «Approve», `/review` перед отправкой запрашивает check-runs и статусы коммитов PR. Если какая-либо проверка завершилась неудачно (или все проверки всё ещё ожидают выполнения), событие API автоматически понижается с `APPROVE` до `COMMENT`, а в теле ревью объясняется причина. Обоснование: LLM-ревью читает код статически и не может увидеть ошибки выполнения тестов; одобрение, пока CI красный, было бы некорректным. Инлайн-результаты всё равно публикуются без изменений. Если вы всё равно хотите одобрить (например, известная ненадёжная ошибка CI), отправьте одобрение вручную после проверки.
## Последующие действия

После проверки появляются контекстные подсказки в виде призрачного текста. Нажмите Tab, чтобы принять:

| Состояние после проверки                           | Подсказка                    | Что происходит                                      |
| -------------------------------------------------- | ---------------------------- | --------------------------------------------------- |
| Локальная проверка с неисправленными замечаниями   | `fix these issues`           | LLM интерактивно исправляет каждое замечание         |
| Проверка PR с замечаниями                          | `post comments`              | Публикует встроенные комментарии в PR (без повторной проверки) |
| Проверка PR без замечаний                          | `post comments`              | Утверждает PR на GitHub (LGTM)                      |
| Локальная проверка, всё чисто                      | `commit`                     | Фиксирует ваши изменения                            |

Примечание: `fix these issues` доступна только для локальных проверок. Для проверок PR используйте Autofix (Шаг 8) — рабочее дерево очищается после проверки, поэтому интерактивное исправление после проверки невозможно.

## Правила проверки проектов

Вы можете настроить критерии проверки для каждого проекта. `/review` читает правила из следующих файлов (в порядке приоритета):

1. `.qwen/review-rules.md` (нативный для Qwen Code)
2. `.github/copilot-instructions.md` (предпочтительный) или `copilot-instructions.md` (запасной — загружается только один, не оба)
3. `AGENTS.md` — раздел `## Code Review`
4. `QWEN.md` — раздел `## Code Review`

Правила внедряются в агенты проверки LLM (1-6) как дополнительные критерии. Для проверок PR правила читаются из **базовой ветки**, чтобы предотвратить внедрение вредоносным PR правил обхода.

Пример `.qwen/review-rules.md`:

```markdown
# Review Rules

- All API endpoints must validate authentication
- Database queries must use parameterized statements
- React components must not use inline styles
- Error messages must not expose internal paths
```

## Инкрементальная проверка

При проверке PR, который уже был проверен ранее, `/review` анализирует только изменения с момента последней проверки:

```bash
# First review — full review, cache created
/review 123

# PR updated with new commits — only new changes reviewed
/review 123
```

### Проверка с разными моделями

Если вы переключаете модели (через `/model`) и повторно проверяете тот же PR, `/review` обнаруживает смену модели и запускает полную проверку вместо пропуска:

```bash
# Review with model A
/review 123

# Switch model
/model

# Review again — full review with model B (not skipped)
/review 123
# → "Previous review used qwen3-coder. Running full review with gpt-4o for a second opinion."
```

Кеш хранится в `.qwen/review-cache/` и отслеживает как SHA коммита, так и ID модели. Убедитесь, что этот каталог добавлен в ваш `.gitignore` (более широкое правило, например `.qwen/*`, также работает). Если кешированный коммит был удалён при перебазировании, выполняется полная проверка.

## Отчёты о проверке

Для проверок в рамках одного репозитория результаты сохраняются в виде Markdown-файла в каталоге `.qwen/reviews/` вашего проекта (кросс-репозиторные облегчённые проверки пропускают сохранение отчётов):

```
.qwen/reviews/2026-04-06-143022-pr-123.md
.qwen/reviews/2026-04-06-150510-local.md
```

Отчёты включают: временную метку, статистику изменений, результаты детерминированного анализа, все замечания с статусом верификации и вердикт.

## Анализ влияния на межфайловые изменения

Когда изменения кода затрагивают экспортируемые функции, классы или интерфейсы, агенты проверки автоматически ищут всех вызывающих и проверяют совместимость:

- Изменения количества/типов параметров
- Изменения возвращаемого типа
- Удалённые или переименованные публичные методы
- Критические изменения API

Для больших изменений (>10 изменённых символов) анализ отдаёт приоритет функциям с изменениями сигнатуры.

## Эффективность использования токенов

Конвейер проверки использует ограниченное количество вызовов LLM независимо от количества найденных замечаний:

| Этап                                     | Вызовы LLM       | Примечания                                               |
| ---------------------------------------- | ---------------- | -------------------------------------------------------- |
| Детерминированный анализ (Шаг 3)         | 0                | Только команды оболочки                                  |
| Агенты проверки (Шаг 4)                  | 9 (или 8)        | Запускаются параллельно; Агент 7 пропускается в кросс-репозиторном режиме |
| Пакетная верификация (Шаг 5)             | 1                | Один агент проверяет все замечания сразу                 |
| Итеративный обратный аудит (Шаг 6)       | 1-3              | Циклически до "Проблем не найдено" или лимита в 3 раунда |
| **Итого**                                | **11-13 (10-12)** | Один репозиторий: 11-13; кросс-репозиторий: 10-12 (без Агента 7) |

Большинство PR сходятся к нижней границе диапазона (1 раунд обратного аудита); лимит предотвращает неконтролируемые затраты в патологических случаях.

## Что НЕ помечается

Проверка намеренно исключает:

- Ранее существовавшие проблемы в неизменённом коде (фокус только на изменениях)
- Стиль/форматирование/именование, соответствующие соглашениям вашей кодовой базы
- Проблемы, которые выявил бы линтер или проверка типов (обрабатываются детерминированным анализом)
- Субъективные предложения "подумайте о том, чтобы сделать X" без реальной проблемы
- Незначительный рефакторинг, который не исправляет ошибку или не снижает риск
- Отсутствие документации, если только логика действительно не сбивает с толку
- Проблемы, уже обсуждённые в существующих комментариях к PR (избегает дублирования человеческой обратной связи)
## Философия дизайна

> **Молчание лучше шума.** Каждый комментарий должен стоить времени читателя.

- Если нет уверенности, что это проблема → не сообщайте о ней
- Проблемы линтера/typecheck обрабатываются инструментами, а не догадками LLM
- Одинаковый паттерн в N файлах → объединяется в одно замечание
- Комментарии в PR — только при высокой уверенности
- Проблемы стиля/форматирования, соответствующие конвенциям кодовой базы, исключаются

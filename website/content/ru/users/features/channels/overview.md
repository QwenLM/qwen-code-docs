# Каналы

Каналы позволяют взаимодействовать с агентом Qwen Code через мессенджеры, такие как Telegram, WeChat, QQ или DingTalk, вместо терминала. Вы отправляете сообщения из чата на телефоне или компьютере, а агент отвечает так же, как в CLI.

## Как это работает

Когда вы запускаете `qwen channel start`, Qwen Code:

1. Читает конфигурации каналов из вашего `settings.json`
2. Запускает единый процесс агента с использованием [Agent Client Protocol (ACP)](../../../developers/architecture.md)
3. Подключается к каждому мессенджеру и начинает прослушивать сообщения
4. Направляет входящие сообщения агенту и отправляет ответы обратно в соответствующий чат

Все каналы используют один процесс агента с изолированными сессиями для каждого пользователя. Каждый канал может иметь свою собственную рабочую директорию, модель и инструкции.

## Быстрый старт

1. Настройте бота в выбранном мессенджере (см. руководства по каналам: [Telegram](./telegram), [WeChat](./weixin), [QQ Bot](./qqbot), [DingTalk](./dingtalk))
2. Добавьте конфигурацию канала в `~/.qwen/settings.json`
3. Выполните `qwen channel start` для запуска всех каналов, или `qwen channel start <name>` для одного канала

Хотите подключить платформу, которая не встроена? См. [Плагины](./plugins), чтобы добавить собственный адаптер в качестве расширения.

## Конфигурация

Каналы настраиваются в разделе `channels` файла `settings.json`. Каждый канал имеет имя и набор параметров:

```json
{
  "channels": {
    "my-channel": {
      "type": "telegram",
      "token": "$MY_BOT_TOKEN",
      "senderPolicy": "allowlist",
      "allowedUsers": ["123456789"],
      "sessionScope": "user",
      "cwd": "/path/to/working/directory",
      "instructions": "Optional system instructions for the agent.",
      "groupPolicy": "disabled",
      "groups": {
        "*": { "requireMention": true }
      }
    }
  }
}
```

### Параметры

| Параметр                | Обязателен | Описание                                                                                                                                                       |
| ----------------------- | ---------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `type`                  | Да         | Тип канала: `telegram`, `weixin`, `qq`, `dingtalk`, `feishu` или пользовательский тип из расширения (см. [Плагины](./plugins))                                 |
| `token`                 | Telegram   | Токен бота. Поддерживает синтаксис `$ENV_VAR` для чтения из переменных окружения. Не требуется для WeChat или DingTalk                                         |
| `clientId`              | DingTalk   | AppKey DingTalk. Поддерживает синтаксис `$ENV_VAR`                                                                                                             |
| `clientSecret`          | DingTalk   | AppSecret DingTalk. Поддерживает синтаксис `$ENV_VAR`                                                                                                          |
| `model`                 | Нет        | Модель для использования в этом канале (например, `qwen3.5-plus`). Переопределяет модель по умолчанию. Полезно для мультимодальных моделей, поддерживающих ввод изображений |
| `senderPolicy`          | Нет        | Кто может общаться с ботом: `allowlist` (по умолчанию), `open` или `pairing`                                                                                   |
| `allowedUsers`          | Нет        | Список идентификаторов пользователей, которым разрешено использовать бота (используется политиками `allowlist` и `pairing`)                                     |
| `sessionScope`          | Нет        | Как сессии ограничены: `user` (по умолчанию), `thread` или `single`                                                                                            |
| `cwd`                   | Нет        | Рабочая директория для агента. По умолчанию — текущая директория                                                                                               |
| `instructions`          | Нет        | Пользовательские инструкции, добавляемые к первому сообщению каждой сессии                                                                                     |
| `groupPolicy`           | Нет        | Доступ к групповым чатам: `disabled` (по умолчанию), `allowlist` или `open`. См. [Групповые чаты](#group-chats)                                                |
| `groups`                | Нет        | Настройки для каждой группы. Ключи — это ID групповых чатов или `"*"` для значений по умолчанию. См. [Групповые чаты](#group-chats)                           |
| `dispatchMode`          | Нет        | Что происходит при отправке сообщения, когда бот занят: `steer` (по умолчанию), `collect` или `followup`. См. [Режимы диспетчеризации](#dispatch-modes)       |
| `blockStreaming`        | Нет        | Прогрессивная доставка ответа: `on` или `off` (по умолчанию). См. [Потоковая передача блоками](#block-streaming)                                               |
| `blockStreamingChunk`   | Нет        | Границы размера блока: `{ "minChars": 400, "maxChars": 1000 }`. См. [Потоковая передача блоками](#block-streaming)                                            |
| `blockStreamingCoalesce`| Нет        | Сброс при простое: `{ "idleMs": 1500 }`. См. [Потоковая передача блоками](#block-streaming)                                                                    |

### Политика отправителя

Определяет, кто может взаимодействовать с ботом:

- **`allowlist`** (по умолчанию) — Только пользователи, указанные в `allowedUsers`, могут отправлять сообщения. Остальные игнорируются.
- **`pairing`** — Неизвестные отправители получают код привязки. Оператор бота одобряет их через CLI, и они добавляются в постоянный белый список. Пользователи из `allowedUsers` полностью пропускают привязку. См. [Привязка по DM](#dm-pairing) ниже.
- **`open`** — Любой может отправлять сообщения. Используйте с осторожностью.

### Область действия сессии

Определяет, как управляются сессии диалога:

- **`user`** (по умолчанию) — Одна сессия на пользователя. Все сообщения от одного пользователя объединяются в один диалог.
- **`thread`** — Одна сессия на ветку/тему. Полезно для групповых чатов с темами.
- **`single`** — Одна общая сессия для всех пользователей. Все используют один и тот же диалог.

### Безопасность токенов

Токены ботов не следует хранить напрямую в `settings.json`. Вместо этого используйте ссылки на переменные окружения:

```json
{
  "token": "$TELEGRAM_BOT_TOKEN"
}
```

Установите фактический токен в вашем окружении оболочки или в файле `.env`, который загружается перед запуском канала.

## Привязка по DM (личные сообщения)

Когда `senderPolicy` установлен в `"pairing"`, неизвестные отправители проходят процесс одобрения:

1. Неизвестный пользователь отправляет сообщение боту
2. Бот отвечает 8-символьным кодом привязки (например, `VEQDDWXJ`)
3. Пользователь передает код вам (оператору бота)
4. Вы одобряете его через CLI:

```bash
qwen channel pairing approve my-channel VEQDDWXJ
```

После одобрения ID пользователя сохраняется в `~/.qwen/channels/<name>-allowlist.json`, и все последующие сообщения проходят в обычном режиме.

### CLI-команды привязки

```bash
# List pending pairing requests
qwen channel pairing list my-channel

# Approve a request by code
qwen channel pairing approve my-channel <CODE>
```

### Правила привязки

- Коды состоят из 8 символов в верхнем регистре с использованием однозначного алфавита (без `0`/`O`/`1`/`I`)
- Коды действительны в течение 1 часа
- Максимум 3 ожидающих запроса на канал одновременно — дополнительные запросы игнорируются, пока один не истечет или не будет одобрен
- Пользователи, указанные в `allowedUsers` в `settings.json`, всегда пропускают привязку
- Одобренные пользователи сохраняются в `~/.qwen/channels/<name>-allowlist.json` — относитесь к этому файлу как к конфиденциальному

## Групповые чаты

По умолчанию бот работает только в личных сообщениях. Чтобы включить поддержку групповых чатов, установите `groupPolicy` в `"allowlist"` или `"open"`.

### Политика групп

Определяет, участвует ли бот в групповых чатах вообще:

- **`disabled`** (по умолчанию) — Бот игнорирует все групповые сообщения. Самый безопасный вариант.
- **`allowlist`** — Бот отвечает только в группах, явно перечисленных в `groups` по ID чата. Ключ `"*"` предоставляет настройки по умолчанию, но **не** действует как универсальное разрешение.
- **`open`** — Бот отвечает во всех группах, куда он добавлен. Используйте с осторожностью.

### Ограничение по упоминаниям

В группах по умолчанию бот требует упоминания через `@` или ответа на одно из своих сообщений. Это предотвращает ответ бота на каждое сообщение в групповом чате.

Настройка для каждой группы с помощью параметра `groups`:

```json
{
  "groups": {
    "*": { "requireMention": true },
    "-100123456": { "requireMention": false }
  }
}
```

- **`"*"`** — Настройки по умолчанию для всех групп. Только задает значения по умолчанию, а не запись в белом списке.
- **ID группового чата** — Переопределяет настройки для конкретной группы. Переопределяет значения по умолчанию из `"*"`.
- **`requireMention`** (по умолчанию: `true`) — Если `true`, бот отвечает только на сообщения, в которых его упомянули через @ или ответили на одно из его сообщений. Если `false`, бот отвечает на все сообщения (полезно для специальных рабочих групп).

### Как обрабатываются групповые сообщения

```
1. groupPolicy — эта группа разрешена?           (нет → игнорировать)
2. requireMention — упомянули бота/ответили ему? (нет → игнорировать)
3. senderPolicy — отправитель одобрен?            (нет → процесс привязки)
4. Маршрутизация к сессии
```

### Настройка Telegram для групп

1. Добавьте бота в группу
2. **Отключите режим конфиденциальности** в BotFather (`/mybots` → Bot Settings → Group Privacy → Turn Off) — иначе бот не будет видеть сообщения, не являющиеся командами
3. **Удалите и снова добавьте бота** в группу после изменения режима конфиденциальности (Telegram кэширует эту настройку)

### Поиск ID группового чата

Чтобы найти ID чата группы для белого списка `groups`:

1. Остановите бота, если он запущен
2. Отправьте сообщение с упоминанием бота в группе
3. Используйте Telegram Bot API для проверки ожидающих обновлений:

```bash
curl -s "https://api.telegram.org/bot${TELEGRAM_BOT_TOKEN}/getUpdates" | python3 -m json.tool
```

Найдите `message.chat.id` в ответе — ID групп являются отрицательными числами (например, `-5170296765`).

## Поддержка медиа

Каналы поддерживают отправку изображений и файлов агенту, а не только текста.

### Изображения

Отправьте фото боту, и агент его увидит — полезно для обмена скриншотами, сообщениями об ошибках или диаграммами. Изображение отправляется напрямую модели как визуальный ввод.

Чтобы использовать поддержку изображений, настройте мультимодальную модель для канала:

```json
{
  "channels": {
    "my-channel": {
      "type": "telegram",
      "model": "qwen3.5-plus",
      ...
    }
  }
}
```

### Файлы

Отправьте документ (PDF, файл с кодом, текстовый файл и т.п.) боту. Файл загружается и сохраняется во временную директорию, а агенту сообщается путь к файлу, чтобы он мог прочитать его содержимое с помощью своих инструментов для чтения файлов.

Файлы работают с любой моделью — мультимодальная поддержка не требуется.

### Различия платформ

| Функция  | Telegram                                    | WeChat                           | DingTalk                                      |
| -------- | ------------------------------------------- | -------------------------------- | --------------------------------------------- |
| Изображения | Прямая загрузка через Bot API               | Загрузка через CDN с дешифровкой AES | API downloadCode (два шага)                   |
| Файлы    | Прямая загрузка через Bot API (лимит 20MB)  | Загрузка через CDN с дешифровкой AES | API downloadCode (два шага)                   |
| Подписи  | Подписи к фото/файлам включены как текст сообщения | Не применимо                   | Rich-текст: смешанный текст и изображения в одном сообщении |

## Режимы диспетчеризации

Определяет, что происходит, когда вы отправляете новое сообщение, пока бот еще обрабатывает предыдущее.

- **`steer`** (по умолчанию) — Бот отменяет текущий запрос и начинает работать над вашим новым сообщением. Лучше всего подходит для обычного чата, где последующее сообщение обычно означает, что вы хотите скорректировать или перенаправить бота.
- **`collect`** — Ваши новые сообщения буферизуются. Когда текущий запрос завершается, все буферизованные сообщения объединяются в один последующий промпт. Хорошо подходит для асинхронных рабочих процессов, где вы хотите поставить мысли в очередь.
- **`followup`** — Каждое сообщение ставится в очередь и обрабатывается как отдельный шаг, по порядку. Полезно для пакетных рабочих процессов, где каждое сообщение независимо.

```json
{
  "channels": {
    "my-channel": {
      "type": "telegram",
      "dispatchMode": "steer",
      ...
    }
  }
}
```

Вы также можете установить режим диспетчеризации для каждой группы, переопределяя значение по умолчанию для канала:

```json
{
  "groups": {
    "*": { "requireMention": true, "dispatchMode": "steer" },
    "-100123456": { "dispatchMode": "collect" }
  }
}
```

## Потоковая передача блоками

По умолчанию агент некоторое время работает, а затем отправляет один большой ответ. При включенной потоковой передаче блоками ответ приходит в виде нескольких более коротких сообщений, пока агент еще работает — аналогично тому, как ChatGPT или Claude отображают прогрессивный вывод.

```json
{
  "channels": {
    "my-channel": {
      "type": "telegram",
      "blockStreaming": "on",
      "blockStreamingChunk": { "minChars": 400, "maxChars": 1000 },
      "blockStreamingCoalesce": { "idleMs": 1500 },
      ...
    }
  }
}
```

### Как это работает

- Ответ агента разбивается на блоки по границам абзацев и отправляется как отдельные сообщения
- `minChars` (по умолчанию 400) — не отправлять блок, пока он не достигнет этой длины, чтобы избежать спама мелкими сообщениями
- `maxChars` (по умолчанию 1000) — если блок достиг этой длины без естественного разрыва, отправить его в любом случае
- `idleMs` (по умолчанию 1500) — если агент приостанавливается (например, при запуске инструмента), отправить то, что накоплено
- Когда агент завершает работу, оставшийся текст отправляется немедленно

Требуется только `blockStreaming`. Настройки chunk и coalesce являются необязательными и имеют разумные значения по умолчанию.

## Слэш-команды

Каналы поддерживают слэш-команды. Они обрабатываются локально (без обращения к агенту):

- `/help` — Показать доступные команды
- `/clear` — Очистить сессию и начать заново (псевдонимы: `/reset`, `/new`)
- `/status` — Показать информацию о сессии и политику доступа

Все остальные слэш-команды (например, `/compress`, `/summary`) передаются агенту.

Эти команды работают во всех типах каналов (Telegram, WeChat, QQ, DingTalk).

## Запуск

```bash
# Start all configured channels (shared agent process)
qwen channel start

# Start a single channel
qwen channel start my-channel

# Check if the service is running
qwen channel status

# Stop the running service
qwen channel stop
```

Бот работает на переднем плане. Нажмите `Ctrl+C` для остановки или используйте `qwen channel stop` из другого терминала.

### Многоканальный режим

Когда вы запускаете `qwen channel start` без имени, все каналы, определенные в `settings.json`, запускаются вместе, используя один процесс агента. Каждый канал поддерживает свои собственные сессии — пользователь Telegram и пользователь WeChat получают отдельные диалоги, даже если они используют одного и того же агента.

Каждый канал использует свой собственный `cwd` из своей конфигурации, поэтому разные каналы могут одновременно работать над разными проектами.

### Управление службой

Служба каналов использует PID-файл (`~/.qwen/channels/service.pid`) для отслеживания запущенного экземпляра:

- **Предотвращение дублирования**: Запуск `qwen channel start`, когда служба уже работает, приведет к ошибке вместо запуска второго экземпляра
- **`qwen channel stop`**: Корректно останавливает работающую службу из другого терминала
- **`qwen channel status`**: Показывает, запущена ли служба, время работы и количество сессий на канал

### Восстановление после сбоя

Если процесс агента неожиданно завершается с ошибкой, служба каналов автоматически перезапускает его и пытается восстановить все активные сессии. Пользователи могут продолжить свои диалоги, не начиная заново.

- Сессии сохраняются в `~/.qwen/channels/sessions.json` во время работы службы
- В случае сбоя: агент перезапускается в течение 3 секунд и загружает сохраненные сессии
- После 3 последовательных сбоев служба завершает работу с ошибкой
- При корректном завершении (Ctrl+C или `qwen channel stop`): данные сессий очищаются — следующий запуск всегда начинается с чистого листа

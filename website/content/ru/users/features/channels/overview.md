# Channels

Каналы позволяют вам взаимодействовать с агентом Qwen Code через мессенджеры, такие как Telegram, WeChat, QQ или DingTalk, вместо терминала. Вы отправляете сообщения с телефона или приложения для чатов на рабочем столе, и агент отвечает так же, как и в CLI.

## Как это работает

Когда вы запускаете `qwen channel start`, Qwen Code:

1. Читает конфигурации каналов из вашего `settings.json`
2. Запускает единый процесс агента, используя [Agent Client Protocol (ACP)](../../../developers/architecture.md)
3. Подключается к каждой платформе обмена сообщениями и начинает прослушивание сообщений
4. Маршрутизирует входящие сообщения агенту и отправляет ответы обратно в нужный чат

Все каналы используют один процесс агента с изолированными сессиями для каждого пользователя. Каждый канал может иметь свою собственную рабочую директорию, модель и инструкции.

## Быстрый старт

1. Настройте бота на вашей платформе обмена сообщениями (см. руководства по конкретным каналам: [Telegram](./telegram), [WeChat](./weixin), [QQ Bot](./qqbot), [DingTalk](./dingtalk))
2. Добавьте конфигурацию канала в `~/.qwen/settings.json`
3. Запустите `qwen channel start` для запуска всех каналов, или `qwen channel start <name>` для одного канала

Хотите подключить платформу, которая не встроена? См. [Plugins](./plugins), чтобы добавить собственный адаптер в качестве расширения.

## Конфигурация

Каналы настраиваются под ключём `channels` в `settings.json`. Каждый канал имеет имя и набор опций:

```json
{
  "channels": {
    "my-channel": {
      "type": "telegram",
      "token": "$MY_BOT_TOKEN",
      "senderPolicy": "allowlist",
      "allowedUsers": ["123456789"],
      "sessionScope": "user",
      "cwd": "/path/to/working/directory",
      "instructions": "Optional system instructions for the agent.",
      "groupPolicy": "disabled",
      "groups": {
        "*": { "requireMention": true }
      }
    }
  }
}
```

### Опции

| Опция                   | Обязательный | Описание                                                                                                                                    |
| ------------------------ | -------- | ---------------------------------------------------------------------------------------------------------------------------------------------- |
| `type`                   | Да      | Тип канала: `telegram`, `weixin`, `qq`, `dingtalk`, `feishu` или пользовательский тип из расширения (см. [Plugins](./plugins)) |
| `token`                  | Telegram | Токен бота. Поддерживает синтаксис `$ENV_VAR` для чтения из переменных окружения. Не требуется для WeChat или DingTalk |
| `clientId`               | DingTalk | AppKey для DingTalk. Поддерживает синтаксис `$ENV_VAR` |
| `clientSecret`           | DingTalk | AppSecret для DingTalk. Поддерживает синтаксис `$ENV_VAR` |
| `model`                  | Нет       | Модель для использования в этом канале (например, `qwen3.5-plus`). Переопределяет модель по умолчанию. Полезно для мультимодальных моделей, поддерживающих ввод изображений |
| `senderPolicy`           | Нет       | Кто может общаться с ботом: `allowlist` (по умолчанию), `open` или `pairing` |
| `allowedUsers`           | Нет       | Список идентификаторов пользователей, которым разрешено использовать бота (используется политиками `allowlist` и `pairing`) |
| `sessionScope`           | Нет       | Как сессии ограничены: `user` (по умолчанию), `thread` или `single` |
| `cwd`                    | Нет       | Рабочая директория для агента. По умолчанию текущая директория |
| `instructions`           | Нет       | Пользовательские инструкции, добавляемые к первому сообщению каждой сессии |
| `groupPolicy`            | Нет       | Доступ в групповые чаты: `disabled` (по умолчанию), `allowlist` или `open`. См. [Групповые чаты](#group-chats) |
| `groups`                 | Нет       | Настройки для каждой группы. Ключи — это ID групповых чатов или `"*"` для значений по умолчанию. См. [Групповые чаты](#group-chats) |
| `dispatchMode`           | Нет       | Что происходит, когда вы отправляете сообщение, пока бот занят: `steer` (по умолчанию), `collect` или `followup`. См. [Режимы диспетчеризации](#dispatch-modes) |
| `blockStreaming`         | Нет       | Постепенная доставка ответа: `on` или `off` (по умолчанию). См. [Блоковая потоковая передача](#block-streaming) |
| `blockStreamingChunk`    | Нет       | Границы размера чанков: `{ "minChars": 400, "maxChars": 1000 }`. См. [Блоковая потоковая передача](#block-streaming) |
| `blockStreamingCoalesce` | Нет       | Сброс при простое: `{ "idleMs": 1500 }`. См. [Блоковая потоковая передача](#block-streaming) |
### Политика отправителя (Sender Policy)

Определяет, кто может взаимодействовать с ботом:

- **`allowlist`** (по умолчанию) — Только пользователи, указанные в `allowedUsers`, могут отправлять сообщения. Остальные игнорируются без ответа.
- **`pairing`** — Неизвестные отправители получают код привязки. Оператор бота одобряет их через CLI, после чего они добавляются в постоянный белый список. Пользователи из `allowedUsers` полностью пропускают привязку. См. [Привязка по личным сообщениям](#привязка-по-личным-сообщениям-dm-pairing) ниже.
- **`open`** — Сообщения может отправлять кто угодно. Используйте с осторожностью.

### Область сессии (Session Scope)

Определяет, как управляются сеансы разговора:

- **`user`** (по умолчанию) — Одна сессия на пользователя. Все сообщения от одного пользователя объединяются в один разговор.
- **`thread`** — Одна сессия на ветку/тему. Полезно для групповых чатов с ветками.
- **`single`** — Одна общая сессия для всех пользователей. Все используют один и тот же разговор.

### Безопасность токена (Token Security)

Токены бота не должны храниться непосредственно в `settings.json`. Вместо этого используйте ссылки на переменные окружения:

```json
{
  "token": "$TELEGRAM_BOT_TOKEN"
}
```

Установите фактический токен в вашем окружении терминала или в файле `.env`, который загружается перед запуском канала.

## Привязка по личным сообщениям (DM Pairing)

Если `senderPolicy` установлено в `"pairing"`, неизвестные отправители проходят процедуру одобрения:

1. Неизвестный пользователь отправляет сообщение боту.
2. Бот отвечает 8-символьным кодом привязки (например, `VEQDDWXJ`).
3. Пользователь передаёт код вам (оператору бота).
4. Вы одобряете его через CLI:

```bash
qwen channel pairing approve my-channel VEQDDWXJ
```

После одобрения ID пользователя сохраняется в `~/.qwen/channels/<name>-allowlist.json`, и все последующие сообщения обрабатываются в обычном режиме.

### Команды CLI для привязки

```bash
# Показать ожидающие запросы привязки
qwen channel pairing list my-channel

# Одобрить запрос по коду
qwen channel pairing approve my-channel <CODE>
```

### Правила привязки

- Коды состоят из 8 символов в верхнем регистре, использующих однозначный алфавит (без `0`/`O`/`1`/`I`).
- Коды действуют в течение 1 часа.
- Не более 3 ожидающих запросов на канал одновременно — дополнительные запросы игнорируются, пока один не истечёт или не будет одобрен.
- Пользователи, перечисленные в `allowedUsers` в `settings.json`, всегда пропускают привязку.
- Одобренные пользователи сохраняются в `~/.qwen/channels/<name>-allowlist.json` — относитесь к этому файлу как к конфиденциальному.

## Групповые чаты (Group Chats)

По умолчанию бот работает только в личных сообщениях. Чтобы включить поддержку групповых чатов, установите `groupPolicy` в `"allowlist"` или `"open"`.

### Политика группы (Group Policy)

Определяет, участвует ли бот в групповых чатах вообще:

- **`disabled`** (по умолчанию) — Бот игнорирует все групповые сообщения. Самый безопасный вариант.
- **`allowlist`** — Бот отвечает только в группах, явно перечисленных в `groups` по ID чата. Ключ `"*"` задаёт настройки по умолчанию, но **не** разрешает все группы.
- **`open`** — Бот отвечает во всех группах, куда он добавлен. Используйте с осторожностью.

### Управление упоминаниями

В группах бот по умолчанию требует `@упоминания` или ответа на одно из своих сообщений. Это предотвращает реагирование на каждое сообщение в групповом чате.

Настройка для каждой группы с помощью параметра `groups`:

```json
{
  "groups": {
    "*": { "requireMention": true },
    "-100123456": { "requireMention": false }
  }
}
```

- **`"*"`** — Параметры по умолчанию для всех групп. Только задаёт настройки по умолчанию, не является записью в белый список.
- **ID группового чата** — Переопределить настройки для конкретной группы. Переопределяет значения по умолчанию `"*"`.
- **`requireMention`** (по умолчанию: `true`) — Если `true`, бот отвечает только на сообщения, которые упоминают его через `@` или являются ответом на одно из его сообщений. Если `false`, бот отвечает на все сообщения (полезно для выделенных рабочих групп).

### Как обрабатываются групповые сообщения

```
1. groupPolicy — разрешена ли эта группа?          (нет → игнорировать)
2. requireMention — был ли бот упомянут/вызван ответом? (нет → игнорировать)
3. senderPolicy — одобрен ли отправитель?           (нет → процедура привязки)
4. Перенаправление к сессии
```

### Настройка Telegram для групп

1. Добавьте бота в группу
2. **Отключите приватный режим** в BotFather (`/mybots` → Bot Settings → Group Privacy → Turn Off) — иначе бот не будет видеть сообщения, не являющиеся командами
3. **Удалите и снова добавьте бота** в группу после изменения режима конфиденциальности (Telegram кэширует эту настройку)

### Как узнать ID группового чата

Чтобы узнать ID чата группы для белого списка `groups`:

1. Остановите бота, если он запущен
2. Отправьте в группу сообщение, упоминающее бота
3. Используйте Telegram Bot API для проверки ожидающих обновлений:

```bash
curl -s "https://api.telegram.org/bot${TELEGRAM_BOT_TOKEN}/getUpdates" | python3 -m json.tool
```

Найдите `message.chat.id` в ответе — ID групп представляют собой отрицательные числа (например, `-5170296765`).

## Поддержка медиа (Media Support)

Каналы поддерживают отправку изображений и файлов агенту, а не только текст.

### Изображения

Отправьте фото боту, и агент его увидит — полезно для обмена скриншотами, сообщениями об ошибках или диаграммами. Изображение напрямую передаётся модели как визуальный ввод (vision input).

Чтобы использовать поддержку изображений, настройте мультимодальную модель для канала:

```json
{
  "channels": {
    "my-channel": {
      "type": "telegram",
      "model": "qwen3.5-plus",
      ...
    }
  }
}
```
### Файлы

Отправьте боту документ (PDF, файл с кодом, текстовый файл и т.д.). Файл загружается и сохраняется во временную директорию, а агенту передаётся путь к файлу, чтобы он мог прочитать его содержимое с помощью своих инструментов для чтения файлов.

Файлы работают с любой моделью — мультимодальная поддержка не требуется.

### Различия платформ

| Функция  | Telegram                                     | WeChat                           | DingTalk                                      |
| -------- | -------------------------------------------- | -------------------------------- | --------------------------------------------- |
| Изображения | Прямая загрузка через Bot API                  | Загрузка через CDN с расшифровкой AES | API downloadCode (два шага)                   |
| Файлы    | Прямая загрузка через Bot API (ограничение 20 МБ)     | Загрузка через CDN с расшифровкой AES | API downloadCode (два шага)                   |
| Подписи | Подписи к фото/файлам включены как текст сообщения | Не применимо                   | Rich text: смешанный текст + изображения в одном сообщении |

## Режимы отправки

Управляет тем, что происходит, когда вы отправляете новое сообщение, пока бот ещё обрабатывает предыдущее.

- **`steer`** (по умолчанию) — Бот отменяет текущий запрос и начинает обрабатывать ваше новое сообщение. Лучше всего подходит для обычного чата, где последующее сообщение обычно означает, что вы хотите исправить или перенаправить бота.
- **`collect`** — Ваши новые сообщения буферизируются. Когда текущий запрос завершается, все буферизированные сообщения объединяются в один последующий запрос. Полезно для асинхронных рабочих процессов, где вы хотите выстроить мысли в очередь.
- **`followup`** — Каждое сообщение ставится в очередь и обрабатывается как отдельный шаг по порядку. Полезно для пакетных процессов, где каждое сообщение независимо.

```json
{
  "channels": {
    "my-channel": {
      "type": "telegram",
      "dispatchMode": "steer",
      ...
    }
  }
}
```

Вы также можете задать режим отправки для каждой группы, переопределяя настройки канала по умолчанию:

```json
{
  "groups": {
    "*": { "requireMention": true, "dispatchMode": "steer" },
    "-100123456": { "dispatchMode": "collect" }
  }
}
```

## Потоковая передача блоков

По умолчанию агент работает некоторое время, а затем отправляет один большой ответ. При включённой потоковой передаче блоков ответ приходит в виде нескольких более коротких сообщений, пока агент ещё работает — аналогично тому, как ChatGPT или Claude показывают постепенный вывод.

```json
{
  "channels": {
    "my-channel": {
      "type": "telegram",
      "blockStreaming": "on",
      "blockStreamingChunk": { "minChars": 400, "maxChars": 1000 },
      "blockStreamingCoalesce": { "idleMs": 1500 },
      ...
    }
  }
}
```

### Как это работает

- Ответ агента разбивается на блоки по границам абзацев и отправляется как отдельные сообщения
- `minChars` (по умолчанию 400) — не отправлять блок, пока его длина не достигнет этого значения, чтобы не засорять сообщениями крошечного размера
- `maxChars` (по умолчанию 1000) — если блок достиг этой длины без естественного разрыва, отправить его в любом случае
- `idleMs` (по умолчанию 1500) — если агент приостановился (например, выполняя инструмент), отправить то, что накопилось на данный момент
- Когда агент завершает работу, оставшийся текст отправляется немедленно

Обязательным является только `blockStreaming`. Настройки чанка и объединения необязательны и имеют разумные значения по умолчанию.

## Команды с косой чертой

Каналы поддерживают команды с косой чертой. Они обрабатываются локально (без обращения к агенту):

- `/help` — Список доступных команд
- `/clear` — Очистить сессию и начать заново (псевдонимы: `/reset`, `/new`)
- `/status` — Показать информацию о сессии и политику доступа

Все остальные команды с косой чертой (например, `/compress`, `/summary`) перенаправляются агенту.

Эти команды работают на всех типах каналов (Telegram, WeChat, QQ, DingTalk).

## Запуск

```bash
# Start all configured channels (shared agent process)
qwen channel start

# Start a single channel
qwen channel start my-channel

# Check if the service is running
qwen channel status

# Stop the running service
qwen channel stop
```

Бот запускается в режиме переднего плана. Нажмите `Ctrl+C` для остановки или используйте `qwen channel stop` из другого терминала.

### Многоканальный режим

Когда вы запускаете `qwen channel start` без указания имени, все каналы, определённые в `settings.json`, запускаются вместе и используют общий процесс агента. Каждый канал ведёт собственные сессии — пользователь Telegram и пользователь WeChat получают отдельные диалоги, даже если они используют одного и того же агента.

Каждый канал использует свой собственный `cwd` из конфигурации, поэтому разные каналы могут одновременно работать над разными проектами.

### Управление службой

Служба каналов использует PID-файл (`~/.qwen/channels/service.pid`) для отслеживания запущенного экземпляра:

- **Предотвращение дублирования**: Запуск `qwen channel start` при уже запущенной службе покажет ошибку вместо запуска второго экземпляра
- **`qwen channel stop`**: Корректно завершает запущенную службу из другого терминала
- **`qwen channel status`**: Показывает, запущена ли служба, время её работы и количество сессий по каждому каналу

### Восстановление после сбоя

Если процесс агента неожиданно аварийно завершается, служба каналов автоматически перезапускает его и пытается восстановить все активные сессии. Пользователи могут продолжить свои диалоги, не начиная заново.
- Сеансы сохраняются в `~/.qwen/channels/sessions.json` во время работы службы
- При сбое: агент перезапускается в течение 3 секунд и перезагружает сохраненные сеансы
- После 3 последовательных сбоев служба завершает работу с ошибкой
- При чистом завершении (Ctrl+C или `qwen channel stop`): данные сеансов очищаются — следующий запуск всегда начинается с чистого листа

# Режим демона (`qwen serve`)

Запускайте Qwen Code в качестве локального HTTP-демона, чтобы несколько клиентов (плагины IDE, веб-интерфейсы, CI-скрипты, пользовательские CLI) могли использовать один агент через HTTP + Server-Sent Events, вместо того чтобы каждый порождал свой собственный подпроцесс.

> **🚧 v0.16-alpha**: `qwen serve` впервые появится в npm в версии v0.16-alpha как **только текстовый чат / программирование** с **локальным развертыванием**. Изображения / вложения файлов на пути запроса, контейнеризованное развертывание (Docker / k8s / nginx reverse-proxy) и удаленная / многодемонная защита появятся в последующем патче, когда будет принято корпоративное пилотное внедрение. Полный список отложенных возможностей см. в разделе [Известные ограничения v0.16-alpha](#известные-ограничения-v016-alpha).

> **Статус:** Этап 1 (экспериментальный). Поверхность протокола зафиксирована в таблице маршрутов §04 из issue [#3803](https://github.com/QwenLM/qwen-code/issues/3803). Этап 1.5 (флаг `qwen --serve` — TUI также запускает тот же HTTP-сервер) и Этап 2 (рефакторинг в процессе + доработка `mDNS`/OpenAPI/WebSocket/Prometheus) уже на подходе.
>
> **Честность о масштабе:** Этап 1 рассчитан на **разработчиков, прототипирующих клиенты для поверхности протокола** и на **локальную однопользовательскую / малокомандную совместную работу**. Для production-grade многоклиентских / долгоиграющих / сетевых нестабильных нагрузок (мобильные компаньоны, IM-боты с 1000+ чатами) потребуются гарантии Этапа 1.5+, которых нет в этом релизе. Полный список отличий см. в разделе [Гарантии времени выполнения Этапа 1.5+](#гарантии-времени-выполнения-этапа-15) и roadmap конвергенции в #3803.

## Что он дает

- **Встроенный веб-интерфейс Web Shell** — `qwen serve` автоматически отдаёт браузерный Web Shell на корневом пути (`http://127.0.0.1:4170/`); запустите `qwen serve --open`, чтобы открыть его в браузере. Он обслуживается на том же origin, что и API, поэтому не нужен второй порт или обратный прокси. Используйте `--no-web` для демона только с API.
- **Один процесс агента, много клиентов** — при настройке по умолчанию `sessionScope: 'single'` все клиенты, подключающиеся к демону, используют одну ACP-сессию. Совместная работа между клиентами в реальном времени в одном диалоге, с одними и теми же файловыми diff и одними и теми же запросами разрешений.
- **Безопасное переподключение при стриминге** — SSE с `Last-Event-ID` позволяет клиенту отключиться и возобновить работу с того же места (в пределах окна воспроизведения кольцевого буфера).
- **Разрешения по принципу первого ответившего** — когда агент запрашивает разрешение на выполнение инструмента, все подключенные клиенты видят запрос; какой клиент ответит первым, тот и выигрывает.
- **Один демон — одна рабочая директория** — каждый процесс `qwen serve` привязывается ровно к одной рабочей директории при запуске (согласно [#3803](https://github.com/QwenLM/qwen-code/issues/3803) §02). Развертывания с несколькими рабочими директориями запускают по одному демону на каждую директорию на отдельных портах (или за оркестратором).
- **Управление удаленным временем выполнения** ([#4175](https://github.com/QwenLM/qwen-code/issues/4175) PR 17) — измените режим одобрения сессии (`POST /session/:id/approval-mode`), включите/отключите инструмент для рабочей директории (`POST /workspace/tools/:name/enable`), создайте пустой `QWEN.md` (`POST /workspace/init`, только механически — НЕ вызывает модель; для AI-заполнения отправьте `POST /session/:id/prompt`), перезапустите отдельный MCP-сервер с предварительной проверкой бюджета (`POST /workspace/mcp/:server/restart`) или добавляйте/удаляйте MCP-серверы во время выполнения без перезапуска демона (`POST /workspace/mcp/servers`, `DELETE /workspace/mcp/servers/:name`). Все строго ограничено — сначала настройте `--token`.
- **Резюме сессии** ([#4175](https://github.com/QwenLM/qwen-code/issues/4175) follow-up) — получите одно предложение «на чем я остановился» для активной сессии (`POST /session/:id/recap`). Оборачивает `generateSessionRecap` ядра как побочный запрос к быстрой модели; не загрязняет ни основную историю чата, ни SSE-поток. Не строгое ограничение (такая же позиция, как у `/prompt`); вспомогательная функция SDK `client.recapSession(sessionId)`.
  - **Известное ограничение — увеличение затрат токенов:** маршрут является чисто затратным (каждый вызов — это побочный запрос к LLM, без выгоды для состояния), и у демона нет ограничения на маршрут в v1. При отсутствии ограничений токенов по умолчанию локальный клиент с ошибкой или вредоносный клиент может спамить этим маршрутом, сжигая токены. Настройте `--token` (и, опционально, `--require-auth`) на общих dev-хостах перед тем, как открывать доступ к демону.
  - **Безопасность одновременных резюме:** два одновременных вызова `/recap` для одной сессии запускают два независимых побочных запроса. `generateSessionRecap` читает снимок истории чата через `GeminiClient.getChat().getHistory()` и передает его отдельному вызову `BaseLlmClient.generateText` (через `runSideQuery`); он никогда не добавляет и не изменяет `GeminiChat` сессии. Безопасно вызывать из нескольких клиентов без координации.

## Известные ограничения v0.16-alpha

Первый npm-релиз `qwen serve` (v0.16-alpha) намеренно ограничен — только текстовый чат / программирование для разработчиков, запускающих демон на своей машине. Список ниже делает отложенную поверхность явной, чтобы adopters могли планировать; все перечисленное находится в дорожной карте патчей v0.16.x или ближайшего последующего релиза.

**Поверхность продукта — только текст:**

- ✅ Текстовые запросы и текстовые ответы (чат, программирование, вызовы инструментов, интеграция MCP)
- ❌ **Изображения / вложения файлов на пути запроса** — `MessageEmitter` пока отображает только текст; мультимодальное эхо появится, когда будет принята альфа-цель с потребностями в изображениях (#4175 chiga0 #27 P0 item)
- ❌ **Потоковая загрузка** — то же ограничение, что и мультимодальность

**Поверхность развертывания — только локально:**

- ✅ Loopback (`127.0.0.1`, по умолчанию) — аутентификация не требуется, подходит для рабочих станций разработчиков
- ✅ Локальный запуск через `systemd` / `launchd` / `nohup &` / `tmux` — см. [Шаблоны локального запуска](./qwen-serve-deploy-local.md)
- ✅ Собственный Bearer-токен через переменную окружения `QWEN_SERVER_TOKEN` ([Аутентификация](#аутентификация) для настройки)
- ❌ **Контейнеризованное развертывание** — Docker / Compose / Kubernetes / nginx reverse-proxy с завершением TLS НЕ входят в v0.16-alpha. Отложено до v0.16.x после того, как будет принято корпоративное пилотное внедрение (иначе код устарел бы из-за отсутствия валидации).
- ❌ **Координация нескольких демонов на одном хосте** — применяется правило `1 демон = 1 рабочая директория × N сессий`. Кросс-хостовое объединение, привязка токенов к экземпляру пути и очистка устаревших токенов отложены до v0.16.x.
- ❌ **Автоматическая генерация токенов демона** — альфа-версия использует собственные токены (одна команда `openssl rand -hex 32`). Автогенерация + инфраструктура хранилища токенов отложены до v0.16.x.

**Защита — минимально жизнеспособная для локального однопользовательского режима:**

- ✅ Шлюз безопасности при запуске (отказывается от привязки не к loopback без токена, [PR 15 / #4236](https://github.com/QwenLM/qwen-code/pull/4236))
- ✅ Шлюз аутентификации для маршрутов модификации, маршрутизация разрешений на уровне сессии (Wave 4 PRs)
- ✅ MCP-ограждения + координация разрешений между несколькими клиентами (F2 / F3)
- ✅ **Абсолютный дедлайн для промпта + таймаут простоя SSE-писателя** — опционально через `--prompt-deadline-ms` и `--writer-idle-timeout-ms`; анонсируются через `prompt_absolute_deadline` и `writer_idle_timeout`, когда включены.
- ✅ **Ограничение HTTP-запросов** — опционально через `--rate-limit` и пороговые значения на уровень; анонсируется через `rate_limit`, когда включен.
- ⏸️ **Метрики Prometheus + нагрузочный тестовый фреймворк** — отложено до v0.17 F4 Phase-1 инструментация масштаба, когда 30–50 активных сессий станут реальной целью.
- ⏸️ **Флаг CLI `--max-body-size`** — демон по умолчанию использует `express.json({ limit: '10mb' })`, что с запасом покрывает текстовые промпты (контекстные окна моделей значительно меньше 10 МиБ символов). Настраивается через флаг в v0.16.x.

Для более глубокого перечисления «что мы не будем исправлять в Этапе 1» (однохостовое изменение состояния сессии + N параллельных сессий, использующих одного ACP-ребенка), см. раздел [Границы области Этапа 1](#границы-области-этапа-1--что-мы-не-будем-исправлять-в-этапе-15) ниже.

## Быстрый старт

### 1. Запуск демона (loopback, без аутентификации)

```bash
cd your-project/
qwen serve
# → qwen serve listening on http://127.0.0.1:4170 (mode=http-bridge, workspace=/path/to/your-project)
# → qwen serve: bearer auth disabled (loopback default). Set QWEN_SERVER_TOKEN to enable.
```

Привязка по умолчанию — `127.0.0.1:4170`. Bearer-аутентификация **отключена** на loopback, чтобы локальная разработка «работала из коробки». Демон привязывается к текущей рабочей директории; используйте `--workspace /path/to/dir` для изменения.

**Откройте веб-интерфейс Web Shell.** Перейдите по адресу `http://127.0.0.1:4170/` (или запустите демон с флагом `qwen serve --open` для автоматического открытия) для полноценного браузерного терминала — чат, diffs, вызовы инструментов и запросы разрешений. Интерфейс отдается на корневом пути демона на том же origin, что и API. Остальная часть этого руководства использует нативные HTTP-запросы, чтобы вы могли напрямую работать с API.

### 2. Проверка работоспособности

```bash
curl http://127.0.0.1:4170/health
# → {"status":"ok"}

curl http://127.0.0.1:4170/capabilities
# → {"v":1,"mode":"http-bridge","features":["health","daemon_status","capabilities","session_create",...],"workspaceCwd":"/path/to/your-project"}

curl http://127.0.0.1:4170/daemon/status
# → {"v":1,"detail":"summary","status":"ok","runtime":{...}}
```

Поле `workspaceCwd` показывает привязанную рабочую директорию, чтобы клиенты могли выполнить предварительную проверку и опустить `cwd` в `POST /session`.
Поле `limits.maxPendingPromptsPerSession` показывает активный лимит приема промптов на сессию; `null` означает, что лимит отключен.

Демон также предоставляет read-only снимки состояния выполнения для интерфейсов клиентов и операторов: `GET /daemon/status`, `GET /workspace/mcp`,
`GET /workspace/skills`, `GET /workspace/providers`, `GET /workspace/env`,
`GET /workspace/preflight`,
`GET /session/:id/status`, `GET /session/:id/context`,
`GET /session/:id/supported-commands`,
`GET /session/:id/tasks` и `GET /session/:id/lsp`.

`GET /session/:id/status` возвращает сводку активного моста для одной сессии:
`sessionId`, `workspaceCwd`, `createdAt`, опциональный `displayName`, `clientCount`
и `hasActivePrompt`. Он отвечает `200` со сводкой, если демон содержит активную
сессию с таким id, и `404` (тело `{ "error": …, "sessionId": … }`) в противном
случае. Используйте его для опроса, работает ли известная сессия
(`hasActivePrompt`) или сколько клиентов подключено (`clientCount`), без
загрузки и сканирования всего списка сессий с пагинацией:

```bash
curl http://127.0.0.1:4170/session/$SESSION_ID/status
# → {"sessionId":"…","workspaceCwd":"…","createdAt":"…","clientCount":1,"hasActivePrompt":false}
```

Это необработанное представление активной сессии, поэтому `clientCount` и `hasActivePrompt` соответствуют соответствующей записи в `GET /workspace/:id/sessions` — но два маршрута не побайтово идентичны. Конечная точка списка обогащает каждый элемент данными из хранилища сессий: её `createdAt` — это время первого промпта, и она добавляет `updatedAt` и `displayName`, полученный из сохраненного заголовка или первого промпта. `/status` вместо этого сообщает собственное `createdAt` активной сессии, опускает `updatedAt` и возвращает `displayName` только когда он задан для активной сессии.

`GET /session/:id/lsp` возвращает структурированный статус LSP для сессии. Запустите демон с флагом `--experimental-lsp`, чтобы включить LSP в порожденных сессиях агента; в противном случае маршрут возвращает `enabled: false` без серверов.

`GET /daemon/status` — это консолидированный диагностический снимок. По умолчанию `detail=summary` читает только состояние демона в памяти (сессии, разрешения, количество транспортов SSE/ACP, отклонения по лимиту запросов, память процесса, разрешенные лимиты) и не запускает ACP-ребенка. Используйте `GET /daemon/status?detail=full` для диагностики на уровне сессий, деталей ACP-подключения, количества аутентификаций через device-flow и разделов статуса рабочей директории, когда вы активно расследуете проблему.

`GET /workspace/mcp`, `GET /workspace/skills` и `GET /workspace/providers` сообщают о состоянии активного ACP и не запускают ACP-ребенка, когда он простаивает; в состоянии простоя демон возвращает `initialized: false` с пустым снимком. Как только сессия становится активной, они переключаются на `initialized: true` и отображают реальное состояние.

`GET /workspace/env` и `GET /workspace/preflight` всегда отвечают с `initialized: true` независимо от состояния ACP. `env` никогда не обращается к ACP (только информация о процессе демона); `preflight` отвечает ячейками уровня демона из `process.*` и выдает заполнители `status: 'not_started'` для ячеек уровня ACP, когда ребенок простаивает.

`GET /workspace/env` сообщает о среде выполнения процесса демона, платформе, песочнице, прокси и **наличии** (но не значении) разрешенных переменных окружения, таких как `OPENAI_API_KEY`. URL прокси лишаются учетных данных и сокращаются до `host:port` перед отправкой. Маршрут всегда отвечает напрямую из процесса демона и никогда не порождает ACP-ребенка.

`GET /workspace/preflight` возвращает список проверок готовности. **Ячейки уровня демона** (версия Node, точка входа CLI, директория рабочей области, ripgrep, git, npm) отображаются всегда. **Ячейки уровня ACP** (аутентификация, обнаружение MCP, навыки, провайдеры, реестр инструментов, исходящий трафик) требуют активного ACP-ребенка — когда демон простаивает, они выводят заполнители `status: 'not_started'` вместо запуска ACP только для их заполнения. Сбои сопоставляются с закрытым перечислением `errorKind` (`missing_binary`, `auth_env_error`, `init_timeout`, `protocol_error`, `missing_file`, `parse_error`, `blocked_egress`), чтобы интерфейсы клиентов могли отображать структурированные инструкции по устранению.

Демон также предоставляет вспомогательные функции для работы с файлами рабочей директории:

- `GET /file` читает текстовые файлы и возвращает хеш `sha256:<hex>` в сырых байтах.
- `GET /file/bytes` читает ограниченные окна сырых байтов и возвращает содержимое в base64.
- `POST /file/write` создает или заменяет текстовые файлы.
- `POST /file/edit` применяет одну точную замену текста.

Запись/редактирование являются **строгими маршрутами модификации**: даже на loopback они требуют настроенного Bearer-токена, иначе возвращают `token_required`. Замены и правки требуют последнего `expectedHash` из `GET /file` (или полного окна `GET /file/bytes`). `create` никогда не перезаписывает. Явная запись в игнорируемые пути разрешена, но аудируется. Бинарная запись, удаление/перемещение/mkdir и рекурсивное создание родителей не входят в эту поверхность.

### 3. Открытие сессии

```bash
curl -X POST http://127.0.0.1:4170/session \
  -H 'Content-Type: application/json' \
  -d '{}'
# → {"sessionId":"<uuid>","workspaceCwd":"…","attached":false}
```

Параметр `cwd` можно опустить — маршрут использует привязанную рабочую директорию демона. Если указать `cwd`, не соответствующий привязанной директории, возвращается `400 workspace_mismatch` (демон привязан ровно к одной рабочей директории; запустите отдельный демон для другой).

Второй клиент, отправляющий запрос к `/session` (с любым подходящим `cwd` или без него), получает `"attached": true` — теперь они используют одного агента.

### 4. Подписка на поток событий (сначала в другом терминале)

```bash
SESSION_ID="<из шага 3>"
curl -N http://127.0.0.1:4170/session/$SESSION_ID/events
# → id: 1
#   event: session_update
#   data: {"id":1,"v":1,"type":"session_update","data":{"sessionUpdate":"agent_message_chunk","content":{"type":"text","text":"…"}}}
```

Строка `data:` — это **полный конверт события** — `{id?, v, type, data, originatorClientId?}` — JSON-строка на одной строке. Полезная нагрузка ACP (блок `sessionUpdate` в этом примере) находится внутри `data` этого конверта. Строки `id:` / `event:` на уровне SSE — это удобство для клиентов EventSource; те же значения появляются внутри JSON-конверта, так что потребители с сырым `fetch` тоже их получают.

Откройте это **до** отправки промпта — буфер воспроизведения SSE хранит последние 8000 событий, так что опоздавший подписчик может наверстать через `Last-Event-ID`, но для простого случая «наблюдать за одним промптом» проще сначала подписаться и дать потоку идти вживую.

Поток генерирует `session_update` (чанки LLM, вызовы инструментов, использование), `permission_request` (инструменту требуется одобрение), `permission_resolved` (кто-то проголосовал), `model_switched`, `model_switch_failed` и терминальные фреймы `session_died` (ребенок агента упал — SSE затем закрывается) и `client_evicted` (ваша очередь переполнена — SSE затем закрывается).

### 5. Отправка промпта (в исходном терминале)

```bash
curl -X POST http://127.0.0.1:4170/session/$SESSION_ID/prompt \
  -H 'Content-Type: application/json' \
  -d '{"prompt":[{"type":"text","text":"What does src/main.ts do?"}]}'
# → {"stopReason":"end_turn"}
```

Команда `curl -N` из шага 4 будет выводить фреймы по мере поступления.

## Аутентификация

Для всего, кроме loopback, **необходимо** передавать Bearer-токен:

```bash
export QWEN_SERVER_TOKEN="$(openssl rand -hex 32)"
qwen serve --hostname 0.0.0.0 --port 4170
# → запуск отказывается без QWEN_SERVER_TOKEN
```

Клиенты затем отправляют `Authorization: Bearer $QWEN_SERVER_TOKEN` в каждом запросе. `/health` освобождается **только на loopback-привязках**, чтобы пробы живучести k8s/Compose внутри пода (где демон слушает на `127.0.0.1`) не требовали учетных данных. На не-loopback привязках (`--hostname 0.0.0.0` и т.д.) `/health` требует токен, как и любой другой маршрут — иначе злоумышленник может сканировать произвольные адреса для подтверждения существования демона. Используйте `/capabilities` для проверки токена «от начала до конца» (он всегда требует аутентификации):

> **Защищенный loopback (`--require-auth`).** Поведение по умолчанию «без токена на loopback» подходит для личного ноутбука, но небезопасно на общих dev-хостах, CI-раннерах или многопользовательских рабочих станциях, где любой локальный пользователь может выполнить `curl 127.0.0.1:4170`. Используйте `--require-auth`, чтобы сделать Bearer-токен обязательным на каждом маршруте — включая `/health` и `/capabilities` — даже при привязке к `127.0.0.1`. Запуск завершается ошибкой без токена. С включенным флагом **неаутентифицированный** клиент не может прочитать `/capabilities`, чтобы узнать, что требуется аутентификация; поверхность обнаружения — это само тело ответа 401. После аутентификации тег `caps.features.require_auth` является подтверждением после аутентификации того, что развертывание защищено (полезно для аудита / интерфейсов соответствия):
>
> ```bash
> qwen serve --require-auth --token "$(openssl rand -hex 32)"
> # → /health, /capabilities, /session, … все требуют Authorization: Bearer …
> curl http://127.0.0.1:4170/health
> # → 401
> curl -H "Authorization: Bearer $TOKEN" http://127.0.0.1:4170/capabilities | jq '.features | index("require_auth")'
> # → 13   (или любой другой индекс — не null после аутентификации означает, что тег присутствует)
> ```

```bash
curl -H "Authorization: Bearer $QWEN_SERVER_TOKEN" http://your-host:4170/capabilities
# → {"v":1,"mode":"http-bridge","features":[...],"modelServices":[],"workspaceCwd":"/path/to/your-project"}
# Неверный токен → 401
```

Сравнение токенов выполняется за константное время (SHA-256 + `crypto.timingSafeEqual`); ответы 401 единообразны для «заголовок отсутствует», «неверная схема» и «неверный токен», так что побочный канал не может их различить.

> **Настройка ручек нагрузки.** `--max-sessions` — это ограничение **на нового дочернего процесса**.  
> Есть ещё три уровня, ограничивающих нагрузку — при настройке для развёртывания с высокой конкурентностью настраивайте их вместе:
>
> - **уровень слушателя**: `--max-connections` / `server.maxConnections=256`  
>   ограничивает сырые TCP-соединения (back-pressure от медленных клиентов).  
> - **подписчики на сессию**: EventBus по умолчанию ограничивает SSE‑подписчиков 64 на сессию; 65‑й клиент получает терминальный `stream_error` и закрывается.  
> - **допуск запросов на сессию**: `--max-pending-prompts-per-session=5` ограничивает количество поставленных в очередь + активных запросов, принятых для одной сессии. При переполнении возвращается `503` с `Retry-After: 5`.  
> - **очередь на подписчика**: буфер на 256 фреймов на одного SSE‑клиента; клиент, превысивший ёмкость, получает терминальный фрейм `client_evicted` и закрывается (один медленный потребитель не может заблокировать демон).
>
> Эти ограничения взаимодействуют: `--max-sessions × 64 подписчика × 256 фреймов` — наихудший сценарий памяти в полёте на уровне EventBus, а `--max-sessions × --max-pending-prompts-per-session` ограничивает принятую работу с запросами на уровне допуска. Стандартные настройки предполагают нагрузку одного пользователя / небольшой команды; повышайте постепенно (и следите за RSS) для мультитенантных развёртываний.

> **Защитные ограничения MCP‑клиентов (issue [#4175](https://github.com/QwenLM/qwen-code/issues/4175) PR 14).** Если в воркспейсе объявлено 30 MCP‑серверов в `mcpServers`, будет запущено 30 клиентов без верхнего ограничения, если его не задать. `--mcp-client-budget=N` ограничивает количество активных MCP‑клиентов; `--mcp-budget-mode={enforce,warn,off}` определяет поведение. По умолчанию — `warn`, когда бюджет задан (снимок показывает предупреждение, но ни один клиент не отклоняется — полезно для измерения реального фан-аута перед включением принудительного режима). Отклонённые серверы в режиме `enforce` получают `disabledReason: 'budget'` в своей ячейке на сервер, а ячейка `budgets[0]` показывает `status: 'error'` + `errorKind: 'budget_exhausted'`. Резервирование слота происходит по имени сервера и сохраняется при переподключениях / таймаутах обнаружения — отклонённый сервер не может забрать слот у здорового.
>
> ⚠️ **Область действия v1: на сессию, а не на воркспейс.** Каждая ACP‑сессия внутри демона имеет свой собственный `Config`/`McpClientManager` (создаётся через `newSessionConfig` для каждой сессии). Бюджет ограничивает количество активных MCP‑клиентов **на сессию**, а не суммарно по всем сессиям в воркспейсе. Снимок по `GET /workspace/mcp` отражает представление bootstrap‑сессии (ячейка несёт `scope: 'session'` для честности). Если вы запускаете 5 одновременных ACP‑сессий с `--mcp-client-budget=10`, у вас может быть до 50 активных MCP‑клиентов во всём демоне — ограничение действует на сессию. **Wave 5 PR 23 (общий пул MCP)** вводит менеджер уровня воркспейса и переводит это на настоящее принуждение на уровне воркспейса.
>
> ```sh
> qwen serve --mcp-client-budget=10 --mcp-budget-mode=warn
> # later, after telemetry shows your real-world distribution:
> qwen serve --mcp-client-budget=10 --mcp-budget-mode=enforce
> ```
>
> Это **не** то же самое, что `MCP_SERVER_CONNECTION_BATCH_SIZE` у claude-code (который ограничивает конкурентность запуска); они ортогональны. PR 23 добавит настоящий общий пул MCP (ячейка `scope: 'workspace'` в `budgets[]` рядом с ячейкой на сессию); PR 14 v1 — это счётчик в процессе + мягкое принуждение на существующем менеджере на сессию.
>
> **Push-события (issue [#4175](https://github.com/QwenLM/qwen-code/issues/4175) PR 14b).** SDK‑клиенты, подписанные на `GET /session/:id/events`, получают типизированные фреймы при пересечении порогов бюджета — `mcp_budget_warning` (синтетическое, срабатывает один раз при каждом пересечении 75% вверх с гистерезисом перевооружения на 37.5%, рекламируется через `mcp_guardrail_events`) и `mcp_child_refused_batch` (объединяется один раз за проход обнаружения в режиме `enforce`; длина 1 при отказе от ленивого порождения из `readResource`). Снимок по `GET /workspace/mcp` по-прежнему является источником истины для состояния после переподключения; события — это границы изменений. Полезно для приборной панели в реальном времени без опроса.

## Модель угроз при развёртывании по умолчанию

- **Только 127.0.0.1** — привязка к loopback, аутентификация не требуется.
- **`--hostname 0.0.0.0` требует токен** — запуск отказывается без него.
- **`LOOPBACK_BINDS` включает IPv6** — `::1` и `[::1]` считаются loopback для правила без токена.
- **Белый список заголовка Host** — при привязке к **loopback** демон проверяет, что `Host:` совпадает с `localhost:port` / `127.0.0.1:port` / `[::1]:port` / `host.docker.internal:port` (без учёта регистра, согласно RFC 7230 §5.4) для защиты от DNS‑ребендинга. **Привязки не к loopback (`--hostname 0.0.0.0`) намеренно обходят белый список Host** — оператор выбрал поверхность атаки, поэтому единственным слоем аутентификации является токен-шлюз; обратные прокси / SNI / закрепление клиентских сертификатов — ответственность оператора, а не демона. Если вам нужна изоляция на основе Host на привязке не к loopback, терминируйте TLS + проверяйте Host на фронтальном прокси.
- **CORS по умолчанию запрещает любой браузерный Origin** — возвращает `403` JSON. Используйте **`--allow-origin <pattern>`** (повторяемый, T2.4 #4514), чтобы пропустить определённые браузерные источники. Каждое значение — либо литерал `*` (любой источник — запуск отказывается, если не настроен токен; для полного усиления защиты на loopback рекомендуется `--require-auth`, так как `/health` и `/demo` по умолчанию остаются без аутентификации на loopback), либо канонический URL‑origin (`<scheme>://<host>[:<port>]`, без завершающего слеша / пути / userinfo). Совпавшие источники получают правильные заголовки ответа CORS (`Access-Control-Allow-Origin: <echoed>`, `Vary: Origin`, плюс стандартные методы / заголовки / max-age и открытый `Retry-After`); несовпавшие источники всё равно получают 403 с той же обёрткой, что и стандартная стена. `caps.features.allow_origin` рекламируется условно, чтобы SDK / webui‑клиенты могли предварительно проверить, поддерживает ли демон кросс‑доменные запросы, прежде чем их отправлять. Пример: `qwen serve --allow-origin http://localhost:3000 --allow-origin http://localhost:5173`. Loopback‑запросы с собственным источником (например, страница `/demo`) не затрагиваются — отдельный шим удаления Origin обрабатывает их независимо от `--allow-origin`. **Браузерные webui без настроенного `--allow-origin`** по-прежнему возвращаются к тем же вариантам Stage 1: упаковка в нативную оболочку (Electron/Tauri), чтобы не отправлялся заголовок `Origin`, или размещение демона за обратным прокси с тем же источником.
- **Запущенный дочерний процесс `qwen --acp` наследует окружение демона** с одним явным удалением: `QWEN_SERVER_TOKEN` удаляется перед запуском дочернего процесса (собственный токен демона; агенту он не нужен). Всё остальное — `OPENAI_API_KEY` / `ANTHROPIC_API_KEY` / `QWEN_*` / `DASHSCOPE_API_KEY` / ваш собственный `modelProviders[].envKey` / и т.д. — передаётся, потому что агенту они легитимно нужны для аутентификации в LLM. **Это намеренно, а не песочница.** Агент работает с тем же UID и имеет доступ к shell‑инструментам, поэтому всё в `~/.bashrc` / `~/.aws/credentials` / `~/.npmrc` доступно при инъекции промпта в любом случае. Проброс переменных окружения — это не граница безопасности; ею является доверие к пользователю. Не запускайте `qwen serve` под идентификатором, у которого есть учетные данные в окружении, которые вы не доверили бы агенту.
- **Ограниченные очереди SSE на подписчика** — медленный клиент, переполнивший свою очередь, получает терминальный фрейм `client_evicted` и закрывается; один зависший потребитель не может заблокировать демон.
- **Ограничение на приём запросов на сессию** — по умолчанию 5 принятых, но ещё не завершённых запросов на сессию. Ошибочный клиент не может поставить в очередь бесконечное количество обещаний запросов или временных ожиданий SSE для одной сессии.
- **Graceful shutdown** — SIGINT/SIGTERM сначала завершают дочерние процессы агента, а затем закрывают слушатель (10-секундный дедлайн на дочерний процесс).

> ⚠️ **Известная проблема Stage 1 — разрешения глобальны для демона, а не на сессию (BUy4H).** `pendingPermissions` находится на уровне демона; любой клиент, владеющий токеном, может голосовать по любому `requestId` для любой видимой ему сессии (а SSE‑события `permission_request` содержат requestId в своей полезной нагрузке). Это приемлемо в модели доверия одного пользователя / небольшой команды, где каждый аутентифицированный клиент — это один и тот же человек или коллеги, которым он доверяет. Stage 1.5 перейдёт на `POST /session/:id/permission/:requestId` + карта ожидающих разрешений на сессию + идентификация на клиента (обязательное требование #3 из downstream‑обзора); до тех пор не запускайте `qwen serve` за токеном, общим с недоверенными сторонами.
>
> ⚠️ **Известная проблема Stage 1 — тело `POST /session/:id/prompt` ограничено 10 МБ (BUy4L).** Мультимодальные запросы, содержащие изображения / PDF / аудио размером более 10 МБ, не пройдут этап разбора тела до выполнения логики маршрута (нет потоковой передачи, нет прерывания при загрузке). Обходной путь: уменьшить содержимое на стороне клиента или передать ссылку на путь и позволить агенту прочитать файл через `readTextFile`. Stage 1.5 будет принимать `multipart/form-data` или chunked‑кодирование на `/prompt`, чтобы большие запросы не упирались в потолок.
>
> ⚠️ **Известная проблема Stage 1 — фантомные SSE‑соединения за NAT.** Демон обнаруживает мёртвых клиентов через TCP back-pressure на heartbeats (интервал 15 с). Клиент, который исчезает БЕЗ TCP RST (например, NAT‑устройство молча сбрасывает бездействующие потоки), оставляет сокет на уровне ядра «живым» до истечения тайм-аута keepalive‑проб Node — обычно около 2 часов на настройках Linux по умолчанию. При развёртывании с `--hostname 0.0.0.0` за таким NAT фантомные SSE‑соединения могут накапливаться и в итоге достигнуть потолка в 256 `server.maxConnections`.
>
> Установите [`--writer-idle-timeout-ms <n>`](#deadlines-and-writer-idle-timeout) (issue [#4514](https://github.com/QwenLM/qwen-code/issues/4514) T2.9), чтобы закрыть пробел явным тайм-аутом бездействия на уровне приложения: если ни одна запись не была успешно сброшена в течение `n` мс, демон отправляет терминальный фрейм `client_evicted` с `reason: 'writer_idle_timeout'` и закрывает поток. Флаг отключён по умолчанию для сохранения устаревшего контракта — операторы в сетях, которые поглощают RST, должны выбрать значение значительно выше 15‑секундного интервала heartbeat (например, `60000`–`300000`), чтобы легитимные бездействующие соединения не были вытеснены, а действительно зависшие писатели были своевременно удалены. Предварительно проверьте `caps.features.includes('writer_idle_timeout')` из вашего SDK, чтобы подтвердить поддержку демоном.

### Дедлайны и тайм-аут бездействия писателя

Issue [#4514](https://github.com/QwenLM/qwen-code/issues/4514) T2.9 включает два опциональных флага, которые закрывают пробелы длительных / удалённых развёртываний, не покрываемые 15‑секундным heartbeat + AbortSignal. Оба отключены по умолчанию — однопользовательские loopback‑рабочие процессы остаются без изменений бит-в-бит.

| Флаг                           | Переменная окружения                | По умолчанию | Описание                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                    |
| ------------------------------ | ----------------------------------- | ------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `--prompt-deadline-ms <n>`     | `QWEN_SERVE_PROMPT_DEADLINE_MS`     | не задано    | Серверное ограничение по настенным часам на один `POST /session/:id/prompt`. По истечении демон прерывает AbortController запроса и возвращает HTTP `504` с `{code:"prompt_deadline_exceeded", errorKind:"prompt_deadline_exceeded", deadlineMs:n}`. Поле `deadlineMs` в теле запроса может УМЕНЬШИТЬ эффективный дедлайн ниже значения флага, но никогда не увеличить его. Тег возможности (условный): `prompt_absolute_deadline`.                                                                                                                                                                                                          |
| `--writer-idle-timeout-ms <n>` | `QWEN_SERVE_WRITER_IDLE_TIMEOUT_MS` | не задано    | Тайм-аут бездействия для каждого SSE‑соединения. Если ни одна запись не была УСПЕШНО сброшена в течение `n` мс — ни реальное событие, ни 15‑секундный heartbeat — демон отправляет терминальный фрейм `client_evicted` с `data.reason = 'writer_idle_timeout'` (отражено в `data.errorKind`) и закрывает поток. **Выберите значение значительно выше 15‑секундного heartbeat** (например, `30000`–`300000`), чтобы легитимные бездействующие потоки не были вытеснены; значения `< 15000` ВЫТЕСНЯТ иначе здоровые бездействующие соединения до первого heartbeat (предназначено только для тестов / краткосрочных сессий разработки). Тег возможности (условный): `writer_idle_timeout`. |

Оба флага принимают положительное целое число в миллисекундах; `0`, `NaN`, нецелые или отрицательные значения отклоняются при запуске с чётким сообщением об ошибке. Флаг CLI имеет приоритет над переменной окружения; явное поле `ServeOptions` (встраиваемые вызывающие) имеет приоритет над окружением. Потребители SDK должны предварительно проверять соответствующий тег возможности, прежде чем полагаться на то или иное поведение — демоны, предшествующие этому PR, не включают оба тега, и поле `deadlineMs` в запросе молча игнорируется.

## Развёртывание с несколькими сессиями и несколькими воркспейсами

Согласно [#3803](https://github.com/QwenLM/qwen-code/issues/3803) §02, каждый процесс `qwen serve` привязывается **к одному воркспейсу** при запуске. Внутри этого воркспейса он мультиплексирует N сессий на один дочерний процесс `qwen --acp` через встроенную карту сессий агента — сессии разделяют процесс дочернего элемента / состояние OAuth / кэш чтения файлов / разбор иерархической памяти.

Для размещения **нескольких воркспейсов** (один пользователь, несколько репозиториев; или несколько пользователей на одном хосте) запустите **несколько процессов демона** — по одному на воркспейс, каждый на своём порту, под управлением systemd / docker-compose / k8s / эталонного оркестратора `qwen-coordinator`. Компромисс намеренный: один воркспейс на дочерний процесс означает, что `loadSettings(cwd)` / OAuth / область MCP‑серверов остаются согласованными с привязанной директорией и не дрейфуют между запросами.

> **Подпишитесь ДО отправки `modelServiceId` при аттаче.** Когда клиент отправляет `POST /session` с `modelServiceId`, а в воркспейсе уже есть сессия, работающая с другой моделью, демон выполняет внутренний вызов `setSessionModel` — ошибки НЕ передаются как HTTP‑ошибка (сессия продолжает работать на текущей модели). Видимым сигналом ошибки является событие `model_switch_failed` в SSE‑потоке сессии. Если вы вызываете `POST /session` и только ПОТОМ открываете `GET /session/:id/events`, вы пропустите событие ошибки и будете молча общаться с неправильной моделью. Откройте SSE‑поток сначала или передайте `Last-Event-ID: 0` при подписке, чтобы воспроизвести самое старое доступное событие кольцевого буфера.

Для обработки нескольких **пользователей** (каждый со своей квотой, журналом аудита, песочницей) или масштабирования за пределы одного процесса (бюджет холодного старта, количество файловых дескрипторов, RSS) запускайте один демон на воркспейс на пользователя за внешним оркестратором. Этот оркестратор (мультитенантность / OIDC / квоты / аудит / k8s) **не входит** в рамки проекта qwen-code — см. issue [#3803](https://github.com/QwenLM/qwen-code/issues/3803) «External Reference Architecture» для указаний по проектированию.

## Загрузка и возобновление сохранённой сессии

Демон предоставляет ACP‑потоки `session/load` и возобновления через HTTP по двум маршрутам:

| Маршрут                     | Когда использовать                                                                                                                                                                                                                                                                                                                                                                    |
| --------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `POST /session/:id/load`   | У клиента **нет** отображённой истории (холодное переподключение, выбор-затем-открытие). Демон воспроизводит все сохранённые ходы через SSE, чтобы подписчики видели полную стенограмму. Тег возможности: `session_load`.                                                                                                                                                              |
| `POST /session/:id/resume` | У клиента уже есть ходы на экране, и ему нужен только дескриптор на стороне демона. Контекст модели восстанавливается на стороне агента без воспроизведения UI — SSE‑поток остаётся чистым. Тег возможности: `session_resume` (`unstable_session_resume` остаётся устаревшим псевдонимом для старых клиентов).                                                                     |

TypeScript SDK предоставляет обе функции как статические фабрики на `DaemonSessionClient`:

```ts
import { DaemonClient, DaemonSessionClient } from '@qwen-code/sdk';

const client = new DaemonClient({ baseUrl: 'http://127.0.0.1:4170' });

// Cold reconnect — daemon will replay history through SSE.
const session = await DaemonSessionClient.load(client, 'persisted-id');

// Or, if your UI already has the history, skip the replay:
// const session = await DaemonSessionClient.resume(client, 'persisted-id');

for await (const event of session.events()) {
  // First the replayed `session_update` frames (load only),
  // then live events.
}
```

Предварительно проверяйте `caps.features.session_load` / `caps.features.session_resume` перед вызовом — старые демоны возвращают `404`. `unstable_session_resume` всё ещё рекламируется как устаревший псевдоним для совместимости. Одновременные запросы одного и того же действия для одного идентификатора объединяются; взаимоблокировки действий (гонка `load` и `resume`) получают `409 restore_in_progress` с `Retry-After: 5`. См. [справочник по протоколу](../developers/qwen-serve-protocol.md) для полной обёртки ошибки.

Примечание: воспроизведение истории ограничено кольцевым буфером SSE (по умолчанию 8000 фреймов). Длинные истории с многословными ходами могут превысить это — самые старые фреймы молча отбрасываются. Для очень длинных сессий предпочитайте `resume` и полагайтесь на локально сохранённый UI клиента.

## Модель долговечности

**Сессии всё ещё эфемерны в Stage 1 при перезапусках демона**, но сохранённые на диске сессии можно перезагрузить:

- Сбой дочернего процесса публикует `session_died` и удаляет активную сессию из карт демона. Сохранённая на диске сессия **может** быть перезагружена через `POST /session/:id/load`, если можно породить нового агента.
- Перезапуск демона теряет все активные сессии в полёте. Сохранённые сессии остаются на диске и могут быть загружены новым процессом демона с соблюдением тех же правил привязки к воркспейсу.
- Длительные отключения клиента (>5 мин на многословном ходе) могут выйти за пределы кольцевого буфера SSE (по умолчанию 8000 фреймов) — переподключение с `Last-Event-ID` может удаться, но состояние может быть несогласованным. Для мобильных / нестабильных сетей планируйте повторное открытие SSE при длительных обрывах или вызывайте `POST /session/:id/load` для воспроизведения с диска.
- Файловые операции (`writeTextFile`) атомарны при сбоях (запись-затем-переименование); они не атомарны при перезапусках демона в смысле воспроизведения — запись файла либо состоялась, либо нет.

Если ваша интеграция требует серверной долговечности между перезапусками сверх того, что покрывает `session/load` (например, управляемые сервером очереди повторных попыток), вам всё равно потребуется восстановление состояния на уровне приложения. Не храните долгоживущее, чувствительное к перезапускам состояние внутри сессии демона.

## Гарантии времени выполнения Stage 1.5+

Контракт Stage 1 рассчитан на прототипирование. Согласно [#3889 chiga0 downstream-consumer review](https://github.com/QwenLM/qwen-code/pull/3889#issuecomment-4427875644), следующее **не** входит в Stage 1 — производственные интеграции требуют Stage 1.5+ перед тем, как на них полагаться:
**Блокирующие проблемы для серьёзного downstream-использования:**

1. **`loadSession` / `unstable_resumeSession` по HTTP** — без этого ни одна интеграция не переживёт падения дочернего процесса или перезапуска демона, и любой оркестратор, управляющий демоном, не сможет восстановить состояние.
2. **Постоянная идентификация клиента (pair-токены + отзыв per-клиент)** — в Stage 1 используется один общий bearer; утечка токена отзывает всех, а `originatorClientId` объявляется самим клиентом, а не проставляется демоном на основе аутентифицированной идентичности.

**Базовый уровень надёжности:**

3. ~~**Путь heartbeat, инициируемый клиентом**~~ — реализовано в [#4175](https://github.com/QwenLM/qwen-code/issues/4175), PR 9. `POST /session/:id/heartbeat` записывает timestamp последнего получения на стороне демона (тег capability `client_heartbeat`); хелперы SDK — `DaemonClient.heartbeat()` / `DaemonSessionClient.heartbeat()`.
4. **Событие `permission_already_resolved`**, когда голос проигрывает гонку «первый ответчик» — сейчас UI вынуждены выводить состояние из `404`.
5. ~~**Увеличенное кольцо воспроизведения (replay ring)**~~ — увеличено до 8000. **Настраиваемое per-сессию кольцо** ещё открыто — для мобильных / нагрузок с частыми оборотами могут потребоваться per-сессионные переопределения.
6. **Событие `slow_client_warning` перед `client_evicted`** — мягкое противодавление, чтобы медленные клиенты могли саморегулироваться (уменьшить глубину рендеринга, отбрасывать чанки) до завершения.

**Эргономика интеграции:**

7. **`POST /session/:id/_meta` для контекста в стиле IM** — per-сессионная пара «ключ-значение», прикрепляемая к последующим подсказкам (chat id, sender, thread id), заменяет импровизацию по каждому каналу.
8. **`/capabilities` — фактическое согласование возможностей** — `protocol_versions: { acp: '0.14.x', daemon_envelope: 1 }`, чтобы клиенты могли обнаруживать расхождение, а не падать в «неизвестный фрейм, игнорирую».
9. **Документация по надёжности первого класса** (этот раздел) — уже описана выше.

Полная дорожная карта конвергенции отслеживается в [#3803](https://github.com/QwenLM/qwen-code/issues/3803).

## Границы Stage 1 — что мы не будем исправлять в Stage 1.5

Два структурных решения явно не являются целями для основной дорожной карты Stage 1 / 1.5 / 2. Если ваш сценарий использования зависит от них, планируйте обходные пути, а не ждите от нас.

### Состояние сессии изменяется только локально (согласно [обзору LaZzyMan #4270256721](https://github.com/QwenLM/qwen-code/pull/3889#pullrequestreview-4270256721))

План Stage 1.5 описывает TUI как подписчика in-process EventBus. На практике **TUI UI строго шире, чем wire-протокол**:

- **Локальный UI** — около 15 компонентов Ink-диалогов (`ModelDialog`, `MemoryDialog`, `PermissionsDialog`, `SessionPicker`, `WelcomeBackDialog`, `FolderTrustDialog`, …) и slash-команды `local-jsx` (`/ide`, `/auth`, `/init`, `/resume`, `/rename`, `/delete`, `/language`, `/arena`, …) рендерят терминальный Ink JSX. Удалённые клиенты по HTTP/SSE не могут эквивалентно рендерить Ink, и эти потоки не генерируют wire-события.
- **Мутации состояния сессии без wire-событий** — `/approval-mode`, `/memory add`, `/mcp add-server`, `/agents`, `/tools enable/disable`, `/auth`, `/init` (запись `CLAUDE.md`) — все меняют поведение агента, но только `/model` сейчас публикует событие (`model_switched`).

**Выбор Stage 1 — вариант (A) из обзора**: не продвигать эти мутации до wire-событий. У двух режимов развёртывания разные последствия.

#### Режим 1 — безголовый `qwen serve` (этот PR)

Внутри демона не запускается TUI-оболочка. Перечисленные выше slash-команды **не существуют** в этом режиме — нет терминального UI, из которого их можно отдать. Состояние сессии поэтому:

- **Зафиксировано на момент запуска** для `approval-mode` / `memory` / `agents` / `tools` allowlist / `auth` — всё загружается из настроек + диска при старте дочернего процесса `qwen --acp`; неизменно на время жизни сессии. Определённые в настройках MCP-серверы также заморожены при загрузке, но **добавленные во время выполнения** серверы (через `POST /workspace/mcp/servers`) могут быть добавлены или удалены без перезапуска.
- **Изменяемо по HTTP** через `POST /session/:id/model` (публикует `model_switched`), `POST /workspace/mcp/servers` / `DELETE /workspace/mcp/servers/:name` (публикует `mcp_server_added` / `mcp_server_removed`) и голосование за разрешения (`POST /permission/:requestId`).

**Следствие:** удалённые клиенты в безголовом режиме видят **полное состояние сессии**. Никакой TUI не скрывает дополнительного состояния; расхождение невозможно. Если вы хотите изменить `approval-mode`, перезапустите демон с новыми настройками. MCP-серверы теперь можно добавлять/удалять во время выполнения через маршруты мутации (`POST /workspace/mcp/servers`, `DELETE /workspace/mcp/servers/:name`) — см. [Управление MCP-серверами во время выполнения](#runtime-mcp-server-management-issue-4514).

#### Режим 2 — Stage 1.5 `qwen --serve` с сосуществующим TUI (не в этом PR)

Когда Stage 1.5 запустит `qwen --serve` (процесс TUI сосуществует на том же HTTP-сервере), TUI **существует** наряду с удалёнными клиентами. Локальный оператор, вводящий `/approval-mode yolo` или `/mcp add-server`, мутирует состояние сессии, а у удалённых клиентов по HTTP нет события, чтобы наблюдать это изменение.

В этом режиме TUI является **«супер-клиентом»** — он видит ту же беседу агента, что и удалённые клиенты, И может мутировать состояние сессии, недоступное удалённым клиентам. Асимметрия:

- ✅ И TUI, и удалённые клиенты видят одни и те же сообщения агента, вызовы инструментов, файловые diff, запросы разрешений.
- ❌ Только TUI видит/мутирует состояние approval-mode / memory / списка MCP-серверов / agents / allowlist инструментов / auth.

**Следствие в режиме 2:** если UI удалённого клиента пытается отразить настройки сессии, после любой slash-команды TUI может возникнуть расхождение. Удалённые клиенты должны **перезапрашивать состояние при подключении / переподключении** (используйте `Last-Event-ID: 0`, чтобы воспроизвести самое старое событие из кольца для таких вещей, как `model_switched`); они НЕ должны полагаться на инкрементные события для мутаций со стороны TUI.

#### Почему (A), а не (B) (продвижение мутаций к семейству событий `session_state_changed`)

Вариант (B) более амбициозен, но привязывает Stage 1.5 к существенно большей wire-поверхности, которая также должна чисто проходить через запланированный in-process рефакторинг. Мы предпочитаем честно придерживаться меньшей области. Работа по таксономии событий состояния сессии — перечисление того, какие TUI-потоки являются локальными по дизайну, а какие потенциально могут перейти на wire в рамках будущего опционального расширения (B) — переносится в [#3803](https://github.com/QwenLM/qwen-code/issues/3803), а не в код Stage 1.5.

### N параллельных сессий используют один дочерний процесс `qwen --acp`

Несколько сессий в одной рабочей области **используют один дочерний процесс `qwen --acp`** через собственную поддержку мульти-сессионности агента (`packages/cli/src/acp-integration/acpAgent.ts:194: private sessions: Map<string, Session>`). Мост вызывает `connection.newSession({cwd, mcpServers})` для каждой сессии — агент хранит их в своей карте sessions и демультиплексирует per-вызов sessionId.

Конкретная стоимость при N=5 сессиях в одной рабочей области:

| Ресурс                              | На сессию       | При N=5                      |
| ----------------------------------- | --------------- | ---------------------------- |
| Процесс Node.js демона              | один            | **30–50 MB** (один демон)    |
| Дочерний процесс `qwen --acp`      | общий           | **60–100 MB** (один дочерний) |
| Дочерние процессы MCP-серверов      | per-сессия      | 3×N, если конфиги различаются |
| `FileReadCache` (в куче дочернего)  | общий           | парсится один раз            |
| `CLAUDE.md` / иерархический memory  | общий           | парсится один раз            |
| Состояние OAuth refresh-токена      | общий           | **один путь обновления**     |
| Автоматически изученные факты memory| общий           | одна база знаний на дочерний  |
| Холодный старт                      | только первый   | <200 мс после первой сессии  |

Мост поддерживает **один канал на демон** (один демон на рабочую область, согласно §02). Канал остаётся активным, пока жива хотя бы одна сессия; последний `killSession` (или сбой на уровне канала) убивает дочерний процесс.

**Дочерние процессы MCP-серверов** пока per-сессионные — каждая сессия может указывать разные серверы, поэтому они порождаются независимо. В Stage 1.5 планируется: подсчёт ссылок на дочерние процессы MCP-серверов по `(workspace, config-hash)`, чтобы одинаковые конфиги использовались совместно. Не входит в область этого PR.

**Peer-агенты (Cursor / Continue / Claude Code / OpenCode / Gemini CLI) все используют один процесс для нескольких сессий.** qwen-code соответствует им на уровне агента; мост Stage 1 в этом PR делает ту же архитектуру видимой по HTTP.

## Вход в удалённый демон (issue #4175 PR 21)

Когда демон работает на удалённом поде (нет общего дисплея с вами), клиент может запустить OAuth-поток устройства по HTTP. Демон сам опрашивает IdP; ваша задача — просто открыть URL на любом устройстве, где есть браузер.

> [!note]
>
> Бесплатный тариф Qwen OAuth был прекращён 2026-04-15. Примеры с `qwen-oauth`
> ниже документируют форму протокола device-flow и идентификатор устаревшего провайдера;
> новые настройки должны использовать поддерживаемого в настоящее время провайдера аутентификации.

```bash
# 1. Запустить поток. Демон связывается с IdP, возвращает код + URL.
curl -X POST http://127.0.0.1:4170/workspace/auth/device-flow \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"providerId":"qwen-oauth"}'
# → 201 {
#     "deviceFlowId": "fa07c61b-…",
#     "userCode": "USER-1",
#     "verificationUri": "https://chat.qwen.ai/api/v1/oauth2/device",
#     "verificationUriComplete": "https://chat.qwen.ai/...?user_code=USER-1",
#     "expiresAt": 1700000600000,
#     "intervalMs": 5000,
#     "attached": false
#   }

# 2. Откройте URL на телефоне/ноутбуке, введите код пользователя.
# 3. Опрашивайте завершение (или подпишитесь на SSE-событие auth_device_flow_authorized):
curl http://127.0.0.1:4170/workspace/auth/device-flow/fa07c61b-… \
  -H "Authorization: Bearer $TOKEN"
# → статус: pending → authorized
```

TypeScript SDK оборачивает оба шага в один хелпер:

```ts
import { DaemonClient } from '@qwen-code/sdk';

const client = new DaemonClient({ baseUrl, token });
const flow = await client.auth.start({ providerId: 'qwen-oauth' });
console.log(`Open ${flow.verificationUri}\nCode: ${flow.userCode}`);
const result = await flow.awaitCompletion({ signal: abortCtrl.signal });
// result.status === 'authorized'
```

**Демон никогда не открывает браузер от вашего имени.** Даже при локальной работе демон остаётся пассивным — он возвращает URL и позволяет SDK / пользователю решать, где его открыть. Это намеренно: демон на безголовом поде, который вызвал бы `xdg-open`, молча бы провалился, скрывая реальную поверхность аутентификации. Повторяйте UX `gh auth login` с «Нажмите Enter, чтобы открыть браузер» в вашем клиенте.

**`--require-auth` и удобство разработки.** Маршруты device-flow используют строгий шлюз мутации (PR 15), поэтому на локальной петле без токена возвращается `401 token_required`. Локально самый простой способ обойти это во время разработки — `qwen serve --token=dev-token`; `--require-auth` не нужен, если только вы не ужесточаете настройки loopback.

**Ограничение между демонами.** `oauth_creds.json` является общим для демонов (`~/.qwen/oauth_creds.json`), поэтому успешный вход в демон A автоматически подхватывается следующим обновлением токена демона B — но клиенты SDK демона B не получат событие `auth_device_flow_authorized` (события привязаны к конкретному демону).

**Перехват между клиентами.** Два SDK-клиента на одном демоне, оба выполняющие `POST /workspace/auth/device-flow` для одного провайдера, получают синглтон на провайдера: первый вызов запускает свежий запрос IdP и возвращает `attached: false`; второй вызов возвращает УЖЕ СУЩЕСТВУЮЩУЮ запись с `attached: true`. Перехват записывается в журнал аудита (под `X-Qwen-Client-Id` второго клиента), но НЕ генерирует отдельного события — оба клиента в итоге наблюдают ОДИНАКОВОЕ событие `auth_device_flow_authorized`, когда пользователь завершает страницу IdP. Если ваш UI различает «я это начал» и «чужой поток, к которому я присоединился», проверяйте поле `attached`, возвращаемое `start()`.

## Файл журнала демона

`qwen serve` записывает per-процессный диагностический журнал по пути:

```
${QWEN_RUNTIME_DIR или ~/.qwen}/debug/daemon/serve-<pid>-<workspaceHash>.log
```

Символическая ссылка `latest` в той же директории всегда указывает на текущий журнал процесса, поэтому `tail -f ~/.qwen/debug/daemon/latest` будет следовать за запущенным демоном.

Журнал содержит сообщения жизненного цикла, ошибки маршрутов (с контекстом `route=` и `sessionId=`), stderr дочернего ACP-процесса и — при установке `QWEN_SERVE_DEBUG=1` — дополнительные отладочные данные моста. Строки, которые сейчас идут в stderr, по-прежнему туда идут; файловый журнал **дополнительный**, а не заменяющий.

### Отключение

Установите `QWEN_DAEMON_LOG_FILE=0` (или `false`/`off`/`no`), чтобы полностью пропустить файловое журналирование. Вывод в stderr не затрагивается.

### Отношение к отладочным журналам сессии

По-сессионные отладочные журналы (`~/.qwen/debug/<sessionId>.txt` и символическая ссылка `~/.qwen/debug/latest`) независимы. Журнал демона находится в поддиректории `daemon/`; семантика по-сессионной отладки не меняется этой возможностью.

### Нет ротации

Журнал демона дописывается бесконечно. Ротируйте вручную, если он становится большим. В будущем может быть добавлена автоматическая ротация; отслеживайте через доработки к [#4548](https://github.com/QwenLM/qwen-code/issues/4548).

## Управление MCP-серверами во время выполнения (issue [#4514](https://github.com/QwenLM/qwen-code/issues/4514))

Добавляйте или удаляйте MCP-серверы во время выполнения без перезапуска демона. Записи, добавленные во время выполнения, живут в эфемерном наложении, которое **затеняет** определённые в настройках серверы с тем же именем; базовая конфигурация `settings.json` / `mcpServers` никогда не изменяется.

**Предварительная проверка:** проверьте `caps.features` на наличие `mcp_server_runtime_mutation` перед вызовом любого маршрута. Старые демоны без этого тега вернут `404`.

### `POST /workspace/mcp/servers` — добавить MCP-сервер во время выполнения

Строгий шлюз (требуется bearer-токен). Подключает сервер немедленно через живой `McpClientManager` и обнаруживает его инструменты.

Запрос:

```json
{
  "name": "my-server",
  "config": {
    "command": "npx",
    "args": ["-y", "@my-org/mcp-server"]
  }
}
```

`name` должен содержать только буквенно-цифровые символы, `_` и `-` (максимум 256 символов). `config` — тот же объект конфигурации MCP-сервера, который используется в записях `mcpServers` файла `settings.json` (поля, зависящие от транспорта: `command`/`args` для stdio, `url` для SSE/HTTP). Чувствительные к безопасности поля (`trust`, `env`, `cwd`, `oauth`, `headers`, `authProviderType`, `includeTools`, `excludeTools`, `type`) отбрасываются демоном и игнорируются.

Ответ (200) — успех:

```json
{
  "name": "my-server",
  "transport": "stdio",
  "replaced": false,
  "shadowedSettings": false,
  "toolCount": 3,
  "originatorClientId": "client-1"
}
```

- `replaced: true` — запись времени выполнения с таким же именем уже существовала, и отпечаток конфига отличается; старое соединение разорвано, установлено новое. Когда отпечаток совпадает (идемпотентное повторное добавление), `replaced` равно `false`.
- `shadowedSettings: true` — в настройках существует сервер с таким же именем; запись времени выполнения теперь его затеняет. Запись в настройках не изменяется и вновь появляется, если запись времени выполнения будет удалена.
- `toolCount` — количество инструментов, обнаруженных на вновь подключённом сервере.

Ответ (200) — мягкий отказ (режим предупреждения о бюджете):

```json
{
  "name": "my-server",
  "skipped": true,
  "reason": "budget_warning_only"
}
```

Возвращается, когда `--mcp-budget-mode=warn` и добавление сервера превысило бы настроенный `--mcp-client-budget`. Сервер НЕ подключается. Вызывающие стороны должны сообщить пользователю о давлении бюджета.

Ошибки:

| Статус | Код                        | Когда                                                                                          |
| ------ | -------------------------- | ---------------------------------------------------------------------------------------------- |
| `400`  | `invalid_server_name`      | Имя пустое, превышает 256 символов или содержит символы за пределами `[A-Za-z0-9_-]`          |
| `400`  | `missing_required_field`   | `config` отсутствует или не является ненулевым объектом                                        |
| `400`  | `invalid_client_id`        | Заголовок `X-Qwen-Client-Id` присутствует, но не зарегистрирован для этой рабочей области     |
| `400`  | `invalid_config`           | Форма конфига отклонена валидатором транспорта MCP                                            |
| `401`  | `token_required`           | Не настроен bearer-токен (строгий шлюз)                                                        |
| `409`  | `mcp_budget_would_exceed`  | `--mcp-budget-mode=enforce` и бюджет исчерпан                                                 |
| `502`  | `mcp_server_spawn_failed`  | Процесс сервера завершился или истекло время ожидания при подключении; тело содержит `serverName`, `exitCode`, `stderr` |
| `503`  | `acp_channel_unavailable`  | Нет живого ACP-дочернего процесса (ещё не создана ни одна сессия)                              |

### `DELETE /workspace/mcp/servers/:name` — удалить MCP-сервер времени выполнения

Строгий шлюз. Отключает сервер и удаляет его из наложения времени выполнения. Идемпотентно — удаление имени, которое никогда не добавлялось, возвращает ответ пропуска (не ошибку).

Параметр пути `:name` — URL-кодированное имя сервера.

Ответ (200) — успех:

```json
{
  "name": "my-server",
  "removed": true,
  "wasShadowingSettings": false,
  "originatorClientId": "client-1"
}
```

- `wasShadowingSettings: true` — удалённая запись времени выполнения затеняла определённый в настройках сервер с таким же именем. Эта запись в настройках теперь не затенена и будет использоваться при следующем обнаружении/перезапуске.

Ответ (200) — идемпотентный пропуск:

```json
{
  "name": "ghost",
  "skipped": true,
  "reason": "not_present"
}
```

Возвращается, когда имени нет в наложении времени выполнения (оно всё ещё может существовать в настройках — записи в настройках нельзя удалить через этот маршрут).

Ошибки:

| Статус | Код                        | Когда                                                                          |
| ------ | -------------------------- | ------------------------------------------------------------------------------ |
| `400`  | `invalid_server_name`      | Имя пустое, превышает 256 символов или содержит символы за пределами `[A-Za-z0-9_-]` |
| `400`  | `invalid_client_id`        | Заголовок `X-Qwen-Client-Id` присутствует, но не зарегистрирован для этой рабочей области |
| `401`  | `token_required`           | Не настроен bearer-токен (строгий шлюз)                                       |
| `503`  | `acp_channel_unavailable`  | Нет живого ACP-дочернего процесса                                             |

### Семантика затенения

Записи времени выполнения образуют эфемерное наложение поверх определённых в настройках MCP-серверов:

- **Добавление** сервера времени выполнения с тем же именем, что и запись в настройках, **затеняет** её — конфиг времени выполнения имеет приоритет. Исходная запись в настройках не изменяется.
- **Удаление** сервера времени выполнения, который затенял запись в настройках, **снимает затенение** — конфиг, определённый в настройках, снова становится активным при следующем подключении.
- **Перезапуск демона** теряет все записи времени выполнения. Только определённые в настройках серверы сохраняются между перезапусками. Серверы времени выполнения ограничены временем жизни сессии.
- **`GET /workspace/mcp`** возвращает объединённое представление — и определённые в настройках, и серверы времени выполнения появляются в массиве `servers[]`. В текущем снимке нет wire-level различия между двумя источниками.

### События

Оба маршрута генерируют **события SSE, относящиеся к рабочей области** (все активные шины сессий их получают):

| Событие              | Генерируется когда                   | Поля payload                                                                      |
| -------------------- | ------------------------------------ | --------------------------------------------------------------------------------- |
| `mcp_server_added`   | `POST` успешен (не пропущен)         | `name`, `transport`, `replaced`, `shadowedSettings`, `toolCount`, `originatorClientId` |
| `mcp_server_removed` | `DELETE` успешен (не пропущен)       | `name`, `wasShadowingSettings`, `originatorClientId`                             |
Пропущенные ответы (`budget_warning_only`, `not_present`) НЕ порождают события.

События, связанные с бюджетом, из существующей поверхности `mcp_guardrail_events` (`mcp_budget_warning`, `mcp_child_refused_batch`) также срабатывают, когда изменения во время выполнения превышают пороговое значение бюджета.

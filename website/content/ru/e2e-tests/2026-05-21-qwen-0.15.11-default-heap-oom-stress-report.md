# Qwen Code 0.15.11 — Отчёт о стресс-тестировании с Heap OOM по умолчанию

Дата: 2026-05-21

## Область тестирования

Данный отчёт документирует раунд стресс-тестирования с heap по умолчанию на последней локальной сборке Qwen Code `0.15.11`.
Цель этого раунда — проверить: без искусственного снижения лимита памяти, можно ли ещё воспроизвести
OOM длинных сессий, упомянутый в issue, и существуют ли новые риски при более экстремальных сценариях большого вывода.

В раунде используются три модели:

- `pai/glm-5`
- `qwen3.6-plus`
- `DeepSeek/deepseek-v4-pro`

Тестирование делится на две части:

1. Реальные длительные задачи с многопоточными циклами ревью с несколькими agent.
2. Стресс-тестирование amplified foreground stdout — имитация крупномасштабного вывода shell stdout для усиления нагрузки на путь tool-output.

## Основные выводы

Последняя локальная сборка показывает значительно лучшие результаты на пути, который больше всего волнует issue — «длинная сессия V8 heap OOM». Основываясь на этом раунде стресс-тестирования с heap по умолчанию, множеством моделей, несколькими agent и длительными задачами, можно считать, что данный PR в основном решил проблему long-session heap OOM, по крайней мере, в текущем измерении — исходный heap OOM не воспроизводится.

В реальных длительных задачах с многопоточными agent было выполнено:

- 23 worker turn
- около `719 094 118` сообщённых total tokens
- 77 вызовов agent tool
- 856 всего вызовов tool

В этой части не было зафиксировано ни одного признака традиционного V8 heap OOM:

- `JavaScript heap out of memory`
- `Reached heap limit`
- `Ineffective mark-compacts near heap limit`
- `Allocation failed`

Максимальный process-tree RSS на этапе реальных длительных задач составил `874,7 MiB`, максимальный root-process RSS — `219,1 MiB`. Это означает, что при heap по умолчанию текущий код не воспроизводит легко тот heap OOM длинных задач, который был в исходном issue.

Второй этап — amplified stdout stress — был более агрессивным. Всего было выполнено 18 попыток payload, охватывающих три модели с размерами foreground stdout payload от `128 MiB` до `2048 MiB`.

Результаты:

- Все три модели успешно прошли payload `1536 MiB`.
- Максимальный успешный process-tree RSS составил `5964,7 MiB` (модель `qwen3.6-plus` при payload `1536 MiB`).
- При payload `2048 MiB` появился новый сбой extreme large-output.

Результаты для `2048 MiB`:

- `pai/glm-5`: `exit=1`, stdout пуст, нет стандартного текста OOM.
- `qwen3.6-plus`: `exit=1`, stdout пуст, нет стандартного текста OOM.
- `DeepSeek/deepseek-v4-pro`: фатальная ошибка V8:
  `Check failed: i::kMaxInt >= len`, стек в
  `v8::String::NewFromOneByte` / `node::StringBytes::Encode` /
  `DecodeUTF8`.

Эта новая проблема не является традиционным long-session heap OOM из исходного issue. Она больше похожа на проблему с ограничением длины строки V8 или обработкой большого вывода, возникающую при декодировании/конструировании multi-GiB foreground stdout в JS string. Рекомендуется отслеживать как follow-up по large-output, а не считать это неудачей текущего исправления long-session heap-pressure.

## Phase 1: Стресс-тестирование реальными длительными задачами с несколькими Agent

### Формат теста

Каждый worker модели использует одну и ту же сессию, постоянно выполняя `--resume`. На каждом раунде Qwen Code требуется:

- выполнять read-only code review и поиск кода;
- одновременно запускать не менее 4 вызовов `agent` tool;
- особое внимание уделять участкам памяти, связанным с chat history, compaction, subagent runtime, non-interactive streaming, provider adapters и т.д.;
- сохранять достаточно детальный итоговый ответ, чтобы session history естественно росла.

Runner снимает process-tree RSS каждую секунду, никаких дополнительных heap cap не установлено.

После того как память стала стабильной, эта часть была намеренно остановлена сигналом `SIGTERM`, чтобы переключиться на второй этап amplified stdout stress. Поэтому `SIGTERM` в таблице не является OOM.

### Интерпретация Phase 1

Это наиболее показательные данные в данном раунде, демонстрирующие значительное улучшение исходного long-session OOM.

## Phase 2: Стресс-тестирование Amplified Foreground Stdout

### Формат теста

На втором этапе намеренно усиливается нагрузка на путь shell-output. Для каждой модели и каждого размера payload родительская сессия и параллельные agent выполняют команды shell foreground, выводящие большое количество `x` в stdout:

```bash
node -e "const chunk='x'.repeat(1024*1024); for (let i=0; i<N; i++) process.stdout.write(chunk)"
```

Размеры payload:

- `128 MiB`
- `256 MiB`
- `512 MiB`
- `1024 MiB`
- `1536 MiB`
- `2048 MiB`

Это синтетический стресс, а не нормальная нагрузка code review. Его цель — проверить, может ли очень большой foreground stdout привести Qwen Code на опасные пути памяти.

### Почему RSS превышает Node Heap Limit

Лимит heap Node на этой машине составляет примерно `4144 MiB`, но process-tree RSS достиг примерно `5964,7 MiB`. Это не противоречие:

- RSS включает V8 heap, external buffers, native allocations, загруженные модули, дочерние процессы и память, не учитываемую в old-space heap.
- Пики в amplified-этапе часто приходятся на дочерние Node процессы, а не на root wrapper.
- Поэтому root RSS может оставаться низким, а process-tree RSS — высоким.

## Сравнение с историческими отчётами

В benchmark maintainer'а от 18 мая пики process-tree RSS Qwen при обычных задачах составляли от `942,5 MiB` до `1006,6 MiB`. Эти данные в основном показывали, что footprint Qwen Code во время выполнения задач был заметно выше, чем у Claude Code.

Отчёт о runtime diagnostics от 19 мая дополнительно разделил две линии:

- startup/config RSS: часто увеличивается из-за нормальной конфигурации и MCP дочерних процессов;
- long-session heap OOM: более склонен к давлению со стороны history / compaction / clone.

Данные от 21 мая дополнительно подтверждают это разделение:

- Реальные длительные задачи с несколькими agent и длинными сессиями стабильно держатся ниже `0,9 GiB`, без heap OOM;
- Amplified stdout может поднять process-tree RSS до `5,96 GiB` и вызвать другой тип фатальной ошибки V8 string-length при `2048 MiB`.

Таким образом, по сравнению с традиционным long-session heap OOM из исходного issue, текущая сборка заметно стабильнее. Но это не означает, что у Qwen Code больше нет никаких рисков для памяти при large-output.

## Новое открытие: Фатальный путь Extreme Foreground Stdout

Новая проблема, обнаруженная в этом раунде, — не исходный heap OOM, а фатальная ошибка string/decoding при multi-GiB foreground stdout:

```text
Fatal error in , line 0
Check failed: i::kMaxInt >= len.
...
v8::String::NewFromOneByte
node::StringBytes::Encode
node::encoding_binding::BindingData::DecodeUTF8
```

Условия возникновения:

- Модель: `DeepSeek/deepseek-v4-pro`
- Payload: `2048 MiB`
- Peak tree RSS: `4660,4 MiB`
- Наибольший RSS процесса: `4527,6 MiB`
- Runner записал exit: `SIGTERM`, так как после перехвата фатального вывода оставшиеся дочерние процессы всё ещё работали на высокой загрузке CPU и были вручную завершены.

`pai/glm-5` и `qwen3.6-plus` также завершились с ошибкой на `2048 MiB`: stdout пуст, exit code `1`, но в stderr не было захвачено V8 fatal stack.

### Степень серьёзности

Это реальная проблема устойчивости, но её условие — multi-GiB foreground stdout, не являющееся нормальной задачей code review. Она также не свидетельствует о неудаче текущего исправления long-session heap-pressure.

### Является ли это регрессией, внесённой данным PR?

В этом раунде нет доказательств того, что сбой `2048 MiB` stdout является регрессией, внесённой текущим PR по памяти.

Причина:

- Путь сбоя — foreground shell stdout decode / string construction.
- Путь исходного issue — long-session history, compaction, clone pressure.
- В этом раунде не было baseline до PR для того же payload, поэтому нельзя квалифицировать как регрессию.
- Этот сбой проявляется только при намеренно экстремальном payload `2048 MiB`; payload от `128 MiB` до `1536 MiB` выполняются успешно.

Рекомендуется рассматривать это как dedicated large-output follow-up: более ранний stream / spool / hard-cap для foreground shell output, чтобы избежать конструирования multi-GiB JS string в памяти. Если только цель текущего PR не включает явно «обработку произвольного multi-GiB foreground stdout», не рекомендуется считать это blocker'ом для текущего PR.

## Выводы

1. Последняя локальная сборка `0.15.11` значительно лучше себя проявляет в направлении long-session heap OOM, описанном в issue. На основе текущих результатов стресс-тестирования с heap по умолчанию можно считать, что данный PR в основном решил путь воспроизведения long-session heap OOM.

2. При стандартном Node heap реальные длительные задачи с многопоточным agent review loop не воспроизвели традиционный V8 heap OOM на моделях `pai/glm-5`, `qwen3.6-plus` и `DeepSeek/deepseek-v4-pro`.

3. Синтетический foreground stdout stress всё ещё способен сильно поднять process-tree RSS. Текущая сборка выдержала payload до `1536 MiB` на всех трёх моделях; максимальный успешный tree RSS составил `5964,7 MiB`.

4. По-прежнему существует независимая проблема экстремального large-output: вблизи `2048 MiB` stdout Qwen Code может завершиться с ошибкой до вывода JSON-результата; в случае DeepSeek зафиксирован V8 string-length fatal.

5. Это новое открытие важно, но скорее относится к последующей проблеме large-output robustness и не должно считаться blocker'ом для смягчения long-session heap-pressure.

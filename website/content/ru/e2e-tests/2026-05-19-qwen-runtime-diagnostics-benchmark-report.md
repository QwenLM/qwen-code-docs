# Отчет о бенчмарке runtime-диагностики Qwen Code

Дата: 2026-05-19

## Область

В этом прогоне повторяются предыдущие сценарии бенчмарка Qwen Code с включенной новой опциональной runtime-диагностикой. Тестируется только Qwen Code, не Claude Code.

Исходная матрица моделей:

- `pai/glm-5`
- `qwen3.6-plus`

Дополнительное PR-размерное продолжение:

- `DeepSeek/deepseek-v4-pro` через протокол, совместимый с Anthropic

Сценарии:

- ревью небольшого GitHub PR: PR `#4268`
- навигация по коду: поиск и чтение кода, связанного со сжатием/компактированием
- синтетический локальный diff: около 94.6 KiB
- синтетический локальный diff: около 968.5 KiB
- синтетический локальный diff: около 4.84 MiB

В прогоне использовался локальный встроенный CLI из ветки с диагностикой, с
`QWEN_CODE_PROFILE_RUNTIME=1` и временным домашним каталогом CLI. Глобальные MCP-серверы и хуки не загружались для этого бенчмарка.

Важное замечание: эти абсолютные значения RSS ниже, чем в предыдущих прогонах с `qwen`, найденных через PATH, потому что в этом прогоне использовался `node dist/cli.js` из локальной ветки плюс облегченная временная конфигурация. Относитесь к этому отчету как к внутреннему дистрибутивному прогону диагностики, а не как к прямой замене предыдущего сравнения RSS установленного CLI.

## Проверка согласованности установленного CLI и локальной сборки

Последующая проверка согласованности использовала одинаковый минимальный промпт, модель и неинтерактивный режим как для установленного CLI, так и для локального диагностического пакета. Единственной намеренной переменной было то, загружает ли Qwen Code облегченный временный домашний каталог CLI или обычную конфигурацию пользователя.

| CLI                 | Режим конфигурации | Всего токенов | Tree RSS пик | Root RSS пик | Пик количества процессов | Runtime-диагностика |
| ------------------- | ------------------ | ------------: | -----------: | -----------: | -----------------------: | ------------------- |
| PATH `qwen`         | облегченная конфиг |        33,965 |    542.4 MiB |    249.9 MiB |                        3 | нет                 |
| local `dist/cli.js` | облегченная конфиг |        47,281 |    455.2 MiB |    214.2 MiB |                        4 | да                  |
| PATH `qwen`         | обычная конфиг     |        97,615 |  1,099.9 MiB |    250.1 MiB |                        6 | нет                 |
| local `dist/cli.js` | обычная конфиг     |        97,954 |  1,105.4 MiB |    212.7 MiB |                        8 | да                  |

Эта проверка меняет атрибуцию: более ранний пик в 1 GiB, видимый пользователю, воспроизводим с обычной конфигурацией даже на локальном диагностическом пакете. Следовательно, он в первую очередь не объясняется локальной веткой, включающей PR `#4186`.

При пике с обычной конфигурацией выборка дерева процессов локального пакета была доминирована несколькими дочерними процессами Node/MCP, а не только корневым процессом Qwen:

| Роль  | Форма команды             | RSS при пике дерева |
| ----- | ------------------------- | ------------------: |
| дочерний | Node процесс              |          252.9 MiB |
| дочерний | Chrome DevTools MCP       |          219.7 MiB |
| дочерний | Node процесс              |          219.2 MiB |
| корневой | Qwen Node процесс         |          215.1 MiB |
| дочерний | Настройка Chrome DevTools MCP |          175.2 MiB |

PR `#4186` присутствует в локальной ветке диагностики, но это защитная сетка автоматической компактификации при давлении на V8 heap. Она срабатывает при примерно 70% давления на V8 heap; в этой среде лимит кучи Node составляет около 4.1 GiB, тогда как конечная куча облегченного бенчмарка составила около 99-143 MiB. Основываясь на этих числах, более низкое RSS облегченной конфигурации не вызвано активным сжатием контекста `#4186` во время этих прогонов бенчмарка.

### Проверка атрибуции режима Bare Mode

Второй дополнительный прогон использовал `qwen3.6-plus` с тем же промптом для ревью PR как на установленном CLI, так и на локальном пакете. Это не обычный бизнес-бенчмарк end-to-end. Это контролируемая проверка атрибуции памяти только для запуска/конфигурации.

`--bare` меняет входные данные runtime: он пропускает обнаружение обычных глобальных настроек, запуск MCP, хуки, неявный контекст, навыки и другие интеграции запуска. Поэтому он может завершиться ошибкой или работать по-другому, если провайдер модели настроен только в глобальных настройках. В этом прогоне учетные данные модели были переданы только через окружение дочернего процесса, потому что bare mode намеренно не загружает обычные настройки провайдера. Никакие изменения не записывались обратно в глобальную конфигурацию пользователя.

Этот прогон не дал полезной статистики по токенам/вызовам инструментов: модель завершила работу за один шаг и не вызвала запрошенную команду оболочки. Не используйте эти строки как результаты обычного бенчмарка задач и не сравнивайте их поведение по токенам/вызовам с матрицей выше. Они полезны только для оценки того, какой объем RSS дерева процессов приходится на обычную конфигурацию и настроенные дочерние процессы.

| CLI                 | Режим     | Время | Шаги | Использования инструментов | Tree RSS пик | Root RSS пик | Пик количества процессов |
| ------------------- | --------- | ----: | ---: | -------------------------: | -----------: | -----------: | -----------------------: |
| PATH `qwen`         | normal    | 5.5s  |    1 |                          0 |  1,021.3 MiB |    251.5 MiB |                        5 |
| PATH `qwen`         | `--bare`  | 2.4s  |    1 |                          0 |    525.7 MiB |    246.4 MiB |                        2 |
| local `dist/cli.js` | normal    | 4.9s  |    1 |                          0 |  1,046.2 MiB |    213.3 MiB |                        5 |
| local `dist/cli.js` | `--bare`  | 2.3s  |    1 |                          0 |    454.3 MiB |    216.5 MiB |                        3 |

Результат подтверждает гипотезу о дереве процессов для атрибуции запуска/конфигурации. На этой машине обычная конфигурация добавляет примерно 0.50-0.59 GiB видимого пользователю RSS дерева процессов по сравнению с `--bare`, в то время как корневой RSS остается в диапазоне 0.21-0.25 GiB. При пике с обычной конфигурацией дополнительный RSS снова пришел от дополнительных дочерних процессов Node/MCP, включая процесс Chrome DevTools MCP и его обертку настройки. `--bare` удаляет эти дочерние процессы запуска/конфигурации и возвращает установленные/локальные прогоны в диапазон 0.45-0.53 GiB tree-RSS.

### Временная изоляция настроек MCP/хуков

Поскольку `--bare` меняет слишком много входных данных runtime, чтобы его можно было рассматривать как обычный бенчмарк, был проведен дополнительный прогон с использованием временных каталогов `QWEN_HOME` и сгенерированных файлов настроек на основе обычных настроек. Прогон остался на обычном пути загрузки настроек, но переключал только два измерения конфигурации:

- MCP отключен: `mcpServers` очищен, а списки разрешений/исключений MCP опустошены.
- Хуки отключены: `disableAllHooks` установлен в true.

Глобальные настройки не изменялись. В сценарии использовались `qwen3.6-plus` и минимальный стартовый промпт, поэтому измеряется стоимость дерева процессов при запуске/конфигурации, а не качество рассуждения задачи.

| CLI                 | Временная конфигурация | MCP-серверы | Инструменты | Tree RSS пик | Root RSS пик | Пик количества процессов |
| ------------------- | ---------------------- | ----------: | ----------: | -----------: | -----------: | -----------------------: |
| PATH `qwen`         | полная                 |           4 |          46 |  1,017.4 MiB |    249.8 MiB |                        5 |
| PATH `qwen`         | MCP отключен           |           0 |          17 |    548.7 MiB |    252.4 MiB |                        2 |
| PATH `qwen`         | хуки отключены         |           4 |          46 |  1,003.8 MiB |    246.4 MiB |                        5 |
| PATH `qwen`         | MCP + хуки отключены   |           0 |          17 |    542.5 MiB |    248.0 MiB |                        2 |
| local `dist/cli.js` | полная                 |           4 |          48 |    865.9 MiB |    220.4 MiB |                        6 |
| local `dist/cli.js` | MCP отключен           |           0 |          19 |    442.9 MiB |    209.6 MiB |                        2 |
| local `dist/cli.js` | хуки отключены         |           4 |          48 |    848.3 MiB |    212.6 MiB |                        5 |
| local `dist/cli.js` | MCP + хуки отключены   |           0 |          19 |    447.2 MiB |    217.8 MiB |                        2 |

Интерпретация:

1. Отключение MCP является доминирующим изменением. Оно удаляет 4 MCP-сервера, уменьшает количество объявленных инструментов примерно на 29 и снижает RSS дерева процессов примерно на 0.42-0.47 GiB в этом сценарии запуска/конфигурации.
2. Отключение только хуков практически не меняет RSS в этом сценарии. Это ожидаемо, так как промпт не вызвал вызовов инструментов, поэтому хуки `PreToolUse` / `PostToolUse` не выполнялись.
3. Корневой процесс остается в диапазоне около 0.21-0.25 GiB во всех строках. Большая разница снова объясняется составом дерева процессов, а не корневым RSS Qwen.

Две попытки продолжения навигации по коду с `qwen3.6-plus` и `pai/glm-5` также воспроизвели то же разделение памяти MCP vs без MCP, но ни одна модель не выполнила вызовы инструментов в этих прогонах. Поэтому эти строки не используются как доказательство выполнения хуков. Для валидного бенчмарка хуков все еще требуется комбинация задачи/модели, которая надежно выполняет вызовы инструментов.

### Изоляция по каждому MCP

Предыдущая строка показала, что группа MCP является доминирующим фактором памяти при запуске/конфигурации. В дополнительном прогоне был изолирован каждый настроенный MCP-сервер при отключенных хуках для всех строк. Это оставляет тест на обычном пути загрузки настроек, но изменяет только подмножество MCP-серверов.

Настроенные имена MCP-серверов:

- `approval-bridge`
- `env-center`
- `chrome-devtools`
- `code`

Интерпретация:

1. `chrome-devtools` является доминирующим вкладчиком MCP в этой среде. Его одного достаточно, чтобы воспроизвести высокий RSS дерева процессов.
2. Удаление `chrome-devtools` из полного набора MCP возвращает RSS к диапазону без MCP. Удаление других MCP при сохранении `chrome-devtools` этого не делает.
3. Количество объявленных инструментов следует тому же шаблону: базовый уровень — 19 инструментов, а `chrome-devtools` увеличивает количество инструментов до 48. Это означает, что этот MCP, вероятно, также увеличивает размер схемы инструментов запроса и нагрузку на токены, а не только RSS дерева процессов.
4. `approval-bridge`, `env-center` и `code` по отдельности остаются около базового уровня без MCP в этих прогонах запуска/конфигурации. Они выдавали предупреждения при запуске в этой среде, поэтому этот результат следует интерпретировать как «не наблюдается постоянного владельца RSS при запуске», а не как доказательство отсутствия затрат во всех рабочих процессах.

## Наблюдения

1. RSS дерева процессов почти равен корневому RSS в этом прогоне локального пакета. Разрыв между корнем и деревом обычно ниже 10 MiB. Это означает, что в этих прогонах не наблюдалось постоянного владельца памяти среди дочерних процессов. Доминирующим процессом является основной процесс Node.
2. Пик локального пакета составляет около 0.36-0.41 GiB, а не ранее наблюдаемых 0.83-1.04 GiB, потому что матрица использовала облегченную временную конфигурацию. Последующая проверка с обычной конфигурацией воспроизвела около 1.1 GiB tree RSS как у PATH `qwen`, так и у local `dist/cli.js`, причем дополнительная память приходила от дочерних процессов MCP/Node в дереве процессов.
3. V8 куча намного меньше RSS. Конечная куча составляет около 99-143 MiB, в то время как конечный RSS — около 216-411 MiB. Оставшийся объем, вероятно, составляют загруженные модули, нативные выделения, внешние буферы или накладные расходы runtime вне живой JS-кучи.
4. Статические накладные расходы запроса велики и повторяются. Системный промпт составляет около 39-51 KiB на запрос, а схема инструментов — около 37-40 KiB на запрос. Это объясняет, почему даже небольшие задачи могут приводить к высокому накопленному количеству токенов, если модель делает несколько шагов.
5. Вывод больших diff-ов обрезается до того, как попадет в запрос к модели. Сценарии с diff 968 KiB и 4.84 MiB дали около 772-799 KiB захваченного результата инструмента, но самый большой ответ функции, видимый модели в запросе, оставался в районе 25-41 KiB, а максимальный размер тела — около 105-122 KiB. Это указывает на то, что усечение/обработка сохраненного вывода работают на пути к модели.
6. Память все равно увеличивается в сценариях с большим выводом, даже если размер тела остается ограниченным. Например, прогон GLM с 4.84 MiB достиг 414.2 MiB tree RSS и 410.7 MiB конечного RSS, а прогон qwen3.6-plus с 4.84 MiB завершился с 142.8 MiB кучи. Это говорит о том, что большой вывод инструмента все еще может влиять на локальный захват, нормализацию или сохраненное состояние runtime, даже если конечный payload запроса ограничен.
7. Выбор модели больше влиял на количество шагов и общее количество токенов, чем на RSS в этом прогоне. `qwen3.6-plus` в среднем потреблял больше токенов и шагов, чем `pai/glm-5`, но его средний пик tree RSS был немного ниже. Это подтверждает более ранний вывод, что выбор модели не является основным объяснением потребления памяти процессом.
## Уточнённая рабочая гипотеза

Новая диагностика делает предыдущую гипотезу более точной:

- Пик в 1 GiB, видимый пользователю установленного CLI, теперь воспроизводится с обычной конфигурацией на локальном диагностическом наборе. Урезанный запуск следует использовать для внутренней атрибуции времени выполнения Qwen; запуск с обычной конфигурацией — для атрибуции видимого пользователю дерева процессов.
- Самое большое наблюдаемое различие между урезанной и обычной конфигурацией — форма дерева процессов: обычная конфигурация запускает дополнительные дочерние процессы MCP/Node. Эти дочерние процессы объясняют большую часть абсолютного скачка с примерно 0,35–0,55 GiB до примерно 1,1 GiB в минимальном прогоне проверки.
- Последующая проверка с `--bare` подтверждает то же направление на `qwen3.6-plus`: обычная конфигурация стоит примерно на 0,50–0,59 GiB больше RSS дерева процессов, чем "bare" режим для той же формы промпта, в то время как корневой RSS изменяется незначительно.
- Изоляция временных настроек — лучший тест атрибуции, чем `--bare`: отключение только MCP снижает RSS дерева процессов примерно на 0,42–0,47 GiB, сохраняя при этом обычный путь загрузки настроек. Отключение только хуков не показывает значимого изменения RSS в сценариях без вызова инструментов.
- Поинструментальная изоляция MCP указывает на `chrome-devtools` как на доминирующего участника MCP: его достаточно, чтобы воспроизвести высокий диапазон RSS, а его удаление возвращает прогон близко к базовому уровню без MCP.
- Внутри локального времени выполнения Qwen наиболее подозрительные области больше не являются "сырыми байтами diff'а, отправленными модели". Тело запроса, отправляемое модели, ограничено.
- Более сильные подозрения — это статические затраты контекста на запрос, повторные раунды запросов, размер схемы инструментов и локальное сохранение/захват больших выходных данных инструментов до или вне усечения, применяемого к модели.
- Поскольку RSS остаётся значительно выше кучи V8, следующий уровень профилирования должен включать учёт модулей/запуска, внешнюю память и снимки кучи вокруг выполнения инструментов и выдачи финального ответа.

В прогоне с обычной конфигурацией видимый пользователем пик дерева процессов в 1 GiB — это в основном состав дерева процессов: корень Qwen плюс дочерние процессы MCP/Node. Следующее целевое измерение должно разделить диагностику корневого процесса Qwen и диагностику настроенных серверов MCP, затем добавить контрольные точки запуска/модулей/внешней памяти внутри корневого процесса Qwen.

## Снимок прогресса

Текущие подтверждённые сигналы:

1. Видимый пользователем пик запуска/конфигурации в 1 GiB воспроизводим как с установленным CLI, так и с локальным диагностическим набором при загрузке обычной конфигурации. Он не объясняется в первую очередь диагностической веткой или PR `#4186`.
2. В этом окружении пик в 1 GiB — это в основном состав дерева процессов: корневой процесс Qwen плюс дочерний процесс перезапуска плюс дочерние процессы MCP.
3. `chrome-devtools` является доминирующим настроенным участником MCP в текущей конфигурации. Его достаточно, чтобы воспроизвести высокий диапазон RSS дерева процессов, даже если промпт явно не использует этот MCP.
4. Форма обычного перезапуска без MCP всё ещё находится около 0,45 GiB RSS дерева процессов. Один процесс рантайма Qwen без родителя перезапуска ближе к 0,22-0,24 GiB в проверке атрибуции запуска. Это означает, что базовый уровень 0,45 GiB — это не число RSS однопроцессного корня.
5. В урезанных неинтерактивных прогонах задач выбор модели меняет ходы, количество токенов, задержку и размеры запросов более явно, чем RSS. RSS оставался в относительно узком диапазоне для `pai/glm-5`, `qwen3.6-plus` и `DeepSeek/deepseek-v4-pro`.
6. Текущая диагностика коротких задач показывает, что ответы инструментов/функций, отправляемые модели, ограничены, но локальный захват результатов инструментов и состояние рантайма могут увеличивать кучу/RSS в случаях с большим выводом. Это оставляет удержание большого вывода в пути расследования.

Текущие пробелы:

1. Матрица бенчмарков коротких задач всё ещё краткосрочна. Более поздний интерактивный прогон длительного ревью воспроизвёл сбой через 41,9 минуты, но это пока один образец и требует повторных прогонов плюс атрибуции кучи/объектов.
2. Текущих счётчиков достаточно для атрибуции RSS дерева процессов и размера запроса, но недостаточно для именования удерживаемого графа JS-объектов во время длительных сессий.
3. Стартовое/конфигурационное RSS и OOM длительных сессий должны оставаться отдельными треками. MCP и перезапуск объясняют большой диапазон RSS в простое/при запуске; они сами по себе не объясняют OOM кучи V8 после длительных задач.
4. Интерактивная память TUI всё ещё требует отдельного прогона от неинтерактивного режима, поскольку история UI и статический вывод Ink не тестируются одинаково.

Первоначальная интерпретация:

- Неиспользуемый настроенный MCP может потреблять память, потому что нормальный запуск подключается к настроенным серверам MCP и рекламирует их инструменты до того, как задача будет нуждаться в них. В измеренной конфигурации `chrome-devtools` запускает дополнительные процессы MCP Node/npm и также увеличивает количество схем инструментов с 19 до 48. Это объясняет большой диапазон стартового/конфигурационного RSS и может также увеличивать накладные расходы на повторные запросы.
- Отчёты об OOM в длительных сессиях — это другой уровень. Логи GC, где Mark-Compact освобождает очень мало памяти, предполагают, что куча заполнена достижимым состоянием. Самые сильные кандидаты — это удерживаемые объекты истории/инструментов/UI, клоны полной истории, промежуточные данные сжатия и аккумуляторы потоковой передачи/логирования.
- PR `#4186` является полезным смягчением, поскольку он может выполнять компактизацию на основе давления на кучу до срабатывания порогов токенов и удаляет один ненужный клон полной истории. Его не следует рассматривать как доказательство того, что удержание большого вывода инструментов, удержание истории UI или пиковая память сжатия уже решены.

## План валидации длительных задач

Следующий бенчмарк должен держать два трека отдельно:

1. Атрибуция запуска/конфигурации: обычная конфигурация vs отключённый MCP vs только `chrome-devtools` vs атрибуция без перезапуска. Это объясняет, что видят пользователи до начала осмысленной работы.
2. Рост во время выполнения длительной задачи: повторные вызовы инструментов, большие выводы, сжатие, возобновление и интерактивная история UI. Это объясняет OOM после реальной работы.

Рекомендуемые кейсы для длительных задач:

| Кейс                          | Форма                                                                                                | Почему это важно                                                                                    |
| ----------------------------- | ---------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------- |
| Длинный цикл ревью PR           | Повторять промпты ревью среднего/крупного PR в течение 30, 60 и 120 минут с фиксированной моделью и конфигурацией | Ближе всего к сообщаемым агентским рабочим процессам; захватывает ходы, вызовы инструментов, рост токенов и тренд RSS/кучи |
| Удержание большого вывода инструментов   | Повторно генерировать ограниченные выводы команд 1 MiB / 5 MiB / 20 MiB, затем задавать уточняющие вопросы      | Проверяет, сохраняется ли сырой вывод локально после усечения, отправляемого модели                        |
| Давление сжатия          | Использовать более низкий контролируемый лимит старой кучи и промпты с большим контекстом для запуска компактизации по давлению на кучу | Проверяет, срабатывает ли PR `#4186` до OOM и создаёт ли само сжатие новый пик         |
| Интерактивная история TUI       | Запустить тот же длинный цикл в режиме TUI (tmux) и сравнить с неинтерактивным режимом                        | Изолирует историю UI, статический вывод Ink, отображаемые diff'ы и удержание вывода терминала     |

Метрики, которые должны записываться за каждый ход или за каждый интервал дискретизации:

- Текущий/пиковый корневой RSS и RSS дерева процессов.
- Количество дочерних процессов и формы команд верхнего уровня.
- V8 `heapUsed`, `heapTotal`, `heap_size_limit`, `external` и `arrayBuffers`.
- Количество ходов, количество запросов, количество вызовов инструментов и количество раундов вызовов инструментов.
- Входные/выходные/кэш/общие токены по запросу и по всей задаче.
- Байты тела запроса, байты системного промпта, байты схемы инструментов и байты ответов функций.
- Количество результатов инструментов, общее количество захваченных байтов результатов инструментов, максимальное количество байтов результатов инструментов и (если доступно) количество удерживаемых байтов результатов инструментов.
- Количество сообщений истории разговора и приблизительный размер истории в байтах.
- Только для интерактивного режима: количество элементов истории UI и приблизительный удерживаемый размер отображения.
- Попытки сжатия, причина запуска сжатия, токены до/после, давление на кучу до/после и статус ошибки сжатия.
- Артефакты снимка кучи или ограниченной временной шкалы памяти, когда давление на кучу превышает настроенный порог.

Критерии валидации:

1. Повторить хотя бы ключевые кейсы длительных задач дважды. Стартовое RSS имеет заметную вариативность, поэтому выводы из одного прогона следует избегать.
2. Сообщать корневое RSS и RSS дерева процессов отдельно. Видимое пользователем давление на память может исходить от дочерних процессов, в то время как V8 OOM исходит из кучи корневого процесса Qwen.
3. Считать плоскую линию RSS важным доказательством. Если токены и вызовы инструментов растут, а куча/RSS остаётся плоским, проблема, вероятно, в другом месте.
4. Когда RSS или куча растут, коррелировать рост с конкретным сигналом: байты результатов инструментов, байты истории, количество элементов истории UI, событие сжатия, размер аккумулятора потоковой передачи или запуск процесса MCP.
5. Если делается снимок кучи, сначала записать структурированный JSON с диагностикой, затем снимок. Снимки кучи могут быть большими и могут содержать чувствительные строки, поэтому они должны оставаться опциональными и локальными.

Процесс CLI оставался живым в одной сессии, в то время как контроллер отправлял по одному реальному ходу ревью PR за раз. Следующий ход отправлялся только после того, как ассистент выдавал маркер завершения этого хода. Это предотвращает трактовку короткого одноразового промпта как воспроизведения длительной задачи.

5. Наиболее вероятные участки кода для проверки: хранение транскриптов сабагентов, слияние результатов агентов, клонирование полной истории, запись контрольных точек/сессии и финальная сборка истории/итогов.

## Детерминированная репродукция давления клонирования при большой задаче

Детерминированный стресс-стенд был добавлен как `scripts/memory-pressure-repro.mjs`. Он не вызывает модель. Вместо этого он строит объектный граф, похожий на длительную сессию Qwen: корневые раунды ревью, транскрипты сабагентов, большие результаты инструментов, JSON контрольных точек и сохранённые копии `structuredClone()`. Это даёт повторяемую репродукцию для пика клонирования и контрольных точек, подозреваемого из стека OOM, предоставленного пользователем.

Стенд включает лёгкий скриптовый тест:

```bash
npx vitest run --config ./scripts/tests/vitest.config.ts \
  scripts/tests/memory-pressure-repro.test.js
```

Результат: пройдено, 1 тест.

Контролируемые запуски выполнялись с `node --max-old-space-size=256`, если не указано иное.

| Случай                                           | Форма истории                                                           | Давление клонированием/контрольными точками                     | Результат                           |   Макс. RSS |
| ------------------------------------------------ | ----------------------------------------------------------------------- | --------------------------------------------------------------- | ----------------------------------- | ----------: |
| Маленький (проверка)                             | 2 раунда, 2 KiB результат инструмента, 1 сабагент                      | 1 клон + 1 контрольная точка                                    | пройдено; 2.6 MiB JSON истории      |    89.7 MiB |
| Большой, только сборка                          | 12 раундов, 256 KiB результат инструмента, 2 сабагента x 12 раундов     | нет сохранённых клонов/контрольных точек                        | пройдено; 76.2 MiB JSON истории     |   491.5 MiB |
| Большой + 1 клон                                | то же, что выше                                                        | 1 сохранённый `structuredClone()`                               | пройдено                            |   569.6 MiB |
| Большой + 2 клона                               | то же, что выше                                                        | 2 сохранённые копии `structuredClone()`                          | OOM, выход 134                      |   496.5 MiB |
| Большой + 1 контрольная точка                   | то же, что выше                                                        | одна контрольная точка с исходной и клонированной историей JSON | пройдено; 152.5 MiB JSON к.т.       |   926.9 MiB |

| Большой + 2 клона, без сохранения транскриптов сабагентов | тот же сгенерированный вывод сабагентов, но родительская история хранит только сводки | пройдено; JSON родительской истории уменьшается до 3.8 MiB      |   136.8 MiB |

Запуск с 2 клонами, который привёл к сбою, выдал:

```text
FATAL ERROR: Reached heap limit Allocation failed - JavaScript heap out of memory
```

Нативный стек включал:

- `v8::internal::ValueDeserializer::ReadObjectInternal`
- `v8::internal::ValueDeserializer::ReadDenseJSArray`
- `node::worker::Message::Deserialize`
- `node::worker::StructuredClone`

Это совпадает с тем же семейством стеков, что и в OOM-логе пользователя.
Контролируемая репродукция также показывает, почему отчёты о 4 GiB / 8 GiB правдоподобны: сбой вызван не одним большим объектом, а большим объёмом удерживаемой истории/результатов инструментов/состояния сабагентов плюс одной или несколькими копиями полной истории (клон или контрольная точка). Увеличение `--max-old-space-size` может отложить крах, сохраняя ту же схему амплификации.

Важные выводы из этого детерминированного запуска:

1. Сборка JSON родительской истории размером 76.2 MiB может быть успешной при уменьшенной куче. OOM появляется, когда добавляются дополнительные копии полной истории (клон/контрольная точка).
2. Одна копия контрольной точки может поднять RSS почти до 1 GiB ещё до OOM.
3. Удаление сохранённых транскриптов сабагентов из горячей родительской истории превращает ту же сгенерированную нагрузку из OOM в небольшой запуск с RSS 136.8 MiB. Это самый чёткий сигнал для смягчения на данный момент.
4. Этот репродуцент синтетический и намеренно недоброжелательный, но он воспроизводит ту же форму объектного графа, что и длительное интерактивное ревью: родительская сессия, сабагенты, большие результаты инструментов, слияние транскриптов и давление клонирования полной истории.

1. Размер PR чётко масштабировал количество раундов, токенов, размер тела проводного запроса Anthropic и размер результатов инструментов, но не привёл к пропорциональному росту RSS. Пики RSS дерева для малого/среднего/большого остались в узком диапазоне `340.7–360.0 MiB`.
2. Большой PR был затратен в основном по количеству раундов модели и объёму токенов: 8 запросов и 386,891 токенов. Его максимальное тело Anthropic составило 289.5 KiB, что значительно больше, чем у запусков через OpenAI-совместимый протокол, но RSS остался вблизи той же локальной полосы.
3. Статическая стоимость запроса Anthropic также заметна: системный prompt ~50.8 KiB, схема инструментов ~47.6 KiB на запрос. Повторные раунды являются поэтому основным усилителем токенов.
4. Большой PR дал 235.0 KiB захваченных результатов инструментов и 182.4 KiB максимального ответа функции в запросе. Это выше, чем в ранних случаях с малым PR / навигацией по коду, и показывает, что большие PR всё ещё создают нагрузку на локальную обработку результатов инструментов и сборку запросов, даже если RSS не растёт.
5. Запуск DeepSeek подтверждает вывод о выборе модели: выбор провайдера/модели сильно меняет количество раундов, задержку, объём токенов и форму проводных данных, но пик RSS локального пакета остаётся доминируемым формой выполнения Qwen Code, а не линейно масштабируется с размером PR.

## Воспроизведение длительного ревью через JSONL: давление клонирования истории

Недавняя запись чата длительного ревью PR была проанализирована как post-mortem форма для отчётного класса OOM. Сырой JSONL не включён сюда, так как содержит текст prompt и вывода инструментов. Агрегированная форма:

| Сигнал                               | Значение                        |
| ------------------------------------ | ------------------------------- |
| Длительность                         | 87.0 мин                        |
| Версия Qwen Code                     | 0.15.10                         |
| Модель                               | qwen-latest-series beta model   |
| Ответов API                          | 380                             |
| Телеметрия вызовов инструментов      | 507 событий                     |
| Телеметрия вызовов инструментов MCP  | 4 события                       |
| Ответов API сабагентов               | 313                             |
| Ответов API корневого                | 67                              |
| Рост prompt корневого                | 38,622 -> 168,555 токенов       |
| Максимум токенов prompt               | 168,555                         |
| Всего токенов ответов                | 31.28M                          |

Эта форма не подтверждает MCP как основную причину OOM в этом случае. Только 4 из 507 событий телеметрии вызовов инструментов были MCP, и все четыре имели `content_length=0`. Доминирующая форма — амплификация длительной сессии/сабагентов: 15 вызовов `agent` породили 313 ответов API сабагентов и 403 события вызова инструментов сабагентов.

Затем воспроизведение перестроило форму сообщений `Content[]` чата из JSONL и провело контролируемые тесты на давление клонирования/сериализации. Базовая полезная нагрузка удерживаемых сообщений мала, поэтому сама по себе она недостаточна для OOM:

| Масштаб воспроизведения | Сохранённых клонов | JSON истории | JSON контрольной точки | Heap в конце | RSS в конце |
| ----------------------- | -----------------: | -----------: | ---------------------: | -----------: | ----------: |
| 1x                      |                  8 |      0.54 MB |               1.08 MB  |    18.0 MB   |    88.8 MB  |
| 30x                     |                  8 |     14.46 MB |              28.92 MB  |   260.0 MB   |   577.8 MB  |
| 60x                     |                  8 |     28.86 MB |              57.71 MB  |   510.3 MB   |   960.8 MB  |

Масштабированное воспроизведение не является утверждением о пользовательских данных; это контролируемая амплификация наблюдаемой формы JSONL для проверки, может ли сериализация полной истории (клон и контрольная точка) создать тот же тип сбоя, что и в отчётах.

Воспроизведение с низкой кучей (`--max-old-space-size=256`) подтверждает механизм:

| Случай                              | JSON истории | Результат                                                |
| ----------------------------------- | -----------: | -------------------------------------------------------- |
| Только сборка истории               |      38.4 MB | Успешно; heap 131.6 MB, RSS 378.2 MB                     |
| Сборка + один клон                  |      38.4 MB | Успешно; heap 183.3 MB, RSS 463.4 MB                     |
| Сборка + повторные клоны            |      38.4 MB | OOM после нескольких сохранённых копий `structuredClone()` |
| Контрольная точка + двойная история |      38.4 MB | OOM при удержании истории плюс клонированной истории клиента |

Стек OOM с повторными клонами содержит `ValueDeserializer::ReadObjectInternal`, `ValueDeserializer::ReadDenseJSArray`, `node::worker::Message::Deserialize` и `node::worker::StructuredClone`, что совпадает с тем же семейством стеков, что и в OOM-логе пользователя. Это доказывает, что `structuredClone()` для полной истории может быть непосредственным триггером OOM без участия MCP-сервера.

Текущая рабочая гипотеза для этого класса JSONL:

1. MCP может объяснять стартовый RSS в нормальной конфигурации в отдельных бенчмарках, но вряд ли является триггером для этой формы OOM длительного ревью.
2. Рост длительной задачи происходит из-за удерживаемой истории чата, больших результатов инструментов, историй сабагентов, наблюдаемых сообщений агентов и состояния UI/результатов инструментов.
3. Непосредственным триггером OOM может быть клонирование полной истории или двойная сериализация в стиле контрольной точки, когда куча уже велика.
4. Сжатие может смягчить хранение истории, но само сжатие может создать временный пик, если сначала клонируется или сериализуется большая история.

### Локальная проверка смягчения: случай ревью PR с отключённым MCP

Два целенаправленных смягчения были применены локально и проверены перед повторным запуском случая ревью PR с отключённым MCP:

1. `checkNextSpeaker()` теперь читает только последнее курированное сообщение с помощью `getHistoryTail(1, true)` и отправляет только это сообщение в побочный запрос выбора следующего говорящего. Побочный prompt спрашивает только о непосредственно предыдущем ответе модели, поэтому отправка полной истории была ненужным клонированием и давлением токенов.
2. `AgentToolInvocation` больше не сохраняет полные массивы `responseParts` внутри виджета `task_execution.toolCalls` в реальном времени. Реальные части ответа по-прежнему проходят по путям транскриптов/истории, но родительский UI теперь хранит только ограниченную текстовую сводку для вложенного стриминга результатов инструментов, вместо хранения ещё одной полной копии больших выводов сабагентов во время длительных запусков.
3. `GeminiChat.sendMessageStream()` теперь строит содержимое запроса модели через внутреннее курированное представление истории вместо вызова публичного `getHistory(true)`. Публичный `getHistory()` всё ещё возвращает защитную копию через `structuredClone()` для внешних вызывающих, но горячий путь запроса больше не делает глубокое клонирование всей удерживаемой истории чата перед каждым вызовом модели.

Для этих смягчений добавлены TDD-проверки:

| `AgentTool > should not retain responseParts in live tool call display after TOOL_RESULT`                      | Предотвращает хранение больших ответов инструментов в виджете сабагента в реальном времени   |

## Примечания

- Первый запрос навигации по коду допускал свободное исследование и достиг `maxSessionTurns`; успешные строки выше используют ограниченный список команд.
- Первая попытка синтетического diff использовала относительный путь к бандлу внутри временных репозиториев; эти попытки сразу провалились и исключены из таблиц. Успешные строки используют абсолютный путь к локальному бандлу.
- Сырые потоки JSONL не сохраняются, так как содержат промпты, команды инструментов и вывод инструментов. Отчёт содержит только агрегированные диагностические данные.

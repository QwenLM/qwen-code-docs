# Qwen Code — Отчёт о тестировании памяти в рантайме

Дата: 2026-05-18

## Краткое описание

Этот отчёт фиксирует локальные измерения памяти для поведения Qwen Code в рантайме. В нём сравнивается Qwen Code на разных моделях, а также сравнивается Qwen Code с Claude Code на тех же типах задач, где были доступны эквивалентные конечные точки моделей.

Ключевой результат стабильно воспроизводится во всей последней матрице (один прогон на ячейку, без статистического повторения):

- Пиковое RSS дерева процессов Qwen Code: около `852–1062 MiB` (`0.83–1.04 GiB`).
- Пиковое RSS дерева процессов Claude Code: около `279–366 MiB` (`0.27–0.36 GiB`).
- Qwen Code показал примерно в `2.3x–3.6x` больше в протестированных неинтерактивных CLI-задачах.

Примечание: RSS дерева процессов включает дочерние процессы MCP (~350 MiB накладных расходов на стороне Qwen). Это увеличивает абсолютные числа, но относительное сравнение остаётся информативным, поскольку оба CLI измерялись одинаковым способом.

Различие воспроизвелось на задачах небольшого PR-ревью, навигации по коду и синтетических diff-нагрузках. Следовательно, маловероятно, что оно объясняется только одним большим PR или одним провайдером модели.

Этот отчёт предназначен для того, чтобы сделать текущее исследование производительности прозрачным: что было измерено, какой вывод уже поддерживается, что остаётся неизвестным и какие диагностики следует добавить в следующий раз.

Для прогонов матрицы использовались временные конфигурационные директории CLI, чтобы бенчмарки не зависели от глобального состояния CLI.

## Артефакты бенчмарка

До составления этого сводного отчёта было подготовлено пять локальных отчётов:

1. Замер памяти Qwen Code при PR-ревью.
2. Сравнение моделей Qwen Code.
3. Строгое сравнение Qwen Code и Claude Code с `pai/glm-5`.
4. Qwen Code и Claude Code — два CLI, две модели.
5. Qwen Code и Claude Code — матрица из пяти случаев.

Этот сводный отчёт охватывает выводы и ключевые метрики всех пяти отчётов. Он не включает каждую сырую строку выборки, стенограмму терминала или временный артефакт раннера. Эти сырые артефакты остались в локальных каталогах `tmp/`, поскольку они являются результатами экспериментов, а не стабильными фикстурами репозитория.

Последняя матрица является самым весомым доказательством, поскольку покрывает несколько типов задач, а не только одну нагрузку PR-ревью.

## Предварительный вывод

Текущих данных достаточно, чтобы утверждать, что Qwen Code имеет большее потребление памяти в рантайме, чем Claude Code, в данных локальных неинтерактивных CLI-задачах. Однако их недостаточно, чтобы назвать единственную корневую причину.

Основное объяснение — различие в рантайме/путях выполнения Qwen Code, а не различие в провайдере модели:

- различие воспроизводится как с `pai/glm-5`, так и с `qwen3.6-plus`;
- различие воспроизводится в задачах с небольшим PR и навигацией по коду, а не только в задачах с большими diff'ами;
- Qwen Code многократно отправляет или учитывает больше токенов, чем Claude Code, для аналогичной работы;
- наибольший наблюдаемый компонент Qwen Code — дочерний процесс Node/Qwen-воркера, что указывает на процессорный след во время выполнения задачи, загрузку модулей, сборку контекста, живую историю, сохранение результатов инструментов или подпроцессы/пути сохранённых результатов.

Следовательно, самым полезным следующим измерением будет не очередной прогон только с внешним RSS. Следующее измерение должно разделить RSS на V8-кучу, нативную память, размер сессии/истории, размер сохранённых результатов инструментов и активность подпроцессов/дерева процессов.

## Анализ начальных причин

Бенчмарк ещё не доказывает единственную корневую причину, но он сужает область вероятной проблемы.

| Сигнал                                                                                           | Что это предполагает                                                                              | Что это не доказывает                                                                                          |
| ------------------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------- |
| Qwen остаётся около `1 GiB` в задачах с небольшим PR и навигацией по коду                        | Вероятно, присутствуют высокие затраты рантайма во время выполнения неинтерактивной задачи         | Не выявляет, является ли следствие V8-кучей, нативной памятью, загрузкой модулей или сохранённым состоянием    |
| Размер diff от 100 KiB до 5 MiB не масштабируется линейно с RSS                                  | Сырые байты diff, вероятно, не являются основным драйвером                                         | Большие выходные данные всё ещё могут усиливать память в реальных сценариях PR-ревью                             |
| Qwen использует больше токенов, чем Claude, в каждой ячейке матрицы                              | Qwen, вероятно, конструирует или сохраняет больший промпт/контекст/состояние результатов инструментов | Количество токенов не равно памяти процесса и может быть следствием, а не причиной                             |
| Количество вызовов инструментов схоже, и иногда Claude использует больше шагов/вызовов при меньшем RSS | Более длинная цепочка вызовов инструментов вряд ли сама по себе является основным объяснением      | Размер выходных данных инструментов и их сохранение всё ещё необходимо измерить                                |
| Более ранние прогоны с большим PR показали восстановление сохранённых результатов и усиление подпроцессами | Усечение выходных данных инструментов и пути сохранённых результатов, вероятно, являются усилителями для тяжёлых рабочих нагрузок | Они не объясняют весь след выполнения на маленьких задачах                                                     |

Таким образом, текущее наилучшее объяснение:

1. **В первую очередь — затраты рантайма во время выполнения задачи**: Qwen Code, вероятно, инициализирует или сохраняет больше состояния рантайма во время выполнения неинтерактивной CLI-задачи, чем Claude Code. Это может включать агентный рантайм, реестр инструментов, адаптеры провайдеров, сервисы сессий или структуры UI/истории, которые не являются строго необходимыми для короткой неинтерактивной задачи.
2. **Во вторую очередь — объём контекста/результатов инструментов**: Qwen Code, по-видимому, несёт больше контекста, обращённого к модели или сессии, для аналогичной работы. Разрыв в токенах делает сборку контекста, нормализацию результатов инструментов и сохранение истории важными подозреваемыми.
3. **В третью очередь — усиление при больших выходных данных**: Большое PR-ревью может активировать дополнительные пути сохранённых результатов и подпроцессов. Это, вероятно, не единственная причина, но может усугубить нагрузку на память и токены в реалистичных задачах ревью.

Следующий диагностический прогон должен ответить, где находится `~1 GiB`:

- высокое сразу после запуска: затраты на запуск модуля/рантайма;
- скачок после выполнения инструмента: сохранение выходных данных инструмента или нормализация результатов;
- скачок во время сборки запроса: конструирование контекста или дублированные истории;
- рост после стриминга/сжатия: сохранение ответов или состояние сжатия;
- большая часть RSS вне V8-кучи: нативные буферы, загруженные модули или внешняя память.

## Анализ использования токенов

Разрыв в токенах — одна из самых сильных подсказок, но для того, чтобы считать его коренной причиной, необходимы внутренние метрики запросов.

Что данные поддерживают на сегодня:

- Qwen Code использовал больше суммарных токенов, чем Claude Code, в каждой ячейке матрицы.
- Разрыв проявляется даже при схожих количествах вызовов инструментов.
- Иногда Claude использовал больше шагов или вызовов инструментов, при этом потребляя меньше памяти.

Что это предполагает:

- Разница в токенах вряд ли вызвана только более длинной цепочкой вызовов инструментов.
- Qwen, возможно, несёт больший статический промпт/контекст, более крупные схемы инструментов, большие сериализованные результаты инструментов или больше сохранённого содержимого разговора/сессии.
- Потоки с большими выходными данными могут добавлять ещё один слой через усечение, восстановление сохранённых результатов или подпроцессы.

Чего всё ещё не хватает:

- разбивка входных токенов на запрос;
- размеры системного промпта и схем инструментов в токенах;
- размеры сохранённых сообщений и результатов инструментов перед каждым запросом к модели;
- сохраняются ли большие выходные данные в нескольких местах: история модели, история UI, запись сессии или хранилище сохранённых результатов.

Именно из-за этих недостающих метрик следующий шаг должен включать внутреннюю диагностику, а не просто повторять внешний RSS-бенчмарк.

## Более ранний сигнал от большого PR-ревью

Более ранний строгий бенчмарк PR-ревью использовал PR `#4186` и показал ту же общую картину:

| Модель          | CLI         | Пиковое RSS дерева процессов |
| -------------- | ----------- | -------------------------: |
| `pai/glm-5`    | Qwen Code   |                 1000,7 MiB |
| `pai/glm-5`    | Claude Code |                  349,0 MiB |
| `qwen3.6-plus` | Qwen Code   |                 1095,8 MiB |
| `qwen3.6-plus` | Claude Code |                  341,1 MiB |

Сам по себе этот более ранний прогон не был достаточным, потому что большой PR может вызывать необычные пути для выходных данных инструментов и сохранённых результатов. Последняя матрица из пяти случаев делает вывод более убедительным, поскольку разрыв воспроизводится также в задачах с небольшим PR и навигацией по коду.

## Рабочая гипотеза

Текущие данные поддерживают следующие гипотезы в порядке приоритета:

1. Qwen Code имеет более высокий процессный след во время выполнения неинтерактивной задачи, чем Claude Code. Дочерний Node-воркер Qwen обычно был самым большим процессом в локальной выборке, часто около `0.7–0.8 GiB`.
2. Выбор модели не является основным объяснением. Как `pai/glm-5`, так и `qwen3.6-plus` показали одинаковый общий разрыв Qwen против Claude.
3. Сам по себе большой размер diff не является основным объяснением. Размер синтетического diff не масштабировался линейно от 100 KiB до 5 MiB, вероятно, из-за того, что усечение выходных данных инструментов ограничивает объём данных, достигающих модели.
4. Обработка контекста/результатов инструментов по-прежнему является вероятным фактором. Qwen Code использовал больше токенов, чем Claude Code, в каждой ячейке матрицы, а более ранние прогоны с большим PR показали пути восстановления сохранённых результатов инструментов и усиления подпроцессов.
5. Следующий диагностический слой должен разделить V8-кучу, нативный RSS, стоимость запуска загруженных модулей/рантайма, историю сессии, историю UI, сохранение результатов инструментов и активность подпроцессов. Один внешний RSS не может различить эти причины.

## Оговорки

- Это одиночные прогоны на ячейку матрицы, а не повторяемые статистические выборки.
- RSS — это внешний RSS процесса. Он не может различить V8-кучу, нативные буферы, загрузку модулей, сохранённые выходные данные инструментов, состояние UI или историю сессии.
- Claude Code и Qwen Code используют разные реализации рантайма и адаптеры протокола, даже если метки моделей одинаковы.
- Бенчмарк запускался локально на macOS. Перед тем как делать выводы, специфичные для развёртывания, следует протестировать на Linux-серверах.

## Рекомендуемые последующие измерения

Следующая ветвь локального исследования должна добавить или использовать диагностику для:

- `process.memoryUsage()` до и после запуска, выполнения инструментов, стриминга, сжатия и завершения сессии.
- Статистики V8-кучи и пространств кучи.
- Активных дескрипторов и запросов.
- Количества сообщений сессии и приблизительного объёма сохранённых символов/токенов.
- Количества результатов инструментов, общего размера сохранённых результатов инструментов, размера самого большого результата инструмента и того, сохраняются ли большие выходные данные в истории UI или истории модели.
- Количества подпроцессов и RSS дерева процессов/подпроцессов.
- Событий усечения выходных данных инструментов и восстановления сохранённых результатов.

Эти измерения следует собирать с той же матрицей бенчмарков, чтобы текущее сравнение RSS можно было связать с внутренним состоянием Qwen Code.

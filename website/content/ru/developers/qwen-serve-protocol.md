# Справочник по HTTP-протоколу `qwen serve`

Этап 1 [проекта демона qwen-code](https://github.com/QwenLM/qwen-code/issues/3803). Все маршруты расположены по базовому URL демона (по умолчанию `http://127.0.0.1:4170`).

## Аутентификация

Если демон был запущен с флагом `--token` или переменной `QWEN_SERVER_TOKEN`, **каждый маршрут, кроме `/health` на loopback-привязках**, должен содержать:

```
Authorization: Bearer <token>
```

Без настроенного токена (loopback-режим разработки по умолчанию) заголовок необязателен. Сравнение токенов выполняется за константное время. Ответы 401 единообразны для случаев `отсутствует заголовок` / `неверная схема` / `неверный токен`.

**Исключение для `/health`** (Bctum): на loopback-привязках (`127.0.0.1` / `localhost` / `::1` / `[::1]`) маршрут `/health` регистрируется **до** промежуточного слоя bearer, поэтому проверки живучести внутри пода могут не передавать токен, даже если демон был запущен с `--token`. При не-loopback привязках (`--hostname 0.0.0.0` и т.д.) доступ к `/health` также защищается bearer, как и любой другой маршрут — см. раздел [`GET /health`](#get-health) для обоснования.

**`--require-auth` (#4175 PR 15).** Передайте этот флаг при запуске, чтобы расширить правило «необходим токен» также на loopback. Запуск завершится ошибкой без токена; исключение для `/health` отключается (т.е. `/health` также требует `Authorization: Bearer …`).

Когда флаг включён, глобальный промежуточный слой `bearerAuth` защищает **каждый** маршрут — включая `/capabilities`. Поэтому **неаутентифицированный** клиент не может предварительно запросить `caps.features`, чтобы узнать, что требуется аутентификация: поверхность обнаружения в этом случае — это **тело ответа 401** (единообразное для всех маршрутов, согласно разделу [Аутентификация](#authentication)). Тег возможности `require_auth` является **подтверждением после аутентификации** — как только клиент успешно аутентифицируется и прочитает `/capabilities`, наличие тега подтверждает, что демон был запущен с `--require-auth` (полезно для аудита / интерфейсов соответствия, а также для SDK-клиентов, чтобы отобразить «это развёртывание защищено» на панели настроек). Маршруты, изменяющие состояние, которые используют строгий режим для конкретного маршрута (последующие волны Wave 4), отказывают с `401 { code: "token_required", error: "…" }`, если они достигнуты на loopback по умолчанию без токена — но с включённым `--require-auth` глобальный bearer-слой прерывает запрос до проверки маршрута, поэтому тело `Unauthorized` — это то, что видят неаутентифицированные вызывающие.

**`--allow-origin <pattern>` (T2.4 [#4514](https://github.com/QwenLM/qwen-code/issues/4514)).** Браузерные web-интерфейсы, обращающиеся к демону с другого источника, блокируются по умолчанию — любой запрос, содержащий заголовок `Origin`, возвращает `403 {"error":"Request denied by CORS policy"}`. Клиенты CLI/SDK никогда не отправляют `Origin`, и демон рассматривает его наличие как признак того, что запрос пришёл из браузерного контекста, на который оператор не дал согласия. Передайте `--allow-origin <pattern>` (можно повторять) при запуске, чтобы установить белый список вместо блокировки. Каждый шаблон — это одно из:

- Буквальный `*` — разрешить любой источник. **Рискованно**: запуск отклоняется, если настроен `*`, но не установлен bearer-токен (любой источник: `--token`, `QWEN_SERVER_TOKEN` или `--require-auth`, который требует токен при запуске). При запуске в stderr выводится предупреждение, если `*` находится в списке. **Рекомендация**: используйте вместе с `--require-auth` на loopback-привязках, чтобы `/health` и `/demo` также защищались bearer — они регистрируются до bearer-промежуточного слоя на loopback по умолчанию (чтобы зонды k8s/Compose могли достичь `/health` без токена), а белый список `*` делает их доступными из любого кросс-доменного браузера. На не-loopback привязках bearer уже обязателен при запуске, поэтому поверхность экспозиции `*` — это только `/health` (JSON-статус) и `/demo` (статическая страница, чей JS всё равно вызывает защищённые токеном маршруты) — фактическая поверхность API всё равно защищена.
- Канонический URL-источник — `<scheme>://<host>[:<port>]`. **Без завершающего слеша, без пути, без userinfo, без query.** Запуск отклоняется с ошибкой `InvalidAllowOriginPatternError`, если запись не проходит проверку `new URL(pattern).origin === pattern`; в сообщении об ошибке указывается неправильный шаблон и каноническая форма. Строгость намеренная: молчаливая нормализация (например, удаление завершающего слеша) позволила бы опечаткам проскользнуть и принять неоднозначный ввод.

Совпавшие источники получают стандартные CORS-заголовки ответа для каждого запроса:

```
Access-Control-Allow-Origin: <эхо исходного источника>
Vary: Origin
Access-Control-Allow-Methods: GET, POST, PATCH, DELETE, OPTIONS
Access-Control-Allow-Headers: Authorization, Content-Type, X-Qwen-Client-Id, Last-Event-ID
Access-Control-Max-Age: 86400
Access-Control-Expose-Headers: Retry-After
```

`Access-Control-Allow-Origin` дословно повторяет исходный источник запроса (в нижнем/верхнем регистре, как отправил браузер), а не буквальный `*`, даже при шаблоне `*` — браузерные кеши связывают ответы с этим заголовком в паре с `Vary: Origin`, а эхо оставляет возможность добавить `Access-Control-Allow-Credentials` в будущем релизе без изменения схемы. `Access-Control-Expose-Headers: Retry-After` позволяет браузерным web-интерфейсам учитывать подсказки демона о повторных попытках из ответов `429` / `503`. `Access-Control-Allow-Credentials` **НЕ** отправляется сегодня: демон аутентифицируется через bearer в `Authorization`, который работает кросс-доменно без `credentials: 'include'`.

OPTIONS-предзапросы (OPTIONS с `Access-Control-Request-Method` или `Access-Control-Request-Headers`) прерываются с `204 No Content` плюс указанные выше заголовки. Это стандартный CORS-шаблон и безопасен — предзапрос только подтверждает, какие методы/заголовки примет демон; фактический последующий запрос всё равно проходит полную цепочку (белый список хостов → bearer-аутентификация → маршруты), поэтому защита от DNS-ребдиндинга и проверка bearer всё равно срабатывают до чтения или изменения состояния. Обычные OPTIONS-запросы от совпавших источников продолжают обрабатываться, но с прикреплёнными CORS-заголовками.

Источники, не совпадающие с белым списком, всё равно получают `403 {"error":"Request denied by CORS policy"}` — та же обёртка, что и при блокировке по умолчанию, чтобы клиенты, уже разбирающие ответ блокировки, не должны были обрабатывать демоны с белым списком отдельно. Путь отклонения **не** отправляет никаких `Access-Control-*` заголовков (браузер бы их проигнорировал, а отправка косвенно рекламировала бы размер белого списка через наличие заголовков).

Настроенный список шаблонов намеренно **не** отражается в `/capabilities` — браузерный web-интерфейс уже знает свой собственный источник (он ведь обратился к демону), а отображение списка позволило бы неаутентифицированному читателю `/capabilities` перечислить все доверенные источники (полезная разведка для неправильно настроенного развёртывания). SDK-клиенты проверяют тег `caps.features.allow_origin` для понимания «этот демон обрабатывает кросс-доменные браузерные запросы», не зная конкретных источников.

Запросы с loopback-собственного источника (например, страница `/demo`, вызывающая демон на том же `127.0.0.1:port`) обрабатываются **отдельным** прокси-слоем, который удаляет заголовок Origin, и запускается **до** CORS-промежуточного слоя. Он удаляет заголовок `Origin` для `127.0.0.1:port` / `localhost:port` / `[::1]:port` / `host.docker.internal:port`. Таким образом, они проходят независимо от настройки `--allow-origin` — операторам не нужно указывать собственный порт демона, чтобы демонстрационная страница работала.

## Общая структура ошибок

Ответы 5xx содержат `code` и `data` исходной ошибки, если они есть (стиль JSON-RPC — SDK ACP передаёт `{code, message, data}` от агента):

```json
{
  "error": "Внутренняя ошибка",
  "code": -32000,
  "data": { "reason": "превышена квота модели" }
}
```

Некорректный JSON в теле запроса возвращает:

```json
{ "error": "Некорректный JSON в теле запроса" }
```

со статусом `400`.

`SessionNotFoundError` для неизвестного идентификатора сессии возвращает:

```json
{ "error": "Нет сессии с id \"<sid>\"", "sessionId": "<sid>" }
```

со статусом `404`.

`WorkspaceMismatchError` для `POST /session`, чей `cwd` не канонизируется к привязанной рабочей области демона (#3803 §02 — 1 демон = 1 рабочая область), возвращает `400` с:

```json
{
  "error": "Несоответствие рабочей области: демон привязан к \"…\", но запрос запросил \"…\". …",
  "code": "workspace_mismatch",
  "boundWorkspace": "/путь/к/привязке/демона",
  "requestedWorkspace": "/путь/в/запросе"
}
```

Используйте это для предварительного обнаружения несоответствия: прочитайте `workspaceCwd` из `/capabilities` и опустите `cwd` в `POST /session` (он вернётся к привязанной рабочей области) или направьте запрос к демону, привязанному к `requestedWorkspace`.

`POST /session`, превышающий лимит демона `--max-sessions`, возвращает `503` с заголовком `Retry-After: 5` и:

```json
{
  "error": "Достигнут лимит сессий (20)",
  "code": "session_limit_exceeded",
  "limit": 20
}
```

Повторные подключения к существующим сессиям НЕ учитываются в лимите, поэтому повторные подключения к бездействующему демону продолжают работать, даже если лимит исчерпан.

`RestoreInProgressError` — выдается только маршрутами `POST /session/:id/load` и `POST /session/:id/resume` — возвращает `409` с заголовком `Retry-After: 5` (как и для `session_limit_exceeded`) и:

```json
{
  "error": "Сессия \"<sid>\" уже восстанавливается через session/<resume|load>; повторите session/<load|resume> после завершения",
  "code": "restore_in_progress",
  "sessionId": "<sid>",
  "activeAction": "load",
  "requestedAction": "resume"
}
```

Возникает, когда для идентификатора, для которого уже выполняется `session/resume`, выдается `session/load` (или наоборот). Подождите не менее `Retry-After` секунд и повторите — фактическое восстановление завершается в течение `initTimeoutMs` (по умолчанию 10 с). Конкурирующие запросы одного и того же действия (`load` vs `load`, `resume` vs `resume`) объединяются, а не порождают ошибку.

## Возможности

Демон сообщает о поддерживаемых тегах функций из реестра возможностей сервера. Клиенты **должны** проверять UI на основе `features`, а не `mode` (согласно дизайну §10).

 'session_list', 'session_prompt', 'session_cancel', 'session_events',
 'slow_client_warning', 'typed_event_schema',
 'session_set_model', 'client_identity', 'client_heartbeat',
 'session_permission_vote', 'permission_vote', 'workspace_mcp', 'workspace_skills',
 'workspace_providers', 'auth_provider_install', 'workspace_memory',
 'workspace_agents', 'workspace_agent_generate', 'workspace_env',
 'workspace_preflight', 'session_context', 'session_context_usage',
 'session_supported_commands', 'session_tasks', 'session_stats',
 'session_lsp', 'session_status',
 'session_close', 'session_metadata', 'mcp_guardrails',
 'workspace_mcp_manage', 'mcp_guardrail_events',
 'mcp_server_runtime_mutation',
 'workspace_file_read', 'workspace_file_bytes', 'workspace_file_write',

 'workspace_settings', 'workspace_init', 'workspace_mcp_restart',

 'workspace_hooks', 'session_hooks', 'workspace_extensions',
 'session_branch', 'rate_limit', 'workspace_reload']
```

> Условные теги появляются только тогда, когда включена соответствующая опция развёртывания (см. таблицу ниже). Тег F3 `permission_mediation` всегда включён и несёт `modes: ['first-responder', 'designated', 'consensus', 'local-only']`, чтобы SDK-клиенты могли исследовать поддерживаемый сборкой набор; активная во время выполнения стратегия находится в `body.policy.permission`.

`session_scope_override` — это дескриптор согласования для поля `sessionScope` для каждого запроса в `POST /session` (см. ниже). Старые демоны молча игнорируют это поле, поэтому SDK-клиенты должны предварительно проверять `caps.features` на наличие этого тега перед отправкой.

`session_load` и `session_resume` рекламируют маршруты явного восстановления (`POST /session/:id/load` и `POST /session/:id/resume`). Старые демоны возвращают `404` для этих путей, поэтому SDK-клиенты должны предварительно проверять `caps.features` перед вызовом. `unstable_session_resume` всё ещё рекламируется как устаревший псевдоним для совместимости с SDK, которые были выпущены, пока базовый метод ACP назывался `connection.unstable_resumeSession`; новые клиенты должны ориентироваться на `session_resume`.

`slow_client_warning` охватывает два совместно выпущенных механизма противодавления SSE, введённых в #4175 Wave 2.5 PR 10: (а) демон генерирует синтетический кадр потока событий `slow_client_warning`, когда очередь подписчика превышает 75% заполнения, один раз за эпизод переполнения (перезаряжается после того, как очередь опускается ниже 37,5%); (б) `GET /session/:id/events` принимает параметр запроса `?maxQueued=N` (диапазон `[16, 2048]`) для предварительного задания размера бэклога для каждого подписчика при холодных переподключениях против большого кольца воспроизведения. Размер кольца демона управляется флагом `--event-ring-size` (по умолчанию **8000**, согласно #3803 §02). Старые демоны молча не имеют обоих — проверяйте этот тег перед включением.

`typed_event_schema` сообщает, что полезные данные событий демона соответствуют схеме `KnownDaemonEvent` SDK. Старые демоны всё равно могут передавать совместимые кадры, но SDK-клиенты должны предварительно проверять этот тег перед тем, как предполагать типизированное покрытие событий.

`client_heartbeat` рекламирует `POST /session/:id/heartbeat`. Старые демоны возвращают `404`; проверяйте этот тег перед выполнением периодических heartbeat-запросов.

`session_close` и `session_metadata` рекламируют `DELETE /session/:id` и `PATCH /session/:id/metadata`. Старые демоны возвращают `404`; проверяйте эти теги перед тем, как открывать возможности закрытия или переименования.

`session_lsp` рекламирует `GET /session/:id/lsp`, доступный только для чтения снимок состояния структурированного LSP для клиентов демона. Старые демоны возвращают `404`; проверяйте этот тег перед тем, как показывать удалённый статус LSP.

`session_status` рекламирует `GET /session/:id/status`, сводку живого моста для одной сессии по идентификатору (`clientCount` / `hasActivePrompt` и основные поля). Старые демоны возвращают `404`; проверяйте этот тег перед опросом статуса одной сессии вместо сканирования полного списка сессий.

`session_approval_mode_control`, `workspace_tool_toggle`, `workspace_init` и `workspace_mcp_restart` (issue [#4175](https://github.com/QwenLM/qwen-code/issues/4175) PR 17) рекламируют четыре маршрута управления мутациями, описанные ниже в разделе «Мутации: подтверждение, инструменты, инициализация, перезапуск MCP». Все четыре строго защищены шлюзом мутаций из PR 15 (демон, настроенный без bearer-токена, отклоняет их с 401 `token_required`). Старые демоны возвращают `404`; проверяйте каждый тег перед тем, как открывать соответствующую возможность.

`mcp_guardrails` (issue [#4175](https://github.com/QwenLM/qwen-code/issues/4175) PR 14) охватывает бюджетную поверхность MCP: поля `clientCount` / `clientBudget` / `budgetMode` / `budgets[]` в `GET /workspace/mcp`, поле `disabledReason` в ячейках каждого сервера и флаги CLI `--mcp-client-budget` / `--mcp-budget-mode`. Старые демоны полностью опускают новые поля; SDK-клиенты проверяют этот тег перед тем, как полагаться на семантику `budgets[]`. Дескриптор реестра также содержит `modes: ['warn', 'enforce']` для будущего отображения режимов функций — пока клиенты определяют режим из поля `budgetMode` снимка. Отказ сервера в режиме `enforce` детерминирован порядком объявления `Object.entries(mcpServers)`; будущий слой приоритетов области (если qwen-code его примет) изменит это на «наименьший приоритет первым», чтобы отразить соглашение claude-code `plugin < user < project < local`.

> ⚠️ **Область PR 14 v1: на сессию, а не на рабочую область.** Каждая ACP-сессия внутри демона создаёт собственный `Config` + `McpClientManager` (через `acpAgent.newSessionConfig`). Бюджетные лимиты влияют на активные MCP-клиенты **на сессию**; каждая сессия независимо читает `QWEN_SERVE_MCP_CLIENT_BUDGET` из переданного окружения. При `--mcp-client-budget=10` и 5 одновременных ACP-сессиях фактическое количество активных MCP-клиентов может достигать 5 × 10 = 50 по всему демону. Снимок `GET /workspace/mcp` читает только учёт `McpClientManager` **загрузочной сессии** — значение `budgets[0].scope: 'session'` является честным сигналом того, что это на сессию, а не агрегировано. **Wave 5 PR 23 (общий пул MCP)** представит менеджер области рабочей области и добавит ячейку `scope: 'workspace'` рядом с ячейкой на сессию для истинной агрегации между сессиями. v1 — это основа счётчика в процессе и мягкого принуждения, на которой строится PR 23.

`workspace_file_read` охватывает маршруты работы с файлами текста/списка/статистики/glob (`GET /file`, `GET /list`, `GET /glob`, `GET /stat`). `workspace_file_bytes` охватывает `GET /file/bytes`, который был добавлен позже, чтобы клиенты могли предварительно проверять поддержку необработанных окон байтов для демонов эпохи PR19. `workspace_file_write` охватывает маршруты текстовых мутаций с учётом хэша (`POST /file/write`, `POST /file/edit`). Тег записи означает, что контракт маршрута существует; это не означает, что текущее развёртывание открыто для анонимных мутаций. Запись/редактирование являются строгими маршрутами мутаций и требуют настроенного bearer-токена даже на loopback.

`daemon_status` рекламирует `GET /daemon/status`, консолидированный диагностический снимок оператора только для чтения, описанный ниже.

**Условные теги.** Небольшое количество тегов функций рекламируются только тогда, когда включена соответствующая опция развёртывания. Наличие тега = поведение включено; отсутствие = либо старый демон, предшествующий тегу, либо текущий демон, где оператор не включил опцию. В настоящее время:

Операторы, не установившие `--mcp-client-budget`, всё равно получают новые поля (с `budgetMode: 'off'`, `budgets: []`).

`mcp_guardrail_events` (issue [#4175](https://github.com/QwenLM/qwen-code/issues/4175) PR 14b) рекламирует типизированные SSE push-события, которые отображают пересечения состояний бюджета MCP без цикла опроса. На `GET /session/:id/events` приходят два типа фреймов:

- `mcp_budget_warning` — срабатывает один раз при пересечении 75% вверх от `reservedSlots.size / clientBudget`. Повторно активируется только после того, как отношение упадёт ниже 37,5% (`MCP_BUDGET_REARM_FRACTION`). Отражает гистерезис `slow_client_warning` из PR 10, но на уровне менеджера, а не на уровне бэклога подписчика. Полезная нагрузка: `{ liveCount, reservedCount, budget, thresholdRatio: 0.75, mode: 'warn' | 'enforce' }`. Срабатывает в обоих режимах `warn` и `enforce`; никогда не срабатывает в режиме `off`.
- `mcp_child_refused_batch` — срабатывает в конце каждого прохода `discoverAllMcpTools*`, когда один или несколько серверов были отклонены, А ТАКЖЕ как пакет длины 1 на пути отказа ленивого порождения `readResource`. Полезная нагрузка: `{ refusedServers: [{ name, transport, reason: 'budget_exhausted' }, ...], budget, liveCount, reservedCount, mode: 'enforce' }`. `mode` — это литерал `'enforce'`, потому что режим `warn` никогда не отказывает.

Оба события живут в кольце повторной передачи SSE на сессию (они имеют `id`), поэтому клиент, переподключающийся с `Last-Event-ID`, проходит через них; снимок на `GET /workspace/mcp` по-прежнему является источником истины для состояния после длительного отключения. Всегда включён после рекламы — условного переключателя нет. Состояние в SDK-редьюсере (`DaemonSessionViewState`) предоставляет `mcpBudgetWarningCount`, `lastMcpBudgetWarning`, `mcpChildRefusedBatchCount`, `lastMcpChildRefusedBatch` для адаптеров, которым нужен простой UI с задержкой.

## Маршруты

### `GET /health`

Проверка живости. Форма по умолчанию возвращает `200 {"status":"ok"}`, если слушатель запущен — дешёвая, без доступа к мосту, подходит для высокочастотных проверок живости в k8s/Compose.

Передайте `?deep=1` (также принимается `?deep=true` или просто `?deep`) для проверки, которая выставляет **счётчики** моста (только информационные, не настоящая проверка живости):

```json
{ "status": "ok", "sessions": 3, "pendingPermissions": 1 }
```

> ⚠️ Глубокая проверка является **информационной**, а не реальной верификацией живости. Она читает аксессоры счётчиков (`bridge.sessionCount`, `bridge.pendingPermissionCount`), которые являются простыми геттерами размера Map; они не пингуют отдельные дочерние процессы/каналы и поэтому не обнаружат зависшую, но всё ещё учитываемую сессию. Используйте её для панелей ёмкости (текущая параллельность vs. `--max-sessions`, глубина очереди), а не как триггер «изъять этого демона из ротации». Ответ `503 {"status":"degraded"}` теоретически возможен, если геттеры пользовательской реализации моста выбрасывают исключение, но геттеры реального моста никогда этого не делают — при нормальной работе глубокая проверка всегда возвращает 200. Для реальной живости полагайтесь на то, принимает ли слушатель TCP-соединение вообще (т.е. на `/health` по умолчанию без `?deep`).

**Аутентификация:** требуется **только на не-loopback привязках**. На loopback (`127.0.0.1`, `::1`, `[::1]`) `/health` регистрируется до middleware с bearer-токеном, поэтому проверки k8s/Compose внутри пода не должны носить токен. На не-loopback (`--hostname 0.0.0.0` и т.д.) маршрут регистрируется после middleware с bearer-токеном и возвращает 401 без действительного токена — иначе неаутентифицированный вызывающий мог бы опрашивать произвольные адреса, чтобы подтвердить существование `qwen serve`, что является утечкой информации низкой серьёзности, плохо сочетающейся со сканированием портов. CORS deny + Host allowlist всё ещё применяются к исключению для loopback.

### `GET /daemon/status`

Диагностика оператора только для чтения. В отличие от `/health`, это обычный API демона:
он регистрируется после bearer-аутентификации и ограничения скорости, в том числе на loopback-привязках. Параметр запроса:

- `detail=summary` (по умолчанию) читает только состояние демона в памяти.
- `detail=full` дополнительно включает диагностику живых сессий, диагностику ACP-соединения, счётчики auth device-flow и разделы статуса рабочей области.
- любое другое `detail` возвращает `400 { "code": "invalid_detail" }`.

`summary` намеренно не запрашивает методы статуса рабочей области, не запускает ACP-дочерний процесс и не порождает сессию. `full` запрашивает каждый раздел рабочей области независимо; таймаут или исключение помечают только этот раздел как `unavailable` и добавляют проблему `workspace_status_unavailable`.

    "maxSessions": 20,

В коротком окне после того, как слушатель готов, но до монтирования полного рантайма, `/daemon/status` может сообщить `daemon_runtime_starting`; если асинхронное монтирование рантайма не удалось, он сообщает `daemon_runtime_failed`, в то время как маршруты, не связанные со статусом, возвращают `503`.

### `GET /capabilities`

```json
{
  "v": 1,
  "protocolVersions": {
    "current": "v1",
    "supported": ["v1"]
  },
  "mode": "http-bridge",
  "features": ["health", "daemon_status", "capabilities", "..."],
  "modelServices": [],
  "workspaceCwd": "/canonical/path/to/workspace"
}
```

Стабильный контракт: когда `v` увеличивается, формат фрейма изменился обратно несовместимым образом.

> **`protocolVersions`** описывает версии протокола serve, которые может использовать демон. `current` — предпочитаемая версия протокола демона, а `supported` — совместимое множество. Клиенты, которым требуется конкретная версия протокола, должны проверять `supported`; UI, ориентированный на функции, всё равно должен проверять `features`. Дополнение к v=1: более старые демоны v=1 опускают это поле, поэтому SDK-клиенты, нацеленные на старые сборки, должны обрабатывать его как опциональное.

> **`modelServices` всегда равен `[]` в Стадии 1.** Агент использует свой единственный сервис модели по умолчанию и не перечисляет его по сети. Стадия 2 заполнит это поле из зарегистрированных адаптеров моделей, чтобы SDK-клиенты могли строить средства выбора сервисов; до тех пор НЕ полагайтесь на то, что это поле не пустое.

> **`workspaceCwd`** — это канонический абсолютный путь, к которому привязан данный демон (#3803 §02 — 1 демон = 1 рабочая область). Используйте его для (а) обнаружения несоответствия перед отправкой `POST /session` и (б) опускания `cwd` в `POST /session` (маршрут использует этот путь по умолчанию). Мультирабочие развёртывания выставляют несколько демонов на разных портах, каждый со своим `workspaceCwd`. Дополнение к v=1: пред-§02 демоны v=1 опускают поле — клиенты, нацеленные на старые сборки, должны проверять на null перед использованием.

### Маршруты статуса рантайма только для чтения

Эти маршруты сообщают снимки состояния рантайма со стороны демона. Они являются дополнительными маршрутами v1, не изменяют состояние и не меняют версию протокола serve. Маршруты статуса рабочей области намеренно **не** запускают ACP-дочерний процесс только потому, что клиент опрашивает GET-маршрут: если демон бездействует, они возвращают `initialized: false` с пустым снимком. Маршруты статуса сессии требуют активной сессии и используют стандартную форму `404 SessionNotFoundError` для неизвестных идентификаторов.

Теги возможностей:

- `workspace_mcp` → `GET /workspace/mcp`
- `workspace_skills` → `GET /workspace/skills`
- `workspace_providers` → `GET /workspace/providers`
- `workspace_env` → `GET /workspace/env`
- `workspace_preflight` → `GET /workspace/preflight`
- `session_context` → `GET /session/:id/context`
- `session_supported_commands` → `GET /session/:id/supported-commands`
- `session_tasks` → `GET /session/:id/tasks`
- `session_status` → `GET /session/:id/status`

Общая ячейка статуса:

```ts
type DaemonStatus =
  | 'ok'
  | 'warning'
  | 'error'
  | 'disabled'
  | 'not_started'
  | 'unknown';

type DaemonErrorKind =
  | 'missing_binary'
  | 'blocked_egress'
  | 'auth_env_error'
  | 'init_timeout'
  | 'protocol_error'
  | 'missing_file'
  | 'parse_error';

interface DaemonStatusCell {
  kind: string;
  status: DaemonStatus;
  error?: string;
  errorKind?: DaemonErrorKind;
  hint?: string;
}
```

`errorKind` — это закрытое перечисление, общее для `/workspace/preflight`, `/workspace/env` и (в конечном итоге) MCP guardrails, чтобы SDK-клиенты могли отображать исправления по категориям вместо разбора свободных сообщений. PR 13 (#4175) ввёл семь литералов, перечисленных выше; PR 14 заполнит `blocked_egress` после того, как зонд egress будет реализован.

Полезные нагрузки статуса никогда не раскрывают значения MCP env, заголовки, детали OAuth/учётной записи службы, ключи API провайдеров, `baseUrl` / `envKey` провайдера, тело навыка, пути файловой системы навыка, определения хуков или значения секретных переменных окружения. `/workspace/env` сообщает только о **наличии** разрешённых переменных env; URL-адреса прокси очищаются от учётных данных и сводятся к `host:port` перед отправкой по сети.

`budgetMode` — одно из `enforce`, `warn` или `off`. `clientBudget` отсутствует, когда бюджет не установлен. `budgets[]` **всегда является массивом** на демонах после PR 14 (возможно, пустым, когда `budgetMode === 'off'`); демоны до PR 14 опускают это поле полностью. v1 выдаёт одну ячейку с `scope: 'session'` (применение на сессию — см. раздел возможностей выше, почему). Потребители ОБЯЗАНЫ переносить дополнительные записи `budgets[]` с нераспознанными значениями `scope` — в PR 23 Волны 5 будет добавлен `scope: 'workspace'` (или `'pool'`) вместе с ячейкой на сессию без изменения схемы.

`disabledReason` на ячейках отдельных серверов различает отключение оператором (`'config'` — список конфигурации `disabledMcpServers`) и отказ по бюджету (`'budget'` — обнаружен, но никогда не подключался из-за режима `enforce`). Отказы детерминированы порядком объявления `Object.entries(mcpServers)`. `status: 'error', errorKind: 'budget_exhausted'` на сервере затеняет сырой `mcpStatus: 'disconnected'` (что правда, но не серьёзность, видимая оператору).

Применение бюджета в PR 14 v1 является **на сессию, а не на рабочую область**. Хотя демоны режима B — это `1 демон = 1 рабочая область × N сессий` после #4113 на уровне процесса, `McpClientManager` конструируется внутри каждой сессии ACP через `acpAgent.newSessionConfig`, поэтому N сессий каждая применяют свою собственную копию лимита. Снимок представляет представление сессии начальной загрузки. PR 23 Волны 5 вводит общий пул MCP на уровне рабочей области, который переводит это на настоящее применение на рабочую область.

**Обнаружение давления бюджета.** Две поверхности, обе заполняются после PR-14b:

- **Push-события** (рекламируются через `mcp_guardrail_events`): подпишитесь на `GET /session/:id/events` и отфильтруйте фреймы `mcp_budget_warning` / `mcp_child_refused_batch` через `KnownDaemonEvent`. Конечный автомат срабатывает один раз при пересечении 75% вверх (повторно активируется ниже 37,5%); отказы объединяются один раз за проход обнаружения в режиме `enforce`.
- **Опрос снимка** (рекламируется через `mcp_guardrails`): `GET /workspace/mcp` и проверьте ячейку бюджета сессии (`budgets[0]`):

- `budgets[0].status === 'warning'` ⇔ `liveCount >= 0.75 * clientBudget` (соответствует порогу гистерезиса push-события PR 14b).
- `budgets[0].status === 'error'` ⇔ `refusedCount > 0` (один или несколько серверов отклонены на этом проходе обнаружения).
- `budgets[0].status === 'ok'` ⇔ ниже порога 75% И нет отказов.

Рекомендуемая периодичность опроса: совмещена с тем, что уже опрашивает `/workspace/mcp`; снимок дёшев, а ячейка бюджета не несёт дополнительных затрат на обнаружение. SDK-клиенты, которые подписываются на push-события, всё равно выигрывают от снимка для состояния после длительного отключения (глубина кольца повторной передачи SSE конечна — `--event-ring-size`, по умолчанию 8000 — поэтому клиент, отключённый дольше, чем покрывает кольцо, возвращается к повторной синхронизации по снимку).

```ts

  | 'runtime' // name: 'node' | 'bun' | 'unknown'; value: process.versions.node
  | 'platform' // name: process.platform; value: process.arch
  | 'sandbox' // name: 'SANDBOX' | 'SEATBELT_PROFILE'; value опционален

interface DaemonEnvCell extends DaemonStatusCell {
  kind: DaemonEnvKind;
  name: string;
  present?: boolean;
  value?: string;
}
```

**Политика редактирования.** Ячейки с `kind: 'env_var'` никогда не включают поле `value`; клиенты видят только `present: boolean`. Белый список перечисленных секретных переменных env в настоящее время включает `OPENAI_API_KEY`, `ANTHROPIC_API_KEY`, `GEMINI_API_KEY`, `GOOGLE_API_KEY`, `DASHSCOPE_API_KEY`, `OPENROUTER_API_KEY` и `QWEN_SERVER_TOKEN`. Другие переменные env не перечисляются, поэтому случайно установленные секреты остаются невидимыми.

Семантика `errorKind`:

- `missing_binary` — версия Node ниже требуемой, отсутствует `QWEN_CLI_ENTRY`,

- `parse_error` — ошибка парсинга `SKILL.md`, некорректный JSON конфигурации.

  или подкласс `ModelConfigError`, проброшенный из разрешения провайдера.
- `init_timeout` — отклонение `withTimeout` в bridge (фактический таймаут
  при ожидании ACP roundtrip). Распознаётся через типизированный класс
  `BridgeTimeoutError`. Примечание: транзиентное предупреждение ячейки `mcp_discovery`
  с `connecting > 0` НЕ несёт этот kind — это обычное состояние рукопожатия,
  отличное от реального таймаута.
- `protocol_error` — ACP `extMethod` отклонён из-за закрытия канала
  посередине запроса, или потому что реестр инструментов неожиданно отсутствовал.
- `blocked_egress` — зарезервировано для PR 14 (#4175). PR 13 оставляет
  ячейку `egress` в статусе `status: 'not_started'`.

preflight-запроса (например, закрытие канала в середине запроса), массив `errors`
в envelope содержит одну ячейку `ServeStatusCell`, описывающую ошибку, а остальные
ячейки возвращаются в состояние `not_started` ACP-заполнителей. Ячейки уровня daemon
по-прежнему возвращаются.

### Маршруты файлов рабочей области

Все пути к файлам разрешаются через привязанную рабочую область (bound workspace) демона. В ответах используются
относительные пути в рабочей области, и для обычных успешных случаев никогда не возвращаются абсолютные пути файловой системы. Успешные файловые ответы включают:

```http
Cache-Control: no-store
X-Content-Type-Options: nosniff
```

Ошибки файловой системы используют такую JSON-форму:

```json
{
  "errorKind": "hash_mismatch",
  "error": "expected sha256:..., found sha256:...",
  "hint": "re-read the file and retry with the latest hash",
  "status": 409
}
```

Значения `errorKind` включают: `path_outside_workspace`, `symlink_escape`,
`path_not_found`, `binary_file`, `file_too_large`, `untrusted_workspace`,
`permission_denied`, `parse_error`, `hash_mismatch`,
`file_already_exists`, `text_not_found` и `ambiguous_text_match`.

#### `GET /file`

Читает текстовый файл. Параметры запроса: `path` (обязательный), `maxBytes`, `line` и
`limit`. Демон отклоняет бинарные файлы и файлы, превышающие лимит чтения текста.
Ответ включает `hash` — SHA-256 дайджест по сырым байтам на диске для
всего файла, даже если `line`, `limit` или `maxBytes` вернули фрагмент.

```json
{
  "kind": "file",
  "path": "src/index.ts",
  "content": "export {};\n",
  "encoding": "utf-8",
  "bom": false,
  "lineEnding": "lf",
  "sizeBytes": 11,
  "returnedBytes": 11,
  "truncated": false,
  "hash": "sha256:...",
  "matchedIgnore": null,
  "originalLineCount": null
}
```

#### `GET /file/bytes`

Читает сырые байты из файла без декодирования. Параметры запроса: `path` (обязательный),
`offset` (по умолчанию `0`) и `maxBytes` (по умолчанию `65536`, максимум `262144`). Этот
маршрут поддерживает ограниченные окна для больших бинарных файлов без загрузки всего
файла. Ответ включает `hash` только если возвращённое окно покрывает весь файл.

```json
{
  "kind": "file_bytes",
  "path": "assets/logo.png",
  "offset": 0,
  "sizeBytes": 3912,
  "returnedBytes": 3912,
  "truncated": false,
  "contentBase64": "...",
  "hash": "sha256:..."
}
```

#### `POST /file/write`

Создаёт или заменяет текстовый файл. Это строгий мутационный маршрут: при работе по loopback
без настроенного токена возвращается `401 { "code": "token_required" }`.
С флагом `--require-auth` глобальное bearer-промежуточное ПО отклоняет неаутентифицированные
запросы до выполнения маршрута.

Тело:

```json
{
  "path": "src/new.ts",
  "content": "export const value = 1;\n",
  "mode": "create"
}
```

```json
{
  "path": "src/existing.ts",
  "content": "export const value = 2;\n",
  "mode": "replace",
  "expectedHash": "sha256:..."
}
```

`mode` должен быть `create` или `replace`. `create` никогда не перезаписывает существующий
файл (`409 file_already_exists`). `replace` требует `expectedHash`; отсутствующий или
некорректный хеш даёт `400 parse_error`, а устаревший хеш — `409 hash_mismatch`. `expectedHash` имеет формат `sha256:` плюс 64 символа шестнадцатеричного числа в нижнем регистре,
вычисленный по сырым байтам на диске.

Могут быть указаны `bom`, `encoding` и `lineEnding`. Замена по умолчанию сохраняет
существующий профиль кодировки файла; явные поля переопределяют его.
Бинарная запись не поддерживается.

Демон записывает во временный файл со случайным именем в целевой директории, выполняет fsync там, где поддерживается, повторно проверяет текущий хеш непосредственно перед `rename()`, а затем
переименовывает на место. Это предотвращает наблюдение частичного файла и сериализует
запись в один и тот же файл, инициированную демоном, но это не является атомарной операцией compare-and-swap на уровне ядра: внешний редактор всё ещё может вмешаться в узком окне
между финальной проверкой хеша и переименованием.

```json
{
  "kind": "file_write",
  "path": "src/existing.ts",
  "mode": "replace",
  "created": false,
  "sizeBytes": 24,
  "hash": "sha256:...",
  "encoding": "utf-8",
  "bom": false,
  "lineEnding": "lf",
  "matchedIgnore": null
}
```

#### `POST /file/edit`

Применяет одну точную замену текста в существующем текстовом файле. Это также
строгий мутационный маршрут и требует `expectedHash`.

```json
{
  "path": "src/config.ts",
  "oldText": "timeout: 30000",
  "newText": "timeout: 60000",
  "expectedHash": "sha256:..."
}
```

`oldText` должен быть непустым и встречаться ровно один раз. Если совпадения нет — возвращается
`422 text_not_found`; несколько совпадений — `422 ambiguous_text_match`.
Маршрут сохраняет кодировку, BOM и концы строк, а также повторно проверяет
`expectedHash` непосредственно перед атомарным переименованием.

Явные записи/редактирования путей, проигнорированных игнор-файлами, разрешены, поскольку
аутентифицированный вызывающий указал путь. Успешные ответы и события аудита включают
`matchedIgnore: "file" | "directory" | null`.

```json
{
  "kind": "file_edit",
  "path": "src/config.ts",
  "replacements": 1,
  "sizeBytes": 128,
  "hash": "sha256:...",
  "encoding": "utf-8",
  "bom": false,
  "lineEnding": "lf",
  "matchedIgnore": null
}
```

Этот маршрут — это снимок «только для чтения» вне основного потока. Он намеренно не является
промптом и может быть запрошен во время стриминга сессии. Ответ содержит только
разрешённые метаданные из реестров задач агента, оболочки и монитора;
контроллеры, таймеры, смещения, ожидающие сообщения и необработанные объекты реестра
никогда не раскрываются.

### `GET /session/:id/lsp`

```json
{
  "v": 1,
  "sessionId": "<sid>",
  "workspaceCwd": "/canonical/path",
  "enabled": true,
  "configuredServers": 1,
  "readyServers": 1,
  "failedServers": 0,
  "inProgressServers": 0,
  "notStartedServers": 0,
  "servers": [
    {
      "name": "typescript",
      "status": "READY",
      "languages": ["typescript", "javascript"],
      "transport": "stdio",
      "command": "typescript-language-server"
    }
  ]
}
```

`status` — одно из `NOT_STARTED`, `IN_PROGRESS`, `READY` или `FAILED`.
Опциональное поле `error` присутствует для отказавших серверов, когда оно доступно. Отключённый LSP
(включая bare mode) возвращает HTTP 200 с `enabled: false`, нулевыми счётчиками и
`servers: []`. LSP включён, но не настроен ни один сервер — возвращает `enabled: true`,
`configuredServers: 0` и `servers: []`. Если инициализация завершается ошибкой до того, как
клиент существует, ответ может включать `initializationError`; если живой клиент
не может предоставить снимок, ответ включает `statusUnavailable: true`.

Этот маршрут раскрывает только стабильные поля, видимые клиенту. Он намеренно опускает
отладочные внутренности, такие как идентификаторы процессов, аргументы запуска, хвост stderr, корневые URI и
пути к папкам рабочей области.

### `POST /session`

Порождает нового агента или подключается к существующему (при `sessionScope: 'single'`, значение по умолчанию).

Запрос:

```json
{
  "cwd": "/absolute/path/to/workspace",
  "modelServiceId": "qwen-prod",
  "sessionScope": "thread"
}
```

| Поле              | Обязательное | Примечания                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                              |
| ----------------- | ------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `cwd`             | нет          | Абсолютный путь, совпадающий с привязанной рабочей областью демона. Если опущен, маршрут использует `boundWorkspace` (читается из `/capabilities.workspaceCwd`). Несовпадающий непустой `cwd` возвращает `400 workspace_mismatch` (#3803 §02 — 1 daemon = 1 workspace). Пути рабочей области канонизируются через `realpathSync.native` (с запасным resolve-only для несуществующих путей), чтобы файловые системы без учёта регистра не отклоняли сессии из-за разного написания.                                                                                                                                                          |
| `modelServiceId`  | нет          | Выбирает, через какой настроенный _модельный сервис_ будет работать агент (провайдер — Alibaba ModelStudio, OpenRouter и т.д.). Если опущен, агент использует значение по умолчанию. Если у рабочей области уже есть сессия, вызывает `setSessionModel` на ней и рассылает уведомление `model_switched`. Отличается от `modelId` в `POST /session/:id/model`, который выбирает модель **внутри** уже привязанного сервиса. Массив `modelServices` в `/capabilities` зарезервирован для рекламирования настроенных сервисов; в Stage 1 он всегда `[]` (сервис по умолчанию агента используется и не перечисляется через HTTP). |
| `sessionScope`    | нет          | Переопределение для разделения сессии. `'single'` (значение по умолчанию в демона) заставляет второй вызов `POST /session` для той же рабочей области повторно использовать существующую сессию (`attached: true`); `'thread'` принудительно создаёт новую отдельную сессию каждый раз. Опустите, чтобы унаследовать значение по умолчанию. Значения вне enum возвращают `400 { code: 'invalid_session_scope' }`. Старые демоны (до #4175 PR 5) игнорируют это поле — сначала проверьте `caps.features.session_scope_override`. Значение по умолчанию демона жёстко задано как `'single'` в продакшне; #4175 может добавить флаг `--sessionScope` в CLI в будущем.     |

Ответ:

```json
{
  "sessionId": "<uuid>",
  "workspaceCwd": "/canonical/path",
  "attached": false
}
```

`attached: true` означает, что сессия для этой рабочей области уже существовала, и вы теперь её разделяете.

Одновременные вызовы `POST /session` для одной рабочей области **объединяются** в один запуск — оба вызывающих получают одинаковый `sessionId`, ровно один сообщает `attached: false`. Если запуск снизу завершается ошибкой (init timeout, некорректный вывод агента, OOM), **все объединённые вызывающие получают одинаковую ошибку** — слот in-flight очищается, чтобы последующий вызов мог повторить попытку с нуля.

> ⚠️ **Отклонение `modelServiceId` при создании новой сессии происходит беззвучно в HTTP-ответе.** Неправильный `modelServiceId` (опечатка, ненастроенный сервис) НЕ вызывает 500 при создании — сессия остаётся работоспособной на модели по умолчанию агента, так что вызывающий всё равно получает `sessionId`, по которому можно повторить смену модели (через `POST /session/:id/model`). Видимым сигналом ошибки является событие `model_switch_failed` в SSE-потоке сессии, отправленное между handshake порождения и вашей первой подпиской. **Подписчикам, которым нужно наблюдать это событие, следует передать `Last-Event-ID: 0` в первом запросе `GET /session/:id/events`**, чтобы воспроизвести события из самого старого доступного события в кольце (это покрывает `model_switch_failed` во время порождения, даже если подписка приходит через несколько мс после ответа create).

### `POST /session/:id/load`

Восстанавливает сохранённую ACP-сессию по идентификатору и воспроизводит её историю через SSE. Идентификатор в пути является авторитетным; поле `sessionId` в теле игнорируется. Предварительная проверка: `caps.features.session_load` — старые демоны возвращают `404` для этого маршрута.

Запрос:

```json
{
  "cwd": "/absolute/path/to/workspace"
}
```

| Поле | Обязательное | Примечания                                                                                                                                                                                                                                              |
| ---- | ------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `cwd` | нет          | Те же правила канонизации и `workspace_mismatch`, что и в `POST /session`. Опустите, чтобы унаследовать `/capabilities.workspaceCwd`. `mcpServers` намеренно НЕ принимается здесь — MCP на уровне демона управляется настройками (соответствует `POST /session`). |

Ответ:

```json
{
  "sessionId": "persisted-1",
  "workspaceCwd": "/canonical/path",
  "attached": false,
  "state": {
    "models": { ... },
    "modes": { ... },
    "configOptions": [ ... ]
  }
}
```

`state` зеркалирует `LoadSessionResponse` из ACP — `models` это `SessionModelState`, `modes` это `SessionModeState`, `configOptions` это массив `SessionConfigOption`. Отсутствующие поля являются решением агента. Поздние подключающиеся (пути с `attached: true` ниже) получают ТОТ ЖЕ снимок `state`, который видел первый вызывающий load — демон кэширует его при входе; мутации во время выполнения (например, `model_switched`) доставляются через SSE-поток, а не в последующих ответах attach.

`attached: true` означает, что сессия уже была активна (либо из предыдущего `session/load`/`session/resume`, либо из-за объединённого конкурентного вызова, который опередил).

**Воспроизведение истории через SSE.** Пока `loadSession` выполняется на стороне агента, агент отправляет уведомления `session_update` для каждого сохранённого шага. Демон буферизирует их в шину событий сессии до возврата ответа маршрута, поэтому подписчики, которые сразу вызывают `GET /session/:id/events` с `Last-Event-ID: 0`, видят полное воспроизведение. **Кольцо воспроизведения имеет ограничение** (по умолчанию 8000 фреймов на сессию). Длинные истории со многими вызовами инструментов / мыслительными потоками могут превысить этот лимит — самые старые фреймы отбрасываются беззвучно. Клиентам, которым нужна полная история, следует подписываться сразу после возврата `load`; альтернативно они могут сохранить идентификаторы SSE-событий и использовать `Last-Event-ID` для возобновления с более поздней границы шага.

**Ошибки:**

- `404` — идентификатор сохранённой сессии не существует (`SessionNotFoundError`).
- `400` — `workspace_mismatch` (та же форма, что и в `POST /session`).
- `503` — `session_limit_exceeded` (учитывается `--max-sessions`; восстановления in-flight также учитываются).
- `409` — `restore_in_progress` (вызов `session/resume` для того же идентификатора уже выполняется). `Retry-After: 5`. Конкуренция одинаковых действий (два одновременных вызова `session/load` для одного идентификатора) объединяются — ровно один возвращает `attached: false`, остальные возвращают `attached: true` с тем же `state`.

### `POST /session/:id/resume`

Восстанавливает сохранённую ACP-сессию по идентификатору БЕЗ воспроизведения истории через SSE. Модельный контекст восстанавливается внутри на стороне агента (через `geminiClient.initialize`, читающую `config.getResumedSessionData`); SSE-поток остаётся чистым для клиентов, у которых уже отрисована история. Предварительная проверка: `caps.features.session_resume`; `unstable_session_resume` остаётся устаревшим совместимым псевдонимом для старых клиентов.

Та же форма запроса, что и у `/load`. Та же форма ответа — `state` зеркалирует `ResumeSessionResponse` из ACP. Тот же конверт ошибок, включая `409 restore_in_progress` (возникает, когда выполняется `session/load`; `session/resume`, отстающий от другого `session/resume`, объединяется).

Используйте `/load`, когда у клиента нет отрисованной истории (холодное переподключение, выбор из пикера → открыть). Используйте `/resume`, когда у клиента уже есть шаги на экране и ему нужен только дескриптор сессии на стороне демона.

> ⚠️ **Почему `unstable_session_resume` всё ещё рекламируется?** HTTP-маршрут демона и capability `session_resume` стабильны для v1, но bridge всё ещё вызывает `connection.unstable_resumeSession` из ACP. Старый тег остаётся только для того, чтобы SDK, выпущенные до `session_resume`, продолжали работать.

### `GET /workspace/:id/sessions`

Список всех активных сессий, каноническая рабочая область которых совпадает с `:id` (абсолютный cwd, закодированный для URL).

```bash
curl http://127.0.0.1:4170/workspace/$(jq -rn --arg c "$PWD" '$c|@uri')/sessions
```

Ответ:

```json
{
  "sessions": [
    {
      "sessionId": "<uuid>",
      "workspaceCwd": "/canonical/path",
      "createdAt": "2026-05-17T08:30:00.000Z",
      "displayName": "My Session",
      "clientCount": 2,
      "hasActivePrompt": false
    }
  ]
}
```

Пустой массив (не 404), когда сессий нет — UI выбора сессии не должен выдавать ошибку только потому, что рабочая область простаивает.

### `POST /session/:id/prompt`

Отправляет промпт агенту. Вызывающие с несколькими промптами ставятся в FIFO-очередь на сессию (ACP гарантирует один активный промпт на сессию).

Запрос:

```json
{
  "prompt": [{ "type": "text", "text": "What does src/main.ts do?" }]
}
```

Валидация: `prompt` должен быть непустым массивом объектов. Другие ошибки возвращают `400` до обращения к bridge.

Ответ:

```json
{ "stopReason": "end_turn" }
```

Другие причины остановки: `cancelled`, `max_tokens`, `error`, `length` (согласно спецификации ACP).

Если HTTP-клиент отключается во время промпта, демон отправляет агенту уведомление ACP `cancel`, которое завершает промпт с `stopReason: "cancelled"`.
> **Ограничение Этапа 1 — отсутствие тайм-аута промпта на стороне сервера.** Мост
> только соревнует `prompt()` агента с `transportClosedReject`
> (падение дочернего процесса агента) и сигналом прерывания HTTP-клиента
> (`AbortSignal`). Зависший, но живой агент (например, зависший вызов модели)
> блокирует сессионный FIFO до тех пор, пока HTTP-клиент не завершится
> по своему тайм-ауту и не отключится. Длительные промпты легитимны
> (глубокие исследования, анализ больших кодовых баз), поэтому дедлайн по умолчанию
> намеренно не устанавливается; Этап 2 представит опциональный
> `promptTimeoutMs` для явной настройки. А до тех пор вызывающие стороны должны
> устанавливать собственный тайм-аут на клиенте и отключаться (или вызывать
> `POST /session/:id/cancel`) по истечении.

### `POST /session/:id/cancel`

Отменяет **текущий активный** промпт в сессии. Со стороны ACP это уведомление, а не запрос — агент подтверждает отмену, разрешая активный `prompt()` со значением `cancelled`.

```bash
curl -X POST http://127.0.0.1:4170/session/$SID/cancel
# → 204 No Content
```

> **Контракт для нескольких промптов:** cancel влияет только на активный промпт. Любые промпты, ранее отправленные тем же клиентом и всё ещё стоящие в очереди за активным, продолжат выполняться. Постановка нескольких промптов в очередь — это поведение, введённое демоном (не в спецификации ACP); контракт для промптов в очереди: «они продолжают работу, если вы не отмените каждый из них или не завершите сессию через channel exit».

### `DELETE /session/:id`

Явно закрывает живую сессию. Принудительно закрывает, даже если подключены другие клиенты — отменяет любой активный промпт, разрешает ожидающие разрешения как отменённые, публикует событие `session_closed`, закрывает EventBus и удаляет сессию из карт демона. Сессии, сохранённые на диск, НЕ удаляются — их можно перезагрузить через `POST /session/:id/load`. Предварительная проверка `caps.features.session_close`.

```bash
curl -X DELETE http://127.0.0.1:4170/session/$SID
# → 204 No Content
```

Идемпотентность: возвращает `404` для неизвестных сессий (та же форма `SessionNotFoundError`, что и у других маршрутов).

> **Событие `session_closed`.** Подписчики SSE получают терминальное событие `session_closed` с полями `{ sessionId, reason: 'client_close', closedBy?: '<clientId>' }` до завершения потока. Редукторы SDK обрабатывают его идентично `session_died` (устанавливает `alive: false`, очищает `pendingPermissions`).

### `PATCH /session/:id/metadata`

Обновляет изменяемые метаданные сессии. В настоящее время поддерживает только `displayName`. Предварительная проверка `caps.features.session_metadata`.

Запрос:

```json
{ "displayName": "My Investigation Session" }
```

| Поле          | Обязательно | Примечания                                                                   |
| ------------- | ----------- | ---------------------------------------------------------------------------- |
| `displayName` | нет         | Строка, максимум 256 символов. Пустая строка очищает имя. Опустите для сохранения без изменений. |

Ответ:

```json
{ "sessionId": "<uuid>", "displayName": "My Investigation Session" }
```

Публикует событие `session_metadata_updated` в SSE-потоке сессии с полями `{ sessionId, displayName }`.

### `POST /session/:id/heartbeat`

Обновляет учёт последнего взаимодействия демона с этой сессией. Долгоживущие адаптеры (TUI/IDE/web) отправляют этот запрос через интервал, чтобы будущая политика отзыва (Wave 5 PR 24) могла отличать мёртвых клиентов от молчаливых.

Заголовки:

| Заголовок          | Обязательно | Примечания                                                                                                                                                                |
| ------------------ | ----------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `X-Qwen-Client-Id` | нет         | Возвращает идентификатор, выданный демоном в `POST /session`. Идентифицированные клиенты также обновляют свою временную метку; анонимные heartbeats обновляют только метку сессии. Должен удовлетворять тому же шаблону `[A-Za-z0-9._:-]{1,128}`, что и везде. |

Тело запроса пустое (`{}` вполне подходит — сегодня поля не читаются).

Ответ:

```json
{
  "sessionId": "<sid>",
  "clientId": "<cid>",
  "lastSeenAt": 1700000000123
}
```

`clientId` возвращается только при указании доверенного `X-Qwen-Client-Id`. `lastSeenAt` — это эпоха `Date.now()` на стороне демона (мс), сохранённая мостом.

Ошибки:

- `400` — `{ code: 'invalid_client_id' }`, когда заголовок имеет неправильный формат (правило формы заголовка) или когда он содержит `clientId`, не зарегистрированный для данной сессии (мост выбрасывает `InvalidClientIdError` до обновления любой временной метки).
- `404` — неизвестная сессия.

Шлюзование по возможностям: предварительная проверка `caps.features.client_heartbeat`. Старые демоны возвращают `404` для этого пути.

### `POST /session/:id/model`

Переключает активную модель **внутри** текущей привязанной модели сервиса сессии. Сериализуется через очередь изменения модели для данной сессии.

(Для переключения самого _сервиса_ — Alibaba ModelStudio vs OpenRouter и т.д. — передайте `modelServiceId` в `POST /session` для создания новой сессии. На Этапе 1 нет маршрута для живого переключения сервиса.)

Запрос:

```json
{ "modelId": "qwen-staging" }
```

Ответ:

```json
{ "modelId": "qwen-staging" }
```

В случае успеха публикует `model_switched` в SSE-поток. В случае неудачи публикует `model_switch_failed` (чтобы пассивные подписчики также видели сбой, а не только вызывающая сторона). Соревнует с выходом из канала агента, чтобы зависший дочерний процесс не заблокировал HTTP-обработчик.

### `POST /session/:id/recap`

Тег возможности: `session_recap`. Мост → ACP extMethod `qwen/control/session/recap`.

Генерирует одно предложение «на чём я остановился» — сводку по сессии. Обёртка над `generateSessionRecap` ядра (`packages/core/src/services/sessionRecap.ts`), которая выполняет побочный запрос к быстрой модели с отключёнными инструментами, `maxOutputTokens: 300` и строгим форматом вывода `<recap>...</recap>`. Побочный запрос читает существующую историю чата GeminiClient сессии и **не** добавляет к ней.

Тело запроса игнорируется (отправляйте `{}` или пустое). Нестрогий шлюз мутаций — поза аналогична `/session/:id/prompt` (вызов расходует токены, но не изменяет состояние). SSE-событие не публикуется.

Ответ (200):

```json
{
  "sessionId": "sess:42",
  "recap": "Debugging the auth retry race. Next: add deterministic timing to the integration test."
}
```

`recap` равен `null` (обычный 200, не ошибка), когда:

- в сессии ещё менее двух витков диалога,
- побочный запрос не вернул извлекаемый payload `<recap>...</recap>`,
- или произошла какая-либо ошибка модели (вспомогательный модуль ядра работает по принципу best-effort и никогда не выбрасывает исключения).

Ошибки:

- `400 {code: 'invalid_client_id'}` — неправильный формат заголовка `X-Qwen-Client-Id`.
- `404` — сессия неизвестна.

Отмена: **нет в v1**. Маршрут не слушает отключение HTTP-клиента, `AbortSignal` не передаётся в мост, и дочерний процесс ACP выполняет побочный запрос до завершения независимо от того, отключился ли вызывающий. Единственные ограничения — резервный тайм-аут моста в 60 секунд (`SESSION_RECAP_TIMEOUT_MS`) и соревнование с закрытием транспорта ACP. Это приемлемо, потому что recap короткий (одна попытка, `maxOutputTokens: 300`, обычно ~1–5 секунд); ext-метод отмены на основе ID запроса может обеспечить полную сквозную отмену в будущем релизе, если пропускная способность когда-нибудь это оправдает.

### Мутация: approval, tools, init, MCP restart

Задача [#4175](https://github.com/QwenLM/qwen-code/issues/4175) Волна 4 PR 17 добавляет четыре маршрута управления мутациями, которые позволяют удалённым клиентам изменять состояние выполнения без доступа к CLI хоста демона. Все четыре:

- Защищены **строгим** шлюзом мутаций из PR 15. Демон, настроенный без bearer-токена, отклоняет их с `401 {code: 'token_required'}`. Настройте `--token` (или `QWEN_SERVER_TOKEN`) перед включением.
- Принимают и штампуют заголовок `X-Qwen-Client-Id` (цепочка аудита PR 7). Когда заголовок содержит доверенный идентификатор, демон добавляет `originatorClientId` в соответствующее SSE-событие, чтобы межклиентские UI могли подавлять эхо собственных мутаций.
- Предварительно проверяют каждую возможность перед открытием доступа. Старые демоны возвращают `404` для маршрута.

Три из четырёх маршрутов (`tools/:name/enable`, `init`, `mcp/:server/restart`) генерируют события **в рабочей области (workspace-scoped)**: каждая активная SSE-шина сессии получает событие, независимо от того, к какой сессии был подключён клиент при запуске мутации. `approval-mode` генерирует событие **в рамках сессии**, потому что изменение локально для `Config` одной сессии.

#### `POST /session/:id/approval-mode`

Тег возможности: `session_approval_mode_control`. Мост → ACP extMethod `qwen/control/session/approval_mode`.

Изменяет режим утверждения живой сессии. Новый режим немедленно применяется в `Config` дочернего процесса ACP. По умолчанию настройки НЕ записываются на диск — передайте `persist: true`, чтобы также записать `tools.approvalMode` в настройки рабочей области.

Запрос:

```json
{ "mode": "auto-edit", "persist": false }
```

`mode` должен быть одним из `'plan' | 'default' | 'auto-edit' | 'auto' | 'yolo'` (зеркало перечисления `ApprovalMode` ядра; SDK экспортирует `DAEMON_APPROVAL_MODES` для проверки во время выполнения). `persist` по умолчанию равен `false`.

Ответ (200):

```json
{
  "sessionId": "sess:42",
  "mode": "auto-edit",
  "previous": "default",
  "persisted": false
}
```

Ошибки:

- `400 {code: 'invalid_approval_mode', allowed: [...]}` — неизвестный литерал режима.
- `400 {code: 'invalid_persist_flag'}` — `persist` не является булевым значением.
- `403 {code: 'trust_gate', errorKind: 'auth_env_error'}` — запрошенный режим требует доверенной папки (привилегированные режимы в недоверенных рабочих областях отклоняются `Config.setApprovalMode`).
- `404` — сессия неизвестна.

SSE-событие (в рамках сессии): `approval_mode_changed` с полями `{sessionId, previous, next, persisted, originatorClientId?}`.

#### `POST /workspace/tools/:name/enable`

Тег возможности: `workspace_tool_toggle`. Чистый файловый ввод-вывод — нет обхода ACP.

Переключает имя инструмента в списке `tools.disabled` настроек рабочей области. Инструменты, перечисленные там, **не регистрируются** вовсе (отличается от `permissions.deny`, который сохраняет инструмент зарегистрированным и отклоняет вызовы). Как встроенные инструменты, так и инструменты, обнаруженные через MCP, проходят через `ToolRegistry.registerTool`, который проверяет отключённый набор.

> ⚠️ **Имена должны точно соответствовать идентификатору, опубликованному реестром.** Разрешения псевдонимов нет — маршрут сохраняет любую строку, указанную в параметре пути, в `tools.disabled`, и следующий дочерний процесс ACP сравнивает её с `tool.name` во время регистрации. Встроенные инструменты используют своё каноническое имя реестра (глагол в snake_case): `run_shell_command`, `read_file`, `write_file`, `list_directory`, `glob`, `grep_search`, `web_fetch` и т.д. — НЕ отображаемые метки (`Shell`, `Read`, `Write`), которые показывает CLI. Инструменты, обнаруженные через MCP, используют полную форму `mcp__<server>__<name>` (именно в таком виде они транслируются в событиях `tool_toggled` и отображаются в `GET /workspace/mcp`). Отключение `Bash` НЕ предотвратит регистрацию `run_shell_command` в следующей сессии.

Живые дочерние процессы ACP сохраняют уже зарегистрированные инструменты — переключение вступает в силу при **следующем** создании дочернего процесса ACP. Комбинируйте с `POST /workspace/mcp/:server/restart` (для инструментов из MCP) или созданием новой сессии, чтобы изменение вступило в силу в текущем демоне.

Неизвестные имена инструментов принимаются: предварительное отключение ещё не установленного инструмента MCP является допустимым сценарием использования.

Запрос:

```json
{ "enabled": false }
```

Ответ (200):

```json
{ "toolName": "run_shell_command", "enabled": false }
```

Ошибки:

- `400 {code: 'invalid_tool_name'}` — пустой параметр пути или параметр пути превышает лимит в 256 символов.
- `400 {code: 'invalid_enabled_flag'}` — `enabled` отсутствует или не является булевым значением.

SSE-событие (в рабочей области): `tool_toggled` с полями `{toolName, enabled, originatorClientId?}`.

#### `POST /workspace/init`

Тег возможности: `workspace_init`. Чистый файловый ввод-вывод — нет обхода ACP, **нет вызова LLM**.

Создаёт пустой `QWEN.md` (или то, что возвращает `getCurrentGeminiMdFilename()` при переопределении `--memory-file-name`) в корне привязанной рабочей области демона. Только механическое действие — для заполнения содержимого с помощью ИИ используйте `POST /session/:id/prompt`.

По умолчанию отказывается перезаписывать файл, если он существует и содержит не только пробельные символы. Файлы, содержащие только пробелы, считаются отсутствующими (аналогично локальной команде `/init`).

Запрос:

```json
{ "force": false }
```

Ответ (200):

```json
{ "path": "/work/bound/QWEN.md", "action": "created" }
```

`action` может быть `'created'` для новых файлов, `'noop'`, когда существующий файл с пробелами остаётся нетронутым (запись не выполнялась), и `'overwrote'`, когда `force: true` заменил непустое содержимое. SSE-событие `workspace_initialized` отражает действие ответа — наблюдатели могут фильтровать по `action !== 'noop'`, чтобы реагировать только на фактические изменения на диске.

Ошибки:

- `400 {code: 'invalid_force_flag'}` — `force` не является булевым значением.
- `409 {code: 'workspace_init_conflict', path, existingSize}` — файл существует с непустым содержимым, а `force` опущен/равен `false`. Тело содержит абсолютный путь и размер (в байтах), чтобы SDK-клиенты могли отобразить запрос «перезаписать N байт?» без повторной проверки статистики.

SSE-событие (в рабочей области): `workspace_initialized` с полями `{path, action, originatorClientId?}`.

#### `POST /workspace/mcp/:server/restart`

Тег возможности: `workspace_mcp_restart`. Мост → ACP extMethod `qwen/control/workspace/mcp/restart`.

Перезапускает настроенный MCP-сервер через `McpClientManager.discoverMcpToolsForServer` дочернего процесса ACP (отключение + повторное подключение + повторное обнаружение). Предварительно проверяет текущий снимок бюджета из учёта PR 14 v1, так что перезапуск в рабочей области с насыщенным бюджетом возвращает мягкий отказ, а не вызывает каскад `BudgetExhaustedError`.

Тело запроса пустое (`{}`). Параметр пути — URL-кодированное имя сервера, как оно отображается в конфигурации `mcpServers`.

Ответ (200) — размеченное объединение с полем `restarted`:

```json
{ "serverName": "docs", "restarted": true, "durationMs": 1234 }
```

```json
{
  "serverName": "docs",
  "restarted": false,
  "skipped": true,
  "reason": "budget_would_exceed"
}
```

Причины мягкого пропуска (все возвращают 200):

| `reason`                  | Значение                                                                                                                                                                                    |
| ------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `'in_flight'`            | Для этого сервера уже выполняется другое обнаружение/перезапуск. Маршрут возвращается немедленно, не дожидаясь исходного обещания. Вызывающей стороне следует повторить после короткой задержки. |
| `'disabled'`             | Сервер настроен, но указан в `excludedMcpServers`. Включите перед перезапуском.                                                                                                             |
| `'budget_would_exceed'`  | Демон в режиме `--mcp-budget-mode=enforce`, целевой сервер в настоящее время не находится в `reservedSlots`, а общее количество живых слотов достигло `clientBudget`. Вызывающей стороне следует сначала освободить слот. |

Ошибки (не 2xx):

- `400 {code: 'invalid_server_name'}` — пустой параметр пути.
- `404` — имя сервера отсутствует в конфигурации `mcpServers` или нет живого канала ACP (перезапуск по своей сути требует живого экземпляра `McpClientManager`).
- `500` — внутренняя ошибка (например, `ToolRegistry` не инициализирован).

SSE-события (в рабочей области): `mcp_server_restarted` с полями `{serverName, durationMs, originatorClientId?}` при успехе; `mcp_server_restart_refused` с полями `{serverName, reason, originatorClientId?}` при мягком пропуске.

### `GET /session/:id/events` (SSE)

Подписка на поток событий сессии.

Заголовки:

```
Accept: text/event-stream
Last-Event-ID: 42        ← опционально, воспроизводит события после id 42
```

Параметры запроса:

| Параметр    | Обязательно | Примечания                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                           |
| ----------- | ----------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `maxQueued` | нет         | **Лимит** живых отложенных событий на подписчика. Диапазон `[16, 2048]`, по умолчанию 256. Кадры воспроизведения, принудительно отправленные при подписке, не учитываются в этом лимите; его фактически потребляют живые события, поступающие, пока подписчик ещё обрабатывает большой реплей с `Last-Event-ID: 0`. Увеличьте для холодных переподключений, чтобы живой хвост не вызвал предупреждение о медленном клиенте / вытеснение до того, как потребитель наверстает упущенное. Выход за пределы диапазона / не десятичное / присутствующее, но пустое значение возвращает `400 invalid_max_queued` до открытия SSE-рукопожатия. Предварительная проверка `caps.features.slow_client_warning` — старые демоны молча игнорируют параметр. |

Формат кадра. Строка `data:` — это **полная обёртка события**, сериализованная в JSON в одну строку — `{id?, v, type, data, originatorClientId?}`. Полезная нагрузка, специфичная для ACP (аргументы `sessionUpdate`, `requestPermission` и т.д.), находится в поле `data` обёртки; собственное поле `type` обёртки соответствует строке `event:` в SSE.

```
id: 7
event: session_update
data: {"id":7,"v":1,"type":"session_update","data":{"sessionUpdate":"agent_message_chunk","content":{"type":"text","text":"…"}}}

id: 8
event: permission_request
data: {"id":8,"v":1,"type":"permission_request","data":{"requestId":"<uuid>","sessionId":"<sid>","toolCall":{...},"options":[...]}}

: heartbeat              ← каждые 15 секунд, без полезной нагрузки

event: client_evicted    ← терминальный кадр, без id (синтетический)
data: {"v":1,"type":"client_evicted","data":{"reason":"queue_overflow","droppedAfter":42}}
```

Строки `id:` / `event:` на уровне SSE дублируют `envelope.id` / `envelope.type` для совместимости с EventSource. Потребители, использующие сырой `fetch` (SDK-шный `parseSseStream`), читают всё из JSON-обёртки и игнорируют строки преамбулы SSE.

| Тип события                   | Триггер                                                                                                                                                                                                                                                                                                        |
| ---------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `session_update`             | Любое уведомление ACP `sessionUpdate` (чанки LLM, вызовы инструментов, использование)                                                                                                                                                                                                                          |
| `permission_request`         | Агент запросил утверждение инструмента                                                                                                                                                                                                                                                                         |
| `permission_resolved`        | Какой-то клиент проголосовал по разрешению через `POST /permission/:requestId`                                                                                                                                                                                                                                 |
| `permission_partial_vote`    | (только консенсус) Голос зафиксирован, но кворум ещё не достигнут. Содержит `{requestId, sessionId, votesReceived, votesNeeded, quorum, optionTallies}`. Предварительная проверка `caps.features.permission_mediation`.                                                                                         |
| `permission_forbidden`       | Голос был отклонён активной политикой (несовпадение `designated`, `local-only` не на loopback, или голосующий `consensus` не в снимке). Содержит `{requestId, sessionId, clientId?, reason}`. Предварительная проверка `caps.features.permission_mediation`.                                                         |
| `model_switched`             | `POST /session/:id/model` успешно выполнен                                                                                                                                                                                                                                                                     |
| `model_switch_failed`        | `POST /session/:id/model` отклонён                                                                                                                                                                                                                                                                             |
| `session_died`               | Дочерний процесс агента неожиданно завершился. **Терминальный: SSE-поток закрывается после этого кадра; сессия удалена из `byId`.** Подписчикам следует восстановить сессию через `POST /session`, чтобы создать новую.                                                                                         |
| `slow_client_warning`        | Локально для подписчика: очередь заполнена ≥ 75%. **Не терминальный** — поток продолжается; предупреждение — это намёк перед вытеснением. Содержит `{queueSize, maxQueued, lastEventId}`. Срабатывает ОДИН раз за эпизод переполнения; переключается после очистки очереди ниже 37.5%. Без `id` (синтетический). |
| `client_evicted`             | Локально для подписчика: переполнение очереди. **Терминальный: SSE-поток закрывается после этого кадра** (без `id` — синтетический). Другие подписчики на той же сессии продолжают.                                                                                                                             |
| `stream_error`               | Ошибка на стороне демона при рассылке. **Терминальный: SSE-поток закрывается после этого кадра** (без `id` — синтетический).                                                                                                                                                                                  |
Семантика переподключения:

- Отправлять `Last-Event-ID: <n>` для воспроизведения событий с `id > n` из кольца текущей сессии (глубина по умолчанию **8000**, настраивается через `qwen serve --event-ring-size <n>`)
- **Обнаружение пропусков (на стороне клиента):** если `<n>` старше самого старого события, всё ещё находящегося в кольце (например, вы переподключаетесь с `Last-Event-ID: 50`, но кольцо теперь содержит 200–1199), демон воспроизводит всё, начиная с самого старого доступного события, без ошибки. Сравните `id` первого воспроизведённого события с `n + 1`; любая разница — размер потерянного окна. В фазе 2 будет внедрён явный синтетический кадр `stream_gap` на стороне демона; в фазе 1 обнаружение — ответственность клиента.
- Идентификаторы монотонны в рамках сессии, начинаются с 1
- Синтетические кадры (`client_evicted`, `slow_client_warning`, `stream_error`) намеренно не содержат `id`, чтобы не занимать порядковый номер для других подписчиков

Обратное давление:

- Очередь каждого подписчика по умолчанию — `maxQueued: 256` «живых» элементов (кадры воспроизведения при переподключении обходят лимит). Переопределяется через `?maxQueued=N` (диапазон `[16, 2048]`) в SSE-запросе.
- Когда очередь подписчика заполняется на 75%, шина принудительно отправляет этому подписчику синтетический кадр `slow_client_warning` (один раз за эпизод переполнения; повторная активация после снижения ниже 37,5%). Поток остаётся открытым — предупреждение даётся для того, чтобы клиент мог быстрее опустошить очередь или корректно отключиться и переподключиться.
- Если очередь действительно переполняется после предупреждения, шина отправляет терминальный кадр `client_evicted` и закрывает подписку.

### `POST /permission/:requestId`

Проголосовать по ожидающему `permission_request`. Активная **политика посредничества** определяет, кто побеждает:

| Политика                     | Поведение                                                                                                                                                                                                                                           |
| ---------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `first-responder` (по умолчанию) | Любой валидированный голосующий побеждает; более поздние голосующие получают `404`. Базовый вариант до F3.                                                                                                                                         |
| `designated`                 | Только инициатор запроса (`originatorClientId`) принимает решение; не-инициаторы получают `403 permission_forbidden / designated_mismatch`. Для анонимных запросов fallback на first-responder.                                                      |
| `consensus`                  | N из M голосующих должны согласиться (по умолчанию `N = floor(M/2) + 1`, переопределяется через `policy.consensusQuorum`). Первый вариант, набравший `N`, побеждает. Неразрешающие голоса получают `200` + SSE-кадры `permission_partial_vote`.       |
| `local-only`                 | Только голосующие через loopback принимают решение; удалённые вызывающие получают `403 permission_forbidden / remote_not_allowed`.                                                                                                                   |

Активная политика настраивается в `settings.json` в разделе `policy.permissionStrategy` и отображается на `/capabilities` в `body.policy.permission`. Предварительно проверяйте `caps.features.permission_mediation` (с `modes: [...]`) для набора, поддерживаемого сборкой.

> **F3 (#4175): координация разрешений между несколькими клиентами.** F3 добавил четыре указанные выше политики. Демоны до F3 использовали жёстко заданный first-responder; формат провода остаётся побитово неизменным, когда настроенная политика — `first-responder`. Новые события (`permission_partial_vote`, `permission_forbidden`) являются аддитивными — старые SDK видят их как `unrecognized_known_event` и корректно игнорируют.

> **Тайм-аут разрешения (по умолчанию 5 минут).** Запрос `permission_request`
> остаётся в ожидании до тех пор, пока: (a) какой-либо клиент не проголосует здесь, (b) не сработает `POST /session/:id/cancel`
> , (c) HTTP-клиент, инициировавший запрос, не отключится
> (отмена во время запроса разрешает все ожидающие разрешения как `cancelled`),
> (d) сессия не будет завершена, (e) демон не выключится, **или
> (f) не сработает тайм-аут разрешения для данной сессии** (`DEFAULT_PERMISSION_TIMEOUT_MS`,
> 5 минут). При срабатывании тайм-аута `requestPermission` агента
> разрешается как `{outcome: 'cancelled'}`, кольцо аудита записывает
> запись `permission.timeout`, stderr демона выводит однострочную
> заметку, а SSE-шина рассылает стандартный
> кадр `permission_resolved` cancelled, чтобы подписчики могли очиститься. Этот
> тайм-аут настраивается через `BridgeOptions.permissionResponseTimeoutMs`;
> headless-вызывающие, выполняющие длинные запросы, могут захотеть увеличить его.

Запрос:

```json
{
  "outcome": {
    "outcome": "selected",
    "optionId": "proceed_once"
  }
}
```

Результаты:

- `{ "outcome": "selected", "optionId": "<один-из-вариантов>" }` — принять / отклонить / выполнить один раз и т.д., согласно предложенным агентом вариантам
- `{ "outcome": "cancelled" }` — отменить запрос (соответствует тому, что делают `cancelSession` / `shutdown` внутри)

Ответы:

- `200 {}` — ваш голос принят (разрешён ИЛИ зафиксирован при консенсусе)
- `403 { "code": "permission_forbidden", "reason": "designated_mismatch" | "remote_not_allowed", "requestId", "sessionId" }` — F3: активная политика отклонила ваш голос
- `404 { "error": "..." }` — requestId неизвестен (уже разрешён, никогда не существовал или сессия завершена)
- `500 { "code": "cancel_sentinel_collision", ... }` — F3: `allowedOptionIds` агента содержит зарезервированный sentinel `'__cancelled__'`; нарушение контракта агент / демон
- `501 { "code": "permission_policy_not_implemented", "policy": "<имя>" }` — F3 forward-compat: литерал политики попал в схему, но её ветвь посредника ещё не построена (в настоящее время недостижимо; зарезервировано для будущих политик)

После успешного голосования каждый подключённый клиент видит `permission_resolved` с тем же `requestId` и выбранным `outcome`. При `consensus` промежуточные голоса дополнительно рассылают `permission_partial_vote` до достижения кворума.

### Маршруты аутентификации с device-потоком (issue #4175 PR 21)

Демон выступает посредником для OAuth 2.0 Device Authorization Grant (RFC 8628), так что удалённый SDK-клиент может инициировать вход, токены которого сохраняются в **файловой системе демона** — а не на клиенте. Демон сам опрашивает IdP; единственная задача клиента — отобразить URL верификации + код пользователя и (опционально) подписаться на SSE для получения событий завершения.

Тег возможности: `auth_device_flow` (всегда объявляется). Поддерживаемые провайдеры в
v1: `qwen-oauth`.

> [!note]
>
> Бесплатный уровень Qwen OAuth был прекращён 15 апреля 2026 года. Рассматривайте `qwen-oauth` как
> идентификатор устаревшего провайдера v1 в этом протоколе; новые клиенты должны предпочитать
> поддерживаемый в настоящее время провайдер аутентификации, если таковой доступен.

**Локальность выполнения.** Демон никогда не открывает браузер — даже если может. Клиент решает, нужно ли вызывать `open(verificationUri)` локально; на headless-поде (каноническое развёртывание Mode B) пользователь открывает URL на любом устройстве, где есть браузер. См. `docs/users/qwen-serve.md` для рекомендуемого UX.

**Без утечки токенов в событиях.** `auth_device_flow_started` содержит только `{deviceFlowId, providerId, expiresAt}`. Код пользователя и URL верификации возвращаются точка-точка в теле ответа POST 201 и через `GET /workspace/auth/device-flow/:id`; они никогда не транслируются по SSE.

**Одиночный экземпляр на провайдера.** Второй `POST` для того же провайдера, пока поток ещё ожидает, является идемпотентным перехватом — он возвращает существующую запись с `attached: true` вместо запуска нового запроса к IdP.

#### `POST /workspace/auth/device-flow`

Строгий шлюз для изменений: требуется bearer-токен даже при настройках loopback по умолчанию без токена (`401 token_required`).

Запрос:

```json
{ "providerId": "qwen-oauth" }
```

Ответ (`201` свежий запуск, `200` идемпотентный перехват):

```json
{
  "deviceFlowId": "fa07c61b-…",
  "providerId": "qwen-oauth",
  "status": "pending",
  "userCode": "USER-1",
  "verificationUri": "https://chat.qwen.ai/api/v1/oauth2/device",
  "verificationUriComplete": "https://chat.qwen.ai/api/v1/oauth2/device?user_code=USER-1",
  "expiresAt": 1700000600000,
  "intervalMs": 5000,
  "attached": false
}
```

Ошибки:

- `400 unsupported_provider` — неизвестный `providerId` (ответ включает `supportedProviders`)
- `409 too_many_active_flows` — достигнут лимит workspace (4); отмените один с помощью `DELETE`
- `401 token_required` — строгий шлюз отклонил запрос без токена
- `502 upstream_error` — IdP вернул неожиданную ошибку

#### `GET /workspace/auth/device-flow/:id`

Прочитать текущее состояние. Ожидающие записи возвращают `userCode/verificationUri/expiresAt/intervalMs`; терминальные записи (льготный период 5 минут) удаляют их и показывают `status` + опциональные `errorKind/hint`.

Возвращает `404 device_flow_not_found` для неизвестных id и записей, удалённых после льготного периода.

#### `DELETE /workspace/auth/device-flow/:id`

Идемпотентная отмена:

- ожидающая запись → `204` + отправка `auth_device_flow_cancelled`
- терминальная запись → `204` без операции (событие не переотправляется)
- неизвестный id → `404`

#### `GET /workspace/auth/status`

Снимок ожидающих потоков + поддерживаемых провайдеров:

```json
{
  "v": 1,
  "workspaceCwd": "/work/bound",
  "providers": [],
  "pendingDeviceFlows": [
    {
      "deviceFlowId": "fa07c61b-…",
      "providerId": "qwen-oauth",
      "expiresAt": 1700000600000
    }
  ],
  "supportedDeviceFlowProviders": ["qwen-oauth"]
}
```

#### События SSE для device-потока

Пять типизированных событий (область видимости workspace, рассылаются по всем активным шинам сессий):

- `auth_device_flow_started` `{deviceFlowId, providerId, expiresAt}` — POST успешен; SDK должен подписаться (здесь нет userCode, получите через GET если нужно)
- `auth_device_flow_throttled` `{deviceFlowId, intervalMs}` — демон выполнил upstream `slow_down`; клиенты, опрашивающие GET, должны увеличить свой интервал до этого значения
- `auth_device_flow_authorized` `{deviceFlowId, providerId, expiresAt?, accountAlias?}` — учётные данные сохранены; `accountAlias` — это метка без личной информации (никогда не email/телефон)
- `auth_device_flow_failed` `{deviceFlowId, errorKind, hint?}` — терминальное; `errorKind` — одно из `expired_token | access_denied | invalid_grant | upstream_error | persist_failed`. `persist_failed` — внутренняя ошибка демона: обмен с IdP прошёл успешно, но демон не смог надёжно сохранить учётные данные (EACCES / EROFS / ENOSPC). Пользователь должен повторить попытку после устранения проблемы с диском.
- `auth_device_flow_cancelled` `{deviceFlowId}` — DELETE успешно выполнен для ожидающей записи

> **Несовместимо с MCP.** Спецификация авторизации MCP (2025-06-18) требует OAuth 2.1 + PKCE auth-code с callback-редиректом, что не работает для демонов на headless-подах. Поверхность device-потока Mode B является приватной для демона — клиенты, нацеленные на серверы, совместимые с MCP, должны использовать другой путь аутентификации.

## Формат потоковых данных

События отправляются как стандартные фреймы EventSource. Демон записывает одну строку `data:` на фрейм (JSON не содержит встроенных переносов строк после `JSON.stringify`); парсер SDK в `packages/sdk-typescript/src/daemon/sse.ts` обрабатывает как эту форму, так и разрешённую спецификацией форму с несколькими `data:` на стороне приёма.

## Фреймы ошибок во время потоковой передачи

Если итератор моста выбрасывает исключение при обслуживании SSE-подписчика, демон отправляет терминальный фрейм `stream_error` (без `id`). Строка `data:` содержит полный конверт (той же формы, что и любой другой SSE-фрейм в этом документе); фактическое сообщение об ошибке находится в `envelope.data.error`:

```
event: stream_error
data: {"v":1,"type":"stream_error","data":{"error":"<сообщение>"}}
```

После этого соединение закрывается.

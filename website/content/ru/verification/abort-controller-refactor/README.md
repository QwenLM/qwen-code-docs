# AbortController рефакторинг — план верификации

Сценарии, используемые для ручной проверки изменений перед открытием PR. Каждый
сценарий записывается в свою панель tmux через `tmux pipe-pane -o 'cat >> <log>'`.

## Однократная настройка

```sh
# Укажите WT на вашу локальную копию проверяемой ветки.
WT=/path/to/qwen-code/worktree
LOGDIR=$WT/docs/verification/abort-controller-refactor/logs
mkdir -p "$LOGDIR"

# Соберите CLI один раз (пропустите sandbox image, пропустите vscode).
( cd "$WT" && npm run build:packages )
```

## Сценарии

Для каждого сценария:

```sh
tmux new-session -d -s qwen-verify-XX
tmux pipe-pane -t qwen-verify-XX -o "cat >> $LOGDIR/XX-name.log"
tmux send-keys -t qwen-verify-XX "cd /path/to/your/test/workspace && exec node $WT/packages/cli/dist/index.js" C-m
tmux attach -t qwen-verify-XX
```

Затем выполните действия в сессии вручную в соответствии с матрицей ниже. Нажмите `C-b d`, чтобы отключиться,
когда закончите; `tmux kill-session -t qwen-verify-XX` — чтобы остановить панель.

### 00 — Базовая линия (до исправления)

- **Настройка:** переключитесь на `main`, соберите, запустите с `NODE_OPTIONS=--trace-warnings`.
- **Вход:** длинная сессия из 50 раундов со смешанными инструментами (shell + edit + grep + agent).
- **Ожидаемый результат:** после ~30–40 раундов в stderr выводится `MaxListenersExceededWarning: ... 1500+ abort listeners added to [AbortSignal]`.
- **Лог:** `00-baseline-reproduction.log`.

### 01 — Длинная сессия, режим DEBUG (эта ветка)

- **Настройка:** `NODE_OPTIONS=--trace-warnings DEBUG=1 qwen`.
- **Вход:** тот же скрипт на 50 раундов, что и в #00.
- **Ожидаемый результат:** `MaxListenersExceededWarning` не выводится; любые другие предупреждения по-прежнему выводятся.
- **Лог:** `01-long-session-debug.log`.

### 02 — Длинная сессия, режим prod (эта ветка)

- **Настройка:** `qwen` (без debug env).
- **Вход:** тот же скрипт на 50 раундов.
- **Ожидаемый результат:** чистый вывод; временный `console.error` probe внутри обработчика (добавлен, затем удалён) подтверждает, что фильтр срабатывает.
- **Лог:** `02-long-session-prod.log`.

### 03 — Ctrl-C при прерывании потока

- **Настройка:** эта ветка, интерактивный режим.
- **Вход:** запросите долгую генерацию (>30 с); нажмите Ctrl-C в процессе.
- **Ожидаемый результат:** поток останавливается в течение ~200 мс, отображается баннер "Cancelled", следующий промпт принимает ввод. `process._getActiveHandles()` возвращается к базовому значению (используйте `:debug handles`).
- **Лог:** `03-ctrlc-streaming.log`.

### 04 — Отмена долгого выполнения shell

- **Настройка:** эта ветка.
- **Вход:** запустите `sleep 60` через инструмент shell; отмените выполнение до завершения.
- **Ожидаемый результат:** дочерний процесс завершён (проверьте `pgrep -f sleep`, результат пуст), результат инструмента показывает отмену, агент принимает следующий промпт.
- **Лог:** `04-shell-cancel.log`.

### 05 — Отмена под-агента

- **Настройка:** эта ветка.
- **Вход:** запустите длительную задачу агента через инструмент agent; отмените её от родительского.
- **Ожидаемый результат:** незавершённые вызовы инструментов под-агента прерываются, поток модели под-агента останавливается, родитель получает событие отмены.
- **Лог:** `05-subagent-cancel.log`.

### 06 — Headless / неинтерактивный abort

- **Настройка:** `qwen --prompt "do a long task"`; отправьте `SIGINT` извне через `kill -INT <pid>`.
- **Ожидаемый результат:** чистое завершение, код выхода 130, без предупреждений.
- **Лог:** `06-headless-abort.log`.

### 07 — Фоновый поток агента

- **Настройка:** интерактивный режим.
- **Вход:** запустите фонового агента (`run_in_background: true`); дайте ему завершиться; запустите второго; отмените второго в процессе работы.
- **Ожидаемый результат:** первый агент завершается нормально; второй прерывается чисто; утечка слушателей между ними отсутствует.
- **Лог:** `07-background-agent.log`.

### 08 — Базовая линия памяти

- **Настройка:** `qwen --inspect`, подключите Chrome devtools.
- **Вход:** сессия из 100 раундов.
- **Ожидаемый результат:** снимки heap на раундах 0/50/100. Количество экземпляров `AbortSignal` и количество слушателей на сигнал стабильно (нет монотонного роста).
- **Лог:** `08-memory-snapshots/`.

### 09 — Существующий потребитель combinedAbortSignal

- **Настройка:** запустите HTTP-хук с внешним сигналом и таймаутом.
- **Вход:** (a) отмените внешний сигнал во время выполнения хука; (b) дайте таймауту сработать в отдельном запуске.
- **Ожидаемый результат:** хук прерывается чисто в обоих случаях; путь shim для устаревшей функциональности отработан.
- **Лог:** `09-http-hook-shim.log`.

## Автоматические (неинтерактивные) проверки

Автоматические проверки, описанные ниже, были выполнены во время разработки и записаны в
`automated-results.md`:

- Все unit-тесты abortController проходят (`abortController.test.ts`, 26 тестов; 1 тест GC пропущен, если не задан `--expose-gc`).
- Все тесты warningHandler проходят (`warningHandler.test.ts`, 13 тестов, включая интеграционный тест stderr дочернего процесса).
- Все тесты потребителей `combineAbortSignals` проходят (`httpHookRunner.test.ts`); устаревший shim `createCombinedAbortSignal` вместе со своим тестовым файлом были удалены после миграции единственного вызывающего кода.
- Все тесты agent runtime / followup / openaiContentGenerator / hooks проходят.
- Область миграции (намеренно): только цепочка родитель→потомок в agent-runtime (`agent-interactive.ts`, `agent-core.ts`, `agent-headless.ts`) плюс `promptHookRunner.ts` (реальная утечка слушателей) была переведена на хелпер. Независимые короткоживущие контроллеры (для каждой команды shell, для каждого fetch, для каждого recall и т.д.) остаются на чистом `new AbortController()` — они быстро собираются GC и не накапливают слушателей на долгоживущем родителе. См. `migration-completeness.txt` для полного grep и обоснования.
- TypeScript в строгом режиме проходит для обоих `packages/core` и `packages/cli`.
- Prettier проходит для всех изменённых файлов.

Результаты выполнения команд см. в `automated-results.md`.

## Как собрать артефакты для тела PR

После выполнения каждого сценария прикрепите файл с транскриптом (или соответствующий фрагмент)
к PR. Для #08 (память) экспортируйте снимки heap и приложите разницу в количестве
слушателей между снимками.

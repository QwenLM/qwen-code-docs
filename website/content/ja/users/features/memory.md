# メモリ

Qwen Code の各セッションは、新しいコンテキストウィンドウから始まります。セッションをまたいで知識を引き継ぐための2つの仕組みがあり、毎回同じことを説明する必要がありません。

- **QWEN.md** — _あなた_ が一度記述し、Qwen が毎セッション読み込む指示
- **Auto-memory** — Qwen があなたから学んだことに基づいて自ら書き込むメモ

---

## QWEN.md: Qwen への指示

QWEN.md は、Qwen がプロジェクトやあなたの好みについて常に知っておくべきことを記述するプレーンテキストファイルです。これは、すべての会話の開始時に読み込まれる永続的なブリーフィングだと考えてください。

### QWEN.md に記述すること

毎セッション繰り返さなければならないようなことを追加します。

- ビルドおよびテストコマンド（`npm run test`、`make build`）
- チームが従うコーディング規約（「すべての新規ファイルには JSDoc コメントを必須とする」など）
- アーキテクチャの決定事項（「リポジトリパターンを使用し、コントローラーから直接データベースを呼び出さない」など）
- 個人の好み（「npm ではなく常に pnpm を使用する」など）

Qwen がコードを読むだけで理解できることは記述しないでください。QWEN.md は短く具体的であるほど効果的です。長くなるほど、Qwen が指示に正確に従わなくなります。

### QWEN.md の作成場所

| ファイル                          | 適用範囲                                |
| ----------------------------- | ------------------------------------------------ |
| `~/.qwen/QWEN.md`             | すべてのプロジェクトに適用される個人用                    |
| プロジェクトルートの `QWEN.md` | チーム全体用（ソース管理にコミットする）    |
| `.qwen/QWEN.local.md`         | このプロジェクトのみの個人用（git に含めない） |

これらは任意の組み合わせで配置できます。Qwen はセッション開始時にすべてを読み込みます。

リポジトリに他の AI ツール用の `AGENTS.md` ファイルがすでに存在する場合、Qwen はそれも読み込みます。指示を重複させる必要はありません。

#### `.qwen/QWEN.local.md` を使用する場合

**プロジェクト固有だが個人用** の指示、つまりこのプロジェクトに属するがチームと共有すべきではないことに使用します。

- 独自のクラスター ID、コンテナレジストリの名前空間、またはクラウドアカウント
- ローカル環境をハードコードする個人用のデバッグコマンド
- 作業中のことについて Qwen に知らせておきたいが、コミットしたくないメモ

これは共有プロジェクトの `QWEN.md` の **後** に読み込まれるため、ローカルの指示でチームの指示を補足または上書きできます。

**`.gitignore` への登録は自分で行う必要があります。** `.qwen/` はローカルディレクトリとして扱われることが多いですが、qwen-code は `.gitignore` を自動生成せず、プロジェクトによっては `.qwen/settings.json` をコミットすることもあります。以下の行を `.gitignore`（またはグローバルの git ignore）に追加してください。

```
.qwen/QWEN.local.md
```

### `/init` による自動生成

`/init` を実行すると、Qwen がコードベースを分析し、見つかったビルドコマンド、テスト手順、規約を含む初期状態の QWEN.md を作成します。すでに存在する場合は、上書きせずに追加を提案します。

### 他のファイルの参照

QWEN.md から他のファイルを指すことで、Qwen にそれらも読み込ませることができます。

```markdown
プロジェクトの概要については @README.md を参照してください。

# 規約

- Git ワークフロー: @docs/git-workflow.md
```

QWEN.md 内の任意の場所で `@path/to/file` を使用します。相対パスは QWEN.md ファイル自体からの相対パスとして解決されます。

## Auto-memory: Qwen があなたについて学ぶこと

Auto-memory はバックグラウンドで実行されます。各会話の後、Qwen は学んだ有用な情報（あなたの好み、与えたフィードバック、プロジェクトのコンテキストなど）を静かに保存し、あなたが繰り返し説明することなく、将来のセッションでそれらを使用できるようにします。

これは QWEN.md とは異なります。あなたが書くのではなく、Qwen が書きます。

### Qwen が保存する内容

Qwen は、記憶する価値のある4種類のものを探します。

| 項目                    | 例                                                 |
| ----------------------- | -------------------------------------------------------- |
| **あなたについて**           | 役割、経歴、作業スタイル              |
| **あなたのフィードバック**       | 行った修正、確認したアプローチ           |
| **プロジェクトのコンテキスト**     | 進行中の作業、コードからは明らかではない決定事項や目標 |
| **外部リファレンス** | 言及したダッシュボード、チケットトラッカー、ドキュメントへのリンク    |

Qwen はすべてを保存するわけではなく、次回実際に役立つものだけを保存します。

### 保存場所

Auto-memory ファイルは `~/.qwen/projects/<project>/memory/` に保存されます。同じリポジトリのすべてのブランチとワークツリーは同じメモリフォルダを共有するため、あるブランチで Qwen が学んだことは他のブランチでも利用可能です。

保存されるものはすべてプレーンなマークダウンです。いつでも任意のファイルを開いて編集または削除できます。

### 定期クリーンアップ

Qwen は定期的に保存されたメモリを確認し、重複を削除して古いエントリをクリーンアップします。これは、十分なセッションが蓄積された後、1日1回バックグラウンドで自動的に実行されます。今すぐ実行したい場合は、`/dream` で手動でトリガーできます。

クリーンアップの実行中、画面の隅に **✦ dreaming** と表示されます。セッションは通常通り継続されます。

### 有効化と無効化

Auto-memory はデフォルトで有効です。切り替えるには、`/memory` を開き、上部のスイッチを使用します。自動保存のみ、定期クリーンアップのみ、またはその両方をオフにできます。

`~/.qwen/settings.json`（全プロジェクトに適用）または `.qwen/settings.json`（このプロジェクトのみ）で設定することもできます。

```json
{
  "memory": {
    "enableManagedAutoMemory": true,
    "enableManagedAutoDream": true
  }
}
```

### チームメモリ（共同作業者と共有）

デフォルトでは、auto-memory は **あなた専用のプライベート** なものであり、ホームディレクトリ下に保存され、共有されることはありません。チームメモリは、チーム全体が **git を介して** 共有するオプトインの階層です。

有効にすると、Qwen は **リポジトリ内** の `.qwen/team-memory/` に3番目のメモリディレクトリを追加します。これはプライベート階層と同じ、1メモリにつき1ファイルのレイアウトと `MEMORY.md` インデックスを使用します。リポジトリにコミットされるため、通常の git の方法で共同作業者全員と共有されます。`git pull` でチームメイトのメモリを受け取り、commit/push で自分のメモリを共有します。Qwen は、すべての貢献者が従うべき規約や共有リファレンスポインタ（トラッカー、ダッシュボード）など、プロジェクト全体にわたる永続的な知識をここにルーティングし、個人的なメモや賞味期限の短いメモはプライベートに保ちます。

`settings.json` でプロジェクトごと（またはグローバル）に有効化します。

```json
{
  "memory": {
    "enableTeamMemory": true
  }
}
```

デフォルトでは **オフ** です。以下の注意事項に留意してください。

- **ソース管理され、リポジトリにアクセスできる全員に表示されます。** チームメモリはリポジトリへのコミットと同様に扱ってください。
- **シークレットはブロックされます。** `.qwen/team-memory/` への書き込みは認証情報（API キー、トークン、秘密鍵）についてスキャンされ、検出されたシークレットは拒否され、書き込まれません。このスキャンは最後の砦であり、保証ではありません。機密データはそこに置かないでください。
- **変更はレビュー可能です。** チームメモリへの書き込みは他のファイルと同様に `git status` や PR の差分に表示されるため、コミット前にレビューできます。デフォルトの承認モードでは、Qwen はチームへの書き込みの前に毎回確認を求めます。`AUTO_EDIT`/YOLO モード（自動承認をオプトインしている場合）では、プロンプトなしで適用されますが、それでも差分に表示されます。
- **ディレクトリは git で追跡されている必要があります。** プロジェクトの `.gitignore` が `.qwen/*` を除外している場合は、共有できるようにパスを再包含してください。

  ```gitignore
  !.qwen/team-memory/
  !.qwen/team-memory/**
  ```

  注意: 末尾にスラッシュのあるディレクトリ形式（`.qwen/`）ではなく、ファイルグロブ無視形式（`.qwen/*`）を使用してください。ディレクトリ形式の無視は git がフォルダ全体をスキップするようにするため、その下の `!` による再包含は無効となり、チーム階層は git 内で静かに空のままになります。階層が有効になっているにもかかわらず、そのディレクトリが git 無視対象であるか、git リポジトリ外にある場合、Qwen は起動時に1回警告を出すため、この設定ミスは見逃されません。

`QWEN_CODE_MEMORY_TEAM=1` / `=0` は、1回の実行に対して設定を上書きします。

### 自動 git 同期（オプション）

デフォルトでは、通常の git ワークフロー（受け取るには `pull`、共有するには `commit`/`push`）を使用してチームメモリを共有します。Qwen に自動で実行させるには、同期を有効にします。

```json
{
  "memory": {
    "enableTeamMemory": true,
    "enableTeamMemorySync": true
  }
}
```

有効にすると、セッション開始時に Qwen は `.qwen/team-memory/` ディレクトリのベストエフォート同期を行います。共有の `MEMORY.md` インデックスを再構築し、**まず** 共同作業者の更新を fast-forward pull し、その上にあなたのチームメモリの変更をコミットして、**その同期コミットのみを**（明示的なシングルブランチの refspec を介して）プッシュします。これにより、読み込まれるインデックスが最新の状態を反映します。チームディレクトリのみを **ステージング** し（他の作業中の変更がコミットされることはありません）、git の失敗でセッションをブロックすることはありません。デフォルトではオフです。`QWEN_CODE_MEMORY_TEAM_SYNC=1` / `=0` は、1回の実行に対して設定を上書きします。

有効にする前に知っておくべき2つのこと:

- **fast-forward pull は `.qwen/team-memory/` だけでなく、現在のブランチ全体に対して作用します**（git にはパススコープの pull が存在しません）。したがって、同期によってあなたのブランチはリモートの最新まで fast-forward されます。対照的に、プッシュはスコープ限定されています。**この同期が作成したコミットのみ** をパブリッシュするため、他の未プッシュのコミットがプッシュされることはありません。ブランチがすでに upstream よりも進んでいる場合、同期はローカルでコミットし、プッシュをスキップします。fast-forward pull が問題ないブランチで有効にするか、専用のチェックアウトで実行してください。
- **分岐したブランチはそのまま残されます**（`--ff-only` はマージを行いません）。その場合、同期はそのセッションでは何も行いません。分岐を解決（`git pull`）すれば再開されます。upstream がない（追跡設定がない）ブランチでもローカルでコミットは行いますが、プッシュはスキップされます。プッシュ先が存在しないためです。

## コマンド

### `/memory`

メモリパネルを開きます。ここから以下の操作が可能です。

- auto-memory の保存のオン/オフ
- 定期クリーンアップ（dream）のオン/オフ
- 個人用 QWEN.md (`~/.qwen/QWEN.md`) を開く
- プロジェクト用 QWEN.md を開く
- auto-memory フォルダを閲覧する

### `/init`

プロジェクト用の初期 QWEN.md を生成します。Qwen がコードベースを読み込み、見つかったビルドコマンド、テスト手順、規約を入力します。

### `/remember <text>`

Qwen が自動的に拾うのを待たずに、すぐに auto-memory に何かを保存します。

```
/remember always use snake_case for Python variable names
/remember the staging environment is at staging.example.com
```

### `/forget <text>`

説明に一致する auto-memory エントリを削除します。

```
/forget old workaround for the login bug
```

### `/dream`

自動スケジュールを待たずに、今すぐメモリクリーンアップを実行します。

```
/dream
```

## トラブルシューティング

### Qwen が QWEN.md に従わない

`/memory` を開き、どのファイルが読み込まれているかを確認します。ファイルがリストにない場合、Qwen はそれを認識できません。プロジェクトルートまたは `~/.qwen/` にあることを確認してください。

指示は具体的であるほど効果的です。

- ✓ `TypeScript ファイルには 2 スペースのインデントを使用する`
- ✗ `コードをきれいにフォーマットする`

競合する指示を含む複数の QWEN.md ファイルがある場合、Qwen の動作が一貫しなくなる可能性があります。それらを確認し、矛盾を解消してください。

### Qwen が保存した内容を確認したい

`/memory` を実行し、**Open auto-memory folder** を選択します。保存されたメモリはすべて読み取り可能なマークダウンファイルであり、閲覧、編集、削除ができます。

### Qwen が何度も忘れる

auto-memory がオンになっているにもかかわらず、Qwen がセッションをまたいで物事を記憶していないように見える場合は、`/dream` を実行してクリーンアップを強制的に実行してみてください。また、`/memory` を確認して、両方のトグルが有効になっていることを確認してください。

Qwen に常に記憶させておきたいことは、代わりに QWEN.md に追加してください。auto-memory はベストエフォート型ですが、QWEN.md は確実です。

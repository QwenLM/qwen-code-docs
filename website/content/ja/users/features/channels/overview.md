# Channels

Channels を使うと、Qwen Code エージェントとターミナルではなく Telegram、WeChat、QQ、DingTalk などのメッセージングプラットフォームからやり取りできます。スマートフォンやデスクトップのチャットアプリからメッセージを送ると、CLI と同様にエージェントが応答します。

## 仕組み

`qwen channel start` を実行すると、Qwen Code は以下を行います:

1. `settings.json` からチャンネル設定を読み込む
2. [Agent Client Protocol (ACP)](../../../developers/architecture.md) を使って単一のエージェントプロセスを起動する
3. 各メッセージングプラットフォームに接続してメッセージの受信を開始する
4. 受信したメッセージをエージェントにルーティングし、適切なチャットに応答を返す

すべてのチャンネルは、ユーザーごとに分離されたセッションを持つ単一のエージェントプロセスを共有します。各チャンネルは独自の作業ディレクトリ、モデル、インストラクションを持てます。

## クイックスタート

1. メッセージングプラットフォームでボットをセットアップする（チャンネル別ガイドを参照: [Telegram](./telegram)、[WeChat](./weixin)、[QQ Bot](./qqbot)、[DingTalk](./dingtalk)）
2. `~/.qwen/settings.json` にチャンネル設定を追加する
3. `qwen channel start` ですべてのチャンネルを起動するか、`qwen channel start <name>` で特定のチャンネルを起動する

組み込みにないプラットフォームを接続したい場合は、[Plugins](./plugins) を参照してカスタムアダプターを拡張として追加してください。

## 設定

チャンネルは `settings.json` の `channels` キー以下で設定します。各チャンネルには名前と一連のオプションがあります:

```json
{
  "channels": {
    "my-channel": {
      "type": "telegram",
      "token": "$MY_BOT_TOKEN",
      "senderPolicy": "allowlist",
      "allowedUsers": ["123456789"],
      "sessionScope": "user",
      "cwd": "/path/to/working/directory",
      "instructions": "Optional system instructions for the agent.",
      "groupPolicy": "disabled",
      "groups": {
        "*": { "requireMention": true }
      }
    }
  }
}
```

### オプション

| オプション               | 必須     | 説明                                                                                                                                           |
| ------------------------ | -------- | ---------------------------------------------------------------------------------------------------------------------------------------------- |
| `type`                   | はい     | チャンネルタイプ: `telegram`、`weixin`、`qq`、`dingtalk`、`feishu`、または拡張機能のカスタムタイプ（[Plugins](./plugins) 参照）               |
| `token`                  | Telegram | ボットトークン。環境変数から読み込む `$ENV_VAR` 構文をサポート。WeChat や DingTalk では不要                                                  |
| `clientId`               | DingTalk | DingTalk AppKey。`$ENV_VAR` 構文をサポート                                                                                                    |
| `clientSecret`           | DingTalk | DingTalk AppSecret。`$ENV_VAR` 構文をサポート                                                                                                 |
| `model`                  | いいえ   | このチャンネルで使用するモデル（例: `qwen3.5-plus`）。デフォルトモデルを上書きします。画像入力をサポートするマルチモーダルモデルに有用         |
| `senderPolicy`           | いいえ   | ボットと話せるユーザー: `allowlist`（デフォルト）、`open`、または `pairing`                                                                   |
| `allowedUsers`           | いいえ   | ボットの使用を許可するユーザー ID のリスト（`allowlist` および `pairing` ポリシーで使用）                                                    |
| `sessionScope`           | いいえ   | セッションのスコープ: `user`（デフォルト）、`thread`、または `single`                                                                         |
| `cwd`                    | いいえ   | エージェントの作業ディレクトリ。デフォルトはカレントディレクトリ                                                                              |
| `instructions`           | いいえ   | 各セッションの最初のメッセージの前に付加されるカスタムインストラクション                                                                      |
| `groupPolicy`            | いいえ   | グループチャットのアクセス: `disabled`（デフォルト）、`allowlist`、または `open`。[グループチャット](#group-chats) 参照                       |
| `groups`                 | いいえ   | グループごとの設定。キーはグループチャット ID または `"*"`（デフォルト設定用）。[グループチャット](#group-chats) 参照                         |
| `dispatchMode`           | いいえ   | ボットが処理中にメッセージを送った場合の動作: `steer`（デフォルト）、`collect`、または `followup`。[ディスパッチモード](#dispatch-modes) 参照  |
| `blockStreaming`         | いいえ   | プログレッシブなレスポンス配信: `on` または `off`（デフォルト）。[ブロックストリーミング](#block-streaming) 参照                              |
| `blockStreamingChunk`    | いいえ   | チャンクサイズの範囲: `{ "minChars": 400, "maxChars": 1000 }`。[ブロックストリーミング](#block-streaming) 参照                               |
| `blockStreamingCoalesce` | いいえ   | アイドル時のフラッシュ: `{ "idleMs": 1500 }`。[ブロックストリーミング](#block-streaming) 参照                                                |

### 送信者ポリシー

ボットとやり取りできるユーザーを制御します:

- **`allowlist`**（デフォルト）— `allowedUsers` に登録されたユーザーのみメッセージを送れます。それ以外のユーザーは無視されます。
- **`pairing`** — 未登録の送信者にはペアリングコードが送信されます。ボットのオペレーターが CLI で承認すると、永続的な許可リストに追加されます。`allowedUsers` に登録済みのユーザーはペアリングをスキップします。[DM ペアリング](#dm-pairing) 参照。
- **`open`** — 誰でもメッセージを送れます。使用には注意が必要です。

### セッションスコープ

会話セッションの管理方法を制御します:

- **`user`**（デフォルト）— ユーザーごとに 1 セッション。同一ユーザーのすべてのメッセージが同じ会話を共有します。
- **`thread`** — スレッド/トピックごとに 1 セッション。スレッド機能があるグループチャットに有用です。
- **`single`** — すべてのユーザーが 1 つのセッションを共有します。全員が同じ会話を共有します。

### トークンのセキュリティ

ボットトークンは `settings.json` に直接保存しないでください。代わりに環境変数参照を使用してください:

```json
{
  "token": "$TELEGRAM_BOT_TOKEN"
}
```

実際のトークンはシェル環境か、チャンネル起動前に読み込まれる `.env` ファイルに設定してください。

## DM ペアリング

`senderPolicy` が `"pairing"` に設定されている場合、未登録の送信者は承認フローを経由します:

1. 未登録ユーザーがボットにメッセージを送信する
2. ボットが 8 文字のペアリングコード（例: `VEQDDWXJ`）を返信する
3. ユーザーがそのコードをボットオペレーターに共有する
4. オペレーターが CLI で承認する:

```bash
qwen channel pairing approve my-channel VEQDDWXJ
```

承認されると、ユーザーの ID が `~/.qwen/channels/<name>-allowlist.json` に保存され、以降のメッセージは通常通り処理されます。

### ペアリング CLI コマンド

```bash
# 保留中のペアリングリクエストを一覧表示
qwen channel pairing list my-channel

# コードでリクエストを承認
qwen channel pairing approve my-channel <CODE>
```

### ペアリングのルール

- コードは 8 文字の大文字で、紛らわしいアルファベット（`0`/`O`/`1`/`I`）を使用しない
- コードは 1 時間後に有効期限切れになる
- チャンネルあたり最大 3 件の保留リクエスト — 1 件が期限切れか承認されるまで追加リクエストは無視される
- `settings.json` の `allowedUsers` に登録されたユーザーはペアリングを常にスキップする
- 承認されたユーザーは `~/.qwen/channels/<name>-allowlist.json` に保存される — このファイルは機密情報として扱うこと

## グループチャット

デフォルトでは、ボットはダイレクトメッセージでのみ動作します。グループチャットのサポートを有効にするには、`groupPolicy` を `"allowlist"` または `"open"` に設定します。

### グループポリシー

ボットがグループチャットに参加するかどうかを制御します:

- **`disabled`**（デフォルト）— ボットはすべてのグループメッセージを無視します。最も安全なオプションです。
- **`allowlist`** — ボットは `groups` でチャット ID が明示的に指定されたグループにのみ応答します。`"*"` キーはデフォルト設定を提供しますが、ワイルドカードとして許可する機能は**ありません**。
- **`open`** — ボットは追加されたすべてのグループで応答します。使用には注意が必要です。

### メンションゲーティング

グループでは、デフォルトでボットへの `@mention` またはボットのメッセージへの返信が必要です。これにより、グループチャットのすべてのメッセージに応答することを防ぎます。

`groups` 設定でグループごとに設定できます:

```json
{
  "groups": {
    "*": { "requireMention": true },
    "-100123456": { "requireMention": false }
  }
}
```

- **`"*"`** — すべてのグループのデフォルト設定。設定のデフォルト値を指定するのみで、許可リストへの追加ではありません。
- **グループチャット ID** — 特定のグループの設定を上書きします。`"*"` のデフォルトより優先されます。
- **`requireMention`**（デフォルト: `true`）— `true` の場合、ボットは @mention またはボットへの返信のみに応答します。`false` の場合、すべてのメッセージに応答します（専用タスクグループに有用）。

### グループメッセージの評価フロー

```
1. groupPolicy — is this group allowed?           (no → ignore)
2. requireMention — was the bot mentioned/replied to? (no → ignore)
3. senderPolicy — is this sender approved?         (no → pairing flow)
4. Route to session
```

### Telegram のグループ設定

1. ボットをグループに追加する
2. BotFather で**プライバシーモードを無効にする**（`/mybots` → Bot Settings → Group Privacy → Turn Off）— そうしないとボットはコマンド以外のメッセージを受信できません
3. プライバシーモードを変更した後、**ボットをグループから削除して再追加する**（Telegram はこの設定をキャッシュします）

### グループチャット ID の確認方法

`groups` 許可リストに使うグループのチャット ID を確認するには:

1. ボットが実行中の場合は停止する
2. グループでボットへのメンションを含むメッセージを送信する
3. Telegram Bot API でキューに入ったアップデートを確認する:

```bash
curl -s "https://api.telegram.org/bot${TELEGRAM_BOT_TOKEN}/getUpdates" | python3 -m json.tool
```

レスポンスの `message.chat.id` を確認してください — グループ ID は負の数です（例: `-5170296765`）。

## メディアサポート

Channels はテキストだけでなく、画像やファイルのエージェントへの送信もサポートしています。

### 画像

ボットに写真を送ると、エージェントがそれを確認できます — スクリーンショット、エラーメッセージ、図の共有に有用です。画像はビジョン入力としてモデルに直接送信されます。

画像サポートを使用するには、チャンネルにマルチモーダルモデルを設定します:

```json
{
  "channels": {
    "my-channel": {
      "type": "telegram",
      "model": "qwen3.5-plus",
      ...
    }
  }
}
```

### ファイル

ドキュメント（PDF、コードファイル、テキストファイルなど）をボットに送信します。ファイルは一時ディレクトリにダウンロード・保存され、エージェントにファイルパスが通知されるため、ファイル読み込みツールを使ってコンテンツを読み取れます。

ファイルはどのモデルでも使用できます — マルチモーダルサポートは不要です。

### プラットフォームの違い

| 機能         | Telegram                                     | WeChat                           | DingTalk                                      |
| ------------ | -------------------------------------------- | -------------------------------- | --------------------------------------------- |
| 画像         | Bot API 経由で直接ダウンロード               | AES 復号化付き CDN ダウンロード  | downloadCode API（2 ステップ）                |
| ファイル     | Bot API 経由で直接ダウンロード（20MB 制限）  | AES 復号化付き CDN ダウンロード  | downloadCode API（2 ステップ）                |
| キャプション | 写真/ファイルのキャプションをメッセージテキストとして含む | 非対応                   | リッチテキスト: テキストと画像を 1 メッセージに混在 |

## ディスパッチモード

ボットが前のメッセージを処理中に新しいメッセージを送った場合の動作を制御します。

- **`steer`**（デフォルト）— ボットは現在のリクエストをキャンセルし、新しいメッセージの処理を開始します。通常のチャットに最適で、フォローアップはボットの修正や方向転換を意味することが多いです。
- **`collect`** — 新しいメッセージはバッファリングされます。現在のリクエストが完了すると、バッファリングされたメッセージがすべて 1 つのフォローアッププロンプトにまとめられます。考えをキューに入れたい非同期ワークフローに適しています。
- **`followup`** — 各メッセージはキューに入れられ、順番に個別のターンとして処理されます。各メッセージが独立しているバッチワークフローに有用です。

```json
{
  "channels": {
    "my-channel": {
      "type": "telegram",
      "dispatchMode": "steer",
      ...
    }
  }
}
```

チャンネルのデフォルトを上書きして、グループごとにディスパッチモードを設定することもできます:

```json
{
  "groups": {
    "*": { "requireMention": true, "dispatchMode": "steer" },
    "-100123456": { "dispatchMode": "collect" }
  }
}
```

## ブロックストリーミング

デフォルトでは、エージェントは処理を完了してから 1 つの大きなレスポンスを送信します。ブロックストリーミングを有効にすると、エージェントが処理中に複数の短いメッセージとしてレスポンスが届きます — ChatGPT や Claude のプログレッシブ出力と同様の動作です。

```json
{
  "channels": {
    "my-channel": {
      "type": "telegram",
      "blockStreaming": "on",
      "blockStreamingChunk": { "minChars": 400, "maxChars": 1000 },
      "blockStreamingCoalesce": { "idleMs": 1500 },
      ...
    }
  }
}
```

### 仕組み

- エージェントのレスポンスは段落の境界でブロックに分割され、別々のメッセージとして送信されます
- `minChars`（デフォルト 400）— 小さなメッセージの乱発を防ぐため、このサイズに達するまでブロックを送信しない
- `maxChars`（デフォルト 1000）— 自然な区切りがないままこのサイズに達した場合、強制的に送信する
- `idleMs`（デフォルト 1500）— エージェントが一時停止した場合（ツール実行中など）、バッファリングされた内容をここまで送信する
- エージェントが完了すると、残りのテキストはすぐに送信される

必須なのは `blockStreaming` のみです。チャンクとコアレス設定はオプションで、適切なデフォルト値があります。

## スラッシュコマンド

Channels はスラッシュコマンドをサポートしています。これらはローカルで処理されます（エージェントへのラウンドトリップなし）:

- `/help` — 使用可能なコマンドを一覧表示
- `/clear` — セッションをクリアして最初からやり直す（エイリアス: `/reset`、`/new`）
- `/status` — セッション情報とアクセスポリシーを表示

その他のスラッシュコマンド（例: `/compress`、`/summary`）はエージェントに転送されます。

これらのコマンドはすべてのチャンネルタイプ（Telegram、WeChat、QQ、DingTalk）で動作します。

## 実行

```bash
# 設定されたすべてのチャンネルを起動（共有エージェントプロセス）
qwen channel start

# 特定のチャンネルを起動
qwen channel start my-channel

# サービスが実行中かどうか確認
qwen channel status

# 実行中のサービスを停止
qwen channel stop
```

ボットはフォアグラウンドで実行されます。停止するには `Ctrl+C` を押すか、別のターミナルから `qwen channel stop` を使用してください。

### マルチチャンネルモード

名前なしで `qwen channel start` を実行すると、`settings.json` で定義されたすべてのチャンネルが単一のエージェントプロセスを共有して起動します。各チャンネルは独自のセッションを維持します — Telegram ユーザーと WeChat ユーザーは同じエージェントを共有していても、それぞれ別の会話を持ちます。

各チャンネルは設定の `cwd` を使用するため、異なるチャンネルが同時に異なるプロジェクトで作業できます。

### サービス管理

チャンネルサービスは PID ファイル（`~/.qwen/channels/service.pid`）を使って実行中のインスタンスを管理します:

- **重複起動防止**: サービスがすでに実行中の状態で `qwen channel start` を実行すると、2 つ目のインスタンスを起動するのではなくエラーを表示します
- **`qwen channel stop`**: 別のターミナルから実行中のサービスを正常に停止します
- **`qwen channel status`**: サービスが実行中かどうか、稼働時間、チャンネルごとのセッション数を表示します

### クラッシュ復旧

エージェントプロセスが予期せずクラッシュした場合、チャンネルサービスは自動的に再起動してすべてのアクティブセッションの復元を試みます。ユーザーは最初からやり直すことなく会話を継続できます。

- サービス実行中はセッションが `~/.qwen/channels/sessions.json` に永続化される
- クラッシュ時: エージェントは 3 秒以内に再起動して保存済みセッションをリロードする
- 3 回連続でクラッシュした場合、サービスはエラーで終了する
- 正常シャットダウン時（Ctrl+C または `qwen channel stop`）: セッションデータはクリアされる — 次回起動は常に新規状態で始まる

# Commands

このドキュメントでは、Qwen Code がサポートするすべてのコマンドを詳しく説明します。セッションの管理、インターフェースのカスタマイズ、動作の制御を効率的に行うための参考にしてください。

Qwen Code のコマンドは特定のプレフィックスで起動され、3 つのカテゴリに分類されます。

| プレフィックス種別              | 機能説明                                              | 主な用途                                                         |
| ------------------------------- | ----------------------------------------------------- | ---------------------------------------------------------------- |
| スラッシュコマンド (`/`)        | Qwen Code 自体のメタレベル制御                        | セッション管理、設定変更、ヘルプ表示                             |
| @ コマンド (`@`)                | ローカルファイルの内容を会話に素早く注入              | 指定ファイルやディレクトリ内のコードを AI に分析させる           |
| エクスクラメーションコマンド (`!`) | システム Shell との直接やり取り                     | `git status`、`ls` などのシステムコマンドの実行                  |

## 1. スラッシュコマンド (`/`)

スラッシュコマンドは、Qwen Code のセッション、インターフェース、基本動作を管理するために使用します。

### 1.6 サイドクエスチョン (`/btw`)

`/btw` コマンドを使用すると、メインの会話フローを中断したり影響を与えたりすることなく、素早いサイドクエスチョンを聞くことができます。

| コマンド               | 説明                               |
| ---------------------- | ---------------------------------- |
| `/btw <your question>` | 素早いサイドクエスチョンを聞く     |
| `?btw <your question>` | サイドクエスチョンの代替構文       |

**仕組み:**

- サイドクエスチョンは、最近の会話コンテキスト（最大で直近 20 件のメッセージ）を含む個別の API 呼び出しとして送信されます
- レスポンスは Composer の上に表示されます — 待っている間も入力を続けられます
- メインの会話は**ブロックされません** — 独立して継続します
- サイドクエスチョンのレスポンスはメインの会話履歴には**追加されません**
- 回答はコードブロック、リスト、テーブルなどを含む完全な Markdown で表示されます

**キーボードショートカット（インタラクティブモード）:**

| ショートカット       | アクション                                              |
| -------------------- | ------------------------------------------------------- |
| `Escape`             | キャンセル（読み込み中）または非表示（完了後）          |
| `Space` または `Enter` | 回答を非表示にする（入力が空のとき）                  |
| `Ctrl+C` または `Ctrl+D` | 処理中のサイドクエスチョンをキャンセルする          |

**例:**

```
(メインの会話でコードのリファクタリングを行っている間)

> /btw What's the difference between let and var in JavaScript?

  ╭──────────────────────────────────────────╮
  │ /btw What's the difference between let   │
  │     and var in JavaScript?               │
  │                                          │
  │ + Answering...                           │
  │ Press Escape, Ctrl+C, or Ctrl+D to cancel│
  ╰──────────────────────────────────────────╯
  > (Composer remains active — keep typing)

(回答が届いた後)

  ╭──────────────────────────────────────────╮
  │ /btw What's the difference between let   │
  │     and var in JavaScript?               │
  │                                          │
  │ `let` is block-scoped, while `var` is    │
  │ function-scoped. `let` was introduced    │
  │ in ES6 and doesn't hoist the same way.   │
  │                                          │
  │ Press Space, Enter, or Escape to dismiss │
  ╰──────────────────────────────────────────╯
  > (Composer still active)
```

**サポートされている実行モード:**

| モード               | 動作                                             |
| -------------------- | ------------------------------------------------ |
| インタラクティブ     | Markdown レンダリング付きで Composer の上に表示  |
| 非インタラクティブ   | テキスト結果を返す: `btw> question\nanswer`      |
| ACP (Agent Protocol) | stream_messages 非同期ジェネレーターを返す        |

> [!tip]
>
> メインのタスクから外れることなく素早い回答が必要なときは `/btw` を使いましょう。概念の確認、事実の調査、メインのワークフローに集中しながら素早い説明を求める場合に特に便利です。

### 1.7 セッション要約 (`/recap`)

`/recap` コマンドは現在のセッションの「どこまで作業したか」という短い要約を生成します。これにより、過去の会話を何ページもスクロールして戻ることなく、中断した会話を再開できます。

| コマンド  | 説明                               |
| --------- | ---------------------------------- |
| `/recap`  | セッションの1行要約を生成して表示する |

**仕組み:**

- 利用可能な場合は設定済みの高速モデル（`fastModel` 設定）を使用し、なければメインセッションモデルにフォールバックします。要約には小型で安価なモデルで十分です。
- 最近の会話（最大 30 件のメッセージ、テキストのみ — ツール呼び出しとツールレスポンスは除外）がタイトなシステムプロンプトとともにモデルに送信されます。
- 要約は `❯` プレフィックスを付けてぼんやりとした色で表示されるため、実際のアシスタントの返信と区別できます。
- モデルターンが実行中または別のコマンドが処理中の場合は、インラインエラーで拒否されます。使用可能な会話がない場合や、基盤となる生成が失敗した場合、`/recap` は要約の代わりに短い情報メッセージを表示します — 手動コマンドは常に何らかのレスポンスを返します。

**離席後に戻ってきたときの自動トリガー:**

ターミナルが **5 分以上**非アクティブになった後にフォーカスされると、要約が自動的に生成・表示されます（モデルレスポンスが進行中でない場合のみ。進行中の場合は現在のターンが完了するまで待機して実行されます）。手動コマンドとは異なり、自動トリガーは失敗時に完全にサイレントです。生成エラーや要約するものがない場合は履歴にメッセージが追加されません。`general.showSessionRecap` 設定（デフォルト: `false`）で制御されます。手動 `/recap` コマンドはこの設定に関わらず常に機能します。

**例:**

```
> /recap

❯ Refactoring loopDetectionService.ts to address long-session OOM caused by
  unbounded streamContentHistory and contentStats. The next step is to
  implement option B (LRU sliding window with FNV-1a) pending confirmation.
```

> [!tip]
>
> `/model --fast <model>`（例: `qwen3-coder-flash`）で高速モデルを設定すると、`/recap` が速くて安価になります。自動トリガーを有効にするには `general.showSessionRecap` を `true` に設定してください。手動 `/recap` コマンドはこの設定に関わらず常に機能します。

### 1.8 差分ビューアー (`/diff`)

`/diff` コマンドは、コミットされていない変更とターンごとの差分を表示するインタラクティブな差分ビューアーを開きます。←/→ で現在の git diff と各会話ターンを切り替え、↑/↓ でファイルを閲覧し、Enter でインライン差分を表示できます。

**仕組み:**

インタラクティブモードでは、`/diff` は上部に**ソースピッカー**を持つダイアログを開きます。

- **Current** — 作業ツリーと HEAD の比較（`git diff HEAD`）。ステージ済み、未ステージ、未追跡ファイルを含むすべてのコミットされていない変更を表示します。
- **T1、T2、T3、…** — ターンごとの差分。ファイルを変更したモデルターンごとに 1 つのタブ。最新のターンが最初に表示されます。各タブにはコンテキスト用の元のプロンプトのプレビューが表示されます。

ファイル一覧には特殊な状態のタグ（`new`、`deleted`、`untracked`、`binary`、`truncated`、`oversized`）とともに、ファイルごとの統計（追加/削除行数）が表示されます。ファイルで Enter を押すと、シンタックスハイライトされたハンクでインライン差分を表示できます。

ターンごとの差分を使用するには、ファイルチェックポイントが有効になっている必要があります（インタラクティブモードではデフォルトで有効）。ファイルチェックポイントがオフの場合、「Current」ソースのみ使用可能です。

**キーボードショートカット:**

| キー       | アクション                                      |
| ---------- | ----------------------------------------------- |
| `←` / `→` | ソース間の切り替え（Current / T1 / T2…）        |
| `↑` / `↓` | ファイル一覧のナビゲーション                    |
| `j` / `k` | ファイル一覧のナビゲーション（vim スタイル）    |
| Enter     | 選択ファイルのインライン差分を表示              |
| `←` / Esc | インライン差分ビューからファイル一覧に戻る      |
| Esc       | ダイアログを閉じる                              |

**例:**

```
┌ /diff · Turn 3 "refactor the auth middleware" ──── 3 files +45 -12 ┐
│                                                                     │
│ ◀ Current · T3 · T2 · T1 ▶                                         │
│                                                                     │
│ › src/utils/parser.ts                              +30 -8           │
│   src/utils/parser.test.ts                         +12 -2           │
│   README.md                                        +3 -2            │
│                                                                     │
│ ←/→ source · ↑/↓ file · Enter view · Esc close                     │
└─────────────────────────────────────────────────────────────────────┘
```

**非インタラクティブモード:**

ヘッドレス（`--prompt`）または非インタラクティブのコンテキストでは、`/diff` は作業ツリーと HEAD の比較のプレーンテキスト要約を出力します。ターンごとのナビゲーションは使用できません。

```
3 files changed, +45 / -12
  +30  -8  src/utils/parser.ts
  +12  -2  src/utils/parser.test.ts
   +3  -2  README.md
```

## 2. @ コマンド（ファイルの導入）

@ コマンドは、ローカルのファイルまたはディレクトリの内容を会話に素早く追加するために使用します。

| コマンド形式          | 説明                                   | 例                                               |
| --------------------- | -------------------------------------- | ------------------------------------------------ |
| `@<file path>`        | 指定ファイルの内容を注入する           | `@src/main.py Please explain this code`          |
| `@<directory path>`   | ディレクトリ内のすべてのテキストファイルを再帰的に読み込む | `@docs/ Summarize content of this document` |
| 単独の `@`            | `@` 記号自体について議論する際に使用   | `@ What is this symbol used for in programming?` |

注意: パスにスペースが含まれる場合はバックスラッシュでエスケープする必要があります（例: `@My\ Documents/file.txt`）

## 3. エクスクラメーションコマンド (`!`) — Shell コマンドの実行

エクスクラメーションコマンドを使用すると、Qwen Code 内でシステムコマンドを直接実行できます。

| コマンド形式       | 説明                                                           | 例                                     |
| ------------------ | -------------------------------------------------------------- | -------------------------------------- |
| `!<shell command>` | サブ Shell でコマンドを実行する                                | `!ls -la`、`!git status`               |
| 単独の `!`         | Shell モードに切り替え、任意の入力を Shell コマンドとして直接実行する | `!`（入力）→ コマンド入力 → `!`（終了） |

環境変数: `!` で実行されるコマンドには `QWEN_CODE=1` 環境変数が設定されます。

## 4. カスタムコマンド

よく使うプロンプトをショートカットコマンドとして保存し、作業効率を向上させて一貫性を確保します。

> [!note]
>
> カスタムコマンドは、オプションの YAML フロントマターを持つ Markdown 形式を使用するようになりました。TOML 形式は非推奨ですが、後方互換性のために引き続きサポートされています。TOML ファイルが検出された場合、自動移行プロンプトが表示されます。

### クイックオーバービュー

| 機能             | 説明                                     | メリット                        | 優先度 | 適用シナリオ                                         |
| ---------------- | ---------------------------------------- | ------------------------------- | ------ | ---------------------------------------------------- |
| 名前空間         | サブディレクトリがコロン区切りのコマンドを作成する | コマンドの整理が向上する    |        |                                                      |
| グローバルコマンド | `~/.qwen/commands/`                    | すべてのプロジェクトで利用可能  | 低     | 個人的によく使うコマンド、クロスプロジェクト使用     |
| プロジェクトコマンド | `<project root directory>/.qwen/commands/` | プロジェクト固有、バージョン管理可能 | 高 | チーム共有、プロジェクト固有のコマンド           |

優先度ルール: プロジェクトコマンド > ユーザーコマンド（名前が同じ場合はプロジェクトコマンドが使用される）

### コマンド命名規則

#### ファイルパスからコマンド名へのマッピング表

| ファイルの場所                           | 生成されるコマンド | 呼び出し例            |
| ---------------------------------------- | ----------------- | --------------------- |
| `~/.qwen/commands/test.md`               | `/test`           | `/test Parameter`     |
| `<project>/.qwen/commands/git/commit.md` | `/git:commit`     | `/git:commit Message` |

命名規則: パス区切り文字（`/` または `\`）がコロン（`:`）に変換される

### Markdown ファイル形式の仕様（推奨）

カスタムコマンドはオプションの YAML フロントマターを持つ Markdown ファイルを使用します。

```markdown
---
description: Optional description (displayed in /help)
---

Your prompt content here.
Use {{args}} for parameter injection.
```

| フィールド    | 必須     | 説明                                          | 例                                         |
| ------------- | -------- | --------------------------------------------- | ------------------------------------------ |
| `description` | オプション | コマンドの説明（/help に表示される）          | `description: Code analysis tool`          |
| プロンプト本文 | 必須    | モデルに送信されるプロンプトの内容            | フロントマター後の任意の Markdown コンテンツ |

### TOML ファイル形式（非推奨）

> [!warning]
>
> **非推奨:** TOML 形式は引き続きサポートされますが、将来のバージョンで削除される予定です。Markdown 形式に移行してください。

| フィールド    | 必須     | 説明                                          | 例                                             |
| ------------- | -------- | --------------------------------------------- | ---------------------------------------------- |
| `prompt`      | 必須     | モデルに送信されるプロンプトの内容            | `prompt = "Please analyze code: {{args}}"` |
| `description` | オプション | コマンドの説明（/help に表示される）          | `description = "Code analysis tool"`           |

### パラメータ処理メカニズム

| 処理方法              | 構文               | 適用シナリオ                 | セキュリティ機能                   |
| --------------------- | ------------------ | ---------------------------- | ---------------------------------- |
| コンテキスト対応注入  | `{{args}}`         | 正確なパラメータ制御が必要   | 自動 Shell エスケープ              |
| デフォルトパラメータ処理 | 特別なマーキングなし | シンプルなコマンド、パラメータ追記 | そのまま追記                   |
| Shell コマンド注入    | `!{command}`       | 動的コンテンツが必要         | 実行前に確認が必要                 |

#### 1. コンテキスト対応注入 (`{{args}}`)

| シナリオ        | TOML 設定                               | 呼び出し方法          | 実際の効果               |
| --------------- | --------------------------------------- | --------------------- | ------------------------ |
| 生の注入        | `prompt = "Fix: {{args}}"`              | `/fix "Button issue"` | `Fix: "Button issue"`    |
| Shell コマンド内 | `prompt = "Search: !{grep {{args}} .}"` | `/search "hello"`     | `grep "hello" .` を実行  |

#### 2. デフォルトパラメータ処理

| 入力状況   | 処理方法                                               | 例                                             |
| ---------- | ------------------------------------------------------ | ---------------------------------------------- |
| パラメータあり | プロンプトの末尾に追記（2 行空けて区切る）         | `/cmd parameter` → 元のプロンプト + パラメータ |
| パラメータなし | プロンプトをそのまま送信                           | `/cmd` → 元のプロンプト                        |

🚀 動的コンテンツ注入

| 注入タイプ        | 構文           | 処理順序     | 目的                             |
| ----------------- | -------------- | ------------ | -------------------------------- |
| ファイルコンテンツ | `@{file path}` | 最初に処理   | 静的参照ファイルを注入する       |
| Shell コマンド    | `!{command}`   | 途中で処理   | 動的実行結果を注入する           |
| パラメータ置換    | `{{args}}`     | 最後に処理   | ユーザーパラメータを注入する     |

#### 3. Shell コマンド実行 (`!{...}`)

| 操作                        | ユーザーインタラクション |
| --------------------------- | ------------------------ |
| 1. コマンドとパラメータを解析する | -                   |
| 2. 自動 Shell エスケープ    | -                        |
| 3. 確認ダイアログを表示する | ✅ ユーザーの確認        |
| 4. コマンドを実行する       | -                        |
| 5. 出力をプロンプトに注入する | -                      |

例: Git コミットメッセージの生成

````markdown
---
description: Generate Commit message based on staged changes
---

Please generate a Commit message based on the following diff:

```diff
!{git diff --staged}
```
````

#### 4. ファイルコンテンツ注入 (`@{...}`)

| ファイル種別     | サポート状況           | 処理方法                    |
| ---------------- | ---------------------- | --------------------------- |
| テキストファイル | ✅ 完全サポート        | コンテンツを直接注入        |
| 画像/PDF         | ✅ マルチモーダルサポート | エンコードして注入          |
| バイナリファイル | ⚠️ 限定サポート        | スキップまたは切り捨てられる場合あり |
| ディレクトリ     | ✅ 再帰注入            | .gitignore のルールに従う   |

例: コードレビューコマンド

```markdown
---
description: Code review based on best practices
---

Review {{args}}, reference standards:

@{docs/code-standards.md}
```

## 5. CLI サブコマンド

これらのコマンドは、インタラクティブセッションを開始する前に `qwen <subcommand>` としてシェルから実行します。

### セッション管理

| コマンド             | 説明                         | 使用例                                                       |
| -------------------- | ---------------------------- | ------------------------------------------------------------ |
| `qwen sessions list` | 最近の会話セッションを一覧表示する | `qwen sessions list`、`qwen sessions list --json --limit 50` |

#### `qwen sessions list`

最近の Qwen Code セッションをメタデータ付きで一覧表示します。

**フラグ:**

| フラグ    | 型      | デフォルト | 説明                                              |
| --------- | ------- | ---------- | ------------------------------------------------- |
| `--json`  | boolean | `false`    | JSON Lines として出力する（1 行につき 1 JSON オブジェクト） |
| `--limit` | number  | `20`       | 表示するセッションの最大数                        |

**人間が読みやすい出力（デフォルト）:**

SESSION ID、STARTED（UTC タイムスタンプ）、TITLE、BRANCH、PROMPT の列を持つテーブル。

**JSON 出力 (`--json`):**

stdout に JSON Lines を出力します。各行は以下のフィールドを持つ JSON オブジェクトです。

```
sessionId, startTime, mtime, prompt, gitBranch, customTitle, titleSource, filePath, cwd
```

「さらにセッションがあります」というヒントは stderr に出力されるため、`jq` へのパイプは安全に機能します。

**例:**

```bash
# Show last 20 sessions (default)
qwen sessions list

# Show last 50 sessions
qwen sessions list --limit 50

# Output as JSON for scripting
qwen sessions list --json | jq .
```

# コマンド

このドキュメントでは、Qwen Code がサポートするすべてのコマンドについて詳しく説明します。これにより、セッションの管理、インターフェースのカスタマイズ、および動作の制御を効率的に行えるようになります。

Qwen Code のコマンドは特定の接頭辞で起動され、以下の 3 つのカテゴリに分類されます。

| 接頭辞の種類         | 機能の説明                                      | 主な使用例                                                         |
| --------------------- | ----------------------------------------------- | ------------------------------------------------------------------ |
| スラッシュコマンド (`/`) | Qwen Code 自体のメタレベル制御                   | セッションの管理、設定の変更、ヘルプの表示など                      |
| アットコマンド (`@`)    | ローカルファイルの内容を会話に素早く挿入         | AI に指定したファイルやディレクトリ内のコードを解析させるなど        |
| エクスクラメーションコマンド (`!`) | システムシェルとの直接的な対話                 | `git status`、`ls` などのシステムコマンドを実行するなど             |

## 1. スラッシュコマンド (`/`)

スラッシュコマンドは、Qwen Code セッション、インターフェース、および基本的な動作を管理するために使用されます。

### 1.1 セッションおよびプロジェクト管理

これらのコマンドは、作業の進捗状況を保存・復元・要約する際に役立ちます。

| コマンド       | 説明                                                         | 使用例                                 |
| -------------- | ------------------------------------------------------------ | -------------------------------------- |
| `/init`        | 現在のディレクトリを分析し、初期コンテキストファイルを作成します | `/init`                                |
| `/summary`     | 会話履歴に基づいてプロジェクトの要約を生成します             | `/summary`                             |
| `/compress`    | チャット履歴を要約に置き換えてトークン使用量を節約します     | `/compress`                            |
| `/resume`      | 前回の会話セッションを再開します                             | `/resume`                              |
| `/restore`     | ツール実行前のファイル状態に復元します                       | `/restore`（一覧表示）または `/restore <ID>` |

### 1.2 インターフェースとワークスペースの制御

インターフェースの外観および作業環境を調整するためのコマンド。

| コマンド         | 説明                                       | 使用例                              |
| ---------------- | ------------------------------------------ | ----------------------------------- |
| `/clear`         | ターミナル画面の内容をクリアします         | `/clear`（ショートカット: `Ctrl+L`） |
| `/theme`         | Qwen Code のビジュアルテーマを変更します   | `/theme`                            |
| `/vim`           | 入力エリアの Vim 編集モードをオン/オフに切り替えます | `/vim`                              |
| `/directory`     | マルチディレクトリ対応ワークスペースを管理します | `/dir add ./src,./tests`            |
| `/editor`        | サポートされているエディタを選択するダイアログを開きます | `/editor`                           |

### 1.3 言語設定

インターフェースおよび出力言語を制御するためのコマンドです。

| コマンド               | 説明                              | 使用例                     |
| --------------------- | --------------------------------- | -------------------------- |
| `/language`           | 言語設定の表示または変更          | `/language`                |
| → `ui [言語]`         | UI インターフェースの言語を設定   | `/language ui zh-CN`       |
| → `output [言語]`     | LLM の出力言語を設定            | `/language output Chinese` |

- 利用可能な組み込み UI 言語: `zh-CN`（簡体字中国語）、`en-US`（英語）、`ru-RU`（ロシア語）、`de-DE`（ドイツ語）  
- 出力言語の例: `Chinese`（中国語）、`English`（英語）、`Japanese`（日本語）など

### 1.4 ツールおよびモデルの管理

AI ツールおよびモデルを管理するためのコマンド。

| コマンド             | 説明                                           | 使用例                                          |
| -------------------- | ---------------------------------------------- | ----------------------------------------------- |
| `/mcp`               | 設定済みの MCP サーバーおよびツールの一覧表示   | `/mcp`, `/mcp desc`                             |
| `/tools`             | 現在利用可能なツールの一覧表示                 | `/tools`, `/tools desc`                         |
| `/skills`            | 利用可能なスキルの一覧表示および実行           | `/skills`, `/skills <名前>`                     |
| `/approval-mode`     | ツール使用時の承認モードを変更                 | `/approval-mode <モード (auto-edit)> --project` |
| →`plan`              | 分析のみ実行（実行は行わない）                 | セキュアなレビュー                              |
| →`default`           | 編集操作には承認を要求                         | 日常的な利用                                    |
| →`auto-edit`         | 編集操作を自動的に承認                         | 信頼された環境                                  |
| →`yolo`              | すべての操作を自動的に承認                     | クイックプロトタイピング                        |
| `/model`             | 現在のセッションで使用するモデルを切り替え     | `/model`                                        |
| `/extensions`        | 現在のセッションで有効なすべての拡張機能を一覧表示 | `/extensions`                                 |
| `/memory`            | AI の指示コンテキストを管理                   | `/memory add 重要な情報`                        |

### 1.5 情報、設定、およびヘルプ

利用可能な情報の取得およびシステム設定を行うためのコマンドです。

| コマンド       | 説明                                             | 使用例                             |
| -------------- | ------------------------------------------------ | ---------------------------------- |
| `/help`        | 利用可能なコマンドに関するヘルプ情報を表示します     | `/help` または `/?`               |
| `/about`       | バージョン情報を表示します                         | `/about`                          |
| `/stats`       | 現在のセッションに関する詳細な統計情報を表示します   | `/stats`                          |
| `/settings`    | 設定エディターを開きます                           | `/settings`                       |
| `/auth`        | 認証方法を変更します                             | `/auth`                           |
| `/bug`         | Qwen Code に関する問題を報告します                 | `/bug ボタンクリックが応答しない`   |
| `/copy`        | 最後の出力内容をクリップボードにコピーします       | `/copy`                           |
| `/quit`        | Qwen Code を即座に終了します                       | `/quit` または `/exit`            |

### 1.6 よく使うショートカット

| ショートカット       | 機能                    | 備考                     |
| ------------------ | ----------------------- | ------------------------ |
| `Ctrl/cmd+L`       | 画面をクリア            | `/clear` と同等         |
| `Ctrl/cmd+T`       | ツールの説明を切り替え  | MCP ツール管理           |
| `Ctrl/cmd+C`×2     | 終了確認                | 安全な終了メカニズム      |
| `Ctrl/cmd+Z`       | 入力を元に戻す          | テキスト編集             |
| `Ctrl/cmd+Shift+Z` | 入力をやり直す          | テキスト編集             |

## 2. @ コマンド（ファイルの挿入）

@ コマンドは、ローカルのファイルまたはディレクトリの内容を会話に素早く追加するためのものです。

| コマンド形式         | 説明                                       | 例                                                |
| --------------------- | ------------------------------------------ | ------------------------------------------------- |
| `@<ファイルパス>`     | 指定されたファイルの内容を挿入します       | `@src/main.py このコードを説明してください`      |
| `@<ディレクトリパス>` | ディレクトリ内のすべてのテキストファイルを再帰的に読み込みます | `@docs/ このドキュメントの内容を要約してください` |
| 単独の `@`            | `@` 記号自体について議論する場合に使用します | `@ プログラミングでこの記号は何に使われますか？`   |

注: パスにスペースが含まれる場合は、バックスラッシュでエスケープする必要があります（例：`@My\ Documents/file.txt`）。

## 3. 感嘆符コマンド (`!`) — シェルコマンドの実行

感嘆符コマンドを使用すると、Qwen Code 内で直接システムコマンドを実行できます。

| コマンド形式         | 説明                                                                 | 例                                      |
| ------------------ | ------------------------------------------------------------------ | --------------------------------------- |
| `!<シェルコマンド>` | サブシェル内でコマンドを実行                                         | `!ls -la`, `!git status`                |
| 単独の `!`          | シェルモードに切り替え。以降の入力はすべてシェルコマンドとして直接実行されます | `!`(Enter) → コマンド入力 → `!`(終了) |

環境変数: `!` 経由で実行されたコマンドには、`QWEN_CODE=1` という環境変数が設定されます。

## 4. カスタムコマンド

作業効率の向上と一貫性の確保のため、頻繁に使用するプロンプトをショートカットコマンドとして保存します。

> [!note]
>
> カスタムコマンドは、オプションの YAML フロントマターを含む Markdown 形式で記述するようになりました。TOML 形式は非推奨となりましたが、下位互換性のため引き続きサポートされています。TOML ファイルが検出された場合、自動移行のプロンプトが表示されます。

### クイック概要

| 機能             | 説明                                       | 利点                                   | 優先度 | 適用されるシナリオ                                 |
| ---------------- | ------------------------------------------ | -------------------------------------- | ------ | -------------------------------------------------- |
| 名前空間         | サブディレクトリによりコロン区切りのコマンドを作成 | コマンドの整理がしやすくなる           |        |                                                  |
| グローバルコマンド | `~/.qwen/commands/`                        | すべてのプロジェクトで利用可能       | 低     | 個人で頻繁に使うコマンド、プロジェクト横断的な利用 |
| プロジェクトコマンド | `<プロジェクトのルートディレクトリ>/.qwen/commands/` | プロジェクト固有、バージョン管理可能   | 高     | チーム間での共有、プロジェクト固有のコマンド      |

優先順位ルール：プロジェクトコマンド > ユーザーコマンド（同名のコマンドが存在する場合、プロジェクトコマンドが使用される）

### コマンド名付け規則

#### ファイルパスからコマンド名への対応表

| ファイルの場所                              | 生成されるコマンド | 例（呼び出し方）        |
| ------------------------------------------ | ------------------ | ----------------------- |
| `~/.qwen/commands/test.md`                 | `/test`            | `/test パラメーター`    |
| `<project>/.qwen/commands/git/commit.md` | `/git:commit`      | `/git:commit メッセージ` |

命名規則：パス区切り文字（`/` または `\`）はコロン（`:`）に変換されます。

### Markdown ファイル形式仕様（推奨）

カスタムコマンドでは、任意の YAML フロントマターを含む Markdown ファイルを使用します。

```markdown
---
description: オプションの説明（/help で表示されます）
---

ここにプロンプトの内容を記述します。
パラメーター挿入には `{{args}}` を使用します。
```

| フィールド         | 必須   | 説明                                       | 例                                          |
| ------------------ | ------ | ------------------------------------------ | ------------------------------------------- |
| `description`      | オプション | コマンドの説明（/help で表示されます）       | `description: コード解析ツール`              |
| プロンプト本文     | 必須   | モデルに送信されるプロンプトの内容           | フロントマター以降の任意の Markdown コンテンツ |

### TOML ファイル形式（非推奨）

> [!warning]
>
> **非推奨：** TOML 形式は現時点ではサポートされていますが、今後のバージョンで削除されます。Markdown 形式への移行をお願いします。

| フィールド        | 必須   | 説明                                     | 例                                        |
| ----------------- | ------ | ---------------------------------------- | ----------------------------------------- |
| `prompt`          | 必須   | モデルに送信されるプロンプトの内容       | `prompt = "以下のコードを解析してください: {{args}}"` |
| `description`     | 任意   | コマンドの説明（`/help` で表示されます） | `description = "コード解析ツール"`        |

### パラメーター処理メカニズム

| 処理方法                 | 構文         | 適用されるシナリオ               | セキュリティ機能                     |
| ------------------------ | ------------ | -------------------------------- | ------------------------------------ |
| コンテキスト認識型インジェクション | `{{args}}`   | パラメーターを正確に制御する必要がある場合 | 自動シェルエスケープ                 |
| 既定のパラメーター処理     | 特殊なマークなし | 単純なコマンド、パラメーター追加   | そのまま追加                         |
| シェルコマンドインジェクション | `!{command}` | 動的なコンテンツが必要な場合       | 実行前に確認が必要                   |

#### 1. コンテキストを意識した挿入 (`{{args}}`)

| シナリオ         | TOML 設定                              | 呼び出し方法           | 実際の効果                |
| ---------------- | -------------------------------------- | --------------------- | ------------------------- |
| 生の挿入         | `prompt = "修正: {{args}}"`            | `/fix "ボタンの問題"` | `修正: "ボタンの問題"`    |
| シェルコマンド内 | `prompt = "検索: !{grep {{args}} .}"` | `/search "hello"`     | `grep "hello" .` を実行 |

#### 2. デフォルトのパラメーター処理

| 入力状況       | 処理方法                                              | 例                                             |
| ---------------- | ----------------------------------------------------- | ---------------------------------------------- |
| パラメーターあり   | プロンプト末尾に追加（2 行の空行で区切り）              | `/cmd パラメーター` → 元のプロンプト + パラメーター |
| パラメーターなし   | プロンプトをそのまま送信                                | `/cmd` → 元のプロンプト                         |

🚀 動的コンテンツの挿入

| 挿入タイプ           | 構文             | 処理順序     | 目的                                 |
| -------------------- | ---------------- | ------------ | ------------------------------------ |
| ファイルの内容         | `@{ファイルパス}` | 最初に処理     | 静的な参照ファイルを挿入               |
| シェルコマンド         | `!{コマンド}`     | 中間に処理     | 動的な実行結果を挿入                   |
| パラメーター置換       | `{{args}}`        | 最後に処理     | ユーザーが指定したパラメーターを挿入       |

#### 3. シェルコマンドの実行（`!{...}`）

| 操作                             | ユーザーとのインタラクション |
| -------------------------------- | -------------------------- |
| 1. コマンドとパラメーターの解析   | —                          |
| 2. 自動シェルエスケープ           | —                          |
| 3. 確認ダイアログの表示           | ✅ ユーザーによる確認       |
| 4. コマンドの実行                 | —                          |
| 5. 出力をプロンプトに挿入         | —                          |

例：Git コミットメッセージの生成

````markdown
---
description: ステージング済みの変更に基づいてコミットメッセージを生成する
---

以下の差分に基づき、コミットメッセージを生成してください：

```diff
!{git diff --staged}
```
````

#### 4. ファイル内容の挿入（`@{...}`）

| ファイル種別 | サポート状況         | 処理方法                     |
| ------------ | ---------------------- | ---------------------------- |
| テキストファイル | ✅ 完全サポート        | 内容を直接挿入               |
| 画像／PDF    | ✅ マルチモーダル対応  | エンコードして挿入           |
| バイナリファイル | ⚠️ 制限付きサポート    | 無視されるか、切り捨てられる場合あり |
| ディレクトリ | ✅ 再帰的挿入          | `.gitignore` ルールに従う    |

例：コードレビュー用コマンド

```markdown
---
description: ベストプラクティスに基づくコードレビュー
---

{{args}} をレビューし、以下の基準を参照してください：

@{docs/code-standards.md}
```

### 実践的な作成例

#### 「純粋関数リファクタリング」コマンド作成手順表

| 操作                     | コマンド／コード                              |
| ----------------------------- | ----------------------------------------- |
| 1. ディレクトリ構造の作成 | `mkdir -p ~/.qwen/commands/refactor`      |
| 2. コマンドファイルの作成        | `touch ~/.qwen/commands/refactor/pure.md` |
| 3. コマンド内容の編集       | 以下の完全なコードを参照してください。         |
| 4. コマンドのテスト               | `@file.js` → `/refactor:pure`             |

```markdown
---
description: コードを純粋関数にリファクタリングする
---

現在のコンテキスト内のコードを分析し、純粋関数へとリファクタリングしてください。
要件：

1. リファクタリング後のコードを提示すること
2. 主な変更点および純粋関数の特性がどのように実装されたかを説明すること
3. 関数の振る舞いは変更しないこと
```

### カスタムコマンドのベストプラクティスまとめ

#### コマンド設計に関する推奨事項の表

| 実践上のポイント     | 推奨されるアプローチ                | 避けるべきこと                              |
| -------------------- | ----------------------------------- | ------------------------------------------- |
| コマンド名の付け方   | 組織化のために名前空間を使用        | あまりに汎用的すぎる名前を付ける            |
| パラメーター処理     | `{{args}}` を明確に使用            | デフォルトによる追加に依存（混同しやすい）  |
| エラー処理           | シェルのエラー出力を活用            | 実行失敗を無視                              |
| ファイルの構成       | 機能ごとにディレクトリで整理        | 全コマンドをルートディレクトリに配置      |
| 説明フィールド       | 常に明確な説明を記述                | 自動生成された説明に依存                    |

#### セキュリティ機能のリマインダーテーブル

| セキュリティ機構         | 保護効果                     | ユーザー操作               |
| ---------------------- | ---------------------------- | -------------------------- |
| シェルエスケープ         | コマンドインジェクションを防止 | 自動処理                   |
| 実行確認                 | 間違った実行を回避           | ダイアログによる確認       |
| エラー報告               | 問題の診断を支援             | エラー情報の表示           |

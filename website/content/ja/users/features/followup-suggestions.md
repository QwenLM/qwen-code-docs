# フォローアップ候補

Qwen Code は次に入力する内容を予測し、入力エリアにプレースホルダーテキストとして表示できます。この機能は LLM 呼び出しを使って会話のコンテキストを分析し、自然な次のステップの候補を生成します。

この機能は CLI でエンドツーエンドで動作します。WebUI ではフックと UI の配線は用意されていますが、候補を表示するにはホストアプリケーション側で候補生成のトリガーとフォローアップ状態の接続が必要です。

## 仕組み

Qwen Code が応答を完了すると、短い遅延（約300ms）の後、入力エリアにグレーアウトされたプレースホルダーテキストとして候補が表示されます。たとえばバグを修正した後には次のように表示されることがあります：

```
> run the tests
```

候補は会話履歴をモデルに送信することで生成され、次に自然に入力するであろう内容を予測します。レスポンスに明示的なヒント（例：`Tip: type post comments to publish findings`）が含まれている場合、そのアクションが自動的に抽出されます。

## 候補の承認

| キー           | アクション                                     |
| ------------- | --------------------------------------------- |
| `Tab`         | 候補を承認して入力欄に入力する                  |
| `Enter`       | 候補を承認して入力欄に入力する                  |
| `Right Arrow` | 候補を承認して入力欄に入力する                  |
| 何か入力する    | 候補を閉じて通常どおり入力する                  |

`Enter` は送信ではなく入力欄への入力を行うため、候補のスラッシュコマンド（例：`/clear`）を承認しても自動実行されることはありません。2回目の `Enter` で自分で送信します。

## 候補が表示される条件

以下の条件がすべて満たされたときに候補が生成されます：

- モデルが応答を完了している（ストリーミング中は対象外）
- 会話内でモデルのターンが2回以上発生している
- 直近のレスポンスにエラーがない
- 確認ダイアログが待機していない（例：シェル確認、パーミッション）
- 承認モードが `plan` に設定されていない
- 機能が有効になっている（デフォルトで有効 — 無効にするには `ui.enableFollowupSuggestions` を `false` に設定）

非インタラクティブモード（例：ヘッドレス/SDK モード）では候補は表示されません。

候補は以下の場合に自動的に閉じられます：

- 入力を開始したとき
- 新しいモデルのターンが始まったとき
- 候補が承認されたとき

## ファストモデル

デフォルトでは、候補はメイン会話と同じモデルを使用します。低レイテンシの候補を生成するには、専用のファストモデルを設定してください：

### コマンドで設定

```
/model --fast qwen3-coder-flash
```

または `/model --fast`（モデル名なし）で選択ダイアログを開きます。

### settings.json で設定

```json
{
  "fastModel": "qwen3-coder-flash"
}
```

ファストモデルはプロンプト候補と投機的実行に使用されます。設定されていない場合、メイン会話モデルがフォールバックとして使用されます。

> **コストに関する注意:** ファストモデルはレイテンシを下げますが、コストを常に下げるわけではありません。候補生成は会話のプレフィックスキャッシュを再利用します（`ui.enableCacheSharing` はデフォルトで有効）— ただし、プレフィックスキャッシュはモデルごとに独立しています。`fastModel` に別のモデルを指定すると別のキャッシュに分岐するため、会話履歴全体がファストモデルでキャッシュなしの入力として再課金されます。長い会話では、デフォルト設定（メインモデル＋共有キャッシュ）の方がファストモデルより**安くなる**場合があります。履歴の大部分がキャッシュ済みの割引料金で処理されるためです。ターンごとのコストよりもレイテンシを優先する場合に `fastModel` を設定してください。

思考/推論モードは、バックグラウンドタスク（候補生成と投機）に対してメインモデルの思考設定に関わらず自動的に無効になります。これらのタスクでは不要な内部推論にトークンを消費しないようにするためです。

## 設定

以下の設定を `settings.json` で構成できます：

| 設定                           | 型      | デフォルト | 説明                                                          |
| ------------------------------ | ------- | --------- | ------------------------------------------------------------ |
| `ui.enableFollowupSuggestions` | boolean | `true`    | フォローアップ候補の有効/無効                                  |
| `ui.enableCacheSharing`        | boolean | `true`    | キャッシュ対応のフォーク済みクエリでコスト削減（実験的）        |
| `ui.enableSpeculation`         | boolean | `false`   | 送信前に候補を投機的に実行（実験的）                           |
| `fastModel`                    | string  | `""`      | プロンプト候補と投機的実行に使用するモデル                     |

### 設定例

```json
{
  "fastModel": "qwen3-coder-flash",
  "ui": {
    "enableFollowupSuggestions": true,
    "enableCacheSharing": true
  }
}
```

## モニタリング

候補モデルの使用状況は `/stats` の出力に表示され、候補生成でファストモデルが消費したトークン数を確認できます。

ファストモデルは `/about` の出力の「Fast Model」にも表示されます。

## 候補の品質

候補は有用性を確保するための品質フィルターを通過します：

- 2〜12語（CJK：2〜30文字）、合計100文字以内であること
- 評価的な表現でないこと（「looks good」「thanks」など）
- AI 的な言い回しを使用しないこと（「Let me...」「I'll...」など）
- 複数の文や書式（マークダウン、改行）を含まないこと
- メタコメントでないこと（「nothing to suggest」「silence」など）
- エラーメッセージやラベル付きのプレフィックスでないこと（「Suggestion: ...」など）
- 1語の候補は一般的なコマンド（yes、commit、push など）のみ許可
- スラッシュコマンド（例：`/commit`）は常に1語の候補として許可

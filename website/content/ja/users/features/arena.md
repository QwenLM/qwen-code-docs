# Agent Arena

> 複数の AI モデルを同時にディスパッチして同じタスクを実行させ、その解決策を並べて比較し、最良の結果をワークスペースに適用できます。

> [!warning]
> Agent Arena は実験的な機能です。表示モードとセッション管理に関して[既知の制限事項](#limitations)があります。

Agent Arena では、複数の AI モデルを同じタスクで競わせることができます。各モデルは独立した Git ワークツリー内で完全に独立したエージェントとして動作するため、ファイル操作が互いに干渉することはありません。すべてのエージェントが完了したら、結果を比較して勝者を選択し、メインワークスペースにマージします。

[サブエージェント](./sub-agents.md)が単一セッション内で集中したサブタスクを委譲するのとは異なり、Arena エージェントは完全なトップレベルのエージェントインスタンスであり、それぞれが独自のモデル、コンテキストウィンドウ、および完全なツールアクセスを持ちます。

## Agent Arena を使う場面

Agent Arena は、異なるモデルが同じ問題にどのように取り組むかを**評価・比較**したい場合に最も効果的です。主なユースケースは次のとおりです:

- **モデルのベンチマーク**: 合成ベンチマークではなく、実際のコードベースで実際のタスクに対して異なるモデルの能力を評価する
- **Best-of-N 選択**: 複数の独立した解決策を取得して最良の実装を選ぶ
- **アプローチの探索**: 異なるモデルが同じ問題をどのように考え解決するかを確認する — 学習や洞察に有用
- **リスク低減**: 重要な変更に対して、コミット前に複数のモデルが同様のアプローチに収束することを検証する

Agent Arena は単一セッションよりも大幅に多くのトークンを消費します（各エージェントが独自のコンテキストウィンドウとモデル呼び出しを持つため）。比較の価値がコストを正当化する場合に最適です。デフォルトモデルを信頼しているルーティンタスクでは、単一セッションの方が効率的です。

## Arena セッションの開始

`/arena` スラッシュコマンドを使用してセッションを起動します。競わせるモデルとタスクを指定します:

```
/arena --models qwen3.5-plus,glm-5,kimi-k2.5 "Refactor the authentication module to use JWT tokens"
```

`--models` を省略すると、設定済みプロバイダーから選択できるインタラクティブなモデル選択ダイアログが表示されます。

### 開始時の動作

1. **ワークツリーのセットアップ**: Qwen Code は各エージェントのために `~/.qwen/arena/<session-id>/worktrees/<model-name>/` に独立した Git ワークツリーを作成します。各ワークツリーは現在の作業ディレクトリの状態を完全に反映します（ステージ済みの変更、未ステージの変更、未追跡のファイルを含む）。
2. **エージェントのスポーン**: 各エージェントは完全なツールアクセスと設定済みモデルを持つ独自のワークツリーで起動します。エージェントは順次起動されますが、並列で実行されます。
3. **実行**: すべてのエージェントが共有状態や通信なしで独立してタスクに取り組みます。進捗を監視し、任意のエージェントとやり取りできます。
4. **完了**: すべてのエージェントが完了（または失敗）すると、結果比較フェーズに入ります。

## エージェントとのやり取り

### 表示モード

Agent Arena は現在**インプロセスモード**をサポートしており、すべてのエージェントが同じターミナルプロセス内で非同期に実行されます。ターミナル下部のタブバーでエージェントを切り替えられます。

> [!note]
> **分割ペイン表示モードは将来的に計画されています。** tmux ベースおよび iTerm2 ベースの分割ペインレイアウトをサポートする予定であり、各エージェントが真のサイドバイサイド表示のために独自のターミナルペインを持ちます。現在は、インプロセスのタブ切り替えのみ利用可能です。

### エージェント間のナビゲーション

インプロセスモードでは、キーボードショートカットを使用してエージェントビューを切り替えます:

| ショートカット | アクション                         |
| :------- | :-------------------------------- |
| `Right`  | 次のエージェントタブに切り替え      |
| `Left`   | 前のエージェントタブに切り替え      |
| `Up`     | 入力ボックスにフォーカスを移動      |
| `Down`   | エージェントタブバーにフォーカスを移動 |

タブバーには各エージェントの現在のステータスが表示されます:

| インジケーター | 意味                |
| :-------- | :--------------------- |
| `●`       | 実行中またはアイドル        |
| `✓`       | 正常に完了              |
| `✗`       | 失敗                   |
| `○`       | キャンセル済み              |

### 個別エージェントとのやり取り

エージェントのタブを表示中に以下のことができます:

- **メッセージの送信** — 入力エリアに入力してエージェントに追加の指示を与える
- **ツール呼び出しの承認** — エージェントがツール承認を要求した場合、確認ダイアログがそのタブに表示される
- **完全な履歴の表示** — モデルの出力、ツール呼び出し、結果を含むエージェントの完全な会話をスクロールして確認する

各エージェントは完全な独立したセッションです。メインエージェントでできることはすべて Arena エージェントでもできます。

## 結果の比較と勝者の選択

すべてのエージェントが完了すると、Arena は結果比較フェーズに入ります。以下が表示されます:

- **ステータスサマリー**: どのエージェントが成功、失敗、またはキャンセルされたか
- **実行メトリクス**: 各エージェントの実行時間、推論ラウンド数、トークン使用量、ツール呼び出し回数
- **Arena 比較サマリー**: 共通で変更されたファイルと一方のエージェントのみが変更したファイル、行変更数、トークン効率、各エージェントの差分・メトリクス・会話履歴から生成された高レベルなアプローチサマリー

選択ダイアログに成功したエージェントが表示されます。変更をメインワークスペースに適用するエージェントを選択するか、すべての結果を破棄します。`p` を押すとハイライトされたエージェントのクイックプレビューを切り替えられ、`d` を押すと勝者を選択する前にそのエージェントの詳細な差分を切り替えられます。

### 勝者を選択したときの動作

1. 勝者エージェントの変更がベースラインに対する差分として抽出される
2. 差分がメインの作業ディレクトリに適用される
3. すべてのワークツリーと一時ブランチが自動的にクリーンアップされる

決定前に完全な推論経路を確認したい場合、選択ダイアログがアクティブな間も各エージェントの完全な会話履歴はタブバーから引き続き利用できます。

## 設定

Arena の動作は [settings.json](../configuration/settings.md) でカスタマイズできます:

```json
{
  "arena": {
    "worktreeBaseDir": "~/.qwen/arena",
    "maxRoundsPerAgent": 50,
    "timeoutSeconds": 600
  }
}
```

| 設定                      | 説明                               | デフォルト値         |
| :------------------------ | :--------------------------------- | :-------------- |
| `arena.worktreeBaseDir`   | Arena ワークツリーのベースディレクトリ | `~/.qwen/arena` |
| `arena.maxRoundsPerAgent` | エージェントあたりの最大推論ラウンド数 | `50`            |
| `arena.timeoutSeconds`    | 各エージェントのタイムアウト（秒）  | `600`           |

## ベストプラクティス

### 互いを補完するモデルを選ぶ

Arena は意味のある異なる強みを持つモデルを比較する場合に最も価値があります。例:

```
/arena --models qwen3.5-plus,glm-5,kimi-k2.5 "Optimize the database query layer"
```

同じモデルファミリーの3つのバージョンを比較しても、異なるプロバイダー間での比較ほどの洞察は得られません。

### タスクを自己完結型に保つ

Arena エージェントは通信なしで独立して動作します。タスクはやり取りなしにプロンプトで完全に記述できる必要があります:

**良い例**: "Refactor the payment module to use the strategy pattern. Update all tests."

**効果的でない例**: "Let's discuss how to improve the payment module" — これは会話から恩恵を受けるものであり、単一セッションの方が適しています。

### エージェント数を制限する

最大5つのエージェントを同時に実行できます。実際には、2〜3つのエージェントが比較価値とリソースコストのバランスが最適です。エージェントが多いほど:

- トークンコストが高くなる（各エージェントが独自のコンテキストウィンドウを持つ）
- 合計実行時間が長くなる
- 比較する結果が増える

2〜3つから始めて、比較の価値が正当化される場合のみスケールアップしましょう。

### 影響の大きい決定に Arena を使う

Arena は複数のモデルを実行する価値があるほど重要な場面で威力を発揮します:

- 新しいモジュールのアーキテクチャの選択
- 複雑なリファクタリングのアプローチの選択
- 複数の観点からの重要なバグ修正の検証

変数名の変更や設定ファイルの更新のようなルーティンな変更には、単一セッションの方が速く安価です。

## トラブルシューティング

### エージェントが起動しない

- `--models` の各モデルが有効な API 認証情報で適切に設定されているか確認する
- 作業ディレクトリが Git リポジトリであることを確認する（ワークツリーには Git が必要）
- ワークツリーベースディレクトリ（デフォルトは `~/.qwen/arena/`）への書き込みアクセスがあることを確認する

### ワークツリーの作成に失敗する

- `git worktree list` を実行して以前のセッションで残った古いワークツリーを確認する
- `git worktree prune` で古いワークツリーをクリーンアップする
- Git バージョンがワークツリーをサポートしているか確認する（`git --version`、Git 2.5+ が必要）

### エージェントが時間がかかりすぎる

- タイムアウトを増やす: settings で `arena.timeoutSeconds` を設定する
- タスクの複雑さを減らす — Arena タスクは集中した明確なものにすべきです
- エージェントが多すぎるラウンドを費やしている場合は `arena.maxRoundsPerAgent` を下げる

### 勝者の適用に失敗する

- メインの作業ディレクトリに競合する可能性のある未コミットの変更がないか確認する
- 差分はパッチとして適用されます — セッション中に作業ディレクトリが変更された場合、マージの競合が発生する可能性があります

## 制限事項

Agent Arena は実験的です。現在の制限事項:

- **インプロセスモードのみ**: tmux または iTerm2 による分割ペイン表示はまだ利用できません。すべてのエージェントはタブ切り替えで単一のターミナルウィンドウ内で動作します。
- **選択前の差分プレビューなし**: 各エージェントの会話履歴は確認できますが、勝者を選ぶ前に解決策をサイドバイサイドで比較するための統合差分ビューアはありません。
- **ワークツリーの保持なし**: ワークツリーは選択後に常にクリーンアップされます。さらなる検査のために保持するオプションはありません。
- **セッションの再開不可**: Arena セッションは終了後に再開できません。セッション途中でターミナルを閉じた場合、ワークツリーはディスクに残り、`git worktree prune` で手動クリーンアップが必要です。
- **最大5エージェント**: 5つの同時エージェントのハードリミットは変更できません。
- **Git リポジトリが必要**: Arena はワークツリー分離のために Git リポジトリが必要です。Git 以外のディレクトリでは使用できません。

## 他のマルチエージェントモードとの比較

Agent Arena は Qwen Code で計画されているいくつかのマルチエージェントモードの一つです。**Agent Team** と **Agent Swarm** はまだ実装されていません — 以下の表は参考として意図された設計を説明します。

|                   | **Agent Arena**                                        | **Agent Team**（計画中）                           | **Agent Swarm**（計画中）                                |
| :---------------- | :----------------------------------------------------- | :------------------------------------------------- | :------------------------------------------------------- |
| **目標**          | 競争的: _同じ_ タスクに対する最良の解決策を見つける | 協調的: _異なる_ 側面を一緒に取り組む | バッチ並列: 大量タスクのためにワーカーを動的にスポーン |
| **エージェント**        | 設定済みモデルが独立して競争            | チームメンバーが割り当てられたロールで協力          | ワーカーはオンザフライでスポーンされ、完了時に破棄      |
| **通信** | エージェント間通信なし                           | ダイレクトなピアツーピアメッセージング                      | 一方向: 結果が親によって集約                    |
| **分離**     | 完全: 独立した Git ワークツリー                           | 共有タスクリストを持つ独立したセッション         | ワーカーごとの軽量なエフェメラルコンテキスト                 |
| **出力**        | 選択された一つの解決策がワークスペースに適用される             | 複数の観点からの総合的な結果     | 並列処理からの集約結果              |
| **最適な用途**      | ベンチマーク、モデルアプローチ間の選択        | リサーチ、複雑な協力作業、クロスレイヤーの作業  | バッチ操作、データ処理、マップリデュースタスク      |

## 次のステップ

並列および委譲作業の関連アプローチを探る:

- **軽量な委譲**: [サブエージェント](./sub-agents.md)はセッション内で集中したサブタスクを処理します — モデルの比較が不要な場合に適しています
- **手動並列セッション**: [Git ワークツリー](https://git-scm.com/docs/git-worktree)を使用して別々のターミナルで複数の Qwen Code セッションを手動で実行し、完全な手動制御を行う

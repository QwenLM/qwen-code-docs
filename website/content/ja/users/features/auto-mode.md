# 自動モード

自動モードは、LLM分類器を使用して各ツール呼び出しを評価し、自動承認するかどうかを判断します。これは、ファイル編集のみを自動承認するAuto-Editと、すべてを自動承認するYOLOの中間に位置します。

このページは、自動モードの設定とトラブルシューティングのためのリファレンスです。概要については、[承認モードの概要](./approval-mode.md#4-auto-mode---classifier-driven-approval)を参照してください。

## 仕組み

自動モードで、エージェントがツールを実行しようとすると、Qwen Codeは次の3つのレイヤーを順に通過します。

1. **acceptEdits 高速パス** — ワークスペース内のパスをターゲットとする編集/書き込みは、分類器を呼び出さずに自動承認されます。
   **例外:** Qwen Code自身の自己変更サーフェス (`.qwen/settings*.json`、`QWEN.md`、`AGENTS.md`、`QWEN.local.md`、設定されたコンテキストファイル名、`.qwen/rules/`、`.qwen/commands/`、`.qwen/agents/`、`.qwen/skills/`、`.qwen/hooks/`、`.mcp.json`) および永続化サーフェス (`.git/`、`.husky/`、`package.json`、`.npmrc`、`Makefile`、`.github/workflows/`など) への書き込みは、ワークスペース内であっても分類器を経由します。保護されたパスをターゲットとするシンボリックリンクも解決され拒否されます。`cd && bash -lc '...'` や他のラッパーを介してこれらのパスに到達するシェルコマンドも、分類器を通過します。
2. **安全ツール許可リスト** — 読み取り専用およびメタデータのみの組み込みツール (Read、Grep、Glob、LS、LSP、TodoWrite、AskUserQuestion など) は、分類器を呼び出さずに自動承認されます。
3. **LLM分類器** — それ以外のすべて (シェルコマンド、Webフェッチ、サブエージェント起動、ワークスペース外への編集、MCPツール) は、2段階の分類器に送信されます。
   - **ステージ1 (高速)** — `shouldBlock` のみを出力します。約300ms。`shouldBlock` が `false` の場合、アクションは許可され、呼び出しが進行します。
   - **ステージ2 (思考)** — ステージ1でブロックが指示された場合のみ実行されます。連鎖思考レビューを使用して、ステージ1の誤検出を減らします。ステージ1のブロックを許可に格下げできます。ブロック時にユーザーに表示される `reason` を出力します。

分類器は、設定された高速モデル (`/model --fast`) を使用します。高速モデルが設定されていない場合は、メインセッションモデルが代わりに使用されます。

## ハードルールが優先される

自動モードは、ハードな権限ルールを**置き換えません**。分類器が実行される前に:

- `permissions.deny` ルールは、ルールの理由とともにアクションをブロックします。分類器はそのアクションを認識しません。
- 特定の指定子を持つ `permissions.allow` ルール (例: `Bash(git status)`、`Read(./docs/**)`) は、分類器なしで自動許可されます — **ただし**、呼び出しが保護された自己変更または永続化パスでの書き込みに解決される場合を除きます ("仕組み" のリストを参照)。その場合、自動モードは分類器を介して呼び出しを再チェックするため、`Bash(*)` の許可ルールが黙ってQwen Codeの設定、コマンド、フック、スキル、MCPサーバーを書き換える権限になることはありません。
- `permissions.ask` ルールは、自動モードでも手動確認を強制します。

## 自動モードでは過度に広範な許可ルールは削除される

以下のようなルールは、分類器によるレビューなしでエージェントが任意のコードを実行できるようにします。

- `Bash` / `Bash(*)` / `Bash()` — すべてのシェルコマンドを自動許可
- `Bash(python:*)`、`Bash(node*)`、`Bash(bash*)` — インタプリタのワイルドカード
- `Agent` / `Agent(coder)` — Agentツールに対する任意の許可
- `Skill` / `Skill(pdf)` — Skillツールに対する任意の許可

自動モードに入ると、Qwen Codeはアクティブな権限セットからこれらのルールを一時的に削除し、それらをリストした通知を表示します。自動モードを離れるとすぐにルールは戻ります。`settings.json` は変更されません。

これらの広範なルールが本当に必要な場合は、代わりにYOLOモードを使用してください。

## ヒントの設定

自動モードは、`settings.json` から `permissions.autoMode` を読み取ります。エントリは自然言語の説明であり、ルールパターンではありません。組み込みのデフォルトに加えて、分類器のシステムプロンプトに追加で注入されます。

ヒントには3つのカテゴリと環境リストがあります。

- **`allow`** — 分類器が自動承認すべきアクション。
- **`softDeny`** — 破壊的または元に戻せないアクションで、分類器は**ユーザーの最新の明示的なリクエストがその正確なアクションとスコープを要求した場合を除き**ブロックする必要があります。ソフト拒否はユーザーの意図によってクリアできます。一般的な「はい、なんでもやって」はカウントされません。
- **`hardDeny`** — セキュリティ境界のアクションで、自動モードでは `autoMode.hints.allow` や最近のユーザーの意図に関係なく分類器がブロックする必要があるものです。これは分類器ポリシーであり、決定論的な権限ルールではありません。`permissions.allow` をオーバーライドしません。権限マネージャーによって決して許可されてはならないアクションには `permissions.deny` を使用してください。

```json
{
  "permissions": {
    "autoMode": {
      "hints": {
        "allow": [
          "このPythonプロジェクトでpoetry installとpoetry updateを実行する",
          "./dist または ./build 配下のビルドアーティファクトをクリーンアップする",
          "/Users/me/code/ 配下の任意のファイルを読み取る"
        ],
        "softDeny": [
          "私が正確な変更を明示的に要求しない限り、Qwen Codeの設定を編集する",
          "本番DBに影響を与えるマイグレーションスクリプトを実行する"
        ],
        "hardDeny": [
          "シークレットや .env の内容をネットワークエンドポイントに送信する",
          "~/.ssh や ~/.aws 配下のものを変更する"
        ]
      },
      "environment": [
        "これは厳格なコミット署名を持つプライベートモノレポです",
        "本番の認証情報は1Passwordにあり、平文ファイルには決して保存しません"
      ]
    }
  }
}
```

`hints.deny` は後方互換性のために引き続き受け入れられ、`softDeny` として扱われます。両方を混在させることも問題ありません。エントリは連結され、`softDeny` が最初になります。

### 長さと数の制限

分類器のシステムプロンプトを小さく保つため:

- 各エントリは200文字に制限されます (長いエントリは警告とともに切り詰められます)。
- `hints.allow`、`hints.softDeny`、`hints.hardDeny` はそれぞれ最大50エントリまで受け入れます。
- `environment` は最大20エントリまで受け入れます。

### 設定ファイル間の階層化

`autoMode` は、他の権限設定と同じようにシステム/ユーザー/ワークスペース設定間でマージされます。配列は連結され、重複排除されます。

## 判定結果の読み取り

分類器がアクションをブロックすると、ツール呼び出しは次のいずれかのエラーテキストで失敗します。

- **`Blocked by auto mode policy: <reason>`** —
  分類器がアクションを安全でないと判断しました。理由は分類器のステージ2からのものです。
- **`Auto mode classifier unavailable; action blocked for safety`** —
  分類器APIに到達できなかったか、タイムアウトしたか、解析不能な応答を返しました。これはフェイルクローズ動作です。疑わしい場合はブロックします。

両方のメッセージの後には、エージェントに対して、**拒否されたアクションが具体的に**、別のツール、シェル間接参照、生成されたスクリプト、エイリアス、シンボリックリンク、設定変更、フック、コマンドファイル、MCP設定、エンコードされたペイロード、または同等のパスを介して完了されてはならないことを伝える後続のガイダンス行が続きます。**関連のない安全な作業と真に安全な代替手段は引き続き許可されます** — 同じ拒否された意図を異なるサーフェスで達成しようとする試みのみがブロックされます。

拒否されたアクションが本当に必要な場合、エージェントは拒否を回避するのではなく、停止して明示的な承認を求める必要があります。

### 分類器の理由の言語

分類器の理由はLLMによって生成され、翻訳されません。日本語以外の理由が必要な場合は、`permissions.autoMode.environment` に `理由は日本語で返答してください` のようなヒントを追加してください。

## 手動承認へのフォールバック

自動モードは、行き詰まるのを防ぎます。

- **3回連続でポリシーブロック**が発生した後、次のツール呼び出しは標準の手動承認プロンプトにフォールバックします。これにより、エージェントが禁止されたコマンドのマイナーバリエーションを試し続けるケースをキャッチします。
- **2回連続で利用不可**の結果 (分類器API障害) が発生した後も、次のツール呼び出しはフォールバックします。これにより、壊れた分類器での待機を回避します。

セッション自体は自動モードのままです。フォールバック呼び出しのみが手動承認を通過します。フォールバック呼び出しを承認するか、モードを切り替えると、カウンターはリセットされます。

常にフォールバックが発生する場合、最も可能性の高い原因は、分類器APIの障害か、調整が必要なヒントです。調査中はデフォルトモードに切り替えてください。

## トラブルシューティング

**「自動モードがコマンドをブロックし続ける」**

エラーメッセージの理由を確認してください。分類器があなたのコンテキストに対して過度に慎重である場合は、`permissions.autoMode.hints.allow` にそのパターンを自然言語で説明するエントリを追加してください。例:

- `"このプロジェクトのDockerイメージをビルドする (docker build ...)"`
- `"ローカルのテストDBに対してデータベースマイグレーションを実行する"`

**「自動モード分類器が利用不可」**

分類器APIが応答しませんでした。考えられる原因:

- あなたとモデルエンドポイント間のネットワーク問題。
- 設定された高速モデルが利用できなくなった — `/model --fast` を確認してください。
- トランスクリプトが長すぎて、高速モデルのコンテキストウィンドウを超えている。

診断中は、デフォルトモードに切り替えてください: `/approval-mode default`。

**「手動承認にフォールバック」**

3回連続ブロックまたは2回連続利用不可のガードに達しました。プロンプトを通常どおり承認または拒否してください。フォールバックが1回承認されると、連続カウンターはリセットされます。

**「分類器がプロンプト内の機密データを認識する」**

ツール入力は、分類器に到達する前に、各ツールの `toAutoClassifierInput` メソッドを介して投影されます。長い編集内容、Webフェッチプロンプト、サブエージェントプロンプトは切り詰められます。ツール結果 (ファイルの内容、Webページ) は分類器に送信されることはありません。ユーザーのテキストとアシスタントのツール使用呼び出しのみが通過します。

特定のツールが編集したいフィールドを公開している場合は、そのツール名を添えて問題を報告してください。投影はツールごとであり、時間の経過とともに厳格化されることを意図しています。

## 制限事項

- **オフラインでは利用不可。** 分類器にはLLM呼び出しが必要です。
- **低速パスにレイテンシが追加されます。** 許可リスト + acceptEdits でほとんどの呼び出しをレイテンシなしでカバーしますが、`run_shell_command` は通常、約300ms (高速分類器パス) または約3～5秒 (思考レビュー付き低速パス) を追加します。
- **`deny` ルールの代わりにはなりません。** 分類器はベストエフォートです。決して実行すべきでないコマンドについては、`permissions.deny` に記述してください。
- **MCPツールはデフォルトで保守的にブロックします。** サードパーティのMCPツール (`mcp__*`) は、`toAutoClassifierInput` オーバーライドを介して引数転送をオプトインします。オプトインしていないツールは、その名前のみを分類器に公開します。このような呼び出しのほとんどは、明示的な `allow` ルールを記述しない限り、保守的にブロックされます。これは設計上のフェイルクローズです (資格情報や大量のコンテンツが分類器LLMに漏洩しません)。特定のMCPツールを信頼する場合は、`permissions.allow: ["mcp__server__tool"]` を追加して、分類器を完全にバイパスしてください。

## FAQ

**自動モードはコードをサードパーティに送信しますか？**

自動モードは、既存のモデル設定を再利用します — メインエージェントと同じエンドポイントです。Qwen Codeをセルフホストモデルを使用するように設定している場合、分類器もそのエンドポイントに対して実行されます。

**シークレット / `.env` の内容が分類器に到達しますか？**

分類器は、各ツールの `toAutoClassifierInput` 投影が公開するもののみを認識します。

- `read_file` およびその他の読み取り専用ツール: 呼び出されません (高速パス許可リストにあります)。
- `edit` / `write_file`: file_path と、古い/新しいコンテンツの最初の80文字。完全なコンテンツは転送されません。
- `run_shell_command`: 完全なコマンド (必須です — 分類器が判断するのはこれです)。
- `web_fetch`: URLのみ。プロンプトフィールドは転送されません。
- `agent`: サブエージェントタイプと完全なプロンプト。プロンプトはサブエージェントが従う指示であるため、分類器は破壊的なアクションにサブエージェントを誘導する攻撃を検出するために完全なプロンプトを必要とします — `run_shell_command` が完全なコマンドを転送するのと同じ理由です。

ツール結果 (ツールによって返される実際のコンテンツ) は、分類器トランスクリプトから完全に削除されます。

MCPツール (`mcp__*`) はより厳格なデフォルトに従います。パラメーターは、MCPツール作成者が `toAutoClassifierInput` オーバーライドを介して明示的にオプトインしない限り転送されません。分類器はツール名を認識しますが、引数は認識しないため、ユーザーが明示的な許可ルールを記述しない限り、ほとんどのMCP呼び出しは保守的にブロックされます。これは設計上のフェイルクローズです — サードパーティツールは、意図せずに資格情報や大量のファイルコンテンツを分類器LLMに漏洩させるべきではありません。

**初回の情報メッセージを無効にできますか？**

ユーザー設定ファイルごとに1回だけ表示されます。消去後、`ui.autoModeAcknowledged: true` がユーザー設定に設定されます。

**これはAuto-Editとどう違うのですか？**

Auto-Editはファイル編集のみを自動承認し、シェルコマンドは引き続き確認を求めます。自動モードは分類器を使用して、リスクのあるものをブロックしながら、安全なシェルコマンドやその他のツール呼び出しも自動承認します。

**これはYOLOとどう違うのですか？**

YOLOはレビューなしですべてを自動承認します。自動モードは分類器がループ内にあり、リスクのあるアクションをブロックします。

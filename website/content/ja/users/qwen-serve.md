# デーモンモード（`qwen serve`）

Qwen Code をローカルの HTTP デーモンとして実行することで、複数のクライアント（IDE プラグイン、Web UI、CI スクリプト、カスタム CLI）が個別にサブプロセスを生成する代わりに、HTTP + Server-Sent Events を介して 1 つのエージェントセッションを共有できるようになります。

> **🚧 v0.16-alpha**: `qwen serve` は v0.16-alpha で npm に初めて出荷され、**テキストのみのチャット / コーディング** と **ローカルのみのデプロイ** を提供します。プロンプトパス上の画像 / ファイル添付、コンテナ化デプロイ（Docker / k8s / nginx リバースプロキシ）、リモート / マルチデーモンの堅牢化は、エンタープライズパイロットが確定した後のフォローアップパッチで対応予定です。延期された全リストは [v0.16-alpha の既知の制限](#v016-alphaの既知の制限) を参照してください。

> **ステータス:** ステージ 1（実験的）。プロトコルサーフェスは issue [#3803](https://github.com/QwenLM/qwen-code/issues/3803) の §04 ルートテーブルで固定されています。ステージ 1.5（`qwen --serve` フラグ — TUI が同じ HTTP サーバーをホスト）およびステージ 2（インプロセスリファクタ + `mDNS`/OpenAPI/WebSocket/Prometheus の仕上げ）は直後に予定されています。
>
> **スコープの正直さ:** ステージ 1 は、**プロトコルサーフェスに対してクライアントをプロトタイピングする開発者** と、**ローカルな単一ユーザー / 小規模チームのコラボレーション** を対象としています。本番グレードのマルチクライアント / 長時間実行 / ネットワーク不安定なワークロード（モバイルコンパニオン、1000 以上のチャットに達する IM ボット）には、このリリースには含まれていないステージ 1.5+ の保証が必要です。完全なギャップリストは [ステージ 1.5+ ランタイム保証](#ステージ-15-ランタイム保証) 、収束ロードマップは #3803 を参照してください。

## 提供される機能

- **組み込みの Web シェル UI** — `qwen serve` は、ブラウザベースの Web シェルをそのルート（`http://127.0.0.1:4170/`）でそのまま提供します。`qwen serve --open` を実行すると、自動的にブラウザで起動します。API と同じオリジンで提供されるため、別のポートやリバースプロキシは不要です。API のみのデーモンにするには `--no-web` を指定します。
- **1 つのエージェントプロセス、多数のクライアント** — デフォルトの `sessionScope: 'single'` では、デーモンに接続するすべてのクライアントが 1 つの ACP セッションを共有します。同じ会話、同じファイル差分、同じ権限プロンプトで、ライブなクロスクライアントコラボレーションが可能です。
- **再接続可能なストリーミング** — `Last-Event-ID` を使用した SSE 再接続により、クライアントは（リングのリプレイウィンドウ内で）中断したところから正確に再接続できます。
- **ファーストレスポンダー権限** — エージェントがツールを実行するための権限を要求すると、接続されているすべてのクライアントがリクエストを確認します。最初に応答したクライアントが勝ちます。
- **1 つのデーモン、1 つのワークスペース** — 各 `qwen serve` プロセスは、起動時に（#3803 §02 に従い）正確に 1 つのワークスペースにバインドされます。マルチワークスペースのデプロイでは、ワークスペースごとに 1 つのデーモンを別々のポート（またはオーケストレーターの背後）で実行します。
- **リモートランタイムコントロール**（[#4175](https://github.com/QwenLM/qwen-code/issues/4175) PR 17）— セッションの承認モードの変更（`POST /session/:id/approval-mode`）、ワークスペースごとのツールの有効/無効切り替え（`POST /workspace/tools/:name/enable`）、空の `QWEN.md` のスキャフォールディング（`POST /workspace/init`、機械的なみ — モデルは呼び出しません。AI による入力には `POST /session/:id/prompt` を続けて使用）、予算事前チェック付きの単一 MCP サーバーの再起動（`POST /workspace/mcp/:server/restart`）、またはデーモン再起動なしでの MCP サーバーの追加/削除（`POST /workspace/mcp/servers`、`DELETE /workspace/mcp/servers/:name`）が可能です。すべて厳格にゲートされています。最初に `--token` を設定してください。
- **セッションの要約**（[#4175](https://github.com/QwenLM/qwen-code/issues/4175) フォローアップ）— アクティブなセッションの「どこで中断したか」を一文で要約したものを取得できます（`POST /session/:id/recap`）。コアの `generateSessionRecap` を、高速モデルに対するサイドクエリとしてラップします。メインのチャット履歴や SSE ストリームを汚染しません。非厳格ゲート（`/prompt` と同じ姿勢）。SDK ヘルパー `client.recapSession(sessionId)`。
  - **既知の制限 — トークンコストの増大:** このルートは純粋なコストエンドポイントです（各呼び出しは LLM サイドクエリであり、状態の利点はありません）。また、v1 ではデーモンにルートごとのレート制限がありません。トークンなしのループバックデフォルトでは、バグがあるか悪意のあるローカルクライアントがこれをスパムしてトークンを消費する可能性があります。デーモンを公開する前に、共有開発ホストで `--token`（およびオプションで `--require-auth`）を設定してください。
  - **同時 recap の安全性:** 同じセッションに対する 2 つの同時の `/recap` 呼び出しは、2 つの独立したサイドクエリを実行します。`generateSessionRecap` は、`GeminiClient.getChat().getHistory()` を介してチャット履歴のスナップショットを読み取り、それを別の `BaseLlmClient.generateText` 呼び出し（`runSideQuery` 経由）に渡します。セッションの `GeminiChat` に追加したり変更したりすることはありません。複数のクライアントから調整なしで安全に呼び出すことができます。

## v0.16-alpha の既知の制限

`qwen serve` の最初の npm リリース（v0.16-alpha）は意図的に狭い範囲です。開発者が自分のマシンでデーモンを実行するための、テキストのみのチャット / コーディングです。以下のリストは延期されたサーフェスを明確にし、採用者が計画を立てられるようにするためのものです。ここに記載されているすべての機能は、v0.16.x パッチロードマップまたは近い将来のフォローアップリリースで対応予定です。

**プロダクトサーフェス — テキストのみ:**

- ✅ テキストプロンプトとテキスト応答（チャット、コーディング、ツール呼び出し、MCP 統合）
- ❌ **プロンプトパス上の画像 / ファイル添付** — `MessageEmitter` は現在テキストのみをレンダリングします。画像ニーズのあるアルファターゲットが確定したときに、マルチモーダルエコーが追加されます（#4175 chiga0 #27 P0 項目）
- ❌ **ストリーミングアップロード** — マルチモーダルと同じゲート

**デプロイサーフェス — ローカルのみ:**

- ✅ ループバック（`127.0.0.1`、デフォルト）— 認証不要。開発ワークステーションに適しています。
- ✅ `systemd` / `launchd` / `nohup &` / `tmux` によるローカル起動 — [ローカル起動テンプレート](./qwen-serve-deploy-local.md) を参照
- ✅ `QWEN_SERVER_TOKEN` 環境変数による独自のベアラートークンの持ち込み（設定方法は[認証](#認証) を参照）
- ❌ **コンテナ化デプロイ** — Docker / Compose / Kubernetes / nginx リバースプロキシ + TLS 終端は v0.16-alpha には含まれません。エンタープライズパイロットが確定したら v0.16.x に延期（そうしないと誰も検証しないため腐る可能性があります）。
- ❌ **1 ホスト上の複数デーモンの調整** — `1 デーモン = 1 ワークスペース × N セッション` が強制されます。クロスホストフェデレーション、インスタンスパストークンキーイング、期限切れトークンのクリーンアップは v0.16.x に延期。
- ❌ **自動生成デーモントークン** — アルファ版は BYO トークン（`openssl rand -hex 32` で簡単に生成可能）。自動生成 + トークンストア基盤は v0.16.x に延期。

**堅牢化 — ローカル単一ユーザー向け最小限:**

- ✅ 起動時セキュリティゲート（トークンなしでの非ループバックバインドを拒否、[PR 15 / #4236](https://github.com/QwenLM/qwen-code/pull/4236)）
- ✅ ミューテーションルートの認証ゲート、セッションスコープの権限ルーティング（Wave 4 PR）
- ✅ MCP ガードレール + マルチクライアント権限調整（F2 / F3）
- ✅ **プロンプト絶対デッドライン + SSE ライターアイドルタイムアウト** — `--prompt-deadline-ms` と `--writer-idle-timeout-ms` でオプトイン。有効時に `prompt_absolute_deadline` と `writer_idle_timeout` でアドバタイズされます。
- ✅ **HTTP レート制限** — `--rate-limit` と階層別しきい値でオプトイン。有効時に `rate_limit` でアドバタイズされます。
- ⏸️ **Prometheus メトリクス + ロードテストハーネス** — 30～50 のアクティブセッションが実際のターゲットになった際、v0.17 F4 Phase-1 スケール計装に延期。
- ⏸️ **`--max-body-size` CLI フラグ** — デーモンはデフォルトで `express.json({ limit: '10mb' })` を強制します。これはテキストのみのプロンプト（モデルのコンテキストウィンドウは 10 MiB の文字数をはるかに下回ります）を十分にカバーします。v0.16.x でフラグにより調整可能になります。

「ステージ 1 で修正しないもの」の詳細な列挙（シングルホストのセッション状態変更モデル + 1 つの ACP 子プロセスを共有する N 並列セッション）については、以下の[ステージ 1 スコープ境界](#ステージ-1のスコープ境界--ステージ-15で修正しないもの) を参照してください。

## クイックスタート

### 1. デーモンを起動する（ループバック、認証なし）

```bash
cd your-project/
qwen serve
# → qwen serve listening on http://127.0.0.1:4170 (mode=http-bridge, workspace=/path/to/your-project)
# → qwen serve: bearer auth disabled (loopback default). Set QWEN_SERVER_TOKEN to enable.
```

デフォルトのバインドは `127.0.0.1:4170` です。ベアラー認証は、ローカル開発が「そのまま動く」ように、ループバックでは **オフ** になっています。デーモンはカレントワーキングディレクトリにバインドされます。`--workspace /path/to/dir` で上書きできます。

**Web シェル UI を開く。** `http://127.0.0.1:4170/` にアクセスするか、デーモンを `qwen serve --open` で起動すると自動的に開き、フルブラウザターミナル（チャット、差分、ツール呼び出し、権限プロンプト）が利用できます。UI は API と同じオリジンのデーモンルートで提供されます。このガイドの残りの部分では、HTTP を直接使用して API をスクリプト化できるように説明します。

### 2. 健全性確認

```bash
curl http://127.0.0.1:4170/health
# → {"status":"ok"}

curl http://127.0.0.1:4170/capabilities
# → {"v":1,"mode":"http-bridge","features":["health","daemon_status","capabilities","session_create",...],"workspaceCwd":"/path/to/your-project"}

curl http://127.0.0.1:4170/daemon/status
# → {"v":1,"detail":"summary","status":"ok","runtime":{...}}
```

`workspaceCwd` フィールドはバインドされたワークスペースを公開するため、クライアントは事前チェックを行い、`POST /session` で `cwd` を省略できます。
`limits.maxPendingPromptsPerSession` フィールドは、アクティブなセッションあたりのプロンプト受付上限を通知します。`null` は上限が無効であることを意味します。

デーモンは、クライアント UI やオペレーター向けに読み取り専用のランタイムスナップショットも公開します: `GET /daemon/status`、`GET /workspace/mcp`、
`GET /workspace/skills`、`GET /workspace/providers`、`GET /workspace/env`、
`GET /workspace/preflight`、
`GET /session/:id/status`、`GET /session/:id/context`、
`GET /session/:id/supported-commands`、
`GET /session/:id/tasks`、`GET /session/:id/lsp`。

`GET /session/:id/status` は、単一セッションのライブブリッジサマリー（`sessionId`、`workspaceCwd`、`createdAt`、オプションの `displayName`、`clientCount`、`hasActivePrompt`）を返します。デーモンがその ID のライブセッションを保持している場合は `200` とサマリーを返し、そうでない場合は `404`（ボディ `{ "error": …, "sessionId": … }`）を返します。これを使用して、既知のセッションがまだ実行中か（`hasActivePrompt`）、または何人のクライアントが接続しているか（`clientCount`）を、ページ分割されたセッションリスト全体を取得してスキャンせずにポーリングできます。

```bash
curl http://127.0.0.1:4170/session/$SESSION_ID/status
# → {"sessionId":"…","workspaceCwd":"…","createdAt":"…","clientCount":1,"hasActivePrompt":false}
```

これは生のライブセッションビューであるため、`clientCount` と `hasActivePrompt` は `GET /workspace/:id/sessions` の対応するエントリと一致しますが、2 つのルートはバイト単位で同一ではありません。リストエンドポイントは各項目に永続化されたセッションストアデータ（`createdAt` は永続化された最初のプロンプト時刻、`updatedAt` と保存されたタイトルまたは最初のプロンプトから派生した `displayName` を追加）を追加します。`/status` は代わりにライブセッション自身の `createdAt` を報告し、`updatedAt` を省略し、ライブセッションに設定されている場合にのみ `displayName` を返します。

`GET /session/:id/lsp` は、セッションごとの構造化された LSP ステータスを返します。`--experimental-lsp` を指定してデーモンを起動すると、生成されたエージェントセッションで LSP が有効になります。それ以外の場合、ルートは `enabled: false` とサーバーなしを返します。

`GET /daemon/status` は統合トラブルシューティングスナップショットです。デフォルトの `detail=summary` はメモリ内のデーモン状態（セッション、権限、SSE/ACP トランスポート数、レート制限拒否、プロセスメモリ、解決済み制限）のみを読み取り、ACP 子プロセスを起動しません。問題を能動的に調査している場合は、`GET /daemon/status?detail=full` を使用して、セッションごとの診断、ACP 接続詳細、認証デバイスフロー数、ワークスペースステータスセクションを取得します。

`GET /workspace/mcp`、`GET /workspace/skills`、`GET /workspace/providers` はライブ ACP ランタイムを報告し、アイドル時には ACP 子プロセスを起動しません。アイドル状態のデーモンは `initialized: false` と空のスナップショットを返します。セッションがアクティブになると、`initialized: true` に切り替わり実際の状態を公開します。

`GET /workspace/env` と `GET /workspace/preflight` は、ACP 状態に関係なく常に `initialized: true` で応答します。`env` は ACP を参照しません（デーモンプロセス情報のみ）。`preflight` はデーモンレベルのセルを `process.*` から応答し、ACP レベルのセルについては、子プロセスがアイドル状態のときに `status: 'not_started'` のプレースホルダーを発行します。

`GET /workspace/env` は、デーモンプロセスのランタイム、プラットフォーム、サンドボックス、プロキシ、およびホワイトリストに登録されたシークレット環境変数（`OPENAI_API_KEY` など）の **存在（値ではありません）** を報告します。プロキシ URL は認証情報が取り除かれ、ワイヤーに送信される前に `host:port` に縮小されます。このルートは常にデーモンプロセスから直接応答し、ACP 子プロセスを生成しません。

`GET /workspace/preflight` は、準備完了チェックのリストを返します。**デーモンレベルのセル**（Node バージョン、CLI エントリ、ワークスペースディレクトリ、ripgrep、git、npm）は常にレンダリングされます。**ACP レベルのセル**（認証、MCP ディスカバリ、スキル、プロバイダ、ツールレジストリ、エグレス）はライブ ACP 子プロセスが必要です。デーモンがアイドル状態の場合、それらを埋めるために ACP を生成するのではなく、`status: 'not_started'` のプレースホルダーを発行します。障害は閉じた `errorKind` 列挙型（`missing_binary`、`auth_env_error`、`init_timeout`、`protocol_error`、`missing_file`、`parse_error`、`blocked_egress`）にマッピングされるため、クライアント UI は構造化された修正をレンダリングできます。

デーモンはワークスペースファイルヘルパーも公開します:

- `GET /file` はテキストファイルを読み取り、raw バイトの `sha256:<hex>` ハッシュを返します。
- `GET /file/bytes` は境界付きの raw バイトウィンドウを読み取り、base64 コンテンツを返します。
- `POST /file/write` はテキストファイルを作成または置き換えます。
- `POST /file/edit` は 1 つの正確なテキスト置換を適用します。

書き込み/編集は **厳格なミューテーションルート** です。ループバックであっても、設定されたベアラートークンが必要です。設定されていない場合は `token_required` を返します。置換と編集には、`GET /file`（またはフルウィンドウの `GET /file/bytes`）からの最新の `expectedHash` が必要です。`create` は上書きしません。無視されたパスへの明示的な書き込みは許可されますが、監査されます。バイナリ書き込み、削除/移動/mkdir、再帰的な親ディレクトリ作成はこのサーフェスの一部ではありません。

### 3. セッションを開く

```bash
curl -X POST http://127.0.0.1:4170/session \
  -H 'Content-Type: application/json' \
  -d '{}'
# → {"sessionId":"<uuid>","workspaceCwd":"…","attached":false}
```

`cwd` は省略可能です。ルートはデーモンがバインドされたワークスペースにフォールバックします。バインドされたワークスペースと一致しない `cwd` を送信すると、`400 workspace_mismatch` が返ります（デーモンは正確に 1 つのワークスペースにバインドされています。別のワークスペースには別のデーモンを起動してください）。

2 番目のクライアントが `/session` にポストすると（`cwd` が一致するか null かに関わらず）、`"attached": true` が返ります。これでエージェントを共有することになります。

### 4. イベントストリームを購読する（最初に別のターミナルで）

```bash
SESSION_ID="<step 3 の値>"
curl -N http://127.0.0.1:4170/session/$SESSION_ID/events
# → id: 1
#   event: session_update
#   data: {"id":1,"v":1,"type":"session_update","data":{"sessionUpdate":"agent_message_chunk","content":{"type":"text","text":"…"}}}
```

`data:` 行は **完全なイベントエンベロープ** です — `{id?, v, type, data, originatorClientId?}` — 1 行に JSON 文字列化されています。ACP ペイロード（この例では `sessionUpdate` ブロック）はそのエンベロープ内の `data` の下にあります。SSE レベルの `id:` / `event:` 行は EventSource クライアントの便宜のためのものです。同じ値が JSON エンベロープ内にも現れるため、生の `fetch` コンシューマもそれらを取得できます。

これを **プロンプトを送信する前** に開いてください。SSE リプレイバッファは最後の 8000 イベントを保持するため、遅れて参加したサブスクライバは `Last-Event-ID` で追いつくことができます。ただし、単純な「1 つのプロンプトを監視する」ケースでは、最初に購読してライブストリームを受信するのが最も簡単です。

ストリームは、`session_update`（LLM チャンク、ツール呼び出し、使用量）、`permission_request`（ツールに承認が必要）、`permission_resolved`（誰かが投票した）、`model_switched`、`model_switch_failed`、および終端フレーム `session_died`（エージェント子プロセスがクラッシュ — SSE はその後閉じる）、`client_evicted`（キューがオーバーフロー — SSE はその後閉じる）を発行します。

### 5. プロンプトを送信する（元のターミナルに戻って）

```bash
curl -X POST http://127.0.0.1:4170/session/$SESSION_ID/prompt \
  -H 'Content-Type: application/json' \
  -d '{"prompt":[{"type":"text","text":"src/main.ts は何をしますか？"}]}'
# → {"stopReason":"end_turn"}
```

ステップ 4 の `curl -N` は、フレームが到着するたびに表示します。

## 認証

ループバック以外では、**ベアラートークンを渡す必要があります**:

```bash
export QWEN_SERVER_TOKEN="$(openssl rand -hex 32)"
qwen serve --hostname 0.0.0.0 --port 4170
# → QWEN_SERVER_TOKEN がないと起動を拒否
```

クライアントはすべてのリクエストで `Authorization: Bearer $QWEN_SERVER_TOKEN` を送信します。`/health` は **ループバックバインドでのみ** 免除されます。これにより、ポッド内（デーモンが `127.0.0.1` でリッスンしている）の k8s/Compose の liveness probe が認証情報を必要としなくなります。非ループバックバインド（`--hostname 0.0.0.0` など）では、`/health` も他のすべてのルートと同様にトークンが必要です。そうしないと、攻撃者が任意のアドレスをプローブしてデーモンの存在を確認できるためです。`/capabilities` を使用して、トークンがエンドツーエンドで正しいことを確認してください（常に認証が必要です）。

> **強化されたループバック（`--require-auth`）。** デフォルトのループバックのトークン不要動作は単一ユーザーのラップトップでは問題ありませんが、共有開発ホスト、CI ランナー、またはマルチテナントワークステーションでは危険です。そこでは任意のローカルユーザーが `curl 127.0.0.1:4170` を実行できます。`--require-auth` を渡すと、`127.0.0.1` にバインドされている場合でも、すべてのルート（`/health` や `/capabilities` を含む）でベアラートークンが必須になります。トークンがないと起動に失敗します。このフラグがオンになっている場合、**認証されていない** クライアントは `/capabilities` を読み取って認証が必要であることを発見できません。発見サーフェスは 401 応答ボディそのものです。認証されると、`caps.features.require_auth` タグは認証後確認として機能し、デプロイが強化されていることを示します（監査 / コンプライアンス UI に便利です）:
>
> ```bash
> qwen serve --require-auth --token "$(openssl rand -hex 32)"
> # → /health, /capabilities, /session, … すべてに Authorization: Bearer … が必要
> curl http://127.0.0.1:4170/health
> # → 401
> curl -H "Authorization: Bearer $TOKEN" http://127.0.0.1:4170/capabilities | jq '.features | index("require_auth")'
> # → 13   (または任意のインデックス — 認証後に null 以外であればタグが存在することを意味します)
> ```

```bash
curl -H "Authorization: Bearer $QWEN_SERVER_TOKEN" http://your-host:4170/capabilities
# → {"v":1,"mode":"http-bridge","features":[...],"modelServices":[],"workspaceCwd":"/path/to/your-project"}
# 間違ったトークン → 401
```

トークンの比較は定数時間（SHA-256 + `crypto.timingSafeEqual`）です。401 応答は「ヘッダー欠落」、「スキーム不一致」、「トークン不一致」で統一されているため、サイドチャネルで区別できません。

> **負荷調整用のつまみ。** `--max-sessions` は **新しい子プロセス** の上限です。
> さらに3つの層が負荷を制限します。高並行デプロイメント向けにサイジングする際は、これらをまとめて調整してください。
>
> - **リスナーレベル**: `--max-connections` / `server.maxConnections=256` は生のTCP接続を制限します（低速クライアントへのバックプレッシャー）。
> - **セッションあたりのサブスクライバー数**: EventBusはSSEサブスクライバーをデフォルトでセッションあたり64に制限します。65番目のクライアントは終端 `stream_error` を受け取り、切断されます。
> - **セッションあたりのプロンプト受付数**: `--max-pending-prompts-per-session=5` は1つのセッションで受け付けるキュー待ち＋アクティブなプロンプトを制限します。超過すると `503` と `Retry-After: 5` が返されます。
> - **サブスクライバーあたりのバックログ**: SSEクライアントあたり256フレームのキュー。容量を超えたクライアントは終端 `client_evicted` フレームを受け取り切断されます（1つの低速コンシューマーがデーモンを占有することはありません）。
>
> これらの上限は相互作用します。`--max-sessions × 64 サブスクライバー × 256 フレーム` がEventBus層での最悪のメモリ使用量（転送中）であり、`--max-sessions × --max-pending-prompts-per-session` が受付層で受け付けるプロンプト作業の上限です。デフォルトのサイジングはシングルユーザー/小規模チームの負荷を想定しています。マルチテナントデプロイメントでは段階的に引き上げ（RSSを監視しながら）てください。

> **MCPクライアントのガードレール（issue [#4175](https://github.com/QwenLM/qwen-code/issues/4175) PR 14）。** ワークスペースが `mcpServers` に30のMCPサーバーを宣言している場合、上限を設定しないと30のクライアントが無制限に起動されます。`--mcp-client-budget=N` で生存中のMCPクライアント数を制限し、`--mcp-budget-mode={enforce,warn,off}` で動作を選択します。デフォルトは、予算が設定されている場合 `warn` です（スナップショットに警告が表示されますが、クライアントは拒否されません。実際のファンアウトを計測してから強制に切り替えるのに便利です）。`enforce` モードで拒否されたサーバーは、per-server cellに `disabledReason: 'budget'` が設定され、`budgets[0]` cellには `status: 'error'` + `errorKind: 'budget_exhausted'` が表示されます。スロットの予約はサーバー名で行われ、再接続やディスカバリータイムアウト後も維持されます。拒否されたサーバーが健全なサーバーのスロットを奪うことはありません。
>
> ⚠️ **v1のスコープ: ワークスペース単位ではなく、セッション単位。** デーモン内の各ACPセッションは、独自の `Config`/`McpClientManager` を持ちます（セッションごとに `newSessionConfig` で作成）。予算は生存中のMCPクライアントを **セッションごと** に制限し、ワークスペース全体で集約されるわけではありません。`GET /workspace/mcp` のスナップショットはブートストラップセッションのビューを反映します（cellには `scope: 'session'` が付与され、正直に示されます）。`--mcp-client-budget=10` で5つの同時ACPセッションを実行する場合、デーモン全体で最大50の生存MCPクライアントが存在する可能性があります。上限はセッションごとに保持されます。**Wave 5 PR 23（共有MCPプール）** でワークスペーススコープのマネージャーが導入され、真のワークスペース単位の強制に格上げされます。
>
> ```sh
> qwen serve --mcp-client-budget=10 --mcp-budget-mode=warn
> # 後で、テレメトリーで実際の分布が分かったら:
> qwen serve --mcp-client-budget=10 --mcp-budget-mode=enforce
> ```
>
> これは **claude-code の `MCP_SERVER_CONNECTION_BATCH_SIZE`（起動時の同時実行数を制限）とは異なります**。これらは直交しています。PR 23では実際の共有MCPプール（`budgets[]` 内の `scope: 'workspace'` cell。セッション単位のcellと共存）が追加されます。PR 14 v1は、既存のセッション単位マネージャーに対するインプロセスカウンター＋ソフト強制です。
>
> **プッシュイベント（issue [#4175](https://github.com/QwenLM/qwen-code/issues/4175) PR 14b）。** `GET /session/:id/events` にサブスクライブしているSDKクライアントは、予算のしきい値を超えたときに型付きフレームを受信します。`mcp_budget_warning`（合成イベント。75%を超える上昇ごとに1回発生、37.5%でヒステリシス再アームあり。`mcp_guardrail_events` で通知）と `mcp_child_refused_batch`（`enforce` モードのディスカバリーパスごとに1回まとめて発生。`readResource` からの遅延生成拒否では長さ1）です。`GET /workspace/mcp` のスナップショットは依然として再接続後の状態の信頼できる情報源であり、イベントは変化のエッジです。ポーリングなしでリアルタイムにダッシュボード表示する場合に便利です。

## デフォルトのデプロイメント脅威モデル

- **127.0.0.1 のみ** — ループバックバインド、認証不要。
- **`--hostname 0.0.0.0` にはトークンが必要** — トークンなしでは起動を拒否。
- **`LOOPBACK_BINDS` にIPv6が含まれる** — `::1` と `[::1]` はトークン不要ルールのループバックとみなされる。
- **Hostヘッダー許可リスト** — **ループバック** バインドでは、デーモンは `Host:` が `localhost:port` / `127.0.0.1:port` / `[::1]:port` / `host.docker.internal:port` に一致するか確認し（RFC 7230 §5.4 に従い大文字小文字を区別しない）、DNSリバインディングを防御する。**非ループバックバインド（`--hostname 0.0.0.0`）では意図的にHost許可リストをバイパス** — オペレーターが攻撃対象領域を選択しているため、ベアラートークンゲートが唯一の認証層となる。リバースプロキシ / SNI / クライアント証明書ピン留めはオペレーターの責任であり、デーモンの責任ではない。非ループバックバインドでHostベースの分離が必要な場合は、フロントプロキシでTLSを終端しHostを確認すること。
- **CORSはデフォルトですべてのブラウザOriginを拒否** — `403` JSONを返す。**`--allow-origin <パターン>`**（繰り返し指定可能、T2.4 #4514）を渡して、特定のブラウザOriginを許可する。各値はリテラルの `*`（任意のOrigin — ベアラートークンが設定されていない場合は起動を拒否。ループバックでは `/health` と `/demo` がデフォルトで認証前のまま残るため、完全な強化には `--require-auth` を推奨）または正規のURL Origin（`<scheme>://<host>[:<port>]`、末尾スラッシュ / パス / userinfo なし）。一致したOriginには適切なCORSレスポンスヘッダー（`Access-Control-Allow-Origin: <echoed>`、`Vary: Origin`、標準のメソッド / ヘッダー / max-age、公開された `Retry-After`）が返される。一致しないOriginにはデフォルトの壁と同じエンベロープで403が返される。`caps.features.allow_origin` は条件的に通知されるため、SDK / WebUIクライアントは発行前にデーモンがクロスオリジンヒットを許可するかどうかを事前確認できる。例: `qwen serve --allow-origin http://localhost:3000 --allow-origin http://localhost:5173`。ループバック自己Originヒット（例: `/demo` ページ）は影響を受けない。`--allow-origin` とは関係なく、別のOrigin削除シムが処理する。**`--allow-origin` が設定されていないブラウザWebUI** は、従来と同じStage 1のオプションにフォールバックする：ネイティブシェル（Electron/Tauri）としてパッケージ化して `Origin` ヘッダーが送信されないようにするか、同一オリジンのリバースプロキシをデーモンの前に配置する。
- **生成された `qwen --acp` 子プロセスはデーモンの環境を継承**するが、1つだけ明示的に除去される：`QWEN_SERVER_TOKEN`（デーモン自身のベアラー。エージェントは不要）。それ以外の `OPENAI_API_KEY` / `ANTHROPIC_API_KEY` / `QWEN_*` / `DASHSCOPE_API_KEY` / カスタムの `modelProviders[].envKey` などはすべて通過する。これはエージェントがLLMへの認証にそれらを正当に必要とするためである。**これは意図的であり、サンドボックスではない。** エージェントは同じUIDでシェルツールへのアクセス権を持って実行されるため、`~/.bashrc` / `~/.aws/credentials` / `~/.npmrc` 内のものはプロンプトインジェクションによってアクセス可能である。環境変数の透過はセキュリティ境界ではない。ユーザーを信頼のルートとすることこそが境界である。エージェントに委ねたくない環境内の認証情報を持つIDで `qwen serve` を実行してはならない。
- **サブスクライバーごとに制限されたSSEキュー** — キューをオーバーフローさせた低速クライアントは終端 `client_evicted` フレームを受け取り切断される。1つのスタックしたコンシューマーがデーモンを占有することはない。
- **セッションあたりのプロンプト受付上限** — デフォルトではセッションあたり5つの受付済み未解決プロンプト。バグのあるクライアントが1つのセッションで無制限にプロンプトのプロミスや一時的なSSE待ちをエンキューすることはできない。
- **Graceful shutdown** — SIGINT/SIGTERMはリスナーを閉じる前にエージェント子プロセスをドレインする（子プロセスあたり10秒の期限）。

> ⚠️ **Stage 1の既知のギャップ — パーミッションはデーモングローバルであり、セッション単位ではない（BUy4H）。** `pendingPermissions` はデーモンスコープに存在する。ベアラートークンを持つクライアントは、自身が見える任意のセッションの任意の `requestId` に投票でき（SSE `permission_request` イベントはペイロードにrequestIdを含む）。これは、認証されたすべてのクライアントが同じ人間または信頼するコラボレーターであるというシングルユーザー/小規模チームの信頼モデルの下では許容できる。Stage 1.5では `POST /session/:id/permission/:requestId` + セッションスコープの保留マップ + クライアント単位のID（下流レビューの必須事項#3）に移行する。それまでは、信頼できない関係者と共有するベアラーで `qwen serve` を実行しないこと。
>
> ⚠️ **Stage 1の既知のギャップ — `POST /session/:id/prompt` のボディは10MBに制限されている（BUy4L）。** 画像/PDF/音声を含むマルチモーダルプロンプトが10MBを超えると、ルートロジック実行前のボディパース時に失敗する（ストリーミングなし、アップロード途中での中断なし）。回避策：クライアント側でコンテンツを縮小するか、パス参照を渡してエージェントに `readTextFile` でファイルを読み取らせる。Stage 1.5では `/prompt` で `multipart/form-data` またはチャンクエンコーディングを受け入れ、大きなプロンプトが崖にぶつからないようにする。
>
> ⚠️ **Stage 1の既知のギャップ — NAT背後でのファントムSSE接続。** デーモンはハートビート（15秒間隔）でのTCPバックプレッシャーにより、死んだクライアントを検出する。TCP RSTなしで消えたクライアント（例：NATボックスがアイドルフローをサイレントにドロップする）は、Nodeのkeepaliveプローブがタイムアウトするまでカーネルレベルのソケットを「生存」状態に保つ。Linuxデフォルトでは通常約2時間。そのようなNAT背後での `--hostname 0.0.0.0` デプロイメントでは、ファントムSSE接続が蓄積され、最終的に256の `server.maxConnections` 上限に達する可能性がある。
>
> [`--writer-idle-timeout-ms <n>`](#deadlines-and-writer-idle-timeout)（issue [#4514](https://github.com/QwenLM/qwen-code/issues/4514) T2.9）を設定して、明示的なアプリケーションレベルのアイドル期限でギャップを埋める：`n` ミリ秒間書き込みのフラッシュ成功がない場合、デーモンは終端 `client_evicted` フレームを `reason: 'writer_idle_timeout'` とともに発行し、ストリームを閉じる。このフラグはデフォルトではオフで、従来の契約を維持する。RSTを飲み込むネットワーク上のオペレーターは、15秒のハートビート間隔を十分に上回る値（例：`60000`–`300000`）を選択することで、正当なアイドル接続が排除されず、本当にスタックしたライターが迅速に刈り取られるようにする。SDKから `caps.features.includes('writer_idle_timeout')` を事前確認し、デーモンがこの機能をサポートしていることを確認する。

### デッドラインとライターアイドルタイムアウト

Issue [#4514](https://github.com/QwenLM/qwen-code/issues/4514) T2.9 は、15秒のハートビート + AbortSignal ではカバーできない長期実行/リモートデプロイメントのギャップを埋める2つのオプトインフラグを提供します。両方ともデフォルトではオフです。シングルユーザーのループバックワークフローはビット単位で変更されません。

| フラグ                         | 環境変数                             | デフォルト | 説明                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                         |
| ------------------------------ | ----------------------------------- | ------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `--prompt-deadline-ms <n>`     | `QWEN_SERVE_PROMPT_DEADLINE_MS`     | 未設定   | 単一の `POST /session/:id/prompt` に対するサーバー側の壁時計上限。期限が切れると、デーモンはプロンプトの AbortController を中断し、HTTP `504` と `{code:"prompt_deadline_exceeded", errorKind:"prompt_deadline_exceeded", deadlineMs:n}` を返します。プロンプトごとのリクエストボディフィールド `deadlineMs` は、実際の期限をフラグ以下に **短縮** できますが、延長することはできません。機能タグ（条件付き）：`prompt_absolute_deadline`。                                                                                                                                                                                                |
| `--writer-idle-timeout-ms <n>` | `QWEN_SERVE_WRITER_IDLE_TIMEOUT_MS` | 未設定   | SSE接続ごとのアイドル期限。`n` ミリ秒間書き込みが **正常に** フラッシュされなかった場合（実際のイベントも15秒のハートビートもない）、デーモンは終端 `client_evicted` フレームを `data.reason = 'writer_idle_timeout'`（`data.errorKind` にも反映）とともに発行し、ストリームを閉じます。**15秒のハートビートを十分に上回る値を選択してください**（例：`30000`–`300000`）。これにより、正当なアイドルストリームが排除されません。`< 15000` の値は、最初のハートビートが発火する前に、そうでなければ健全なアイドル接続を排除します（テストや短期間の開発セッションでのみ意図的）。機能タグ（条件付き）：`writer_idle_timeout`。 |

両方のフラグはミリ秒単位の正の整数を受け入れます。`0`、`NaN`、非整数、負の値は起動時に明確なエラーメッセージとともに拒否されます。CLIフラグは環境変数よりも優先されます。明示的な `ServeOptions` フィールド（埋め込み呼び出し元）は環境変数よりも優先されます。SDK利用者は、これらの動作に依存する前に、該当する機能タグを事前確認する必要があります。このPRより前のデーモンは両方のタグを省略し、リクエストの `deadlineMs` フィールドはサイレントに無視されます。

## マルチセッション & マルチワークスペースデプロイメント

[#3803](https://github.com/QwenLM/qwen-code/issues/3803) §02 によると、各 `qwen serve` プロセスは起動時に **1つのワークスペース** にバインドされます。そのワークスペース内で、エージェントのネイティブセッションマップを介して、N個のセッションを単一の `qwen --acp` 子プロセスに多重化します。セッションは子プロセスのプロセス / OAuth状態 / ファイル読み取りキャッシュ / 階層メモリパースを共有します。

複数の **ワークスペース**（1ユーザーで複数のリポジトリ、または同一ホスト上の複数ユーザー）をホストするには、**複数のデーモンプロセス** を実行します。ワークスペースごとに1つのデーモン、それぞれ独自のポートで、systemd / docker-compose / k8s / `qwen-coordinator` リファレンスオーケストレーターで管理します。このトレードオフは意図的です。ワークスペースごとに1つの子プロセスとすることで、`loadSettings(cwd)` / OAuth / MCPサーバースコープがバインドされたディレクトリと整合し、リクエスト間でずれることがありません。

> **`modelServiceId` をアタッチで投稿する前にサブスクライブしてください。** クライアントが `modelServiceId` を指定して `POST /session` を呼び出し、ワークスペースにすでに異なるモデルで実行中のセッションがある場合、デーモンは内部で `setSessionModel` 呼び出しを発行します。失敗はHTTPエラーとして伝播されません（セッションは現在のモデルで動作を継続します）。可視化された失敗信号は、セッションのSSEストリーム上の `model_switch_failed` イベントです。`POST /session` を呼び出してから `GET /session/:id/events` を開くと、失敗イベントを見逃し、間違ったモデルと話し続けることになります。まずSSEストリームを開くか、サブスクライブ時に `Last-Event-ID: 0` を渡してリングの最も古い利用可能なイベントをリプレイしてください。

複数の **ユーザー**（それぞれに独自のクォータ、監査ログ、サンドボックスがある）を処理する場合、または1つのプロセスの到達範囲（コールドスタート予算、FD数、RSS）を超えてスケールするには、外部オーケストレーターの背後でワークスペースごとユーザーごとに1つのデーモンを生成します。そのオーケストレーター（マルチテナンシー / OIDC / クォータ / 監査 / k8s）は qwen-code プロジェクトの **スコープ外** です。設計のポインタについては、issue [#3803](https://github.com/QwenLM/qwen-code/issues/3803)「外部リファレンスアーキテクチャ」を参照してください。

## 永続化セッションのロードと再開

デーモンは、ACP の `session/load` と再開フローを2つのルートでHTTP経由で公開します。

| ルート                      | 使用するタイミング                                                                                                                                                                                                                                                                                      |
| -------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `POST /session/:id/load`   | クライアントに **表示された履歴がない** 場合（コールド再接続、ピッカーから開く）。デーモンは永続化されたすべてのターンをSSEでリプレイし、サブスクライバーが完全なトランスクリプトを確認できるようにします。機能タグ：`session_load`。                                                                                        |
| `POST /session/:id/resume` | クライアントがすでにターンを画面に表示しており、デーモン側のハンドルだけが必要な場合。モデルコンテキストはUIリプレイなしでエージェント側で復元され、SSEストリームはクリーンなままです。機能タグ：`session_resume`（`unstable_session_resume` は古いクライアント用の非推奨エイリアスとして残ります）。 |

TypeScript SDK は両方を `DaemonSessionClient` の静的ファクトリメソッドとして公開します。

```ts
import { DaemonClient, DaemonSessionClient } from '@qwen-code/sdk';

const client = new DaemonClient({ baseUrl: 'http://127.0.0.1:4170' });

// コールド再接続 — デーモンがSSEで履歴をリプレイします。
const session = await DaemonSessionClient.load(client, 'persisted-id');

// または、UIがすでに履歴を持っている場合は、リプレイをスキップします。
// const session = await DaemonSessionClient.resume(client, 'persisted-id');

for await (const event of session.events()) {
  // 最初にリプレイされた `session_update` フレーム（loadのみ）、
  // その後ライブイベント。
}
```

呼び出す前に `caps.features.session_load` / `caps.features.session_resume` を事前確認してください。古いデーモンは `404` を返します。`unstable_session_resume` は非推奨の互換性エイリアスとして引き続き通知されます。同じIDに対する同時の同一アクションリクエストは統合されます。クロスアクションの競合（`load` と `resume` の競合）は `409 restore_in_progress` と `Retry-After: 5` になります。完全なエラーエンベロープについては[プロトコルリファレンス](../developers/qwen-serve-protocol.md)を参照してください。

注：履歴リプレイはSSEリング（デフォルト8000フレーム）によって制限されます。ターン数が多い長い履歴はこれを超える可能性があり、最も古いフレームはサイレントにドロップされます。非常に長いセッションの場合は、`resume` を使用し、クライアントのローカルに永続化されたUIに依存することをお勧めします。

## 耐久性モデル

**Stage 1のセッションはデーモン再起動後もエフェメラルですが**、ディスクに永続化されたセッションはリロードできます。

- 子プロセスのクラッシュは `session_died` を発行し、ライブセッションをデーモンのマップから削除します。ディスクに永続化されたセッションは、新しいエージェント子プロセスが生成可能であれば、`POST /session/:id/load` で **リロード可能** です。
- デーモンの再起動は、進行中のすべてのライブセッションを失います。ディスクに永続化されたセッションは残り、同じワークスペースバインディングルールに従って、新しいデーモンプロセスに対してロードできます。
- 長時間のクライアント切断（チャットの多いターンで5分以上）は、SSEリプレイリング（デフォルト8000フレーム）を超える可能性があります。`Last-Event-ID` による再接続は成功しますが、状態が不整合になる可能性があります。モバイル / 不安定なネットワークのクライアントの場合は、長時間の切断でSSEを再開するか、`POST /session/:id/load` を呼び出してディスクからリプレイするように計画してください。
- ファイル操作（`writeTextFile`）はクラッシュに対してアトミックです（書き込み後に名前変更）。ただし、デーモン再起動をまたいでリプレイするという意味ではアトミックではありません。ファイル書き込みは成功するか、行われないかのどちらかです。

統合で `session/load` がカバーする以上のサーバー側の再起動間耐久性（例：サーバー管理のリトライキュー）が必要な場合でも、アプリケーションレベルの状態復旧が必要です。デーモンのセッション内に、再起動に敏感な長時間実行状態を保持しないでください。

## Stage 1.5+ のランタイム保証

Stage 1 の契約はプロトタイピング向けにサイジングされています。[#3889 chiga0 ダウンストリームコンシューマーレビュー](https://github.com/QwenLM/qwen-code/pull/3889#issuecomment-4427875644) によると、以下は **Stage 1では提供されません**。本番グレードの統合は、これらに依存する前に Stage 1.5+ を必要とします。
**本格的な下流利用のためのブロッカー：**

1. **`loadSession` / `unstable_resumeSession` over HTTP** — これがないと、子プロセスのクラッシュやデーモンの再起動を乗り越えられる統合は存在せず、デーモンを調整するオーケストレーターも状態を復旧できません。
2. **永続的なクライアントID（ペアトークン＋クライアント単位の失効）** — Stage 1では共有のベアラートークンを使用します。トークンが漏洩すると全員が失効し、`originatorClientId` はデーモンが認証済みIDからスタンプするのではなく、クライアントが自己宣言します。

**信頼性ベースライン：**

3. ~~**クライアント起動のハートビートパス**~~ — [#4175](https://github.com/QwenLM/qwen-code/issues/4175) PR 9 で出荷済み。`POST /session/:id/heartbeat` はデーモンに最終確認タイムスタンプを記録します（機能タグ `client_heartbeat`）。SDKヘルパーは `DaemonClient.heartbeat()` / `DaemonSessionClient.heartbeat()` です。
4. **`permission_already_resolved` イベント** — 投票が最初の応答者レースに負けた場合。現在、UIは `404` から状態を推測する必要があります。
5. ~~**より大きなリプレイリング**~~ — 8000 に増加。**セッションごとに設定可能なリング**はまだ未対応です。モバイルやチャットターンワークロードでは、セッションごとのオーバーライドが必要になる場合があります。
6. **`slow_client_warning` イベント（`client_evicted` の前）** — ソフトなバックプレッシャーにより、適切に動作する低速クライアントが強制終了される前に自己調整（レンダリング深度のトリミング、チャンクのドロップ）できます。

**統合のエルゴノミクス：**

7. **IMスタイルのコンテキストのための `POST /session/:id/_meta`** — セッションごとのキー・バリューをその後のプロンプト（チャットID、送信者、スレッドID）に付加し、チャンネルごとの即興を置き換えます。
8. **`/capabilities` による実際の機能ネゴシエーション** — `protocol_versions: { acp: '0.14.x', daemon_envelope: 1 }` により、クライアントはドリフトを検出でき、「不明なフレーム、無視」にフォールスルーしなくなります。
9. **第一級の永続性ドキュメント**（このセクション） — 上記で既に出荷済み。

完全なコンバージェンスロードマップは [#3803](https://github.com/QwenLM/qwen-code/issues/3803) で追跡されています。

## Stage 1 スコープの境界 — Stage 1.5 で修正しないもの

2つの構造的な選択は、Stage 1 / 1.5 / 2 のメインラインロードマップでは明示的な非目標です。あなたのユースケースがこれらのいずれかに依存する場合は、私たちを待つのではなく、それに応じて計画してください。

### セッション状態はローカルミューテーションのみ（[LaZzyMan レビュー #4270256721](https://github.com/QwenLM/qwen-code/pull/3889#pullrequestreview-4270256721) による）

Stage 1.5 計画では、TUI をインプロセスの EventBus サブスクライバーとして説明しています。実際には **TUI UI はワイヤープロトコルよりも厳密に大きい**です。

- **ローカルのみの UI** — 約15個の Ink ダイアログコンポーネント（`ModelDialog`、`MemoryDialog`、`PermissionsDialog`、`SessionPicker`、`WelcomeBackDialog`、`FolderTrustDialog`、…）および `local-jsx` スラッシュコマンド（`/ide`、`/auth`、`/init`、`/resume`、`/rename`、`/delete`、`/language`、`/arena`、…）は、ターミナル固有の Ink JSX をレンダリングします。HTTP/SSE 上のリモートクライアントは Ink を同等にレンダリングできず、これらのフローはワイヤーイベントを発行しません。
- **ワイヤーイベントなしのセッション状態ミューテーション** — `/approval-mode`、`/memory add`、`/mcp add-server`、`/agents`、`/tools enable/disable`、`/auth`、`/init`（`CLAUDE.md` への書き込み）はすべてエージェントの動作を変更しますが、現在 `/model` だけがイベント（`model_switched`）を公開しています。

**Stage 1 の選択 — レビューからのオプション (A)**：これらのミューテーションをワイヤーイベントに昇格させません。2つのデプロイモードには異なる結果があります。

#### モード 1 — ヘッドレス `qwen serve`（このPR）

デーモン内で TUI シェルは実行されません。上記のスラッシュコマンドはこのモードには **存在しません** — それらを発行するターミナルUIがないからです。したがって、セッション状態は次のようになります。

- **起動時に固定**：`approval-mode` / `memory` / `agents` / `tools` 許可リスト / `auth` — すべてデーモンの `qwen --acp` 子プロセスが起動するときに設定 + ディスクからロードされ、セッションの存続期間中は不変です。設定で定義された MCP サーバーも起動時に固定されますが、**実行時に追加されたサーバー**（`POST /workspace/mcp/servers` 経由）は再起動せずに追加・削除できます。
- **HTTP 経由で可変**：`POST /session/:id/model`（`model_switched` を公開）、`POST /workspace/mcp/servers` / `DELETE /workspace/mcp/servers/:name`（`mcp_server_added` / `mcp_server_removed` を公開）、および許可投票（`POST /permission/:requestId`）。

**結果：** ヘッドレスモードのリモートクライアントは **完全なセッション状態** を表示します。TUI が追加の状態を隠すことはなく、ドリフトの可能性はありません。`approval-mode` を変更したい場合は、新しい設定でデーモンを再起動してください。MCP サーバーは、ミューテーションルート（`POST /workspace/mcp/servers`、`DELETE /workspace/mcp/servers/:name`）を介して実行時に追加/削除できるようになりました — [ランタイム MCP サーバー管理](#runtime-mcp-server-management-issue-4514) を参照してください。

#### モード 2 — Stage 1.5 `qwen --serve` 共存 TUI（このPRには含まれません）

Stage 1.5 で `qwen --serve`（TUI プロセスが同じ HTTP サーバーをホストする）が登場すると、TUI はリモートクライアントと **共存** します。ローカルオペレーターが `/approval-mode yolo` や `/mcp add-server` と入力すると、セッション状態がミューテーションされ、HTTP 上のリモートクライアントにはその変更を観測するイベントがありません。

このモードでは、TUI は **「スーパークライアント」** です。リモートクライアントと同じエージェント会話を観測する **と同時に**、リモートクライアントが変更できないセッション状態をミューテーションできます。非対称性は次のとおりです。

- ✅ TUI とリモートクライアントの両方が、同じエージェントメッセージ、ツール呼び出し、ファイル差分、許可プロンプトを表示します。
- ❌ TUI だけが、approval-mode / memory / MCP サーバーリスト / agents / tools 許可リスト / auth 状態を表示/ミューテーションします。

**モード 2 の結果：** リモートクライアント UI がセッション設定をミラーリングしようとすると、TUI スラッシュコマンドの後にドリフトする可能性があります。リモートクライアントは、アタッチ/再接続時に状態を **再取得** する必要があります（`Last-Event-ID: 0` を使用して、`model_switched` のようなもののためにリングの最も古いイベントをリプレイします）。TUI 側のミューテーションについては、インクリメンタルイベントに依存すべきでは **ありません**。

#### なぜ (A) であり (B) ではないのか（ミューテーションを `session_state_changed` イベントファミリーに昇格させる）

(B) はより野心的な答えですが、Stage 1.5 をかなり大きなワイヤーサーフェスにロックし、計画されているインプロセスリファクタリングをクリーンに通過する必要もあります。より小さなスコープを正直に進めるほうがよいでしょう。セッション状態イベントの分類作業（どの TUI フローが設計上ローカルのみか、将来的なオプトイン (B) タイプの拡張でワイヤーに昇格できる可能性があるかを列挙する）は、[#3803](https://github.com/QwenLM/qwen-code/issues/3803) に移動し、Stage 1.5 のコードには含まれません。

### N 個の並列セッションが1つの `qwen --acp` 子プロセスを共有

同じワークスペース上の複数のセッションは、エージェントのネイティブマルチセッションサポート（`packages/cli/src/acp-integration/acpAgent.ts:194: private sessions: Map<string, Session>`）を介して **1つの `qwen --acp` 子プロセスを共有** します。ブリッジは、各セッションに対して `connection.newSession({cwd, mcpServers})` を呼び出します。エージェントはこれをセッションマップに保存し、呼び出しごとの sessionId で多重分離します。

同じワークスペースで N=5 セッションの場合の具体的なコスト：

| リソース                             | セッションごと | N=5 の場合                      |
| ------------------------------------ | -------------- | ------------------------------- |
| デーモン Node プロセス               | 1              | **30–50 MB**（デーモン1つ）     |
| `qwen --acp` 子プロセス              | 共有           | **60–100 MB**（子プロセス1つ）  |
| MCP サーバーの子プロセス             | セッションごと | 設定が異なる場合 3×N           |
| `FileReadCache`（子プロセス内ヒープ） | 共有           | 1回パース                       |
| `CLAUDE.md` / 階層メモリパース       | 共有           | 1回パース                       |
| OAuth リフレッシュトークン状態       | 共有           | **1つのリフレッシュパス**       |
| 自動メモリ学習済みファクト           | 共有           | 子プロセスあたり1つの知識ベース |
| コールドスタート                     | 最初のみ       | 最初のセッション後 <200 ms      |

ブリッジはデーモンごとに **1つのチャネル** を保持します（ワークスペースごとに1つのデーモン、§02 参照）。少なくとも1つのセッションが生きている間、チャネルは存続します。最後の `killSession`（またはチャネルレベルのクラッシュ）によって子プロセスが強制終了されます。

**MCP サーバーの子プロセス** は現在もセッションごとです。各セッションの設定で異なるサーバーを指定できるため、独立して起動されます。Stage 1.5 のフォローアップ：`(workspace, config-hash)` で MCP サーバーの子プロセスを参照カウントし、同一設定のものを共有できるようにします。この PR のスコープには含まれません。

**ピアエージェント（Cursor / Continue / Claude Code / OpenCode / Gemini CLI）はすべてシングルプロセスマルチセッションです。** qwen-code はエージェントレイヤーでこれらと一致します。この PR の Stage 1 ブリッジは、同じアーキテクチャを HTTP 経由で可視化します。

## リモートデーモンへのログイン（issue #4175 PR 21）

デーモンがリモートポッド上で動作している場合（あなたとディスプレイを共有していない場合）、クライアントは HTTP 経由で OAuth デバイスフローをトリガーできます。デーモンが IdP をポーリングします。あなたの仕事は、ブラウザを持つ任意のデバイスで URL を開くことだけです。

> [!note]
>
> Qwen OAuth 無料ティアは 2026-04-15 に終了しました。以下の `qwen-oauth`
> の例は、デバイスフロープロトコルの形状とレガシープロバイダー識別子を文書化したものです。
> 新しいセットアップでは、現在サポートされている認証プロバイダーを使用してください。

```bash
# 1. フローを開始します。デーモンが IdP に問い合わせ、コードと URL を返します。
curl -X POST http://127.0.0.1:4170/workspace/auth/device-flow \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"providerId":"qwen-oauth"}'
# → 201 {
#     "deviceFlowId": "fa07c61b-…",
#     "userCode": "USER-1",
#     "verificationUri": "https://chat.qwen.ai/api/v1/oauth2/device",
#     "verificationUriComplete": "https://chat.qwen.ai/...?user_code=USER-1",
#     "expiresAt": 1700000600000,
#     "intervalMs": 5000,
#     "attached": false
#   }

# 2. 携帯電話/ノートパソコンで URL にアクセスし、ユーザーコードを入力します。
# 3. 完了をポーリングします（または auth_device_flow_authorized イベントの SSE を購読します）：
curl http://127.0.0.1:4170/workspace/auth/device-flow/fa07c61b-… \
  -H "Authorization: Bearer $TOKEN"
# → ステータス遷移：pending → authorized
```

TypeScript SDK は両方のステップを1つのヘルパーにラップします：

```ts
import { DaemonClient } from '@qwen-code/sdk';

const client = new DaemonClient({ baseUrl, token });
const flow = await client.auth.start({ providerId: 'qwen-oauth' });
console.log(`Open ${flow.verificationUri}\nCode: ${flow.userCode}`);
const result = await flow.awaitCompletion({ signal: abortCtrl.signal });
// result.status === 'authorized'
```

**デーモンは決してあなたの代わりにブラウザを開きません。** ローカルで実行している場合でも、デーモンはパッシブなままです。URL を返し、SDK/ユーザーがどこで開くかを選択できるようにします。これは意図的なものです。ヘッドレスポッド上のデーモンが `xdg-open` を呼び出すと、静かに失敗し、実際の認証サーフェスを隠してしまいます。クライアントで `gh auth login` の「Enter を押してブラウザを開く」UX をミラーリングしてください。

**`--require-auth` と開発の利便性。** デバイスフロールートは厳格なミューテーションゲート（PR 15）を使用するため、トークンなしのループバックデフォルトは `401 token_required` を返します。ローカルで開発中にこれを回避する最も簡単な方法は `qwen serve --token=dev-token` です。ループバックデフォルトを強化しない限り、`--require-auth` は必要ありません。

**クロスデーモンの制限。** `oauth_creds.json` はデーモン間で共有されます（`~/.qwen/oauth_creds.json`）。そのため、デーモン A でのログイン成功は、デーモン B の次のトークンリフレッシュで自動的に検出されます。ただし、デーモン B の SDK クライアントは `auth_device_flow_authorized` イベントを受信しません（イベントはデーモンごとです）。

**クロスクライアントの引き継ぎ。** 同じデーモン上の2つの SDK クライアントが同じプロバイダーに対して `POST /workspace/auth/device-flow` を呼び出すと、プロバイダーごとのシングルトンが取得されます。最初の呼び出しは新しい IdP リクエストを開始し、`attached: false` を返します。2番目の呼び出しは、既存の処理中のエントリを `attached: true` で返します。引き継ぎは監査証跡に記録されます（2番目のクライアントの `X-Qwen-Client-Id` 配下）が、別個のイベントは発行されません。両方のクライアントは、ユーザーが IdP ページを完了すると、最終的に **同じ** `auth_device_flow_authorized` を観測します。UI で「自分が開始した」と「他の誰かが参加したフロー」を区別する場合は、`start()` から返された `attached` フィールドで分岐してください。

## デーモンログファイル

`qwen serve` は、プロセスごとの診断ログを次の場所に書き込みます：

```
${QWEN_RUNTIME_DIR または ~/.qwen}/debug/daemon/serve-<pid>-<workspaceHash>.log
```

同じディレクトリ内の `latest` シンボリックリンクは、常に現在のプロセスのログを指します。そのため、`tail -f ~/.qwen/debug/daemon/latest` を実行すると、実行中のデーモンを追跡できます。

ログには、ライフサイクルメッセージ、ルートエラー（`route=` および `sessionId=` コンテキスト付き）、ACP 子プロセスの stderr、および `QWEN_SERVE_DEBUG=1` が設定されている場合の追加のブリッジブレッドクラムが含まれます。現在 stderr に出力される行は引き続き stderr に出力されます。ファイルログは **追加的** であり、置き換えではありません。

### 無効化

`QWEN_DAEMON_LOG_FILE=0`（または `false`/`off`/`no`）を設定すると、ファイルログを完全にスキップします。Stderr 出力は影響を受けません。

### セッションデバッグログとの関係

セッションスコープのデバッグログ（`~/.qwen/debug/<sessionId>.txt` および `~/.qwen/debug/latest` シンボリックリンク）は独立しています。デーモンログは、兄弟ディレクトリ `daemon/` 内にあります。セッションごとのデバッグセマンティクスは、この機能によって変更されません。

### ローテーションなし

デーモンログは無期限に追記されます。大きくなった場合は手動でローテーションしてください。将来の機能強化として自動ローテーションが追加される可能性があります。[#4548](https://github.com/QwenLM/qwen-code/issues/4548) のフォローアップで追跡してください。

## ランタイム MCP サーバー管理（issue [#4514](https://github.com/QwenLM/qwen-code/issues/4514)）

デーモンを再起動せずに、実行時に MCP サーバーを追加または削除します。ランタイムエントリは、同じ名前の設定定義サーバーを **シャドウ** するエフェメラルオーバーレイに存在します。基になる `settings.json` / `mcpServers` 設定は決して書き込まれません。

**事前確認：** ルートを呼び出す前に、`caps.features` に `mcp_server_runtime_mutation` があるか確認してください。このタグがない古いデーモンは `404` を返します。

### `POST /workspace/mcp/servers` — ランタイム MCP サーバーの追加

厳格ゲート（ベアラートークンが必要）。ライブの `McpClientManager` を介してサーバーに即座に接続し、そのツールを検出します。

リクエスト：

```json
{
  "name": "my-server",
  "config": {
    "command": "npx",
    "args": ["-y", "@my-org/mcp-server"]
  }
}
```

`name` は英数字 + `_` と `-` である必要があります（最大256文字）。`config` は、`settings.json` の `mcpServers` エントリで使用されるものと同じ MCP サーバー設定オブジェクトです（トランスポート依存のフィールド：stdio の場合は `command`/`args`、SSE/HTTP の場合は `url`）。セキュリティに関わるフィールド（`trust`、`env`、`cwd`、`oauth`、`headers`、`authProviderType`、`includeTools`、`excludeTools`、`type`）はデーモンによって除去され、無視されます。

レスポンス (200) — 成功：

```json
{
  "name": "my-server",
  "transport": "stdio",
  "replaced": false,
  "shadowedSettings": false,
  "toolCount": 3,
  "originatorClientId": "client-1"
}
```

- `replaced: true` — 同じ名前のランタイムエントリがすでに存在し、設定フィンガープリントが異なる場合。古い接続は切断され、新しい接続が確立されます。フィンガープリントが一致する場合（べき等な再追加）、`replaced` は `false` です。
- `shadowedSettings: true` — 同じ名前の設定定義サーバーが存在する場合。ランタイムエントリがそれをシャドウします。設定エントリは変更されず、ランタイムエントリが後で削除されると再有効化されます。
- `toolCount` — 新しく接続されたサーバーで検出されたツールの数。

レスポンス (200) — ソフト拒否（予算警告モード）：

```json
{
  "name": "my-server",
  "skipped": true,
  "reason": "budget_warning_only"
}
```

`--mcp-budget-mode=warn` が設定されており、サーバーを追加すると設定済みの `--mcp-client-budget` を超過する場合に返されます。サーバーは接続されません。呼び出し元は、予算の圧力をユーザーに伝える必要があります。

エラー：

| ステータス | コード                        | 状況                                                                                               |
| ---------- | ----------------------------- | ------------------------------------------------------------------------------------------------- |
| `400`      | `invalid_server_name`         | 名前が空、256文字超過、または `[A-Za-z0-9_-]` 以外の文字を含む                                   |
| `400`      | `missing_required_field`      | `config` が欠落しているか、null 以外のオブジェクトではない                                         |
| `400`      | `invalid_client_id`           | `X-Qwen-Client-Id` ヘッダーが存在するが、このワークスペースに登録されていない                    |
| `400`      | `invalid_config`              | 設定形状が MCP トランスポートバリデーターによって拒否された                                       |
| `401`      | `token_required`              | ベアラートークンが設定されていない（厳格ゲート）                                                   |
| `409`      | `mcp_budget_would_exceed`     | `--mcp-budget-mode=enforce` で、予算が満杯                                                    |
| `502`      | `mcp_server_spawn_failed`     | サーバープロセスが終了したか、接続中にタイムアウト。ボディに `serverName`、`exitCode`、`stderr`    |
| `503`      | `acp_channel_unavailable`     | ライブな ACP 子プロセスがない（まだセッションが作成されていない）                                  |

### `DELETE /workspace/mcp/servers/:name` — ランタイム MCP サーバーの削除

厳格ゲート。サーバーを切断し、ランタイムオーバーレイから削除します。べき等 — 追加されたことのない名前を削除すると、スキップ応答（エラーではない）が返されます。

`:name` パスパラメータは URL エンコードされたサーバー名です。

レスポンス (200) — 成功：

```json
{
  "name": "my-server",
  "removed": true,
  "wasShadowingSettings": false,
  "originatorClientId": "client-1"
}
```

- `wasShadowingSettings: true` — 削除されたランタイムエントリが、同じ名前の設定定義サーバーをシャドウしていた場合。その設定エントリはシャドウ解除され、次回の検出/再起動時に使用されます。

レスポンス (200) — べき等スキップ：

```json
{
  "name": "ghost",
  "skipped": true,
  "reason": "not_present"
}
```

名前がランタイムオーバーレイに存在しなかった場合に返されます（設定には存在する可能性があります。設定エントリはこのルートでは削除できません）。

エラー：

| ステータス | コード                      | 状況                                                                 |
| ---------- | --------------------------- | -------------------------------------------------------------------- |
| `400`      | `invalid_server_name`       | 名前が空、256文字超過、または `[A-Za-z0-9_-]` 以外の文字を含む     |
| `400`      | `invalid_client_id`         | `X-Qwen-Client-Id` ヘッダーが存在するが、このワークスペースに登録されていない |
| `401`      | `token_required`            | ベアラートークンが設定されていない（厳格ゲート）                     |
| `503`      | `acp_channel_unavailable`   | ライブな ACP 子プロセスがない                                       |

### シャドウのセマンティクス

ランタイムエントリは、設定定義の MCP サーバーの上にエフェメラルオーバーレイを形成します：

- **追加**：設定エントリと同じ名前のランタイムサーバーを追加すると、それを **シャドウ** します。ランタイム設定が優先されます。元の設定エントリは変更されません。
- **削除**：設定エントリをシャドウしていたランタイムサーバーを削除すると、**シャドウが解除** されます。設定定義の設定は、次回の接続時に再びアクティブになります。
- **デーモン再起動**：すべてのランタイムエントリが失われます。再起動後も存続するのは設定定義のサーバーのみです。ランタイムサーバーはセッションの有効期間にスコープされます。
- **`GET /workspace/mcp`** はマージされたビューを報告します。設定定義とランタイムの両方のサーバーが `servers[]` 配列に表示されます。現在のスナップショットでは、2つの起点のワイヤーレベルの区別はありません。

### イベント

両方のルートは **ワークスペーススコープ** の SSE イベントを発行します（すべてのアクティブなセッションバスが受信します）：

| イベント                | 発行タイミング       | ペイロードフィールド                                                                 |
| ----------------------- | ------------------- | ---------------------------------------------------------------------------------- |
| `mcp_server_added`      | `POST` 成功時（スキップなし） | `name`、`transport`、`replaced`、`shadowedSettings`、`toolCount`、`originatorClientId` |
| `mcp_server_removed`    | `DELETE` 成功時（スキップなし） | `name`、`wasShadowingSettings`、`originatorClientId`                               |
スキップされたレスポンス（`budget_warning_only`、`not_present`）はイベントを発行しません。

既存の `mcp_guardrail_events` サーフェスからの予算関連イベント（`mcp_budget_warning`、`mcp_child_refused_batch`）は、ランタイムの追加が予算しきい値を超えた場合にも発火します。

# Qwen Code Companion プラグイン：インターフェース仕様

> 最終更新日：2025年9月15日

本ドキュメントでは、Qwen Code の IDE モードを有効化するためのコンパニオンプラグインを構築する際の契約（インターフェース仕様）を定義します。VS Code の場合、これらの機能（ネイティブな差分比較、コンテキスト認識）は公式拡張機能（[マーケットプレイス](https://marketplace.visualstudio.com/items?itemName=qwenlm.qwen-code-vscode-ide-companion)）によって提供されます。本仕様は、JetBrains IDE や Sublime Text などの他のエディタにも同様の機能を実装したいコントリビューター向けです。

## I. 通信インターフェース

Qwen Code と IDE プラグインは、ローカルの通信チャネルを介して相互に通信します。

### 1. トランスポート層：HTTP 上の MCP

プラグインは、**Model Context Protocol (MCP)** を実装するローカル HTTP サーバーを実行する**必要があります**。

- **プロトコル:** サーバーは有効な MCP サーバーでなければなりません。利用可能な場合は、使用言語向けの既存の MCP SDK の利用を推奨します。
- **エンドポイント:** サーバーは、すべての MCP 通信に対して単一のエンドポイント（例：`/mcp`）を公開する必要があります。
- **ポート:** サーバーは、動的に割り当てられたポート（つまり、ポート `0` でリッスン）で待機する**必要があります**。

### 2. 発見メカニズム：ロックファイル

Qwen Code がサーバーに接続するには、そのサーバーが使用しているポートを特定する必要があります。このため、プラグインは**必ず**「ロックファイル」を作成し、ポートを環境変数に設定する必要があります。

- **CLI によるファイルの検出方法：** CLI はまず環境変数 `QWEN_CODE_IDE_SERVER_PORT` からポート番号を読み取り、その後 `~/.qwen/ide/<PORT>.lock` を読み込みます。（旧バージョンの拡張機能向けにレガシーなフォールバックが存在します。詳細は下記の注釈を参照してください。）
- **ファイルの保存場所：** ファイルは特定のディレクトリ `~/.qwen/ide/` 内に作成する必要があります。該当ディレクトリが存在しない場合、プラグイン側で**必ず**作成してください。
- **ファイル名の命名規則：** ファイル名は極めて重要であり、**必ず**以下の形式に従う必要があります：
  `<PORT>.lock`
  - `<PORT>`：MCP サーバーが待ち受けているポート番号です。
- **ファイルの内容とワークスペースの検証：** ファイルには、以下の構造を持つ JSON オブジェクトを**必ず**含める必要があります：

  ```json
  {
    "port": 12345,
    "workspacePath": "/path/to/project1:/path/to/project2",
    "authToken": "a-very-secret-token",
    "ppid": 1234,
    "ideName": "VS Code"
  }
  ```
  - `port`（数値、必須）：MCP サーバーのポート番号。
  - `workspacePath`（文字列、必須）：開かれているすべてのワークスペースのルートパスを OS 固有の区切り文字で連結した文字列（Linux/macOS では `:`、Windows では `;`）。CLI はこのパスを用いて、自身が IDE で開かれているプロジェクトフォルダーと同じ場所で実行されているかを確認します。CLI の現在の作業ディレクトリが `workspacePath` のいずれかのサブディレクトリでない場合、接続は拒否されます。プラグインは、開かれているワークスペースのルートパスを**正しい絶対パス**で提供する**必要があります**。
  - `authToken`（文字列、必須）：接続を保護するための秘密トークン。CLI はすべてのリクエストにおいて、このトークンを `Authorization: Bearer <token>` ヘッダーに含めます。
  - `ppid`（数値、必須）：IDE プロセスの親プロセス ID。
  - `ideName`（文字列、必須）：IDE のユーザーフレンドリーな名称（例：`VS Code`、`JetBrains IDE`）。

- **認証：** 接続を保護するために、プラグインは**必ず**一意かつ秘密のトークンを生成し、それを発見ファイルに含める必要があります。CLI はその後、このトークンを MCP サーバーへのすべてのリクエストの `Authorization` ヘッダーに含めます（例：`Authorization: Bearer a-very-secret-token`）。サーバー側は、すべてのリクエストでこのトークンを検証し、不正なトークンを含むリクエストは**必ず**拒否しなければなりません。
- **環境変数（必須）：** プラグインは、統合ターミナル内で `QWEN_CODE_IDE_SERVER_PORT` 環境変数を**必ず**設定する必要があります。これにより、CLI が適切な `<PORT>.lock` ファイルを検出できます。

**レガシーに関する注記：** v0.5.1 より前のバージョンの拡張機能では、Qwen Code がシステムの一時ディレクトリ内にある `qwen-code-ide-server-<PID>.json` または `qwen-code-ide-server-<PORT>.json` という名前の JSON ファイルを読み込むフォールバック処理が実装されています。新規の連携実装では、これらのレガシーファイルに依存してはいけません。

## II. コンテキスト・インターフェース

コンテキスト認識を可能にするために、プラグインは IDE 内でのユーザーの活動に関するリアルタイム情報を CLI に提供 **できます**。

### `ide/contextUpdate` 通知

プラグインは、ユーザーのコンテキストが変更された際に、CLI に対して `ide/contextUpdate` [通知](https://modelcontextprotocol.io/specification/2025-06-18/basic/index#notifications) を**送信してもよい（MAY）**。

- **トリガーとなるイベント：** 次のいずれかのイベント発生時に（推奨デバウンス時間：50ms）、この通知を送信する必要があります。
  - ファイルが開かれたり、閉じられたり、フォーカスされたりしたとき。
  - アクティブなファイル内で、ユーザーのカーソル位置またはテキスト選択範囲が変更されたとき。
- **ペイロード（`IdeContext`）：** この通知のパラメーターは、**必ず** `IdeContext` オブジェクトでなければなりません。

  ```typescript
  interface IdeContext {
    workspaceState?: {
      openFiles?: File[];
      isTrusted?: boolean;
    };
  }

  interface File {
    // ファイルへの絶対パス
    path: string;
    // 最後にフォーカスされた時刻（Unix タイムスタンプ、並べ替え用）
    timestamp: number;
    // 現在フォーカス中のファイルである場合 true
    isActive?: boolean;
    cursor?: {
      // 行番号（1 基準）
      line: number;
      // 文字番号（1 基準）
      character: number;
    };
    // ユーザーが現在選択しているテキスト
    selectedText?: string;
  }
  ```

  **注意：** `openFiles` リストには、ディスク上に実際に存在するファイルのみを含める必要があります。仮想ファイル（例：パスを持たない未保存ファイル、エディター設定ページなど）は**必ず除外しなければなりません**。

### CLI がこのコンテキストをどのように使用するか

CLI は `IdeContext` オブジェクトを受信した後、情報をモデルに送信する前に、いくつかの正規化および切り捨て処理を行います。

- **ファイルの並べ替え:** CLI は `timestamp` フィールドを用いて、最も最近使用されたファイルを特定します。`openFiles` リストはこの値に基づいてソートされます。したがって、プラグインはファイルが最後にフォーカスされた時刻を正確な Unix タイムスタンプで提供 **しなければなりません**。
- **アクティブなファイル:** CLI はソート後の最も新しいファイルのみを「アクティブ」なファイルと見なします。他のすべてのファイルについて、`isActive` フラグは無視され、その `cursor` および `selectedText` フィールドはクリアされます。プラグインは、現在フォーカスされているファイルに対してのみ `isActive: true` を設定し、カーソル位置および選択範囲の詳細を提供するようにしてください。
- **切り捨て:** トークン数制限を管理するため、CLI はファイルリスト（最大 10 ファイル）および `selectedText`（最大 16KB）をそれぞれ切り捨てます。

最終的な切り捨て処理は CLI が行いますが、プラグイン側でも送信するコンテキスト量を事前に制限することを強く推奨します。

## III. 差分表示インターフェース

対話的なコード変更を可能にするために、プラグインは**オプションで**差分表示インターフェースを公開できます。これにより、CLI が IDE に対してファイルへの提案変更を表示する差分ビューを開くよう要求できるようになります。ユーザーはこの差分ビュー内で変更内容を確認・編集し、最終的に IDE 内でそれらの変更を承認または却下できます。

### `openDiff` ツール

このプラグインは、自身の MCP サーバー上で `openDiff` ツールを**必ず**登録しなければなりません。

- **説明:** このツールは、IDE に対して特定のファイル用の編集可能な差分ビュー（diff view）を開くよう指示します。
- **リクエスト (`OpenDiffRequest`):** このツールは `tools/call` リクエストによって呼び出されます。リクエストの `params` 内の `arguments` フィールドは、**必ず** `OpenDiffRequest` オブジェクトでなければなりません。

  ```typescript
  interface OpenDiffRequest {
    // 差分表示対象ファイルの絶対パス。
    filePath: string;
    // ファイルに適用する予定の新しい内容。
    newContent: string;
  }
  ```

- **レスポンス (`CallToolResult`):** このツールは、リクエストを受諾し、差分ビューが正常に開かれたかどうかを報告するために、**即座に** `CallToolResult` を返さなければなりません。
  - **成功時:** 差分ビューが正常に開かれた場合、レスポンスには空のコンテンツ（つまり `content: []`）を含める必要があります。
  - **失敗時:** 何らかのエラーにより差分ビューの開きに失敗した場合、レスポンスの `isError` を `true` に設定し、`content` 配列内にエラー内容を記述した `TextContent` ブロックを含める必要があります。

  差分ビューにおける実際の操作結果（変更の適用または破棄）は、非同期の通知を通じて通知されます。

### `closeDiff` ツール

このプラグインは、その MCP サーバー上で **必ず** `closeDiff` ツールを登録しなければなりません。

- **説明:** このツールは、特定のファイルに対して開かれている差分ビュー（diff view）を IDE に閉じさせる指令を送ります。
- **リクエスト (`CloseDiffRequest`):** このツールは `tools/call` リクエストによって呼び出されます。リクエストの `params` 内の `arguments` フィールドは、**必ず** `CloseDiffRequest` オブジェクトでなければなりません。

  ```typescript
  interface CloseDiffRequest {
    // 差分ビューを閉じる対象ファイルの絶対パス。
    filePath: string;
  }
  ```

- **レスポンス (`CallToolResult`):** このツールは **必ず** `CallToolResult` を返さなければなりません。
  - 成功時: 差分ビューが正常に閉じられた場合、レスポンスの `content` 配列には、閉じる直前のファイルの最終内容を含む単一の **TextContent** ブロックを含める必要があります。
  - 失敗時: 何らかのエラーにより差分ビューの閉じ込みが失敗した場合、レスポンスの `isError` を `true` に設定し、エラー内容を記述した `TextContent` ブロックを `content` 配列に含める必要があります。

### `ide/diffAccepted` 通知

ユーザーが差分ビュー内の変更を承認した場合（例：「適用」または「保存」ボタンをクリック）、プラグインは **必ず** `ide/diffAccepted` 通知を CLI に送信しなければなりません。

- **ペイロード：** 通知のパラメーターには、ファイルパスとファイルの最終的な内容を含める **必要があります**。この内容は、ユーザーが差分ビュー内で手動で編集した場合、元の `newContent` とは異なる可能性があります。

  ```typescript
  {
    // 差分が取られたファイルの絶対パス。
    filePath: string;
    // 承認後のファイルの完全な内容。
    content: string;
  }
  ```

### `ide/diffRejected` 通知

ユーザーが変更を拒否した場合（例：差分ビューを承認せずに閉じる）、プラグインは **必ず** `ide/diffRejected` 通知を CLI に送信しなければなりません。

- **ペイロード：** 通知のパラメーターには、拒否された差分の対象ファイルのパスを含める **必要があります**。

  ```typescript
  {
    // 差分が取られたファイルの絶対パス。
    filePath: string;
  }
  ```

## IV. ライフサイクル インターフェース

プラグインは、IDE のライフサイクルに応じて、自身のリソースおよびディスカバリー ファイルを**必ず**適切に管理しなければなりません。

- **有効化時（IDE の起動時／プラグインが有効化されたとき）:**
  1.  MCP サーバーを起動します。
  2.  ディスカバリー ファイルを作成します。
- **無効化時（IDE の終了時／プラグインが無効化されたとき）:**
  1.  MCP サーバーを停止します。
  2.  ディスカバリー ファイルを削除します。

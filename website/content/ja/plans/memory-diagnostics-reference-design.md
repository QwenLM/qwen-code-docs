# メモリ診断リファレンス設計

## 背景

メモリとパフォーマンスの診断に関するIssue #3000は、長時間実行されるQwen Codeセッションを対象としています。最初のPRでは、より重いプロファイリングや保持期間の変更を追加する前に、小規模で低リスクな診断用のインターフェースを確立する必要があります。

この設計はリファレンスファーストです：

- Claude Codeは、メモリ診断をヒープスナップショット生成から分離して保持しています。その診断には、プロセスメモリ、V8ヒープ統計、ヒープ領域、リソース使用量、アクティブなハンドル/リクエスト、ファイルディスクリプタ、Linux `smaps_rollup`、リークヒントが含まれます。
- Codexは、長期プロセス状態に対するバウンデッドリテンションと遅延ローディングに重点を置いています。これらのアイデアは、会話、コマンド出力、履歴の保持に対処する後続のPRの指針となるはずです。

## 最初のPRの範囲

`/doctor memory` という診断パスを追加し、単一の時点のスナップショットを取得します：

- `process.memoryUsage()`
- V8 heap statistics and heap spaces
- `process.resourceUsage()`
- active handle/request counts
- 利用可能な場合の `/proc/self/fd` によるオープンファイルディスクリプタ数
- 利用可能な場合の Linux `smaps_rollup`
- ヒープ圧力、デタッチされたコンテキスト、過剰なハンドル、過剰なリクエスト、ファイルディスクリプタの多さ、ネイティブメモリ圧力に対する基本的なリスクヒント

このコマンドは、通常のセッションで実行するのに十分軽量で、Linux専用のプローブが利用できないプラットフォームでも安全である必要があります。

## 非目標

このPRは意図的に以下を行いません：

- ヒープスナップショットの書き出し
- 継続的なポーリングの実行
- プロンプト/履歴保持の変更
- ツール出力保持の変更
- モジュール読み込み動作の変更

これらは診断ベースラインが存在した後のフォローアップPRです。

## フォローアップPR

1. より詳細なローカル調査のための明示的なスナップショット/エクスポートサポートを追加します。
2. Codexのキャップ付き出力保持を主なリファレンスとして、大きなコマンド/ツール出力に対するバウンデッドリテンションを追加します。
3. 測定結果がホットスポットを特定した後、遅延ローディングとモジュール起動パスを監査します。
4. 長時間実行セッション用の再現可能なメモリ/パフォーマンスベンチマークシナリオを追加します。

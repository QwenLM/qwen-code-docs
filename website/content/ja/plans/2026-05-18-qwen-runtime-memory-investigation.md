# Qwen Code ランタイムメモリ調査計画

日付: 2026-05-18

## 背景

ローカルベンチマークでは、Qwen Code が同様の非対話型 CLI タスク形状において、Claude Code よりも大幅に多くのプロセスツリー RSS を使用していることが示されています。最新の5ケースのマトリックスでは、Qwen Code が約 `0.83-1.04 GiB` でピークに達するのに対し、Claude Code は約 `0.27-0.36 GiB` に留まっています。

このドキュメントは、調査と最適化の方向性を草案として提案します。まだ根本原因を断定するものではありません。当面の目標は、メモリギャップをレビュー可能かつ再現可能にし、内部診断によって説明可能にすることです。

## これまでの進捗

調査は証拠と方向性の段階に達しています。

- 小さな PR レビュー、コードナビゲーション、および合成 diff ワークロードに対して、再現可能なローカルマトリックスが構築されました。
- Qwen Code は複数のモデル間で比較されました。
- Qwen Code と Claude Code は、同等のモデルエンドポイントが利用可能な同じタスク形状で比較されました。
- 観測された RSS ギャップは、より深いランタイム診断を正当化するのに十分な一貫性があります。
- 関連する上流の作業がマッピングされ、この取り組みが既存の `/doctor memory` およびメモリ診断のフォローアップに基づいて構築できるようになりました。

調査はまだ根本原因の段階に達していません。なぜなら、外部プロセス RSS だけでは、保持されているメモリが V8 ヒープ、ネイティブメモリ、ロードされたモジュール、ライブ履歴、ツール結果、リクエスト組み立て状態のいずれであるかを示すことができないからです。

## 現在のエビデンス

付属のベンチマークレポートは以下の通りです。

- `docs/e2e-tests/2026-05-18-qwen-memory-benchmark-report.md`

主なエビデンスは以下の通りです。

- Qwen 対 Claude の RSS ギャップは、小さな PR レビュー、コードナビゲーション、および合成 diff ワークロード全体で再現されました。
- ギャップは `pai/glm-5` と `qwen3.6-plus` の両方で再現されました。
- Qwen Code は、テストしたすべてのマトリックスセルにおいて Claude Code よりも多くのトークンを使用しました。
- 大きな diff サイズは明確な線形メモリ増加を生み出しませんでした。これは、生の diff バイト数よりも、ベースラインと境界/切り詰められた出力パスの方が重要であることを示唆しています。

## なぜこのドラフトはドキュメントから始めるのか

このドラフトは、ランタイムコードの変更をバンドルする代わりに、意図的にベンチマークエビデンスと調査計画から始めます。

理由:

1. 現在の目標は、パフォーマンス問題と方向性を可視化することであり、同日中の修正を主張することではありません。
2. インストルメンテーションと最適化を同じ PR に含めると、測定、診断、動作変更が混在するため、レビューが難しくなります。
3. 既存のベンチマークは、より深い診断の必要性をすでに裏付けています。
4. 次の PR はより狭く、検証しやすくなります。診断のみにし、同じマトリックスを再実行し、内部メトリクスを比較します。

次の実装 PR では、不足しているカウンターとタイムラインポイントを追加し、ベンチマークマトリックスを再実行する必要があります。その後にのみ、ターゲットを絞った最適化 PR でメモリ削減を試みるべきです。

## 作業仮説

現在のデータは、モデルプロバイダー問題というよりも、Qwen Code のランタイム/パス問題を示しています。

最も有力な仮説は次の通りです。

> Qwen Code は、非対話型 CLI タスクの実行フットプリントが高いように見え、おそらくより大きなコンテキスト/ツール結果/セッション処理によって増幅されています。問題領域は、選択されたモデル単独ではなく、CLI ランタイムとエージェントデータパスである可能性が高いです。

より具体的には、エビデンスは「ツール呼び出しが多すぎる」ことが主な原因ではないことを示しています。ツール呼び出し回数は CLI 間で同程度であり、Claude はより低い RSS を維持しながら、より多くのターンまたはツール呼び出しを使用する場合もありました。より可能性の高い問題は、Qwen Code が同じ短い非対話型 CLI タスクに対してより重い状態を初期化または保持し、その実行フットプリントをより大きなコンテキスト、ツール結果、保存出力、またはセッション履歴データで増幅していることです。

最も可能性の高いバケットは次の通りです。

1. **プロセスおよびモジュールの起動/実行コスト**: Qwen Code は、非対話型 CLI タスクに必要なよりも多くのランタイム、ツール、UI/セッションインフラ、またはプロバイダー機構を初期化する可能性があります。
2. **履歴とコンテキストの組み立て**: Qwen Code は、同じタスク形状に対して Claude Code よりも大きなモデル向けコンテキストを保持または構築する可能性があります。
3. **ツール結果の保持**: 大きなまたは繰り返しのツール結果が、ライブ履歴、UI 履歴、チャット記録、または保存出力復元パスに保持される可能性があります。
4. **サブエージェントと保存出力の増幅**: 以前の大規模 PR テストでは、保存出力の復元とサブエージェントアクティビティが示されており、メモリとトークンの圧力を追加する可能性があります。
5. **MCP 子プロセス**: 付属の診断レポートにより、MCP サーバー（例: chrome-devtools）がプロセスツリー RSS に約 350 MiB 寄与していることが明らかになりました。これは絶対値を膨らませますが、セッション長とは無関係の一定のオーバーヘッドです。
6. **ネイティブメモリと JS ヒープの分割**: 外部 RSS では、圧力が V8 ヒープ、ネイティブバッファ、ロードされたモジュール、または保持されたデータのいずれかを判断できません。

これは意図的に仮説として表現されています。次のステップは、各バケットを確認または除外するのに十分な内部測定を追加することです。

## 提案するドラフト PR の範囲

最初のドラフト PR は、エビデンスと診断に焦点を当てるべきです。

1. ベンチマークレポートと調査計画をコミットする。
2. ローカル診断出力を追加または拡張して、Qwen Code が以下を報告できるようにする。
   - V8 ヒープおよびヒープスペース統計。
   - RSS とヒープの分割。
   - セッションメッセージ数と推定保持サイズ。
   - ツール結果の数、合計保持サイズ、最大保持結果サイズ。
   - 切り詰めと保存出力復元のカウンター。
   - 利用可能な場合のサブエージェント/プロセスツリーアクティビティ。
3. 既存のマトリックスを以下に対して再実行する。
   - 現在公開されている Qwen Code、
   - 現在の `main`、
   - 診断のみのブランチ、
   - 候補最適化ブランチ。
4. これらの測定値を使用して、1 つの小さな最適化ターゲットを選択する。

最初の PR は、いくつかの無関係な最適化を混在させるべきではありません。ドキュメントのみに留めるか、診断のみのコードを追加する必要があります。原因がより明確になったら、別の PR で最初のランタイムメモリ削減を実施する必要があります。

## 最適化の候補方向

これらは結論ではなく、候補です。

1. **境界付きツール出力保持**: 大きな出力をホットパスから外して保存し、ライブ履歴にはプレビュー、メタデータ、および取得ポインタのみを保持する。
2. **非対話型レイジーローディング**: 非対話型 CLI タスク実行中に TUI 専用または対話専用のサブシステムを初期化しない。
3. **セッション/UI 履歴の上限**: 古いまたは重い履歴アイテムをコンパクトなトランスクリプトエントリに劣化させる。
4. **コンテキスト組み立てのアカウンティング**: モデルリクエスト構築前に大きなツール結果を測定し上限を設定する。
5. **サブエージェントのアカウンティング**: 診断でサブエージェントのライフサイクルとメモリへの影響を公開する。

Claude Code と OpenAI Codex（OpenAI の CLI コーディングエージェント）は、診断の分離、境界付き出力保持、およびレイジー履歴読み込みの設計リファレンスとして使用する必要があります。実装は依然として Qwen Code 独自のアーキテクチャとテストに従う必要があります。

## 検証計画

調査では、before/after の結果を比較可能にするために、同じベンチマークマトリックスを維持する必要があります。

- 小さな PR レビュー
- コードナビゲーション
- 約 100 KiB の合成 diff
- 約 1 MiB の合成 diff
- 約 5 MiB の合成 diff

各実行について、以下を記録します。

- プロセスツリー RSS ピーク
- ルートプロセス RSS ピーク
- V8 ヒープピーク
- ヒープスペースサマリー
- 実行時間
- ターン数
- トークン数
- ツール呼び出し回数
- 最大保持ツール結果サイズ
- 保持ツール結果の合計サイズ
- セッション/履歴アイテム数
- サブエージェント数

候補修正の最小成功条件は、単に「RSS が減少した」ことだけではありません。どの内部メトリクスが変更され、その理由も特定できる必要があります。

## 次期 PR 候補

次の PR は診断のみとし、ランタイム動作を変更しないようにする必要があります。最小限の有用なスライスは以下を追加します。

- モデルリクエストの入力サイズのアカウンティング。
- システムプロンプトとツールスキーマサイズのアカウンティング。
- 保持メッセージ数と推定保持文字サイズ。
- 保持ツール結果の数、合計サイズ、最大アイテムサイズ。
- 起動、最初のリクエスト組み立て、ツール実行、ストリーミング完了、圧縮、最終応答の周辺のライフサイクルサンプル。
- RSS、ヒープ使用量、ヒープ合計、外部、ヒープスペース統計を含むプロセスメモリサンプル。

これがローカルに導入された後、同じ Qwen モデルマトリックスを再実行し、以下を比較します。

- 公開済みの Qwen Code。
- 現在の `main`。
- 診断のみのブランチ。
- 候補最適化ブランチ。

## 非目標

このドラフトは以下を主張するものではありません。

- すべてのメモリ圧力がツール出力によって引き起こされている。
- 既存の未解決 PR の1つが観測されたタスク実行時フットプリントを解決する。
- モデルプロバイダーの違いがすべての環境で無関係である。
- 単一実行のローカル測定がリリースレベルのパフォーマンス主張に十分である。

意図された主張はより狭いものです。Qwen Code はテストされたワークロードにおいて一貫したローカル RSS ギャップを示しており、プロジェクトはそのギャップを説明し削減するための内部診断を必要としています。

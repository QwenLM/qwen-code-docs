# OpenRouter 認証とモデル管理の設計

このドキュメントは、OpenRouter の認証フローとそれに伴って導入されたモデル管理の変更に関する設計意図をまとめたものです。実装の経緯ではなく、プロダクトおよびアーキテクチャ上の選択に焦点を当てています。

## 目標

- CLI と `/auth` の両方から OpenRouter で認証できるようにする。
- OpenRouter 専用の新しい認証タイプを追加するのではなく、既存の OpenAI 互換プロバイダーパスを再利用する。
- 初回起動時に、何百ものモデルをすぐに管理させることなく使えるエクスペリエンスにする。
- `/manage-models` を通じたより豊かなモデル管理への明確な拡張パスを維持する。

## OpenRouter 認証

OpenRouter は OpenAI 互換プロバイダーとして統合されています：

- auth type: `AuthType.USE_OPENAI`
- provider settings: `modelProviders.openai`
- API key env var: `OPENROUTER_API_KEY`
- base URL: `https://openrouter.ai/api/v1`

ランタイムのモデルプロバイダーパスがすでに OpenAI 互換であるため、OpenRouter 専用の `AuthType` を導入しないようにしています。これにより、認証ステータス・モデル解決・プロバイダー選択・設定スキーマが既存のプロバイダー抽象化と一致した状態を保ちます。

ユーザー向けのフローは以下の通りです：

- `/auth` → インタラクティブな TUI フロー用の OpenRouter。
- 自動化または直接 API キー設定用の環境変数：
  `OPENROUTER_API_KEY` と `OPENAI_BASE_URL=https://openrouter.ai/api/v1`。
- 明示的なモデルプロバイダーエントリが必要なスクリプトセットアップ用の `~/.qwen/settings.json`。

ブラウザ OAuth は OpenRouter の PKCE フローを使用し、交換した API キーを設定に書き込んでから、`AuthType.USE_OPENAI` として認証を更新します。

## モデル管理

OpenRouter は大規模な動的モデルカタログを公開しています。発見されたすべてのモデルを `modelProviders.openai` に書き込むと、`/model` が煩雑になり、長期的な設定フィールドがリモートカタログのキャッシュになってしまいます。

重要な設計上の分割は以下の通りです：

- **カタログ**: OpenRouter などのソースから発見されたモデルの全セット。
- **有効セット**: `/model` に表示され、ユーザー設定に永続化されるモデルの小さなセット。

OpenRouter の初回フローでは、大きなピッカーでユーザーを中断させるのではなく、認証完了後に使いやすいデフォルトの有効セットを用意する必要があります。推奨セットは小さく安定しており、利用可能な場合は無料モデルを含め、ユーザーがプロダクトを正常に試せるモデルに偏っているべきです。

`/model` は高速なモデルスイッチャーであり続けるべきです。ユーザーがプロバイダーの全カタログを閲覧・整理する場所にはなるべきではありません。

## `/manage-models`

より豊かなモデル管理は、専用の `/manage-models` エントリーポイントに属します。そのフローではユーザーが以下を行えるようにすべきです：

- 発見されたモデルを閲覧する；
- ID・表示名・プロバイダープレフィックス・`free` や `vision` などの派生タグで検索する；
- 現在有効なモデルを確認する；
- バッチ単位でモデルを有効化または無効化する。

ソースの次元はこの設計の一部として残す必要があります。OpenRouter は最初の動的カタログソースにすぎず、ModelScope や ModelStudio などの将来のソースも同じ形状に収まるべきです。UI の複雑さは軽減できますが、基盤となるソース抽象化は拡張ポイントとして利用可能な状態を維持すべきです。

## 現在の境界

この変更は、OpenRouter の認証とモデルセットアップを快適にするために最低限必要なことだけを行うべきです：

- OAuth またはキーベースの認証が、既存の OpenAI 互換プロバイダーパスを通じて OpenRouter を設定する。
- 初期の有効モデルセットは、全カタログを設定にダンプするのではなく、精選されたものにする。
- 全カタログの保存・閲覧・フィルタリング・バッチ管理は `/manage-models` に延期する。

設計原則はシンプルです：認証はユーザーをすばやく動作状態に導くべきであり、モデルの整理は専用の管理フローに委ねるべきです。

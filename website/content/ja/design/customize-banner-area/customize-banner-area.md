# バナー領域のデザインをカスタマイズする

> ユーザーが QWEN の ASCII アートを置き換え、ブランドタイトルを置き換え、バナー全体を非表示にできるようにする一方で、Qwen Code のデバッグ可能性と信頼性を支える運用データ（バージョン、認証、モデル、作業ディレクトリ）は非表示にできないようにします。

## 概要

Qwen Code CLI は起動時に、QWEN の ASCII ロゴと枠線付きの情報パネルを含むバナーを表示します。実際のユースケースでは、この表示面をある程度制御したいという要望があります。

- **ホワイトラベル / サードパーティブランド統合**: Qwen Code を自社製品に組み込む企業やチームは、デフォルトの「Qwen Code」ではなく、自社のブランドアイデンティティを表示したいと考えます。
- **パーソナライゼーション**: 個人がターミナルのバナーをチーム標準や自分の好みに合わせたいと考えます。
- **マルチテナント / マルチインスタンスの区別**: 共有環境では、異なるチームが自分たちがどのインスタンスにいるかをすばやく視覚的に確認したいと考えます。

設計のスタンスはシンプルです。**ブランドの飾り部分は置き換え可能、運用データはロックされています**。カスタマイズにより、ユーザーは独自のブランディングを前面に出すことができますが、セッションのデバッグを可能にする情報を非表示にすることはできません。このスタンスが、このドキュメントの残りの部分における「何を変更でき、何がロックされているか」というすべての判断を決定します。

これは [issue #3005](https://github.com/QwenLM/qwen-code/issues/3005) で追跡されています。

## バナー領域の分類

現在、バナーは `Header`（`AppHeader` からマウント）によってレンダリングされ、以下の領域に分類されます。

```
  marginX=2                                                           marginX=2
  │                                                                          │
  ▼                                                                          ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│                                                                             │
│   ┌──── ロゴカラム ─────┐  gap=2  ┌──── 情報パネル (枠線付き) ────────────┐  │
│   │                      │         │                                       │  │
│   │  ███ QWEN ASCII ███  │         │  ① タイトル:    >_ Qwen Code (vX.Y.Z) │  │
│   │  ███   アート   ███  │         │  ② サブタイトル: «空白、またはオーバーライド›│  │
│   │  ███ QWEN ASCII ███  │         │  ③ ステータス: Qwen OAuth | qwen-…   │  │
│   │                      │         │  ④ パス:     ~/projects/example      │  │
│   └──────── A ───────────┘         └──────────────── B ──────────────────┘  │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
                              region: AppHeader
                          │ Tips コンポーネントは下に表示 (ui.hideTips で制御) │
```

2つのトップレベルのボックスは次のとおりです。

- **A. ロゴカラム** — グラデーションが適用された単一の ASCII アートブロック。現在は `packages/cli/src/ui/components/AsciiArt.ts` の `shortAsciiLogo` から取得されています。
- **B. 情報パネル** — 4つの行を含む枠線付きのボックス。2行目はデフォルトでは空白の視覚的スペーサーですが、呼び出し元が指定したサブタイトルに置き換えることもできます。
  - **B①** タイトル: `>_ Qwen Code (vX.Y.Z)` — ブランドテキスト + バージョンサフィックス。
  - **B②** サブタイトル / スペーサー: デフォルトでは空白の1行スペース。`ui.customBannerSubtitle` が設定されている場合、その文字列がこの行を占有します（例：フォークの場合「Built-in DataWorks Official Skills」など）。
  - **B③** ステータス: `<認証表示タイプ> | <モデル> ( /model で変更)`。
  - **B④** パス: チルダで短縮された作業ディレクトリ。

全体は `<AppHeader>` でラップされており、`showBanner = !config.getScreenReader()` (スクリーンリーダーモードではプレーンな出力にフォールバック) によってバナーが制御されます。

## カスタマイズルール — 何を変更でき、何がロックされているか

| 領域                                           | 現在のソース                        | カスタマイズカテゴリ         | 根拠                                                                                                                                                                                                       |
| ---------------------------------------------- | ----------------------------------- | --------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **A. ロゴカラム**                               | `shortAsciiLogo` (`AsciiArt.ts`)    | **置き換え可能 + 自動非表示可能** | 純粋なブランド表示面。ホワイトラベルでは視覚表現を完全に制御する必要があります。既存の「狭いターミナルでは自動非表示」フォールバックは維持されます。                                                              |
| **B①. タイトル — ブランドテキスト** (`>_ Qwen Code`) | `Header.tsx` にハードコード         | **置き換え可能**              | ブランド表示面。先頭の `>_` グリフは既存のブランドの一部です。ユーザーが不要であれば、`customBannerTitle` から単に省略できます。                                                                          |
| **B①. タイトル — バージョンサフィックス** (`(vX.Y.Z)`) | `version` プロパティ                | **ロック**                    | バグレポートに不可欠です。非表示にすると、「どのバージョンを使っていますか？」という質問に `--version` でしか答えられなくなり、サポートワークフローにおいて大きなコストが発生します。小さなホワイトラベルの損失と引き換えに、サポートの追跡可能性を確保します。 |
| **B②. サブタイトル / スペーサー行**             | デフォルトでは空白                  | **置き換え可能**              | 純粋なブランド・コンテキスト表示面。ホワイトラベルのフォークがビルドのラベル付けに使用します（例：「Built-in DataWorks Official Skills」）。タイトルと同様にサニタイズされ、1行のみ — レイアウトを壊す改行は禁止です。 |
| **B③. ステータス行** (認証 + モデル)            | `formattedAuthType`, `model` プロパティ | **ロック**                    | 運用・セキュリティシグナル。ユーザーは常にどの認証情報が使用されており、どのモデルがトークンを消費するかを確認できなければなりません。これを抑制することは、ホワイトラベルのシナリオでも危険です。             |
| **B④. パス行** (作業ディレクトリ)               | `workingDirectory` プロパティ         | **ロック**                    | 運用データ。「どのディレクトリにいるのか？」は常に疑問に上がる質問であり、バナーがその正規の答えを提供します。                                                                                              |
| **バナー全体** (A + B)                          | `AppHeader.tsx` の `<Header>` マウント | **非表示可能**              | 単一の `ui.hideBanner: true` で両方の領域をスキップします — 既存のスクリーンリーダーゲートと同じ形状です。`<Tips>` は引き続き `ui.hideTips` によって独立して制御されます。                                         |

このマトリックスは、以下の4つの設定に対応します。これ以上はありません。

| 設定                      | デフォルト | 効果                                                                                                                                | 影響を受ける領域 |
| ------------------------- | ---------- | ----------------------------------------------------------------------------------------------------------------------------------- | ---------------- |
| `ui.hideBanner`           | `false`    | バナー全体 (領域 A + B) を非表示にします。                                                                                          | A + B            |
| `ui.customBannerTitle`    | 未設定     | B①のブランドテキストを置き換えます。バージョンサフィックスは引き続き追加されます。トリミングされます。空文字列の場合は「デフォルトを使用」を意味します。 | B① ブランドテキスト |
| `ui.customBannerSubtitle` | 未設定     | B②の空白スペーサー行を1行のサブタイトルで置き換えます。サニタイズされ、160文字に制限されます。空の場合は「空白スペーサーを維持」を意味します。          | B② スペーサー    |
| `ui.customAsciiArt`       | 未設定     | 領域 A を置き換えます。3つの入力形式をサポートします（下記参照）。エラー時はデフォルトにフォールバックします。                             | A                |

設計上、**提供されない**もの:

- バージョンサフィックスのみを非表示にする設定はありません。
- 認証/モデル行のみを非表示にする設定はありません。
- パス行のみを非表示にする設定はありません。
- ロゴのグラデーションカラーを変更する設定はありません（テーマが管理します）。
- 情報パネルの順序や構造を変更する設定はありません。

将来的に実装でこれらのいずれかを公開する必要が生じた場合、上記3つのフィールドから派生するものではなく、独自の正当性を持つ新しいフィールドとして追加する必要があります。

## ユーザー設定ガイド — 変更方法

### 制限の概要

すべてのバナーカスタマイズにはいくつかの上限が適用されます。リゾルバが入力を切り詰めたり拒否したりしないよう、アートを手作りする前にこれらの制限を覚えておいてください。

| 項目                                   | 制限                                                                                                                                                                         |
| -------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **タイトルの文字数**                   | **最大80文字**（サニタイズ後）。それより長い場合は切り詰められ、`[BANNER]` 警告が記録されます。改行と制御文字はこの長さがカウントされる前に削除されます。                         |
| **サブタイトルの文字数**               | **最大160文字**（サニタイズ後）。タイトルと同じクリーンアップパイプライン。切り詰め時には同じ `[BANNER]` 警告が記録されます。                                                      |
| **ASCII アートブロックのサイズ**       | **最大200行 × 200列**（階層ごと）。それを超えると収まるように切り詰められ、`[BANNER]` 警告が記録されます。                                                                      |
| **ASCII アートファイルのディスクサイズ** | **最大64 KB**。それより大きなファイルは上限まで読み込まれ、残りは無視されます。                                                                                                  |
| **レンダリングされる ASCII アートの幅** | 起動時のターミナルの列数に依存し、**固定の文字数制限ではありません**。具体的な計算式とターミナルごとの数値については、下記の「ロゴの幅はどのくらいにできるか？」を参照してください。 |

ASCII アートに**固定の文字数制限はありません** — 上記の列/行の上限と起動ごとの幅の予算のみが適用されます。あるフォントでは問題なく収まる17文字のブランド名でも、別のフォントでは積み重ねやより密なフォントが必要になる場合があります。制限要因は文字数ではなく、視覚的な幅です。

### 設定の保存場所

4つの設定はすべて、`settings.json` の `ui` 以下にあります。ユーザーレベル（`~/.qwen/settings.json`）とワークスペースレベル（プロジェクトルートの `.qwen/settings.json`）の両方がサポートされており、標準的なマージの優先順位（ワークスペースがユーザーを上書き、システムがワークスペースを上書き）が適用されます。

`customAsciiArt` は特別な扱いを受けます。オブジェクト全体を1つの値として扱い、優先順位の高いスコープが置き換えるのではなく、リゾルバはスコープごとに階層を走査します。ユーザー設定が `{ small }` を定義し、ワークスペース設定が `{ large }` を定義している場合、両方が寄与します — `small` はユーザーから、`large` はワークスペースから。これにより、次の2つが同時に機能します。

1. 各 `{ path }` エントリは、それを宣言したファイル（ワークスペースの `.qwen/` またはユーザーの `~/.qwen/`）に対して相対的に解決されます。マージされたビューだけでは、このスコープ情報が失われます。
2. ユーザーは個人設定にデフォルトの `large` 階層を保持し、ワークスペースごとに `small` のみを上書きできます。オブジェクト全体を再記述する必要はありません。

同じ階層が複数のスコープで定義されている場合、通常の優先順位が適用されます（システム > ワークスペース > ユーザー）。いずれかのスコープで `customAsciiArt` を単なる文字列または `{ path }` に設定すると、そのスコープ内の両方の階層が埋められます。

### バナー全体を非表示にする

```jsonc
{
  "ui": {
    "hideBanner": true,
  },
}
```

起動時の出力では、ロゴカラムと情報パネルの両方がスキップされます。`ui.hideTips` も `true` に設定されていない限り、Tips は引き続き表示されます。

### ブランドタイトルを置き換える

```jsonc
{
  "ui": {
    "customBannerTitle": "Acme CLI",
  },
}
```

情報パネルに `Acme CLI (vX.Y.Z)` として表示されます。カスタムタイトルが設定されている場合、`>_` グリフは削除されます。もしグリフを残したい場合は、自分で含めてください：`"customBannerTitle": ">_ Acme CLI"`。

### ブランドサブタイトルを追加する

```jsonc
{
  "ui": {
    "customBannerSubtitle": "Built-in DataWorks Official Skills",
  },
}
```

サブタイトルは独自の行に、セカンダリテキスト色で、通常タイトルと認証/モデル行の間にある空白スペーサーの代わりにレンダリングされます。

```
┌─────────────────────────────────────────────────────────┐
│ DataWorks DataAgent (vX.Y.Z)                            │  ← B① タイトル
│ Built-in DataWorks Official Skills                      │  ← B② サブタイトル
│ Qwen OAuth | qwen-coder ( /model で変更)                │  ← B③ ステータス
│ ~/projects/example                                      │  ← B④ パス
└─────────────────────────────────────────────────────────┘
```

制約:

- 1行のみ。改行やその他の制御バイトは削除/スペースに変換され、貼り付けの事故で情報パネルのレイアウトが崩れるのを防ぎます。
- サニタイズ後、最大160文字に制限されます（タイトルの上限よりも緩やかです。タグラインや「powered by」行はやや長くなることが多いため）。
- フィールドを未設定のままにするか（または空文字列/空白に設定する）と、既存の空白スペーサー行が維持されます — デフォルトは後方互換です。
- サブタイトルはロックされた行には影響しません。サブタイトルの状態にかかわらず、認証、モデル、作業ディレクトリは常に表示されます。

### ASCII アートを置き換える — インライン文字列

```jsonc
{
  "ui": {
    "customAsciiArt": "  ___  _    _  ____ \n / _ \\| |  / |/ _\\\n| |_| | |__| | __/\n \\___/|____|_|___|",
  },
}
```

JSON 文字列内に改行を埋め込むには `\n` を使用します。アートはデフォルトのロゴと同じように、アクティブなグラデーションテーマでレンダリングされます。

> **ASCII アートが手元にありませんか？** 任意の外部ジェネレータを使用して、結果を貼り付けてください。最も簡単な方法は `figlet` です：
> `npx figlet -f "ANSI Shadow" "xxxCode" > brand.txt` を実行し、`customAsciiArt: { "path": "./brand.txt" }` でそれを参照します。CLI は実行時にテキストからアートをレンダリングしません — 理由については「スコープ外」のセクションを参照してください。

### ASCII アートを置き換える — 外部ファイル

```jsonc
{
  "ui": {
    "customAsciiArt": { "path": "./brand.txt" },
  },
}
```

複数行の文字列を JSON エスケープする必要がなくなります。パス解決ルール:

- **ワークスペース設定**: 相対パスはワークスペースの `.qwen/` ディレクトリからの相対パスとして解決されます。
- **ユーザー設定**: 相対パスは `~/.qwen/` からの相対パスとして解決されます。
- 絶対パスはそのまま使用されます。
- ファイルは**起動時に1回**読み込まれ、サニタイズされ、キャッシュされます。セッション中にファイルを編集してもバナーは再レンダリングされません — CLI を再起動してください。

### ASCII アートを置き換える — 幅に応じて切り替え

```jsonc
{
  "ui": {
    "customAsciiArt": {
      "small": "  ACME\n  ----",
      "large": { "path": "./brand-wide.txt" },
    },
  },
}
```

ターミナルの幅が十分に広い場合は `large` が優先され、そうでない場合は `small` が使用されます。どちらも使用できない場合はロゴカラムが非表示になります（既存の2カラムフォールバック）。各階層は文字列または `{ path }` のいずれかです。どちらかの階層を省略することもできます。省略された階層は単に次のステップにフォールスルーします。

### ロゴの幅はどのくらいにできるか？ — サイズ予算

タイトルやアートにハードな文字数制限はありません。**幅の予算**はターミナルの列数と、不正なファイルがレイアウトをフリーズさせるのを防ぐための絶対的なハードキャップによって決まります。

| 調整対象                                          | 制限値                                                               |
| ------------------------------------------------- | -------------------------------------------------------------------- |
| 起動時のターミナルの列数                          | ユーザーのターミナルが報告する値                                      |
| コンテナ外側のマージン                            | 4 列 (左2 + 右2)                                                     |
| ロゴと情報パネルの間のギャップ                    | 2 列                                                                 |
| 情報パネルの最小幅                                | 44 列 (パス40 + 境界線 + パディング)                                  |
| **利用可能なロゴ幅** (階層ごと、レンダリング時)      | `terminalCols − 4 − 2 − 44 = terminalCols − 50`                      |
| 各アート階層のハードキャップ（サニタイズ後）       | 200列 × 200行。それを超えると切り詰め + `[BANNER]` 警告。            |
| `customBannerTitle` のハードキャップ（サニタイズ後） | 80 文字。それを超えると切り詰め + `[BANNER]` 警告。                  |

一般的なターミナル幅での予算の読み取り:

| ターミナル列数 | レンダリングされる最大ロゴ幅 | 実際の意味                                                                        |
| -------------- | ---------------------------- | --------------------------------------------------------------------------------- |
| 80             | 30                           | ほとんどの figlet "ANSI Shadow" の文字は約7～11列 — 最大3文字。                     |
| 100            | 50                           | ANSI Shadow の短い単語（約6文字）、または2つの短い単語を積み重ねたもの。             |
| 120            | 70                           | 積み重ねた複数行のワードアートが快適に収まります。                                |
| 200            | 150                          | ANSI Shadow の完全な製品名のような長いインライン文字列が収まります。                |

アートをデザインする際の2つの実用的な意味:

1. **複数単語のブランドは、ほとんどのターミナルで1行の ANSI Shadow としてレンダリングされません。** ANSI Shadow の文字あたり約7～9列の場合、`Custom Agent` のような12文字のブランドでも1行で約95列のアートになります — これは100列のターミナルでは情報パネルと並べて表示できる幅をすでに超えています。単語を複数行に積み重ねるか、より密度の高い figlet フォントを選ぶか、`▶ Custom Agent ◀` のようなコンパクトな1行テキスト装飾を使用してください。
2. **幅に応じて切り替える `{ small, large }` 形式を使用する**と、1つの階層だけで「広いときは見栄えが良いが狭いと使えない」と「狭いときは見栄えが良いが広いときにスペースを無駄にする」の選択を強いられる状況を回避できます。以下の例では、`large` で ≥104列のターミナル向けに単語を積み重ね、`small` で16列の1行装飾にフォールスルーしています。

```jsonc
{
  "ui": {
    "customBannerTitle": "Custom Agent",
    "customAsciiArt": {
      "small": "▶ Custom Agent ◀",
      "large": { "path": "./banner-large.txt" },
    },
  },
}
```

ここで、`banner-large.txt` には積み重ねた単語の ANSI Shadow 出力（約54列 × 12行）が含まれています。例えば、次のように生成されます:

```bash
( npx figlet -f "ANSI Shadow" CUSTOM
  npx figlet -f "ANSI Shadow" AGENT ) > banner-large.txt
```

### 3つすべてを組み合わせる

```jsonc
{
  "ui": {
    "hideBanner": false,
    "customBannerTitle": "Acme CLI",
    "customAsciiArt": {
      "small": "  ACME\n  ----",
      "large": { "path": "./brand-wide.txt" },
    },
  },
}
```

### 変更を確認する方法

1. `settings.json` を保存し、新しい `qwen` セッションを開始します — バナーの解決は起動時に1回実行されます。
2. ターミナルをリサイズして、`small` / `large` の階層が期待通りに切り替わること、および非常に狭い幅でロゴカラムが消えることを確認します。
3. 何かが期待通りに表示されない場合は、`~/.qwen/debug/<sessionId>.txt`（シンボリックリンク `latest.txt` が現在のセッションを指しています）を参照し、`[BANNER]` を grep してください — すべてのソフト障害は、根本的な理由とともに警告行を記録します。

## 解決パイプライン

```
   settings.json                              packages/cli/src/ui/components/
   ─────────────                              ──────────────────────────────
   {                                          AppHeader.tsx
     "ui": {                                    │
       "hideBanner": false,                     │  showBanner =
       "customBannerTitle": "Acme",             │      !screenReader
       "customBannerSubtitle": "Built-in …",    │   && !ui.hideBanner
       "customAsciiArt": …                      │
     }                                          │
   }                                            ▼
        │                              <Header
        ▼                                customAsciiArt={resolved.asciiArt}
   loadSettings()                        customBannerTitle={resolved.title}
   merge user / workspace                customBannerSubtitle={resolved.subtitle}
        │                                version=… model=… authType=…
        ▼                                workingDirectory=… />
   resolveCustomBanner(settings)                  │
   ┌─────────────────────────┐                    ▼
   │ 1. { small, large } に   │         packages/cli/src/ui/components/
   │    正規化               │         Header.tsx
   │ 2. 各階層を解決:        │           │
   │    string → そのまま    │           │ 利用可能なターミナル幅に
   │    {path} → fs.read     │           │ 応じて階層を選択
   │      O_NOFOLLOW          │           ▼
   │      ≤ 64 KB             │         ロゴカラムをレンダリング
   │ 3. アートをサニタイズ:  │         情報パネルをレンダリング:
   │    制御シーケンス削除   │           タイトル    = customBannerTitle
   │    ≤ 200 行 × 200 列    │                   ?? '>_ Qwen Code'
   │ 4. タイトル + サブタイトル│           サブタイトル= customBannerSubtitle
   │    をサニタイズ         │                   ?? 空白スペーサー行
   │    (1行、≤ 80 / 160 文字)│           ステータス  = ロック
   │ 5. ソースごとにメモ化   │           パス      = ロック
   └─────────────────────────┘
```
設定が読み込まれたときに5段階の解決アルゴリズムが一度実行され、設定の再読み込みイベントでのみ再度実行されます：

1. **正規化**。単なる `string` または `{ path }` は `{ small: x, large: x }` になります。`{ small, large }` オブジェクトはそのまま通過します。
2. **各階層の解決**。各 `AsciiArtSource` に対して：
   - 文字列の場合はそのまま使用します。
   - `{ path }` の場合は、`O_NOFOLLOW` 防御を適用してファイルを同期的に読み取ります（Windows: 単なる読み取り専用 — 定数は公開されていません）。上限は64 KBです。相対パスは、*所有する設定ファイルのディレクトリ* を基準に解決します。ワークスペース設定はワークスペース `.qwen/` に対して、ユーザー設定は `~/.qwen/` に対して解決します。読み取りに失敗した場合は `[BANNER]` 警告が記録され、その階層のデフォルトにフォールバックします。
3. **サニタイズ**。バナー固有のストリッパーがOSC / CSI / SS2 / SS3のリーダーを削除し、その他のC0 / C1制御バイト（およびDEL）をスペースに置き換えますが、`\n` は保持されるため、複数行のアートはそのまま残ります。行末の空白をトリミングし、その後200行×200列にキャップします。上限を超えた部分は切り捨てられ、`[BANNER]` 警告が記録されます。
4. **レンダリング時の階層選択**。`Header.tsx` では、解決された `small` と `large` を基に、既存の幅予算（`availableTerminalWidth ≥ logoWidth + logoGap + minInfoPanelWidth`）を評価します：
   - 収まる場合は `large` を優先。
   - 収まらない場合は、収まるなら `small` にフォールバック。
   - 収まらない場合、**ユーザーがカスタムアートを提供した場合**は、ロゴ列を完全に非表示にします（既存の `showLogo = false` ブランチ）— ここでバンドルされているQWENロゴにフォールバックすると、狭い端末でのホワイトラベルデプロイが無言で元に戻されてしまいます。情報パネルは引き続きレンダリングされます。
   - 収まらない場合（カスタムアートがまったく提供されなかった場合）は `shortAsciiLogo` にフォールスルーし、既存の幅ゲートによってデフォルトロゴを表示するかどうかが決定されます。
5. **フォールバック**。ソフト障害（ファイル欠落、サニタイズですべて拒否、設定の不正など）により両方の階層が空または無効になった場合は、カスタマイズが設定されていないかのように動作します。`shortAsciiLogo` をレンダリングし、デフォルトロゴの幅ゲートに従います。CLIはバナー設定エラーでクラッシュしてはいけません。

階層選択の擬似コード：

```ts
function pickTier(
  small: string | undefined,
  large: string | undefined,
  availableWidth: number,
  logoGap: number,
  minInfoPanelWidth: number,
): string | undefined {
  for (const candidate of [large, small]) {
    if (!candidate) continue;
    const w = getAsciiArtWidth(candidate);
    if (availableWidth >= w + logoGap + minInfoPanelWidth) {
      return candidate;
    }
  }
  return undefined; // ロゴ列を非表示
}
```

## 検討した代替アプローチ

この機能の5つの形状が検討されました。将来の貢献者が設計空間を理解し、制約が変わった場合に選択を再検討できるように、ここにリストされています。

### オプション1 — 3つのフラット設定（推奨、issueと一致）

```jsonc
{
  "ui": {
    "customAsciiArt": "...", // string | {path} | {small,large}
    "customBannerTitle": "Acme CLI",
    "hideBanner": false,
  },
}
```

- **効果**: ユーザー向けの表面を最小限に抑える。まさにissueが要求するもの。
- **長所**: 学習曲線ゼロ。文書化が簡単。既存のフラットな `ui.*` プロパティ（`hideTips`、`customWittyPhrases` など）と一貫性がある。
- **短所**: 概念的には一緒に属する3つのトップレベルキーがグループ化されていない。将来のバナー専用ノブ（グラデーション、サブタイトル）は、ネスト化ではなく `ui` にさらに多くの兄弟を追加することになる。

### オプション2 — ネストされた `ui.banner` 名前空間

```jsonc
{
  "ui": {
    "banner": {
      "hide": false,
      "title": "Acme CLI",
      "asciiArt": { "path": "./brand.txt" },
    },
  },
}
```

- **効果**: オプション1と同じ機能だが、機能ごとに整理される。
- **長所**: 将来のバナー専用ノブのためのクリーンな名前空間。`/settings` 経由での発見が容易。
- **短所**: issueの正確な表現から逸脱する。既存のUI設定はほとんどがフラット（`ui.accessibility` と `ui.statusLine` のみネスト）であり、一貫性が混在している。ユーザーが覚えるネストレベルが1つ増える。

### オプション3 — バナープロファイルプリセット + スロットオーバーライド

```jsonc
{
  "ui": {
    "bannerProfile": "minimal" | "default" | "branded" | "hidden",
    "banner": { /* 'branded' のスロットオーバーライド */ }
  }
}
```

- **効果**: ユーザーは名前付きプリセットから選択。高度なユーザーは選択したプロファイル内のスロットをオーバーライド。
- **長所**: 優れたオンボーディングUX。プリセットはCLIに同梱。
- **短所**: かなりの複雑さ。プリセットはメンテナンスのコミットメント。issueはキュレーションではなく生のカスタマイズを要求している。

### オプション4 — バナー全体のオーバーライド（単一文字列テンプレート）

```jsonc
{
  "ui": {
    "bannerTemplate": "{{logo}}\n>_ {{title}} ({{version}})\n{{auth}} | {{model}}\n{{path}}",
  },
}
```

- **効果**: ロックされた変数が埋め込まれた単一の自由形式テンプレート。
- **長所**: 非標準レイアウトに対して最大の柔軟性。
- **短所**: ユーザー空間でレイアウトを再実装する。Inkの2列ウィンドウサイズ耐性を失う。狭い端末で壊れるテンプレートを簡単に書けてしまう。小さな機能に対して影響範囲が大きすぎる。

### オプション5 — プラグイン/フックAPI

拡張システムを通じてバナーレンダラーフックを公開。

- **効果**: コードレベルのカスタマイズ。拡張機能は何でもレンダリング可能。
- **長所**: 最大のパワー。企業は封印されたブランディングプラグインを出荷可能。
- **短所**: 大規模なAPIサーフェス。任意の端末レンダリングのセキュリティレビューが必要。issueには過剰にスコープが大きすぎる。

### 推奨

**オプション1** が推奨されます。issueをそのまま満たし、既存の `ui.*` スタイルに適合し、他のバナー専用ノブが実際にどのようになるかを知る前にネスト名前空間の決定を強制しません。将来的に兄弟が蓄積され始めた場合、オプション2への移行は追加的です。`ui.banner.title` と `ui.customBannerTitle` は非推奨期間中に共存できます。

## セキュリティと障害処理

カスタムバナーコンテンツは端末にそのままレンダリングされ、パス形式ではディスクから読み取られます。両方のサーフェスは、悪意のあるまたは改ざんされた設定ファイルが読み込まれた場合に攻撃可能です。セッションタイトル機能を駆動するのと同じ脅威モデルがここにも適用されます。

| 懸念事項                                                 | ガード                                                                                                                                                                                                             |
| ------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| アート、タイトル、サブタイトルへのANSI / OSC-8 / CSIインジェクション | バナー固有のストリッパー（`sanitizeArt` / `sanitizeSingleLine`）：OSC / CSI / SS2 / SS3リーダーを削除し、その他のC0 / C1制御バイト（およびDEL）をスペースに置き換えます。レンダリングおよびキャッシュ書き込みの前に適用されます。 |
| 巨大ファイルによる起動フリーズ                           | ファイル読み取りに64 KBのハードリミット。                                                                                                                                                                                     |
| 病理学的なアートによるレイアウトフリーズ                 | 解決された各文字列に対して200行×200列の上限。超過分は切り捨てられ、`[BANNER]` 警告が記録されます。                                                                                                               |
| パス形式でのシンボリックリンクリダイレクト                 | ファイル読み取り時に `O_NOFOLLOW`（Windows: 単なる読み取り専用。定数は公開されていません）。                                                                                                                      |
| ファイルの欠落または読み取り不可                         | 捕捉し、`[BANNER]` 警告を記録し、デフォルトにフォールバック。UIにスローされることはありません。                                                                                                                        |
| 改行を含むタイトルまたはサブタイトル / 長すぎる         | 改行はスペースに変換。80文字（タイトル）/ 160文字（サブタイトル）に制限。                                                                                                                                      |
| 信頼されていないワークスペースによるレンダリングやファイル読み取りの操作 | `settings.isTrusted` が false の場合、リゾルバーは `settings.workspace` を完全にスキップします（`settings.merged` が適用する信頼ゲートをミラーリング）。                                                             |
| 設定の再読み込み時の競合                                   | 解決は呼び出しごとにソース（パスまたは文字列ハッシュ）によってメモ化されます。再読み込み時にリゾルバーが再実行され、影響を受けるファイルが再読み取りされます。                                                          |

障害モードの概要：すべてのソフト障害は `shortAsciiLogo`（またはロックされたデフォルトタイトル）とデバッグログ警告で終了します。ハード障害（スローされたエラー）はリゾルバーのどのブランチでも許可されていません。

| ファイル変更時の動的再読み込み                                      | リゾルバーは起動時と設定の再読み込み時のみ実行されます。セッション中のアート変更は十分に稀であり、「再起動して反映」が許容可能なトレードオフです。                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                |

## 検証計画

最終的な実装 PR では、以下のエンドツーエンドのチェックが合格する必要があります。

1. `~/.qwen/settings.json` に `customBannerTitle: "Acme CLI"` とインラインの `customAsciiArt` 文字列が設定されている → `qwen` が新しいタイトルとアートを表示する。バージョンサフィックスは引き続き表示される。
2. `customBannerSubtitle: "Built-in Acme Skills"` → サブタイトル行がタイトルと認証/モデル行の間にセカンダリテキスト色でレンダリングされる。認証、モデル、パスは引き続き表示される。設定を解除すると、空白のスペーサー行に戻る（後方互換性）。
3. `hideBanner: true` → `qwen` がバナーなしで起動する。ヒントとチャットは通常通りレンダリングされる。
4. ワークスペースの `settings.json` に `customAsciiArt: { "path": "./brand.txt" }` が設定され、`.qwen/` 内に `brand.txt` が存在する → ワークスペースを開いたときにディスクから読み込まれる。
5. `customAsciiArt: { "small": "...", "large": "..." }` → ターミナルの幅を広/中/狭に変更する。広幅では large が、中幅では small が使用され、狭幅ではロゴ列が非表示になり、情報パネルは常に表示される。
6. `customBannerTitle` と `customBannerSubtitle` に `\x1b[31mhostile` を注入する → 両方ともリテラルテキストとしてレンダリングされ、赤色としては解釈されない。
7. `path` に存在しないファイルを指定する → CLI は起動する。`~/.qwen/debug/<sessionId>.txt` に `[BANNER]` 警告が表示され、デフォルトのアートがレンダリングされる。
8. ワークスペーストラストをオフにしてワークツリーを開く → ワークスペースで定義された `customAsciiArt`（`{ path }` エントリを含む）は静かに無視される。ユーザースコープの設定は引き続き適用される。
9. `npm test` と `npm run typecheck` が CLI パッケージでパスする。`customBanner.test.ts` の単体テストが、受け入れ可能な各形状と各失敗パス（ファイル欠落、ファイルサイズ超過、ANSI インジェクション、不正なオブジェクト）をカバーしている。

# Qwen Code 0.15.11 デフォルト Heap OOM 負荷テストレポート

日付：2026-05-21

## テスト範囲

本レポートは、Qwen Code `0.15.11` の最新ローカルビルドに対する、デフォルトヒープでの負荷テスト（stress test）の結果を記録したものです。
このテストの目的は、人為的にメモリ上限を引き下げない状態で、現在のコードが issue で報告された長セッション OOM を再現できるかどうか、またより極端な大出力シナリオにおいて新たなリスクが存在するかを検証することです。

今回のテストでは、以下の3つのモデルを対象としました。

- `pai/glm-5`
- `qwen3.6-plus`
- `DeepSeek/deepseek-v4-pro`

テストは2つのパートに分かれています。

1.  実際の長タスク、マルチエージェント並行レビューループ。
2.  amplified foreground stdout 負荷テスト：大規模なフォアグラウンドシェル stdout を用いて tool-output パスへの負荷を増幅。

## 核心的結論

最新のローカルビルドは、issue が最も懸念する「長セッション V8 heap OOM」のパスにおいて、明らかに優れたパフォーマンスを示しています。
今回のデフォルトヒープ、マルチモデル、マルチエージェント、長タスクの負荷テストに基づくと、本 PR は以前遭遇した long-session heap OOM 問題を基本的に解決しており、少なくとも現在の再現次元では元の heap OOM を再現できないと言えます。

実際の長タスク、マルチエージェント並行テストは、合計で以下を実行しました。

- 23 の worker turn
- 約 `719,094,118` の reported total tokens
- 77 回の agent tool call
- 856 回の総 tool call

この部分では、従来の V8 heap OOM の特徴は一切再現されませんでした。

- `JavaScript heap out of memory`
- `Reached heap limit`
- `Ineffective mark-compacts near heap limit`
- `Allocation failed`

実際の長タスクフェーズにおける最大 process-tree RSS は `874.7 MiB`、最大 root-process RSS は `219.1 MiB` でした。これは、デフォルトヒープの下で、現在のコードが元の issue に見られたような長タスクで停止する heap OOM を容易には再現しないことを示しています。

第二フェーズの amplified stdout 負荷テストはより過激です。合計18回のペイロード試行を実行し、3つのモデルと `128 MiB` から `2048 MiB` までのフォアグラウンド stdout ペイロードをカバーしました。

結果は以下の通りです。

- 3つのモデル全てが `1536 MiB` ペイロードを成功裏に処理しました。
- 成功した最大 process-tree RSS は `5964.7 MiB` で、`qwen3.6-plus` の `1536 MiB` ペイロード時に記録されました。
- `2048 MiB` ペイロード時に、新たな extreme large-output failure が発生しました。

`2048 MiB` の結果：

- `pai/glm-5`：`exit=1`、stdout は空、標準的な OOM テキストはなし。
- `qwen3.6-plus`：`exit=1`、stdout は空、標準的な OOM テキストはなし。
- `DeepSeek/deepseek-v4-pro`：V8 fatal が発生：`Check failed: i::kMaxInt >= len`、スタックは `v8::String::NewFromOneByte` / `node::StringBytes::Encode` / `DecodeUTF8`。

この新しい問題は、元の issue における従来の long-session heap OOM ではありません。これは、マルチギガバイトのフォアグラウンド stdout がデコード/JS 文字列に構築される際に発生する、V8 の文字列長制限または大出力処理の問題のように見えます。これは、現在の長セッションのヒーププレッシャー修復の失敗として扱うのではなく、large-output フォローアップとして追跡することを推奨します。

各モデルワーカーは同じセッションを再利用し、`--resume` を繰り返します。各ラウンドで Qwen Code に要求する内容：

- 読み取り専用のコードレビューとコード検索を実行すること。

### Phase 1 の解釈

これは、今回のテストにおいて、元の long-session OOM が明らかに改善されたことを最もよく示すデータです。

このテスト群は、5月18日の小規模な PR レビュー / code navigation よりも負荷が高く、より多くの `--resume`、より多くの subagent アクティビティ、より多くの reported token 量、そしてより多くの tool call を含んでいます。しかし、process-tree RSS は一貫して `0.9 GiB` 未満に留まり、従来の V8 heap OOM も発生しませんでした。

これは、すべてのユーザー OOM が今後発生しないことを証明するものではありませんが、少なくとも現在のビルドがデフォルトヒープの下で、元の issue に見られたような長セッションのヒーププレッシャー OOM を容易には再現できないことを示しています。

## Phase 2：Amplified Foreground Stdout 負荷テスト

### テスト形態

第二フェーズでは、シェル出力パスへの負荷を意図的に増幅します。各モデル、各ペイロードサイズにおいて、親セッションと並行エージェントにフォアグラウンドシェルコマンドを実行させ、大量の `x` を stdout に出力させます。

```bash
node -e "const chunk='x'.repeat(1024*1024); for (let i=0; i<N; i++) process.stdout.write(chunk)"
```

ペイロードサイズ：

- `128 MiB`
- `256 MiB`
- `512 MiB`
- `1024 MiB`
- `1536 MiB`
- `2048 MiB`

これは合成的なストレス負荷であり、通常のコードレビューワークロードではありません。その目的は、極めて大きなフォアグラウンド stdout が Qwen Code を危険なメモリパスに押し込むかどうかを検証することです。

### RSS が Node Heap Limit を超える理由

このマシンの Node heap limit は約 `4144 MiB` ですが、process-tree RSS は最大約 `5964.7 MiB` に達しています。これは矛盾しません。

- RSS には、V8 heap、external buffers、native allocations、loaded modules、子プロセス、そして old-space heap に計上されないメモリが含まれます。
- amplified フェーズの最大ピークは通常、ルートラッパーではなく、子 Node プロセスで発生します。
- そのため、root RSS は低く抑えられつつ、process-tree RSS は非常に高くなることがあります。

## 過去のレポートとの比較

5月18日の maintainer benchmark では、通常タスクにおける Qwen の process-tree RSS ピークは概ね `942.5 MiB` から `1006.6 MiB` の範囲でした。これらのデータは主に、当時の Qwen Code のタスク実行時フットプリントが Claude Code よりも著しく高かったことを示しています。

5月19日の runtime diagnostics report では、さらに2つの系統に分類されました。

- startup/config RSS：通常の設定と MCP 子プロセスによって押し上げられることが多い。
- long-session heap OOM：history / compaction / clone pressure に起因する傾向がより強い。

今回の5月21日のデータは、この分類をさらに裏付けています。

- 実際の長タスク、マルチエージェント、長セッションのシナリオでは、`0.9 GiB` 未満で安定しており、heap OOM に達しなかった。
- amplified stdout は process-tree RSS を `5.96 GiB` まで押し上げ、`2048 MiB` では別の種類の V8 string-length fatal を引き起こした。

したがって、元の issue に見られた従来の long-session heap OOM と比較して、現在のビルドは明らかに安定しています。しかし、これは Qwen Code に large-output メモリリスクが全くないことを意味するわけではありません。

## 新たな発見：極端な Foreground Stdout Fatal Path

今回新たに発見された問題は、元の heap OOM ではなく、マルチギガバイトのフォアグラウンド stdout 下での文字列/デコード fatal です。

```text
Fatal error in , line 0
Check failed: i::kMaxInt >= len.
...
v8::String::NewFromOneByte
node::StringBytes::Encode
node::encoding_binding::BindingData::DecodeUTF8
```

トリガー条件：

- Model：`DeepSeek/deepseek-v4-pro`
- Payload：`2048 MiB`
- Peak tree RSS：`4660.4 MiB`
- Largest process RSS：`4527.6 MiB`
- runner 記録の exit：`SIGTERM`。fatal 出力がキャプチャされた後、残りの子プロセスが高い CPU でアイドル状態になったため、手動で終了されました。

`pai/glm-5` と `qwen3.6-plus` も `2048 MiB` で失敗し、stdout が空で exit code `1` を示しましたが、stderr には V8 fatal スタックがキャプチャされませんでした。

### 深刻度

これは実際のロバスト性問題ですが、トリガー条件はマルチギガバイトのフォアグラウンド stdout であり、通常のコードレビュータスクではありません。また、現在の long-session heap-pressure 修復の失敗を示すものでもありません。

### 本 PR によって導入されたものか？

今回のテストでは、`2048 MiB` stdout 障害が現在のメモリ PR によって導入された回帰であるという証拠はありません。

理由：

- 障害パスはフォアグラウンドシェル stdout のデコード/文字列構築です。
- 元の issue のパスは long-session history、compaction、clone pressure です。
- 今回のテストでは、同一ペイロードでの PR 前のベースラインを取っていないため、回帰と断定できません。
- この障害は、意図的に極端な `2048 MiB` ペイロードでのみ発生し、`128 MiB` から `1536 MiB` までは全て完了しました。

これは、専用の large-output フォローアップとして扱うことを推奨します。具体的には、フォアグラウンドシェル出力をより早期にストリーム/スプール/ハードキャップし、メモリ内でマルチギガバイトの JS 文字列を構築しないようにします。現在の PR の目的が「任意のマルチギガバイトのフォアグラウンド stdout を処理可能でなければならない」というものでない限り、これを現在の PR のブロッカーとして扱うことは推奨しません。

## 結論

1.  最新のローカル `0.15.11` ビルドは、issue で報告された long-session heap OOM の方向において明らかに改善されています。現在のデフォルトヒープでの負荷テスト結果に基づくと、本 PR は以前遭遇した long-session heap OOM の再現パスを基本的に解決したと見なせます。

2.  デフォルトの Node heap 下で、実際の長タスク + マルチエージェントレビューループは、`pai/glm-5`、`qwen3.6-plus`、`DeepSeek/deepseek-v4-pro` の3つのモデルにおいて、従来の V8 heap OOM を再現しませんでした。

3.  合成された foreground stdout 負荷テストは、依然として process-tree RSS を非常に高く押し上げる可能性があります。現在のビルドは3モデル全てで `1536 MiB` ペイロードを処理し、最大成功ツリー RSS は `5964.7 MiB` でした。

4.  独立した極端な大出力問題が依然として存在します。`2048 MiB` stdout 付近で、Qwen Code は JSON 結果を出力する前に失敗する可能性があります。DeepSeek のケースでは V8 string-length fatal がキャプチャされました。

5.  この新たな発見は重要ですが、これはむしろ後続の large-output ロバスト性の問題であり、long-session heap-pressure 軽減策のブロッカーとして直接扱うべきではありません。

# Qwen Code ランタイムメモリベンチマークレポート

日付: 2026-05-18

## サマリー

このレポートは、Qwen Code のランタイム動作に関するローカルメモリベンチマークを記録したものです。Qwen Code を複数のモデル間で比較し、同等のモデルエンドポイントが利用可能だった場合には、同じタスク形状で Qwen Code と Claude Code の比較も行っています。

最新のマトリックス（セルごとに1回の実行、統計的に反復されたものではない）を通じて、主要な結果は一貫しています。

- Qwen Code プロセスツリー RSS ピーク: 約 `852-1062 MiB` (`0.83-1.04 GiB`)
- Claude Code プロセスツリー RSS ピーク: 約 `279-366 MiB` (`0.27-0.36 GiB`)
- テストした非インタラクティブ CLI タスクベンチマークにおいて、Qwen Code は約 `2.3x-3.6x` 高い値でした。

注: プロセスツリー RSS には MCP 子プロセス（Qwen 側で約350 MiB のオーバーヘッド）が含まれます。このため絶対値は大きくなりますが、両 CLI を同じ方法で測定しているため、相対比較は有意義です。

この差は、小さな PR レビュー、コードナビゲーション、および合成 diff ワークロードでも再現しました。したがって、単一の大きな PR や単一のモデルプロバイダーだけで説明できる可能性は低いです。

このレポートの目的は、現在のパフォーマンス調査の状況を可視化することです。何が測定され、どのような結論がすでに裏付けられており、何が未知で、次にどの診断を追加すべきかを示します。

プロセスツリー RSS を主要指標とした理由は、Qwen Code がルートラッパーと子 Node/Qwen ワーカーを起動するためです。ルートプロセスだけを見ると、ユーザーが体感するメモリフットプリントを過小評価する可能性があります。

マトリックス実行には一時的な CLI 設定ディレクトリを使用したため、ベンチマークはグローバルな CLI 状態に依存しません。

## ベンチマーク成果物

この統合レポートの前に、5つのローカルレポートが作成されました。

1. Qwen Code PR レビューメモリ実行
2. Qwen Code モデル比較実行
3. 厳密な Qwen Code と Claude Code の比較（`pai/glm-5` を使用）
4. Qwen Code と Claude Code、2つの CLI × 2つのモデル
5. Qwen Code と Claude Code、5ケースのマトリックス

この統合レポートでは、5つのレポートすべての結論と主要指標をカバーしています。生のサンプル行、ターミナルのトランスクリプト、一時的なランナー成果物はすべて埋め込んでいません。これらの生の成果物は、安定したリポジトリのフィクスチャではなく実験出力であるため、ローカルの `tmp/` ディレクトリに残しています。

最新のマトリックスが最も強力な証拠です。なぜなら、単一の PR レビューワークロードだけでなく、複数のタスク形状をカバーしているからです。

## 暫定的な結論

現在のデータからは、これらのローカル非インタラクティブ CLI タスクベンチマークにおいて、Qwen Code が Claude Code よりもランタイムメモリフットプリントが高いと断言するには十分です。ただし、最終的な根本原因を1つ特定するにはまだ十分ではありません。

有力な説明は、モデルプロバイダーの違いではなく、Qwen Code のランタイム/パスの違いです。

- `pai/glm-5` と `qwen3.6-plus` の両方で差が再現する。
- 大きな diff タスクだけでなく、小さな PR やコードナビゲーションタスクでも差が再現する。
- Qwen Code は、同様の作業に対して Claude Code よりも多くのトークンを繰り返し送信または計上している。
- Qwen Code で観測された最大のコンポーネントは子 Node/Qwen ワーカープロセスであり、タスク時のプロセスフットプリント、モジュール読み込み、コンテキストアセンブリ、ライブ履歴、ツール結果の保持、サブエージェント/保存出力パスを示唆している。

したがって、次に最も有用な測定は、単なる外部 RSS のみの実行ではありません。次の測定では、RSS を V8 ヒープ、ネイティブメモリ、セッション/履歴サイズ、保持されたツール結果サイズ、サブエージェント/プロセスツリーアクティビティに分割すべきです。

## 初期原因分析

ベンチマークはまだ根本原因を1つに特定していませんが、問題領域を絞り込んでいます。

| シグナル                                                                                       | 示唆すること                                                                                   | 証明しないこと                                                                                                |
| ---------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------- |
| 小さな PR やコードナビゲーションケースでも Qwen が約 `1 GiB` に留まる                             | 非インタラクティブタスク時のランタイムコストが高い可能性が高い                                 | フットプリントが V8 ヒープ、ネイティブメモリ、モジュール読み込み、保持状態のいずれかであるかを特定しない     |
| Diff サイズが 100 KiB から 5 MiB に変化しても RSS は線形にスケーリングしない                     | 生の diff バイトだけが主要なドライバーではおそらくない                                         | 大きな出力は実際の PR レビューフローでメモリを増幅させる可能性がある                                          |
| Qwen はすべてのマトリックスセルで Claude よりも多くのトークンを使用する                           | Qwen は同様の作業に対してより大きなプロンプト/コンテキスト/ツール結果状態を構築または保持している可能性が高い | トークン数はプロセスメモリと同じではなく、原因ではなく結果である可能性がある                                  |
| ツール呼び出し数は同程度で、Claude がより多くのターン/ツール呼び出しを使用しながらも RSS が低い場合がある | より長いツール呼び出しチェーンだけが主な説明である可能性は低い                               | ツール出力サイズと保持はまだ測定する必要がある                                                                |
| 以前の大規模 PR 実行では、保存出力の復元とサブエージェントの増幅が見られた                         | ツール出力のトランケーションと保存出力パスは、おそらく高負荷ワークロードでの増幅要因である       | 小さなタスクの実行フットプリント全体を説明するものではない                                                    |

したがって、現在の最善の説明は次のとおりです。

1. **タスク時のランタイムコストが第一**: Qwen Code は、非インタラクティブ CLI タスクの実行中に、Claude Code よりも多くのランタイム状態を初期化または保持している可能性が高い。これには、エージェントランタイム、ツールレジストリ、プロバイダーアダプター、セッションサービス、または短い非インタラクティブタスクには厳密には不要な UI/履歴構造が含まれる可能性がある。
2. **コンテキスト/ツール結果のボリュームが第二**: Qwen Code は、同様の作業に対して、より大きなモデル向けまたはセッション向けのコンテキストを保持しているように見える。トークンギャップは、コンテキストアセンブリ、ツール結果の正規化、履歴保持を重要な疑いのあるものにしている。
3. **大きな出力による増幅が第三**: 大規模な PR レビューは、追加の保存出力パスやサブエージェントパスをトリガーする可能性がある。これらはおそらく唯一の原因ではないが、現実的なレビュータスクではメモリとトークンの負荷を悪化させる可能性がある。

次の診断実行では、`~1 GiB` がどこにあるかを回答すべきです。

- 起動直後に高い: モジュール/ランタイムの起動コスト
- ツール実行後にジャンプ: ツール出力の保持または結果の正規化
- リクエストアセンブリ中にジャンプ: コンテキスト構築または重複した履歴
- ストリーミング/圧縮後に増加: レスポンス保持または圧縮状態
- RSS が V8 ヒープ外にほとんどある: ネイティブバッファ、読み込まれたモジュール、外部メモリ

## トークン使用量分析

トークンギャップは最も強力な手がかりの1つですが、根本原因として扱うには内部リクエストメトリクスが必要です。

現在のデータがサポートする点:

- Qwen Code はすべてのマトリックスセルで Claude Code よりも多くの総トークンを使用した。
- ツール呼び出し数が同程度でもギャップが現れている。
- Claude がより多くのターンやツール呼び出しを使用しても、メモリは少ない場合がある。

これが示唆すること:

- トークンの差は、より長いツール呼び出しチェーンだけから生じている可能性は低い。
- Qwen は、より大きな静的プロンプト/コンテキスト状態、より大きなツールスキーマ、より大きなシリアル化されたツール結果、またはより多くの保持された会話/セッションコンテンツを保持している可能性がある。
- 大出力フローでは、トランケーション、保存出力の復元、またはサブエージェントパスによって別のレイヤーが追加される可能性がある。

まだ不足している点:

- リクエストごとの入力トークンの内訳
- システムプロンプトとツールスキーマのトークンサイズ
- 各モデルリクエスト前の保持されたメッセージとツール結果のサイズ
- 大きな出力がモデル履歴、UI 履歴、セッション記録、保存出力ストレージなどの複数の場所に保持されているかどうか

これらの不足メトリクスが、次のステップで外部 RSS ベンチマークを繰り返すのではなく、内部診断を追加すべき理由です。

## 作業仮説

現在のエビデンスは、以下の仮説を優先順位順に支持しています。

1. Qwen Code は Claude Code よりも非インタラクティブタスク時のプロセスフットプリントが高い。Qwen の子 Node ワーカーはローカルサンプリングで通常最大のプロセスであり、多くの場合約 `0.7-0.8 GiB` でした。
2. モデルの選択が主な説明ではない。`pai/glm-5` と `qwen3.6-plus` の両方で、Qwen と Claude の間に同じ大まかなギャップが見られた。
3. 大きな diff サイズだけが主な説明ではない。合成 diff サイズは 100 KiB から 5 MiB に線形にスケーリングしなかった。これはおそらく、ツール出力のトランケーションがモデルに到達する出力を制限するためです。
4. コンテキスト/ツール結果の処理は依然として寄与要因である可能性が高い。Qwen Code はすべてのマトリックスセルで Claude Code よりも多くのトークンを使用しており、以前の大規模 PR 実行では、保存されたツール出力の復元とサブエージェントの増幅パスが示された。
5. 次の診断レイヤーでは、V8 ヒープ、ネイティブ RSS、読み込まれたモジュール/ランタイムの起動コスト、セッション履歴、UI 履歴、ツール結果の保持、サブエージェントアクティビティを分離すべき。外部 RSS だけではこれらの原因を区別できない。

## 注意点

- これらはマトリックスセルごとに1回の実行であり、統計的に反復されたサンプルではありません。
- RSS は外部プロセス RSS です。V8 ヒープ、ネイティブバッファ、モジュール読み込み、保持されたツール出力、UI 状態、セッション履歴を区別できません。
- Claude Code と Qwen Code は、同じモデルラベルの場合でも、異なるランタイム実装とプロトコルアダプターを使用しています。
- ベンチマークは macOS でローカルに実行されました。デプロイメント固有の結論を導く前に、Linux サーバーでもテストする必要があります。

## 推奨されるフォローアップ測定

次のローカル調査ブランチでは、以下の診断を追加または使用する必要があります。

- `process.memoryUsage()` の起動前後、ツール実行前後、ストリーミング前後、圧縮前後、セッション終了前後
- V8 ヒープ統計とヒープスペース
- アクティブハンドルとリクエスト
- セッションメッセージ数とおおよその保持文字/トークン量
- ツール結果の数、保持されたツール結果の合計サイズ、最大ツール結果サイズ、および大きな出力が UI 履歴またはモデル履歴のどちらに保持されているか
- サブエージェント数と子プロセス/プロセスツリー RSS
- ツール出力のトランケーションと保存出力の復元イベント

これらの測定は、現在の RSS 比較を Qwen Code の内部状態に関連付けることができるように、同じベンチマークマトリックスで収集する必要があります。

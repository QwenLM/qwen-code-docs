Qwen Code Runtime Diagnostics ベンチマークレポート

日付: 2026-05-19

スコープ

この実行では、新しいオプトインのランタイム診断を有効にして、以前の Qwen Code ベンチマークの形状を繰り返します。Qwen Code のみをテストし、Claude Code はテストしません。

初期モデルマトリックス:

  • pai/glm-5
  • qwen3.6-plus

追加の PR サイズフォローアップ:

  • DeepSeek/deepseek-v4-pro（Anthropic 互換プロトコル経由）

ケース:

  • 小さな GitHub PR レビュー: PR #4268
  • コードナビゲーション: 圧縮 / コンパクション関連のコード検索と読み取り
  • 合成ローカル diff: 約 94.6 KiB
  • 合成ローカル diff: 約 968.5 KiB
  • 合成ローカル diff: 約 4.84 MiB

この実行では、診断ブランチのローカルバンドル CLI を QWEN_CODE_PROFILE_RUNTIME=1 と一時的な CLI ホームディレクトリで使用しました。このベンチマークでは、グローバル MCP サーバーとフックは読み込まれませんでした。

重要な注意点: これらの絶対 RSS 値は、以前の PATH 解決された qwen の実行よりも低くなっています。これは、この実行がローカルブランチの node dist/cli.js と、ストリップされた一時的な設定を使用したためです。このレポートは内部診断配布実行として扱い、以前のインストール済み CLI との直接的な RSS 比較の代替としないでください。

インストール済み CLI vs ローカルバンドルの健全性チェック

フォローアップの健全性チェックでは、インストール済み CLI とローカル診断バンドルで、同じ最小限のプロンプト、モデル、非対話モードを使用しました。唯一の意図的な変数は、Qwen Code がストリップされた一時的な CLI ホームを読み込むか、通常のユーザー設定を読み込むかです。

通常設定のピーク時、ローカルプロセスツリーサンプルは、Qwen ルートプロセス単独ではなく、複数の Node/MCP プロセスによって支配されていました:

PR #4186 はローカル診断ブランチに存在しますが、これは V8 ヒーププレッシャー自動コンパクションのセーフティネットです。V8 ヒーププレッシャーが約 70% でトリガーされます。この環境では Node ヒープ制限は約 4.1 GiB ですが、ストリップされたベンチマーク終了時のヒープは約 99-143 MiB でした。これらの数値に基づくと、ストリップ設定の RSS が低いのは、これらのベンチマーク実行中に #4186 がコンテキストを積極的に圧縮したことが原因ではありません。

ベアモード設定の原因特定チェック

2 番目のフォローアップでは、インストール済み CLI とローカルバンドルの両方で、同じ PR レビュープロンプト形状で qwen3.6-plus を使用しました。これは通常のエンドツーエンドのビジネスベンチマークではありません。起動/設定メモリのみを制御された原因特定チェックです。

--bare はランタイム入力を変更します。通常のグローバル設定の検出、MCP 起動、フック、暗黙のコンテキスト、スキル、その他の起動統合をスキップします。そのため、モデルプロバイダーがグローバル設定でのみ構成されている場合、失敗したり異なる動作をしたりする可能性があります。この実行では、モデル認証情報は子プロセス環境を介してのみ提供されました。ベアモードは意図的に通常のプロバイダー設定を読み込まないためです。ユーザーのグローバル設定には何も書き戻されませんでした。

この実行では有用なトークン/ツール呼び出し統計は生成されませんでした。モデルは 1 ターンで完了し、要求されたシェルコマンドを呼び出しませんでした。これらの行を通常のタスクベンチマーク結果として使用したり、上記のマトリックスとトークン/ツール呼び出し動作を比較したりしないでください。これらは、通常の設定と設定された子プロセスからプロセスツリー RSS がどの程度発生するかを見積もるためにのみ役立ちます。

一時設定の MCP / フック分離

--bare はランタイム入力を多く変更するため、通常のベンチマークとして扱えません。そのため、フォローアップでは、通常設定から派生した生成設定ファイルを使用して、一時的な QWEN_HOME ディレクトリを使用しました。実行は通常の設定読み込みパスを維持しましたが、設定の 2 つの次元のみを切り替えました:

  • MCP 無効: mcpServers をクリアし、MCP 許可/除外リストを空にしました。
  • フック無効: disableAllHooks を true に設定。

グローバル設定は変更されていません。このケースでは qwen3.6-plus と最小限の起動プロンプトを使用したため、タスク推論品質ではなく、起動/設定のプロセスツリーコストを測定します。

観察結果

  1. このローカルバンドル実行では、プロセスツリー RSS はルート RSS とほぼ同じです。ルート/ツリー間のギャップは通常 10 MiB 未満です。つまり、これらの実行では永続的な子プロセスメモリ所有者は見られませんでした。支配的なプロセスはメインの Node プロセスです。
  2. ローカルバンドル実行のピークは約 0.36-0.41 GiB であり、以前の 0.83-1.04 GiB ではありません。これは、マトリックスがストリップされた一時設定を使用したためです。フォローアップの通常設定健全性チェックでは、PATH qwen とローカル dist/cli.js の両方で約 1.1 GiB のツリー RSS が再現され、追加メモリはプロセスツリー内の子 MCP/Node プロセスから発生しました。
  3. V8 ヒープは RSS よりもはるかに小さいです。終了時ヒープは約 99-143 MiB ですが、終了時 RSS は約 216-411 MiB です。残りのフットプリントは、読み込まれたモジュール、ネイティブアロケーション、外部バッファ、またはライブ JS ヒープ外のランタイムオーバーヘッドである可能性があります。
  4. 静的リクエストオーバーヘッドは大きく、繰り返し発生します。システムプロンプトはリクエストあたり約 39-51 KiB、ツールスキーマはリクエストあたり約 37-40 KiB です。これは、モデルが複数ターン行う場合、小さなタスクでも累積トークン数が高くなる可能性があることを説明しています。
  5. 大きな diff 出力は、モデルリクエストに到達する前にキャップされます。968 KiB および 4.84 MiB の diff ケースでは、キャプチャされたツール結果は約 772-799 KiB でしたが、リクエスト内でモデルに表示される最大の関数応答は約 25-41 KiB に留まり、最大ワイヤボディは約 105-122 KiB に留まりました。これは、モデル対向パスでトランケーション/保存出力処理が機能していることを示しています。
  6. ワイヤボディが制限されているにもかかわらず、大出力ケースではメモリが依然として増加しています。たとえば、4.84 MiB の GLM 実行では 414.2 MiB のツリー RSS と 410.7 MiB の終了時 RSS に達し、4.84 MiB の qwen3.6-plus 実行では 142.8 MiB のヒープで終了しました。これは、最終リクエストペイロードがキャップされている場合でも、大きなツール出力がローカルキャプチャ、正規化、または保持されたランタイム状態に影響を与える可能性があることを示唆しています。
  7. この実行では、モデルの選択により、RSS よりもターン数とトークン合計が変化しました。qwen3.6-plus は pai/glm-5 よりも平均トークン数とターン数が多かったものの、平均ツリー RSS ピークはわずかに低くなりました。これは、モデルの選択がプロセスメモリの主な説明ではないという以前の結論を裏付けています。

更新された動作推論

新しい診断により、初期の仮説がより正確になりました。

  • インストール済みCLIにおけるユーザーが見る1 GiBのピークは、ローカル診断バンドルで通常構成を使用しても再現可能です。内部のQwenランタイム属性評価にはストリップ版の実行、ユーザーから見えるプロセスツリー属性評価には通常構成の実行を使用する必要があります。
  • ストリップ版と通常構成の最大の差異はプロセスツリーの形状です。通常構成では追加のMCP/Node子プロセスが起動します。これらの子プロセスが、最小プロンプトのサニティチェックにおいて約0.35-0.55 GiBから約1.1 GiBへの絶対的なジャンプの大部分を説明します。
  • --bare フォローアップテストでも qwen3.6-plus で同様の傾向が確認されました。同じプロンプト形状で通常構成はベアモードよりもプロセスツリーRSSが約0.50-0.59 GiB多く消費されますが、ルートRSSの変化はわずかです。
  • 一時設定分離は --bare よりも優れた属性テストです。MCPのみを無効にすると、通常の設定読み込みパスを維持したままプロセスツリーRSSが約0.42-0.47 GiB削減されます。フックのみを無効にした場合、ツール呼び出しなしのケースでは有意なRSS変化は見られません。

  • ローカルのQwenランタイム内部では、もはや「モデルに送信される生の差分バイト数」は疑わしい領域ではありません。モデル向けのリクエスト本文は制限されています。
  • より強い容疑者は、リクエストごとの静的コンテキストコスト、繰り返しリクエストラウンド、ツールスキーマサイズ、モデル側の切り詰め前または外部での大きなツール出力のローカル保持/キャプチャです。
  • RSSがV8ヒープよりもはるかに高いままであるため、次のプロファイリングレイヤーにはモジュール/起動アカウンティング、外部メモリ、ツール実行および最終応答発行前後のヒープスナップショットを含める必要があります。

初期解釈：

  • 未使用の設定済みMCPはメモリを消費する可能性があります。通常の起動では設定済みMCPサーバーに接続し、タスクが必要とする前にそれらのツールをアドバタイズするためです。測定された構成では、chrome-devtools が追加のNode/npm MCPプロセスを起動し、ツールスキーマ数が19から48に増加します。これは大きな起動/構成RSS帯域を説明し、繰り返しリクエストのオーバーヘッドも増加させる可能性があります。
  • 長時間セッションOOMレポートは別のレイヤーです。Mark-Compactがほとんどメモリを解放しないGCログは、ヒープが到達可能な状態で満たされていることを示唆します。最も有力な候補は、保持された履歴/ツール/UIオブジェクト、全履歴クローン、圧縮中間データ、ストリーミング/ログ集約器です。
  • PR #4186 は有用な緩和策です。トークン閾値がトリガーされる前にヒープ圧力に基づいてコンパクションを実行でき、不要な全履歴クローンを1つ除去します。しかし、大きなツール出力保持、UI履歴保持、または圧縮ピークメモリがすでに解決されたという証明として扱うべきではありません。

長時間タスク検証計画

  1. 起動/構成属性評価：通常構成 vs MCP無効 vs chrome-devtoolsのみ vs 再起動なしの属性評価。これにより、有意義な作業が始まる前にユーザーが見るものを説明します。
  2. 長時間タスクの実行時増加：繰り返しツール呼び出し、大きな出力、圧縮、再開、対話型UI履歴。これにより、実際の作業後のOOMを説明します。

検証基準：

  1. 少なくとも主要な長時間タスクケースを2回繰り返すこと。起動RSSには可視のバラつきがあるため、単回実行の結論は避けるべき。

  5. ヒープスナップショットを取得する場合、まず構造化診断JSONを書き、その後スナップショットを取得すること。ヒープスナップショットは大きく、機密文字列を含む可能性があるため、オプトインかつローカルに留めるべき。

対話型長時間レビューの再現

短い非対話型プロンプトが目標ウィンドウ前に終了し続けた後、リモート入力を使った対話型TUIベンチマークが実行されました。CLIプロセスは1つのセッションで存続し、コントローラが一度に1つの実際のPRレビューターンを送信しました。次のターンは、アシスタントがそのターンの完了マーカーを発行した後にのみ送信されました。これにより、短いワンショットプロンプトを長時間タスクの再現として扱うことを避けています。 セットアップ:

  • Qwen Code 0.15.11 をインストール、モデルは qwen-latest-series-invite-beta-v28。
  • 通常の設定から一時的なCLIホームを生成し、MCPとフックの設定を削除。グローバル設定は変更なし。
  • インタラクティブTUIモード、デュアルJSONイベント出力、リモートJSONL入力。
  • 静的PRレビューのみ。プロンプトで依存関係のインストール、ビルド、テスト、Playwright、Docker、その他の長時間の外部ビルドコマンドを禁止。

この決定論的実行からの重要な帰属:

  1. 76.2 MiBの親履歴JSONを削減されたヒープで構築することは成功する。OOMは追加の全履歴クローン/チェックポイントコピーが保持されたときに現れる。
  2. 1つのチェックポイントコピーだけで、OOM前でもRSSを1 GiB近くまで押し上げることができる。
  3. 親ホット履歴から保持されたサブエージェントトランスクリプトを削除すると、同じ生成ワークロードがOOMから136.8 MiBの小さなRSS実行に変わる。これが今のところ最も明確な緩和シグナルである。
  4. この再現器は合成であり、意図的に敵対的だが、長いインタラクティブレビューと同じオブジェクトグラフ形状（親セッション、サブエージェント、大きなツール出力、トランスクリプトマージバック、全履歴クローン圧力）を実行している。

  2. 大きなPRは主にモデルラウンド数とトークン量の点でコストが高かった。8リクエスト、合計386,891トークン。最大Anthropicボディは289.5 KiBで、OpenAI互換実行よりもはるかに大きかったが、RSSは依然として同じローカルバンドル帯域付近に留まった。
  3. 静的Anthropicリクエストコストも顕著である。システムプロンプトは約50.8 KiB、ツールスキーマはリクエストあたり約47.6 KiB。そのため、ラウンドの繰り返しは主要なトークン増幅器となる。
  4. 大きなPRでは、キャプチャされたツール結果が235.0 KiB、リクエスト内の最大関数応答が182.4 KiBになった。これは以前の小さなPR/コードナビゲーションケースよりも高く、RSSが急上昇しなくても、大きなPRはローカルツール結果処理とリクエストアセンブリに圧力をかけることを示している。
  5. DeepSeek実行はモデル選択の結論を強化する。プロバイダー/モデルの選択はターン数、レイテンシ、トークン量、ワイヤーペイロード形状を強く変えるが、ローカルバンドルRSSピークはQwen Codeランタイム形状に支配され、PRサイズに線形に比例しない。

  1. MCPは通常設定の起動RSSを別のベンチマークで説明できるが、この長時間レビューOOM形状の引き金としては可能性が低い。

  3. 即時のOOMトリガーは、ヒープがすでに高い状態での全履歴クローンまたはチェックポイントスタイルの二重シリアライゼーションである可能性がある。
  4. 圧縮は保持履歴を軽減できるが、圧縮自体が最初に大きな履歴をクローンまたはシリアライズする場合、一時的なピークを生み出す可能性がある。

ローカル緩和検証: MCP無効PRレビューケース

2つの対象を絞った緩和策をローカルで適用し、MCP無効PRレビューケースを再実行する前に検証した:

  1. checkNextSpeaker() は現在、最後のキュレーションメッセージのみを getHistoryTail(1, true) で読み取り、そのメッセージのみを次のスピーカーサイドクエリに送信する。次のスピーカープロンプトは直前のモデル応答についてのみ尋ねるため、全履歴を送信するのは不要なクローンとトークン圧力だった。
  2. AgentToolInvocation は、ライブの task_execution.toolCalls 表示内に完全な responseParts 配列を保持しなくなった。実際の応答パーツはトランスクリプト/履歴パスを依然として流れるが、長時間実行中に大きなサブエージェントツール出力の別の完全コピーを保持する代わりに、親UI表示はネストされたツール結果ストリーミングのために制限されたテキストサマリーのみを保持する。
  3. GeminiChat.sendMessageStream() は、パブリックな getHistory(true) を呼び出す代わりに、内部のキュレーション履歴ビューを介してモデルリクエストコンテンツを構築するようになった。パブリックな getHistory() は外部呼び出し元に対して防御的な structuredClone() を依然として返すが、リクエストホットパスはすべてのモデル呼び出しの前に保持されたチャット履歴全体をディープクローンしなくなった。
